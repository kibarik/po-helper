# 05 — Плохие A/B-тесты (4 ловушки)

> **Источник:** [blog.aleksishmanov.ru/a-b-leaks](https://blog.aleksishmanov.ru/a-b-leaks)
> **Дата:** 2026-06-02 · **Раздел:** Аналитика данных · **Slug:** `a-b-leaks`

---

## TL;DR
A/B-тест — инструмент оптимизации известного, **не** проверки новых гипотез с малой базой. Интуиция о данных работает иначе, чем математика за ними (Канеман/Тверская, 1970-е). Без зафиксированного протокола команда систематически принимает решения на основе шума, ощущая при этом «data-driven культуру».

---

## 4 ловушки

### Ловушка 01 — Peeking (остановка в удобный момент)
- **Интуиция:** смотреть каждый день — разумно; если уже значимо, зачем ждать.
- **Почему ломает:** значимость флуктуирует. При досрочной остановке команда ловит именно случайные всплески. Смотреть 20 дней → вероятность хотя бы раз увидеть p<0.05 без реального эффекта растёт с **5% до 54%**.
- **Метрика-индикатор:** % тестов, где фактическая длительность = запланированной. >50% остановлено раньше → система заражена peeking-ом.

### Ловушка 02 — P-hacking (перебор метрик до победного)
- **Интуиция:** первичная не выросла — проверим retention/чек/шаги онбординга.
- **Почему ломает:** при p<0.05 одна из ~20 метрик «победит» случайно. Проблема множественных сравнений. Опаснее всего — выбор primary metric после просмотра данных.
- **Метрика-индикатор:** доля победивших тестов. >40–50% — подозрительно. Зрелые системы: **10–30%**. Win rate >60% — красный флаг.
- Кейс: Spotify публично признавали слишком высокий ранний win rate как плохой знак.

### Ловушка 03 — Novelty Effect (иллюзия от новизны)
- **Интуиция:** +15% кликов в первую неделю → работает.
- **Почему ломает:** пользователи реагируют на новое просто потому что оно новое (особенно 3–7 дней). Через месяц эффект испаряется.
- **Метрика-индикатор:** сравнение эффекта на неделях 1 и 2. Резкое падение = novelty.
- Кейс: кнопка +22% кликов, +8% конверсии за 5 дней → после выкатки вернулось к базе.

### Ловушка 04 — Загрязнение выборки
- **Интуиция:** случайное распределение 50/50 = корректный эксперимент.
- **Почему ломает:** не гарантирует изоляцию. Один юзер видит B на работе, A дома. B2B: один аккаунт — несколько человек в разных группах. Социальные продукты: network interference (контент из B попадает в A). Плюс **Simpson's Paradox**.
- **Метрика-индикатор:** cross-contamination rate. >5% — результат под вопросом.
- Кейс: LinkedIn — переход к **cluster randomization** (по сетевым кластерам, не по юзерам).

## Анатомия: плохой vs хороший тест

| Параметр | ❌ Плохой | ✅ Хороший |
|---|---|---|
| Гипотеза | «улучшит онбординг» | «email на день 3 с конкретным шагом → +% создавших первый проект за 7 дней» |
| Primary metric | не зафиксирована | % создавших проект за 7 дней |
| Длительность | «пока не наберём» | 21 день |
| Sample size | не рассчитан | ~3200/группу (power 80%, α 0.05) |
| Рандомизация | по user ID | по user ID + SRM-тест |
| Промеж. проверки | каждый день | запрещены (Statsig с блокировкой ранней остановки) |

Хороший тест даёт реалистичное ожидание **+3–3,5%**, а не обманчивое +12%.

## Конкурентный контекст
Крупный игрок с миллионной базой получает значимость за 48 часов. Стартап: либо тестирует с недостаточной выборкой, либо ждёт слишком долго. **Третий путь:** понять, какие вопросы решаются A/B, а какие — качественными методами/custdev/наблюдением.

## Цена ошибок
- **False positive:** внедряют неработающее → потеря времени, техдолг понимания (следующие гипотезы на неверных основаниях).
- **False negative:** отклоняют работающее (особенно при ранней остановке с низкой мощностью) — к идее не возвращаются.
- **Накопленный эффект:** 50 тестов/год с peeking + p-hacking → продукт оптимизируется в случайном направлении при ощущении data-driven культуры. **Хуже, чем отсутствие данных.**

## Фреймворк: 6 шагов «как правильно»

1. **Зафиксируйте все параметры до запуска** (письменно, с датой): primary metric (одна), secondary (только контекст), длительность, MDE, sample size. Документ закрыт от редактирования после старта.
2. **Рассчитайте sample size, не угадывайте.** Калькулятор (Evan Miller, Statsig). Вход: базовая конверсия, MDE, power 80%, α 0.05. Если выборка недостижима за разумный срок → A/B не подходит. *Критерий: расчётное время ≤ 4 недель.*
3. **Фиксированный горизонт, не смотрите раньше.** Минимум 2 полных бизнес-цикла (обычно 14 дней) — покрывает эффект дня недели + стабилизацию novelty. Дата окончания в календаре до запуска.
4. **Sequential testing, если нужна гибкость.** SPRT / mSPRT / Always Valid Inference (Johari et al.) — смотреть в любой момент без inflation α. Реализовано в Statsig, Optimizely (Stats Engine), VWO. Нет sequential testing → не смотрите раньше срока.
5. **Проверьте загрязнение до интерпретации.** Cross-contamination, network interference, SRM-тест (Sample Ratio Mismatch). SRM есть → ненадёжно даже при красивом p-value. *Критерий: contamination <5%, SRM пройден.*
6. **Анализ по неделям, не только итог.** Эффект недели 1 в 2× > недели 2 = novelty. *Критерий: эффект недели 2 ≥ 70% от недели 1.*
