# WORKFLOW: Планирование спринта (v1, зафиксировано по кейсу 2026Q3-S1)



> Канонический процесс формирования спринта в JIRA из «образа результата PO» (доска) с переиспользованием беклога.

> Выведен ретроспективно из реальной сессии 2026-07-02 (спринт 2026Q3-S1, GDSLV/TLND).

> Три источника истины: **Доска = образ результата PO**, **Беклог = что уже есть**, **JIRA = система записи**. Процесс = свести их без дублей и без орфанов.



## Принципы

- **Discovery до записи.** Почти весь ретрозафиксированный rework — из-за пропуска discovery. Front-load его.

- **Два прохода: PLAN (vault, обратимо) → BUILD (JIRA, необратимо).** Между ними — обозримый dry-run на согласование PO.

- **Reuse-first.** Ничего не создавать, пока не проверен беклог.

- **Ноль орфанов.** Каждая задача под историей/эпиком и под измеримой поставкой.

- **Человек — на решениях, автономия — внутри стадии.** Чекпоинты только на реальных развилках PO, не на тривиальном.

- **Гигиена.** Секреты — только через MCP, не в чат. Артефакты — под git.

## СТАДИЯ 0 — Discovery окружения (без записей) 🚪gate: пройдена

- [ ] **Vault path**: определить реальную папку `20XX-Q?/Исполнение` (проверить корень vault / вложенный workspace / worktree). Писать только туда, что под git — не терять артефакты.

- [ ] **JIRA — только через MCP** (Atlassian). Нет MCP → остановиться, попросить поднять MCP/VPN. **Токены руками не собирать.**

- [ ] **Слоты спринтов**: список active/future + **их состав (кол-во задач)**. Выбрать/создать ПУСТОЙ слот. Имя+даты зафиксировать один раз.

- [ ] **Метаданные проекта**: валидные issuetype id (Story, типы Sub-task), id поля Epic Link, список **назначаемых** пользователей (роли).

- [ ] **Crosswalk доска↔OKR**: соответствие KR доски и KR из INDEX.



## СТАДИЯ 1 — Интейк образа результата PO

- Разобрать доску (по людям/целям) → перечень требований, привязка к KR, **критерий демо** каждого.

- Диалог с PO по методике (образ результата, демо, кто на что, риски) — по одному вопросу.



## СТАДИЯ 2 — Сверка с беклогом (reuse-first) 🚪gate: reuse-check до любого create

- По каждому требованию: поиск в беклоге (`text ~`, оба проекта) →

  - **REUSE** существующей задачи; **NEW** только если активного дубля нет; **activity-без-тикета** отдельно.

  - Закрытые аналоги не переиспользовать, но показывать как контекст.

- Артефакт: матрица покрытия «требование доски → ключ/NEW».



## СТАДИЯ 3 — Черновик ПЛАН в vault (обратимо)

- По шаблону: **OBJ → KR → История → Подзадача**; capacity по людям; риски/блокеры.

- У каждой истории — **измеримый критерий демо** (Definition of Ready ниже).

- PO подтверждает состав (Main/Stretch, что уводим).



## СТАДИЯ 4 — Сборка в JIRA (необратимо) 🚪gate: dry-run → approve → write, верификация после каждого батча

- 🚪**slot-safety**: не репойнтить непустой слот без явного решения PO.

- 🚪**cascade-preview**: перед добавлением эпика/родителя — показать, сколько детей подтянется.

- Добавить существующие истории в спринт (подзадачи наследуют спринт от родителя).

- Переназначить исполнителей по доске (валидные учётки).

- Снять `On Hold`/`В ожидании` где нужно (через доступный transition).

- Создать NEW как подзадачи под родными историями (валидный тип подзадачи).

- Кросс-проектное переиспользование (напр. TLND↔GDSLV) — допустимо, помечать.



## СТАДИЯ 5 — Группировка под измеримые User Story 🚪gate: орфанов 0

- Найти орфанов: не подзадача, `Epic Link` пуст, не эпик.

- Свести в **измеримые US** (образ результата + критерий демо).

- Привязать: **Epic Link** к эпику (+ link «Relates» на umbrella-US).

- ⚠️ Существующую историю **нельзя** конвертировать в подзадачу через REST (только Move-визард UI) — проектировать группировку через Epic Link сразу.



## СТАДИЯ 6 — Актуализация и финализация

- Закрыть неактуальные подзадачи: родитель закрыт / работа superseded / протухло по дате. Список → согласование PO → закрытие корректным transition.

- (Опц.) Убрать закрытые легаси-деревья из спринта.

- 🚪**activation-gate**: Start sprint — отдельным явным подтверждением, не «заодно».



## СТАДИЯ 7 — Отчёт и хендофф

- Заполнить `ПЛАН-{sprint}.md` реальными ключами + **ссылками JIRA на истории и подзадачи** (не «на словах»).

- Закоммитить артефакты (не оставлять незакоммиченными в вычищаемых путях).

- Чистый handoff (без секретов, без сырого транскрипта). Секреты сессии — ротировать.

- Замкнуть цикл: в конце спринта — `/sprint-fact {sprint}`.

## Definition of Ready истории спринта (усиленная)

1. Образ результата текстом. 2. **Измеримый критерий демо.** 3. Валидный назначенный исполнитель. 4. Заведена в JIRA под эпиком (Epic Link не пуст → «Других задач» нет).



## JIRA know-how (Jira Server/DC, GDSLV/TLND)

- Связь история→эпик = **Epic Link** (`customfield_10101`), НЕ parent.

- Подзадача наследует спринт от родителя; независимо в спринт не встаёт.

- Типы подзадач GDSLV: аналитика `12007`, разработка `10107`, девопс `12702`, тестирование `10109`. Story = `10203`.

- Существующую задачу нельзя сделать подзадачей через REST.

- Connectivity: убрать `HTTP(S)_PROXY/ALL_PROXY` → прямой маршрут, системный curl, PAT bearer; корп-серт валиден напрямую. (Но приоритет — MCP.)
