# 🧩 SUITE — каркас набора скиллов по фреймворку Byndyusoft (методы Шапиро/Бындю)

> Архитектура **всей связки** методов «фреймворка проектирования социотехнических систем» (Byndyusoft) и её оцифровка в **набор отдельных скиллов**. Опирается на deep-research в [[RESEARCH]]. Бизнес-процесс каждого скилла — зеркало `bft-writer` (конвейер ролей, STOP-паузы, ноль галлюцинаций, adversarial-валидация).

---

## 1. Фреймворк как конвейер (результат одного → вход другого)

Методы фреймворка **тесно переплетены**: выход одного идёт на вход следующего [HM-KB]. Discovery-фаза Byndyusoft:

```
┌─ АНАЛИЗ ────────────┐   ┌─ СТРАТЕГИЯ ──────┐   ┌─ ПРОЕКТИРОВАНИЕ ──────────────┐
│ Связанный SWOT      │──▶│ Карта гипотез    │──▶│ Карта процесса-опыта (XPM)    │
│ (ставки)            │   │ (цели→гипотезы→  │   │ (процесс+опыт)                │
│                     │   │  задачи)         │   │ Карта реализации историй (SIM)│
└─────────────────────┘   └──────────────────┘   │ (истории→требования)          │
        Бындю                   Бындю             └───────────────────────────────┘
                                  │                          Шапиро
                                  ▼
                        Дорожная карта / HADI · USM (Шапиро 2019)
```

- **Связанный SWOT** → стратегические ставки → питает гипотезы Карты гипотез.
- **Карта гипотез** → цели/субъекты/гипотезы/задачи → задаёт «зачем» для проектирования.
- **XPM** → процесс-механизм услуги + опыт → «как устроен процесс».
- **SIM** → рабочие истории → требования к реализации.
- Общий инструмент-реализация всех методов — **Социотех** (`sociotech.center`).

> Это и есть «главная особенность — соединение всего опыта воедино» на уровне **всего фреймворка**, а не одной диаграммы.

## 2. Методы и их ядро (что оцифровываем)

| Метод | Автор | Ядро (элементы) | Источник |
|---|---|---|---|
| **Связанный SWOT** | Бындю | 3 области: Внутренние (сильные/слабые) · Внешние (тренды/угрозы) · Стратегические ставки `<действие>, чтобы <ценность>`; связи многие-ко-многим; приоритет позицией | ✅ [SWOT-KB] |
| **Карта гипотез** | Бындю | Цель+метрики(+баланс.) · Субъект+боли/желания(+негативный) · Гипотеза `Если/то/потому что/тогда` · Задача · Блокеры · Заметки; топологии; каденции | ✅ [HM-KB] |
| **XPM** ⭐ | Шапиро | Акторы/агенты · Ключевые точки ● (решение/изменение/контакт) · Линии тока ⎯/┈ · Линии взаимодействия ⦙ · Триггеры/события ◇ · аннотации точек. Нотация 4+5 | ✅ [XPM-KB] |
| **SIM** | Шапиро | Рабочая история из 5 элементов `Н·С·Д·О→О'·Ц` («Н в С выполняет Д, меняя О на О′ для Ц»); матрица истории × формы реализации | ✅ [SIM-KB] |
| **USM** | Шапиро | User Story Mapping (backbone/walking skeleton, срезы релизов) | ⛔ gated (+ Паттон 2014) |
| **Дерево гипотез развития** | Шапиро | проектирование развития потоковой системы (шаг развития) | ⛔ gated |

Полные разборы ядра — в [[RESEARCH]] §3–§5 (для XPM) и в клонах репозиториев (scratchpad).

## 3. Набор скиллов (декомпозиция)

Каждый метод = **отдельный скилл** в `.claude/skills/`, по архитектуре `bft-writer`: `SKILL.md` (роль, конвейер, принципы, индекс ресурсов) + `resources/` (нотация, стандарт, каталог ошибок→гейты, глоссарий) + `examples/` (golden-карты).

| Скилл | Метод | Готовность источника | Приоритет |
|---|---|---|---|
| `xpm-writer` | XPM | 100% | ⭐ флагман |
| `sim-writer` | SIM | 100% | высокий |
| `hypothesis-map` | Карта гипотез | 100% | высокий |
| `linked-swot` | Связанный SWOT | 100% | средний |
| `usm-writer` | USM | ⛔ ждёт источник | отложен |
| `hypotheses-tree` | Дерево гипотез | ⛔ ждёт источник | отложен |
| `sts-suite` (оркестратор) | связка | — | после 2+ скиллов |

**Оркестратор `sts-suite`** связывает артефакты на стыках (SWOT→гипотезы→XPM/SIM), как единый Discovery-конвейер Byndyusoft.

## 4. Общий шаблон скилла (зеркало bft-writer)

Каждый скилл-метод строится по одному лекалу:

```
0 frame     → рамка (что моделируем, цель, модальность как есть/как будет, участники)  [СТОП]
1 …         → наполнение элементов метода (свои для каждого метода)
…           → связывание элементов (стрелки/линии/матрица)                             [СТОП]
N validate  → hard-гейты из «каталога типовых ошибок» метода + Светофор
N+1 deliver → рендер (Excalidraw MCP) + публикация                                     [СТОП]
```

**Общие принципы (из всех источников):**
1. **Ноль галлюцинаций** — каждый элемент ← источник (интервью/наблюдение/регламент), иначе `[УТОЧНИТЬ]`.
2. **Модальность** — всегда фиксировать «как есть» / «как будет» / гибрид.
3. **Мыслительная позиция** — фиксировать «как кто смотрим» (рефлексивный выход).
4. **Метод «набегающей волны»** — наполнять волнами, не за один заход (общее для КГ, SWOT, XPM).
5. **Каталог типовых ошибок = hard-гейты** — у каждого метода свой (напр. XPM: липовая синхронность, система-на-дорожке, глагол/канал/артефакт в имени точки).
6. **STOP-паузы** — human-in-the-loop на границах стадий.
7. **Соединение воедино** — ценность в связях элементов, не в списках.
8. **Атрибуция + AGPL** — методология излагается своими словами со ссылкой на автора; дословный текст источников (AGPL-3.0) не копируется.

## 5. Движок рендера — Excalidraw MCP

Выбран **Excalidraw MCP** (согласовано): в среде доступен, у автора есть готовая **библиотека элементов XPM** (`xpm-library.excalidrawlib` в [XPM-KB]). План: собрать/переиспользовать библиотеки фигур под каждый метод (ключевые точки, линии, карточки гипотез, секторы SWOT) и рендерить схемы программно. Fallback — генерация структуры для Holst/Figma-шаблонов автора или PlantUML (для XPM — кастомная эмуляция нотации).

## 6. Service Blueprint — предшественник XPM

Первый проход (папка `docs/BLUEPRINT/`) описывал **generic Service Blueprint** — исторический предшественник XPM (Wreiner 2009, откуда взят референс `Blueprint 24h`). Эта папка **удалена**: целевой метод — точный **XPM** (человеческое руководство — в [[README]], пошагово — в [[postroenie/README]]). Service Blueprint остаётся упомянут как один из корней метода в [[RESEARCH]] и [[osnovy/chto-takoe-kpo]].

## 7. Порядок сборки (предлагаемый)

1. `xpm-writer` (флагман, источник 100%) — полный скилл + ресурсы + golden-примеры.
2. `hypothesis-map` и `sim-writer` — параллельно (источники 100%).
3. `linked-swot` — короткий скилл (источник компактный).
4. `sts-suite` — оркестратор поверх готовых.
5. `usm-writer`, `hypotheses-tree` — после разблокировки/присылки источников.
