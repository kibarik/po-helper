# 1.2 instruction — Нексус Продукта через product analytics

> Этап **RB-STEP-1 — BIG IDEA** (прелюдия) · Линза: **Product** (Product Sprint → Нексус продукта → состояние → Банч) [S1] VI.1 · Инструкция для **Инженера Продукта нового поколения** (PAF, [S1],[S2]).
> Преемственность метода: исходный раннбук `RB-STEP-1-IDEA/1.2 Нексус Продукта.md` — два сценария (существующий продукт → Current State + Gap; новый → Vision + метрики), ручное заполнение. Эта инструкция: (а) для существующего продукта включает **авто-заливку Current State из product analytics** (data base методики Und-Id-Ex [S1] VI.5), (б) генерацию Vision Кортекс-агентом из Узлов рынка/роста, (в) **авто-gap** как разность Current State и Vision в графе Узлов.

## 🎯 Цель в PAF-оптике

Насытить **Нексус продукта** до CR, достаточного для входа в Product Sprint:
Current State (для существующего продукта) или Vision (для нового) + измеримый **Гэп** [S1] базовая логика.

В методике Und-Id-Ex это **data base**
(есть продукт → извлекаем знания о поведении, находим бутылочные горлышки)
или **null base** (нет данных → Vision с нуля) [S1] VI.5.
Цель — не «описать продукт в Notion», а получить **цифровой профиль продукта** [S2] III.5
(единая точка инфо, аналог claude.md), с которым будут работать агенты Банча.

## 👤 Роль и ответственность

- **Product Engineer** — формулирует Vision (Видение = образ продукта из фич [S1] базовая логика);
  валидирует авто-gap; определяет, какие метрики — mNSM-кандидаты.
- **Product Ops** — подключает product analytics (Amplitude/Mixpanel/и т.п.) к Nexus-Builder
  для авто-заливки Current State; поддерживает рёбра `метрика → Узел`.
- **Discovery/Launcher PM** (если есть, для нового продукта) — вайбкодит прототип-Видение
  как спелый узел-визуализация [S2] III.6.
- **Зона человека:** формулировка **Vision** — стратегический образ продукта,
  не делегируется ИИ целиком.
  Кортекс аугментирует (предлагает альтернативы), инженер выбирает [S2] III.4.

## 🤖 Что делает Кортекс-агент

**Nexus-Builder Agent** (продолжение 1.1):
- Для **data base:** тянет из product analytics текущие метрики
  (DAU/MAU, retention, funnels, Aha-moment) → создаёт `Current-State`-узлы с `timestamp` живых данных.
- Для **null base:** генерирует **кандидат-Vision** из Узлов Нексуса рынка
  (`Gap-рынка`, `Trend`) и Нексуса роста (`Growth-Lever`) —
  как аугментация (3 альтернативы), не финал [S2] III.4.
- **Авто-gap:** Кортекс вычисляет разность между `Current-State`-узлами и `Vision`-узлами →
  создаёт `Gap`-узлы (что отсутствует, что превышает, что расходится).

- **Вход:** product analytics API (data base) или пустой скелет (null base)
  + Узлы рынка/роста для генерации Vision.
- **Выход:** `Current-State`, `Vision`, `Gap`, `mNSM-candidate` (метрика, которую можно монетизировать [S1] Принцип 6).

**Промпт-идея:**
> «Для data base — злей Current State из analytics с `timestamp`.
> Сгенерируй 3 альтернативы Vision из Gap-рынка и Growth-Lever (аугментация, не выбор).
> Вычисли авто-gap. Предложи 2–3 mNSM-кандидата
> с пометкой, монетизируемы ли они (ребро → NPV).»

## 🧠 Что делает человек (метапознание)

- **Чекпоинт:** Vision, сгенерированный Кортексом, — это **челендж-кандидат**, не истина.
  Инженер проверяет: соответствует ли Vision цели бизнес-юнита?
  Не подменяет ли продуктовую линзу стратегической?
- **Veto-право:** mNSM-кандидат должен быть **монетизируемым** [S1] Принцип 6.
  Если Кортекс предложил NSM, который нельзя превратить в деньги (например, «время в ленте»), —
  инженер помечает его `NSM (не mNSM)` и понижает CP.
- **Вопрос:**
  - «Эта метрика — ценность или vanity? Если вырастет в 10 раз — вырастет ли NPV?» [S1] Принцип 6.
  - «Vision из Кортекса — какая из 3 альтернатив ближе к цели бизнес-юнита?»

## 📦 Артефакт = узлы Нексуса

- **Создаются/обновляются:** `Current-State` (data base) или `Vision-draft` (null base),
  `Vision` (финал, за человеком), `Gap`, `mNSM-candidate`,
  `Metric-Tree-seed` (зерно дерева метрик с корнем mNSM).
- **Метаданные:** `источник` (analytics/интервью/синтез), `CP`, `timestamp`
  (для Current State — live), `статус увядания`.
- **Прирост Context Ripeness:** 0% → ≥60% по Нексусу продукта;
  Current State на data base авто-обновляется событийно.

## 🚦 Confidence Point-гейт

- **Критерий перехода к 1.3:** Vision зафиксирован (человек) + авто-gap существует +
  минимум один `mNSM-candidate` с CP≥4.
- **Стоимость риска:** «Если Vision ошибочен —
  сколько циклов Sprint мы потратим на закрытие неверного Гэпа?» — фиксируется в `Vision`-узле.

**Порог CP по типам метрик** (контроль vanity):
- mNSM-кандидат с ребром → NPV: CP может расти до 10;
- NSM без ребра → NPV (vanity): потолок CP = 3;
- метрика разработки (DAU без связи к ценности): CP = 0, отсев [S1].

## 🔭 Связь с Тремя Линзами (3PL)

- **Активная:** **Product Lens** [S1] VI.1.
- **Связь со Strategy Lens:** Vision черпает из `Bet`-узлов рынка (1.1).
- **Связь с Business Lens:** `mNSM-candidate` → потенциал NPV в Нексусе роста (1.3).

## 🛡️ Анти-workslop чекпоинт

- **Workslop-риск:** **Vision как красивый слоган без измеримого gap** [S2] III.7;
  NSM→mNSM-путаница (главный риск D9 для Value-стадий, [reformation] §4.9).
- **Признак:** gap описан качественно («нет collaboration»), без метрик;
  mNSM = vanity-метрика.
- **Правило верификации:** каждый `Gap`-узел обязан иметь метрику-критерий закрытия;
  mNSM проверяется на монетизируемость (ребро → NPV).
  Vision без измеримого Гэпа → CP=0 (слоган, не Видение).
