# 4.1 instruction — Гипотезы через агента

> Этап **RB-STEP-1 — BIG IDEA** (прелюдия) · Линза: **Product** (валидация фич Банча) · Инструкция для **Инженера Продукта нового поколения** (PAF, [S1],[S2]).
> Преемственность метода: исходный раннбук `RB-STEP-1-IDEA/4.1 Генерация гипотез.md` — ручные Hypothesis Cards по шаблону («We believe that [feature] for [segment] will achieve [mNSM outcome] if [success criteria]»), метрики, метод валидации, CP, зависимости. Эта инструкция: генерация **Hypothesis Cards Кортекс-агентом** с обязательной **связкой с mNSM** (в исходнике mNSM в шаблоне, но без числовой связи — [reformation] §1.3 разрыв 3) и трассировкой к Узлам (защита от галлюцинаций [reformation] §1.8).

## 🎯 Цель в PAF-оптике

Для каждой фичи Банча (3.3) сформулировать **Hypothesis Card** как тестируемое утверждение,
связанное с **mNSM** (монетизируемой ценностью [S1] Принцип 6).

Гипотеза — не «что мы проверяем»,
а **способ поднять CP** фичи по stage-gate (Принцип 4 [S1]):
каждая гипотеза = шаг роста доверия в колесе Гилада [S1] VI.7.

В Identify/Execute-фазе Und-Id-Ex [S1] VI.5 гипотезы → тест → рост CP → gate.

## 👤 Роль и ответственность

- **Product Engineer** — формулирует success criteria (измеримые пороги);
  валидирует связку гипотезы с mNSM;
  veto на гипотезы без метрики.
- **Product Ops** — поддерживает шаблон Hypothesis Card как Узел
  с рёбрами к `Feature` и `mNSM-numeric`.
- **Growth Engineer** — валидирует, что success criteria гипотезы
  действительно двигают mNSM (а не vanity).
- **Зона человека:** выбор **проверяемой метрики** —
  что считать успехом — за человеком (инвестиционная логика).

## 🤖 Что делает Кортекс-агент

**Hypothesis-генератор** (режим Scouting Agent [reformation] §1.5):

- Для каждой `Feature` Банча (3.3) генерирует Hypothesis Card
  по шаблону исходника 4.1 (We believe… for… will achieve… if…).
- **Связка с mNSM:**
  success criteria обязан включать метрику,
  связанную ребром с `mNSM-numeric` (не vanity).
- **Трассировка к Узлам:**
  каждая гипотеза — ребро к `Opportunity-Cluster` (3.1)
  → `JTBD-Opportunity`/`Strategic-Opportunity`/`Monetization-Hypothesis`.
  Гипотеза без трассировки → CP=0
  (защита от галлюцинаций [reformation] §1.8).
- Предлагает **метод валидации** по текущему CP фичи
  (smoke для CP 1–3, prototype для 4–6, customer validation для 7–8 — из исходника 4.2).

- **Вход:** `Feature`-узлы Банча (3.3) + `Opportunity-Cluster` (3.1) + `mNSM-numeric` (1.3).
- **Выход:** `Hypothesis-Card`-узлы с рёбрами к `Feature`, `mNSM-numeric`, `Opportunity-Cluster`.

**Шаблон Hypothesis Card** (преемственность из исходника 4.1):
- Hypothesis Statement: «We believe that [feature] for [segment] will achieve [mNSM outcome] if [success criteria]»;
- We'll Know This Is True When — метрика + порог + timeframe;
- Validation Method — smoke/prototype/customer validation;
- Current CP + Risk level;
- Dependencies (must exist / blocked by).

**Промпт-идея:**
> «Для каждой фичи Банча сгенерируй Hypothesis Card.
> Success criteria — метрика с ребром к mNSM-numeric.
> Трассируй гипотезу к Opportunity-Cluster.
> Предложи метод валидации по CP фичи.
> Гипотеза без трассировки → CP=0.»

## 🧠 Что делает человек (метапознание)

- **Чекпоинт:** Кортекс может сгенерировать **правдоподобные, но неверные** гипотезы
  (Harvard × BCG [S2] III.4).
  Инженер проверяет трассировку:
  к какому Узлу ведёт ребро? Есть ли evidence?
- **Veto-право:** гипотеза без ребра к `Opportunity-Cluster` → CP=0;
  success criteria = vanity (DAU без связи к mNSM) → отсев.
- **Вопрос:**
  - «Если эта гипотеза подтвердится — вырастет ли mNSM?
    Какая метрика — leading indicator для NPV?»
  - «Success criteria = "40% retention" — это связано с mNSM или это vanity?»

## 📦 Артефакт = узлы Нексуса

- **Создаются/обновляются:** `Hypothesis-Card`-узлы (по числу фич Банча)
  с рёбрами к `Feature`, `mNSM-numeric`, `Opportunity-Cluster`, `метод-валидации`.
- **Метаданные:** `источник` (Feature + Opportunity), `CP`,
  `success-criteria` (с mNSM-ребром), `метод-валидации`, `timestamp`, `статус увядания`.
- **Прирост Context Ripeness:** +5%
  (гипотезы уточняют связи фич→mNSM).

## 🚦 Confidence Point-гейт

- **Критерий перехода к 4.2:** каждая фича Банча имеет `Hypothesis-Card`
  с success criteria (mNSM-ребро) и методом валидации.
- **Стоимость риска:** «Какой бюджет на валидацию каждой гипотезы?
  Какая гипотеза — самая дешёвая для проверки CP-роста?» —
  приоритизация по стоимости/росту CP.

## 🔭 Связь с Тремя Линзами (3PL)

- **Активная:** **Product Lens** (валидация ценности фичи).
- **Связь с Business Lens:** success criteria → mNSM → NPV.
- **Связь со Strategy Lens:** гипотеза обслуживает `Bet`.

## 🛡️ Анти-workslop чекпоинт

- **Workslop-риск:** **правдоподобные, но неверные гипотезы** (галлюцинации);
  success criteria = vanity без mNSM-ребра.
- **Признак:** гипотеза без трассировки к Узлу;
  success criteria = DAU/registrations (не монетизируемые);
  CP назначен без evidence.
- **Правило верификации:** каждая Hypothesis Card —
  ребро к Opportunity-Cluster (трассировка);
  success criteria — mNSM-ребро; CP=0 без трассировки.
