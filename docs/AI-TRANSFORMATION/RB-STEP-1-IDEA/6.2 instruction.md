# 6.2 instruction — Pre-mortem через агента: Монте-Карло по NPV

> Этап **RB-STEP-1 — BIG IDEA** (прелюдия) · Линза: **над 3PL** (рисковая по всему Банчу) · Инструкция для **Инженера Продукта нового поколения** (PAF, [S1],[S2]).
> Преемственность метода: исходный раннбук `RB-STEP-1-IDEA/6.2 Pre-mortem.md` — brainstorm рисков по 4 категориям (Market/Product/Team/Business), оценка (Likelihood/Impact), Risk Matrix (Critical/High/Medium), Monitoring Plan (metric/frequency/owner/trigger), final Go/No-Go Decision. Эта инструкция: Pre-mortem через **Кортекс-агента с симуляцией Монте-Карло по NPV Банча** (NPV-модель из 1.3, Банч из 4.3), **monitoring-план с триггерами** (event-based пересчёт CP/CR [S1] VI.2), связь с **стоимостью риска** (3.2) и **Principle 4** (управление знаниями через риски [S1]).

## 🎯 Цель в PAF-оптике

Pre-mortem — не «brainstorm, что может пойти не так»,
а **управление рисками как управление знаниями** (Принцип 4 [S1]):
Кортекс через **симуляцию Монте-Карло по NPV Банча** quantifies риск
(какое распределение NPV при варьировании параметров —
CAC/churn/COGS токенов/конверсии?),
строит **monitoring-план с event-based триггерами**
(что мониторить, какой порог → пересчёт CP/CR/Банча).

Final Go/No-Go — это **осознанный отказ или подтверждение gate** [S1] VI.3
на основе quantified риска, не интуитивной Risk Matrix.
Риски — будущее, не потери
(Принцип 4: потери — прошлое, риски — будущее [S1]).

## 👤 Роль и ответственность

- **Product Engineer** — отвечает за Go/No-Go;
  валидирует допущения Монте-Карло;
  определяет триггеры пересмотра.
- **Product Ops** — внедряет monitoring-план с event-based триггерами
  (CP/CR change → re-evaluation).
- **Growth Engineer** — валидирует риски юнит-экономики
  (CAC/churn/COGS-эксплозия [S2] III.6).
- **Portfolio Manager** — сверяет residual-риск с аппетитом к риску портфеля.
- **Зона человека:** **Go/No-Go decision** —
  стратегическое инвестиционное решение, не передаётся ИИ [S2] III.4.
  Кортекс симулирует, человек решает.

## 🤖 Что делает Кортекс-агент

**Pre-mortem Agent** (симуляция + monitoring) [reformation] §1.5:

- **Монте-Карло по NPV Банча:**
  берёт NPV-модель из `NPV-estimate` (1.3) + `Bunch-v2` (4.3),
  варьирует параметры (CAC, LTV, churn, цена, COGS токенов, конверсия)
  по распределениям → распределение NPV
  (P(NPV>0), P(NPV<0), ожидаемый NPV, worst-case).
- Идентификация **«убийц экономики»**
  (параметры с наибольшей эластичностью NPV → главные риски [reformation] §5.9).
- Для AI-продуктов — **стресс-тест COGS ×2/×5**
  (COGS-эксплозия [reformation] §5.9, [S2] III.6).
- **Monitoring-план с триггерами:**
  для каждого критического риска — метрика (из Узлов), порог,
  **event-based триггер**
  (при пересечении порога → переоценка CP фичи / переформирование Банча /
   переcчёт CR Нексуса [S1] VI.2).
- Генерация **Go/No-Go recommendation**
  на основе P(NPV>0) и residual-риска
  (но решение — за человеком).

- **Вход:** `NPV-estimate` (1.3) + `Bunch-v2` (4.3) + `Risk-Cost` (3.2)
  + Узлы юнит-экономики.
- **Выход:** `Monte-Carlo-NPV`-узел (распределение, P(NPV>0)),
  `Risk-Register`-узлы (с триггерами),
  `Monitoring-Plan`-узел,
  `Go-NoGo-recommendation`-узел.

**4 категории рисков** (преемственность из исходника 6.2):
- Market (конкурент, масштаб, цена, регуляция);
- Product (execution, ценность, масштабируемость, качество);
- Team (ключевой сотрудник, навыки, коммуникация);
- Business (CAC, LTV, churn, runway).

**Промпт-идея:**
> «Симулируй NPV Банча Монте-Карлом
> (варьируй CAC/churn/COGS/конверсию).
> Дай P(NPV>0), worst-case, убийц экономики.
> Для AI — стресс COGS ×2/×5.
> Для каждого риска — метрика + порог + event-триггер (CP/CR/Банч переоценка).
> Дай Go/No-Go recommendation (решение за человеком).»

## 🧠 Что делает человек (метапознание)

- **Критический чекпоинт:** NPV-модель как чёрный ящик
  ([reformation] §1.3 разрыв 3) →
  Монте-Карло усиливает ложную точность.
  Инженер проверяет: какие распределения параметров?
  Из бенчмарков или с потолка?
- **Veto-право:** Go при P(NPV>0) = 51% с огромным worst-case →
  инженер вправе потребовать ещё валидацию
  (преждевременное решение — сквозной риск [reformation] Часть VIII).
- **Вопрос:**
  - «P(NPV>0) = 70% — но какие допущения?
    Если COGS токенов ×5 (AI-COGS-эксплозия) — какой NPV?
    Какой метрикой поймём, что риск сработал?»

## 📦 Артефакт = узлы Нексуса

- **Создаются/обновляются:** `Monte-Carlo-NPV`-узел
  (распределение, P(NPV>0), worst-case, убийцы экономики),
  `Risk-Register`-узлы
  (risk + likelihood + impact + метрика + триггер — расширенный из исходника 6.2),
  `Monitoring-Plan`-узел (event-based),
  `Go-NoGo-Decision`-узел (за человеком).
- **Метаданные:** `источник` (NPV-модель + параметры),
  `P(NPV>0)`, `worst-case`, `триггеры`, `CP-связь`,
  `timestamp`, `appetite-to-risk`.
- **Прирост Context Ripeness:** +5%
  (quantified риски — инкремент знания о будущем).

## 🚦 Confidence Point-гейт

- **Финальный gate стадии** ([reformation] §1.6 Fit-точка):
  Go/No-Go из Pre-mortem.
  Критерий Go: P(NPV>0) ≥ порога
  **и** критические риски имеют monitoring-триггеры
  **и** аппетит к residual-риску подтверждён.
  No-Go = осознанный отказ [S1] VI.3.
- **Стоимость риска:** это **центральный** артефакт стоимости риска
  (Принцип 4 [S1]) —
  worst-case NPV = стоимость риска всего Банча.

## 🔭 Связь с Тремя Линзами (3PL)

- Над 3PL: риски по всему Банчу (все три линзы).
- **Связь:** Product Lens (execution risk),
  Business Lens (unit-economics/COGS risk),
  Strategy Lens (`Bet`/market risk).
  Monitoring-триггеры замыкают цикл на следующий Product Sprint [S1] VI.4.

## 🛡️ Анти-workslop чекпоинт

- **Workslop-риск:** **Монте-Карло как ложная точность**
  («P(NPV>0) = 73.4%» при допущениях с потолка);
  pre-mortem как ритуал без триггеров;
  Go без residual-риска.
- **Признак:** распределения параметров без источника
  (не бенчмарк/не гипотеза);
  monitoring без event-триггеров;
  worst-case не исследован.
- **Правило верификации:** параметры — маркировка fact/hypothesis/benchmark;
  monitoring — event-триггеры обязательны;
  COGS-стресс для AI;
  Go — по P(NPV>0) + residual + аппетит.
