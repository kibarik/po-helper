# 3.3 instruction — Bunch Architect Agent

> Этап **RB-STEP-1 — BIG IDEA** (прелюдия) · Линза: **над 3PL** (Банч собирает все три линзы) · Инструкция для **Инженера Продукта нового поколения** (PAF, [S1],[S2]).
> Преемственность метода: исходный раннбук `RB-STEP-1-IDEA/3.3 AI Bunch Formation.md` — промпт к ChatGPT/Claude с контекстом 3 Нексусов + критериями отбора (max mNSM, min time-to-market, fit, synergy) → Bunch v1.0 из 5–7 features с CP и mNSM-impact (low/med/high). Эта инструкция: превращает точечный промпт в **Bunch Architect Agent** ([reformation] §1.3 разрыв 1, §1.5) с **RAG по 3 Нексусам**, **NPV Банча** (комплексный, [S1] VI.6), **Bunch Size/Window**, **anti-workslop фильтр**.

## 🎯 Цель в PAF-оптике

Сформировать **Банч (Feature Bunch)** — связку фич **в моменте** под текущее состояние Нексуса,
чтобы выиграть конкуренцию
(а не «распределить разработку» — Беклог упразднён [S1] Принцип 1, VI.6).

Банч определяется **только текущим состоянием** (Нексусом), не накопленным списком.
В Identify-фазе Und-Id-Ex [S1] VI.5 Кортекс генерирует фич-кандидаты из графа opportunities (3.1).

**Лимиты Банча** [S1] VI.6:
- **Bunch Size** — ограничен скоростью восприятия рынка, не разработки;
- **Bunch Window** — период ожидания бизнес-результата.

**Три критерия отбора** [S1] VI.6:
① макс. mNSM; ② мин. рисков (CP); ③ эффект в окне → **комплексный NPV**.

## 👤 Роль и ответственность

- **Product Engineer** — ставит задачу Bunch Architect
  (порог CP, Bunch Size, Bunch Window);
  валидирует комплексный NPV Банча;
  veto на фичи без mNSM-ребра.
- **Product Ops** — поддерживает **Nexus RAG** (индекс по 3 Нексусам);
  следит, что Банч формируется из текущего состояния, а не из старого.
- **Growth Engineer** — валидирует mNSM-потенциал фич Банча.
- **Зона человека:** **состав Банча** как инвестиционное решение
  (какие фичи максимизируют NPV) — за человеком [S2] III.4.

## 🤖 Что делает Кортекс-агент

**Bunch Architect Agent** [reformation] §1.5 — ядро стадии:

- RAG по 3 Нексусам (рынка, продукта, роста) + граф opportunities (3.1) с CP (3.2).
- Генерация фич-кандидатов из opportunities (Identify-фаза [S1] VI.5).
- Отбор 5–7 фич по 3 критериям [S1] VI.6:
  ① макс. mNSM; ② мин. рисков (CP); ③ эффект в Bunch Window.
- **Комплексный NPV Банча** (сумма NPV фич с учётом синергии) [S1] VI.6 —
  ключевой разрыв vs исходника (там mNSM только low/med/high).
- **Bunch Size** (5–7, по скорости восприятия рынка [S1] VI.6) и
  **Bunch Window** (из исходника 3.3).
- **Anti-workslop фильтр:**
  фичи без mNSM-ребра → отсев;
  фичи с CP < порога → отсев; дубликаты → отсев.

- **Вход:** Nexus RAG (3 Нексуса) + `Opportunity-Cluster` (3.1) + CP (3.2).
- **Выход:** `Feature`-узлы Банча (5–7),
  `Bunch-v1`-мета-узел с комплексным NPV, Size, Window.

**Промпт-идея:**
> «Сформируй Банч из 3 Нексусов.
> Отбор: макс mNSM ∧ CP≥порог ∧ эффект в Window.
> Посчитай комплексный NPV Банча с учётом синергии.
> Bunch Size = 5–7 (по восприятию рынка).
> Anti-workslop: отсев фич без mNSM-ребра или с CP<порога.
> Для каждой фичи — NPV-вклад в Банч.»

## 🧠 Что делает человек (метапознание)

- **Критический чекпоинт:** NPV Банча — потенциальный чёрный ящик
  ([reformation] §1.3 разрыв 3).
  Инженер проверяет: каков NPV-вклад каждой фичи?
  Какие допущения? Что если синергия переоценена?
- **Veto-право:** фича без явного mNSM-ребра (vanity) — отсев,
  даже если CP высокий.
- **Вопрос:**
  - «Банч формирован из текущего состояния Нексуса или из старого списка?
    Bunch Size = 6 — это по восприятию рынка или по вместимости разработки?»
    (должно по рынку [S1] VI.6).

## 📦 Артефакт = узлы Нексуса

- **Создаются/обновляются:** `Feature`-узлы (5–7)
  с `mNSM-ребро`, `CP`, `NPV-вклад`, `Bunch-Window`;
  `Bunch-v1`-мета-узел (комплексный NPV, Size, Window).
- **Метаданные:** `источник` (opportunity[]), `CP`, `mNSM-numeric`,
  `NPV-вклад`, `timestamp`, `Bunch-Window`, `статус увядания`.
- **Прирост Context Ripeness:** +5–10%
  (Банч замыкает связи между Нексусами).

## 🚦 Confidence Point-гейт

- **Критерий перехода к 4.1:** `Bunch-v1` существует,
  комплексный NPV > 0 (базовый сценарий),
  каждая фича имеет mNSM-ребро и CP≥порога.
- **Стоимость риска:** «Какой NPV мы теряем, держа в Банче фичу,
  чья валидация failed?» — drop по NPV-логике.

## 🔭 Связь с Тремя Линзами (3PL)

- Над 3PL: Банч собирает фичи из opportunities всех трёх линз.
- **Связь:** Product Lens (фичи → mNSM),
  Business Lens (NPV Банча), Strategy Lens (Банч обслуживает `Bet`).

## 🛡️ Анти-workslop чекпоинт

- **Workslop-риск:** **Банч как старый беклог под новым именем**
  (накопленный, а не в моменте);
  mNSM = low/med/high без числа; NPV без разложения.
- **Признак:** фича без mNSM-ребра;
  CP фичи > CP opportunity-источника;
  Банч формирован из списка, а не из текущего состояния.
- **Правило верификации:** Банч — только из текущего состояния Нексуса [S1] VI.6;
  каждая фича — mNSM-ребро + NPV-вклад;
  комплексный NPV Банча обязателен.
