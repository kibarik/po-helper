# 2.5 instruction — Nexus Builder (graph, not doc)

> Этап **RB-STEP-2 — CUSTOMER VALIDATION** (Stage 1) · Линза: Product (контекст: Business/Strategy) · Инструкция для **Инженера Продукта нового поколения** (PAF, [S1],[S2]).
> Преемственность метода: исходный раннбук `RB-STEP-2-CUSTOMER-VALIDATION/2.5 Фиксация поведения и контекста потребителя.md` («единственный контекст потребителя» — 8 сущностей, единицы знания, Value Stream, альтернативные пути).

## 🎯 Цель в PAF-оптике
Заменить сборку «единого контекста потребителя» в один статичный документ на **Nexus-Builder: граф узлов (8 сущностей) с метрикой Context Ripeness как вычисляемым gate**. Разрозненные карточки интервью превращаются в связный подграф Нексуса потребителя, чья спелость = критерий перехода дальше [S1] III.1, reformation 2.5. Документация исчезает → цифровой профиль продукта [S2] III.5.

## 👤 Роль и ответственность
- **Product Engineer** — владелец графа; проверяет связность (каждая проблема привязана к пути и триггеру; каждая мотивация — к цели).
- **Product Ops** — поддерживает структуру графа, метрику CR (полнота × актуальность), флаги увядания.
- **Growth Engineer** — добавляет ребро Value Stream → mNSM (дельта ценности как потенциал монетизации).
- **Зона человека**: решение «достаточно ли спел контекст» для перехода к сегментации — стратегический вызов.

## 🤖 Что делает Кортекс-агент
**Nexus-Builder Agent (сборка графа):**
- **Вход**: единицы знания о поведении из 2.4 + карточки интервью + обновлённые CP.
- **Действие**: собирает 8 сущностей в граф (Сегмент → Триггер → Начальное состояние → Целевое состояние → Альтернативные пути → Проблемы+Причины → Мотивация → Value Stream); выявляет ситуации-триггеры, пары «начальное→целевое», проблемы на шагах путей (по структуре исходного `2.5`); считает CR; флагует узлы на увядание; строит ребро Value Stream→mNSM.
- **Выход**: спелый подграф Нексуса потребителя с CR как live-метрикой.

### Сборка по фазам исходного `2.5`
**Phase 1 — единицы знания**: из карточек интервью выписываются только факты реального поведения (не мнения/пожелания). ~15–30 единиц. Выявляются ситуации-триггеры (событие, запускающее потребность) и пары «начальное→целевое состояние» (точка A → точка B).
**Phase 2 — альтернативные пути и Value Stream**: для каждой пары состояний описываются реальные способы добраться до цели (шаг за шагом), с проблемами на шагах и их причинами. Value Stream = дельта ценности текущих решений (что потребитель получает/теряет).
**Phase 3 — связность**: каждая проблема привязана к пути и триггеру; каждая мотивация — к цели; ребро Value Stream→mNSM (Growth Engineer).

### Альтернативные пути (по исходному `2.5`)
Аналогия Москва–Петербург: Сапсан / машина / самолёт / ночной поезд — каждый путь со своими шагами, проблемами, ценой. Задача: описать, как потребители РЕАЛЬНО добираются до цели сейчас. Один путь — один подграф: шаги → кто использует (частота) → проблемы на шагах → причины. Это основа для проектирования ценностного предложения (RB-STEP-4).

### Примеры триггеров и состояний (по исходному `2.5`)
| Триггер | Начальное состояние | Целевое состояние |
|---|---|---|
| Ребёнок получил плохую оценку | Не понимает материал, отстаёт | Понимает, успевает, уверен |
| Экзамен через 3 месяца | Не готов, боится | Готов, сдал |
| Текущий репетитор не подошёл | Без репетитора, нужен новый | С подходящим репетитором |

Один способ решения может использоваться для разных пар состояний → у потребителя несколько контекстов → все фиксируются как отдельные подграфы.

## 🧠 Что делает человек (метапознание)
- **Чекпоинт**: Product Engineer проверяет, что «контекст важнее демографии» — узлы описывают поведение/ситуацию, а не возраст/пол (по тезису исходного `2.5`).
- **Veto-право**: отклонить узлы, основанные на мнениях/пожеланиях («хотел бы, чтобы…»), а не на реальном поведении.
- **Метапознание**: узлы без источника-интервью → workslop.

### Почему граф, а не документ (по [S2] III.5)
Документация исчезает → цифровой профиль продукта (аналог claude.md, единая точка) [S2]. Разрозненный markdown/Sheets-документ: не обновляется, теряет актуальность, дублируется. Граф узлов: обновляется событийно агентами, CR вычисляется автоматически, увядание флагуется, стейкхолдеры работают с одним двойником. «Контекст — новый способ конкуренции» [S1] Принцип 2: без живого двойника Кортекс усиливает мусор.

### Проверка связности графа
Чек-лист Product Engineer: у каждой проблемы есть путь и триггер? · у каждой мотивации — цель? · каждый путь имеет ≥1 проблемы с причиной? · ребро Value Stream→mNSM построено? · нет «висящих» узлов без источника? Если нет — граф неспел, CR ниже порога, возвращаемся к интервью (2.3–2.4).

### 8 сущностей контекста — почему именно столько (по исходному `2.5`)
PAF фиксирует 8 сущностей, потому что они образуют замкнутый цикл «почему человек действует»: Сегмент (кто) → Триггер (что запустило) → Начальное/Целевое состояния (откуда/куда) → Альтернативные пути (как) → Проблемы+Причины (где спотыкается/почему) → Мотивация (зачем) → Value Stream (что получает/теряет). Меньше — теряется причинность; больше — избыточность и workslop. Все 8 = полнота CR.

### Цитата-доказательство как первоклассный узел
По исходному `2.4`+`2.5`, прямые цитаты респондентов — самый ценный материал. В PAF цитата = отдельный узел типа `quote`, привязанный к проблемному/поведенческому узлу ребром `evidence`. Цитата без привязки → workslop. Цитата с ≥3 подтверждениями из разных интервью → высокий CP доказуемого узла. Это превращает «базу цитат» в доказательную базу Нексуса, а не в маркетинговый копи-паст.

## 📦 Артефакт = узлы Нексуса потребителя
Полный граф из 8 сущностей (типы — см. таблицу в [[2.0 instruction]]). Метаданные каждого узла: `источник-интервью[]`, `CP`, `timestamp`, `статус увядания`, `частота`. Ребра: триггер→состояние, путь→проблема, проблема→причина, мотивация→цель, Value Stream→mNSM. **Context Ripeness = полнота × актуальность** — вычисляется.

## 🚦 Confidence Point-гейт
- **Критерий**: CR Нексуса потребителя ≥ порога (все 8 сущностей представлены, узлы верифицированы, CP фокус-проблем ≥ gate).
- **Стоимость риска**: «Сколько стоит проектирование продукта на неспелом контексте (отсутствует, например, альтернативный путь или причины)?»

## 🔭 Связь с Тремя Линзами (3PL)
**Product Lens** — Нексус потребителя = вход в Product Sprint (Understand). **Business Lens** — ребро Value Stream→mNSM связывает с бизнес-линзой. Подаёт контекст в Strategy Lens через узлы рынка.

## 🛡️ Анти-workslop чекпоинт
- Запрет на «портрет среднего пользователя» (мешанина сегментов) → только поведенческие узлы конкретных контекстов.
- Цитаты-доказательства обязательны для каждого проблемного узла.
- CR не растёт от «дописывания текста» — только от добавления верифицированных узлов.

## 🔄 Преемственность: классика → PAF
| Было (исходный `2.5`) | Стало (PAF) |
|---|---|
| «Единый контекст потребителя» в одном документе | **Граф узлов** (8 сущностей), цифровой двойник |
| Context Ripeness = самооценка | CR = **вычисляемая метрика** (полнота × актуальность) |
| Ручная сборка после всех интервью | **Событийная сборка** агентом внутри цикла (2.4→2.5) |
| Фиксация = дописать раздел | Фиксация = **commit узлов** в Нексус |
| Нет «увядания» | Флаги `статус=увядает` → авто-refresh |
| Документация как артефакт | **Цифровой профиль продукта** (документация исчезает) |

## 📚 Ключевые термины PAF на этом шаге (по naming_conventions)
- **Нексус (Nexus)** — живая модель/цифровой двойник; контекст для решений [S1] III.1.
- **Context Ripeness** — полнота × актуальность; метрика качества Нексуса [S1].
- **Узел (Node)** — атомарная единица Нексуса с метаданными (источник, CP, timestamp, статус увядания).
- **Value Stream** — поток ценности текущих решений; дельта = потенциал продукта.
- **mNSM** — монетизируемая NSM; ребро Value Stream→mNSM связывает с Business Lens [S1].
- **Прозрачность до очевидности** — Принцип 1: рост доверия требует роста прозрачности (полнота + непротиворечивость + доступ стейкхолдеров) [S1].

## 📖 Пример альтернативного пути как подграфа (по исходному `2.5`)
**Путь: «Поиск через интернет-площадки»** (AI-репетитор):
- Шаги: заход на Profi.ru → просмотр 30+ анкет → фильтрация по цене → написать 3–5 → ждать ответа → пробный урок → выбор.
- Кто использует: большинство родителей без рекомендаций (8/12).
- Проблемы на шагах: «просмотр 30+ анкет» → проблема «не может объективно сравнить», причина «все анкеты одинаковые, нет реальных кейсов»; «ждать ответа» → проблема «теряет темп», причина «репетиторы отвечают нерегулярно».
- Value Stream: что получает — широкий выбор; что теряет — время, уверенность в выборе. Ребро → mNSM: если продукт даёт рекомендацию + критерии сравнения, дельта ценности высокая → потенциал монетизации.

## ❓ Контрольные вопросы (метапознание)
1. У каждой проблемы есть путь и триггер? (связность графа)
2. У каждой мотивации — цель?
3. У каждого пути ≥1 проблемы с причиной?
4. Построено ли ребро Value Stream→mNSM (Growth Engineer)?
5. Нет ли «висящих» узлов без источника-интервью?
6. Описывают ли узлы контекст/поведение, а не демографию?
7. CR ≥ порога для перехода к сегментации (2.6)? Если нет — вернуться к интервью.

## ⚠️ Типичные ошибки 2.5
- **Граф с «висящими» узлами** (проблема без пути) → CR ниже порога, возвращаемся к 2.3–2.4.
- **Демография как ядро профиля** → контекст первичен («контекст важнее демографии»).
- **Value Stream без ребра mNSM** → теряется связь с Business Lens (Growth Engineer).
- **CR растёт от «дописывания текста»** → только от верифицированных узлов.
- **Нет увядания** → устаревшие узлы не флагуются; Product Ops следит за `статус=увядает`.
- **«Портрет среднего пользователя»** → мешанина сегментов; сегментация (2.6) разделит.
- **Цитаты без привязки к узлам** → workslop; цитата = узел `quote` с ребром `evidence`.
- **Меньше 8 сущностей** → теряется причинность цикла «почему человек действует»; полнота CR падает.
