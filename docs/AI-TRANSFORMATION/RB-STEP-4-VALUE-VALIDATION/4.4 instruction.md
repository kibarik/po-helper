# 4.4 instruction — Фиксация need/value fit как commit в цифровой профиль продукта

> Этап **RB-STEP-4 — VALUE VALIDATION** (Stage 3) · Линза: Product (контекст: Business) · Инструкция для **Инженера Продукта нового поколения** (PAF, [S1],[S2]).
> Преемственность метода: исходный раннбук `RB-STEP-4-VALUE-VALIDATION/4.4 Фиксация ценностного предложения (need-value fit).md`.

## 🎯 Цель в PAF-оптике
Зафиксировать подтверждённое ценностное предложение как **commit в цифровой профиль продукта** (Нексус): не «запись в базе знаний», а версионированный узел VP/VPC с доказательной базой, который становится **якорем** всего дальнейшего продуктового процесса. **Need/value fit = CP ≥ gate**: пора прекратить ставить под сомнение само ЦП и работать с ним как с фундаментом.

## 👤 Роль и ответственность
- **Product Engineer** — формулирует VP в широком и узком смысле (Остервальдер), заполняет Value Proposition Canvas (VPC), определяет границы применимости и anti-scope, версионирует (v1.0 — первая фиксация; v1.1 — уточнение формулировки; v2.0 — смена сути). Синхронизирует команду: каждый участник формулирует VP «своими словами», расхождения выявляют скрытые предположения.
- **Product Ops** — поддерживает версионирование узлов Нексуса, следит за фазой «увядания» (если с валидации прошло >3 месяцев — флаг переподтверждения), настраивает коммиты в цифровой профиль как аналог claude.md для продукта.
- **Зона человека**: стратегическое решение «достигнут ли need/value fit» — опирается на конвергенцию ≥2 источников (количественный + качественный), ответственность не делегируется.

## 🤖 Что делает Кортекс-агент
**Агент-фиксация в Нексус (версионирование, anti-workslop)**:
1. Сводит результаты экспериментов из 4.3 в единую таблицу: эксперимент / метод / сегмент / метрика / цель / факт / подтверждено.
2. Расставляет доказательства по **иерархии достоверности**: пилот (использовал и готов продолжать) > оплата/предзаказ > заявка > клик/CTR > цитата.
3. Генерирует VPC (Customer Profile: Jobs/Pains/Gains ↔ Value Map: Products & Services/Pain Relievers/Gain Creators) с трассировкой каждого Pain Reliever на Pain, Gain Creator на Gain.
4. Формирует **commit** VP-node + VPC-node + доказательная база + границы + ключевые цитаты в цифровой профиль продукта (Нексус).
5. **Anti-workslop-фильтр формулировок VP**: широкий смысл фокусирует на профите (изменении состояния), не на фичах; Pain/Gain подтверждены интервью (не придуманы).
6. Триггерит обновление связанных узлов: CP гипотезы → статус «Подтверждено, зафиксировано»; связи в концепт продукта.
7. Флаг «увядания»: если доказательная база устарела — требует переподтверждения перед коммитом.

### Иерархия доказательств (по убыванию достоверности — из ранбука 4.4)
1. 🏆 **Пилот / образец** — потребитель реально использовал продукт и готов продолжать.
2. 🥈 **Оплата / предзаказ** — «проголосовал рублём».
3. 🥉 **Заявка / оставил контакт** — выразил намерение.
4. 4️⃣ **Клик / CTR** — проявил интерес.
5. 5️⃣ **Цитата из интервью** — словесное подтверждение.

Зачем важно: если есть только CTR-подтверждение, уверенность в VP ниже, чем при предзаказах/пилоте. Это влияет на уровень CP и формулировку VP (с оговорками или без).

### Value Proposition Canvas (Остервальдер — трассировка ценности на потребность)
Суть: **левая часть (карта ценности) должна соответствовать правой части (профиль клиента)**. Минимально достаточный VPC: 3–5 Customer Jobs, 3–5 Pains, 3 Gains + соответствующие Pain Relievers и Gain Creators. Гнаться за точностью, а не за полнотой: 3 подтверждённых интервью Pain лучше 10 придуманных.

| Профиль клиента (Customer Profile) | Карта ценности (Value Map) |
|---|---|
| **Customer Jobs** — что клиент делает / хочет достичь (функциональное, социальное, эмоциональное) | **Products & Services** — что конкретно предлагаем |
| **Pains** — что мешает / болит / риски / опасения | **Pain Relievers** → как снимаем каждый Pain (с привязкой) |
| **Gains** — выгоды, которых ищет; результат «вау» | **Gain Creators** → как создаём каждый Gain (с привязкой) |

**Fit Score** по каждой потребности: закрыта VP? (да/нет) · сила закрытия (высокая/средняя/низкая).

### Границы применимости (anti-scope)
Фиксировать строго для подтверждённого сегмента. Для второго сегмента VP — пока гипотеза, отдельная валидация, не смешивать подтверждённое и предполагаемое в одном узле. Что явно НЕ входит в VP — отдельное поле (anti-scope), чтобы избежать scope creep.

## 🧠 Что делает человек (метапознание)
- **Отличать «подтверждено» от «нам кажется»**: need/value fit достигнут при конкретных измеримых сигналах — конверсия достигла порога, клиенты пилота готовы платить/продолжать, ≥2 источников в одном направлении. Один слабый сигнал → CP «Низкий», планируется доп. валидация.
- **Различать фиксацию VP и упаковку для инвесторов**: сначала зафиксировать суть (что подтверждено), потом «упаковать». Не путать с питчем.
- **Ловить расхождения в команде**: если участники формулируют VP по-разному — это сигнал о разных сегментах/ценностях, нужна сессия с данными, а не «продавливание» формулировки.
- **Границы применимости**: фиксировать строго для подтверждённого сегмента; для второго сегмента VP — пока гипотеза, отдельная валидация.
- **Veto** на фиксацию без иерархии доказательств: только CTR-подтверждение → уровень уверенности «Низкий».

## 📦 Артефакт = узлы Нексуса продукта
- **VP-node** (версионированный) — широкий смысл (Остервальдер) + узкий смысл (с метрикой) + статус «✅ Подтверждено — need/value fit» + дата + сегмент + версия.
- **VPC-node** — Customer Profile (Jobs/Pains/Gains) ↔ Value Map (Products & Services/Pain Relievers/Gain Creators) + Fit Score по каждой потребности.
- **Evidence-node** — сводная таблица валидации + ключевые цитаты с источником/датой.
- **Boundary-node** — для какого сегмента, при каких условиях, что НЕ входит (anti-scope).
- **Context Ripeness** по подшагу: целевое ≥70%.

## 🚦 Confidence Point-гейт
**Need/value fit = CP ≥ gate**. Критерий: по иерархии доказательств есть сигнал уровня «оплата/предзаказ» или «пилот с готовностью продолжать» (а не только CTR/заявка) + конвергенция ≥2 источников. CP подтверждённой гипотезы проходит gate → узлы коммитятся в Нексус как v1.0. Если подтверждено несколько VP — фиксируются все, выбор финального на 4.5 (вторичный скоринг).

## 🔭 Связь с Тремя Линзами (3PL)
**Product Lens**: зафиксированный VP = сердцевина концепта продукта и будущего Видения (Vision), якорь Product Sprint. Связь с **Business Lens**: узкий смысл VP уже содержит метрику — это кандидат на mNSM (проверка монетизируемости на 4.5). Связь со **Strategy Lens**: VP с границами применимости определяет, на какой сегмент/рынок делается стратегическая Ставка.

## 🧭 Модель зрелости Кортекса на этом подшаге
- **Фаза 1 (отдельные задачи):** Кортекс формирует сводную таблицу валидации и черновик VPC по шаблону, инженер редактирует.
- **Фаза 2 (процессы команд):** агент-фиксация коммитит VP-node + VPC-node + Evidence-node в Нексус с версионированием; anti-workslop-фильтр проверяет формулировки; флаг увядания срабатывает при устаревании доказательной базы.
- **Фаза 3 (процессы компании):** commit в цифровой профиль продукта — единая точка истины (аналог claude.md); обновление связанных узлов концепта/карты продукта происходит автоматически; версия v2.0 (смена сути VP) триггерит переподтверждение.

## ❓ Типовые ошибки (метапознание-чекпоинты)
- **«Подтверждено три VP — что фиксировать?»** → зафиксировать все с доказательной базой; финальный выбор — на 4.5 (вторичный скоринг). В концепте временно: «рассматриваем несколько VP».
- **«Результаты неоднозначны»** → разобрать каждый эксперимент отдельно; проверить, не связан ли провал с исполнением (плохой лендинг, неверная аудитория); если ЦП в целом подтверждено, но оффер-формулировки не работают — фиксировать VP, отмечая эффективные формулировки.
- **«Команда не соглашается с формулировкой»** → не «продавливать»: вернуться к доказательной базе (цитаты, цифры); попросить каждого предложить свою формулировку; найти общее. Расхождение в сути → сигнал о разных сегментах/ценностях, нужна сессия с данными.
- **«Давно не валидировали — данные устарели»** → если >3 месяцев: проверить рынок (конкуренты, тренды), 3–5 проверочных интервью; актуально — фиксировать с пометкой «переподтверждено [дата]»; устарело — не фиксировать, вернуться к валидации.

### Success Criteria подшага
- Сводная таблица валидации заполнена; вывод о подтверждении сделан.
- VP в широком и узком смысле сформулированы (Остервальдер).
- VPC заполнен с трассировкой Pain Relievers→Pain, Gain Creators→Gain.
- VP-node, VPC-node, Evidence-node, Boundary-node коммичены в Нексус как v1.0.
- Команда синхронизирована: каждый формулирует VP «своими словами», расхождения выявлены.
- Context Ripeness по подшагу: ≥70%.

## 🛡️ Анти-workslop чекпоинт
- Документация исчезает → артефакт = **узел цифрового профиля** (Нексус), не разрозненный документ. Запрещённый синоним: НЕ «документация по продукту» / «запись в базе знаний» → **commit в цифровой профиль продукта**.
- VPC заполняется точно: 3–5 подтверждённых интервью Jobs/Pains лучше 10 придуманных.
- Формулировки VP не перекладывают когнитивную нагрузку — профит, а не фичи.
- Версионирование обязательно: любое изменение VP после v1.0 обосновывается новыми данными, не «нам так кажется».
- Увядание контекста: данные >3 месяцев → флаг переподтверждения, иначе фиксация не коммитится.

## 📐 Привязка к 6 принципам PAF
- **Принцип 1 (Прозрачность)**: версионированный VP-node — единый якорь для всей команды; расхождения в формулировках выявляют скрытые предположения.
- **Принцип 2 (Совместная унификация контекста)**: commit в Нексус, а не в личный документ; стейкхолдеры работают с одним артефактом.
- **Принцип 4 (Риск = знание)**: need/value fit = CP ≥ gate — момент, когда риск «не того ЦП» достаточно снижен, чтобы работать с ним как с фундаментом.
- **Принцип 6 (Прибыль через ценность)**: узкий смысл VP с метрикой — кандидат на mNSM (проверка на 4.5).

## 🧪 Сквозной пример (из ранбука 4.4)
**Repeto** — сервис подбора репетиторов. После валидации (4.3): конверсия в заявку 11,2% (пор. 8%), 6 из 8 семей пилота готовы платить, CTR таргета 3,8% (пор. 2,5%).

**Иерархия доказательств:** пилот (6/8 готовы продолжать) + оплата (готовность платить) — верхние уровни → уровень уверенности «Высокий».
**VP (широкий смысл):** Наш сервис Repeto помогает родителям школьников 7–11 класса, которые хотят исправить оценки ребёнка, но тратят слишком много времени на поиск специалиста, тем, что подбирает репетитора за 24 часа по параметрам предмета, расписания и стиля обучения, в отличие от Авито или «сарафанного радио».
**VP (узкий смысл):** Мы считаем, что подбор за 24 часа с гарантией первого урока будет интересен родителям 7–11 класса, что подтверждает конверсия в заявку 11,2% (пор. 8%), ведь 6 из 8 семей пилота готовы платить за продолжение.
**VPC (фрагмент):** Pain «трачу 2–3 недели на поиск» ↔ Pain Reliever «алгоритм подбора за 24 часа»; Pain «не знаю, насколько хорош репетитор» ↔ Pain Reliever «верификация дипломов + пробный урок».
**Commit:** VP-node v1.0, статус «✅ Подтверждено — need/value fit», сегмент «родители 7–11 кл.», anti-scope: «результаты ребёнка в школе — за границами VP».

### Версионирование VP в Нексусе
- **v1.0** — первая фиксация после need/value fit (CP ≥ gate).
- **v1.1** — уточнение формулировки без смены сути VP.
- **v2.0** — значимое изменение VP (новый сегмент, другой механизм ценности) — триггерит переподтверждение и mNSM-переформулировку.

Любое изменение VP после v1.0 обосновывается новыми данными из валидации, а не «потому что нам так кажется». Commit в цифровой профиль продукта — единая точка истины (аналог claude.md), обновление связанных узлов концепта/карты продукта происходит автоматически.

### Синхронизация команды (ритуал после коммита)
Фиксация VP в узле — половина дела. Чеклист синхронизации:
- Встреча/синк с командой (30–60 мин).
- Показ доказательной базы — почему именно это VP подтверждено (цитаты, цифры).
- Обсуждение вопросов и сомнений.
- **Форма «своими словами»**: каждый участник пишет одно предложение «Мы помогаем [кому] [что делать / чего достичь]». Сравнить ответы. Если сильно расходятся — вернуться к формулировке (расхождение в сути → сигнал о разных сегментах/ценностях).
- Явное «да» от всех стейкхолдеров.

Если VP подтверждено только для одного сегмента — фиксировать строго для него; для второго сегмента VP — пока гипотеза, отдельная валидация, не смешивать подтверждённое и предполагаемое в одном узле.
