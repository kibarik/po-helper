# Системные требования: bft-fast (СА / FNR-высота)

Высота ниже БФТ: **как** устроен `/bft-fast`. Каждое СТ трассируется к ФТ/ИТ/НФТ upstream-БФТ. Не дублирует «что/зачем» — только механику.

## 0. Границы системных требований

**В зоне СТ:** новый навык `bft-fast` (промпт-конвейер поверх Claude Code command), парсинг Summary, синтез 3 блоков письма, сборка и вывод, гейт галлюцинаций, резолв конфига.

**Вне зоны СТ (не реализуем в MVP):**
- Запись в JIRA/Confluence (письмо — только в чат/файл).
- Генерация полного БФТ-документа канон-структуры MTS.
- Запуск deep-pipeline `bft-*` (глубина — отдельный шаг PO после ответов).
- Реальная отправка письма в мессенджер/почту через MCP.
- Фактический per-step выбор модели и замер tokens/sec (#112) — только конфиг-хук.

## 1. Компонентная архитектура

Навык — промпт-конвейер (как существующие `bft-*`), без исполняемого кода. Артефакты репо:

```
.claude/
├── skills/bft-fast/
│   ├── SKILL.md                    роль, принципы, порядок этапов, резолв конфига
│   ├── resources/
│   │   ├── letter_format.md        шаблон письма: 3 блока + правила голоса
│   │   ├── extraction_schema.md    схема сущностей извлечения из Summary
│   │   └── demo_synthesis.md       как строить How to demo (E2E от лица пользователя)
│   └── examples/
│       ├── golden_summary.md       эталонный вход (Приложение A upstream-БФТ)
│       ├── golden_letter.md        эталонный выход (Приложение A upstream-БФТ)
│       └── thin_summary.md         тонкий Summary — крайний случай (§12)
└── commands/bft-fast.md            точка входа /bft-fast <источник>
```

Конвейер `/bft-fast` (один проход, без STOP-пауз):

```
resolve-source → extract-entities → synth(How to demo | Вопросы | Границы) → assemble-letter → emit(chat [+file])
```

Стиль/голос переиспользует `.claude/skills/bft-writer/resources/writing_style.md` по ссылке (не копия).

## 2. Интерфейс команды (СТ, ← ИТ-1/ИТ-2)

| СТ | Требование |
|---|---|
| СТ-1 | `/bft-fast <источник>` — единственный аргумент. Формы: (а) путь к `#summary`-заметке; (б) путь к файлу транскрипта; (в) инлайн-текст; (г) пусто |
| СТ-2 | Резолв источника: путь существует и это `.md` со `#summary` → режим «заметка»; путь существует иначе → «транскрипт»; непустой текст без пути → «инлайн»; пусто → спросить у PO путь/текст (не выдумывать) |
| СТ-3 | Опц. 2-й аргумент `[epic_slug]` для имени файла-копии; не задан → slug из главной темы Summary (`{ГГГГ-ММ-ДД}-{slug}`), как у `/summary` |
| СТ-4 | `description:` frontmatter команды соответствует стилю прочих `bft-*`; per-command `model:` не задаётся (недоступно) — см. СТ-24 |

## 3. Извлечение сущностей из Summary (СТ, ← ФТ-1)

| СТ | Требование |
|---|---|
| СТ-5 | Парсер читает Summary и извлекает по `extraction_schema.md`: `проблема`, `для_кого`, `что_делаем`, `участники[]` (ФИО + роль), `договорённости[]`, `затронутые_сущности[]`, `out_of_scope_hints[]` |
| СТ-6 | Каждая сущность несёт `source` — цитату/якорь из Summary. Сущность без источника не создаётся (идёт в вопросы, СТ-14) |
| СТ-7 | `участники[]` матчатся с ростером `capacity.roster_source` (`GROUND/NEXUS/team`) для полного ФИО/роли; не сматченное имя не додумывается (берётся как есть из Summary) |
| СТ-8 | Режим «транскрипт»: сначала внутренняя выжимка до эквивалента `#summary` (переиспользовать логику навыка `summary`), затем извлечение; не парсить сырой транскрипт напрямую |

## 4. Синтез блока «How to demo» (СТ, ← ФТ-2, НФТ-2)

| СТ | Требование |
|---|---|
| СТ-9 | Строит нумерованный E2E-сценарий от лица пользователя: операционные глаголы 1-го лица («открываю», «добавляю», «вижу»); шаги в порядке пользовательского пути |
| СТ-10 | Каждый шаг демо ← факт из `что_делаем`/`договорённости`/`затронутые_сущности`; шаг без факта-источника не выдумывается |
| СТ-11 | Последний шаг — верификация результата (внешняя, если применимо: поиск/индексация/интеграция), извлечённая из Summary; нет данных о верификации → шаг опускается, не выдумывается |
| СТ-12 | Демо покрывает только in-scope сущности; сущности из `out_of_scope_hints` в демо не попадают (уходят в «Границы», СТ-16) |

## 5. Синтез блока «Открытые вопросы» (СТ, ← ФТ-3, НФТ-3)

| СТ | Требование |
|---|---|
| СТ-13 | Открытые точки (нерешённые развилки, неясности, `out_of_scope_hints`, факты без источника) → конкретные вопросы-развилки, не общие формулировки |
| СТ-14 | Вопросы группируются по стейкхолдеру-владельцу: заголовок `ФИО:` + список `- [ ]`. Владелец определяется по теме вопроса ↔ роль участника (роутинг как People Graph); неоднозначно → блок `[кому?]:` |
| СТ-15 | Каждый факт, который не удалось заякорить (СТ-6/СТ-10), обязан появиться как вопрос — тихого пропуска нет (No Silent Skip) |

## 6. Синтез блока «Границы» (СТ, ← ФТ-4, НФТ-2)

| СТ | Требование |
|---|---|
| СТ-16 | Строка in-scope: перечень `затронутые_сущности[]`. Строка out-of-scope: `out_of_scope_hints[]` с явной формулировкой «… не входят в зону БФТ» |
| СТ-17 | Операционные заметки (сроки/эффекты, напр. «правки появляются в течение суток») — из `договорённости[]`, отдельными строками |
| СТ-18 | Пусто по out-of-scope в Summary → строку не выдумывать; оставить только in-scope + заметки |

## 7. Сборка и вывод письма (СТ, ← ФТ-5, НФТ-4)

| СТ | Требование |
|---|---|
| СТ-19 | Сборка в один текст по `letter_format.md`, порядок: `How to demo` → `Открытые вопросы` → `Границы`; markdown/plain, копируемый в мессенджер без правок |
| СТ-20 | Без служебной обвязки (frontmatter, метки пайплайна, `[СТ-*]`) в теле письма; голос по `writing_style.md`, без AI-примет |
| СТ-21 | Вывод — текст в чат PO. Опц.: копия в файл `{bft_store}/{epic_slug}/letter.md` (создать папку при отсутствии); путь конфиг-резолв, при отсутствии `bft_store` → рядом с источником + `[УТОЧНИТЬ путь]` |
| СТ-22 | Финал — короткая подсказка PO о развилке: «ответы получены → запусти глубину отдельно»; сам deep-запуск команда не делает (граница §0) |

## 8. Гейт нулевых галлюцинаций (СТ, ← НФТ-5)

| СТ | Требование |
|---|---|
| СТ-23 | Перед выводом — самопроверка: каждое утверждение письма ← сущность с `source` ИЛИ оформлено как вопрос. Найдено утверждение без источника и не-вопрос → переоформить в вопрос, не публиковать как факт |

## 9. Роутинг модели (СТ, ← ФТ-7, #112)

| СТ | Требование |
|---|---|
| СТ-24 | MVP: fast-lane реализован **дизайном** — один лёгкий проход без тяжёлых стадий; фактической смены модели нет (per-command `model:` недоступен в Claude Code command) |
| СТ-25 | Зарезервировать конфиг-хук `models.fast_lane` в `domain-profile` (опц.); SKILL.md документирует его как точку расширения #112. Отсутствует → текущая модель сессии, без ошибки |

## 10. Поток данных

```

        │
        ▼
   Summary (заметка│транскрипт→выжимка│инлайн)
        │

        │                       │(source на каждой)

        ├──▶ [Вопросы СТ-13..15] ◀── участники (атрибуция)
        └──▶ [Границы СТ-16..18]
                 │
                 ▼
        [assemble СТ-19..20] ──▶ [gate СТ-23] ──▶ [emit СТ-21..22] ──▶ письмо в чат PO [+ файл-копия]
```

## 11. Конфигурация (резолв на «Этапе 1» команды)

| Ключ | Источник | Дефолт / фолбэк |
|---|---|---|

| `paths.bft_store` | domain-profile | рядом с источником + `[УТОЧНИТЬ]` |
| `capacity.roster_source` | domain-profile | `GROUND/NEXUS/team`; нет → имена как в Summary |
| `stakeholders` | domain-profile | для сверки ролей владельцев вопросов; нет → `[кому?]` |

Профиль отсутствует → `domain-profile.template.md` + честные `[УТОЧНИТЬ]`, без остановки.

## 12. Обработка ошибок / negative flows

| Ситуация | Поведение |
|---|---|
| Источник пуст | Спросить путь/текст (СТ-2), не выдумывать Summary |
| Файл не читается / не найден | Сообщить путь-ошибку, попросить корректный источник; не эмулировать содержимое |
| Тонкий Summary (мало фактов) | Собрать письмо из имеющегося; недостающее — целиком в «Открытые вопросы»; не добивать выдумкой |
| Нет участников в Summary | Вопросы — под `[кому?]`; НФТ-3 соблюдён через явную пометку |
| Нет domain-profile | Дефолты §11 + `[УТОЧНИТЬ]`; работа не блокируется |
| Нет out-of-scope в Summary | «Границы» = только in-scope + заметки (СТ-18) |

## 13. Артефакты репо (что создаём/меняем)

Создаём: `skills/bft-fast/{SKILL.md, resources/letter_format.md, resources/extraction_schema.md, resources/demo_synthesis.md, examples/golden_summary.md, examples/golden_letter.md, examples/thin_summary.md}`, `commands/bft-fast.md`.
Меняем (аддитивно, опц.): `domain-profile.template.md` — зарезервировать `models.fast_lane` с комментарием.
Не трогаем: существующие `bft-*` команды и `skills/bft-writer/*` (НФТ-6).

## 14. Верификация (как проверяем перед Production)

| Проверка | Критерий |
|---|---|
| Голден-прогон | `/bft-fast` на эталонном Summary (событие/вопрос-ответ) → письмо структурно совпадает с Приложением A (3 блока, порядок, чекбоксы, «не входят в зону БФТ») |
| Гейт галлюцинаций | Внедрить в тест-Summary факт-дыру → в письме она как вопрос, не как утверждение (СТ-23) |
| Атрибуция | Вопросы привязаны к участникам Summary; участник без роли → `[кому?]` (НФТ-3) |
| Пересылаемость | Тело письма копируется как есть, без frontmatter/меток (НФТ-4) |
| Аддитивность | Существующий `bft-*` прогон не изменился (НФТ-6) |
| Время | Прогон ≤5 мин на типовом Summary (НФТ-1) |
