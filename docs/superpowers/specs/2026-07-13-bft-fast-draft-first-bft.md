# [БФТ] bft-fast: письмо-результат за 5 минут из Summary (How to demo + вопросы + границы)

> Итерация **MVP** эпика «БФТ-процесс: быстрый draft-first, приёмка-first, итеративный, тиринг+гейты».
> В этой итерации: **R1 (#113)** draft-first инверсия (результат = одно письмо в чат) + **R2 (#112)** тиринг как требование скорости; частично подтянут **R3 (#98)** — How to demo = образ приёмки/scope.
> Вне итерации (следующие MVP+1..+3): полный R3-формат детализации (#98), R4 итер-дробление (#97), R5 DoR-гейт (#96), R6 строгое ревью (#89), генерация полного БФТ-документа канон-структуры MTS.

## 1. Бизнес описание

**Образ результата.** PO даёт на вход Summary встречи (заметка `/summary` или ссылка/файл транскрипта) и запускает `/bft-fast`. За **≤5 минут** на выходе — **одно готовое письмо в чат**, которое PO пересылает коллегам как есть. Письмо = краткий БФТ из трёх блоков:

1. **How to demo** — нумерованный E2E-сценарий демонстрации от лица пользователя (что открыть, что сделать, что увидеть, как проверить результат, вплоть до внешней верификации).
2. **Открытые вопросы** — сгруппированы **по стейкхолдеру** (ФИО из встречи), под каждым — чекбокс-вопросы для уточнения именно к этому человеку.
3. **Границы** — что в зоне ответственности БФТ и что явно **вне** («… не входят в зону БФТ»), плюс операционные заметки (напр. «правки появляются в течение суток»).

**Ядро = How to demo + Границы как ограничитель scope.** Сценарий демо очерчивает, что демонстрируем и принимаем — то и в зоне ответственности; блок «Границы» явно отсекает лишнее. Незакрытое Summary становится не «дырой», а **адресным вопросом** к конкретному стейкхолдеру.

**Развилка.** Быстрый путь заканчивается письмом. **Всё остальное — глубинная проработка (полный сбор контекста и т.д.) — идёт только после ответов на вопросы письма.** `/bft-fast` на этом свою работу завершает; запуск глубины — отдельный последующий шаг PO, когда ответы получены.

**Зачем делать.** Убрать 30-минутное ожидание и слепую глубину наперёд: 80% ясно из встречи, письмо собирается быстро и сразу очерчивает границу ответственности через демо. Метрика — время `конец встречи → готовое письмо` (цель ≤5 мин против ~30 мин) и доля кейсов, закрытых быстрым путём без глубокого прогона.

<details><summary>Подробный контекст</summary>

Текущий `bft-writer` — 7-стадийный тяжёлый pipeline (context-gen → problem → concept → debate → draft → validate → deliver), STOP-пауза между стадиями. Глубина применяется ко всему сразу и с самого начала: PO ждёт полный прогон прежде чем увидит форму документа. Инверсия: **сначала быстрый полный черновик из известного (Summary), глубина — только адресный fallback, если не попали.** Deep-pipeline не выбрасывается — переиспользуется, запускается от уже согласованной PO формы, а не с нуля вслепую.

Тиринг моделей (#112): быстрый проход draft-first — классический *fast-lane* (минимальная задержка, PO ждёт вживую); fallback deep-проработка — *deep-lane* (тяжёлая модель, большое окно, PO не ждёт вживую). В этой итерации тиринг входит как требование скорости (НФТ) + техническая развилка, без замера каталога моделей tokens/sec (это отдельная работа #112).

</details>

## 2. Проблема

- **ДЛЯ КОГО:** PO, оформляющий БФТ по итогам встречи с бизнес-заказчиком.
- **ЧТО ДЕЛАЕМ:** быстрый вход `Summary → одно письмо в чат` одним проходом: How to demo + Открытые вопросы (по стейкхолдерам) + Границы. Письмо пересылается коллегам как есть. Глубина — отдельным шагом после ответов.
- **AS-IS:** первый артефакт БФТ появляется через ~30 мин; глубина применяется вслепую наперёд ко всему объёму; scope ответственности не очерчен до конца проработки.
- **ПРОБЛЕМА:** долго до первого артефакта; PO простаивает в ожидании тяжёлого прогона; глубина тратится там, где хватило бы демо-скоупа; заказчик получает детальный документ вместо образа приёмки и границ.
- **TO-BE:** за ≤5 мин — письмо, очерчивающее scope через How to demo + Границы, с адресными вопросами к стейкхолдерам; глубокая проработка запускается только после ответов.

| Шаг | Было (As-Is) | Стало (To-Be) |
|---|---|---|

| Глубина | всегда, по всему объёму, наперёд | отдельным шагом, только после ответов на вопросы |

## 4. Бизнес-Требования

### Вводные для разрабатываемого функционала (открытые вопросы)

- Порог достаточности Summary — минимум, при котором draft-first «попадает». В MVP: не блокируем, недостающее помечаем `[УТОЧНИТЬ]`, критерий достаточности PO оценивает глазами.
- Критерий «попали / не попали» — в MVP решает **PO вручную** на развилке. Авто-чек полноты по бизнес-процессу — следующая итерация (стыковка с R4 #97).
- Атрибуция вопросов по стейкхолдерам — если во встрече роль/владелец вопроса не ясны, вопрос идёт в блок без адресата с пометкой `[кому?]`.
- Вывод письма в MVP — текст в чат (готов к копипасту) + опц. копия в файл рядом с рабочей папкой; авто-отправка в мессенджер/почту через MCP — вне MVP.
- Выбор конкретных fast/deep моделей и их замер — вне MVP (#112); берём модель fast-lane из конфигурации, при отсутствии — текущая модель сессии.

## 9. Зависимости

- Навык `summary` — источник входа (формат заметки `#summary`, участники/ФИО для атрибуции вопросов).
- Навык `bft-writer` (`.claude/skills/bft-writer`) — общий принцип нулевых галлюцинаций; deep-проработка, запускаемая PO отдельным шагом после ответов.

- #112 (тиринг моделей) — выбор fast-модели; в MVP из конфига, без замера; при отсутствии — модель сессии.

## 10. Риски

- **How to demo не ограничивает scope** (демо раздут / всё подряд) → scope-эффект теряется. Митигация: НФТ-2 — каждый пункт демо ← факт Summary; лишнее в «Границы»/вопросы.
- **Вопрос без владельца** → PO не знает, кого спросить. Митигация: НФТ-3 — 100% вопросов адресны либо `[кому?]`.
- **Галлюцинации при быстром проходе** (fast-модель слабее) → НФТ-5 гейт: факт без источника → вопрос, не выдумка.
- **Письмо неудобно пересылать** (служебная обвязка, разметка ломается в мессенджере) → НФТ-4 самодостаточный текст без обвязки.

## Приложение A. Эталонное письмо (от PO)

Референс формата и голоса результата `/bft-fast`. Структура: `How to demo` (нумерованный E2E) → `Открытые вопросы` (по стейкхолдерам, чекбоксы) → `Границы` (in/out + заметки).

```
How to demo:
1. Открываю в Direct карточку события (потом площадки, потом персоны) → есть раздел «Вопрос-ответ».
2. Добавляю вопрос, правлю, меняю порядок, один скрываю.
3. Захожу на страницу на сайте → вижу опубликованные вопросы; скрытый не показан.
4. На странице без вопросов блока нет, страница работает как обычно.
5. Проверяю в Google Search Console / Яндекс.Вебмастер, что вопросы попали в поиск (в течение суток обновится)

Открытые вопросы:

- [ ] Надписи покупателю: «возврат оформляется» (сразу после ухода) → «возвращён» (когда вернул) — формулировки ок?
- [ ] Если билет вернуть нельзя — показываем надпись вместо кнопки — ок?
- [ ] Что показать покупателю, пока возврат ещё не подтвердился с их стороны?
- [ ] Есть какие-то требования к метрикам кроме функционала?

Дмитрий Ананьев:
- [ ] вопрос
- [ ] вопрос

Фамилия Имя:
- [ ] вопрос

Границы: событие, площадка, персона (заполнение от MRS и FAQ для категорий - не входят в зону БФТ).
Правки на сайте появляются в течение суток.
```

**Наблюдения для генератора:**
- How to demo — от лица пользователя, операционными глаголами («открываю», «добавляю», «вижу»); заканчивается внешней верификацией результата, если применимо.
- Вопросы — не общие, а конкретные развилки решения («кнопка Вернуть — ок?»), сгруппированы по владельцу.
- Границы — одна строка in-scope сущностей + явное «не входят в зону БФТ» + операционная заметка (сроки).
- Голос — деловой, короткий, без AI-примет (см. `writing_style.md` из bft-writer).
