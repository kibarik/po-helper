# [БФТ] bft-fast: Draft-first генерация БФТ за 5 минут из Summary

> Итерация **MVP** эпика «БФТ-процесс: быстрый draft-first, приёмка-first, итеративный, тиринг+гейты».
> В этой итерации: **R1 (#113)** draft-first инверсия + **R2 (#112)** тиринг как требование скорости.
> Вне итерации (следующие MVP+1..+3): R3 приёмка-first формат (#98), R4 итер-дробление (#97), R5 DoR-гейт (#96), R6 строгое ревью (#89).

## 1. Бизнес описание

**Образ результата.** PO даёт на вход Summary встречи (заметка `/summary` или ссылка/файл транскрипта) и запускает `/bft-fast`. За **≤5 минут** на выходе — **полный черновик БФТ** по канонической структуре MTS (Бизнес описание → Проблема → БТ/ПТ/ИТ/ФТ/НФТ → Зависимости/Риски), а не заготовка. Всё, что Summary не покрыл, помечено естественной прозой `[УТОЧНИТЬ у <кто>]`. PO правит форму и стоит на явной **развилке**: «форма ок → отправить в JIRA» (быстрый путь закрыт, глубокий прогон не нужен) либо «форма не попала → запустить существующий deep-pipeline `bft-*` от уже согласованной формы».

**Зачем делать.** Убрать 30-минутное ожидание первого артефакта и слепую глубину наперёд: 80% требований уже ясны из встречи, форму нужно собрать быстро. Метрика — время `конец встречи → готовый черновик` (цель ≤5 мин против ~30 мин) и доля кейсов, закрытых быстрым путём без глубокого прогона.

<details><summary>Подробный контекст</summary>

Текущий `bft-writer` — 7-стадийный тяжёлый pipeline (context-gen → problem → concept → debate → draft → validate → deliver), STOP-пауза между стадиями. Глубина применяется ко всему сразу и с самого начала: PO ждёт полный прогон прежде чем увидит форму документа. Инверсия: **сначала быстрый полный черновик из известного (Summary), глубина — только адресный fallback, если не попали.** Deep-pipeline не выбрасывается — переиспользуется, запускается от уже согласованной PO формы, а не с нуля вслепую.

Тиринг моделей (#112): быстрый проход draft-first — классический *fast-lane* (минимальная задержка, PO ждёт вживую); fallback deep-проработка — *deep-lane* (тяжёлая модель, большое окно, PO не ждёт вживую). В этой итерации тиринг входит как требование скорости (НФТ) + техническая развилка, без замера каталога моделей tokens/sec (это отдельная работа #112).

</details>

## 2. Проблема

- **ДЛЯ КОГО:** PO, оформляющий БФТ по итогам встречи с бизнес-заказчиком.
- **ЧТО ДЕЛАЕМ:** быстрый вход `Summary → полный БФТ-черновик` одним проходом + явная развилка «в JIRA / в глубокую проработку».
- **AS-IS:** первый артефакт БФТ появляется через ~30 мин; глубина применяется вслепую наперёд ко всему объёму, даже когда форма уже понятна из встречи.
- **ПРОБЛЕМА:** долго до первого артефакта; PO простаивает в ожидании тяжёлого прогона; глубина тратится там, где хватило бы быстрого черновика.
- **TO-BE:** полный черновик БФТ за ≤5 мин из Summary; PO правит и решает — быстрый путь (в JIRA) закрыт сразу, либо адресный запуск deep-pipeline от согласованной формы.

## 3. Изменение в UJM

| Шаг | Было (As-Is) | Стало (To-Be) |
|---|---|---|
| Старт БФТ | `/bft-context-gen` → ждать тяжёлый сбор контекста | `/bft-fast <summary>` → полный черновик за ≤5 мин |
| Первая форма документа | после 4+ стадий pipeline | сразу, на первом проходе |
| Правки PO | по ходу многих STOP-пауз | на едином черновике, один цикл |
| Глубина | всегда, по всему объёму, наперёд | опционально, адресно, от согласованной формы |
| Отправка в JIRA | в конце `/bft-deliver` | сразу после правок PO, если форма попала |

## 4. Бизнес-Требования

### Вводные для разрабатываемого функционала (открытые вопросы)

- Порог достаточности Summary — минимум, при котором draft-first «попадает». В MVP: не блокируем, недостающее помечаем `[УТОЧНИТЬ]`, критерий достаточности PO оценивает глазами.
- Критерий «попали / не попали» — в MVP решает **PO вручную** на развилке. Авто-чек полноты по бизнес-процессу — следующая итерация (стыковка с R4 #97).
- Точка отправки в JIRA — черновик уходит после правок PO. Нужна ли лёгкая валидация перед JIRA (усечённый `/bft-validate`) — открытый вопрос, в MVP по умолчанию без неё, но развилка это допускает.
- Выбор конкретных fast/deep моделей и их замер — вне MVP (#112); берём модель fast-lane и deep-lane из конфигурации.

### Ценность разрабатываемого функционала

| ID | Ценность | Связанные |
|---|---|---|
| БТ-1 | PO получает полный черновик БФТ за ≤5 мин после встречи, не простаивая в ожидании тяжёлого прогона | #113 |
| БТ-2 | Глубокая проработка тратится адресно — только когда быстрый черновик не попал, и стартует от согласованной формы, а не с нуля | #113 |
| БТ-3 | Интерактивная фаза идёт на быстрой модели (минимальная задержка), автономная глубокая — на тяжёлой в фоне | #112 |

## 10. Риски

- **Draft-first «не попадает» слишком часто** → быстрый путь не экономит время. Митигация: развилка всегда даёт откат в deep-pipeline от согласованной формы; замер доли быстрых закрытий (метрика БТ-1).
- **Галлюцинации при быстром проходе** (fast-модель слабее на генерации) → НФТ-4 гейт `[УТОЧНИТЬ]`, факт без источника не выдумывается.
- **Расхождение staging-формата** `/bft-fast` и `bft-*` → deep-pipeline не продолжит с черновика. Митигация: ФТ-3 пишет в ту же рабочую папку эпика.
