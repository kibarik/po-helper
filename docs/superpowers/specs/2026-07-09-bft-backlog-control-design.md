# BFT Backlog Control — design spec

**Статус:** дизайн согласован в брейншторме, реализация не начата.
**Ветка:** `feat/bft-backlog-controll`.
**Дата:** 2026-07-09.
**Триггер:** ревью с внешним PO по требованиям. Нужно, чтобы работа с БФТ велась через Backlog.md — по пайплайну и отдельно под контролем процесса, чтобы PO не терял ни один БФТ из операционного контекста.

Контроль БФТ-требований — важнейшая часть, ради которой создан po-helper. Этот инструмент делает состояние каждого БФТ видимым на доске.

---

## 1. Что уже есть (исходное состояние)

- **Backlog.md = операционный штаб.** Конвенция `backlog-ops.template.md`: модель «1 задача = 1 артефакт», стадии пайплайна = чек-лист AC, статусы `To Do / In Progress / Done`. Инициализируется `install.sh` (`backlog init … --integration-mode mcp`), MCP-сервер `backlog` в `.mcp.json`.
- **Пайплайн генерации БФТ = навык `bft-writer`.** 10 стадий, каждая — отдельная команда в `.claude/commands/bft-*.md`, своя роль/артефакт, STOP-пауза для PO. Внутри команд **нет ни одного упоминания Backlog** сегодня.
- **Контекст БФТ живёт** в `bft_documentation/<epic>/` (`value.md`, `bft-context-pack.md`, `problem.md`, `concept.md`, `constraints.md`, `validation.md` + финальный `<epic>.md`).
- **Механика Backlog.md:** статусы кастомизируются через `backlog/config.yml`; `backlog task edit` умеет `--status`, `--check-ac/--uncheck-ac`, `--priority high|medium|low`, `--append-notes` (повторяемый). **Отдельного треда комментариев в Backlog.md нет** — «комментарии» кладём в notes задачи через `--append-notes`.

Визуальный референс пайплайна: `docs/bft-pipeline/` (диаграмма потока + жизненного цикла).

## 2. Ключевые решения

### 2.1 Два измерения
- **Статус задачи Backlog = контроль процесса** (жизненный цикл, 5 состояний).
- **Чек-лист AC = стадии пайплайна** (value…deliver) — как сейчас.

### 2.2 Пять статусов контроля
`To Do → In Progress → Wait for Review → Accepted → Done`

| Статус | Значение | Переход |
|---|---|---|
| **To Do** | взят на контроль, пайплайн не начат | `backlog task create -s "To Do"` (парковка будущего БФТ, чтоб не забыть) |
| **In Progress** | идёт пайплайн (value…validate) | старт `/bft-value` (find-or-create задачи) |
| **Wait for Review** | черновик у внешнего PO на приёмке требований | `/bft-validate` 🟢/🟡 |
| **Accepted** | внешний PO одобрил; **все правки внесены и учтены в БФТ** | ветка «одобрено» (см. 2.4) |
| **Done** | опубликовано в JIRA + Confluence | `/bft-deliver` |

**Wait for Review = приёмка внешним PO** — это заказчик, который принимает результат работ. Одно состояние, один рецензент.

### 2.3 Прикрепление к пайплайну (не отдельный процесс)
Никакого нового навыка и новых команд. Доска — операционное зеркало существующего пайплайна `bft-writer`.

- В `bft-writer/SKILL.md` — одна секция **«§ Синхронизация с доской»** с точными вызовами (`backlog`/`mcp__backlog__*`): find-or-create задачи, `--check-ac`, `-s <status>`, `--append-notes`. Единственный источник истины по механике.
- Каждый из 11 командных файлов `.claude/commands/bft-*.md` получает **одну строку** в конце «Инструкции для LLM»: «Синхронизация с доской — по протоколу SKILL.md §, стадия = `<name>`, статус = `<status>`». Минимальный след, без дублирования.

### 2.4 Развилка приёмки (Wait for Review) — обязательный аудит
После `/bft-validate` 🟢/🟡 статус → **Wait for Review**, черновик уходит внешнему PO.

- **ОДОБРЕНО** → **обязательно** `--append-notes` «одобрил {кто} · {когда} · замечания {…}». Статус → **Accepted** **только когда все правки внесены и учтены в БФТ**. Одобрение с замечаниями само по себе ≠ Accepted, пока замечания не внесены.
- **ОТКАЗАНО** → **обязательно** `--append-notes` «отказал {кто} · какие правки на следующую итерацию {…}». Затем: `--uncheck-ac 9` (снять `validate`; backlog 1.44 принимает индекс, не имя), `--priority high`, статус → **In Progress** (новая итерация с повышенным приоритетом).

Замечания ревью также питают существующий цикл Lessons Learned (`bft-writer/resources/review_feedback.md`).

### 2.5 Хранение контекста
Не заводим дублирующий индекс-файл. Реестр БФТ на контроле = доска Backlog (задачи label `bft`) + папки `bft_documentation/<epic>/`.
- **Обзор всех на контроле** → нативная `backlog board` / `backlog task list --plain | grep -i "БФТ"` (в backlog 1.44 `task list` не фильтрует по label — отбор грепом по заголовку).
- **Фокус/восстановление контекста** → `backlog task view <id> --plain` (статус + чек-лист стадий + notes «где остановились») ⨯ папка эпика. Карточка доски = точка восстановления.
- **«Где остановились / чего ждём»** и аудит-лог статусов — в notes задачи (`--append-notes`). Контент требований не дублируем.

### 2.6 Конфигурация
- `backlog/config.yml`: статусы `To Do, In Progress, Wait for Review, Accepted, Done` (общий список доски; Wait for Review/Accepted реально использует только label `bft`).
- `install.sh`: выставляет эти статусы при init, идемпотентно.
- `backlog-ops.template.md`: дополнить BFT-жизненным циклом (5 состояний) + маппинг статус/стадия + правило обязательного аудита на приёмке.

## 3. Границы (YAGNI)
- 5-состояний контроль — **только BFT**. OKR/sprint/request/release не трогаем (остаются `To Do/In Progress/Done`).
- Никаких хуков, новых MCP-серверов, новых команд/навыков.
- «Комментарии» = `--append-notes` (треда комментариев в Backlog.md нет).
