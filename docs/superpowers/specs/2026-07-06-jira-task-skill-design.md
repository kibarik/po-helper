# Дизайн навыка `jira-task`

**Дата:** 2026-07-06
**Ветка:** `feat/jira-task`
**Статус:** утверждён к реализации

## Проблема

PO формулирует задачу для исполнителя в свободной форме (чат, почта) и хочет
оперативно перенести её в JIRA в правильном, единообразном виде. Ручное
перекладывание текста в шаблон и заполнение полей JIRA — рутина.

## Цель

Навык `jira-task`: берёт сырой текст, раскладывает по корпоративному шаблону
описания задачи, при нехватке критичных полей проводит короткое интервью,
показывает превью и по явному «ок» PO создаёт Issue в JIRA.

## Роль

Ты — **оформитель задачи PO**. Превращаешь свободный текст в структурированную
задачу и заносишь её в трекер. Соблюдаешь два принципа проекта:
- **Нулевой допуск к галлюцинациям** — каждое поле либо из текста/ответа PO,
  либо `[УТОЧНИТЬ у {кого}]`. Ничего не выдумывать.
- **Read-only до явного «ок» PO** — JIRA не пишем, пока PO не подтвердил превью.

## Вход / Выход

- **Вход:** `/jira-task [PROJ] [текст]` — сырой текст задачи. Текст можно
  вставить в вызов или следующим сообщением. `PROJ` — опц. ключ проекта.
- **Выход:** созданный JIRA Issue — ключ + URL. Файл-артефакт не пишем
  (навык «оперативный»).

## Пайплайн (5 шагов)

### 1. Parse
Разложить сырой текст по 5 полям шаблона:
`Образ результата`, `ASIS`, `ПРОБЛЕМА`, `TOBE`, `Описание`.
Дополнительно вывести `Summary` (заголовок Issue) из «Образа результата» —
краткая императивная формулировка.

### 2. Gate-интервью
- **Критичные поля:** `Образ результата`, `ПРОБЛЕМА`, `TOBE`.
  Если пусты после парсинга → задать 1–3 точечных вопроса PO, дождаться ответа.
- **Некритичные поля:** `ASIS`, `Описание`.
  Пусты → пометить `[УТОЧНИТЬ у {кого}]`, вопрос не задавать.

### 3. Classify
- **Тип Issue:** `Task` по умолчанию; `Bug`, если текст явно про дефект/поломку
  (ошибка, не работает, регресс, падает и т.п.). При неоднозначности — Task,
  и упомянуть предположение в превью.
- **Проект:** аргумент `PROJ` → иначе `tracker.projects[0]` →
  иначе `bft.default_project` из `.claude/domain-profile.md`.
- **Labels:** пусто по умолчанию (не выдумывать).

### 4. Preview (такт 1)
Показать PO: `project_key`, `issue_type`, `Summary` и собранное описание
в шаблоне. Явно спросить подтверждение. **JIRA read-only на этом шаге.**

### 5. Create (такт 2)
По явному «ок» PO:
`jira_create_issue(project_key, summary, issue_type, description, labels)`.
Вернуть ключ + URL созданного Issue.

**Обработка недоступности MCP:** если сервер `tracker.mcp` (напр. `MCP_DOCKER`
/ `atlassian`) недоступен или инструменты `jira_*` отсутствуют — честно вывести
`[УТОЧНИТЬ: MCP <tracker.mcp> недоступен]`, отдать готовый payload (поля для
ручного создания) и НЕ эмулировать успех.

## Формат описания (тело Issue)

Строго по шаблону:

```
Образ результата:
<краткое описание образа результата>

ASIS: <описание текущего состояния>
ПРОБЛЕМА: <ключевая проблема, почему текущее решение не устраивает>
TOBE: <целевое состояние>

Описание:
<остальной дополнительный контекст для решения>
```

Незаполненное некритичное поле → `[УТОЧНИТЬ у {кого}]`.

## Конфигурация (из `.claude/domain-profile.md`)

- `tracker.type` — `jira` (иначе навык предупреждает, что тип трекера не jira).
- `tracker.mcp` — имя MCP-сервера с инструментами `jira_*`.
- `tracker.projects` / `bft.default_project` — дефолтный проект.
- `tracker.base_url` — для формирования URL Issue в выводе (если MCP не вернул).

## Границы (YAGNI)

- Не создаём Epic/подзадачи/связи — только один плоский Issue.
- Не назначаем assignee/sprint/story points — не выдумываем то, чего нет в тексте.
- Не пишем файл-артефакт и не логируем — задача оперативная.
- Не редактируем существующие Issue — только создание.

## Примеры кодов

Коды проектов (`PROJ`), ключи Issue — иллюстративные. Реальные значения —
из `.claude/domain-profile.md`.
