# Spec — External Request Intake (Pipeline обработки внешних запросов) · v1

> **Цель:** автоматизировать первичную обработку внешних запросов к командам под управлением PO. Сегодня PO делает это руками: первичная встреча → custdev-распаковка ценности → разрыв ожиданий As-Is/To-Be → SMART-задача → занесение в JIRA. Дорого по времени. Этот пайплайн снимает с PO первичный скоринг и оформление БФТ-ready запроса.
> **North star:** отдельная **domain-grounded intake-система с веб-чатом** — *Repowise, но для продуктовых требований, а не кода*. Любой внешний стейкхолдер приходит, общается с экосистемой, и система — потому что обучена контекстом конкретной команды — распаковывает запрос, скорит его и выдаёт структурированное требование.
> **Status:** design v1 approved → Фаза 1 (движок) **реализована**. Дата 2026-06-29. Ветка `feat/external-request-processing`. Скилл `request-intake` (SKILL + 7 команд `req-*` + 6 resources + 2 examples) собран; зарегистрирован в README и `domain-profile.template.md` (`paths.intake_workspace`).
> **Ground:** строится на существующих конвенциях po-helper (sa-helper FNR-pipeline: стадия = команда + роль + артефакт + STOP), GROUND Vault (NEXUS-граф), CORTEX, `domain-profile.md`, и upstream к `bft-writer` (`/bft-context-gen`).

## 1. Решения (зафиксированы в обсуждении с PO)

| Развилка | Решение |
|---|---|
| Что строим первым | **Design-док всей системы** (этот документ) → затем движок-скилл → затем кластер/веб. |
| Слой 1 — движок | **Скилл `request-intake`** внутри po-helper. Multi-step pipeline по конвенции дома (команда+роль+артефакт+STOP). Работает в CLI сегодня, портируется в веб позже. |
| Модель интервью | **Транспортно-агностичный** промт интервью. Стадия `/req-interview` элиситит у «запрашивающего», кто бы ни был за клавиатурой (PO-прокси в CLI сегодня; внешний стейкхолдер в веб-чате завтра). Тот же логический промт, разный транспорт. |
| Routing на выходе | **Да, first-class.** Скоринг выдаёт решение: *консультация-и-закрыть / отклонить / редирект другой команде / → оформить в БФТ*. Это главная экономия времени PO — отсев на входе. |
| Граница движка | **Только локальный артефакт** `smart-task.md` + хэндофф-подсказка. JIRA PO заводит сам (без авто-записи на старте — меньше прав/MCP-зависимостей, безопаснее). |
| Грунтовка знаниями | Knowledge-domain = текущая связка po-helper: `domain-profile.md` + GROUND (NEXUS-граф) + CORTEX. Один настроенный po-helper = один домен знаний = одна команда. |
| Кластер/веб | **Слой 2-3, позже.** Кластер = workspace из множества domain-конфигов; веб-чат = транспорт поверх движка. Архитектура зафиксирована здесь, реализация — после валидации движка. |

## 2. Картина целиком — три слоя

Система строится **движок → поверхность**, не наоборот. Сначала методология (brain), потом UI/оркестрация (transport).

```
┌─────────────────────────────────────────────────────────────┐
│ СЛОЙ 3 — Веб-чат (поверхность, позже)                         │
│   Внешний стейкхолдер ↔ чат-виджет                            │
│   Адресация: весь кластер ИЛИ конкретная команда             │
└───────────────────────────┬─────────────────────────────────┘
                            │ вызывает движок per-domain
┌───────────────────────────┴─────────────────────────────────┐
│ СЛОЙ 2 — Кластер / Workspace (оркестрация, позже)            │
│   Реестр доменов (= команд). Роутер: какой домен(ы)          │
│   отвечает. Fan-out по кластеру или адресная команда.        │
└───────────────────────────┬─────────────────────────────────┘
                            │ каждый домен = один po-helper config
┌───────────────────────────┴─────────────────────────────────┐
│ СЛОЙ 1 — Intake-движок (скилл request-intake, СЕЙЧАС)        │
│   Grounded custdev → ImpactMap → Gap → SMART → Scoring+Route │
│   Грунт: domain-profile + GROUND(NEXUS) + CORTEX             │
│   Выход: smart-task.md + routing-решение → хэндофф в bft     │
└─────────────────────────────────────────────────────────────┘
```

**Аналогия Repowise (референс PO):**

| Repowise | Эта система |
|---|---|
| Workspace | Кластер (все команды под управлением PO) |
| Knowledge domain (репозиторий) | Команда (её домен знаний) |
| Запрос ко всему workspace / к конкретному репо | Запрос ко всему кластеру / к конкретной команде |
| LLM отвечает grounded в домене | Интервьюер ведёт *информированную* распаковку, зная контекст команды |
| Подключённый репо | Настроенный `po-helper` config = один подключённый домен |

## 3. Слой 1 — Intake-движок (`request-intake`)

### 3.1 Принцип-дифференциатор: информированный интервьюер

Это **не форма обратной связи**. Ценность — в том, что агент уже знает домен команды (NEXUS-граф + CORTEX + история беклога), поэтому на «мне нужен X» отвечает *«у нас уже есть Y, закрывающий половину — вы про это или про…?»*. Без грунтовки получился бы тупой опросник; с грунтовкой — Repowise-grade распаковка. Поэтому стадия `/req-context` критична и богата, а не формальна.

### 3.2 Pipeline — 7 стадий (зеркало sa-helper FNR + routing)

Каждая стадия = **отдельная команда, отдельная роль, отдельный артефакт, STOP-пауза** для ревью. Не прогонять весь пайплайн за один промт.

| # | Команда | Роль | Артефакт | = шаг реального процесса PO |
|---|---|---|---|---|
| 0 | `/req-context` | Context Builder (grounded) | `context-pack.md` — кто спрашивает, его домен/команда, смежные JIRA/CORTEX, **похожие прошлые запросы** | прайминг (информированный интервьюер) |
| 1 | `/req-interview` | Custdev Interviewer (Socratic, 5-whys, без наводящих) | `interview.md` — сырые ответы + цитаты-якоря | шаг 1a (custdev-интервью) |
| 2 | `/req-impact-map` | Product Analyst | `impact-map.md` — цель; для кого; что делаем; почему сейчас; какие пути видит PO; почему именно этот путь | шаг 1b (заполнение ImpactMap) |
| 3 | `/req-gap` | Problem→Change Analyst | `gap.md` — As-Is / Проблема / To-Be → краткое понятное продуктовое изменение | шаг 2 (разрыв ожиданий) |
| 4 | `/req-smart` | Task Writer | `smart-task.md` — SMART-задача + контекст + критерии приёмки | шаг 3 (SMART-задача) |
| 5 | `/req-score` | Validator + Router | `scoring.md` — hard gates + Светофор + **routing-решение** | первичный скоринг |
| 6 | `/req-handoff` | Deliverer (local) | финальный `smart-task.md` + подсказка «дальше — `/bft-context-gen`» | передача в БФТ |

```
/req-context → context-pack            [СТОП]
     ↓
/req-interview → interview.md          (интерактивно, multi-turn внутри стадии)  [СТОП]
     ↓
/req-impact-map → impact-map.md        [СТОП]
     ↓
/req-gap → gap.md                      [СТОП]
     ↓
/req-smart → smart-task.md             [СТОП]
     ↓
/req-score → scoring.md + РЕШЕНИЕ ──┬─ консультация → ответить grounded и ЗАКРЫТЬ (early exit)
     ↓ (только "→ БФТ")            ├─ отклонить → краткое обоснование, СТОП
/req-handoff → хэндофф в bft         ├─ редирект → какой команде + почему, СТОП
                                     └─ → БФТ → /req-handoff
```

### 3.3 Routing-решение (стадия 5) — главный time-saver

Скоринг классифицирует запрос и **может закоротить пайплайн (early exit)**:

| Решение | Что делает агент | Артефакт |
|---|---|---|
| **Консультация** | Запрос — это вопрос, не изменение. Агент отвечает grounded (NEXUS/CORTEX) и закрывает. SMART-задача не нужна. | ответ + `scoring.md` |
| **Отклонить** | Вне scope / дубль / нет ценности. Краткое обоснование для стейкхолдера. | `scoring.md` |
| **Редирект** | Не наша команда. Куда передать и почему. | `scoring.md` |
| **→ БФТ** | Валидный change-request, ценность и приёмка ясны. Идёт в `/req-handoff`. | `smart-task.md` + handoff |

Лёгкий триаж-сигнал ловится уже на `/req-interview` (это «просто вопрос» vs «запрос на изменение»), но **полное** routing-решение — на скоринге, когда ценность распакована.

### 3.4 Адаптация anchor-дисциплины (нулевой допуск к галлюцинациям)

Принцип дома сохраняется: каждый факт ← источник. Для intake **источник ценностных утверждений = собственные слова запрашивающего** (цитата из интервью).

| Ранг | Источник | Где применим |
|---|---|---|
| R1 | Прямая цитата запрашивающего (`interview.md`) | ценность, боль, образ результата/приёмки |
| R2 | Домен знаний — NEXUS / CORTEX / JIRA-история | As-Is, смежные системы, похожие запросы |
| R3 | Вывод агента (`[вывод — проверить]`) | гипотезы, классификация |
| — | Нет источника | `[УТОЧНИТЬ у запрашивающего]` |

### 3.5 Структура скилла (по конвенции дома)

```
.claude/skills/request-intake/
├── SKILL.md                  роли + pipeline-карта + принципы + routing-модель
├── resources/
│   ├── interview_guide.md     custdev-скрипт: open-вопросы, 5-whys, без наводящих
│   │                          (грунт: docs/TRADITIONAL/RB-STEP-2-CUSTOMER-VALIDATION)
│   ├── impact_map.md          шаблон + правила ImpactMap (5 полей)
│   ├── gap_rules.md           As-Is / Проблема / To-Be — высота, краткость
│   ├── smart_rules.md         SMART-критерии + формат + критерии приёмки
│   ├── hard_gates.md          бинарные 🔴-гейты + Светофор + routing-матрица
│   ├── anchor_rules.md        R1 цитата / R2 домен / R3 вывод
│   └── writing_style.md       стиль (или reuse из bft-writer)
└── examples/
    ├── ideal_impact_map.md    эталон ImpactMap
    └── ideal_smart_task.md    эталон SMART-задачи

.claude/commands/req-*.md      7 тонких команд (грузят SKILL + один resource)
```

Грунт интервью — существующая методология `docs/TRADITIONAL/RB-STEP-2-CUSTOMER-VALIDATION/` (глубинные интервью: open-вопросы, копать «почему», без наводящих). Не выдумываем custdev с нуля — он уже в репо.

### 3.6 Рабочая папка запроса

```
<intake_workspace>/<request_id>/
├── smart-task.md             ← финальный артефакт (если решение «→ БФТ»)
└── artefacts/
    ├── context-pack.md
    ├── interview.md
    ├── impact-map.md
    ├── gap.md
    └── scoring.md
```

Новый ключ в `domain-profile.md`: `paths.intake_workspace` (напр. `CORTEX/_intake/{request_id}`).

## 4. Слой 2 — Кластер / Workspace (позже)

### 4.1 Модель

- **Домен = команда.** Один настроенный `po-helper` (его `domain-profile.md` + GROUND + CORTEX) = один домен знаний.
- **Кластер = реестр доменов.** Workspace знает список подключённых команд.
- **Адресация:** стейкхолдер обращается *к конкретной команде* (один домен) или *ко всему кластеру* (роутер выбирает домен(ы) по теме запроса — «кто это вообще делает?»).

### 4.2 Роутер

Поверх `/req-context`: классифицирует запрос → какой домен(ы) релевантен. Для кластерного обращения — fan-out лёгкого контекста по доменам, выбор владельца, дальше движок в выбранном домене. Это естественное обобщение routing-стадии 5 (редирект) на уровень кластера.

### 4.3 Открытый вопрос

Кластер = монорепо из нескольких po-helper-конфигов? Или внешний реестр, ссылающийся на отдельные репозитории команд (ближе к Repowise-«подключённый репо»)? → §7.

## 5. Слой 3 — Веб-чат (позже)

- Чат-виджет: внешний стейкхолдер ведёт диалог; под капотом — `/req-interview` промт без изменения логики (транспорт-агностичность из §1).
- Выбор адресата: команда или кластер (§4.1).
- Сессия чата = одна рабочая папка запроса (§3.6); по завершении — артефакты + routing-решение, видимые PO.
- Стек — **открытый вопрос** (§7). Кандидаты: Claude Agent SDK как бэкенд движка + тонкий веб-фронт.

### Контракт портируемости (важно с самого начала)

Чтобы веб не потребовал переписывания движка:

1. Промты стадий **не предполагают, кто за клавиатурой** — элиситят у «запрашивающего».
2. Стадии общаются через **файлы-артефакты** (`artefacts/*.md`), а не через состояние чата → веб-сессия просто пишет в ту же папку.
3. Грунтовка идёт через **явные пути** (`domain-profile.md`), а не хардкод → веб подставляет домен выбранной команды.

Соблюдение этих трёх правил при написании скилла = движок поднимается в веб без переписывания, только меняется транспорт.

## 6. Хэндофф в bft-writer

Движок — **front door** перед существующим БФТ-пайплайном. Сегодня `/bft-context-gen <epic> <jira_key>` предполагает, что эпик JIRA с ясным запросом *уже существует* — а это ровно то, что PO делает руками. `request-intake` производит этот вход.

- Финальный `smart-task.md` (решение «→ БФТ») = seed для эпика.
- `/req-handoff` выводит подсказку: PO заводит JIRA-задачу/эпик руками (решение §1: без авто-записи), затем `/bft-context-gen <epic> <jira_key>`.
- Стык чистый: разные высоты (intake = первичный скоринг и ценность; bft = декомпозиция эпика в БТ/ПТ/ИТ/ФТ/НФТ).

## 8. Фазирование

- **Фаза 1 (сейчас):** скилл `request-intake` — 7 стадий, resources, examples. CLI-режим, PO-прокси. Валидируем методологию на реальных запросах.
- **Фаза 2:** кластер/роутер — реестр доменов, адресация команда/кластер.
- **Фаза 3:** веб-чат — транспорт поверх движка, отдельный проект, внешние стейкхолдеры напрямую.
