# Дизайн: `/okr-present` — финальная стадия квартала (презентация + roadmap)

**Дата:** 2026-07-08 · **Ветка:** `fix/quarter-presentation` · **Скилл:** `okr-planner` (новая финальная стадия)

**Эталоны (target-структура, не инстанс):**
`GDS-Q3-2026.pptx` (30 слайдов, 16:9) и `Roadmap-GDS-Q3-2026.xlsx` — из vault MTS. Роль как у `examples/ideal_*.md`: показывают целевую форму, сам скилл — generic.

---

## 1. Проблема и ценность

Сегодня квартальный пайплайн заканчивается vault-markdown: `okr-planner` даёт `OKR-{quarter}.md` + `KR-EPIC-MAP`, `sprint-planner` — `SPRINT-ROADMAP-{quarter}.md` + `ПЛАН-*.md` (+ опц. JIRA). Нет стадии «упаковать квартал для комнаты».

`/okr-present` — эта недостающая финальная стадия. Она:
1. **Собирает всё из хранилища** и рендерит артефакты: `deck.html` (промежуточная поверхность правок), затем финал `.pptx` (нарратив для стейкхолдеров простым языком) и `.xlsx` (roadmap инициативы×недели с роль·стек-чипами).
2. Работает как **линза консистентности**: при доработке дека всплывают дыры и противоречия в артефактах (KR без образа результата, спринт без исполнителя, число без источника).
3. **Принятый дек = метрика выравнивания** контекста и ожиданий между AI-системой и живым PO. Это цель стадии, а не побочка.

**HTML-first (референс: Claude Design).** Вся сутевая доработка идёт в `deck.html` — LLM на порядок точнее и быстрее правит вёрстку и содержание в HTML/CSS/JS, чем в python-pptx. `.pptx` собирается **только после «принято по сути»**, из того же канонического `deck-spec` + общей `theme`, поэтому визуально совпадает с согласованным HTML.

**Не** делает: не перепланирует OKR, не трогает JIRA, не выдумывает факты.

## 2. Модель истины: vault → render, HTML как поверхность правок (решение PO)

- **Источник истины — vault.** `deck-spec` — каноническая модель дека (vault-якоря). `deck.html` / `.pptx` / `.xlsx` — производные витрины, всегда пересобираемые из `deck-spec` + `theme`.
- **Поверхность правок — `deck.html`.** PO смотрит HTML (файл или опубликованный Artifact), называет правки; правки классифицируются и текут в канон:
  - Правка **факта** (состав KR, исполнитель, срок) → в vault-артефакт (`OKR-*.md` / `SPRINT-ROADMAP` / `ПЛАН-*`) → в `deck-spec`.
  - Правка **рамки/подачи** (имя направления, формулировка «простым языком», порядок слайдов) → в `deck-spec.md` (`[PO-authored]`).
  - Правка **вёрстки** (отступы, разбивка, размер) → в `theme` / HTML-шаблон архетипа.
- **Финализация в `.pptx` — только после «принято по сути» в HTML.** PPTX и HTML рендерятся из одного `deck-spec` + `theme` → паритет.
- **Ноль галлюцинаций:** каждый факт на слайде/в ячейке несёт vault-якорь либо явную метку `[PO-authored]`. Нет источника → `[УТОЧНИТЬ у {кого}]`, попадает в реестр пробелов, а не выдумывается.

## 3. Границы данных: что из vault, что доавторится

**Выводится из vault (машинный маппинг):**
| Слайд/блок | Источник |
|---|---|
| СЕЙЧАС / СТАНЕТ по KR | «образ результата» KR (конвенция `БЫЛО→СТАЛО`) |
| Образ действия (шаги с ролями) | «образ действия» KR/истории из `OKR-*` / `ПЛАН-*` |
| Риски | «метрики&риски&зависимости» KR + риски sprint-плана |
| Sprint-lifecycle таблица (направление × спринт) | `SPRINT-ROADMAP-{quarter}.md` (матрица KR×спринт) + фазы |
| Роли×спринт таблица | `ПЛАН-*.md` (исполнители, SP) + `GROUND/NEXUS/team/*` |
| Ответственные (по экспертизе/людям) | team-нексус (`full_name`, `role_title`, `expertise_topics`) |
| xlsx roadmap (инициативы×недели, чипы) | `SPRINT-ROADMAP` + `ПЛАН-*` + `cadence`/`sprints_per_quarter` |
| Внешние команды / зависимости | `landscape-{quarter}.md` |

**Доавторится в диалоге (не в vault — `[PO-authored]`):**
- Группировка OBJ → **направления** (имена + цвета).
- Рыночное «зачем» (числа рынка) — **с источником**, иначе `[УТОЧНИТЬ]`.
- Глоссарий «что есть что простым языком».
- Тон «простым языком», closing: порядок работ / «прямо сейчас» / «что запомнить».

## 4. Структура: одна команда `/okr-present <quarter>`, внутренние фазы со STOP

Единственный вход — `/okr-present <quarter>`. Внутри — фазы с STOP-паузой PO (как каденция каркаса). Не гнать до финала за один промт.

- **A · Сбор + Консистентность** (роль: Consistency Auditor). Читает весь vault, строит coverage-map, **выдаёт реестр пробелов** `[УТОЧНИТЬ]`: KR без образа результата, спринт-строка без исполнителя, OBJ без направления, число без источника, риск без KR. Артефакт: `{okr_workspace}/present/coverage.md`. **STOP.**
- **B · Доавторинг рамки** (роль: Storyteller, опрос по одному вопросу). Заполняет не-vault слайды: направления+цвета, рынок+источник, глоссарий, closing. Пишет в `deck-spec` как `[PO-authored]`. **STOP.**
- **C · deck-spec** (роль: Spec Builder). Единая структурированная спека — **источник всех рендереров**. Сухой прогон (перечень слайдов + якоря). Артефакт: `{deck_spec_doc}`. **STOP.**
- **D1 · Рендер HTML** (роль: Deliverer). `render_html.py` → `deck.html` (self-contained, инлайн CSS/JS, 16:9) + HTML-превью roadmap-таблицы. Артефакт: `{quarter_deck_html}`. Можно опубликовать как Artifact для просмотра в браузере. **STOP.**
- **E · Коррекция по сути** (роль: Alignment Loop, **на HTML**). PO смотрит `deck.html` → называет правки → LLM правит vault (факт) / deck-spec (рамка) / theme (вёрстка) → перерендер D1. Цикл до **«принято по сути»**. Это главная петля выравнивания AI↔PO.
- **D2 · Финализация** (роль: Deliverer). После «принято»: `render_pptx.py` → `.pptx` (theme-паритет с HTML), `render_roadmap.py` → `.xlsx`. Оба из того же `deck-spec`. Артефакты: `{quarter_deck_doc}`, `{quarter_roadmap_xlsx}`. **STOP.**

**Почему одна команда, а не пять `/okr-*`:** это единый акт «упаковать квартал», а не отдельные планировочные роли. Дробить на 5 команд перегрузило бы пайплайн okr-planner. Фазы дают чекпоинты без раздувания CLI-поверхности.

## 5. Движок рендера (коммит в скилл, переиспользуемый)

Три бэкенда над **одним** `deck-spec` + общей `theme` + каталогом архетипов → визуальный паритет HTML↔PPTX. HTML — среда правок (LLM силён в HTML/CSS/JS); PPTX — финал; XLSX — roadmap. Правка → перезапуск скрипта, детерминированно.

```
.claude/skills/okr-planner/
  resources/
    deck_spec_schema.md     # схема deck-spec + каталог архетипов слайдов
    render_html.py          # deck-spec + theme → deck.html (поверхность правок)
    render_pptx.py          # python-pptx: deck-spec + theme → .pptx (финал)
    render_roadmap.py       # openpyxl:  deck-spec + theme → .xlsx
    theme.md                # дизайн-система (общая для HTML и PPTX; дефолт = эталон MTS)
    present_gates.md        # Светофор консистентности
  examples/
    ideal_deck_spec.md      # эталонная deck-spec (реверс из GDS-Q3-2026)
```

### 5.0 Паритет HTML → PPTX (фидельность)
`render_html.py` и `render_pptx.py` — **две реализации одного каталога архетипов** поверх общих констант `theme` (цвета, шрифты, размеры, сетка 16:9). Маппинг детерминирован **по-архетипно**: каждый архетип имеет фиксированную раскладку и в HTML (CSS-grid/flex), и в pptx (autoshapes по сетке) → одинаковый результат из одних данных.
- **Дефолт:** нативный редактируемый `.pptx` (текст, автофигуры — как эталон).
- **Fallback (опц., при сложной вёрстке):** пиксель-в-пиксель — headless-скриншот каждого HTML-слайда (1920×1080) → full-bleed картинкой в pptx-слайд. Не редактируемо, зато 100% совпадение. Выбор — флагом на фазе D2.

### 5.1 Каталог архетипов слайдов (из эталона, 30 слайдов)
Каждый архетип имеет две реализации из общей `theme`: HTML-шаблон (CSS-grid/flex) и pptx-маппинг (autoshapes по сетке).
`title` · `big-picture-4` (4 направления карточками) · `why-market` (3 числа + сдвиг квартала) · `glossary` (термин→простыми словами) · `how-to-read` (СЕЙЧАС/СТАНЕТ легенда) · `direction-divider` (NN + название + подводка) · `sprint-lifecycle-table` (строки-шаги × S1..Sn, чипы роль·стек, фазовая раскраска аналитика→разработка→отладка→релиз) · `roles-sprint-table` (экспертиза+люди × спринты) · `now-becomes-detail` (СЕЙЧАС/СТАНЕТ + образ действия по этапам + риски-карточки) · `order-of-work` (сначала/к августу/дальше) · `right-now` (нумерованный список) · `after-meeting` · `takeaways` (N тезисов). Footer «{product} · {quarter} · {subtitle}» + номер.

### 5.2 deck-spec (единый источник всех рендереров)
Структурированный markdown/YAML: шапка (product, quarter, subtitle, footer), `directions[]` (name, color, objs[]), на каждый KR — `now`, `becomes`, `steps[]` (role, text), `risks[]`, `sprint_cells{Sx: [role·stack]}`, `owners`. Плюс `authored{}` блок (market, glossary, closing). Каждое поле — `src:` (vault-якорь) или `authored:true`. `render_html.py` и `render_pptx.py` берут всё; `render_roadmap.py` — `directions[].objs[].kr.sprint_cells`.

### 5.3 Тема (дизайн-система, дефолт = эталон, override в profile)
Дефолты из эталона: accent `#E4002B`, navy heading `#1C2333` (Cambria), body `#434C60` (Calibri), палитра направлений `[#E4002B,#2E4B7A,#0F7A8C,#2F7D54]`, роль→цвет (BA/SA/ADR `#B37D0C`, BE/FE `#2E4B7A`, QA `#7E56A6`, RELEASE `#2F8557`, DBA `#0F7A8C`), muted `#8B94A6`, фон-карточки `#F5F7FB`, 16:9 (13.33×7.5"). Все переопределяются секцией `present` в domain-profile → каркас generic.

## 7. Гейты консистентности (`present_gates.md`, Светофор)

- 🔴 (СТОП, вернуть PO): KR без «образа результата» (нет СЕЙЧАС/СТАНЕТ); sprint-lifecycle строка без исполнителя; OBJ не привязан к направлению; рыночное число без источника.
- 🟡 (продолжить с явным `[УТОЧНИТЬ]`): риск без явной привязки к KR; спринт-ячейка без стека; отсутствует глоссарный термин, встречающийся в деке.
- 🟢: полное покрытие → рендер.
Светофор печатается в фазе A и повторно перед D1.

## 8. Зависимости и переисполняемость

- `render_html.py` — **без зависимостей** (чистый Python-шаблонинг → self-contained HTML с инлайн CSS/JS). Это фаза правок → лёгкая и мгновенная.
- `render_pptx.py` — `python-pptx`; `render_roadmap.py` — `openpyxl` (тот же стек, что произвёл эталоны). Фаза финализации D2.
- Fallback пиксель-в-пиксель (опц.) — headless-браузер (Playwright, уже в окружении) для скриншотов HTML-слайдов.

Рендер детерминирован (без сети), переисполняем: правка vault/spec/theme → повтор D1/D2 даёт обновлённые файлы поверх старых. Скрипты — под git, вывод — в `{planning_root}/presentation/` (не в вычищаемый tmp/worktree).

## 9. Изоляция компонентов

| Компонент | Что делает | Вход | Выход |
|---|---|---|---|
| Фаза A (auditor) | покрытие+пробелы | vault | `coverage.md` + Светофор |
| Фаза B (storyteller) | не-vault рамка | диалог PO | `[PO-authored]` блоки |
| deck-spec | единая модель дека | vault+рамка | `deck-spec.md` |
| `render_html.py` | spec→html (правки) | deck-spec+theme | `deck.html` |
| `render_pptx.py` | spec→pptx (финал) | deck-spec+theme | `.pptx` |
| `render_roadmap.py` | spec→xlsx | deck-spec+theme | `.xlsx` |
| `theme` | дизайн-система | domain-profile | палитра/шрифты/сетка |

Каждый рендерер меняется независимо (переверстать pptx, не трогая xlsx). deck-spec — стабильный контракт между сбором и рендером; `theme` — общий контракт паритета HTML↔PPTX.

## 10. Границы MVP (YAGNI)

**В MVP:** оба файла из одной команды; фазы A–E (HTML-first: D1 html → E правки → D2 финал pptx+xlsx); три рендерера над одним spec+theme; тема-эталон+override; гейты; правки profile.
**НЕ в MVP:** обратная синхронизация из отредактированного вручную .pptx/.xlsx/.html в vault (модель — vault→render, правки называются PO в диалоге); пиксель-в-пиксель fallback (заготовка, но по умолчанию нативный pptx); анимации/переходы; авто-подбор рыночных чисел (только с источником от PO); мультиязычность; отдельная команда `/okr-roadmap`.
