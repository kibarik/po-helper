# [Дизайн] V2 БФТ-конвейер: Fast → Deep → Sync (баланс скорости и качества)

> **Главная инновация: не заменяем отлаженный канонический workflow bft-writer, а добавляем V2-способ работать с БФТ, сокращающий время.** Старый 7-стадийный пайплайн = **ориентир** (уровень качества, глубина проработки, шаблоны, процесс), которого V2 обязан достичь. Fast и Deep — быстрые пути входа; Sync вливает результат в канон, не перезатирая его.
> Цель = **баланс скорости и качества**. Вне итерации: `/bft-deliver` (запись в JIRA/Confluence остаётся ручным гейтом PO), per-step замер каталога моделей (#112).

## 1. Образ результата — три lane

```
1. FAST   — из фактов встречи → письмо-seed              (bft-fast, СУЩЕСТВУЕТ, не тронут)
2. DEEP   — автономно, полный контекст → ОБОГАЩАЕТ seed   (НОВАЯ роль-обогатитель:
              (не регенерирует с нуля)                       ценность + what-if + границы)
3. SYNC   — сверка с существующим bft-writer: укладка       (переиспользует /bft-draft
              в канон-шаблоны + прогон канонических гейтов    + /bft-validate; канон = ОРИЕНТИР,
              качества                                        не перезатирается)
```

PO запускает `/bft-fast <summary>`. Немедленно — письмо в чат (Fast, поток A). Параллельно или следом (см. §4 адаптивный форк) — фоновый Deep обогащает черновик полным контекстом, затем Sync укладывает его в каноническую структуру MTS и прогоняет канонические гейты. Выход: `workspace/<epic_slug>/<epic_slug>.md` — черновик БФТ канон-уровня со всей фактурой и `[УТОЧНИТЬ]`-маркерами. Флаг `--no-deep` отключает Deep+Sync.

## 2. Проблема

- **ДЛЯ КОГО:** PO, которому нужен полный БФТ канон-качества, но без 30-минутного слепого прогона и 7 STOP-пауз.
- **ЧТО ДЕЛАЕМ:** V2-конвейер параллельно каноническому — быстрый вход из фактов + автономное обогащение полным контекстом + синхронизация с отлаженным bft-writer.
- **AS-IS:** один способ — канонический 7-стадийный пайплайн, тяжёлый, STOP-пауза на каждом шаге, глубина применяется вслепую наперёд.
- **ПРОБЛЕМА:** нет баланса скорость/качество — либо быстрое письмо (bft-fast, неполное), либо полный медленный канон. Между ними пусто.
- **TO-BE:** V2 закрывает середину — быстрый seed + автономное обогащение до канон-уровня, сверенное со старым пайплайном как ориентиром качества.

## 3. Роль: БФТ-deep (Обогатитель)

**Не** ре-ран старого пайплайна с нуля. Отдельная роль, берущая быстрый черновик как основу и наращивающая глубину.

- **Вход:** быстрый черновик из bft-fast (письмо: How-to-demo / Открытые вопросы / Границы) = **seed** + Summary + полный контекст (JIRA/Confluence/repowise/ADR/стейкхолдеры).
- **Предусловие «есть основа»:** обогащает, только имея основу — либо готовый полный контекст, либо дождавшийся Fast-черновик (§4).
- **Три оси обогащения:**
  1. **Ценность** — глубинные вопросы «зачем инвестировать»: привязка к KR/стратегии квартала, реальный outcome, что быстрый проход пропустил. Переиспользует `bft-value`.
  2. **«Что если» демонстрация** — расширяет How-to-demo в what-if сценарии: пользователь делает X; крайний случай Y; альтернативный путь приёмки. Стресс демо вскрывает скрытые требования.
  3. **Явные границы** — заостряет «вот это нам здесь не важно»: явный out-of-scope, режет шум из документа.
- **Выход:** обогащённый черновик, ложащийся на канон-структуру MTS; всё без источника → `[УТОЧНИТЬ у {кого}]`, не выдумка.

## 4. Адаптивный форк (баланс скорость/качество)

Стартовый проб контекста в `/bft-fast` определяет топологию запуска:

```
probe context richness (JIRA-эпик? Confluence-страницы? прежние artefacts? насыщенность Summary?)
├─ RICH (есть на что опереться) → форкать DEEP СРАЗУ, ПАРАЛЛЕЛЬНО Fast.
│     Deep грудится на полный контекст напрямую, не ждёт seed от Fast.
│     Максимум скорости: быстрый И глубокий разом.
└─ THIN (опереться не на что)   → сперва дождаться завершения FAST;
      его письмо-черновик = seed; ЗАТЕМ форкать DEEP.
      Deep без seed захлебнётся → Fast сначала строит скелет.
```

`--no-deep` пропускает Deep+Sync в обеих ветках.

## 5. Подложка: ruflo координирует, Claude исполняет

Рыночный скан Q3 2026 (Anthropic MAS orchestrator-worker +90% vs single-agent; консенсус анти-галлюцинаций = типизированный грудинг + forced-citation + verifier-агент): качество рождается в **дизайне оркестрации**, не в бренде движка.

- **ruflo** = координация: `swarm_init` (hierarchical, queen-coordinator), `agent_spawn` (cost-tracking + memory на роль), `memory_store/retrieve` (namespace `bft-deep/<epic_slug>`, shared fact-base + хендофф), `coordination_consensus` (вердикт дебатов + грудинг-споры).
- **Нативные Claude-субагенты** = исполнение с **прямым MCP-грудингом** (JIRA/Confluence/repowise подключены в сессии). Держат качество/грудинг.
- **Failure-режим:** ошибка ruflo-tool → деградация на native-Task + локальный файловый хендофф, лог в error-callback. **ruflo down ≠ прогон прерван** — падает до native-subagent-only.

## 6. Оркестрация Deep+Sync (seed → обогащение → синк)

```
0. FORK: pin repo commit hash → всем субагентам. epic_slug = date-slug из темы Summary
   (всегда выводим, независим от JIRA epic key). Топология запуска — §4.
1. swarm_init (hierarchical, queen-coordinator + role-card §3 + guardrails).
   Seed = Fast-черновик (или полный контекст в RICH-ветке) → ruflo memory.
   Vague/противоречивый вход → шире [УТОЧНИТЬ], не догадки.

── DEEP (обогащение seed, не ре-ран с нуля) ──
2. context   → субагент /bft-context-gen-deep (parallel: JIRA/Confluence/repowise/ADR/стейкхолдеры).
               Per-source fallback: timeout/403 → источник=UNAVAILABLE; claim оттуда → [УТОЧНИТЬ],
               никогда «пусто = нет данных». Memory-audit гейтит фактбазу перед обогащением.
3. value     → Value Investigator (ось 1): глубинные «зачем», привязка KR/стратегия. bft-value.
4. what-if   → Demo Stress (ось 2): what-if расширение How-to-demo, скрытые требования.
5. bounds    → Boundary Sharpener (ось 3): явный out-of-scope «нам здесь не важно».

── SYNC (укладка в канон + канонические гейты, канон = ориентир) ──
6. draft     → /bft-draft: обогащённый seed → канон-структура MTS (БТ/ПТ/ИТ/ФТ/НФТ).
7. verify    → Grounding-Verifier: типизация claim (grounded/ungrounded/contradicted).
               ungrounded → [УТОЧНИТЬ у {кого}]; contradicted → регенерация С reasoning контр-примера;
               после 1 regen всё contradicted → escalate [УТОЧНИТЬ].
8. validate  → /bft-validate: канонические hard-gates + Светофор.
               Evaluator-optimizer: гоняем пока гейты улучшаются; 2 раунда без прогресса
               → стоп, emit 🟡 + validation-notes (convergence-stop, не фикс.лимит).
9. citation  → forced-citation: каждый факт ← якорь (JIRA/Confluence/PO/CORTEX). Нет якоря → [УТОЧНИТЬ],
               никогда инвент, никогда тихий [UNANCHORED].
10. review   → свежий агент, весь документ против канон-ориентира (уровень качества/глубины старого БФТ).
11. emit: workspace/<epic_slug>/<epic_slug>.md + artefacts/ + нотификация.
```

## 7. Границы

- Прогон **останавливается на валидированном черновике** (через `/bft-validate`). **НЕ** запускает `/bft-deliver` — запись в JIRA/Confluence = ручной гейт PO (никаких автономных побочек).
- Существующие `bft-*` команды и `skills/bft-writer/*` **не переписываются** — Sync их **вызывает** (`/bft-draft`, `/bft-validate`). Канон = ориентир, не перезатирается.
- Вход/эпик: от Summary; JIRA-эпика нет → линковка = `[УТОЧНИТЬ]`, заполнимо позже. Валидатор трактует явный `[УТОЧНИТЬ]`-пробел как 🟡-допустимо, только фабрикацию как 🔴.

## 8. Валидация качества — «не хуже PO И на уровне канона» (eval-харнесс, merge-гейт)

Двойной ориентир: (а) не хуже прямого диалога PO, (б) достигает уровня качества/глубины старого канонического БФТ.

Двухъярусный синтетический фикстур:
- `examples/golden_deep_summary.md` — насыщенный Summary (факты сказаны + открытые точки не решены).
- `examples/golden_deep_gold.md` — аннотированный claim-table: каждый claim помечен **SOURCE-DERIVABLE** (swarm ОБЯЗАН грудить) vs **PO-ONLY** (swarm ОБЯЗАН пометить `[УТОЧНИТЬ]`); + answer-key (значения живого PO — swarm НЕ должен их выдать) + рубрика.

**Принцип честности.** У автономного прогона нет живого PO → `[УТОЧНИТЬ]` там, где PO бы ответил, законно. Сравнение двухъярусное, не байт-в-байт:
- SOURCE claim пропущен/неверен → регрессия → **FAIL**.
- PO-ONLY claim выдуман (совпал с answer-key) → галлюцинация → **hard FAIL**.
- PO-ONLY claim помечен `[УТОЧНИТЬ]` → **PASS**.

**Рубрика (hard-gates):**
1. hallucination = 0 · 2. SOURCE-recall = полный · 3. все пробелы помечены (No Silent Skip) · 4. false-gap = 0 · 5. canon MTS структура present · 6. anchor validity = 100% · **7. канон-паритет: обогащённый выход покрывает ценность+what-if+границы и проходит канонические гейты `/bft-validate` на уровне старого БФТ (ориентир)**.

**Судья:** ОТДЕЛЬНЫЙ свежий LLM-агент (не из пайплайна) грейдит swarm-вывод против gold; ниже бара = «хуже PO / ниже канона» = **блок мержа**.

Фикстуры: `docs/superpowers/specs/fixtures/2026-07-14-bft-deep-*` (черновые; при реализации дописывается реальный gold-BFT текст поверх claim-table).

## 9. Нотификация

- Успех: `deep БФТ готов: <path> · N [УТОЧНИТЬ]-точек · валидация 🟢/🟡`.
