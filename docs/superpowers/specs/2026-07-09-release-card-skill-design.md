# Design — навык «Релизная карточка» (`/release-card`)

> Статус: **design draft, ожидает финального ОК PO**. Ветка: `feat/release-cards`.
> Дата: 2026-07-09. Автор сессии: Claude (Opus 4.8).

## Задача

При выкатке в Production у любого эпика, который касается связанных задач
(**Backend + Frontend**) или затрагивает **больше 2 модулей** доработок, должна быть
**релизная карточка**. Нужен отдельный навык для генерации этой карточки.

Пример содержания карточки (эпик «Кино», ticketland.ru) приложен как иллюстрация —
см. `skills/release-guard/examples/example_release_card.md`. Пример **не канон**:
po-helper — настраиваемый каркас, шаблон должен быть дженерик (см. память
`po-helper-is-configurable-framework`).

## Ключевые решения (согласовано с PO)

| Вопрос | Решение |
|---|---|
| Структура | **Суб-навык `release-guard`** — новая команда `/release-card`, фаза Run, общий `{release_workspace}` |
| Отгрузка | **`.md` сейчас**; публикация в Confluence/JIRA — отдельная будущая стадия `/release-card-deliver` (вне scope) |
| Триггер | Навык **сам детектит** Backend+Frontend / >2 модулей и **предупреждает** PO, если карточка не нужна |
| Детект «модуля» | JIRA `components` дочерних задач; fallback — linked issues. Backend/Frontend распознаём по именам компонентов/лейблам |
| Standalone | Работает по любому эпику JIRA, **не требует** прошлого `/release-frame`. Если release workspace релиза есть — пишет туда |

## Архитектура (следует паттерну release-guard)

Два слоя, как у остальных `release-*` команд:

- **Мастер-навык** `.claude/skills/release-guard/SKILL.md` — роли, pipeline, принципы.
- **Тонкая команда** `.claude/commands/release-card.md` — точка входа `/release-card`, полная LLM-инструкция по Этапам.

### Роль
**Deploy Card Author** — собирает из эпика проверяемую карточку выкатки: план деплоя
с атрибуцией ответственных и критериями успеха, условный блок DDL-миграции,
три чек-листа проверок и план отката по FK-зависимостям.

### Принцип нулевого допуска к галлюцинациям
Каждая строка карточки ← источник (задача трекера / решение PO / People Graph team-нексуса).
Нет источника → `[УТОЧНИТЬ у {кого}]`. Ответственные и критерии успеха **не выдумываются**.

## Deliverables (4 файла)

1. **`.claude/commands/release-card.md`** — тонкая команда `/release-card <epic-key> [--force]`.
   Этапы LLM-инструкции:
   - **Э1 Загрузка:** `resources/release_card_template.md`, `resources/release_standards.md`,
     `baseline.md` (если есть), People Graph (team-нексус) для ответственных.
   - **Э2 Gate (авто-детект):** `jira_get_issue` эпика + дочерние/linked → сигналы
     Backend+Frontend / >2 модулей. Сработал → дальше. Не сработал → варн
     («карточка обычно не нужна, эпик в 1 модуле»), стоп; PO может форсировать `--force`.
   - **Э3 Сбор контекста:** этапы деплоя из дочерних задач; ответственный ← assignee +
     People Graph; наличие задач миграции/схемы БД.
   - **Э4 Генерация 4 блоков** (DDL — условный, только если есть миграции).
   - **Э5 Dry-run превью PO** → правки → ОК (паттерн `bft-deliver`).
   - **Э6 Запись** `{release_workspace}/<epic>/release-card.md`.
   - **Запреты:** строка без источника → `[УТОЧНИТЬ]`; выдуманные ответственные/критерии;
     генерация самого `.sql`; отгрузка в Confluence/JIRA (вне scope).

2. **`skills/release-guard/resources/release_card_template.md`** — дженерик-шаблон 4 блоков
   + правила заполнения:
   - **1. План выкатки (Deploy plan)** — таблица `Действие | Ответственный | Критерий успеха`.
   - **2. DDL-миграция** *(условный блок)* — краткое содержание + ссылка на вынесенный
     `*.sql` (forward + rollback, идемпотентный). Карточка **ссылается**, не пишет SQL.
   - **3. Проверки** — 3 чек-листа: Pre-deploy / Dry-run / Post-deploy.
   - **4. План отката (Rollback)** — порядок сверху-вниз по FK-зависимостям: что откатываем + как.

3. **`skills/release-guard/examples/example_release_card.md`** — пример «Кино» из задачи,
   помечен «пример, не канон».

4. **`release-guard/SKILL.md`** — правка: `/release-card` в pipeline-диаграмму (фаза Run),
   таблицу ролей/артефактов, счётчик команд 6 → 7.

## Вне scope (будущие стадии)
- `/release-card-deliver` — публикация карточки в Confluence/JIRA через MCP.
- Генерация самого `ddl_migration_*.sql`.
