# Навык okr-status — подведение статуса квартальных OKR

## Проблема

Идёт экватор или конец квартала. PO нужно дать руководству честный отчёт о
статусе исполнения квартальных планов: что сделано, актуальны ли сроки,
насколько изменился объём, что **реально** будет готово к концу квартала и на
что потребуются дополнительные риски/ресурсы. По каждому KR нужна фактурка
**ПЛАН / ФАКТ / ИЗМЕНЕНИЕ**.

Сегодня в каркасе есть механика по краям, но нет команды, которая подводит
промежуточный статус именно на уровне квартального OKR:

- `okr-planner` рождает ПЛАН (OBJ/KR, IMP, образ результата, progress-шкала,
  KR-EPIC-MAP), но не оценивает исполнение.
- `release-guard` даёт честный статус с метриками дрейфа, но на уровне
  **релиза**, а не квартального OKR.
- `sprint-fact` фиксирует ФАКТ спринта (velocity, carryover), но по одному
  спринту, а не по всему кварталу и не в разрезе KR.
- `bft-context-gen-deep` — эталон «глубокого исследования по источникам»
  (параллельные субагенты + кросс-валидация), но заточен под БФТ, не под статус.

## Образ результата

1. Отдельный навык `okr-status` из 3 стадий-команд (STOP после каждой).
2. На входе — утверждённый ПЛАН (артефакты `okr-planner`), на выходе — **один**
   `.md`-отчёт для руководства: executive-сводка (светофор по OBJ) + секция на
   каждый OBJ с таблицей ПЛАН/ФАКТ/ИЗМЕНЕНИЕ по его KR + сводка «актуальность
   сроков / изменение объёма / прогноз EOQ / доп. риски».
3. ФАКТ собирается глубоким исследованием по источникам (JIRA, прошедшие релизы
   и спринт-ФАКТ, переписки и стейкхолдеры) параллельными субагентами с
   кросс-валидацией — с нулевым допуском к галлюцинациям.
4. Отчёт финализируется вместе с PO (human-in-the-loop), а не отдаётся «как
   сгенерировалось».

## Ключевые решения (зафиксированы с PO)

| # | Решение | Выбор |
|---|---|---|
| 1 | Дом функциональности | **Новый навык `okr-status`** (не расширение okr-planner, не клон release-guard) |
| 2 | Число стадий | **3**: `/okr-status-sync` → `/okr-status-assess` → `/okr-status-report`, STOP после каждой |
| 3 | Обязательные источники ФАКТА | **JIRA** (статусы KR-эпиков) · **прошедшие релизы + спринт-ФАКТ** · **переписки + стейкхолдеры**. Confluence+Код — best-effort обогащение |
| 4 | Формат выхода | **Один документ**, executive-сводка сверху + секция на каждый OBJ |
| 5 | Модель оценки KR | **Светофор 🟢/🟡/🔴 + % готовности (progress-шкала okr-planner) + прогноз EOQ (будет/частично/не будет) + scope-дрейф и сдвиг срока (формулы release-guard) + доп. риски/ресурсы** |
| 6 | Публикация | Сохранение `.md` в нексус `project-management/<quarter>/Статус/` (по образцу `okr-deliver`); выгрузка в Confluence — опциональный best-effort |

## Параметр checkpoint

Команды принимают `<quarter> [checkpoint]`, где `checkpoint ∈ {equator, final}`
(по умолчанию выводится из текущей даты относительно границ квартала). При
`final`, если существует снимок статуса `equator`, в колонке ИЗМЕНЕНИЕ
показывается дельта и к ПЛАНУ, и к прошлому чекпоинту.

## Архитектура и поток данных

```
ПЛАН (артефакты okr-planner)
  OKR-<quarter>.md · INDEX · KR-EPIC-MAP · enriched-okr (образ результата, progress-шкала, IMP, сроки)
  + прошлый снимок статуса (если есть)
        │
        ▼
/okr-status-sync <quarter> [checkpoint]      роль: Fact Investigator
  Resolve capability (role_bindings / source_policy, класс status-critical)
  3 параллельных субагента + кросс-валидация:
    🔵 A JIRA           — по каждому KR-эпику: дочерние, статусы, SP done/total, блокеры, сигналы срыва
    🟢 B Релизы+ФАКТ    — release-guard/status.md + sprint-fact (velocity, carryover, что в проде)
    🔴 C Переписки+люди — info-channels / PULSE/summaries + People Graph (сроки/риски/эскалации)
    (best-effort: Confluence + Код через repowise — обогащение, не жёсткий источник)
  ──► {status_workspace}/fact-pack.md + raw/agent-*.md          [СТОП]
        │
        ▼
/okr-status-assess <quarter>                 роль: Progress Assessor
  По каждому KR: ПЛАН (образ результата, IMP, срок, объём)
               / ФАКТ (% готовности, SP done/total, что в проде — каждый ← источник)
               / ИЗМЕНЕНИЕ (🟢🟡🔴 + scope-дрейф + сдвиг срока + прогноз EOQ + доп. риски)
  ──► {status_workspace}/assessment.md                          [СТОП: PO ревьюит]
        │
        ▼
/okr-status-report <quarter>                 роль: Reporter + Finalizer
  Интерактивная финализация с PO (правки фактуры/прогноза/рисков — по одному вопросу)
  Сборка: executive summary (светофор по OBJ) → секция на OBJ (таблица ПЛАН/ФАКТ/ИЗМЕНЕНИЕ)
          → сводка сроков / объёма / прогноза EOQ / доп. рисков
  ──► {status_report_doc}  (project-management/<quarter>/Статус/)   финал
```

## Компоненты и их границы

Каждая стадия — отдельный запуск, отдельная роль, свой артефакт, STOP-пауза.

### Стадия 1 — `/okr-status-sync <quarter> [checkpoint]` (Fact Investigator)

- **Что делает:** грузит ПЛАН и прошлый снимок статуса; резолвит доступность
  ролей (`role_bindings`, `source_policy` класс `status-critical`); запускает 3
  параллельных субагента; кросс-валидирует их факты; собирает `fact-pack.md`.
- **Субагенты** (запуск одновременно, raw-артефакты сохраняются):
  - 🔵 A JIRA — через KR-EPIC-MAP по каждому KR-эпику: дочерние задачи,
    статусы, SP done/total, блокеры, сигналы срыва срока.
  - 🟢 B Релизы+ФАКТ — `release-guard/<id>/status.md` + `sprint-fact` последних
    спринтов квартала: velocity, carryover, что реально ушло в прод.
  - 🔴 C Переписки+стейкхолдеры — `info-channels` / `PULSE/summaries` + People
    Graph: сигналы о сроках/рисках/эскалациях, кто что говорит.
- **Кросс-валидация:** «JIRA Done» ↔ «в проде» (B) ↔ «в чате жалуются» (C).
  Конфликты выносятся явно в раздел «Разногласия источников», не схлопываются.
- **Вход:** ничего (старт pipeline, но требует наличия ПЛАНа okr-planner).
- **Выход:** `{status_workspace}/fact-pack.md` + `{status_workspace}/raw/agent-*.md`.
- **Зависит от:** артефакты okr-planner (INDEX, KR-EPIC-MAP, enriched),
  release-guard, sprint-fact, info-channels, People Graph (нексус team),
  `role_bindings` / `source_policy` из domain-profile.

### Стадия 2 — `/okr-status-assess <quarter>` (Progress Assessor)

- **Что делает:** для каждого KR считает тройку ПЛАН / ФАКТ / ИЗМЕНЕНИЕ.
  - **ПЛАН** ← enriched-okr: образ результата, IMP, срок, планируемый объём.
  - **ФАКТ** ← fact-pack: % готовности по progress-шкале `okr_standards.md`,
    SP done/total, что реально в проде. Каждый факт ← источник (JIRA-key /
    status.md / спринт-ФАКТ / якорь summary / узел People Graph).
  - **ИЗМЕНЕНИЕ:** светофор 🟢/🟡/🔴 (рубрика `status_standards.md`) +
    scope-дрейф и сдвиг срока (формулы `release_standards.md` §8) + прогноз EOQ
    (будет / частично / не будет, с обоснованием фактом) + доп. риски/ресурсы.
- **Вход:** `fact-pack.md` + ПЛАН.
- **Выход:** `{status_workspace}/assessment.md` (черновик по-KR).
- **Гейты:** каждый ФАКТ ← источник; нет выдуманного прогресса; прогноз EOQ
  обоснован; нет `[УТОЧНИТЬ]` в оценке Critical KR (IMP ≥ 8).

### Стадия 3 — `/okr-status-report <quarter>` (Reporter + Finalizer)

- **Что делает:** интерактивно финализирует отчёт с PO (правки фактуры,
  прогноза, рисков — по одному вопросу за раз), затем собирает финальный
  документ для руководства.
- **Структура отчёта:** executive summary (светофор по каждому OBJ + заголовок
  «что реально будет к концу квартала») → секция на каждый OBJ с таблицей
  ПЛАН/ФАКТ/ИЗМЕНЕНИЕ по его KR → сводные разделы: актуальность сроков,
  изменение объёма (агрегированный дрейф), прогноз EOQ, доп. риски/запросы
  ресурсов.
- **Вход:** `assessment.md` + правки PO.
- **Выход:** `{status_report_doc}` в `project-management/<quarter>/Статус/`.
  Опционально — выгрузка в Confluence (best-effort).

## Сквозные принципы

- **Нулевой допуск к галлюцинациям.** Каждый ФАКТ ← JIRA-key / `status.md` /
  спринт-ФАКТ / якорь summary / узел People Graph. Нет источника → `[УТОЧНИТЬ]`.
  Прогноз EOQ обязан быть обоснован фактом, не «на глаз».
- **Параллельность обязательна.** Субагенты A–C запускаются одновременно.
- **Конфликты не схлопываются.** Разногласие источников → явный раздел.
- **Честно про недоступность.** VPN/MCP недоступен → `[НЕДОСТУПНО: <источник>]`,
  не выдумывать состав ФАКТА.
- **STOP после каждой стадии.** Human-in-the-loop, особенно на финализации.
- **Переиспользование, не дублирование.** progress-шкала и IMP — из
  `okr-planner`; drift-формулы — из `release-guard`; ФАКТ спринта — из
  `sprint-fact`; сигналы каналов — из `info-channels`; люди — из People Graph.
- **Стиль артефактов** — по `skills/bft-writer/resources/writing_style.md`:
  деловая проза, семантические эмодзи только как маркеры (светофор, статус
  субагентов), без декоративных.

## Вне объёма (YAGNI)

- Автоматический пересчёт самого ПЛАНа (это задача `okr-planner` / груминга).
- Отдельный executive one-pager как второй файл (выбран один документ).
- Обязательная выгрузка в Confluence (оставлена опциональной).
- Прогнозные ML-модели velocity — прогноз EOQ делается эвристически на фактах
  (progress + carryover + оставшиеся спринты), с обоснованием.
