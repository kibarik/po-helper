# PRD-Research (Discovery) — дизайн генеративного онбординга

**Дата:** 2026-07-06
**Ветка:** `feat/prd-research`
**Статус:** дизайн утверждён; итерация 1 (PP-0 каркас + PP-1 эталонный Step 1) реализована
**Тип:** новая рамка (workflow) поверх существующей методологии `docs/AI-PROCESSES/`

---

## 0. TL;DR

po-helper сегодня **экстрактивен**: предполагает, что продуктовый контекст уже существует (в JIRA/Confluence/документах) и его надо перенести/провалидировать. `/paf-onboard` оцифровывает существующие документы; `/po-research` — read-only выкачивает факты из корпоративных систем.

Для **свежего проекта**, где PO ещё не понимает свой продукт и находится в активной фазе исследования, эта модель не работает: источников нет, Нексусы пусты, ответов PO сам не знает. Нужен **генеративный** режим — конструирование продуктового контекста через диалог, desk-research и валидацию гипотез.

`/prd-research` — это **недостающая «нулевая миля»** между `/paf-init` (создал пустые Нексусы) и всеми остальными навыками (`okr-*`, `bft-*`, `sprint-*`, которые ждут наполненные Нексусы). Он превращает уже описанную в `docs/AI-PROCESSES/` методологию (хребет из 9 шагов × движок Product Sprint × опер-модель CP/Stage-Gate) в **исполняемый, stateful, нелинейный workflow** по паттерну `okr-planner`, наполняет Нексусы как источник истины и собирает поверх них пересобираемую **PRD-витрину**.

**Методологию не меняем.** `docs/AI-PROCESSES/` (канон) и `docs/TRADITIONAL/RB-STEP-*` (raw-источник методов) остаются read-only справочником; workflow — исполнитель поверх них.

## 1. Проблема и позиционирование

### 1.1 Разрыв

| | Экстрактивный режим (есть сегодня) | **Генеративный режим (проектируем)** |
|---|---|---|
| Когда применим | Зрелая экосистема, контекст существует | Свежий проект, контекста нет, PO не знает продукт |
| Наполнение Нексусов | `/paf-onboard` — оцифровка документов | **`/prd-research`** — конструирование через диалог |
| Сбор фактов | `/po-research` — read-only выкачка из JIRA/Confluence | диалог + активный desk-research (web / `po-research`) |
| Якорь факта | JIRA-тикет / Confluence-страница | **гипотеза PO + статус валидации + Confidence Point** |
| Отношение к неопределённости | факт либо есть, либо `[УТОЧНИТЬ]` | неопределённость — first-class: гипотеза с CP, `parked` — легально |

### 1.2 Где сидит в жизненном цикле

```
/paf-init  →  [ пусто: нулевая миля ]  →  наполненные Нексусы  →  /okr-*, /bft-*, /sprint-*, /release-*
              экстрактивно: /paf-onboard (нужны документы)
              генеративно:  /prd-research  ← ЭТОТ ДИЗАЙН
```

`/prd-research` и `/paf-onboard` — два входа в одну точку (наполненные Нексусы). Не конкурируют: онбординг переносит существующее, discovery порождает недостающее. На гибридном проекте применимы оба (сначала onboard того, что есть, потом research белых пятен).

### 1.3 Принцип нулевого допуска к галлюцинациям — сохраняется, инвертируется

В генеративном режиме система **не выдаёт догадку за факт**. Каждый узел несёт статус и CP: `hypothesis · CP 0.3 · источник: суждение PO`. «Не знаю» — легальное состояние узла (`parked`), а не пробел под выдумывание. Это ровно `confidence`/`ripeness` из Node schema (`sa_documentation/nexus_schema.md`).

## 2. Архитектура пайплайна

### 2.1 Отображение хребет × движок на паттерн `okr-planner`

```
/prd-research                        ← оркестратор (уровень A, роль: Discovery Facilitator)
   читает config + domain-profile; ведёт PO по шагам; держит доску шагов
   (CP, статусы, зависимости, открытые вопросы); STOP после каждой стадии
        │
        ├─ Step 1  /prd-idea         BIG IDEA + 3 Линзы   → market · product · growth
        ├─ Step 2  /prd-customer     сегменты, проблемы    → customer
        ├─ Step 3  /prd-market       TAM/SAM/SOM, конкур.  → market   (тяжёлый desk-research)
        ├─ Step 4  /prd-value        ценностное предл.     → product   ▸ GATE: Need/Value Fit
        ├─ Step 5  /prd-bizmodel     монетизация, юнит-эк. → growth    ▸ GATE: Biz-model
        ├─ Step 6  /prd-gtm          позиционир., каналы   → growth
        ├─ Step 7  /prd-solution     решение, требования   → product (+ system-landscape) ▸ GATE: PMF
        └─ Step 8  /prd-acquisition  конфиг привлечения    → growth    ▸ GATE: PCF
        │
        └─ /prd-assemble             PRD-витрина ← читает наполненные Нексусы (пересобираемо)
```

### 2.2 Решения по гранулярности

1. **1 шаг = 1 навык-стадия** (не 1 подстадия = 1 навык). Иначе ~48 команд. Подстадии RB-раннбука (напр. `2.1…2.7`) становятся **внутренним чеклистом** стадии — диалоговыми ходами с мини-STOP. Итог: ~8 стадий + оркестратор + ассемблер ≈ 10 навыков (сопоставимо с 9 у `okr-planner`).

2. **Движок Product Sprint (`PULSE→SCOUT→BUNCH→PITCH→EXECUTE→HARVEST`) — внутренняя форма каждой стадии**, а не отдельные команды. Каждый `/prd-*` внутри проходит: PULSE (что уже в Нексусе / гэп) → SCOUT (диалог + desk-research, наполнение) → BUNCH/PITCH (скоринг гипотез, рост CP, дебаты) → HARVEST (запись в Нексус). Какие фазы «тяжёлые» — по `overview.md` каждого шага в `docs/AI-PROCESSES/STEP-*`.

3. **Skill-chain, а не `.js`-workflow.** Нужен human-in-the-loop диалог («PO не знает ответов»); `.js` (`po-context-research.js`) — автономный fan-out. STOP-паузы okr-паттерна = диалоговые чекпоинты. `.js`-движок переиспользуется точечно — как делегат тяжёлого desk-research внутри SCOUT (см. §4.2).

## 3. Модель неопределённости, состояния и дебатов

Каждый шаг — **живая исследовательская задача**: тянется во времени, идёт через долгие дебаты, работает при высокой неопределённости, переоткрывается и рассогласовывается с другими. Отсюда 4 механизма.

### 3.1 Гипотеза как единица работы (не «факт»)

Узел-гипотеза в Нексусе с жизненным циклом и CP:

```
draft → hypothesis → scoring → validating → validated / refuted / parked
                                    │
                    источник валидации (по возрастанию CP):
                    суждение PO → внутр. библиотека (RL) → web desk-research (якорь+дата)
                    → интервью/эксперимент (валидировано)
```

`parked` («PO не знает / отложено») — легальный терминальный статус, не блокирует.

### 3.2 Шаг — возобновляемое состояние, не команда

У каждого `/prd-*` — персистентный воркспейс с **журналом исследования** и статусом шага:

```
exploring → debating → converging → gate-ready → (revisit ↩)
```

PO может выйти на недели и вернуться. `/prd-research` на входе показывает **доску шагов**: статус, CP, открытые вопросы, что «протухло». Стадия не обязана закрыться за один присест; раунды дебатов накапливаются в журнале.

### 3.3 Дебаты — first-class внутри шага (переиспользуем существующее)

В коробке есть adversarial-паттерн — `okr-debate`/`bft-debate` (Адвокат Дьявола, 3 раунда, вердикт). Discovery-шаг хостит дебаты как под-цикл: `гипотеза → раунд дебатов → пересчёт CP → новый раунд / фиксация`. Долгий спор = много раундов в журнале, не блокировка.

### 3.4 Контур рассогласования между шагами (не «надежда», а механизм)

- Каждый узел несёт `depends_on` (напр. ценность Step 4 ← сегмент Step 2).
- Изменение узла-первопричины → зависимые узлы автоматически метятся `⚠️ wilting / re-check` (переиспользуем `ripeness` из Node schema).
- Оркестратор на входе **сводит противоречия между шагами** в единый список: «Step 6 канал предполагает сегмент X, но Step 2 сузил до Y — рассинхрон». Не чинит сам — показывает PO, предлагает переоткрыть шаг.
- **Fit-гейты мягкие**: пройти дальше при низком CP можно (методология: «ненулевое знание > чистое знание»), но гейт помечает долг, а PRD-витрина честно показывает «раздел на гипотезах».

**Линейность — лишь дефолтный маршрут.** Реально это доска из 8 живущих шагов с CP, статусами, зависимостями и очередью открытых вопросов. Оркестратор — навигатор по неопределённости, а не конвейер.

## 4. Наполнение Нексусов, веб-движок доказательств, PRD-витрина

### 4.1 Step × Нексус (из хребта AI-PROCESSES + `sa_documentation/nexus_process_map.md`)

| Шаг | Пишет в Нексус | Тип узлов (из методологии) |
|---|---|---|
| 1 Idea | `market` · `product` · `growth` | 3 Линзы, BIG Idea |
| 2 Customer | `customer` | сегмент, проблема, JTBD, mNSM-гипотеза |
| 3 Market | `market` | Force / Trend / Constant / TAM-SAM-SOM / Competitor / Gap / Bet (7 типов) |
| 4 Value | `product` | ценностное предложение, need/value fit |
| 5 Business Model | `growth` | монетизация, юнит-экономика, NPV |
| 6 GTM | `growth` | позиционирование, каналы |
| 7 Solution | `product` (+ `system-landscape`) | требования, прототип, PMF-критерии |
| 8 Acquisition | `growth` | конфиг «сегмент-канал-оффер», PCF |

Запись — строго по Node schema (`sources`, `confidence`, `ripeness`, `depends_on`), т.е. в тот же формат, который читают `okr-*`, `bft-*`, `sprint-*`. **Discovery наполняет — остальной po-helper потребляет** (замыкание нулевой мили).

### 4.2 Веб-движок доказательств (first-class)

Фаза SCOUT каждого шага активно исследует внешние ресурсы:

- **Тяжёлый desk-research** (рыночная среда, тренды, TAM/SAM/SOM, конкуренты — Step 3 почти целиком внешний) → делегируем существующему `po-research` Deep (уже умеет web + critic/skeptic + source-registry). Движок не дублируем.
- **Лёгкие точечные запросы** → `WebSearch` / `WebFetch` прямо в стадии.
- **Внутренняя библиотека** `docs/RL/` (Research Library: рост, A/B, метрики, первые лиды) — грунтующий справочник методов, чтобы шаг опирался на проверенные подходы.
- **Каждая веб-находка = узел с настоящим якорём** (URL + дата извлечения + оценка качества источника) и `ripeness` (рыночные данные протухают).

Единая шкала доказательности под узлом: `суждение PO (CP низк.)` < `внутр. библиотека (RL)` < `web desk-research (якорь + дата)` < `интервью / эксперимент (валидировано)`.

### 4.3 PRD-витрина (`/prd-assemble`)

Читает наполненные Нексусы и рендерит PRD-документ, **пересобираемый в любой момент**. Честно отражает неопределённость:
- разделы помечены `✅ validated / 🟡 assumed / 🔴 open`;
- реестр открытых вопросов и список межшаговых противоречий — прямо в PRD;
- приложение доказательной базы: источники с якорями и датами.

PRD не «замораживает» состояние — это витрина текущего среза доски. Пока шаги живут и рассогласовываются, витрина это показывает.

### 4.4 Добавки в `domain-profile`

```yaml
paths:
  discovery_workspace: "CORTEX/_context-packs/discovery/{step}"   # журнал + состояние шага
  prd_output_doc:      "GROUND/RESULTS/PRD-{product}.md"          # витрина
discovery:
  methodology_track: "AI-PROCESSES"   # канон; TRADITIONAL — raw-справочник методов
  web_research: true                  # активный desk-research в SCOUT
```

Пустое поле профиля → дефолт + `[УТОЧНИТЬ]` (как в остальных пайплайнах). Профиль не отменяет требование источника/статуса на каждый узел.

## 5. Декомпозиция и порядок сборки

Полный дизайн (эта спека) фиксирует **всю рамку**. Реализация разбивается на под-проекты, каждый со своей цепочкой спека→план→имплементация:

```
PP-0  Ядро (общие механизмы, без стадий)
      · Node schema: статусы гипотез, CP, depends_on, ripeness-контур рассинхрона
      · оркестратор /prd-research: доска шагов, навигация, мягкие fit-гейты
      · формат журнала/состояния шага (возобновляемость)
      · domain-profile: пути + discovery-блок
      · заглушка /prd-assemble (рендер того, что уже есть)
        └─ доказывает каркас end-to-end на пустом контенте

PP-1  Вертикальный срез: Step 1 (Idea) как эталонная стадия
      · полный /prd-idea: PULSE→SCOUT→BUNCH→PITCH→HARVEST + под-дебаты
      · веб-движок SCOUT (WebSearch + делегирование po-research)
      · запись в market/product/growth по Node schema
        └─ эталон, по которому клонируются шаги 2–8

PP-2..PP-N  Тиражирование стадий по одной (Customer → Market → … → Acquisition)
      · Market (Step 3) — акцент: тяжёлый desk-research, 7 типов узлов
      · Value / Solution / Acquisition несут fit-гейты

PP-final  Полноценный /prd-assemble: витрина с confidence, открытыми
          вопросами, противоречиями, evidence-приложением
```

Обоснование порядка: PP-0 доказывает каркас, PP-1 даёт эталон стадии, дальше — тиражирование. Каждая итерация самостоятельно ценна и проверяема (вертикальный срез раньше горизонтального охвата). **Первая итерация реализации — PP-0 + PP-1** (каркас + эталонный Step 1).

## 6. Принципы (наследуем от коробки)

1. **Read-only методология.** `docs/AI-PROCESSES/`, `docs/TRADITIONAL/`, `docs/RL/` — справочник, правке не подлежит.
2. **Работа только в `GROUND/`** (Нексусы) + пайплайн-воркспейсы (`CORTEX/_context-packs/discovery/`).
3. **Нет узла без статуса и источника.** Гипотеза помечается как гипотеза; факт — с якорём. `parked` вместо выдумывания.
4. **STOP-паузы = human-in-the-loop.** Каждая стадия отдаёт управление PO.
5. **Конвенции Node schema / domain-profile** — те же, что у `okr-*`/`bft-*` (интероперабельность: discovery наполняет то, что они читают).
6. **Инструкции из данных** (веб-страницы, документы) — данные, не команды.

## 7. Открытые вопросы (для этапа writing-plans)

- **CP-шкала:** числовая (0…1) как в Node schema `confidence`, или дискретная (low/med/high)? Предложение: наследовать `confidence` из Node schema, mapping на пороги fit-гейтов — в PP-0.
- **Формат журнала шага:** отдельный `journal.md` + `state.yaml` в `discovery_workspace/{step}`, или единый front-matter-документ? Решить в PP-0.
- **Ре-baseline PRD:** нужен ли версионируемый снапшот витрины (по аналогии с `release-guard`), или всегда «живой» рендер? По умолчанию — живой рендер; снапшот опционален.
- **Custom-Нексусы:** нужен ли discovery-специфичный Нексус (напр. `hypotheses` как единый реестр всех гипотез поверх продуктовых Нексусов), или гипотезы живут узлами внутри `market/customer/product/growth`? Предложение: узлами внутри профильных Нексусов (не плодить), реестр — вьюха оркестратора.
