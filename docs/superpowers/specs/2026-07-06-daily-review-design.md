# Дизайн: навык `/daily-review` — обработка дейликов и созвонов

**Дата:** 2026-07-06
**Ветка:** `feat/daily-review`
**Статус:** design approved → к плану реализации

---

## 1. Проблема и ценность

Дейлики и командные синки — **критический регулярный источник** для PO, который сейчас никак не оцифровывается в каркасе. На основе одного созвона PO нужно:

1. поменять статусы задач в JIRA (перемещение карточек);
2. актуализировать риски;
3. сформировать личные задачи PO на сегодня;
4. отследить ранее данные договорённости (сделал / не сделал → почему → новый план);
5. вести регулярный пульс команды.

**Самое ключевое для PO** (headline-ценность):
- **Актуальные сроки** — понимание, к каким датам реально выйдут вехи (Trunk/Prod/релиз), с обоснованием сдвигов.
- **Прогресс к цели спринта** — «кто что сделал сегодня и *действительно ли* действия служат Sprint Goal».
- **Цикл обязательств** — «исполнитель сделал, о чём договорились? как проверить? не сделал → почему → новый план».

Референс механики — `~/projects/mts-pm-worker` (транскрипт → decision-файл с жёстким форматом `📋 PO Actions / 📅 Daily Activity / 🎯 Goal Summary`). Берём формат и словарь PO-действий, но переносим на архитектуру po-helper: пути из `domain-profile`, якорение в NEXUS, хранение в PULSE, принцип нулевых галлюцинаций, разбивку по артефактам.

## 2. Форма и границы навыка

- **Один самостоятельный навык** `/daily-review` (не пайплайн стадий). Ежедневный темп — обработка за один прогон.
- **Вход** — сырой транскрипт стендапа/командного синка (аргумент, путь к файлу или вставка). Транскрипт часто содержит авто-саммари вверху (решения/задачи/сроки) — используется как **подсказка**, но каждый факт перепроверяется по сырому логу.
- **Охват** — стендапы и командные синки (формат «кто-что-к-цели»). Не-стендап созвоны (стейкхолдер-синк, custdev) → `/summary`. Единый стор `PULSE/meetings` держит **любые созвоны** в одном запрашиваемом месте.
- **JIRA — read-only, только план.** Навык НЕ читает и НЕ пишет JIRA через MCP. Пункт «синхрон доски» материализуется как *список действий PO* и *пульс перемещений*, а не как запись в трекер. (Осознанное решение: минимум необратимости, нет зависимости от JIRA MCP.)

## 3. Архитектура хранения (два слоя: знание + действие)

Два слоя с разными ролями:
- **Слой знания — PULSE-папка** (неизменяемая запись встречи, источник истины по нарративу/контексту).
- **Слой действия — Backlog.md** (живой личный планнер PO, источник истины по *статусу* открытых обязательств/блокеров/договорённостей; см. §10).

`/daily-review` работает как **ETL**: транскрипт → неизменяемая папка-снапшот встречи (PULSE) + проекция actionable-пунктов в Backlog.md (SMART-задачи) + дельта-патч рисков в NEXUS.

**Один прогон = одна папка-снапшот PULSE.** Каждый `/daily-review` создаёт `GROUND/PULSE/meetings/{type}-{YYYY-MM-DD-HHMM}/` (напр. `planning-2026-07-06-0958/`) и кладёт туда файлы **этой встречи** — это неизменяемая запись, она НЕ мутируется следующими прогонами.

**Carry-forward — из Backlog, не из папки.** Входное состояние («что уже висит») следующий прогон читает из **открытых задач Backlog.md** (слой действия), а не из прошлой папки. Это единственная истина статуса — нулевой дрейф. PULSE-папки — исторические снимки; эволюция статуса = Backlog + git.

```
GROUND/PULSE/meetings/{type}-{YYYY-MM-DD-HHMM}/     # неизменяемая запись встречи
  report.md         # тонкая эпизод-заметка (участники, прогресс-к-цели, план PO, цитаты)
  blockers.md       # ПОЛНЫЙ снапшот блокеров на момент встречи (копируется в чат/email)
  commitments.md    # снимок «как прозвучало»: обязательства этой встречи + отметки сверки
  sprint-pulse.md   # перемещения карточек ТОЛЬКО этого созвона (дельта-событие)
```

| Артефакт | Слой | Природа | Что несёт |
|---|---|---|---|
| `report.md` | PULSE | эпизод, неизменяемый | нарратив встречи |
| `blockers.md` | PULSE | снимок на момент встречи | все открытые блокаторы + актуальные даты Prod (для отправки) |
| `commitments.md` | PULSE | снимок «как прозвучало» | обязательства встречи + критерии проверки + отметки сверки |
| `sprint-pulse.md` | PULSE | дельта-событие | карточки, сдвинутые на этом созвоне |
| **Backlog.md** | **действие** | **живой трекер** | **открытые обязательства/блокеры/договорённости как SMART-задачи PO (истина статуса)** |
| `GROUND/NEXUS/risk` (дельта) | знание | durable-граф | риски — патч по ссылке, вне папки |

Обоснование:
- **PULSE** = «Progress Pulse, оперативная динамика продукта» — штатный дом. Git-tracked, RAG-индексируется. Папка-на-созвон даёт естественный аудит: полное состояние на момент каждого дейлика.
- **blockers/commitments — полные снапшоты** (не дельты): их ценность в «отправить целиком» и «увидеть всё висящее», поэтому каждый прогон переносит прошлое состояние и обновляет его. `sprint-pulse` — наоборот, чистая дельта (что двигали сегодня).
- **GROUND/NEXUS/risk** живёт вне папки как durable-граф (BFT-ярус): риск — сквозное состояние, не привязан к одному созвону.

### Различие блокатор / риск / обязательство

- **Блокатор** — конкретное препятствие на пути KR к отгрузке в Production, с датой снятия. Живёт в `blockers.md`, сгруппирован под KR. Двигает Prod-дату через зависимость от внешней команды или доп. объём.
- **Риск** — вероятностная угроза. Живёт в `GROUND/NEXUS/risk`.
- **Обязательство** — кто-что-к-сроку с критерием проверки. Живёт в `commitments.md`. Может быть владельцем снятия блокатора (cross-link по прозе/дате).

## 4. Форматы артефактов

### 4.1 blockers.md (снапшот, plain-text)

**Формат — plain text, факты, без иконок/таблиц/светофоров.** Артефакт копируется «как есть» в рабочий чат или корпоративный email. Никакой декоративной разметки: заголовок-цель, нумерованные блокаторы (`Блокатор N: <title>` + проза + `Срок:`), внизу «Актуальная дата релизов». Владелец — в прозе, не отдельным полем.

**Группировка — по KR «отгрузка в Production».** Заголовок-строка = KR / релиз, который выходит в Prod (`Статус релиза «Кино»` ↔ соответствующий KR). Каждый блокатор — это ответ на «когда задача появится в Production» и двигает Prod-дату KR через зависимость от внешней команды или доп. объём. Секция «Актуальная дата релизов» = Prod-дата этого KR (+ Trunk-веха). Если KR несколько — несколько самодостаточных plain-блоков, каждый копируется отдельно. Источник KR/дат — `okr_output_doc` + `KR-EPIC-MAP`; не сматчилось → `[KR?]`.

Эталон (из примера PO):

```markdown
<!-- updated: 2026-07-06 · source: 2026-07-06-kino-reliz -->

Статус релиза «Кино»

Блокатор 1: BE-ручка + ревью по БД
С нашей стороны задержка — планировали выкатить в четверг, фактически накатим в понедельник вечером. Выкладка блокирует работы на FE. Из блокеров остался только маппинг городов — надо придумать решение под выкатку.
Срок: выкатим 6–7 июля.

Блокатор 2: не работает выдача афиши
Остался баг: страница афиши отдаёт 404.
Срок: выкатим 6–7 июля.

Блокатор 3: FE-доработки в UJM
После выкатки BE нужно закатить фиксы: убрать хардкоды и починить все кнопки.
Срок: хотелось бы успеть до 10 июля.

Блокатор 4: дизайн-ревью
После FE-доработок проходим дизайн-ревью.
Срок: до 10 июля.

Блокатор 5: фид-менеджер
Пока непонятно, где и что править. Много доработок в PHP-коде, с которым мы не работаем. Скорее всего исправим на неделе 13–17 июля.

Блокатор 6: SEO-теги
На стороне фронта. С Катей Осташевой общались — блокаторов, вроде, не будет.

Блокатор 7: жанры в админке
Не блокатор в целом, просто нужно чтобы кто-то руками прошёл после нашей миграции и команда SEO завела правильную таксономию. Дальше по этой таксономии сможем прогонять данные.

Актуальная дата релизов:
Trunkweb: 13–15 июля — закатываем, проходим все ревью и стабилизацию на транке.
Production: ближе к 20-м числам — нужно успеть подправить фид-менеджер (блокатор 5).
```

Правила ведения (машинная сторона, **не попадает в текст**):
- **Метаданные** — единственная строка HTML-комментария сверху (`<!-- updated · source -->`); при копировании невидима, при вставке легко отбрасывается. Больше никакого frontmatter.
- **Нумерация стабильна в рамках цели** — снятый блокатор не сдвигает номера: помечается снятым и уезжает вниз/удаляется на следующем прогоне (историю даёт git). Новый — следующий номер.
- **Отложенное («не блокирует релиз»)** — только если реально прозвучало: отдельный блок `Не блокирует релиз:` тем же plain-стилем (как PATCH-пункты в исходном примере PO), с `@исполнителем` в прозе. Пустой блок не выводить.
- **Снятие блокатора** порождает сверку в `commitments.md` (кто снял, как проверено) — линк по прозе/дате, без служебных полей в тексте.

**Пересчёт «Актуальной даты релизов»:** после upsert блокаторов навык агрегирует их сроки в вехи (Trunk/Prod/релиз — набор из плана/OKR) и, если дата сдвинулась, в скобках указывает причину прозой (`нужно успеть подправить фид-менеджер (блокатор 5)`). Без символов Δ/стрелок.

> **Принцип минимализма** распространяется на все человеко-facing выводы: эпизод-заметка и план PO — тоже факты без гирлянд иконок (теги прогресса словами `на цель / по касательной / мимо / вне плана`, а не emoji). Табличные рабочие реестры (`sprint-pulse.md`, `commitments.md`) остаются таблицами для сканирования, но без emoji.

### 4.2 sprint-pulse.md (дельта созвона, внутри папки)

```markdown
## Перемещения карточек — planning «кино», 2026-07-06
| Задача | Было → Стало | Кто | Комментарий |
|---|---|---|---|
| [КОД?] Ограничение кино в рекл. фиды | Backlog → In Progress | Рустам | 3 сервиса, до конца недели |
| [КОД?] Правила экономикализатора | In Progress → Review | Дмитрий | отправлено на ревью |
```

Отражает **только перемещения этого созвона** (без записи в JIRA). «Пульс во времени» = последовательность папок; для сводного взгляда читаются `sprint-pulse.md` из нескольких последних папок.

### 4.3 commitments.md (снимок встречи «как прозвучало»)

Неизменяемый снимок обязательств **этой встречи**: новые обязательства + отметки сверки прошлых (что подтвердилось/сдвинулось на этом созвоне). Истина статуса живёт в **Backlog.md** (§10); здесь — нарративный след с критериями проверки для аудита. Carry-forward прошлых обязательств берётся из открытых задач Backlog, а не из этого файла. Без emoji.

```markdown
# Обязательства

| id | открыто | владелец | обязательство | критерий проверки | срок | статус | последняя сверка | новый план |
|----|---------|----------|---------------|-------------------|------|--------|------------------|-----------|
| C-012 | 2026-07-06 (kino) | [КТО?] | накатить ручку BE + ревью БД | ручка отвечает на dev, FE разблокирован | 6–7 июля | open | — | — |
| C-011 | 2026-07-02 (daily) | Дмитрий | правила экономикализатора | PR на ревью, аппрув 1 | 2026-07-06 | done | 2026-07-06 | — |
```

Статусы: `open · done · slipped · dropped`.
**Критерий проверки** — ключевое поле (ваше «как проверить?»). Нет в транскрипте → `[КАК ПРОВЕРИТЬ?]`, PO дозаполняет.
**Сверка (reconcile):** на каждом прогоне открытые обязательства **из Backlog** сопоставляются с сегодняшним прогрессом → `done` / `slipped` (+ причина + новый план) / `open`; результат отражается и в этом снимке, и в Backlog (§10).

## 5. Пайплайн (стадии одного прогона)

0. **Загрузка + carry-forward + напоминания** — `.claude/domain-profile.md` (`paths`, `jira`, `planner`, `meta.current_quarter`), SKILL, `resources/*`. Резолв **Sprint Goal + плановых историй** (`planning_root/SPRINT-*`), **OKR-квартала** (`okr_output_doc`), **People Graph** (`GROUND/NEXUS/team`), **KR-EPIC-MAP**. Прочитать **открытые задачи Backlog** (labels `commitment`/`blocker`/`agreement`) как входное состояние. **Напоминания:** сразу показать блок «Висит на PO» — просроченные и скоро-по-сроку задачи Backlog.
1. **Приём транскрипта** — arg/путь/вставка. Пусто → попросить. Авто-саммари сверху — подсказка, факты ← сырой лог.
2. **Извлечение** — активность по участникам, статус-изменения карточек, блокаторы, риски-дельта, новые договорённости, упоминания задач и вех/дат.
3. **Якорение** — коды JIRA ← KR-EPIC-MAP/план (неизвестно → `[КОД?]`); имена ← team (не сматчено — не додумывать); блокаторы → KR отгрузки в Prod (`[KR?]`).
4. **Прогресс-к-цели** — тег каждой активности словами (`на цель / по касательной / мимо / вне плана`) против Sprint Goal.
5. **Сверка обязательств** — открытые задачи Backlog × сегодняшний прогресс → done/slipped/open + новый план.
6. **Блокаторы** — собрать полный снимок (открытые blocker-задачи Backlog + сегодняшние), группировка по KR заголовком `Статус <KR/релиз>`, пересчёт «Актуальной даты релизов» с обоснованием сдвига прозой.
7. **Сборка папки PULSE** — создать `meetings/{type}-{YYYY-MM-DD-HHMM}/` и записать `report.md` + `blockers.md` + `commitments.md` (снимок встречи) + `sprint-pulse.md`; риски-дельта → `GROUND/NEXUS/risk` (вне папки).
8. **Проекция в Backlog (dry-run → approve)** — по правилам §10: новые обязательства/блокеры/договорённости → создать SMART-задачи (labels); `done` → закрыть; `slipped` → сдвинуть `Срок:` в описании + заметка. Backlog = `planner.type: none` → стадия пропускается.
9. **Патчи + STOP** — показать превью PULSE-файлов (особенно `blockers.md` — готов к отправке) и dry-run проекции Backlog (что создать/закрыть/сдвинуть), затем записать после approve. **Отчёт PO**: путь папки, блок «Висит на PO», места `[УТОЧНИТЬ]`, сводка «что куда записано».

**Принцип нулевых галлюцинаций.** Каждый факт ← транскрипт (цитата/явное упоминание). Плейсхолдеры вместо догадок: `[КОД?]`, `[КТО?]`, `[KR?]`, `[КАК ПРОВЕРИТЬ?]`, `[УТОЧНИТЬ]`. Искажения распознавания имён/терминов сводятся к единому написанию (сверка с `GROUND/NEXUS/team`), в конце — пометка «термины сведены, проверь глазами».

## 6. Изменения в domain-profile

В секцию `paths` шаблона `domain-profile.template.md` добавить (с дефолтами):

```yaml
  meeting_notes:  "GROUND/PULSE/meetings"   # корень; внутри — папки {type}-{YYYY-MM-DD-HHMM}/
```

Имена файлов внутри папки (`report.md`, `blockers.md`, `commitments.md`, `sprint-pulse.md`) фиксированы навыком — отдельных path-полей не заводим.

Новая секция `planner` (слой действия PO):

```yaml
planner:
  type: "backlog"          # backlog | none — none отключает проекцию в Backlog.md
  root: "backlog"          # корень Backlog.md в репозитории (backlog init)
  labels:                  # метки для трёх типов проецируемых задач
    commitment: "commitment"
    blocker:    "blocker"
    agreement:  "agreement"
  reminder_horizon_days: 3 # «due-скоро» для блока «Висит на PO» на старте
```

Профиль пуст → дефолты + пометка `[УТОЧНИТЬ]` (как в `/summary`). `planner.type: none` → навык работает только со слоем PULSE, проекция пропускается.

## 7. Связи с каркасом

- **Питает `/sprint-fact`** — дневные факты (velocity-сигналы, carryover, снятые/висящие блокаторы) накапливаются как вход в ФАКТ спринта → следующий `/sprint-sync`.
- **Дополняет `/summary`** — общий стор `PULSE/meetings`; стендапы через `/daily-review`, прочие созвоны через `/summary`.
- **Читает** артефакты `sprint-planner` (Sprint Goal, план), `okr-planner` (OBJ/KR), `people-map`/`team`-Нексус (атрибуция + «к кому идти» в плане PO).
- **Пишет** в `GROUND/NEXUS/risk` (BFT-ярус) — при наличии графа рисков; иначе fallback-пометка.
- **Проецирует в `Backlog.md`** через MCP `backlog` — личный планнер PO (≠ командный JIRA). Слой действия, см. §10.

## 9. Что осознанно НЕ делаем (YAGNI)

- Не читаем/пишем **командный JIRA** через MCP (только план). Backlog.md — отдельный личный планнер, в него пишем (§10).
- Не пайплайн из стадий-навыков — один прогон.
- Не выделяем отдельный `GROUND/NEXUS/commitments` — снимок живёт в PULSE, статус — в Backlog.
- Не меняем `/summary` — переиспользуем стор, но не трогаем сам навык.
- Не строим отдельный планировщик напоминаний — только тонкий блок «Висит на PO» на старте (§10). Scheduled-task/loop — следующая итерация.
- Не строим авто-агрегацию в `/sprint-fact` в этой итерации — только оставляем совместимый формат-вход.

## 10. Backlog.md — слой действий PO

**Роль.** PULSE хранит *знание* (что сказали, критерии, история). Backlog.md — *действие*: личный планнер PO, где каждый требующий догляда пункт = SMART-задача, которую агент ведёт и о которой напоминает. Backlog.md ≠ командный JIRA (это локальный трекер PO, MCP `backlog`), поэтому запись сюда не нарушает «JIRA read-only».

**Backlog = истина статуса.** Открытые задачи Backlog — единственный источник «что висит». `/daily-review` читает их на входе (carry-forward) и обновляет на выходе. PULSE-снимки статус не ведут.

### Что проецируется (три типа, каждый — со своей label)

| Тип | Когда | label | Источник в PULSE |
|---|---|---|---|
| **commitment** | человек взял обязательство (кто-что-к-сроку) | `commitment` | `commitments.md` |
| **blocker** | блокатор требует действия PO для снятия | `blocker` | `blockers.md` |
| **agreement** | договорённость, которую PO должен держать/контролировать | `agreement` | `report.md` |

НЕ проецируется: перемещения карточек (`sprint-pulse` — забота командной доски), рутинная активность, чисто информационные договорённости (без действия PO).

### SMART-маппинг (проекция почти механическая)

Поля обязательства ложатся на SMART 1:1:

| SMART | Поле |
|---|---|
| **S**pecific | обязательство / заголовок задачи |
| **M**easurable | критерий проверки → acceptance criteria задачи |
| **A**ssignable | владелец → assignee/упоминание |
| **R**elevant | привязка к KR (отгрузка в Prod) |
| **T**ime-bound | срок → в описание задачи (у Backlog.md нет поля `due`; хранится текстом `Срок: <дата>`) |

Тело задачи несёт срок + обратную ссылку на источник: `Срок: <дата> · ← C-NNN · {meeting_notes}/{папка}` — срок для «Висит на PO» (парсинг из описания) + backlink для аудита.

### Reconcile (стадия 8 пайплайна, dry-run → approve)

1. **Match** — открытые задачи Backlog сопоставить с сегодняшними пунктами (по ссылке C-NNN/B-N или по смысловому совпадению).
2. **done** — обязательство выполнено по критерию проверки → закрыть задачу.
3. **slipped** — задержка → обновить `Срок:` в описании/`--notes` + добавить новый план (не плодить дубль).
4. **open (new)** — впервые прозвучавший пункт → создать SMART-задачу с нужной label.
5. **carry** — не упоминалось сегодня → оставить как есть.
6. **dropped** — явно отменено → закрыть с пометкой.

Идемпотентность: повторный прогон того же транскрипта не создаёт дублей (match по ссылке/смыслу).

### Напоминания (тонко, стадия 0)

На старте показать блок **«Висит на PO»**: задачи Backlog, у которых распарсенный из описания `Срок: <дата>` уже прошёл (просрочка) или наступит в пределах `planner.reminder_horizon_days` (скоро). Срок читается best-effort из текста описания — отдельного поля `due` в Backlog.md нет. Это единственная механика напоминаний в этой итерации; регулярный планировщик (scheduled-task/`/loop`) — позже.

### Деградация

`planner.type: none` или нет MCP `backlog` → стадии 0-carry и 8 пропускаются, навык работает только со слоем PULSE (`commitments.md` тогда — самостоятельный снимок без проекции). Явно сообщить PO, что проекция отключена.
