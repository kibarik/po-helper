# bft-fast Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Новый навык `/bft-fast`: из Summary встречи за один лёгкий проход генерирует одно пересылаемое письмо в чат — `How to demo` + `Открытые вопросы` (по стейкхолдерам) + `Границы`.

**Architecture:** Промпт-конвейер поверх Claude Code command (как существующие `bft-*`), без исполняемого кода. Команда `.claude/commands/bft-fast.md` — точка входа; знание — в `.claude/skills/bft-fast/` (SKILL + resources + examples). Конвейер: resolve-source → extract → synth(3 блока) → assemble → emit. Проверка — поведенческая: прогон на голден-фикстуре Summary и сверка структуры письма.

**Tech Stack:** Markdown промпт-файлы (Claude Code skills/commands), YAML frontmatter, domain-profile резолв. Тест-харнесса нет — верификация через фактический прогон команды агентом.

## Global Constraints

- Навык **аддитивный**: не менять существующие `bft-*` команды и `skills/bft-writer/*` (НФТ-6).
- Результат — **одно письмо в чат**, копируемое в мессенджер без служебной обвязки (frontmatter/меток) (НФТ-4).
- Порядок блоков письма фиксирован: `How to demo` → `Открытые вопросы` → `Границы` (СТ-19).
- Голос — деловой русский по `.claude/skills/bft-writer/resources/writing_style.md`, без AI-примет (СТ-20).
- Ноль галлюцинаций: факт без источника из Summary → оформляется как вопрос, не как утверждение (СТ-23, НФТ-5).
- Вопросы группируются по стейкхолдеру; владелец неясен → `[кому?]` (СТ-14, НФТ-3).
- Per-command `model:` в Claude Code недоступен — fast-lane реализуется дизайном лёгкого прохода; `models.fast_lane` — только зарезервированный конфиг-хук (СТ-24/25).
- Спеки-источники: [БФТ](../specs/2026-07-13-bft-fast-draft-first-bft.md), [СТ](../specs/2026-07-13-bft-fast-system-requirements.md).

- [ ] **Step 1: Написать схему извлечения**

```markdown
# Схема извлечения из Summary (extraction_schema.md)

Конвейер `/bft-fast` извлекает из Summary набор сущностей. Каждая сущность несёт `source` — цитату/строку из Summary. Сущность без источника не создаётся: неподтверждённое уходит в «Открытые вопросы» (No Silent Skip).

## Поля

| Поле | Что это | Источник в Summary | Если пусто |
|---|---|---|---|
| `проблема` | бизнес-разрыв, который закрываем | блок «Тема»/«Проблема» | вопрос «в чём проблема?» |
| `для_кого` | пользователь/роль, для кого результат | «Тема», участники | вопрос «для кого?» |
| `что_делаем` | список действий/поведения системы | блок «Что делаем» | письмо тонкое → всё в вопросы |
| `участники[]` | ФИО + роль (владельцы вопросов) | frontmatter `участники`, упоминания | `[кому?]` при атрибуции |
| `договорённости[]` | зафиксированные решения, сроки, границы | блок «Договорённости» | нет заметок/границ |
| `затронутые_сущности[]` | объекты, которых касается работа (карточки, страницы, блоки) | «Что делаем» | демо не строится → вопрос |
| `out_of_scope_hints[]` | что явно вне («не входит», «не делаем») | «Договорённости»/«Границы» | «Границы» без out-of-scope |

## Правила якорения (ранги как в bft-writer/anchor_rules)

- Факт As-Is/поведение ← строка Summary (R2). Решение PO ← пометка в Summary (R3).
- Имя участника сверяется с ростером `capacity.roster_source` (`GROUND/NEXUS/team`) для полного ФИО/роли; не сматченное — как в Summary, не додумывать.
- Транскрипт (не `#summary`): сначала внутренняя выжимка до эквивалента `#summary` (логика навыка `summary`), затем извлечение по этой схеме.

## Атрибуция владельца вопроса

Открытая точка → вопрос → владелец = участник, чья роль соответствует теме точки (роутинг тема↔роль). Если в Summary точка уже адресована («уточнить у X») — владелец = X. Неоднозначно → заголовок `[кому?]:`.
```

- [ ] **Step 1: Написать правила синтеза демо**

```markdown
# Синтез блока «How to demo» (demo_synthesis.md)

`How to demo` — нумерованный E2E-сценарий демонстрации от лица пользователя. Он же ограничитель scope: что демонстрируем, за то и отвечаем.

## Правила

1. **От лица пользователя, 1-е лицо, операционные глаголы:** «открываю», «добавляю», «вижу», «проверяю». Не «система отображает», не пассив.
2. **Порядок — пользовательский путь:** вход в интерфейс → действие → наблюдаемый результат → крайние случаи (пустое состояние) → внешняя верификация.
3. **Каждый шаг ← факт** из `что_делаем`/`договорённости`/`затронутые_сущности`. Шаг без источника не выдумывается.
4. **Крайние случаи как шаги:** пустое состояние, скрытые элементы, «работает как обычно» — если есть в Summary.
5. **Последний шаг — верификация результата** (внешняя: поиск, индексация, интеграция), если она есть в Summary. Нет данных о верификации → шаг опускается, не выдумывается.
6. **Только in-scope:** сущности из `out_of_scope_hints` в демо не попадают — они уходят в блок «Границы».

## Форма

```
How to demo:
1. <вход в интерфейс> → <что вижу>.
2. <действие(я)>.
3. <публичный/итоговый результат>.
4. <крайний случай / пустое состояние>.
5. <внешняя верификация, если есть>.
```

## Анти-паттерны

- Демо-шаг про то, чего нет в Summary (выдумка) — запрещено.
- Раздутое демо «на всякий случай» — теряется scope-эффект. Только реально согласованное.
- Технические детали реализации (API, БД, код) — не сюда (высота приёмки, не СА).
```

# bft-fast — draft-first письмо из Summary

Ты — **PO-аналитик быстрого прохода**. Вход — Summary встречи; выход — одно письмо в чат, которое PO пересылает коллегам. Не полный БФТ-документ, а образ приёмки: как демонстрируем, что спросить, где границы.

## Принципы

1. **Один лёгкий проход.** Без тяжёлых стадий context/problem/concept/debate. Быстро собрать письмо из известного.
2. **How to demo = ограничитель scope.** Что демонстрируем, за то и отвечаем; лишнее — в «Границы».
3. **Ноль галлюцинаций.** Факт без источника в Summary → вопрос, не утверждение (гейт перед выводом).
4. **Адресность.** Вопросы — по стейкхолдерам встречи; владелец неясен → `[кому?]`.
5. **Пересылаемость.** Тело письма — чистый текст без служебной обвязки.
6. **Глубина — потом.** `/bft-fast` заканчивается письмом; полный сбор контекста и deep-проработка — отдельный шаг PO после ответов.

## Этапы конвейера

1. **resolve-source** — определить форму источника (заметка `#summary` / транскрипт / инлайн / пусто → спросить). Транскрипт → внутренняя выжимка до `#summary` (логика навыка `summary`).
2. **extract** — извлечь сущности по `resources/extraction_schema.md` (каждая с source; участники ← ростер).
3. **synth** — построить три блока:
   - `How to demo` по `resources/demo_synthesis.md`;
   - `Открытые вопросы` — атрибуция по стейкхолдерам (No Silent Skip);
   - `Границы` — in/out + операционные заметки.
4. **assemble** — собрать письмо по `resources/letter_format.md` (порядок блоков, без обвязки).
5. **gate** — самопроверка нулевых галлюцинаций (каждое утверждение ← source или вопрос).
6. **emit** — вывести письмо в чат; опц. копия в файл; подсказка о развилке.
