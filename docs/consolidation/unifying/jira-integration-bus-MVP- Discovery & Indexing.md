**Синтез проблемы и механизма решения:**

Текущее взаимодействие с JIRA характеризуется неэффективным поиском (большой расход токенов на чтение целых страниц для проверки), отсутствием знаний о workflow (ошибки при смене статусов) и пробелами в индексации контекста по сравнению с Confluence. Для решения необходимо создать фундаментальный уровень интеграции (**JIRA Integration Bus MVP**). Решение включает реализацию навыка индексации данных JIRA в граф знаний, создание метадата-реестра для быстрого поиска и легкого доступа, а также механизм Discovery для сохранения схем переходов статусов, что позволит агенту понимать структуру внешней системы без активных изменений.

**Требования (агрегированы):**

*   Реализовать новый навык `jira-indexator` (multi-step pipeline из 6 стадий, зеркалящий `confluence-indexator`) для обхода проектов/бордов/эпиков, индексации знаний и их связывания с OKR-целями, спринтами и фразами сотрудников.
  `— from #57` contributed_requirement: Навык индексирования JIRA в граф знаний (jira-indexator, /jindex-*)

*   Создать реестр ссылок (link registry) с легкими карточками (metadata-only) по key/URL JIRA и реализовать metadata-first доступ (тонкий враппер над JIRA MCP), чтобы использовать дорогой `get` только при реальной необходимости контента.
  `— from #60` contributed_requirement: Metadata-first доступ к JIRA/Confluence: реестр ссылок с лёгкими карточками

*   Реализовать Discovery-шаг workflow: агент должен одноразово (или по инвалидации) получать схемы статусов и переходов через JIRA MCP и сохранять карту `issue_type → {statuses, transitions[]}` в GROUND для использования как guardrail перед изменениями.
  `— from #63` contributed_requirement: Discovery workflow JIRA: агент сохраняет схему статусов+переходов по каждому issue type как знание
