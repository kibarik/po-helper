## Синтез проблемы и механизма решения

Текущая архитектура характеризуется избыточной нагрузкой на тяжёлые модели, отсутствием гибкости в выборе инструментов и ручным управлением синхронизацией контекста. Для повышения эффективности и расширения функциональности необходимо внедрить **слоя LLM-оркестрации (LLM Orchestration Layer)**.

Данный механизм выступает в роли интеллектуального диспетчера, который решает следующие задачи:
*   **Умная маршрутизация и тиринг моделей:** распределение запросов между быстрыми (для интерактива, сбора контекста, дайджестов) и тяжёлыми (для глубокой проработки) моделями в зависимости от фазы навыка.
*   **Абстракция инструментов:** переход от хардкода имён к резолву тулов по декларируемым способностям (capabilities), что позволит интегрировать пользовательские скрипты и MCP-серверы.
*   **Автоматизация воркфлоу:** оркестрация фоновых задач и синхронизации данных по расписанию (cron) для подготовки утренних сводок.
*   **Мультиагентность и изоляция:** поддержка запуска изолированных инстансов агентов (микро-PO-helper) для исполнителей с возможностью выравнивания контекста через дебаты.

## Требования (агрегированы)

*   **Роутинг операций и параллелизм:** Реализовать механизм выбора модели (роутинг) и параллельного выполнения независимых внешних вызовов (JIRA/Confluence) для устранения долгого отклика и ускорения работы с PO-helper.
    `— from #61 contributed_requirement: Ускорить взаимодействие с PO-helper: тест альтернативных LLM/роутинг по операциям + кратное ускорение тяжёлых шагов`

*   **Реестр кастомных тулов и Capability-based резолв:** Создать слой реестра, поддерживающий кастомные инструменты пользователя, где скиллы запрашивают «способность» (например, источник транскриптов), а система возвращает конкретный инструмент, избегая хардкода имён.
    `— from #78 contributed_requirement: Интеграция кастомных тулов пользователя (свой MCP JIRA, скрипты транскриптов/чатов): реестр тулов + резолв по capability вместо хардкода имён`

*   **Оркестрация автосинхронизации:** Разработать навык-оркестратор (например, `/morning-brief`), который последовательно или параллельно вызывает команды синхронизации и запускается по расписанию (cron), обеспечивая готовность контекста к началу рабочего дня.
    `— from #79 contributed_requirement: Автоподгрузка и синхронизация данных до старта рабочего дня → утренняя сводка PO`

*   **Матрица выбора модели (LLM Tiering):** Внедрить механику переключения моделей внутри одного навыка на основе матрицы «фаза навыка → модель»: быстрая для интерактива и сбора контекста, тяжёлая с большим окном — для автономной проработки.
    `— from #112 contributed_requirement: Тиринг LLM по задачам: быстрая модель на сбор контекста/мелочь + тяжёлая с большим окном на автономную проработку`

*   **Изоляция доменов и дебаты агентов:** Обеспечить поддержку создания микро-PO-helper для каждого исполнителя (execution-домен) с навыками защиты границ и механизмом эмерджентного выравнивания через adversarial-дебаты агентов.
    `— from #141 contributed_requirement: Микро-PO-helper на каждого исполнителя (execution-домен): защита границ + самовыравнивание контекста через дебаты агентов вместо top-down JIRA/Confluence`

*   **Fast-lane дайджесты для чатов:** Реализовать on-demand навык для создания мобильных дайджестов тредов, используя быструю модель (fast_lane) с детекторами «адресовано мне»/«уже отвечено» и строгим гейтом на галлюцинации.
