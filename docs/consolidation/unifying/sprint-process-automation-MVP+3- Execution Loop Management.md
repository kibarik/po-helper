### Проблема и механизм решения
Текущий процесс управления спринтом требует значительных ручных усилий и характеризуется дискретностью. Закрытие спринта превращается в длительную ручную сверку план/факт (до часа), так как JIRA не является источником правды (закрытые задачи не равны сделанным). Оценка задач выполняется вручную, а актуализация беклога происходит рывками в конце спринта, а не непрерывно.

Механизм решения направлен на автоматизацию жизненного цикла спринта (Execution Loop). Он включает:
1. Внедрение ИИ-оценок в Story Points по умолчанию при создании задач, с возможностью корректировки командой для калибровки системы.
2. Реализацию DayByDay-актуализации для непрерывной синхронизации план/факт и автономного управления беклогом (перенос, приоритизация).
3. Внедрение протокола быстрого закрытия спринта на основе накопленных фактов, что позволяет сократить время согласования до 5 минут.

### Требования (агрегированы)
*   Внедрить протокол закрытия спринта, позволяющий согласовать план/факт за 5 минут вместо часа, так как JIRA не отражает фактическое состояние. Необходимо использовать команду `/sprint-fact` для сбора факта спринта за один проход и задействовать `po-helper` для быстрого согласования статуса изменений вместо длительного выравнивания. — from #132
*   Настроить ИИ-систему так, чтобы она оценивала задачу в Story Points по умолчанию при создании (оценка ИИ считается достаточной). Команда должна иметь возможность изменить оценку самостоятельно, но с обязательным обоснованием; при этом расхождение между оценкой ИИ и финальной оценкой команды должно копиться как сигнал калибровки. — from #136
*   Реализовать DayByDay-актуализацию спринта для устранения разрыва контекста план/факт между спринтами. Обеспечить ежедневный проход по расписанию для считывания сигналов дня, обновления статусов, подсветки дельты план/факт и принятия автономных решений по беклогу (перенос, переприоритизация, флаг блокеров) с прозрачным логом, чтобы к концу спринта факт уже был собран. — from #137
