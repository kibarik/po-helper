### Синтез проблемы и решения

Текущая реализация GROUND Vault имеет ряд критических ограничений: доступ к данным осуществляется через неэффективные прямые обходы (grep/echo), что приводит к высокому расходу токенов; отсутствие системного механизма актуальности (pull-модель) делает контекст устаревшим; а ядро системы (CORTEX) уязвимо для загрязнения данными из внешних доменов без проверки релевантности.

Для решения этих задач в рамках инкремента MVP+1 необходимо реализовать комплексный механизм:
1. Внедрить полноценный поисковый слой (LanceDB) и графовый движок зависимостей для оптимизации сбора контекста.
2. Обеспечить защиту ядра через Relevance-гейт и карантинную зону для фильтрации входящих артефактов.
3. Перейти к событийной модели обновления данных (Freshness) через вебхуки для поддержания достоверности контекста.

### Требования (агрегированы)

*   **Реализовать поисковый слой и графовый движок зависимостей.**
    Необходимо заменить примитивные grep/echo-обходы на нормальный поисковый слой между навыками и Vault. Использование полнотекстового/семантического поиска (LanceDB) и графового движка обеспечит получение навыком отранжированного узкого набора узлов, что существенно снизит расход токенов на сборку контекста.
    — from #59 contributed_requirement: Поисковый слой вместо grep/echo с использованием LanceDB и графового движка для снижения токенозатрат.

*   **Внедрить движок актуальности и достоверности контекста (Freshness Engine).**
    Требуется отказаться от отстающей пул-модели в пользу push-модели: настроить вебхуки от JIRA/Confluence для точечного обновления затронутых узлов. Добавить на узлы метаданные свежести (`updated_at`, `ttl/stale_after`) и достоверности (`confidence`, `verified_by`, `conflicts_with`).
    — from #75 contributed_requirement: Движок актуальности и достоверности контекста на основе вебхуков и метаданных.

*   **Реализовать Relevance-гейт и карантин по происхождению.**
    Необходимо настроить фильтрацию входящих артефактов: артефакт считается релевантным, если есть заякоренный граф-путь от него к strategy/KR-узлу (в пределах N хопов). Входящие данные из чужих доменов (verdict: foreign/adjacent) не должны попадать напрямую в CORTEX, а должны помещаться в карантин (`GROUND/_intake/`) с provenance-тегом.
    — from #100 contributed_requirement: Relevance-гейт + карантин по происхождению для защиты ядра от некорректного ввода.
