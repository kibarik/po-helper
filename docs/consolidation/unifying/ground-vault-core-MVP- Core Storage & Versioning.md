## Синтез проблемы и механизма решения

Текущее решение на базе плоских .md-файлов демонстрирует недостаточную производительность и вызывает чрезмерный расход токенов при масштабировании. Отсутствует прозрачная ретроспектива происхождения данных в памяти, а также существует риск потери контекста между рабочими сессиями, требуя реализации контроля задач.

В рамках инкремента `ground-vault-core` предлагается реализовать фундаментальную архитектуру хранения и управления данными. Механизм решения включает:
1. **Двухъярусное хранилище**: Переход на векторные движки (аналог NotebookLM/Нейроэксперт). Первый ярус обеспечивает быстрый прием файлов, второй (GROUND Vault) — долгосрочное консолидированное хранение.
2. **Git-подобное версионирование**: Организация памяти (CORTEX) как временного ряда с ветвлением (поколения CORTEX → W+n), где каждая запись — коммит с атрибуцией, что обеспечивает полную ретроспективу через entire.io.
3. **BFT Backlog Control**: Внедрение трекера продолжения работы и контроля состояний задач (5 стадий) для сохранения контекста сессий и управления потоками данных без необходимости обучения новым навыкам.

## Требования (агрегированы)

*   Реализовать систему контроля БФТ-требований с трекером продолжения работы (resume-контекст), обеспечивающим полный контекст для старта следующей сессии. Ввести статусную модель задачи из 5 состояний (`To Do → In Progress → Wait for Review → Accepted → Done`) и обеспечить присоединение к существующему пайплайну через §-протокол «Синхронизация с доской» без создания новых команд.
