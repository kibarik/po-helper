### Синтез проблемы и механизма решения

Требуется вывести функциональность движка контент-синтеза на уровень бизнес-ориентированных артефактов. На текущий момент отсутствуют автоматизированные потоки для создания стратегических презентаций на основе данных из Vault, а также инструменты для качественного сопровождения релизов сложных эпиков. Существующие методы коммуникации выкаток ограничиваются статичными списками задач (Jira/GitLab), что не подходит для бизнеса и смежных команд.

Механизм решения подразумевает внедрение трех ключевых компонентов:
1. **Стратегический слой:** Генерация презентаций стратегии платформы через `po-helper` с использованием ресурсов движка `okr-planner/resources/engine` (render_html/pptx) на основе брендбука и актуального контекста из Vault, обеспечивающая выравнивание знаний.
2. **Релизная документация:** Навык `/release-card` для автоматического создания структурированных планов выкатки эпиков (Backend+Frontend, >2 модулей) с включением блоков миграций, проверок и отката.
3. **Коммуникационный слой:** ИИ-нарратив, который в момент выкатки синтезирует технические изменения (с помощью `repowise`) в человекочитаемую историю о поведении системы и доставляет её в рабочий чат.

### Требования (агрегированы)

*   Реализовать генерацию готового презентационного артефакта стратегии платформы из vault (брендбук + актуальный контекст) с использованием `po-helper` и движка рендеринга, где артефакт служит источником выравнивания контекста «сотрудник ↔ хранилище» и позволяет классифицировать правки PO как vision-дрейф или detail-дрейф. — from #101
*   Разработать навык `/release-card` для эпиков (Backend+Frontend / >2 модулей) с автодетектом триггеров по JIRA компонентам и суб-навыком `release-guard`, который генерирует тело карточки из 4 дженерик-блоков: план выкатки, DDL-миграция, проверки (pre/dry-run/post) и Rollback по FK. — from #53
*   Внедрить ИИ-нарратив релиза, который автоматически формируется при выкатке и отправляется в рабочий чатик, представляя собой связную историю «что уехало и как меняется поведение системы» (синтезированную через `repowise: get_why/get_context`), а не просто статичный маппинг JIRA/GitLab. — from #143
