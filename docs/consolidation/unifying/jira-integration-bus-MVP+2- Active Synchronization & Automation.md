### Синтез проблемы и механизма решения

Текущий уровень интеграции страдает от разрыва данных: источник истины расщеплён между po-helper и JIRA, а отчётность в JIRA не отражает фактическую поставку (релизы/MR выкатываются, но статус показывает 15% готовности). Кроме того, управление крупными структурами (эпиками) затруднено отсутствием инструмента для массовых операций над деревьями задач.

Необходимо реализовать механизм **Active Synchronization & Automation** (MVP+2), который обеспечит:
1. **Двустороннюю синхронизацию планов**: через навык `jira-sync` (DIFF → RECONCILE → BUILD) для сверки po-helper и JIRA Gantt, устраняя противоречия в планах.
2. **Мост реальности**: использование релизного журнала и MR-потока как источника правды для спринт-отчётности с подсветкой рассинхронов с JIRA.
3. **Автоматизацию массовых операций**: скриптовый детерминированный обход деревьев эпиков для рекурсивного обновления статусов, замещающий неэффективный последовательный проход LLM.

### Требования (агрегированы)

*   Реализовать навык-синхронизатор плана `jira-sync` (команды `/gsync-*`) с тремя фазами: DIFF (показ дельты), RECONCILE (примирение версий) и BUILD (построение GanttView), чтобы обеспечить полную синхронизацию контекста po-helper ↔ JIRA и решить проблему, где «источник истины расщеплён». — from #58
*   Разработать инструмент массовых операций (скрипт) для рекурсивного обхода и обновления деревьев эпиков JIRA, поскольку текущие решения «не умеют работать с деревом эпика» и не закрывают подзадачи; перейти на детерминированный скрипт вместо LLM для bulk-переходов. — from #62
*   Создать мост «релизный журнал / MR ↔ JIRA», чтобы po-helper отслеживал фактический сигнал поставки (GitLab MR, GitHub PR, CHANGELOG, теги релизов, деплой-лог) как независимый источник правды и сверял его с JIRA-статусами для выявления ситуаций, когда «JIRA показывает 15% готовности, но фактически команда день за днём выкатывала релизы». — from #123
