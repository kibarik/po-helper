# Консолидация бэклога

## Кластеры

### cluster-128 — Введение новой сущности «ожидание/обязательство» (точка контроля) в NEXUS/GROUND с привязкой к стейкхолдерам, целям (OKR) и каналам связи, где напоминание трактуется как коммитмент на проверку статуса. (target: Обеспечение непрерывного контроля исполнения ожиданий и обязательств с пониманием последствий их пропуска (цена риска) и интеграцией в ежедневный отчёт.)
- #128 [primary] — Создание сущности для отслеживания обязательств и оценки рисков в NEXUS/GROUND.

### cluster-127 — Двухфазная система: мгновенный захват (скрин/текст) в черновик (Backlog.md) без интервью + асинхронный роутинг карточек в JIRA-эпик по семантическому образу результата. (target: Быстрый (zero-friction) инбокс задач с отложенным точным размещением.)
- #127 [primary] — Реализация механизма мгновенного захвата и асинхронного семантического роутинга в JIRA.

### cluster-125 — Автоматическое создание GitHub Issue в качестве контейнера и использование оркестратора `gh-issue-agents` с GitHub App для записи результатов синхронного (`/bft-fast`) и фонового (`bft-writer`) агентов в комментарии. (target: Единый GitHub Issue, содержащий быстрый черновик и полную deep-проработку БФТ от первой мысли до канон-документа.)
- #125 [primary] — Оркестрация агентов и агрегация их вывода в едином GitHub Issue.

### cluster-123 — Сверка статусов JIRA с потоком событий поставки (журнал релизов/MR) как независимым источником правды (ground truth) для выявления рассинхрона. (target: Отчётность спринта на основе фактической доставки кода и автоматическое выявление расхождений между планом в JIRA и фактом релизов.)
- #123 [primary] — Механизм верификации статусов JIRA через поток событий поставки (MR/релизы).

### cluster-121 — Внедрение навыка (пайплайна) на основе теории принятия решений, который выполняет фрейминг развилки, генерирует древо вариантов из памяти, проводит сравнительный анализ, взвешенную оценку по критериям и SWOT-анализ. (target: Научить po-helper взвешивать решения и помогать PO выбирать стратегию и тактику, превращая память в структурированную опору для принятия решений.)
- #121 [primary] — Пайплайн принятия решений (фрейминг, генерация вариантов, анализ) для po-helper.

### cluster-120 — Подключение через внешние клиенты (мессенджеры, почта, нативные приложения) (target: Чтение информации и обсуждение с ИИ без использования VPN)
- #120 [primary] — Интеграция с внешними клиентами для доступа к функциям ИИ без VPN.
