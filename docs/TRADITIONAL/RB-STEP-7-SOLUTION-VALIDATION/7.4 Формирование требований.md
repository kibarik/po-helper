# 7.4 Формирование требований — Полная инструкция

## 🎯 Цель этой инструкции

Создать **спецификацию требований к продукту** — формализованный документ, описывающий, какими функциями и свойствами должен обладать продукт (или его части) после разработки. Это главный артефакт, который передаётся команде разработки и тестирования как «контракт» на реализацию.

**Время на выполнение:** 6–10 часов (для новичка: 10–16 часов)
**Уровень:** Средний (нужно понимание того, что было выявлено на этапах валидации)
**После выполнения:** У тебя будет пакет требований v0.1 — Use Cases / User Stories, нефункциональные требования, критерии приёмки и Definition of Done.

---

## 📋 Что такое Формирование требований?

### Простыми словами

**Требования** — это детальное описание того, **что** должен уметь продукт и **как** он должен себя вести. Это не то, как его сделать технически (это задача разработчиков), а именно то, что именно должно произойти с точки зрения пользователя и системы.

Ключевой вопрос, на который отвечает этот шаг:
> «Какими функциями и свойствами должен обладать продукт или его части после процесса разработки?»

### Аналогия

Представь, что ты нанял строительную бригаду переделать кухню. Если ты просто скажешь «сделайте красиво» — результат непредсказуем: бригада понимает «красиво» по-своему, ты по-своему. Но если ты заранее составишь техническое задание: «плитка белая 30×30, столешница из кварца, 4 розетки у фартука, вытяжка над плитой» — у всех одинаковая картинка того, что должно получиться.

**Требования к продукту = техническое задание для кухни.** Они превращают абстрактный замысел («хочу удобное приложение») в конкретный контракт между продуктом и разработкой.

---

## 🎯 Зачем это нужно?

### 3 главные причины

1. **Сокращение времени на разработку и переделку**
   Чем точнее сформулированы требования, тем меньше времени потребуется на проектирование технологического решения и саму разработку. Разница между желаемым и реализацией будет минимальна — не нужно будет переделывать уже готовую работу.

2. **Снижение рисков технического долга**
   Ясные и непротиворечивые функциональные и нефункциональные требования позволяют сформировать масштабируемую и консистентную технологическую систему. Это сокращает риски появления технического долга (Technical Debt) — ситуации, когда продукт работает, но его крайне трудно развивать дальше.

3. **Единый язык команды**
   Требования — это общее понимание между продуктом, дизайном, разработкой и тестированием. Без них каждый участник команды реализует своё представление о продукте, что приводит к конфликтам и несовместимым решениям.

### Что будет плохо без этого шага?

- Разработчики начнут угадывать, что имелось в виду, и реализуют не то
- В готовом продукте обнаружатся «дыры» — забытые сценарии поведения
- Тестирование не сможет проверить продукт объективно — непонятно, что считать успехом
- Каждое изменение требований после начала разработки стоит в 5–10 раз дороже, чем до неё

## 📊 Структура спецификации требований

Пакет требований состоит из 5 типов артефактов:

| Артефакт | Что описывает | Для кого |
|---|---|---|
| User Stories / Use Cases | Функциональные сценарии взаимодействия | Разработка, дизайн |
| Нефункциональные требования (NFR) | Качественные свойства системы | Разработка, архитектура |
| Screen Map | Карта экранов и переходов между ними | Дизайн, разработка |
| Test Cases | Сценарии тестирования | QA-тестирование |
| Definition of Done (DoD) | Критерии приёмки | Вся команда |

## 📝 Phase 1: Функциональные требования — User Stories

### Что такое функциональные требования?

Функциональные требования описывают, **что** система должна делать: какие действия доступны пользователю, как реагирует система, что происходит при различных сценариях. Они описывают поведение продукта.

### Step 1.1: Составь список всех пользовательских сценариев

Перед написанием требований нужно составить карту всех сценариев (user flows), через которые проходит пользователь при взаимодействии с продуктом.

**Действие:** Возьми прототип или Screen Flow из предыдущего шага и выпиши все «точки взаимодействия» — моменты, когда пользователь что-то делает, а система что-то отвечает.

**Шаблон карты сценариев:**

```markdown
## Карта пользовательских сценариев

### Основные потоки (Happy Path)
1. [Сценарий 1: Регистрация нового пользователя]
2. [Сценарий 2: Подбор репетитора по предмету]
3. [Сценарий 3: Запись на урок]
4. [Сценарий 4: Проведение урока]
5. [Сценарий 5: Оставление отзыва]

### Альтернативные потоки (Edge Cases)
- Пользователь ввёл неверный email при регистрации
- Репетитор недоступен в выбранное время
- Платёж отклонён банком
```

**Пример (сервис подбора репетиторов):**

```markdown
## Карта сценариев: AI-сервис подбора репетиторов

### Основные потоки
1. Регистрация ученика / родителя
2. Описание потребности (предмет, цель, бюджет)
3. Просмотр AI-подборки репетиторов
4. Просмотр профиля репетитора
5. Запись на пробный урок
6. Оплата урока
7. Проведение урока (видеозвонок)
8. Оценка урока и репетитора

### Альтернативные потоки
- Ученик не нашёл подходящего репетитора → запрос на ручной подбор
- Репетитор отменил урок → автоматическое предложение альтернативы
- Ученик опоздал → уведомление репетитору
```

### Step 1.2: Напиши User Stories

**User Story (История пользователя)** — формат описания требований, в котором основной акцент стоит на получаемой пользователем ценности. Простые в написании и понятные заказчикам, но требуют критериев приёмки для однозначности.

**Канонический формат User Story:**

```
Как [тип пользователя],
Я хочу [действие / функция],
Чтобы [ценность / результат].
```

**Важно:** User Story должна обладать балансом размерности:
- Не слишком большая — можно реализовать за 1 итерацию (спринт, обычно 1–2 недели)
- Не слишком маленькая — приносит реальную ценность бизнесу, а не является мелкой технической задачей

**Шаблон User Story:**

```markdown
## US-[номер]: [Краткое название]

**Как** [тип пользователя],
**Я хочу** [действие],
**Чтобы** [ценность].

**Приоритет:** [Must Have / Should Have / Could Have / Won't Have]
**Story Points:** [оценка сложности: 1, 2, 3, 5, 8, 13]
**Зависимости:** [US-XX, US-YY]

### Критерии приёмки (Acceptance Criteria)
- [ ] [Критерий 1]
- [ ] [Критерий 2]
- [ ] [Критерий 3]

### Примечания
[Дополнительный контекст, уточнения, ограничения]
```

**Пример заполненной User Story:**

```markdown
## US-007: AI-подборка репетиторов по запросу

**Как** родитель ребёнка школьного возраста,
**Я хочу** описать потребность в свободной форме («нужна помощь с алгеброй 9 класс, экзамен через 2 месяца, бюджет до 1500 рублей в час»),
**Чтобы** получить список релевантных репетиторов без ручного поиска и фильтрации.

**Приоритет:** Must Have
**Story Points:** 8
**Зависимости:** US-001 (авторизация), US-003 (профили репетиторов)

### Критерии приёмки
- [ ] Поле ввода принимает текст до 500 символов
- [ ] AI возвращает от 3 до 10 репетиторов в течение 5 секунд
- [ ] Каждый репетитор в выдаче имеет процент совпадения с запросом (Match Score)
- [ ] Пользователь может запросить новую подборку без перезагрузки страницы
- [ ] Если AI не нашёл подходящих — показывается сообщение с предложением изменить запрос

### Примечания
Для MVP: AI — это простая система тегов + фильтрация. Полноценный NLP-алгоритм — в версии 2.0.
```

### Step 1.3: Напиши Use Cases для сложных сценариев

**Use Case (Прецедент использования)** — формат описания функциональных требований в виде пошагового сценария взаимодействия пользователей с системой по пути к цели. Эффективен при большом количестве шагов или нескольких участниках взаимодействия.

**Структура Use Case:**
- Имя сценария
- Цель
- Участники (Actors)
- Предусловия (что должно быть верным перед стартом)
- Триггер (что запускает сценарий)
- Основной поток (шаги)
- Альтернативные потоки (что если...)
- Постусловия (результат)

**Шаблон Use Case:**

```markdown
## UC-[номер]: [Название сценария]

**Цель:** [Чего хочет достичь пользователь]
**Участники:** [Кто взаимодействует — Пользователь, Система, Оператор...]
**Предусловия:** [Что должно быть выполнено перед началом]
**Триггер:** [Что запускает сценарий]

### Основной поток
1. [Действие участника]
2. [Ответ системы]
3. [Действие участника]
4. [Ответ системы]
...

### Альтернативный поток А: [Название]
3а. [Что происходит иначе]
    3а.1 [Ответ системы]

### Альтернативный поток Б: [Название]
...

**Постусловия:** [Что изменилось в системе после успешного завершения]
```

**Пример заполненного Use Case:**

```markdown
## UC-003: Запись на пробный урок

**Цель:** Ученик записывается на первый урок с выбранным репетитором
**Участники:** Ученик (родитель), Система, Репетитор
**Предусловия:** Ученик авторизован; репетитор имеет доступные слоты
**Триггер:** Ученик нажал «Записаться» в профиле репетитора

### Основной поток
1. Система отображает календарь доступных слотов репетитора
2. Ученик выбирает дату и время урока
3. Система запрашивает подтверждение и показывает итоговую стоимость
4. Ученик подтверждает и переходит к оплате
5. Система проводит оплату
6. Система создаёт запись в расписании ученика и репетитора
7. Оба участника получают уведомление (email + push) с деталями урока

### Альтернативный поток А: Выбранный слот занят
2а. Пока ученик выбирал, другой пользователь забронировал слот
    2а.1 Система отображает сообщение «Это время уже занято»
    2а.2 Система обновляет календарь и предлагает ближайшие свободные слоты

### Альтернативный поток Б: Оплата отклонена
5б. Банк отклонил транзакцию
    5б.1 Система отображает ошибку и предлагает сменить способ оплаты
    5б.2 Слот остаётся зарезервированным на 10 минут

**Постусловия:** 
- Создана запись в расписании ученика и репетитора
- Проведена оплата пробного урока
- Оба участника уведомлены
```

## 📝 Phase 2: Нефункциональные требования (NFR)

### Что такое нефункциональные требования?

Нефункциональные требования (Non-Functional Requirements, NFR) описывают **качественные свойства** системы, не связанные с конкретными пользовательскими функциями. Это «как» система должна себя вести, а не «что» она делает.

**Аналогия:** Если функциональные требования — это список блюд в меню ресторана, то нефункциональные — это требования к скорости обслуживания, чистоте посуды, температуре зала и уровню шума. Блюда могут быть отличными, но если еда приходит через 2 часа — ресторан провалится.

**Основные категории NFR:**

| Категория | Что означает | Пример требования |
|---|---|---|
| Производительность (Performance) | Скорость отклика системы | Страница загружается < 2 секунды |
| Масштабируемость (Scalability) | Способность выдержать рост нагрузки | Система выдерживает 10 000 одновременных пользователей |
| Надёжность (Reliability) | Работоспособность без сбоев | Uptime ≥ 99.5% в месяц |
| Безопасность (Security) | Защита данных и доступа | Пароли хранятся в хэшированном виде (bcrypt) |
| Доступность (Accessibility) | Использование людьми с ограничениями | Поддержка screen readers; соответствие WCAG 2.1 уровень AA |
| Совместимость (Compatibility) | Работа на разных устройствах/браузерах | Поддержка Chrome, Safari, Firefox; iOS 15+, Android 10+ |
| Портируемость (Portability) | Возможность переноса между средами | Развёртывание в Docker-контейнерах |

### Step 2.1: Задокументируй NFR

**Шаблон нефункциональных требований:**

```markdown
# Нефункциональные требования

## 1. Производительность
- Время загрузки главной страницы: ≤ 2 сек (при стабильном 4G)
- Время ответа API-запросов: ≤ 500 мс для 95% запросов
- AI-подборка репетиторов: ≤ 5 сек

## 2. Масштабируемость
- MVP: поддержка до 1 000 одновременных пользователей
- Архитектура должна позволять горизонтальное масштабирование без рефакторинга

## 3. Надёжность
- SLA: Uptime ≥ 99.5% (≤ 3.6 часа простоя в месяц)
- Критические данные (профили, расписание, платежи) — резервное копирование 1 раз в сутки

## 4. Безопасность
- Авторизация: JWT-токены, срок жизни 24 часа + refresh token
- Хранение паролей: bcrypt, salt rounds ≥ 10
- Данные платёжных карт: не хранятся, передаются напрямую платёжному шлюзу (Stripe/ЮКасса)
- HTTPS обязателен для всех соединений

## 5. Доступность
- Поддержка Chrome 110+, Safari 16+, Firefox 110+, Edge 110+
- Адаптивная вёрстка: мобильные устройства от 320px, десктоп от 1024px
- iOS 15+, Android 10+

## 6. Совместимость
- Часовые пояса: все слоты расписания хранятся в UTC, отображаются в локальном tz пользователя
- Языки интерфейса MVP: русский
```

## 📝 Phase 3: Screen Map — карта экранов

### Что такое Screen Map?

**Screen Map (карта экранов)** — визуальное отображение реализации user flow продукта при помощи конкретных интерфейсных решений. Включает перечень состояний каждого элемента интерфейса и переходы между ними, инициированные пользователем или системой.

Используется как для постановки требований в разработку, так и в качестве документации.

### Step 3.1: Составь Screen Map

**Действия:**
1. Открой Figma / Miro / FigJam
2. Перенеси все экраны прототипа
3. Свяжи их стрелками-переходами с указанием триггера (действие пользователя или системы)
4. Для каждого экрана задокументируй список состояний

**Шаблон описания экрана:**

```markdown
## Экран: [Название]

**URL / Route:** [/search или /profile/:id]
**Триггер открытия:** [Действие, которое ведёт на этот экран]

### Состояния экрана
| Состояние | Условие | Что показывается |
|---|---|---|
| Загрузка | Данные запрашиваются | Skeleton / Spinner |
| Успех | Данные загружены | Список репетиторов |
| Пусто | Результатов не найдено | Иллюстрация + текст + CTA «Изменить запрос» |
| Ошибка | Сетевая ошибка | Toast-уведомление + кнопка «Повторить» |

### Элементы интерфейса
- [ ] Поле поиска (placeholder: «Опишите, что нужно...»)
- [ ] Кнопка «Найти» (активна при ≥ 10 символах в поле)
- [ ] Карточка репетитора (фото, имя, предметы, рейтинг, цена, Match Score)
- [ ] Кнопка «Записаться» на карточке

### Переходы
| Действие | Куда ведёт |
|---|---|
| Клик на карточку репетитора | → Экран профиля репетитора |
| Клик «Записаться» | → Экран выбора времени |
| Клик на логотип | → Главный экран |
```

## 📝 Phase 4: Test Cases и Definition of Done

### Step 4.1: Напиши Test Cases

**Test Case (Тест-кейс)** — пошаговый сценарий взаимодействия пользователя с системой, предназначенный для проверки корректности поведения продукта.

**Структура Test Case:**
- Наименование
- Предусловия среды
- Значения переменных
- Последовательность шагов
- Ожидаемый результат
- Действительный результат (заполняет тестировщик)
- Вердикт (Pass / Fail)

**Шаблон Test Case:**

```markdown
## TC-[номер]: [Название]

**Связанная User Story:** US-[номер]
**Предусловия:** [Что должно быть готово перед тестом]
**Тестовые данные:** [Конкретные значения, которые используются]

### Шаги
| № | Действие | Ожидаемый результат | Факт. результат | Вердикт |
|---|---|---|---|---|
| 1 | [Действие] | [Что должно произойти] | | |
| 2 | [Действие] | [Что должно произойти] | | |
| 3 | [Действие] | [Что должно произойти] | | |

**Итоговый вердикт:** [ ] Pass  [ ] Fail
**Примечания:** [Дефекты, скриншоты, ссылки]
```

**Пример заполненного Test Case:**

```markdown
## TC-007: AI-подборка репетиторов — позитивный сценарий

**Связанная User Story:** US-007
```

### Step 4.2: Составь Definition of Done

**Definition of Done (DoD)** — список условий, при выполнении которых User Story, спринт или релиз считаются завершёнными. Делает процесс приёмки прозрачным и измеримым для всех участников.

**Важно:** DoD может быть разным для разных уровней:
- **User Story DoD** — когда считается готовой одна история
- **Sprint DoD** — когда спринт закрыт
- **Release DoD** — когда версия готова к выпуску

**Шаблон Definition of Done:**

```markdown
# Definition of Done

## User Story DoD
- [ ] Код написан и прошёл code review (≥ 1 аппрув от senior)
- [ ] Написаны и проходят unit-тесты (coverage ≥ 80% по новому коду)
- [ ] Пройдены все Test Cases для данной US (все Pass)
- [ ] Нет открытых критических/блокирующих багов
- [ ] Дизайн соответствует макету (pixel-perfect или допустимые отклонения согласованы)
- [ ] Функционал задеплоен на staging-окружение
- [ ] Обновлена документация (если требуется)
- [ ] PM принял работу на staging

## Sprint DoD
- [ ] Все US в спринте соответствуют User Story DoD
- [ ] Регрессионное тестирование критического пути — Pass
- [ ] Метрики производительности на staging в норме (NFR)
- [ ] Демо проведено и принято стейкхолдерами

## Release DoD
- [ ] Все US релиза соответствуют Sprint DoD
- [ ] Полное регрессионное тестирование — Pass
- [ ] Аналитика (трекинг событий) настроена и проверена
- [ ] Маркетинговые материалы подготовлены (если применимо)
- [ ] Runbook для релиза написан
- [ ] Rollback-план подготовлен
```

## 📝 Phase 5: Приоритизация требований по MoSCoW

### Что такое MoSCoW?

Это метод приоритизации требований, который делит их на 4 категории. Название — аббревиатура от первых букв категорий:

| Категория | Русское название | Что означает |
|---|---|---|
| **M**ust Have | Обязательно | Без этого MVP нельзя выпускать |
| **S**hould Have | Желательно | Важно, но не критично для запуска |
| **C**ould Have | По возможности | Хорошо иметь, но только если есть ресурсы |
| **W**on't Have | Не сейчас | Откладываем на будущее |

### Step 5.1: Расставь приоритеты

**Правило:** Must Have — не более 60% от общего объёма требований. Если больше — ты не сокращаешь MVP, а перекладываешь в него весь продукт.

**Шаблон приоритизации:**

```markdown
# Приоритизация требований (MoSCoW)

## Must Have (обязательно для MVP)
- US-001: Регистрация и авторизация пользователя
- US-003: Профиль репетитора
- US-007: AI-подборка репетиторов
- US-009: Запись на урок
- US-010: Онлайн-оплата
- US-012: Видеозвонок

## Should Have (добавим сразу после MVP)
- US-015: Система отзывов и рейтингов
- US-016: История уроков
- US-018: Уведомления (email + push)

## Could Have (если хватит ресурсов)
- US-020: Реферальная программа
- US-021: Промокоды

## Won't Have (следующая версия)
- US-030: Мобильные приложения iOS/Android
- US-031: Групповые уроки
```

## ✅ Success Criteria

После выполнения этой инструкции ты должен иметь:

✅ **Карта сценариев** — все основные и альтернативные user flows задокументированы
✅ **User Stories** написаны для всех Must Have требований с критериями приёмки
✅ **Use Cases** оформлены для сложных многошаговых сценариев (≥ 5 шагов или ≥ 2 участников)
✅ **Нефункциональные требования** описаны по всем ключевым категориям (производительность, безопасность, совместимость)
✅ **Screen Map** создана в Figma/Miro и связана с User Stories
✅ **Test Cases** написаны для всех Must Have US
✅ **Definition of Done** согласован со всей командой
✅ **MoSCoW-приоритизация** выполнена: Must Have ≤ 60% от всех требований

**Context Ripeness (зрелость контекста):** считается достигнутой при ≥ 70% выполнения чеклиста выше.

## 🚨 Troubleshooting

### Problem 1: «Я не знаю, как детально описать требования — это же очевидно»

**Solution:**
Ничто не «очевидно» для разработчика, который не видел обсуждений и интервью. Правило: если ты можешь представить два разных способа реализации — нужно уточнение в требованиях. Спроси себя: «Что именно произойдёт, если пользователь сделает X?» — и запиши ответ.

### Problem 2: «У нас слишком много Must Have, не знаем что убрать»

**Solution:**
Применяй тест «Что произойдёт, если этого не будет на запуске?». Если ответ: «Продукт нельзя будет использовать по основному сценарию» — Must Have. Если «Будет менее удобно, но использовать можно» — Should Have. Ограничь Must Have одним core user flow.

### Problem 3: «Разработчики говорят, что требования противоречат друг другу»

**Solution:**
Это классический признак того, что нефункциональные требования конфликтуют с функциональными (например, «должно работать оффлайн» + «данные всегда актуальны»). Проведи сессию с командой — выяви конфликт, расставь приоритет и зафиксируй решение в требованиях. Противоречивые требования лучше выявить до начала разработки.

### Problem 4: «Мы постоянно меняем требования в процессе разработки»

**Solution:**
Изменения требований — нормально, но дорого. Введи процесс Change Request: любое изменение требований фиксируется, оценивается по влиянию (время, деньги) и только потом принимается решение. Это дисциплинирует команду и стейкхолдеров.

### Problem 5: «Не понимаю, нужны ли Use Cases или достаточно User Stories»

**Solution:**
User Stories достаточно для большинства фич. Use Cases нужны, когда: (1) в сценарии участвуют несколько типов пользователей или внешние системы; (2) сценарий имеет более 7 шагов; (3) важны альтернативные потоки (исключения, ошибки). Для MVP-старта начни с User Stories, Use Cases добавляй по мере роста сложности.

### Problem 6: «QA говорит, что не понимает, что тестировать»

**Solution:**
Это сигнал отсутствия критериев приёмки у User Stories. Вернись к каждой US без чеклиста Acceptance Criteria и добавь конкретные проверяемые условия. Хороший критерий приёмки: описывает конкретное поведение системы, проверяемое без интерпретации, не содержит слов «быстро», «удобно», «правильно».

## ❓ FAQ

### Q: Чем User Story отличается от Use Case?
**A:** User Story фокусируется на ценности для пользователя («Я хочу... чтобы...»), описывается очень коротко и сопровождается критериями приёмки. Use Case детально описывает последовательность взаимодействий — шаг за шагом. User Stories лучше для гибкой разработки (Agile), Use Cases — для сложных сценариев с многими участниками. На практике используют оба формата.

### Q: Сколько User Stories нужно для MVP?
**A:** Нет универсального ответа. Типичный диапазон для первого MVP — 15–40 User Stories. Ориентируйся на Must Have: это один полный core user flow (от входа до получения основной ценности) + минимально необходимая инфраструктура (авторизация, оплата).

### Q: Нужно ли писать требования для всего или только для MVP?
**A:** Для старта достаточно Must Have + краткий список Should Have/Could Have. Подробно документируй только то, что войдёт в ближайшие 2–3 спринта. Остальное — в виде «резервного беклога» без детализации.

### Q: Кто должен писать требования — PM или аналитик?
**A:** В малых командах — PM с помощью разработчиков и дизайнеров. В крупных — системный аналитик (BA) при участии PM. Главное: требования согласованы со всеми участниками разработки до начала работ.

### Q: Можно ли менять требования после начала разработки?
**A:** Можно, но каждое изменение имеет цену. Введи процесс Change Request: изменение фиксируется, оценивается по влиянию на сроки и стоимость, принимается осознанное решение. Изменения на поздних стадиях (во время тестирования) стоят в 10–100 раз дороже, чем до начала.

### Q: Что если требования оказались неверными и продукт уже разработан?
**A:** Это нормальная ситуация в стартапе — она называется «итерация» или «пивот требований». PAF предполагает, что требования уточняются итеративно по мере получения обратной связи от пользователей. Именно для этого существует MVP — чтобы максимально быстро узнать, что нужно скорректировать.
