# Подшаг 0.4 — Progress Pulse: ритуал и Карта Целей

> Четвёртый, завершающий подшаг бутстрапа. Вводит **ритуал Progress Pulse** (старт каждого Sprint-цикла) и **Карту Целей (Goal Map)** — механизм синхронизации между линзами и подами. Без Pulse шаги 1–8 не «заводятся»: первый Sprint нечем начать.
> Источник: [[operating-model]] столп 5 (цикл Product Sprint), 3PL; [S1] VI.1, VI.4.

---

## 🎯 Цель

Поставить **повторяющийся движок** — так, чтобы каждый шаг 1–8 начинался с фиксации состояния Нексуса и переходил в intent (гэп → цель цикла). На Step 0 вводится:

1. **Шаблон Progress Pulse** — ритуал старта Sprint: snapshot состояния Нексуса, фиксация гэпа, формулировка intent шага [S1] VI.4.
2. **Карту Целей (Goal Map) v0** — связывает 3 Линзы (продукт / бизнес / стратегия) и поды без прямого вмешательства [S1] VI.1.
3. **Переход к Step 1** — под готов запустить первый Product Sprint (фазы SCOUT → BUNCH → PITCH → EXECUTE → HARVEST).

> **Асинхронность [S1] VI.2:** Pulse не привязан к каденциям (PI-планирование). Спринты — «сезоны», срабатывают по событиям; отсюда event-based / Kanban, не Scrum.

---

## 👤 Роль и ответственность

**Ведёт Pulse — Product Engineer** (он же инженер продукта, работающий через Кортекс [S1] III.1). **Фасилитирует и обеспечивает качество ритуала — Product Ops** (процессная роль [S2] III.6).

**Карта Целей** — зона **Portfolio Manager** / синхронизации между подами [S1] III.3, VI.1. В solo-поде Product Engineer совмещает.

> ⚠️ Целеполагание на уровне портфеля/стратегии — **зона человека** [S2] III.4. Кортекс помогает оформить, но Ставки (Bets) и распределение ресурсов не делегирует.

---

## 🧩 Структура: Progress Pulse + Goal Map

### Progress Pulse — ритуал старта Sprint [S1] VI.4

Pulse — фиксация состояния (как пилоты у приборной панели): что мы знаем сейчас, где гэп, куда движемся в этом цикле.

**Шаблон Pulse (заполняется в начале каждого шага 1–8):**

```
PROGRESS PULSE — Step N
1. Snapshot Нексусов:
   - Нексус рынка:        Context Ripeness = __ (полнота × актуальность)
   - Нексус потребителя:  Context Ripeness = __
   - Нексус продукта:     Context Ripeness = __
   - Нексус системы роста:Context Ripeness = __
2. Гэп (gap):            разрыв между текущим и Видением (Vision) [S1]
3. Intent шага:          какую часть гэпа закрываем в этом цикле
4. Confidence Point старт:  текущий CP ключевых гипотез
5. Линза (3PL):          продукта / бизнеса / стратегии [S1] VI.1
```

> Pulse НЕ генерирует решения — он **фиксирует исходную точку**. Решения появляются в фазе SCOUT (скаутинг возможностей/угроз [S1] Принцип 5).

### Карта Целей (Goal Map) — синхронизация [S1] VI.1, III.2

Goal Map связывает три Линзы и поды **без прямого вмешательства** друг в друга [S1] III.2:

| Линза [S1] VI.1 | Объект управления | Спринт | На Step 0 |
|---|---|---|---|
| **Продукта** | продукт | Product Sprint (Нексус продукта → Банч) | цель-черновик: BIG Idea (Step 1) |
| **Бизнеса** | модель роста | Business Sprint (Рычаги/Lever → NPV) | цель-черновик: mNSM (Step 2–5) |
| **Стратегии** | портфель бизнес-юнитов | Strategic Sprint (долгосрочные цели + Нексус рынка → Ставки/Bets) | цель-черновик: Ставки (Step 3) |

Goal Map v0 — **каркас целей**: конкретные значения появятся в шагах 1–3 (BIG Idea, mNSM, Ставки). На Step 0 важна **структура связей**, а не цифры.

---

## 🤖 Кортекс

- **Pulse-агент** (черновой) — по команде инженера собирает snapshot Нексусов (читает Context Ripeness, статус увядания) и заполняет шаблон Pulse. Это **автоматизация** (скорость) из 3 оптимизаций [S2] III.4.
- **Goal Map-агент** (черновой) — визуализирует связи 3 Линз; помечает рассинхрон между подами (аугментация — альтернативная точка зрения [S2] III.4).
- **CP-трекер** — ведёт историю Confidence Point по шагам; сигнализирует, когда CP не растёт несмотря на насыщение (кандидат на осознанный отказ [S1] VI.3).

> Pulse — единственная фаза Sprint, где Кортекс **минимально агентивен**: её суть — честная фиксация человеком «где мы», а не генерация. Метапознание особенно строго (см. ниже).

---

## 🧠 Человек (метапознание)

Pulse — момент **максимальной честности** пода: инженер признаёт реальное состояние контекста, а не желаемое. Анти-pattern — приукрасить Context Ripeness, чтобы «пройти» в следующий шаг. Метапознание [S2] III.4 требует:

- верифицировать каждую оценку Context Ripeness (на каких `источник[]` она основана);
- признавать гэп, даже если он неприятен (Stagility: стабильность цели + гибкость пути [S1] Принцип 3);
- не «подкручивать» CP — если доверие не растёт, это кандидат на **осознанный отказ** [S1] VI.3, [[fit-points]].

> **Логика CP-гейтов [S1] VI.3:** рост контекста = рост скорости процессов, **в т.ч. отказа**. Убить слабую гипотезу быстро — это успех, а не провал.

## 📦 Артефакт

- **Шаблон Progress Pulse** — зафиксирован, готов к заполнению в начале Step 1.
- **Goal Map v0** — структура связей 3 Линз с черновыми целями под шаги 1–3.
- **Pulse-агент + Goal Map-агент + CP-трекер** — скелеты подключены.
- **Правила осознанного отказа** — под принял: CP не растёт → честно убить, не «подкручивать» [S1] VI.3.

## 🛡️ Анти-workslop

- **Не «приоритизация списка задач»** → скаутинг возможностей/угроз в фазе SCOUT (Принцип 5 [S1]); Pulse не генерирует to-do.
- **Не приукрашивать Context Ripeness / CP** → честная фиксация; иначе гейты превратятся в формальность [S1] VI.3.
- **Не привязывать Pulse к каденциям (Scrum-спринты)** → event-based / Kanban [S1] VI.2.
- **Не синхронизировать поды прямым вмешательством** → только через Goal Map [S1] III.2.
- **Не «метрика ради метрики»** → Context Ripeness, CP, mNSM, NPV — единственные измерения [S1].
