# 3.bunch — BUNCH / Граф Нексуса и сегменты (Step 2: Customer)

> Фаза Sprint: **BUNCH** (тяжёлая). Источник: [[operating-model]] (Банч вместо Беклога, Принцип 1; Банч из текущего состояния), [S1] VI.6. Термины — по [[sa_documentation/naming_conventions|naming_conventions]].

---

## 🎯 Цель фазы для Step 2

Превратить разрозненные единицы знания (из EXECUTE) и кандидат-узлы (из SCOUT) в **связный подграф Нексуса потребителя из 8 сущностей** с метрикой CR как вычисляемым gate (2.5), затем **кластеризовать узлы в 2–5 поведенческих сегментов** с workslop-фильтром «персона ≥4 интервью» (2.6). Документация исчезает → цифровой профиль продукта [S2] III.5.

Ключевые инверсии PAF:
- «Единый контекст потребителя» в одном документе → **граф узлов (8 сущностей)**, цифровой двойник [2.5].
- Empathy Map / CJM / ручные персоны → **верифицированный подграф** с поведенческими узлами [2.6].
- CR как самооценка → CR = **вычисляемая метрика** (полнота × актуальность).

> Эта фаза питается данными из EXECUTE (единицы знания о поведении). На ранних циклах Sprint может стартовать на каркасе из SCOUT и дозаполняться по мере поступления интервью — цикл event-based [S1] VI.2.

---

## 👤 Роль

- **Product Engineer** — владелец графа; проверяет связность (каждая проблема привязана к пути и триггеру; каждая мотивация — к цели); задаёт разрез сегментации (ситуация-триггер / уровень боли / поведение / цель / мотивация / ролевой контекст / опыт) [2.6].
- **Product Ops** — поддерживает структуру графа, метрику CR, флаги увядания; обслуживает workslop-фильтр (правило ≥4 интервью) и метапознание-гейт.
- **Growth Engineer** — строит ребро **Value Stream → mNSM** (дельта ценности как потенциал монетизации); добавляет mNSM-профиль каждому сегменту [2.5, 2.6].
- **Зона человека:** решение «достаточно ли спел контекст» для сегментации; выбор фокус-сегмента (переходит в PITCH) — стратегический вызов.

---

## 🤖 Кортекс-агенты

**1. Nexus-Builder Agent (сборка графа):** [[AI-TRANSFORMATION/RB-STEP-2-CUSTOMER-VALIDATION/2.5 instruction]]
- **Вход:** единицы знания о поведении из EXECUTE + карточки интервью + обновлённые CP.
- **Действие:** собирает 8 сущностей в граф (Сегмент → Триггер → Начальное → Целевое → Альтернативные пути → Проблемы+Причины → Мотивация → Value Stream); выявляет ситуации-триггеры, пары «начальное→целевое», проблемы на шагах путей; считает CR; флагует узлы на увядание; строит ребро Value Stream→mNSM.
- **Выход:** спелый подграф Нексуса потребителя с CR как live-метрикой.

**2. Сегментатор (AI Segmenter Agent) с workslop-фильтром:** [[AI-TRANSFORMATION/RB-STEP-2-CUSTOMER-VALIDATION/2.6 instruction]]
- **Вход:** спелый подграф Нексуса потребителя.
- **Действие:** кластеризует узлы по поведенческим разрезам; формирует профили сегментов (название/описание/триггеры/начальное→целевое/мотивация/поведение/ключевые проблемы/ценности/уровень боли); применяет **workslop-фильтр**: узел Сегмент допускается **только при ≥4 подтверждающих интервью** с цитатами-доказательствами (иначе отбрасывается как AI-выдумка); метапознание-чек.
- **Выход:** 2–5 поведенческих сегментов как подграфов Нексуса, каждый с mNSM-профилем и CP.

---

## 🧠 Человек (метапознание)

- **Чекпоинт связности графа:** у каждой проблемы есть путь и триггер? у каждой мотивации — цель? у каждого пути ≥1 проблемы с причиной? ребро Value Stream→mNSM построено? нет «висящих» узлов без источника? Если нет — граф неспел, CR ниже порога, возврат к интервью (EXECUTE) [2.5].
- **«Контекст важнее демографии»:** узлы описывают поведение/ситуацию, а не возраст/пол [2.5, 2.6].
- **Триангуляция:** после авто-кластеризации Product Engineer возвращается к карточкам интервью и проверяет — соответствует ли каждый реальный респондент сегменту? «Не вписывающиеся» → возможный новый сегмент (доп. интервью) или выброс (исключить с обоснованием) [2.6].
- **Veto-право:** отбраковать сегмент-«средний пользователь» (мешанина паттернов) или AI-персону без 4 интервью.

---

## 📦 Артефакт = узлы Нексуса потребителя (+CR)

- Полный граф из 8 сущностей. Метаданные каждого узла: `источник-интервью[]`, `CP`, `timestamp`, `статус увядания`, `частота`.
- Ребра: триггер→состояние, путь→проблема, проблема→причина, мотивация→цель, **Value Stream→mNSM**.
- Узлы Сегмент (2–5), каждый — подграф: триггер + путь + проблема + мотивация + ценности + уровень боли + `разрез сегментации`, `список подтверждающих интервью (≥4)`, `готовность платить (mNSM)`, `демография (вспомогательная, в конце)`.
- **Цитата-доказательство** = отдельный узел типа `quote` с ребром `evidence` (без привязки → workslop) [2.5].
- **Context Ripeness = полнота × актуальность** — вычисляется (не самооценка).

---

## 🚦 CP-гейт

- **Критерий выхода из BUNCH (сборка графа, 2.5):** CR Нексуса потребителя ≥ порога — все 8 сущностей представлены, узлы верифицированы, CP фокус-проблем ≥ gate.
- **Критерий выхода из BUNCH (сегментация, 2.6):** каждый сегмент ≥4 интервью; разрезы сегментации осмыслены; нет AI-персон без доказательной базы.
- **Стоимость риска:** «Сколько стоит проектирование продукта на неспелом контексте или фокус на выдуманном сегменте (продукт не для кого)?» [2.5, 2.6].

---

## 🛡️ Анти-workslop (критический фильтр стадии)

- **Правило «персона ≥4 интервью»** — главный workslop-фильтр Step 2 [reformation 2.9]. Сегмент из 1–3 интервью → кандидат на доработку (не персона). Сегмент из «интуиции Кортекса» без интервью → отбрасывается, CP=0.
- Запрет на «портрет среднего пользователя» (мешанина сегментов) → только поведенческие узлы конкретных контекстов.
- Цитаты-доказательства обязательны для каждого проблемного узла.
- CR не растёт от «дописывания текста» — только от добавления верифицированных узлов.
- Трассировка сегмента до конкретных интервью (список в метаданных).

---

## ❓ Контрольные вопросы (метапознание)

1. У каждой проблемы есть путь и триггер? у каждой мотивации — цель? у каждого пути ≥1 проблемы с причиной? (связность графа)
2. Построено ли ребро Value Stream→mNSM (Growth Engineer)?
3. Каждый сегмент подтверждён ≥4 интервью с цитатами? (правило workslop-фильтра)
4. Разрез сегментации — поведенческий (триггер/боль/поведение), а не демографический?
5. Демография — в конце профиля, не в начале? нет сегмента-«среднего пользователя»?
6. Каждый ли реальный респондент отнесён к сегменту? 2–5 чётких профилей (а не 10 размытых)?
7. Есть ли mNSM-профиль у каждого сегмента? CR ≥ порога для перехода к PITCH?

---

## ⚠️ Типичные ошибки фазы

- **Граф с «висящими» узлами** (проблема без пути) → CR ниже порога, возврат к EXECUTE.
- **AI-персона из 1–3 интервью** → главный workslop Step 2; фильтр ≥4 обязателен.
- **Демография как ядро профиля** → контекст первичен; демография в конце.
- **Value Stream без ребра mNSM** → теряется связь с Business Lens.
- **«Портрет среднего пользователя»** → мешанина сегментов; разделить или исключить выбросы.
- **Меньше 8 сущностей** → теряется причинность цикла «почему человек действует»; полнота CR падает.
