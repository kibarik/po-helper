# 5.execute — EXECUTE (Step 7: Solution & PMF) 🔴

> Фаза ТЯЖЁЛАЯ · Линза: Product · **Цифровой профиль продукта (7.4) + вайбкодинг MVP (7.5): Build + Bale + Germination.**
> Источник: [[AI-TRANSFORMATION/RB-STEP-7-SOLUTION-VALIDATION/7.4 instruction]] + [[AI-TRANSFORMATION/RB-STEP-7-SOLUTION-VALIDATION/7.5 instruction]] · [S1] VI.4–VI.5.

## 🎯 Цель фазы для Step 7

**Execute-фаза Und-Id-Ex** и **Build-фаза цикла Product Sprint** [S1] VI.4. Две под-фазы:

**(7.4) Цифровой профиль продукта (Нексус) вместо пакета требований.** Классический «пакет требований v0.1» (User Stories / Use Cases / NFR / MoSCoW / DoD / Test Cases в Notion/Confluence) **прекращает существовать как финальный артефакт** [S2] III.5 — документация исчезает, её заменяет **цифровой профиль продукта (Нексус)**: живой граф узлов, с которым работают агенты, аналог `claude.md` для продукта/процессов. Методы сохраняются (User Stories, Use Cases, NFR, Screen Map, Test Cases, DoD) — но становятся типами узлов.

**(7.5) Вайбкодинг MVP как стратегия разработки.** MVP — **НЕ «продукт с минимумом фич»**. Это концепция разработки на ранних этапах жизни компании (теория Шарпа: диверсифицированный портфель = лучший ROI при меньшем риске; Робинсон применил к продукту — параллельная/синхронная разработка и верификация; Бланк/Дорф: MVP — способ разработки до момента, пока масштабируемая модель не найдена). Каденции (PI-планирование) заменяются **асинхронным Kanban / event-based** [S1] VI.2.

Цель — построить инкремент через Кортекс (вайбкодинг), поднять CP инкремента через **параллельную верификацию** и подготовить к аналитике/PMF-детекции (7.6).

## 👤 Роль

**Product Engineer** — проектирует цифровой профиль продукта; стратегическое решение об уровне MVP (баланс под гипотезы; иногда для верификации нужен «автомобиль», не «скейтборд»). **Discovery/Launcher PM** — **вайбкодинг MVP** как ключевая компетенция [S2] III.6: продуктовые знания + генерация кода; минимизация time-to-market/time-to-PMF; full-stack человек-команда. **AI UX Designer** — дизайн-система v0.1 под ИИ-метафоры; Screen Map (состояния загрузка/успех/пусто/ошибка). **Technical/AI Product Manager** — Entity/Sequence/State для AI-свойств; сшивает агентов; COGS токенов. **Product Ops** — асинхронный Kanban, цикл Build→Change Log, модель зрелости Кортекса.

> **Зона человека** [S2] III.4: стратегическое решение об уровне MVP и NPV-моделирование — не агентное; синхронизация команды.

## 🤖 Кортекс-агент (из 7.4 + 7.5)

**Под-фаза 7.4 — цифровой профиль продукта:**
- **Генератор User Stories / Use Cases** [7.4] — User Stories («Как [роль], я хочу [действие], чтобы [ценность]» с критериями приёмки; баланс размерности); Use Cases для сложных сценариев (≥5 шагов или ≥2 участников).
- **NFR-агент** — производительность, масштабируемость, надёжность (uptime/SLA), безопасность, доступность (WCAG), совместимость; для AI-продуктов — **AI-COGS** (cost-per-token × tokens-per-user).
- **Screen Map-билдер** — экраны с состояниями (Загрузка/Skeleton, Успех, Пусто, Ошибка/Toast).
- **Test Case-генератор** — для Must Have; шаги (действие → ожидаемый → фактический → Pass/Fail).
- **DoD-агент** — Definition of Done трёх уровней (User Story/Sprint/Release).
- **Traceability-агент** — User Story → User Flow → Final Solution → Solution Hypothesis → JTBD; без сквозной трассировки CP=0.

**Под-фаза 7.5 — вайбкодинг MVP:**
- **Вайбкодинг-агент / MVP-builder** [7.5] — из узлов генерирует код инкремента; rapid-итерации Build→Change Log; практики: **feature flags**, CI/CD, готовые компоненты (UI-библиотеки, платёжные SDK), согласованный DoD, timeboxing.
- **Entity Model-агент** — сущности (существительные-объекты), атрибуты, связи (ассоциация/генерализация/композиция/агрегация), кратность 1:1/1:N/N:M; ERD.
- **Sequence-агент** — Sequence Diagram для ключевого Use Case (Пользователь → Frontend → Backend API → БД → внешние сервисы → Email).
- **State-агент** — State Diagrams для сущностей с жизненным циклом (состояния + переходы + триггеры).
- **Верификация-агент** — параллельная верификация: после каждого блока инкремента проверка на пользователях по плану; обновляет CP.
- **Product Ops-агент** — асинхронный Kanban, event-триггеры («CP вырос» → следующий шаг); технический долг (15–20% capacity на рефакторинг); MVP Decision Log.

## 🧠 Человек (метапознание — валидация ИИ-генерации кода/бизнес-логики/требований)

- **Валидация ИИ-генерации кода/бизнес-логики** [S2] III.4: Launcher PM обязан понимать сгенерированный код и Entity Model, быть способным объяснить разработке/команде. Entity Model — всегда гипотеза; спорные связи (1:N vs N:M) разрешаются через реальные сценарии. **Planning Poker** для калибровки Story Points (baseline «добавить поле в форму» = 1 SP).
- **Veto на workslop-код**: сгенерированный код, который выглядит работающим, но не проверяет гипотезы MVP — отбраковка; спагетти-логика, технический долг, неотслеживаемые зависимости.
- **Решение об уровне MVP**: 5 вопросов диагностики (core user flow? без чего flow невозможен? что верифицируем в первую очередь? что ускорит верификацию? что блокирует принятие?).
- **Параллельная верификация**: Product Engineer тратит 2–3 часа в неделю на звонки с 2–3 пользователями по staging — снижает риск «не то рынку».
- **Валидация требований**: каждая User Story — баланс размерности, однозначность критериёв приёмки (без «быстро», «удобно»); правило «если можно представить два способа реализации — нужно уточнение».
- **MVP ≠ «скейтборд вместо машины»**: знаменитая картинка — иллюстрация итеративности, но не «всегда самое минимальное». Иногда для верификации нужен «автомобиль».

## 🚦 CP-гейт фазы

**Gate 7.5 → 7.6 (Change Log готов)** [7.5]:
- инкремент собран по Банчу;
- Entity/Sequence/State Diagrams согласованы;
- MVP Decision Log ведётся;
- цикл Build→Change Log работает;
- параллельная верификация идёт по плану (метрики sprint-верификации в норме);
- дизайн-система v0.1 готова;
- асинхронный Kanban работает.
- CP инкремента растёт от результатов параллельной верификации (реальные транзакции, первые действия, UX-фидбек); не поднимается без реальных данных.

> Типичные сроки MVP: команда 3–5 человек за 2–4 месяца; команда 1–2 fullstack за 4–8 месяцев. Признак неправильного MVP: оценка >6 месяцев — значит, в него попало слишком много.

## 🛡️ Анти-workslop

- **Workslop в коде (вайбкодинг ~15,4%)** [S2] III.7 — сгенерированный код выглядит убедительно, но содержит спагетти-логику, технический долг, «убедительные, но неверные» результаты (Harvard × BCG).
- **Workslop в требованиях** — неконкретные критерии приёмки, противоречивые NFR, User Stories без ценности.
- **Метапознание-фильтр**: Launcher PM проходит code review сгенерированного, объясняет Entity Model; Planning Poker; 15–20% capacity на рефакторинг; каждое требование проверяется на однозначность.
- **Трассировка до Нексуса**: каждый `increment` → `user-story` → `solution-hypothesis`; инкремент без связи с гипотезой — workslop.
- **Ложное ощущение готовности**: «инкремент работает» ≠ гипотезы подтверждены; параллельная верификация обязательна; MVP — не «сырой» продукт, а полностью покрывающий core user flow, надёжный и качественный, без вторичного функционала.
- **Эффект асинхронности**: при росте локальных скоростей каденции ломаются → event-based Kanban с событийными триггерами, не Scrum-спринты фиксированной длины [S1] VI.2.
- **Утрата стратегического контроля**: NPV-моделирование и фиксация уровня MVP — зона человека; агенты = инструменты, не акторы [S2] III.4.
