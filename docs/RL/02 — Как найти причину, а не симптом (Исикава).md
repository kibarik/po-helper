# 02 — Как найти причину, а не симптом (диаграмма Исикавы)

> **Источник:** [blog.aleksishmanov.ru/isikawa-diagram](https://blog.aleksishmanov.ru/isikawa-diagram)
> **Дата:** 2026-05-16 · **Раздел:** Продуктовый менеджмент · **Slug:** `isikawa-diagram`

---

## TL;DR
Диаграмма Исикавы (fishbone) — графический метод структурного анализа причин: проблема последовательно раскладывается на категории и подкатегории, пока команда не доходит до **операционально устранимых источников**. Главная сила — убирает иллюзию быстрого ответа: заставляет копать глубже симптома. Создал Каору Исикава (1960-е, Toyota Production System).

---

## Когда нужна

- **Проблема повторяется** несмотря на точечные фиксы → команда лечит следствия.
- **Команда не договаривается о приоритетах** (4 человека = 4 разные «главные причины»).
- Нужно объяснить инвесторам/стейкхолдерам, **почему цифры падают** («мы разбираемся» звучит слабо).
- Продукт готовится к **масштабированию** (важно, чтобы текущие проблемы не масштабировались вместе с ростом).

---

## Фреймворк: 5 шагов

1. **Сформулируйте проблему как точное утверждение.** Не «плохая конверсия», а «конверсия из регистрации в первое действие = 12% при цели 30% — последние 2 недели».
   - *Критерий:* новый участник сессии понимает, что именно анализируется.
2. **Определите категории причин.** Классика — 6М: Man, Machine, Method, Material, Measurement, Mother Nature. Для продуктовых команд адаптация: **люди, процессы, технология, данные, рынок, продукт**. Начать с 4–6 категорий.
3. **Сессия генерации причин** (60–90 мин). По каждой категории — «что могло привести к проблеме?» Записывать без фильтрации. Метод **«5 почему»** — докапываться до конкретного, изменяемого фактора.
4. **Приоритизация** по двум осям: **вероятность влияния** × **управляемость**. Начинать с высокой вероятности и высокой управляемости → первый **RAT** (Riskiest Assumption Test).
   - *Критерий:* 3–5 приоритетных гипотез с конкретным экспериментом для каждой.
5. **Проверяйте данными, не обсуждением.** Для каждой гипотезы — какие данные подтвердят/опровергнут. Результаты возвращаются в диаграмму и уточняют её.
   - *Критерий:* через 1–2 недели есть данные по ≥2–3 гипотезам.

---

## Типы диаграмм

- **Классическая fishbone (6М)** — производственные/операционные проблемы, качество поставки, процессные сбои.
- **Процессная** — категории = шаги процесса/воронки.
- **4P (Product, Price, Place, Promotion)** — анализ провалов выручки/роста.
- **Голос клиента** — категории = роли пути клиента (пользователь, покупатель, поддержка, партнёр).
- **Impact Mapping** — распаковка истинной потребности.

## 🔗 Связь с древом метрик (ключевая)

Это **ближайший аналог** древа метрик в блоге:
- Дерево причин (Исикава) ↔ дерево метрик — обе структуры: **декомпозиция сверху вниз**, каждая ветка заканчивается **изменяемым фактором**.
- Принцип переноса: метрика-симптом («пользователи не понимают ценность») — не лист дерева метрик, а то, что надо декомпозировать дальше (как и в Исикаве).
- При падении метрики верхнего уровня — спуститься по дереву метрик (как по рыбьему скелету) до операционной причины, которую можно изменить действием.
