# 🗺️ BLUEPRINT — методология построения Service Blueprint (гибридная нотация CJM × Blueprint)

> **Исполняемый StepByStep-фреймворк схематизации опыта.** Собирает разрозненный опыт участников услуги (клиент, фронт-сотрудники, бэк-процессы, системы) **в одну диаграмму** и структурирует его в канонический формат Service Blueprint — как на референсе `Blueprint 24h` (парковочный сервис, [SB4]).
> Опирается на гибридную нотацию Андрея Шапиро «CJM × Service Blueprint» ([SB1], книга «Карта процесса-опыта» [SB2]) и канон NN/g ([SB3]).
> Бизнес-процесс **идентичен `bft-writer`**: multi-step конвейер ролей с STOP-паузами для человека, нулевой допуск к галлюцинациям, отдельная adversarial-стадия.

---

## 🎯 Что это и зачем

**Service Blueprint** — это диаграмма, которая одновременно показывает **все слои услуги**: что делает клиент, что он видит (фронт-сцена), что скрыто за кулисами (бэк-сцена) и какие системы всё это держат (поддержка). В отличие от **CJM** (карта только со стороны клиента), Blueprint описывает **систему доставки ценности целиком — прежде всего внутри компании** ([SB3]).

> **CJM отвечает на вопрос «что переживает клиент».
> Service Blueprint отвечает на вопрос «что должна делать организация, чтобы клиент это пережил».**

**Главная особенность метода (и этого навыка) — соединение всего опыта воедино.** Отдельные интервью, CJM, регламенты, системные схемы, замечания стейкхолдеров — это осколки. Ценность Blueprint в том, что он **сшивает вертикаль**: каждый шаг клиента получает опору во фронт-сцене, каждая фронт-сцена — опору в бэк-сцене, каждая бэк-сцена — в поддерживающих системах. Разрыв в вертикали = дефект услуги (см. [[STEPS/4.weave]]).

**Три слоя фреймворка** (зеркало `AI-PROCESSES`):
1. **Хребет (spine):** анатомия диаграммы — 5 дорожек и 3 линии. См. [[anatomy]].
2. **Движок (engine):** конвейер построения — 7 стадий от рамки сервиса до отгрузки. См. [[STEPS/overview]].
3. **Правила игры:** принципы качества — ноль галлюцinаций, `As-Is → Gap → To-Be`, negative flows, altitude. См. [[principles]].

## 🧬 Хребет — анатомия диаграммы (5 дорожек + 3 линии)

Диаграмма — это таблица: **колонки = шаги во времени** (слева направо), **строки = дорожки** (сверху вниз). Дорожки разделены тремя горизонтальными линиями. Референс — `Blueprint 24h` [SB4].

| # | Дорожка (Lane) | Что внутри | Чья зона |
|---|---|---|---|
| 1 | **Physical Evidence** (Физические свидетельства) | Артефакты, которые клиент видит/держит на каждом шаге: талон, чек, парковочное место, штраф | вещи |
| — | ── **Line of Interaction** (линия взаимодействия) ── | | |
| 2 | **Customer Actions** (Действия клиента) | Шаги клиента во времени + эмоции + сценарии (напр. `Drive in → Find empty → Park → Pay fee → Place ticket → Drive out → Find car → Drive home`) | клиент |
| — | ── **Line of Visibility** (линия видимости) ── | | |
| 3 | **Onstage / Frontstage** (Фронт-сцена) | Действия сотрудников и систем **прямого контакта** — то, что клиент видит: паркомат, инспектор, интерфейс | сотрудники/системы контакта |
| — | ── **Line of Internal Interaction** (линия внутреннего взаимодействия) ── | | |
| 4 | **Backstage** (Бэк-сцена) | Невидимые клиенту действия сотрудников: инкассация, поддержка контакта с субподрядчиками, обновление статистики | внутренние сотрудники |
| 5 | **Supporting Processes** (Поддерживающие процессы) | Системы и регламенты-фундамент: расписание, роутинг обработки платежей, отчётность владельцу | системы/регламенты |

> Три линии — это **границы ответственности**. Они делают неявное явным: *что клиент вообще видит* (линия видимости) и *кто с ним контактирует напрямую* (линия внутреннего взаимодействия). Подробно — [[anatomy]].

## 🔁 Движок — конвейер построения (7 стадий, зеркало bft-writer)

Как и `bft-writer`, это **не «одна генерация»**, а pipeline из стадий: каждая — отдельная роль, отдельный артефакт, **STOP-пауза** для ревью человеком.

```
0. FRAME     → рамка сервиса        (скоуп, актор, as-is/to-be, вход/выход)     [СТОП]
      ↓
1. JOURNEY   → клиентский путь      (Customer Actions + Physical Evidence + эмоции) [СТОП]
      ↓
2. ONSTAGE   → фронт-сцена          (что клиент видит; линии Interaction/Visibility)
      ↓
3. BACKSTAGE → бэк-сцена + поддержка (невидимое; линия Internal Interaction)      [СТОП]
      ↓
4. WEAVE     → соединение опыта      (стрелки, зависимости, fail-points, время)   ← ядро метода
      ↓
5. VALIDATE  → валидация            (hard gates + Светофор)
      ↓ 🔴 нарушен гейт → возврат на нужную стадию
      ↓ 🟢/🟡
6. DELIVER   → отгрузка             (рендер PlantUML/картинка + публикация)       [СТОП]
      ↓
   Done
```

| Стадия | Роль | Артефакт |
|---|---|---|
| 0 [[STEPS/0.frame]] | Service Framer | `frame.md` (скоуп, актор, as-is/to-be, entry/exit) |
| 1 [[STEPS/1.journey]] | Journey Analyst | `journey.md` (шаги клиента + эмоции + физ. свидетельства) |
| 2 [[STEPS/2.onstage]] | Frontstage Designer | `onstage.md` (фронт-сцена по каждому шагу) |
| 3 [[STEPS/3.backstage]] | Backstage Analyst | `backstage.md` (бэк-сцена + поддержка) |
| 4 [[STEPS/4.weave]] | Systems Weaver | `blueprint.puml` (связанная диаграмма) |
| 5 [[STEPS/5.validate]] | Validator (свежий взгляд) | `validation.md` (Светофор) |
| 6 [[STEPS/6.deliver]] | Deliverer | рендер + публикация |

**Почему разделение:** разные роли не «загрязняют» друг друга (путь клиента ≠ фронт ≠ бэк); соединение (WEAVE) вынесено отдельной стадией, потому что это самая сложная и ценная операция; STOP-паузы дают human-in-the-loop. Полная карта — [[STEPS/overview]].

## 🧭 CJM или Blueprint — что и когда

| | **CJM** | **Service Blueprint** |
|---|---|---|
| Точка зрения | клиент | организация (+клиент) |
| Что показывает | переживания, эмоции, боли | всю систему доставки: фронт + бэк + системы |
| Глубина | горизонтальная (путь) | вертикальная (слои под каждым шагом) |
| Когда применять | понять клиента, найти боли | спроектировать/починить сервис, найти где именно ломается |
| Роль в этом методе | **вход** для Blueprint (стадия 1) | **выход** (стадии 2–6) |

Гибридная нотация Шапиро [SB1] снимает противопоставление: **CJM — это верхние две дорожки Blueprint**. Мы строим CJM (стадия JOURNEY), а затем «опускаем вертикали» под каждый шаг. Подробно — [[cjm-vs-blueprint]].

## 📖 Как пользоваться

1. **Сначала прочитай [[anatomy]] и [[principles]]** — без анатомии дорожек и правил качества стадии не поймёшь.
2. **Пройди стадии 0 → 6 по порядку.** Стадия 0 (FRAME) обязательна: без зафиксированного скоупа и актора диаграмма расплывётся.
3. **В каждой стадии** читай её файл в `STEPS/` → делай артефакт → STOP → жди ревью человека.
4. **Гейт не пройден** (стадия 5) → возврат на стадию, где дефект, не «замазывание».
5. **Нотация и рендер** → [[notation]] (символы, цвета, эмоции, PlantUML).
