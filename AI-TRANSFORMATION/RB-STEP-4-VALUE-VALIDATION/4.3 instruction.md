# 4.3 instruction — Stage-gate валидация с динамическим CP и оценкой стоимости риска

> Этап **RB-STEP-4 — VALUE VALIDATION** (Stage 3) · Линза: Product (контекст: Business) · Инструкция для **Инженера Продукта нового поколения** (PAF, [S1],[S2]).
> Преемственность метода: исходный раннбук `RB-STEP-4-VALUE-VALIDATION/4.3 Валидация гипотез ценности.md`.

## 🎯 Цель в PAF-оптике
Провести эксперименты по **четырём методам валидации** (smoke / лендинг / fake door / пилот), поднимая **Confidence Point** динамически по мере поступления данных, с явным различением «оффер сработал» ≠ «ценность подтверждена». Цикл повторяется до подтверждения или осознанного pivot; каждый цикл = event-based обновление узлов Нексуса продукта.

## 👤 Роль и ответственность
- **Product Engineer** — выбирает метод валидации под CP-профиль гипотезы (smoke для быстрой проверки интереса к офферу; лендинг для готовности к следующему шагу; fake door для чистого теста спроса у существующих юзеров; пилот для реальной ценности). Фиксирует **pre-experiment statement** ДО запуска (гипотеза, метрика, порог, размер выборки, критерии «подтверждена/не подтверждена»). Принимает решение go / pivot / следующий цикл.
- **Product Ops** — настраивает event-based мониторинг CP: поступление сигнала → пересчёт CP → триггер следующего гейта или alert.
- **Зона человека**: стратегическое решение о pivot — какой компонент бизнес-модели менять (потребность / сегмент / рынок / цена / формат) и на какой этап Discovery возвращаться. Это решение об инвестициях, не делегируется агенту.

## 🤖 Что делает Кортекс-агент
**Агент-валидатор экспериментов** ведёт весь цикл:
1. **Кортекс-генерация лендингов/офферов** — по VP-node и Offer-node из 4.1/4.2 агент генерирует варианты лендинга (Hero/Problem/Solution/Trust/Pricing/FAQ) и 2–3 варианта объявления для A/B формулировок оффера.
2. **Авто-запуск** smoke/таргета/fake door с UTM-метками, цели в аналитике.
3. **Парсинг интервью** — ASR → кодирование → авто-валидация → кодирование цитат в узлы Нексуса.
4. **Динамический CP**: каждый сигнал (CTR, конверсия, цитата, повторное использование) пересчитывает CP в реальном времени по колесу Гилада. CP растёт от данных, снижающих неопределённость; падает при срабатывании рисков.
5. **Отличает «оффер сработал» от «ценность подтверждена»**: CTR/клик = реакция на оффер; повторное использование/оплата = ценность. Агент не даёт поднять CP до уровня «need/value fit» только по CTR.
6. **Оценка стоимости риска**: «сколько ещё потратим ресурсов и негативного эффекта на клиентскую базу при таком уровне рисков» перед масштабированием метода (smoke → лендинг → пилот).
7. **4 сценария интерпретации**: A (подтверждена) / B (не подтверждена) / C (неоднозначно) / D (неожиданно подтверждена) → триггер следующего действия.

### Четыре метода валидации (от простого к глубокому — преемственность из ранбука 4.3)
| Метод | Что проверяет | Метрика | Когда выбирать |
|---|---|---|---|
| **1. Smoke / Реклама / Брошюра** | Реакция на **оффер** (не на ценность!) | CTR (обычно порог 1,5–3%) | Быстро и дёшево проверить интерес; нет продукта и лендинга; A/B формулировок |
| **2. Лендинг / Сбор контактов** | Готовность к **следующему шагу** (email/заявка) | Конверсия в целевое действие (8–12%) | ЦП требует объяснения; можно собрать качественную обратную связь |
| **3. Fake Door / Пустышка** | Спрос у **существующих юзеров** | CTR по элементу (5–15% активных) | Есть продукт с активными пользователями; тест новой функции без разработки |
| **4. Пилот / Образец** | **Реальную ценность** (не только интерес) | Конверсия + повторное использование + качественная реакция | Другие методы дали неоднозначно; ЦП сложно передать без демонстрации |

Ключевой принцип PAF: **цикл валидации повторяется, пока не найдено подтверждение.** Ни одна гипотеза не подтвердилась — это данные, не провал. Далее **pivot** — осознанная смена компонента бизнес-модели с возвратом на соответствующий этап Discovery. Пилот — **не MVP**: цель проверить гипотезу, а не запустить продукт.

### Диагностика уровня проблемы при отрицательном результате
| Уровень | Признак | Что делать | Куда возвращаться |
|---|---|---|---|
| Оффер | Низкий CTR, но в интервью «хотел бы попробовать» | Переформулировать оффер | остаёмся в Stage 4 |
| Канал | Хороший CTR, нулевая конверсия — нецелевая аудитория | Уточнить таргетинг/сменить канал | остаёмся в Stage 4 |
| ЦП (узкий) | Конверсия есть, но уходят после первого контакта | Оффер привлёк, продукт не дотянул | остаёмся в Stage 4 |
| Потребность | В интервью потребность не такая «горящая» | Пересмотреть данные | → Stage 2 |
| Сегмент | Реагирует другая ЦА | Скорректировать сегмент | → Stage 2 |
| Бизнес-модель | ЦП нравится, но цена отпугивает | Проверить ценовые гипотезы | остаёмся в Stage 4 |
| Рынок | Конкуренция слишком высока | Пересмотреть рынок | → Stage 3 |

## 🧠 Что делает человек (метапознание)
- **Документ до запуска**: pre-experiment statement обязателен ДО эксперимента, иначе соблазн интерпретировать любой результат как подтверждение.
- **Различать декларируемое и реальное поведение**: слова «куплю» дешевле денег. Нет оплаты/предзаказа → тест не пройден, даже если все в интервью говорили «да».
- **Диагностировать уровень проблемы** при отрицательном результате: оффер / канал / ЦП-узкий / потребность / сегмент / бизнес-модель — и только потом объявлять pivot.
- **Veto** на premature «go»: один слабый сигнал ≠ подтверждение. Минимум 2 разных источника (количественный + качественный) в одном направлении.
- **Pivot ≠ провал**: осознанная смена компонента бизнес-модели с возвратом на соответствующий этап Discovery.

## 📦 Артефакт = узлы Нексуса продукта
- **Experiment-node (+CP-дельта)** — метод, гипотеза, метрика, порог, выборка, срок, pre-experiment statement, результаты (количественные + качественные), сценарий интерпретации, CP до/после.
- **Insight-node** — новые знания о потребителе/рынке из каждого эксперимента (даже отрицательного).
- **Pivot-node** (если сработал) — что меняется, на какой этап возвращаемся.
- **Context Ripeness** по подшагу: целевое ≥75% (при подтверждённой гипотезе).

## 🚦 Confidence Point-гейт
Stage-gate с **динамическим CP** между методами валидации:
- smoke → лендинг: CP вырос от CTR + стоимость риска приемлема.
- лендинг → пилот: CP вырос от конверсии в целевое действие, оффер и ценность разведены.
- пилот → 4.4 (фиксация): **need/value fit** в зоне видимости — повторное использование/оплата/готовность продолжать при достаточной выборке.

CP не поднимается без реальных данных (симуляция ≠ a posteriori). Осознанный отказ — равноправный исход gate.

## 🔭 Связь с Тремя Линзами (3PL)
**Product Lens**: валидация — execute-фаза Und-Id-Ex для ценностных гипотез, вход в Product Sprint. Связь с **Business Lens**: метрика валидации (конверсия в оплату, повторное использование) — это прокси будущей mNSM и вход в NPV. Связь со **Strategy Lens**: pivot на уровне «рынок» или «потребность» возвращает в Strategy/Customer-линзу.

## 🧭 Модель зрелости Кортекса на этом подшаге
- **Фаза 1 (отдельные задачи):** Кортекс генерирует лендинг/оффер по шаблону, инженер запускает вручную и интерпретирует.
- **Фаза 2 (процессы команд):** агент-валидатор автозапускает тесты, ведёт лог, парсит интервью (ASR→кодирование→узлы), пересчитывает CP событийно по колесу Гилада.
- **Фаза 3 (процессы компании):** непрерывный мониторинг CP; масштабирование метода (smoke→лендинг→пилот) триггерится автоматически при стабильном CP; агент явно разделяет «оффер сработал» и «ценность подтверждена» без человеческой проверки каждого случая.

## ❓ Типовые ошибки (метапознание-чекпоинты)
- **«CTR хороший, а продаж нет»** → разрыв между интересом к офферу и готовностью платить. 3–5 интервью с теми, кто кликнул, но не купил: непонятна ценность / цена не соответствует воспринимаемой / барьер доверия / сомнение в исполнении.
- **«Люди говорят "хочу", но не платят»** → классический разрыв декларируемого и реального поведения. Слова дешевле денег. Снизить барьер (предзаказ), Wizard of Oz, deeper интервью о причинах.
- **«Результаты противоречивы: половина реагирует хорошо»** → сигнал о неоднородности аудитории. Сегментировать данные, найти сегмент с высокой конверсией — возможно, он и есть приоритетный рынок.
- **«Все три гипотезы не подтвердились»** → перед pivot проверить: честные ли эксперименты (правильная аудитория/метод), не было ли технических проблем, достаточна ли выборка. Если всё корректно — диагностировать уровень проблемы (матрица выше) и pivot.
- **«Pre-experiment statement не записан»** → остановиться и записать ДО запуска; иначе соблазн интерпретировать любой результат как подтверждение.

### Success Criteria подшага
- Для каждой топ-гипотезы проведён минимум 1 эксперимент с достаточной выборкой.
- Зафиксированы конкретные данные: метрика + значение + размер выборки.
- Дана интерпретация (4 сценария) и принято решение: GO / pivot / следующий цикл с обоснованием.
- Experiment-node и Insight-node коммичены в Нексус; CP обновлён.
- Context Ripeness по подшагу: ≥75% (при подтверждённой гипотезе).

## 🛡️ Анти-workslop чекпоинт
- **Главный риск**: принять сработавший оффер за подтверждённую ценность. Кортекс-агент явно разделяет: CTR/email = оффер; повторное использование/оплата/retention = ценность.
- Статистическая гигиена: минимум 30–50 конверсий для вывода о конверсии; <10 конверсий — иллюстрация, не доказательство. (Полная гигиена A/B α/β/MDE — в 4.6.)
- Pre-experiment statement обязателен — защищает от подгонки интерпретации.
- Кортекс-лендинги проходят workslop-фильтр: нет «воды», каждая секция несёт конкретный профит.
- Запрещённые синонимы: НЕ «приоритизация», НЕ «go/no-go чекбокс» → **CP-gate с оценкой стоимости риска** (Принцип 4: риски, не потери).

## 📐 Привязка к 6 принципам PAF
- **Принцип 3 (Stagility)**: стабильность долгосрочной цели (подтвердить ценность) + гибкость пути (разные методы, pivot).
- **Принцип 4 (Риск = знание)**: цикл валидации = сокращение рисков через насыщение Нексуса; CP растёт от данных.
- **Принцип 6 (Прибыль через ценность)**: различение оффер≠ценность гарантирует, что gate проходит именно подтверждённая ценность, а не сработавшая реклама.

## 🧪 Сквозной пример (из ранбука 4.3)
**EduMatch, ЦП #1 «AI-подбор за 24 часа»**. Метод: лендинг + таргет (Validation Cost 4 на скоринге).
**Pre-experiment statement (ДО запуска):** гипотеза — AI-подбор за 24 часа интересен родителям школьников; метрика — конверсия в заявку; порог — ≥12%; срок — 7 дней; выборка — минимум 1000 показов; «подтверждена» = CTR ≥ порога при значимости; «не подтверждена» = ниже порога или качественная аудитория не кликает.
**Результат (сценарий A):** конверсия 14,3% (порог 12%), охват ~3000, 3 интервью с теми, кто посетил, но не оставил заявку. Качественно: «наконец-то не нужно часами листать объявления». **CP поднимается** по колесу Гилада (стадия «лендинг»). Стоимость риска для следующего шага (пилот): 5 семей, 2 недели — приемлемо. GO → 4.4 (фиксация).
**Альтернатива (сценарий B):** конверсия 6% (порог 12%). Диагностика: в интервью говорят «хотел бы попробовать» → проблема в оффере, не в ЦП. Переформулировать оффер, повторить. Не pivot, итерация эксперимента.

### Четыре сценария интерпретации — действия
- **A. Подтверждена однозначно** — метрика превысила порог с достаточной выборкой, качественная обратная связь позитивна → GO к 4.4 (фиксация).
- **B. Не подтверждена однозначно** — метрика не достигла порога / выборка недостаточна / аудитория не та → анализ причины: оффер (переформулировать) / канал (сменить таргетинг) / само ЦП (следующая гипотеза топ-3) / pivot.
- **C. Неоднозначно** — метрика в пограничной зоне / количественное и качественное расходятся → дополнительный цикл с уточнённой гипотезой или более глубокий метод (smoke → пилот).
- **D. Подтверждена, но неожиданно** — метрика достигнута, но поведение отличается от ожидаемого (например, высокий CTR, но оставляют контакты под другим предлогом) → НЕ игнорировать расхождение: 3–5 интервью, скорректировать гипотезу, зафиксировать новое знание.

### Структура лендинга для Кортекс-генерации (преемственность из ранбука 4.3)
1. **Hero Block** — заголовок (ключевой профит, 1 строка) + подзаголовок (как именно) + CTA.
2. **Problem Block** — «Ты сталкиваешься с этим?» + 3–4 боли из Нексуса потребителя.
3. **Solution Block** — «Вот как мы решаем» + описание через выгоды, не функции.
4. **Trust Block** — социальное доказательство (отзывы/цифры/логотипы; даже без продукта — цитаты из интервью с разрешения).
5. **Pricing/Offer Block** — ценовой сигнал + повтор CTA.
6. **FAQ** — 3–5 частых вопросов из интервью.

### Pre-experiment statement (обязателен ДО запуска)
Гипотеза / метод / метрика / порог / срок / что значит «подтверждена» / что значит «не подтверждена» / размер выборки / подписи команды. Без этого документа эксперимент не запускается — защищает от подгонки интерпретации под любой результат.
