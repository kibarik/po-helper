# ЧАСТЬ I. СУТЬ ТРАНСФОРМАЦИИ

PAF отвечает на два c-level вопроса **[S1]**:

1. **Почему продуктовый подход НЕ даёт бизнес-результата?** — потому что компаниям «продали» методологии *разработки* (Scrum, Kanban) как панацею от *бизнес-проблем*. Процесс разработки ≠ процесс развития продукта. Scrum оптимизирует **потери производства**, а развитие продукта отвечает на другой вопрос: как создать продукт с **максимальным возвратом монетизируемой ценности на инвестиции (NPV)** **[S1]**.
2. **Каким должно быть управление продуктами в эпоху ИИ?** — когда практически все задачи менеджера могут быть ИИзированы. ИИ = disruptive-технология раз в сто лет. Discovery решается почти мгновенно, часть задач полностью делегируется **[S1, S2]**.

> **Главный тезис [S2]:** в эпоху ИИ конкуренция смещается с продуктов на **контекст и модель управления им**. *«У кого лучше собран контекст продукта и реализована ИИзация работы с ним — тот и будет лидером на рынке»*.

**Что это значит для библиотеки RB-STEP.** Библиотека `RB-STEP-1 … RB-STEP-8` — это сильная, добротная реализация **классического** Product Discovery (Customer Development, JTBD, Lean, AARRR, MVP по Бланку). Но почти на каждом этапе она:
- хранит контекст как **разрозненные документы** (а не живой Нексус-двойник);
- предполагает **ручную** работу (а не работу через Кортекс);
- мыслит **списками/приоритизацией/беклогом** (а не Банчем в моменте);
- оптимизирует **скорость/потери** (а не риски через Confidence Point);
- оперирует **LTV/CAC/EBITDA** как самоцелью (а не NPV/mNSM как возвратом монетизируемой ценности);
- описывает процесс как **линейный каскад с каденциями** (а не асинхронный event-based цикл Product Sprint);
- использует роли **«продакт-менеджер / маркетолог / SDR»** (а не AI-Native роли).

**Reformation = переписать каждый RB-STEP как AI-native инструкцию для Инженера Продукта нового поколения**, сохранив методологическую силу классики, но заменив операционную модель.

# ЧАСТЬ III. СДВИГ ПАРАДИГМЫ: PO → ИНЖЕНЕР ПРОДУКТА НОВОГО ПОКОЛЕНИЯ

Инструкции нового поколения (`N.M instruction`) пишутся **не для классического Product Owner/Manager**, а для **Инженера Продукта (Product Engineer)** — роли, которая управляет развитием продукта через Нексус посредством Кортекса **[S1, S2]**.

| Параметр | Классический PO (Scrum-эпоха) | Инженер Продукта нового поколения (PAF) | Источник |
|---|---|---|---|
| Базовый артефакт | Беклог (накопительный список) | **Банч** (в моменте, из текущего состояния Нексуса) | [S1] |
| Объект управления | Поток задач в разработку | **Нексусы** (рынок/продукт/рост/…) | [S1] |
| Ключевая метрика процесса | Скорость / story points / потери | **Confidence Point** (риски) | [S1] |
| Бизнес-метрика | velocity, выручка, DAU | **NPV, mNSM** (+ COGS токенов для AI-продуктов) | [S1, S2] |
| Инструмент работы с продуктом | Вручную (Sheets/Notion/Jira/Figma) | **Через Кортекс** (агенты/RAG/промпты) | [S1] |
| Способ принятия решений | Приоритизация списка задач | **Скаутинг возможностей и угроз** | [S1] Принцип 5 |
| Знания | Разрозненные документы | **Цифровой профиль продукта** (единая точка) | [S2] |
| Ритм работы | Каденции (спринты, PI) | **Event-based / Kanban** | [S1] |
| Компетенции | T-shaped | **Comb-shaped** (гребёнка за счёт Кортекса) | [S2] |
| Команда | Scrum-team (7±2) | **Pod** (может быть из 1 человека) | [S1, S2] |
| Критический навык | Процессная дисциплина | **Метапознание** (валидация ИИ) | [S2] III.4 |
| Bottleneck системы | Discovery / разработка | **Дистрибуция (последняя миля)** | [S2] III.5 (Голдратт) |

> **Ключевой фильтр [S2]:** рост качества от ИИ — **только** при достаточном метапознании. Junior без метапознания + ИИ = нулевой прирост. Компетентный, но забивший на верификацию = «убедительные, но неверные» результаты (Harvard × BCG). Поэтому каждая инструкция нового поколения содержит **явные чекпоинты метапознания** и **анти-workslop гейты**.

# ЧАСТЬ IV. 8 СКВОЗНЫХ ТРАНСФОРМАЦИЙ (применимы ко всем этапам)

Эти 8 сдвигов повторяются на каждом RB-STEP. Ниже — что меняется, где (в каких файлах) и почему.

### Сквозная 1. Документ → Нексус (живой цифровой двойник)
- **Что:** разрозненные markdown/Sheets-документы → структурированный граф Узлов Нексуса с метаданными (`источник`, `Confidence Point`, `timestamp`, `статус увядания`); Context Ripeness = полнота × актуальность как **вычисляемая** метрика **[S1] III.1**.
- **Где:** Нексус рынка (`1.1`, `3.x`), Нексус потребителя (`2.5`), Нексус продукта (`1.2`, `7.4`), Нексус системы роста (`1.3`, `5.x`, `6.x`, `8.x`).
- **Почему:** «контекст — новый способ конкуренции» **[S1] Принцип 2**; без живого двойника Кортекс «усиливает мусор».

### Сквозная 2. Ручная работа → Кортекс (ИИ как ОС)
- **Что:** ручное заполнение шаблонов → Кортекс-агенты (RAG по Нексусам, генерация/анализ/скоринг/симуляция); инженер = постановщик задач + валидатор **[S1] III.1**.
- **Где:** **все** этапы — на каждом подшаге определяется, какой Кортекс-агент его ведёт (см. Часть V, блок «Кортекс-инструменты»).
- **Почему:** «Discovery решается почти мгновенно» **[S1]**; 3 оптимизации роли — автоматизация / аугментация / агентизация **[S2] III.4**.

### Сквозная 3. Беклог/приоритизация → Банч (связка в моменте)
- **Что:** списки гипотез, матрицы MoSCoW, «спринт-беклоги», «волны», «банк гипотез» → Банч фич/конфигураций **в моменте**, формируемый анализом текущего состояния Нексуса под Карту Целей; с лимитами Bunch Size/Window **[S1] VI.6**.
- **Где:** `2.1`, `4.1`–`4.2`, `5.6`, `6.3`, `7.4`–`7.5`, `8.1`.
- **Почему:** Принцип 1 (Беклог упразднён) и Принцип 5 (скаутинг, не приоритизация). *Запрещённые синонимы* — `naming_conventions.md`.

### Сквозная 4. Скорость/потери → Confidence Point (управление рисками)
- **Что:** «уверенность 1–10», 5-балльные матрицы, go/no-go-чекбоксы → единый **Confidence Point**, растущий по колесу Гилада (ICE/RICE) и падающий при срабатывании рисков; включает **стоимость риска**; CP = критерий stage-gate **[S1] VI.7, Принцип 4**.
- **Где:** `2.2`, `3.6`, `4.2`–`4.3`, `6.3`, `7.3`, `8.1`–`8.5`.
- **Почему:** управление продуктом = **возделывание, оптимизация рисков (энтропии)**, а не потерь **[S1] Принцип 4**.

### Сквозная 5. Линейный каскад → Цикл Product Sprint
- **Что:** линейные N.1→N.6 каскады с каденциями → цикл Progress Pulse → Bunch Saturation (Und-Id-Ex) → Pitching of Trust → Build → Bale → Germination → Harvesting **[S1] VI.4–VI.5**.
- **Где:** **все** `index.md` этапов (сегодня все линейны).
- **Почему:** PAF — циклический и асинхронный; «эффект асинхронности» ломает каденции **[S1] VI.2**.

### Сквозная 6. Изолированный этап → Три Линзы (3PL)
- **Что:** этап как изолированный конвейер → этап явно привязан к линзе (Продукт/Бизнес/Стратегия) и связан с двумя другими через Карту Целей **[S1] VI.1**.
- **Где:** Customer/Value/Solution Validation → Product Lens; Market → Strategy Lens; BizModel/GTM/Acquisition → Business/Growth Lens.
- **Почему:** инженер переключается между горизонтами через 3PL **[S1]**.

### Сквозная 7. Метрики ради метрик → mNSM / NPV (+ COGS токенов)
- **Что:** LTV/CAC/EBITDA/NSM как самоцель, vanity-метрики → **mNSM** (монетизируемая NSM), **NPV** (возврат монетизируемой ценности на инвестиции), для AI-продуктов — контроль **COGS токенов** как отдельного рычага **[S1] Принцип 6; [S2] III.6**.
- **Где:** `3.3` (SOM→mNSM-потенциал), `4.5`–`4.6` (NSM→mNSM), `5.5` (NPV как вершина), `6.4` (бюджет как инвестиция в NPV), `7.6` (PMF→mNSM/NPV), `8.5` (PCF→NPV рычага).
- **Почему:** Принцип 6 и целевая функция PM из **[S3]**.

### Сквозная 8. Документация → Цифровой профиль + метапознание + анти-workslop
- **Что:** «база знаний»/презентации как финальный артефакт → **цифровой профиль продукта** (аналог claude.md, единая точка); каждый Кортекс-агент имеет **метапознание-чекпоинт** и **workslop-фильтр** **[S2] III.4, III.5, III.7**.
- **Где:** фиксирующие подшаги (`3.6`, `4.4`, `5.6`, `7.4`, `8.5`); генерирующие подшаги (`3.3`, `4.1`, `6.2`, `6.6`, `7.2`, `8.1`).
- **Почему:** workslop ≈ 15,4 % материалов перекладывают когнитивную нагрузку; MIT — 95 % не получают отдачи от инвестиций в ИИ именно из-за workslop **[S2] III.7**.

## 🟦 RB-STEP-1 — BIG IDEA (прелюдия)

### 1.0 Вердикт
**~58% PAF.** Стадия **уже построена на PAF-терминологии** (три Нексуса `1.1–1.3`, скаутинг через 3 Линзы `2.1–2.3`, Банч `3.3`, Confidence Point `3.2`, Pitching of Trust `6.1`, Pre-mortem `6.2`). Самый высокий скоринг в библиотеке. Но: Кортекс — точечный промпт (`3.3`), а не ОС; нет NPV Банча; нет Context Ripeness как live-метрики; нет метапознания/анти-workslop; нет маппинга к Skill Map и профстандарту; нет модели зрелости Кортекса; Product Sprint не встроен (линейный pipeline 0→5).

### 1.1 Что сейчас
Полностью переработанный discovery, изложенный в PAF-терминах (`index.md`, `1.0 Базовые концепции`). Три Нексуса определяются явно; используется скаутинг через Strategy/Business/Product; формируется Банч через AI-анализ (`3.3`); шкала CP 1–10; Pitching of Trust — gate. Роль — «Product Engineer». Но не интегрирован с каркасом PAF: нет Нексуса продукта, нет Кортекса, нет Банча (вместо него backlog/MoSCoW), нет цикла Product Sprint, роли классические.

### 7.1 Что сейчас
Линейная двухфазная последовательность: Фаза А — проектирование и проверка (`7.1` Solution Hypothesis Cards + CJM + User Flow + Lean UX Canvas + RAT-приоритизация; `7.2` прототипы бумага→wireframe→кликабельный→консьерж→Волшебник Оз→no-code→Soft Launch; `7.3` юзабилити-тесты ≥5 + скоринг ценность/стратегия/ресурсы); Фаза Б — разработка (`7.4` User Stories/Use Cases/NFR/MoSCoW/DoD/Test Cases; `7.5` Entity Model + Sequence/State + MVP-разработка с параллельной верификацией; `7.6` AARRR + когортный retention + опрос Шона Эллиса + Aha Moment + критерии PMF). PMF-оптика корректна, но изолирована от Нексуса/Кортекса/Банча.

### 7.2 Цель в PAF-оптике
**Запустить цикл Product Sprint до точки PMF**, где Solution Validation растворяется в механизме: Solution Hypothesis Cards (`7.1`) = **Identify-фаза Und-Id-Ex**; прототипы (`7.2`) = **скаутинг** для поднятия CR Нексуса продукта; валидация (`7.3`) = **Pitching of Trust** (CP как gate); требования (`7.4`) = **цифровой профиль продукта**, с которым работают агенты; MVP (`7.5`) = **execute-фаза + вайбкодинг** через Кортекс (Discovery/Launcher PM); аналитика/PMF (`7.6`) = **Harvesting** (NPV/mNSM как инкремент контекста). Артефакт — **спелый Нексус продукта** с измеримой mNSM, а не «пакет требований v0.1».

### 7.3 Критические разрывы
1. **Банч vs Backlog** [`7.4` MoSCoW, `7.5` Sprint Plan] → Банч из текущего состояния + Size/Window.
2. **Цикл Product Sprint отсутствует** (линейный waterfall А/Б) → Progress Pulse/Pitching/Bale/Germination/Harvesting.
3. **Нет Нексуса продукта** (спецификация-контракт в Notion) → живой двойник с CR.
4. **Нет Кортекса** (Figma/Miro/Linear; точечные Otter.ai/ManyChat) → агенты генерации/прототипов/тестов/требований/PMF.
5. **Роли «PM»** → Product Engineer/**Discovery-Launcher PM (вайбкодинг)**/**AI UX Designer**/Product Ops.
6. **PMF-критерии не привязаны к mNSM/NPV** [`7.6` NSM = «завершённые уроки»] → NSM→**mNSM**; NPV как критерий.
7. **CP не используется как gate-механизм** (разовые оценки 1–10) → единый CP, растущий по колесу Гилада.
8. **Вайбкодинг/anti-workslop не учтены** (`7.2`, `7.5` — естественная зона вайбкодинга).

## ⬛ RB-STEP-8 — ACQUISITION CONFIGURATION (Stage 7)

### 8.0 Вердикт
**~38% PAF.** Честно работает с тем же объектом, что и PAF (конфигурация сегмент-канал-оффер, Product-Channel Fit, CAC/LTV), опирается на первоисточник `productframework.ru/activities/acquisition_configuration_research`. Но реализована как линейный классический discovery с ручными карточками, фиксированными волнами и кадровыми ритуалами, без Нексуса, Кортекса, CP как ядра, метрики CR и AI-Native ролей.

### 8.1 Что сейчас
`index.md` определяет объект = «конфигурация сегмент–канал–оффер», цель — Product/Channel Fit. Линейно: `8.1` карточки гипотез (7 полей, «Уверенность 1–10») → `8.2` валидация 3 фазы (Demand/Purchase/Unit Economics) → `8.3` оптимизация «одна переменная за раз» → `8.4` «лестница бюджетов» +20–30% → `8.5` фиксация + взвешенный скоринг + количественный CAC/LTV для PCF. Роли — «продакт-менеджер/маркетолог-Growth/аналитик»; A/B — классическое 95% или байесовское P2BB≥75%; артефакты — карточки в Sheets/Notion.

### 8.2 Цель в PAF-оптике
Bottleneck сместился на **дистрибуцию** — именно сюда попадает Stage 8 [S2, Голдратт]. Цель = **Harvesting модели роста**: каждая конфигурация = **Бейл** (Bale = инкремент + оффер + план дистрибуции), финал (PCF) = точка, где **NPV модели роста положителен** (LTV окупает CAC). Конфигурация = узлы **Нексуса системы роста**; перебор = **скаутинг**; управление рисками = рост/падение **CP** по колесу Гилада; Кортекс ведёт весь цикл Und-Id-Ex.

### 8.3 Критические разрывы
1. **Нексуса системы роста нет** (CR = декоративная метрика, карточки в Sheets) → узлы с двумя фазами (созревание/увядание); авто-обновление метрик.
2. **Кортекса нет** (всё вручную) → агенты на каждую операцию.
3. **«Уверенность 1–10» вместо CP** [`8.1`] → единый CP со стоимостью риска; уровни `8.4` = рост CP.
4. **Список гипотез + «волны» = Беклог** [`8.1`] → Банч конфигураций в моменте.
5. **Линейность вместо event-based** [`8.1`→`8.5`] → Kanban: событие «CP вырос» → следующий гейт.
6. **Роли PM/маркетолог** [`8.1` Phase 0] → Product Engineer/**Growth PM (time-to-activation, LTV, COGS токенов)**/Product Ops.
7. **3PL не привязаны** → явно Business Lens (рычаг дистрибуции → NPV); PCF связан со Ставками портфеля.

| **COGS-эксплозия AI-фич** | `5.4`/`5.5` · `6.4` · `8.4` | AI-COGS как отдельный рычаг (cost-per-token × tokens-per-user); стресс-тест «usage ×2/×5»; COGS токенов в полный CAC |
| **Bottleneck на дистрибуции** | `6.x` · `8.x` | Явный тезис Голдратта; Germination + COGS; рост adoption/usage под контролем |
| **Дофаминовая петля ИИ** | Все этапы при росте доли ИИ | Явное разделение зон человека (стратегия, синхронизация) и ИИ (транскрипция, скоринг); метрика ритуала = изменение CP/NPV |
| **Ложное ощущение готовности** | `5.5` (симуляция) · `7.6` (premature PMF) · `8.5` (ложный PCF) | Конвергенция ≥3 сигналов; Soft Launch обязателен; CP не поднимается без реальных данных |
| **Иллюзия живого Нексуса** | `7.4` цифровой профиль · все фиксирующие подшаги | CR как метрика с фазами созревание/увядание; обновление только через агентов, не ручное редактирование |
| **Утрата стратегического контроля** | `5.5` NPV-моделирование · `8.5` фиксация PCF | Стратегическое планирование и синхронизация людей — всегда за человеком; агенты = инструменты, не акторы **[S2] III.4** |
