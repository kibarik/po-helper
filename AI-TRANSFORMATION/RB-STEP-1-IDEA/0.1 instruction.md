# 0.1 instruction — Progress Pulse: стартовый ритуал Product Sprint

> Этап **RB-STEP-1 — BIG IDEA** (прелюдия) · Линза: **Strategy + Product + Business** (запускающая, собирает состояние всех трёх) · Инструкция для **Инженера Продукта нового поколения** (PAF, [S1],[S2]).
> Преемственность метода: в исходном ранббуке нет отдельного ритуала Progress Pulse — `index.md` библиотеки открывается фазой «Подготовка Нексусов» как линейный вход. Эта инструкция вводит **Progress Pulse как событие 1 цикла Product Sprint** [S1] VI.4, в которое обёртывается вся прелюдия BIG Idea.

## 🎯 Цель в PAF-оптике

Progress Pulse — это **первое событие цикла Product Sprint** [S1] VI.4:
фиксация текущего состояния Нексусов (рынка, продукта, роста) «как пилоты у приборной панели» [S1] VI.4,
измерение **Context Ripeness** (полнота × актуальность) [S1] III.1,
и формирование из разрыва между текущим состоянием и Видением **первого Гэпа (gap)** [S1] базовой логики.

В классике прелюдия стартует с «открой первый Нексус по списку» — линейный вход.
В PAF-оптике цель прелюдии — **запустить event-based цикл** [S1] VI.2, VI.4:
Progress Pulse → насыщение Банча (Und-Id-Ex) → Pitching of Trust → Build → Bale → Germination → Harvesting → цикл заново.
Эта инструкция задаёт стартовую точку цикла, триггеры перехода и определяет базис старта.

**Два базиса методики Und-Id-Ex** [S1] VI.5 (фиксируется здесь, влияет на весь цикл):
- **Null base** — нет данных/продукта → discovery с нуля: гипотезы потребностей → ЦП → реализации. 1.2 будет генерировать Vision.
- **Data base** — есть продукт → извлекаем знания о поведении, находим бутылочные горлышки. 1.2 будет заливать Current State из product analytics.

## 👤 Роль и ответственность

- **Product Engineer** — запускает цикл Product Sprint; считывает состояние Нексусов через Кортекс;
  формулирует стартовый Гэп между Current State и Vision; ставит задачу Кортексу на насыщение;
  определяет базис (null/data) цикла.
- **Product Ops / Product Architect** — проверяет, что Нексусы существуют как графы Узлов
  (а не разрозненные документы), что **Context Ripeness вычисляется как метрика** (а не самооценка),
  и что у каждого Узла есть метаданные (`источник`, `Confidence Point`, `timestamp`, `статус увядания`) [S1] III.1.
  Миссия — двигать команду по модели зрелости Кортекса [S2].
- **Portfolio Manager** (если есть) — сверяет стартовый Гэп с Картой Целей портфеля (Goal Map) [S1] III.2,
  чтобы прелюдия не отклонилась от Ставок (Bets) стратегической линзы.
- **Зона человека** (не передавать ИИ): **целеполагание** — какой Гэп закрываем этим циклом,
  какие Ставки портфеля обслуживает прелюдия.
  Стратегическое планирование и распределение ресурсов всегда за человеком [S2] III.4.

## 🤖 Что делает Кортекс-агент

**Nexus-Pulse Agent** (фаза зрелости 2+ Кортекса [S1] Часть VII) — событийно считывает три Нексуса и возвращает:

1. Список Узлов с метаданными по каждому Нексусу;
2. Значение Context Ripeness по каждому Нексусу с фазой «созревание»/«увядание» [S1] III.1;
3. Флаги устаревших Узлов (где `timestamp` старше порога свежести);
4. Предварительный список «дыр» (отсутствующих Узлов), критичных для решений BIG Idea.

**Пороги свежести по типу Узла** (настраивает Product Ops):
- `TAM-SAM-SOM` — квартал (медленно меняется);
- `Trend` — месяц;
- `Competitor`-отзывы — неделя (быстро устаревают);
- `Unit-Economics` — месяц;
- `JTBD` — полгода (медленно, если сегмент стабилен).

- **Вход:** RAG-индекс по трём Нексусам [S1] III.1 + (на null base) пустые скелеты.
- **Действие:** вычисляет `Context Ripeness = полнота × актуальность`;
  помечает Узлы в стадии «увядания» [S1] III.1; генерирует стартовый дашборд Pulse.
- **Выход:** **Pulse-узел** в Кортексе со снимком состояния + триггер `GAP_DEFINED`
  → передача в подшаги 1.1–1.3 (насыщение Нексусов).

**Промпт-идея:**
> «Считай три Нексуса BIG Idea. Для каждого верни:
> (1) список Узлов с метаданными;
> (2) CR = полнота(0–1) × актуальность(0–1) с фазой созревание/увядание;
> (3) флаги увядания — узлы старше порога свежести по типу без обновления;
> (4) критические дыры для решения о Банче.
> Не заполняй Узлы — только снимок. Верни Pulse-узел с триггером GAP_DEFINED.»

## 🧠 Что делает человек (метапознание)

- **Чекпоинт валидации ИИ:** Product Engineer проверяет, что Кортекс не «придумал» Узлы на пустом месте —
  каждый Узел обязан иметь `источник` [S1] III.1.
  На null base пустые скелеты — это честно, а не «заполненные галлюцинациями».
- **Veto-право:** если Кортекс показывает CR ≥60% по Нексусу, которого физически не существует, —
  это **иллюзия живого Нексуса** (сквозной риск трансформации, [reformation] Часть VIII).
  Инженер вправе обнулить метрику.
- **Вопросы «откуда эта цифра»:**
  - «Почему CR Нексуса Роста = 45%? Какие Узлы посчитаны в полноту?»
  - «Какой `timestamp` у юнит-экономики? Какой Узел в `stale`?»
  - «Кортекс, ты откуда взял этот Узел тренда? URL и дата?»
- **Чекпоинт базиса:** инженер честно фиксирует null vs data base —
  это определяет, будет ли 1.2 генерировать Vision (null) или заливать Current State (data) [S1] VI.5.
- **Ограничение автономии (для junior):** на низкой зрелости Кортекса инженер не доверяет авто-CR,
  а сверяет выборку Узлов вручную (метапознание как фильтр прироста качества [S2] III.4).

## 📦 Артефакт = узлы Нексуса

- **Создаётся/обновляется:** **Pulse-узел** (мета-узел Кортекса, не Нексуса) —
  снимок состояния трёх Нексусов в момент старта цикла.
- **Метаданные Pulse-узла:**
  - `cycle_id` — идентификатор цикла Product Sprint;
  - `timestamp` — момент Pulse;
  - `CR_рынка`, `CR_продукта`, `CR_роста` — спелость каждого Нексуса (0–1);
  - `gap_summary` — стартовый Гэп (что закрываем);
  - `базис` — null/data;
  - `trigger=GAP_DEFINED`;
  - `бюджет_цикла` — лимит ресурсов на насыщение.
- **Прирост Context Ripeness:** на этом подшаге — **только фиксация**, не насыщение.
  Прирост приходит в 1.1–1.3. Pulse — это «приборная панель», а не источник новых Узлов.

## 🚦 Confidence Point-гейт

- **Критерий перехода к 1.1:** существует Pulse-узел + стартовый Гэп сформулирован
  (даже как «gap = весь Vision, продукта ещё нет» на null base).
  Без Гэпа цикл Product Sprint не запускается [S1] базовая логика (Гэп → Банч).
- **Стоимость риска** (Принцип 4 [S1]): «Сколько ресурсов мы готовы потратить на насыщение,
  прежде чем Гэп станет измеримым?» — фиксируется в Pulse-узле как бюджет цикла.
  Риски — будущее, не потери [S1] Принцип 4.
- **Связь со stage-gate:** рост CP по циклу = рост скорости процессов [S1] VI.3;
  Pulse фиксирует стартовый CP-уровень Нексусов.

## 🔭 Связь с Тремя Линзами (3PL)

- Progress Pulse **не привязан к одной линзе** — он собирает состояние всех трёх
  и определяет, какая линза включается первой для насыщения [S1] VI.1:
  - новый рынок → первой Strategy Lens (1.1 Нексус рынка);
  - новый продукт → первой Product Lens (1.2 Нексус продукта);
  - новая модель роста → первой Business Lens (1.3 Нексус роста).
- Связь с двумя другими — через **Карту Целей (Goal Map)** [S1] III.2:
  Pulse сверяет Гэп с целью бизнес-юнита;
  синхронизация между командами — без прямого вмешательства, через Карту Целей [S1] III.2.

## 🛡️ Анти-workslop чекпоинт

- **Workslop-риск здесь:** «дашборд ради дашборда» — Pulse, который красиво выглядит,
  но Узлы в нём без источников [S2] III.7
  (~15,4% материалов перекладывают когнитивную нагрузку;
  MIT — 95% не получают отдачи от ИИ именно из-за workslop).
- **Признак workslop:** CR посчитан из Узлов, у которых `источник = "Кортекс предположил"`;
  дашборд без возможности drill-down до источника.
- **Правило верификации:** каждый Узел в Pulse-снимке обязан иметь внешний или внутренний источник
  (URL, дата, интервью, product analytics).
  Узлы без источника → CP=0 [reformation] Часть VIII.
  Дашборд должен уметь показать `источник` каждого Узла в один клик.

  (сопровождение, A/01.4 оргсопровождение исследований), уровень квалификации 4 — ассистент/junior.
