# 5.3 instruction — Обновление Нексусов v1.0

> Этап **RB-STEP-1 — BIG IDEA** (прелюдия) · Линза: **над 3PL** (фиксирующая по всем трём Нексусам) · Инструкция для **Инженера Продукта нового поколения** (PAF, [S1],[S2]).
> Преемственность метода: исходный раннбук `RB-STEP-1-IDEA/5.3 Обновление Нексусов.md` — ручное обновление трёх Нексусов до v1.0 (Validation Insights в рынок; Customer Insights/Prioritization/Vision Refinement в продукт; Unit Economics Updates/Growth Strategy в рост) + Summary (CR-таблица по Нексусам). Эта инструкция: **авто-коммит** learnings (не ручное редактирование), **пересчёт Context Ripeness как live-метрики**, **флаг «увядания»** ([reformation] §1.3 разрыв 2, Сквозная 1).

## 🎯 Цель в PAF-оптике

Зафиксировать все learnings цикла BIG Idea в Нексусах как **v1.0** —
не «дописать разделы вручную»,
а **авто-коммит** через Кортекс (event-based обновление Узлов [S1] VI.2),
**пересчёт Context Ripeness** (полнота × актуальность [S1] III.1)
с автоматическим **флагом «увядания»**
для Узлов, потерявших актуальность.

Это подготовка Нексусов к следующему циклу Product Sprint [S1] VI.4:
на выходе CR всех трёх Нексусов ≥60%
([reformation] §1.6 Fit-точка: Нексусы v1.0 с CR≥60%).

## 👤 Роль и ответственность

- **Product Engineer** — отвечает, что Нексусы отражают реальность
  (не приукрашенную);
  инициирует v1.0-коммит.
- **Product Ops** — обеспечивает событийный авто-коммит;
  поддерживает метрику CR как вычисляемую (не самооценку);
  настраивает пороги увядания по типам Узлов.
- **Зона человека:** **решение о достаточности CR для закрытия цикла** —
  за человеком.
  Кортекс считает CR, человек решает «готовы ли Нексусы к gate».

## 🤖 Что делает Кортекс-агент

**Nexus-Commit Agent** (режим Nexus-Builder [reformation] §1.5):

- Авто-коммит всех `Learning`-узлов (4.3) + `Validation-Result` (4.2)
  + `Vision-v1` (5.1) в соответствующие Нексусы
  (рынок/продукт/рост) с обновлением рёбер.
- **Пересчёт Context Ripeness** по каждому Нексусу:
  `CR = полнота (покрытие типов Узлов) × актуальность (по timestamp Узлов)` [S1] III.1.
- **Флаг «увядания»:**
  Узлы с `timestamp` старше порога свежести
  (зависит от типа: TAM — медленно, отзывы — быстро)
  → статус `aging`/`stale` [S1] III.1.
- Генерация Summary (CR-таблица по Нексусам, из исходника 5.3) —
  но как live-узел, не статичный документ.

- **Вход:** все `Learning`/`Validation-Result`/`Vision`-узлы цикла.
- **Выход:** Нексусы v1.0 (обновлённые Узлы),
  пересчитанный CR по каждому, флаги увядания,
  `Nexus-Summary-v1.0`-узел.

**Промпт-идея:**
> «Закоммить все learnings цикла в Нексусы v1.0 (авто, не ручное).
> Пересчитай CR = полнота × актуальность для каждого Нексуса.
> Поставь флаг увядания на Узлы старше порога.
> Сгенерируй Summary (CR-таблица) как live-узел.
> Цель: CR ≥60% по всем трём.»

## 🧠 Что делает человек (метапознание)

- **Чекпоинт:** **иллюзия живого Нексуса**
  (сквозной риск [reformation] Часть VIII) —
  CR посчитан, но Узлы без источников или устаревшие.
  Инженер проверяет выборку Узлов: источники? timestamp? флаги увядания?
- **Veto-право:** Нексус с CR ≥60%,
  но с увядшими/без-источниковыми Узлами → обнулить CR или пометить `иллюзия`.
- **Вопрос:**
  - «CR Нексуса рынка = 65% — сколько Узлов в `stale`?
    Какие источники у топ-5 Узлов?»
  - «Обновление Узлов прошло через агентов (event-based)
    или кто-то правил вручную?»

## 📦 Артефакт = узлы Нексуса

- **Создаются/обновляются:** Нексусы v1.0 (рынка, продукта, роста) —
  обновлённые Узлы;
  `Nexus-Summary-v1.0`-узел
  (CR-таблица, key learnings, Bunch rationale — из исходника 5.3);
  флаги увядания на Узлах.
- **Метаданные каждого Узла:** `источник`, `CP`,
  `timestamp` (обновлённый),
  `статус увядания` (fresh/aging/stale),
  `версия-Нексуса` (v1.0).
- **Прирост Context Ripeness:** это **итоговый** замер цикла;
  цель ≥60% по всем трём [reformation] §1.6.

## 🚦 Confidence Point-гейт

- **Критерий перехода к 6.1:** Нексусы v1.0 с CR ≥60% по каждому;
  `Nexus-Summary-v1.0` существует;
  критичные Узлы — не `stale`.
- **Стоимость риска:** «Какой ущерб решениям,
  если Нексус иллюзорно спелый?» —
  gate не пропускает иллюзии.

## 🔭 Связь с Тремя Линзами (3PL)

- Над 3PL: фиксирующая по всем трём Нексусам.
- **Связь:** Нексус рынка → Strategy Lens;
  продукта → Product Lens;
  роста → Business Lens —
  все замыкаются для следующего цикла.

## 🛡️ Анти-workslop чекпоинт

- **Workslop-риск:** **иллюзия живого Нексуса**
  (CR посчитан, Узлы устарели/без источников) [reformation] Часть VIII;
  ручное редактирование вместо авто-коммита
  (Нексус перестаёт быть живым).
- **Признак:** CR ≥60% при `stale`-Узлах;
  Узлы без источников;
  ручная правка Узлов (не через агентов).
- **Правило верификации:** CR = вычисляемая метрика;
  флаги увядания;
  обновление только через агентов (event-based);
  выборочная проверка источников.
