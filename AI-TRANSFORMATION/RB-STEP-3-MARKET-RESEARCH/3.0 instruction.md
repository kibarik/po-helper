# 3.0 instruction — Нексус рынка как живой двойник

> Этап **RB-STEP-3 — MARKET RESEARCH** (Stage 2) · Линза: Strategy (контекст: Product/Business) · Инструкция для **Инженера Продукта нового поколения** (PAF, [S1],[S2]).
> Преемственность метода: исходный раннбук `RB-STEP-3-MARKET-RESEARCH/index.md` (Context Ripeness 0–100%, область потребностей, go/no-go).

## 🎯 Цель в PAF-оптике
Превратить разрозненные рыночные документы (карта сил, база трендов, расчёт TAM/SAM/SOM, карточки конкурентов) в **Нексус рынка** — живой цифровой двойник, насыщаемый узлами до **Context Ripeness ≥ 60%**, достаточной для скаутинга возможностей/угроз (Принцип 5) и поднятия CP рыночных гипотез до CP-gate стратегической линзы. Нексус рынка — объект **Strategy Lens**: Strategic Sprint → Нексус рынка → Ставки/Bets [S1] VI.1.

## 👤 Роль и ответственность
- **Product Engineer** — определяет структуру Узлов Нексуса рынка, ставит задачу Кортексу на насыщение, валидирует спелость, решает, достаточно ли контекста для перехода к Bet-узлам. Работает **только через Кортекс**, не вручную [S1].
- **Product Ops** — строит инфраструктуру Нексуса (схема Узлов, RAG-индексы, триггеры event-based обновления), настраивает авто-метрику Context Ripeness = полнота × актуальность, поддерживает флаги «увядания» [S1, S2].
- **Portfolio Manager** — потребитель Bet-узлов Нексуса рынка (максимизация NPV портфеля); формирует Ставки на основе спелого подграфа [S1].
- **Зона человека**: интерпретация рыночных сигналов как возможностей/угроз; стратегические Ставки (стратегическое планирование не делегируется ИИ) [S2] III.4.

## 🤖 Что делает Кортекс-агент
**Nexus-Builder Agent** (рынок) на каждом подшаге 3.1–3.6:
- Создаёт Узлы 7 типов: **Force-узел** (Porter/PESTEL), **Trend-узел**, **Constant-узел**, **TAM-SAM-SOM-узел**, **Competitor-узел**, **Gap-узел**, **Bet-узел** (Strategy Lens).
- Считает **Context Ripeness** как вычисляемую метрику (полнота × актуальность), а не галочный чеклист; две фазы — «созревание» (рост инфо) и «увядание» (потеря актуальности по `timestamp`) [S1] III.1.
- Привязывает каждый Узел к **области потребностей** конкретного сегмента (не «рынок в целом», а «задача X клиента Y») — требование исходного `3.6`.
- Триггерит event-based обновление: событие (новый отчёт, новость, refresh) → пересчёт CR → аларм при падении CR ниже порога.

### Шкала Context Ripeness (из исходного `index.md` строк 90–97, переведена в live-метрику)
| CR | Интерпретация | Действие |
|---|---|---|
| 0–40% | Первичный контекст, слишком мало данных | Продолжать насыщение Узлами |
| 40–60% | Достаточно для первого решения | Регулярно обновлять |
| 60–80% | Хороший контекст | Можно двигаться к CP-gate |
| 80–100% | Глубокий контекст (редко на старте) | Поддерживать актуальность |

В PAF CR — **не самооценка документа**, а функция: `CR = f(полнота покрытия 7 типов Узлов, свежесть timestamp, средний CP Узлов)`. Увядание: Узел без обновления N месяцев → флаг `decay_status: увядание`, падение вклада в CR.

### Lifecycle Нексуса рынка
1. **Phase-0 (нулевой шаг любой инструкции)** [S1] Часть IX: сначала Нексус, потом Кортекс. Без спелого контекста Кортекс «усиливает мусор».
2. **Насыщение** (3.1–3.5): агенты создают Узлы из внешних источников → рост CR.
3. **Commit** (3.6): Bet-узлы фиксируются, Нексус рынка v1.0 (CR ≥ 60%).
4. **Event-based поддержка**: после релиза Кортекс обновляет Нексус → меняется Банч → цикл [S1] базовая логика 5.

## 🧠 Что делает человек (метапознание)
- Решает, какая область потребностей является фокусом оценки (кто + что хочет сделать) — это не делегируется ИИ.
- Проверяет, что Нексус рынка — **двойник**, а не «документ с галочками»: наличие Узлов с метаданными (URL+дата источника, CP, timestamp, статус увядания).
- Отличает **факт** (URL+дата, верифицировано) от **гипотезы** (допущение, явная маркировка) от **бенчмарка** (аналогия по другому рынку) [S2] III.4.
- Veto на галлюцинированный Узел (Узел без источника → CP=0, в Нексус не принимается).

## 📦 Артефакт = узлы Нексуса рынка
7 типов Узлов с обязательными метаданными:
| Тип Узла | Что фиксирует | Источник (исходный раннбук) |
|---|---|---|
| **Force-узел** | Одна из 5 сил Портера или PESTEL-фактор + уровень + обоснование | `3.1` |
| **Trend-узел** | Тренд (тип/горизонт/влияние/источник) — меняется | `3.2` |
| **Constant-узел** | Константа рынка (что не изменится 5–10 лет) — фундамент стратегии | `3.2` |
| **TAM-SAM-SOM-узел** | Оценка объёма (клиенты/деньги) + метод + допущения | `3.3` |
| **Competitor-узел** | Конкурент 4 типов + профиль + teardown + оценки | `3.4` |
| **Gap-узел** | Рыночный пробел (где никто не обслуживает хорошо) | `3.4`, `3.6` |
| **Bet-узел** | Ставка стратегического сценария (Strategy Lens) | `3.6` (PAF-дополнение) |

Каждый Узел: `источник[]` (URL+дата), `CP`, `timestamp`, `статус увядания`, `связи с другими Узлами`.

### Схема метаданных Узла (обязательный минимум)
```
node_id: <уникальный идентификатор>
type: Force | Trend | Constant | TAM_SAM_SOM | Competitor | Gap | Bet
title: <название>
scope: <область потребностей: сегмент + задача>   ← НЕ «рынок в целом»
source: [{url, date, publisher, type: fact|hypothesis|benchmark}]   ← ОБЯЗАТЕЛЬНО
cp: <Confidence Point 0..10>
timestamp: <дата создания>
last_updated: <дата обновления>
decay_status: созревание | увядание
links: [<node_id связанных Узлов>]
```
Узел без поля `source` → CP=0, флаг «галлюцинация», в Нексус не принимается.

### Связь с существующей нотой `1.1 Нексус Рынка`
Нексус рынка на Stage 3 — **углубление и операционализация** концепта, зафиксированного в ноте [[1.1 Нексус Рынка]] (Stage 1 / RB-STEP-1). На Stage 1 Нексус рынка определён декларативно (TAM/SAM/SOM, конкуренты 4 типов, тренды, opportunities/threats). На Stage 3 он:
- наполняется **верифицированными Узлами** с источниками (а не тезисами);
- получает **live-метрику Context Ripeness** (вместо галочной шкалы);
- связывается с **Bet-узлами** Strategy Lens (Ставки портфеля);
- переходит из v0.x (Stage 1) в v1.0 (Stage 3, CR ≥ 60%).

## 🚦 Confidence Point-гейт
Нексус рынка считается **спелым для скаутинга**, когда:
- CR ≥ 60% (минимум для стратегического решения; исходный `index.md` Success Criteria) [S1];
- все 7 типов Узлов представлены хотя бы одним экземпляром;
- каждая рыночная цифра в TAM-SAM-SOM-узле имеет URL+дату (иначе CP=0).

CP-gate стратегической линзы = `{CP рыночной гипотезы ≥ порога} ∧ {стоимость риска приемлема} ∧ {mNSM-потенциал SOM > 0}` — см. `3.6 instruction`.

## 🔭 Связь с Тремя Линзами (3PL)
Нексус рынка = объект **Strategy Lens** [S1] VI.1. Связь с двумя другими линзами:
- **Product Lens** — Gap-узлы и Constant-узлы питают Банч фич (насыщение Understand, Und-Id-Ex).
- **Business Lens** — TAM-SAM-SOM-узел связан с Рычагами (Lever) и NPV модели роста.

Strategic Sprint: долгосрочные цели + Нексус рынка → Ставки/Bets [S1] VI.1.

## 🛡️ Анти-workslop чекпоинт
- Нексус рынка ≠ «рыночный отчёт в Obsidian». Документ как финальный артефакт = будущий workslop [S2] III.7.
- Каждый Узел обязан хранить URL+дату источника; Узел без источника → CP=0 и флаг «галюцинация».
- Context Ripeness — **вычисляемая** метрика (полнота × актуальность), а не самооценка документа (в исходном `index.md` строки 90–97 — это декоративная шкала; PAF-трансформация делает её live).
- Запрет на «рынок в целом»: оценка только в контексте области потребностей (требование исходного `3.6`).

### «Эхо» PAF в исходном `index.md` (что сохраняется)
Исходный `RB-STEP-3-MARKET-RESEARCH/index.md` уже содержит PAF-следы (reformation §3.1): прямые пересказы [S1] по TAM/SAM/SOM, трендам vs константам, конкурентам 4 типов, области потребностей, Context Ripeness (0–100% по 4 диапазонам), go/no-go. Stage 3 = **~28% PAF** (reformation §3.0). Эта инструкция не переписывает метод с нуля, а **заменяет операционную модель**: декоративная шкала CR → live-метрика; разрозненные документы → Нексус-двойник; go/no-go по объёму → CP-gate; ручной анализ → Кортекс-агенты. Сильные методы классики (Porter/PESTEL, top-down/bottom-up, правило ФРИИ 10%, 4 типа конкурентов) **сохраняются** полностью.

### Риски трансформации 3.0 (reformation §3.9)
- **Иллюзия живого Нексуса**: CR заявлен как метрика с фазами созревание/увядание, но обновление происходит вручную — «живость» теряется. Снятие: event-based обновление только через Кортекс-агентов.
- **Workslop при генерации Узлов**: TAM-цифры, галлюцинации в конкурентном анализе. Снятие: каждый Узел — URL+дата; CP=0 без источника.
- **Метапознание как bottleneck** для junior на Stage 3 (стратегически сложный этап). Снятие: ограничение автономии агентов; Product Ops как верификатор.
