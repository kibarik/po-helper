# Нексус информационных каналов

Placeholder-Узел каталога `channels`. Карта информационных потоков команды ещё не оцифрована.
Наполнение — через `/channel-map <slug>` (интервью по одному каналу) или добавлением channel-узлов по шаблону [`_template.md`](_template.md).

Каждый Узел нексуса `channels` = один канал поступления информации (рабочий чат, Email, Telegram-канал, регулярный созвон). Реестр даёт ИИ-агенту структурированное представление: **откуда** приходит информация, **для чего** канал, **по каким темам** пишут, **кто** здесь находится и к каким **участкам системы / целям** привязать сообщение.

## Структура графа каналов

| Слой | Поля Узла | Назначение |
|---|---|---|
| **Идентификация** | `channel_type`, `platform`, `handle`, `direction`, `cadence` | Что это за канал и как он течёт |
| **Назначение** | `purpose`, `topics`, `signal_types` | Для чего канал и по каким вопросам здесь пишут |
| **Связи** | `stakeholders`, `system_areas`, `goals` | Роутинг: кто в канале, какие участки системы и цели он питает |
| **Обработка** | `ingest_action` | Правило: что делать с входящей информацией отсюда |

## seed_questions

- Какие каналы поступления информации использует команда (чаты, Email, Telegram, созвоны)?
- Для чего каждый канал — по каким вопросам здесь пишут?
- Кто находится в каждом канале (стейкхолдеры)?
- Какие участки системы и цели затрагивает информация из канала?
- Что делать с входящей информацией из канала (в intake, в решения, в трекер, FYI)?
