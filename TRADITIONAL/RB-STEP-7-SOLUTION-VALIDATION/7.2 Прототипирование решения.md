# 7.2 Прототипирование решения — Полная инструкция

## 🎯 Цель этой инструкции

Создать **прототипы** продукта — упрощённые модели, на которых можно проверить гипотезы решения (из 7.1) без полноценной дорогостоящей разработки. Прототип позволяет поймать ошибки UX на этапе «картона», а не на этапе готового продукта.

**Время на выполнение:** 4–12 часов (для новичка: 8–20 часов)  
**Уровень:** Новичок (объясняем все типы прототипов по нарастающей)  
**На выходе у тебя будет:**
- Прототип(ы) для каждой ключевой гипотезы решения
- Задокументированный user flow внутри прототипа
- Чеклист для проверки каждой части user flow
- Готовность к проведению пользовательского тестирования (шаг 7.3)

---

## 📋 Что такое прототип?

### Простыми словами

Прототип — это **проверяемая модель продукта**, созданная с минимальными затратами для тестирования конкретной гипотезы. Главное свойство прототипа: он не требует разработки, а значит, позволяет быстро проверить идеи и выбросить плохие раньше, чем потратишь на них месяцы.

**Ключевой вопрос, на который отвечает прототип:**  
_Какими свойствами должен обладать продукт, чтобы приносить ценность потребителю?_

### Аналогия

Представь, что ты хочешь сделать ремонт в квартире и расставить мебель по-новому. Есть два подхода:

**Подход 1 (без прототипа):** Купить всю мебель, привезти, расставить. Понять, что диван не помещается, а шкаф закрывает балкон. Везти обратно, терять деньги и время.

**Подход 2 (с прототипом):** Нарисовать план на бумаге, вырезать из картона силуэты мебели, подвигать их по плану. За 30 минут понять то же самое — и не потерять ни рубля.

Прототип продукта — это «картонный план» твоего интерфейса.

---

## 🎯 Зачем это нужно?

### 3 главные причины

**1. Прототипы ловят ошибки дёшево**

Стоимость изменения интерфейса на этапе наброска на бумаге — минуты. На этапе wireframe — часы. На этапе готового кода — дни или недели. Чем раньше ты находишь проблему, тем дешевле её исправить.

**2. Прототипы дают реальную обратную связь**

Люди не могут надёжно ответить на вопрос «вы бы пользовались этим?». Но они могут показать реальное поведение с прототипом в руках — где нажимают, где путаются, где уходят. Это настоящие данные.

**3. Прототипы позволяют сравнивать альтернативы**

Если у тебя есть три гипотезы решения (из 7.1) — прототипы позволяют протестировать все три и выбрать лучший. Это называется скорингом решений (7.3). Без прототипов ты выбираешь «вслепую».

**Что будет плохо без этого шага:**
- Разработчики построят «не то» — потому что никто не видел, как это работает в реальности
- Первые пользователи столкнутся с неудобным интерфейсом и уйдут
- Ты потратишь 2–3 месяца на разработку фичи, которая не работает

## 📊 Типы прототипов: от быстрых к дорогим

В PAF прототипы делятся по стоимости создания и качеству получаемых данных. Чем «быстрее» прототип — тем дешевле, но тем менее точны данные. Чем «дороже» — тем точнее, но тем больше ресурсов.

**Важный инсайт из методологии PAF:** Ресурсы на создание «быстрых» прототипов коррелируют с точностью соответствия конечному интерфейсу, но **не** коррелируют с объёмом знаний, получаемых при валидации гипотез. Другими словами, wireframe может дать те же инсайты, что и кликабельный прототип — но в 10 раз быстрее.

```
┌─────────────────────────────────────────────────────────────┐
│  Типы прототипов (от быстрых к дорогим)                     │
│                                                             │
│  ① Набросок на бумаге        ← 5–30 минут, $0             │
│  ② Wireframe (статичный)     ← 1–4 часа, $0               │
│  ③ Кликабельный прототип     ← 1–3 дня, $0                │
│  ④ Консьерж                  ← 1–3 дня + ручная работа    │
│  ⑤ Волшебник Оз              ← 2–5 дней + ручная работа   │
│  ⑥ Чужой продукт (API)       ← 1–5 дней                   │
│  ⑦ No-code прототип          ← 3–7 дней                    │
│  ⑧ Soft Launch               ← 2–6 недель                  │
└─────────────────────────────────────────────────────────────┘
Правило: начинай с самого быстрого, который даёт нужный ответ
```

## 📝 Phase 1: Выбор типа прототипа (30 мин)

### Step 1.1: Определи, что именно нужно проверить

Перед созданием прототипа ответь на три вопроса:

```markdown
## Решение перед прототипированием

### 1. Какую гипотезу проверяем?
[Скопируй из 7.1 карточку гипотезы с наивысшим приоритетом]

### 2. Что именно нужно проверить в прототипе?
□ Навигация (может ли пользователь найти нужное?)
□ Понятность (понимает ли, что здесь делать?)
□ Желание действовать (хочет ли нажать ключевую кнопку?)
□ Доверие (верит ли информации?)
□ Ценность (видит ли, что это ему поможет?)

### 3. Какой минимальный прототип даст нужный ответ?
[Выбери тип из таблицы ниже]
```

### Step 1.2: Выбери тип прототипа по задаче

| Что нужно проверить | Рекомендуемый тип прототипа | Время создания |
|---------------------|----------------------------|----------------|
| Общая структура / навигация | Бумажный скетч или wireframe | 30–120 мин |
| Понятность интерфейса | Кликабельный прототип в Figma | 4–8 часов |
| Интерес к ценностному предложению | Лендинг на Tilda | 1–2 дня |
| Готовность платить (тест спроса) | Лендинг с кнопкой оплаты | 1–2 дня |
| Сложная логика / AI / алгоритм | Консьерж или Волшебник Оз | 2–5 дней |
| Мобильное приложение | Glide / Adalo или Figma mobile | 1–5 дней |
| Интеграция с внешними данными | Чужой продукт (API / платформа) | 1–3 дня |
| Максимально реалистичный тест | Soft Launch | 2–6 недель |

**Правило выбора:** Начинай с самого простого типа, который даёт ответ на ключевой вопрос гипотезы. Не усложняй без необходимости.

## 📝 Phase 2: Прототип Типа I — Быстрые прототипы (1–4 часа)

### Step 2.1: Бумажный скетч

Самый быстрый способ. Подходит для первичной проверки навигации и структуры интерфейса.

**Как делать:**
1. Возьми листы бумаги А4 (один лист = один экран)
2. Нарисуй схематично каждый экран из User Flow
3. Нарисуй кнопки, текст, изображения — не нужно быть художником, достаточно прямоугольников
4. Соедини стрелками: «нажатие этой кнопки ведёт к этому экрану»

```
Пример скетча экрана «Карточка репетитора»:

┌─────────────────────────────┐
│  ← Назад     [Поделиться]  │
│                             │
│  [ФОТО]    Иван Петров      │
│            ★★★★☆ 4.8 (23)  │
│            Математика       │
│            ОГЭ / ЕГЭ       │
│                             │
│  [Видеовизитка ▶ 1:45]      │
│                             │
│  «О себе» (краткий текст)   │
│                             │
│  📍 Онлайн + Москва, ЮЗАО  │
│  💰 1200 руб/час            │
│                             │
│  ─────────────────────────  │
│  Отзывы (3)                 │
│  «Отлично объясняет...»     │
│                             │
│  ┌──────────────────────┐   │
│  │  Записаться (btn)    │   │
│  └──────────────────────┘   │
└─────────────────────────────┘
```

**Когда использовать:** На самом раннем этапе, для обсуждения с командой, для первой итерации перед Figma.

### Step 2.2: Wireframe (статичный макет)

Wireframe — это схематичный макет в цифровом виде. Без цветов, без финального дизайна — только структура и контент.

**Инструменты:** Figma (бесплатно), Balsamiq, Sketch

**Как делать в Figma:**
1. Создай Frame размером 390×844 (iPhone) или 1440×900 (десктоп)
2. Используй только прямоугольники, текст и иконки-заглушки
3. Цвета: серый (#E0E0E0) для блоков, чёрный для текста — никакого брендинга
4. Не трать время на дизайн: wireframe проверяет структуру, не красоту

**Пример набора wireframe-экранов для тестирования:**

```markdown
## Wireframe-сет: «Запись к репетитору»

### Экран 1: Главная
- Поисковая строка
- Кнопка «Подобрать репетитора»
- Блок «Популярные предметы»

### Экран 2: Квиз — вопрос 1
- «Какой предмет нужен?»
- Варианты: Математика / Русский / Физика / Другое

### Экран 3: Квиз — вопрос 2
- «Для какой цели?»
- Варианты: ОГЭ / ЕГЭ / Школьная программа / Олимпиады

### Экран 4: Квиз — вопрос 3
- «Какой формат?»
- Варианты: Онлайн / Офлайн / Не важно

### Экран 5: Результаты (3 карточки)
- Карточка репетитора: фото, имя, рейтинг, специализация, цена

### Экран 6: Карточка репетитора
- Фото + Имя + Рейтинг
- Блок «Видеовизитка»
- Специализация, опыт, цена
- Блок «Отзывы»
- CTA-кнопка «Записаться на пробный урок»

### Экран 7: Форма записи
- Выбор даты и времени
- Поле для имени
- Поле для телефона
- Кнопка «Подтвердить»

### Экран 8: Подтверждение
- «Вы записаны!»
- Детали записи
- Кнопка «Добавить в календарь»
```

## 📝 Phase 3: Прототип Типа II — Кликабельный прототип (1–3 дня)

### Step 3.1: Создай кликабельный прототип в Figma

Кликабельный прототип — это wireframe с переходами между экранами. Пользователь может нажимать на кнопки и «путешествовать» по интерфейсу как в настоящем приложении.

**Когда использовать:** Когда нужно проверить логику навигации, понятность переходов, ощущение от полного user flow.

**Как настроить переходы в Figma:**
1. Выдели кнопку (элемент, на который нажимают)
2. В правой панели выбери «Prototype» → «+» → «On Click»
3. Укажи, к какому Frame ведёт нажатие
4. Повтори для каждой кнопки в user flow

**Пример настройки переходов:**
```
Экран 1 (Главная)
  → Кнопка «Подобрать репетитора» → Экран 2 (Квиз — вопрос 1)
  
Экран 2 (Квиз — вопрос 1)
  → Нажатие «Математика» → Экран 3 (Квиз — вопрос 2)
  
Экран 3 (Квиз — вопрос 2)
  → Нажатие «ОГЭ» → Экран 4 (Квиз — вопрос 3)
  
Экран 4 (Квиз — вопрос 3)
  → Нажатие «Онлайн» → Экран 5 (Результаты)
  
Экран 5 (Результаты)
  → Нажатие на карточку Ивана Петрова → Экран 6 (Карточка репетитора)
  
Экран 6 (Карточка репетитора)
  → Нажатие «Записаться» → Экран 7 (Форма записи)
  
Экран 7 (Форма записи)
  → Нажатие «Подтвердить» → Экран 8 (Подтверждение)
```

**Как поделиться прототипом:** Figma → Share → «Anyone with the link» → Copy link. Ссылка даёт доступ к интерактивному прототипу без необходимости скачивать приложение.

## 📝 Phase 4: Прототип Типа III — Консьерж и Волшебник Оз (2–5 дней)

### Step 4.1: Консьерж (Concierge Prototype)

**Что это:** Прототип, в котором вместо автоматизации — ручной труд. Пользователь думает, что пользуется автоматической системой, но за ней стоишь ты и вручную выполняешь то, что должна делать логика продукта.

**Когда использовать (по PAF):**
- Если клиенты работают онлайн и плохо разбираются в технике
- Если вы работаете с продуктами с трудно предсказуемой логистикой
- Если масштабирование операционных расходов требует значительных капиталовложений
- Если конкурентоспособность зависит от степени удовлетворённости каждого конкретного потребителя

**Пример для сервиса подбора репетиторов:**

```markdown
## Консьерж-прототип: «Подбор репетитора»

### Что видит пользователь:
Форма на лендинге Tilda:
- «Укажите предмет, цель и формат — мы подберём трёх лучших репетиторов»
- Кнопка «Получить подборку»

### Что происходит на самом деле:
1. Получаю заявку на email
2. Вручную просматриваю базу репетиторов (в Google Sheets)
3. Выбираю трёх подходящих
4. Отправляю письмо: «Вот три репетитора для вас: [карточки]»

### Что проверяем:
- Готов ли пользователь заполнить форму? (конверсия лендинга)
- Устраивают ли его предложенные варианты? (NPS после получения подборки)
- Записывается ли он к кому-то из трёх? (конверсия в запись)

### Продолжительность: 2 недели, 30–50 пользователей
```

**Главное преимущество консьержа:** Ты узнаёшь, что именно происходит в «чёрном ящике» логики (какие репетиторы нравятся пользователям, какие вопросы они задают) — это невозможно получить из автоматического алгоритма на ранней стадии.

### Step 4.2: Волшебник Оз (Wizard of Oz Prototype)

**Что это:** Прототип, в котором пользователь думает, что взаимодействует с автоматикой (AI, алгоритмом), а на самом деле за ней скрывается живой человек, которого пользователь не видит.

**Ключевое отличие от консьержа:** В консьерже пользователь знает, что получит ответ от человека. В «Волшебнике» — думает, что это AI/алгоритм.

**Когда использовать (по PAF):**
- Если конечный продукт требует разработки сложных алгоритмов или автоматизированных процессов
- Если вы работаете в деликатных отраслях (финансы, здравоохранение, знакомства, юриспруденция)
- Если вы работаете на двухсторонних рынках

**Пример:**

```markdown
## Волшебник Оз: «AI-подборщик репетиторов»

### Что видит пользователь:
Чат-бот в Telegram (создан через ManyChat):
«Привет! Я AI-ассистент для подбора репетиторов. Расскажи, что тебе нужно?»

### Что происходит на самом деле:
Менеджер сидит в Telegram Business и вручную отвечает на все сообщения — но пользователи думают, что это AI.

### Что проверяем:
- Хочет ли пользователь общаться с «AI» (vs. формой)?
- Какие вопросы задаёт? (Это покажет нам, какие фильтры нужны в реальном продукте)
- Конвертируется ли в запись?

### Важно:
Обязательно раскрой пользователям правду после теста — это вопрос этики и доверия.
```

## 📝 Phase 5: Прототип Типа IV — No-code прототип (3–7 дней)

### Step 5.1: Лендинг на Tilda

**Что это:** Полноценная страница в интернете, созданная без программирования. Может включать форму захвата, кнопку оплаты, видео, отзывы — всё, что нужно для проверки.

**Когда использовать:** Тест спроса (готовы ли платить), проверка ценностного предложения, сбор лидов.

**Чеклист страницы (лендинга):**

```markdown
## Структура лендинга для теста решения

### 1. Hero-секция (первый экран, главный)
- [ ] Заголовок: «Что потребитель получит?» (выгода, не фича)
- [ ] Подзаголовок: «Для кого это?» (сегментация)
- [ ] Визуал: скриншот / мокап / иллюстрация решения
- [ ] CTA: «Попробовать бесплатно» / «Записаться» / «Оставить заявку»

### 2. Блок «Как это работает»
- [ ] 3 шага пользователя (максимально просто)
- [ ] Акцент на результате каждого шага

### 3. Блок «Что получите»
- [ ] 3–5 ключевых выгод (не фич)
- [ ] Для каждой выгоды — конкретный пример или результат

### 4. Социальные доказательства
- [ ] Отзывы (реальные или от бета-тестеров)
- [ ] Логотипы компаний / фото пользователей

### 5. Блок «FAQ»
- [ ] 3–5 ответов на главные возражения

### 6. Финальный CTA
- [ ] Повтор призыва к действию
- [ ] Форма захвата / кнопка оплаты
```

### Step 5.2: No-code мобильное приложение (Glide / Adalo)

**Когда использовать:** Нужно проверить конкретные функции мобильного приложения с реальной логикой и данными.

**Пример на Glide:**
```markdown
## Приложение на Glide: «Подборщик репетиторов»

### Источник данных: Google Sheets
Лист 1: Репетиторы (ФИО, фото, предмет, цена, рейтинг, специализация)
Лист 2: Отзывы (репетитор_id, текст, оценка)

### Экраны в Glide:
1. Главная — список репетиторов с фильтром по предмету
2. Карточка — детальная страница репетитора
3. Форма — запись на пробный урок (отправляет данные в Google Sheet)
4. Подтверждение — «Вы записаны!»

### Что проверяем:
- Находит ли пользователь нужного репетитора за ≤3 минуты?
- Нажимает ли «Записаться» после просмотра карточки?
- Возвращается ли в приложение после первого урока?
```

## 📝 Phase 6: Проверка каждой части User Flow (1 час)

### Step 6.1: Пройди прототип сам — «Smoke Test»

Перед показом пользователям обязательно пройди прототип сам, имитируя незнакомого человека.

**Чеклист проверки каждого экрана:**

```markdown
## Smoke Test прототипа

### Для каждого экрана проверь:
□ Понятно ли, что здесь нужно делать, без подсказок?
□ Очевидно ли, куда нажать, чтобы продвинуться?
□ Нет ли «тупиков» (экранов, откуда невозможно вернуться)?
□ Есть ли ключевой CTA на каждом экране?
□ Понятна ли ценность действия («зачем мне это делать?»)?

### Критические переходы (проверить отдельно):
□ Первый экран → понятно ли, что это за продукт?
□ Ключевое действие → очевидна ли кнопка/форма?
□ Момент оплаты/записи → нет ли лишних шагов?
□ Финальный экран → понятен ли следующий шаг?
```

### Step 6.2: Составь сценарии для тестирования

Перед пользовательским тестированием (7.3) подготовь конкретные задания для пользователей.

**Шаблон сценария:**

```markdown
## Сценарий тестирования прототипа

### Задание 1: Найти подходящего репетитора
«Представь, что твоему сыну нужен репетитор по математике для подготовки к ОГЭ. 
Попробуй найти подходящего репетитора в этом приложении.»

**Что наблюдаем:**
- Нашёл ли квиз? (нажал «Подобрать»?)
- Завершил ли квиз?
- Открыл ли карточку репетитора?
- Время выполнения задания?

### Задание 2: Записаться на пробный урок
«Теперь запишись на пробный урок к выбранному репетитору.»

**Что наблюдаем:**
- Нашёл ли кнопку «Записаться»?
- Понял ли форму записи?
- Завершил ли заполнение?
- Были ли затруднения?

### Задание 3: Оценить решение
После выполнения заданий:
«Как бы ты оценил удобство приложения по шкале 1–10?»
«Что было непонятно или неудобно?»
«Что понравилось?»
```

## ✅ Success Criteria

✅ Создан прототип для **каждой гипотезы решения с высоким приоритетом** (минимум одна)  
✅ Прототип покрывает **весь user flow** от входа до получения ценности  
✅ Пройден **smoke test** — командой или автором, все тупики устранены  
✅ Подготовлены **сценарии для пользовательского тестирования** (минимум 2 задания)  
✅ Определены **5–8 пользователей для тестирования** из целевого сегмента  
✅ Понятно, **какую метрику измеряем** во время теста (конверсия / время / NPS)

**Context Ripeness на этом шаге:** выполнение всех пунктов даёт ~40% от стадии. Готов к переходу на 7.3.

## 🚨 Troubleshooting

### Problem 1: «Прототип получается слишком сложным»

**Solution:**  
Возможно, ты пытаешься спроектировать весь продукт сразу.  
1. Вернись к карточкам гипотез (7.1) — прототип должен проверять ОДНУ конкретную гипотезу
2. Выбери минимальный user flow: только те экраны, которые нужны для проверки этой гипотезы
3. Срежь всё второстепенное — в прототипе не нужны профиль пользователя, настройки, история и т.д.
4. Правило: если экран не участвует в проверке гипотезы — его не нужно в прототипе

### Problem 2: «Нет навыков дизайна — не могу сделать нормальный прототип»

**Solution:**  
Хорошая новость: прототип не должен быть красивым.  
1. Бумага + ручка — это полноценный прототип для первых тестов
2. В Figma используй только прямоугольники и текст — этого достаточно
3. Balsamiq специально создан так, что wireframe выглядит «нарисованным» — это снижает ожидания пользователей и они дают более честную обратную связь
4. Пользователи тестируют логику и навигацию, а не дизайн

### Problem 3: «Не знаю, какой тип прототипа выбрать»

**Solution:**  
Задай себе три вопроса:  
1. Сколько у меня времени? Если < 2 дней → бумага или wireframe. Если 1 неделя → кликабельный или no-code
2. Что я хочу проверить? Навигацию/UX → Figma. Логику/алгоритм → консьерж. Спрос/конверсию → лендинг
3. Есть ли разработчик в команде? Если нет → no-code или консьерж. Если есть → можно сделать рабочий прототип

### Problem 4: «Пользователи говорят, что им «всё понятно» — но на деле путаются»

**Solution:**  
Не спрашивай «Вам понятно?» — всегда отвечают «Да».  
1. Используй метод Think Aloud: «Расскажи, что ты думаешь, пока пользуешься приложением»
2. Наблюдай за поведением, а не за словами: куда тыкает мышью, на что обращает внимание
3. Задавай задания, а не вопросы: «Попробуй записаться к репетитору» вместо «Вам понятно, как записаться?»
4. Фиксируй: где паузы? Где переспрашивает? Где кликает не туда? Это и есть проблемы

### Problem 5: «Нет пользователей для тестирования»

**Solution:**  
1. Вернись к базе интервью из предыдущих стадий — они уже знакомы с темой
2. Закрытые сообщества (Telegram-группы, форумы, Facebook-группы) по теме продукта
3. Знакомые, которые точно попадают в целевую аудиторию
4. Сервисы для рекрутинга: Яндекс.Толока, Respondent.io, User Interviews
5. Минимально достаточно 5 человек — Якоб Нильсен доказал, что 5 пользователей выявляют ~85% проблем UX

### Problem 6: «Мне кажется, консьерж — это мошенничество»

**Solution:**  
Это нормальное ощущение у новичков. Консьерж — не мошенничество, это исследовательский метод:  
1. Ты не берёшь деньги за автоматику, которой нет
2. Ты исследуешь реальное поведение пользователей в реальных условиях
3. После теста важно раскрыть правду участникам (если использовал «Волшебника Оз»)
4. Этот метод использовали Zappos, Airbnb, Amazon при запуске

## ❓ FAQ

### Q: Сколько пользователей нужно для тестирования прототипа?

**A:** Минимум 5–8. Якоб Нильсен из Nielsen Norman Group доказал, что 5 пользователей выявляют ~85% проблем юзабилити. Для проверки количественных гипотез (конверсии) нужно больше — зависит от показателя. Для качественного тестирования (понятность, ощущения) — 5 человек достаточно для первой итерации.

### Q: Нужно ли делать отдельный прототип для каждой гипотезы?

**A:** Зависит от гипотез. Если гипотезы тестируют разные части одного user flow — их можно включить в один прототип (но запустить A/B тест между версиями). Если гипотезы кардинально разные решения одной и той же проблемы — делай отдельные прототипы.

### Q: Можно ли использовать конкурентский продукт как прототип?

**A:** Да, это тип «Чужой продукт» в PAF. Подходит, если существующий продукт частично реализует нужную функцию. Плюсы: быстро и дёшево. Минусы: трудно изолировать реакцию именно на твою гипотезу.

### Q: Как долго должен существовать прототип консьержа?

**A:** Достаточно долго, чтобы набрать репрезентативную выборку — обычно 2–4 недели, 30–50 пользователей. Если данные перестали давать новые инсайты (последние 5 пользователей не принесли ничего нового) — можно останавливаться.

### Q: Нужен ли дизайнер для создания прототипа?

**A:** Для бумажного скетча и wireframe — нет. Для кликабельного в Figma — желательно, но не обязательно. Любой PM может освоить базовый уровень Figma за 2–4 часа. No-code прототип (Tilda, Glide) вообще не требует навыков дизайна.

### Q: Что если прототип «не работает» — пользователи путаются?

**A:** Это отличный результат! Это именно то, зачем ты делал прототип — найти проблемы до разработки. Зафиксируй все места, где пользователи путались, итерируй прототип и тестируй снова. Обычно достаточно 2–3 итераций, чтобы устранить основные проблемы.
