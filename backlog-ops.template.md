# Операционный штаб po-helper (Backlog.md)

> **Backlog.md — не источник истины.** Истина живёт там, где артефакт: финальный БФТ — в `bft_documentation/<epic>/`, OKR/спринты — в путях планирования (см. `.claude/domain-profile.md`), контекст продукта — в `GROUND/`, задачи/эпики — в трекере, документы — в wiki.
>
> Backlog.md — **операционный штаб**: лёгкая доска, чтобы за 10 секунд восстановить *над какими артефактами мы начали работу и на какой стадии она оборвалась*. Он хранит **указатель и статус**, а не содержимое.

Этот файл — конвенция для агента и PO. Инициализируется `install.sh`; читается локально рядом с доской (`backlog/`).

---

## Модель: 1 задача = 1 артефакт

| Сущность Backlog | Значение в штабе |
|:---|:---|
| **Задача** | Один артефакт в работе (эпик БФТ, квартальный OKR, спринт, внешний запрос, релиз) |
| **label** | Тип артефакта: `bft` · `okr` · `sprint` · `request` · `release` |
| **Acceptance Criteria (чек-лист)** | Стадии пайплайна. Отмеченный AC = пройденная стадия → видно, **на чём оборвалась работа** |
| **status** | Крупная фаза: `To Do` (в очереди) · `In Progress` (в работе) · `Done` (отгружено) |
| **Описание** | Ссылка на реальный артефакт: код эпика, тикер трекера, путь файла. Без дублирования содержимого |

Обзор штаба:
- `backlog board` — канбан по статусам («чем занимались»).
- `backlog task list --plain` — список; `-l bft` — фильтр по типу.
- `backlog task view <id> --plain` — карточка артефакта с чек-листом стадий.

---

## Стадии пайплайнов (готовые чек-листы AC)

- **bft** — `value` · `context-gen` · `ext-teams` · `problem` · `concept` · `debate` · `constraints` · `draft` · `validate` · `deliver`
- **okr** — `context-gen` · `landscape` · `objectives` · `key-results` · `debate` · `enrich` · `validate` · `deliver`
- **sprint** — `sync` · `goal` · `decompose` · `load` · `deliver`
- **request** — `context` · `interview` · `impact-map` · `gap` · `smart` · `score` · `handoff`
- **release** — `frame` · `baseline` · `sync` (цикл) · `gate` · `status`

## Когда обновлять доску (на STOP-паузах пайплайна)

1. **Старт пайплайна** (`/*-context-gen`, `/req-context`, `/sprint-sync`, `/release-frame`) — создать задачу:
   ```bash
   backlog task create "БФТ EPIC-10 — <название>" -l bft -s "In Progress" \
     -d "Артефакт: bft_documentation/EPIC-10/ · трекер: PROJ-101" \
     --ac value --ac context-gen --ac ext-teams --ac problem --ac concept --ac debate --ac constraints --ac draft --ac validate --ac deliver
   ```
2. **Прошли стадию** — отметить соответствующий AC:
   ```bash
   backlog task edit task-1 --check-ac 1        # пройден value (AC 2 = context-gen и т.д.)
   ```
   Вернулись на шаг назад (дебаты забраковали / валидация 🔴) — снять галочку: `--uncheck-ac <n>`.
3. **Отгрузка** (`/*-deliver`, `/req-handoff`) — отметить последний AC и `-s "Done"`.
4. **Работа встала на паузу** — оставить `In Progress`, дописать в описание одну строку «остановились на …/ждём …», чтобы вернуться в контекст.

MCP-сервер `backlog` доступен в проекте — те же операции можно делать инструментами `mcp__backlog__*` без ручного CLI.

## Принципы

- **Штаб отражает, а не решает.** Отгрузка и валидность артефакта определяются пайплайном (hard gates, Светофор), не статусом на доске.
- **Одна задача — один артефакт.** Не заводи под-задачи на каждую стадию; стадии — это AC внутри задачи.
- **Не копируй содержимое.** В описании — только ссылки/коды/пути. Содержание живёт в источнике истины.
- **Штаб можно пересобрать.** Потеряли доску — не страшно: истина в артефактах. Штаб восстанавливается из них.
