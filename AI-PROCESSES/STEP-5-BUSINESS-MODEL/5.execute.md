# 5.execute — EXECUTE

> Шаг **Step 5 — Business Model** · фаза цикла Product Sprint · Линза: **Business** · **ТЯЖЁЛАЯ фаза** · Инструкция для Инженера по развитию бизнеса / Bizdev Architect (PAF [S1],[S2]).
> Референс метода: [[TRADITIONAL/RB-STEP-5-BUSINESS-MODEL-VALIDATION/5.5 Юнит-экономика и моделирование]]. Референс «почему»: [[AI-TRANSFORMATION/RB-STEP-5-BUSINESS-MODEL-VALIDATION/5.5 instruction]] · [[reformation]] §5.

## 🎯 Цель фазы для Step 5

**EXECUTE** в Business Lens = не разработка, а **финмоделирование** — перевод гипотезы БМ в систему взаимосвязанных метрик с **NPV как вершиной** [5.5]. Ключевые сдвиги PAF в этой фазе:

1. **NPV — вершина метрик**, а не LTV/CAC как самоцель [reformation §5.0]; NPV>0 на реалистичном сценарии (P50) — решающий критерий, LTV/CAC≥3 — лишь необходимое [[fit-points]];
2. **Monte-Carlo по распределениям заменяет ручной sensitivity ±50%** [5.5 Phase 3]: каждый параметр (CAC, ARPU, Churn, COGS, AI-COGS, Fixed Costs) — распределение; N прогонов → распределение NPV (P10/P50/P90), месяца breakeven, LTV/CAC;
3. **Ранжирование Рычагов (Lever) по эластичности NPV** (∂NPV/∂параметр) [5.5] — точки максимального рычага инвестиций, кандидаты в бизнес-Банч (HARVEST);
4. **AI-COGS включён в Monte-Carlo как параметр с распределением**; стресс usage ×2/×5 (из PITCH) отражается в хвосте распределения NPV [5.4, 5.5];
5. **Прямая и обратная задача** [5.5 Phase 0]: Forward (входные метрики → NPV) и Inverse (целевой NPV → какие должны быть входные).

Цель EXECUTE — узел `unit-economics` с распределением NPV, ранжированием рычагов и детектом «убийц экономики», готовый к выбору бизнес-Банча в HARVEST.

**Ключевые метрики юнит-экономики** [5.5] (каждая — узел-параметр с распределением):
| Метрика | Формула | Смысл |
|---|---|---|
| **CAC** | (Маркетинг + Продажи за период) / Новые клиенты | Стоимость «покупки» одного клиента |
| **ARPU** | Выручка за период / Пользователи | Средний доход с пользователя |
| **LTV** | ARPU / Churn | Доход за всё время жизни клиента |
| **LTV/CAC** | LTV / CAC | Коэффициент окупаемости (здоровое ≥3) |
| **Payback Period** | CAC / (ARPU × Gross Margin) | Срок окупаемости клиента |
| **NPV** (вершина) | Σ (CF_t / (1+r)^t) − I | Возврат монетизируемой ценности на инвестиции |

**Ключевые формулы PAF-сдвига:** LTV/CAC≥3 — необходимое; **NPV>0 на P50 — решающее** [reformation §5.0]. Monte-Carlo: каждый параметр → распределение (нормальное/логнормальное/треугольное — выбирает **человек**); N прогонов → распределение NPV, месяца breakeven, LTV/CAC. 3 сценария [5.5 Phase 4] = квантили **P10 / P50 / P90** (не точечные оптимист/реалист/пессимист).

**Прямая и обратная задача** [5.5 Phase 0]:
- **Forward (прямая):** входные метрики (CAC, ARPU, Churn, COGS, AI-COGS) → выходные (LTV, LTV/CAC, Payback, NPV, breakeven-месяц). Отвечает: «сойдётся ли при наших допущениях?».
- **Inverse (обратная):** целевой NPV>0 / LTV-CAC≥3 → какие должны быть входные метрики. Отвечает: «какие значения нам нужно достичь, чтобы экономика сошлась?». Обратная задача — основа Growth Forecast в HARVEST.

**Ранжирование Рычагов по эластичности NPV** [5.5]: для каждого параметра считается ∂NPV/∂параметр. Топ-Рычаги (максимальная эластичность) = точки максимального рычага инвестиций — кандидаты в бизнес-Банч (HARVEST). Типичные Рычаги: ARPU (цена), Churn (удержание), CAC (привлечение), conversion воронки, **AI-COGS** (tokens-per-user).

**«Убийцы экономики»** [5.5] — параметры, малое ухудшение которых обращает NPV в отрицательный. Пример: churn вырос с 7% до 12% → NPV уходит в минус → удержание = критический риск, красная черта для мониторинга. AI-COGS при usage ×5 — частый «убийца» для AI-продуктов без кэширования.

## 👤 Роль

- **Инженер по развитию бизнеса / Bizdev Architect** — владелец узла `unit-economics`; NPV как вершина; **NPV-моделирование — зона человека** [reformation §5.9; S2] III.4: выбор ставки дисконтирования (WACC/альтернатива), допущений распределений (нормальное/логнормальное/треугольное), интерпретация результата; veto при NPV<0 на P50.
- **Growth Engineer** — mNSM в NPV-модель; AI-COGS как входной параметр с распределением (cost-per-token снижается по Курцвейлу [S2] III.5, tokens-per-user — гипотеза).
- **Product Ops** — настройка NPV-модели (Monte-Carlo), RAG по ставкам дисконтирования/бенчмаркам LTV-CAC, event-driven пересчёт NPV при обновлении узлов 5.1–5.4.

## 🤖 Кортекс-агент

**NPV-моделировщик (Monte-Carlo)** (5.5):
- Строит финмодель: входные параметры → метрики (LTV = ARPU/Churn, LTV/CAC, Payback = CAC/(ARPU×Gross Margin)) → P&L по месяцам → NPV [формулы 5.5 Phase 1–2].
- **Monte-Carlo** по распределениям вместо ручного sensitivity ±50% [5.5 Phase 3]; N прогонов → распределение NPV, месяца breakeven, LTV/CAC.
- **Ранжирование Рычагов по эластичности NPV** — для каждого параметра считает ∂NPV/∂параметр; ранжирует: Рычаг = свойство БМ, рост NPV при инвестиции [naming_conventions].
- **Детект «убийц экономики»** — параметры, при ухудшении которых на ±X% NPV становится отрицательным (аналог «красной черты» 5.6, но на уровне эластичности).
- **3 сценария** [5.5 Phase 4] — теперь не точки, а квантили распределения NPV: **P10 / P50 / P90**.
- **Прямая/обратная задача** [5.5 Phase 0].
- **Sanity-чек** — входные параметры трассируются до узлов 5.1–5.4; бенчмарки (LTV/CAC≥3) — из RAG; маркировка факт/гипотеза/бенчмарк.

## 🧠 Человек (метапознание) — КРИТИЧНО ДЛЯ EXECUTE

- **NPV-моделирование — зона человека, агенты = инструменты, не акторы** [reformation §5.9, §VIII; S2] III.4. Утрата стратегического контроля при делегировании NPV-моделирования ИИ — **сквозной риск трансформации**. Кортекс считает, человек верифицирует и решает.
- **Отличить «оптимистичный Excel»** (главный workslop-риск Step 5 [reformation §5.9]): модель, где все параметры согласованы слишком гладко и NPV положителен при любых сценариях — красный флаг; человек проверяет, не подогнаны ли допущения под желаемый результат. Каскадная ошибка: ошибка в одном параметре размножается через формулы в breakeven — Monte-Carlo её вскрывает, ручная точечная модель маскирует.
- **Выбор распределений и ставки дисконтирования:** критическое метапознание — НЕ делегируется ИИ. Ставка обоснована (WACC/альтернатива), а не «взяли 15%».
- **Интерпретация NPV:** NPV>0 — необходимое, но не достаточное; человек оценивает стратегический риск (зависимость от одного канала, платформенный риск) — не видный в NPV.
- **Veto:** признание NPV<0 на P50 → осознанный отказ или пересмотр БМ (возврат к 5.1–5.4), а не «подкрутить параметры».

## 📦 Артефакт = узлы Нексуса системы роста (+ Context Ripeness)

- `unit-economics` (агрегирующий) с распределением NPV (P10/P50/P90), CP. CR ≥ 65%.
- Дочерние узлы-параметры (CAC, ARPU, Churn, COGS, **AI-COGS**, Fixed Costs), каждый с распределением, источником, CP, факт/гипотеза/бенчмарк.
- `levers-ranking` — ранжирование Рычагов по эластичности NPV.
- `economy-killers` — параметры, обращающие NPV в отрицательный при ухудшении.
- `npv-scenarios` — P10/P50/P90 с месяцем breakeven и LTV/CAC.

## 🚦 CP-гейт

**CP-gate юнит-экономики (5.5) — второй из трёх CP-gate стадии [reformation §5.7]:**
{LTV/CAC ≥ 3} **∧** **{NPV > 0 на реалистичном сценарии (P50)}** ∧ {Рычаги ранжированы по эластичности NPV} ∧ {«убийцы экономики» идентифицированы} ∧ **{AI-COGS включён в модель как параметр с распределением}** ∧ {CR ≥ 65%}.

**Ключевой сдвиг PAF:** LTV/CAC≥3 — необходимое, но NPV>0 — решающее (LTV/CAC как самоцель — разрыв классики [reformation §5.0]). При NPV≤0 на P50 — осознанный отказ или пересмотр БМ.

## 🛡️ Анти-workslop

- **Workslop в финмоделях (главный риск стадии) [reformation §5.9]:** «убедительные, но неверные» числа; каскадная ошибка. Гейт: каждый параметр — трассировка до узла-источника; Monte-Carlo вскрывает чувствительность, ручная точечная модель маскирует.
- **Галлюцинации цифр:** бенчмарки (LTV/CAC≥3, отраслевой Churn) — только из RAG с источником; ставка дисконтирования — обоснована, не «взяли 15%».
- **NPV как чёрный ящик:** формула NPV и ставка раскрываются; входные параметры видны и верифицируемы.
- **COGS-эксплозия в NPV:** AI-COGS включён в Monte-Carlo как параметр с распределением; стресс usage ×2/×5 отражён в хвосте NPV.
- **Ложное ощущение готовности:** NPV из симуляции ≠ a posteriori; реальная экономика подтверждается только в Stage 6–8 (Soft Launch, когортный CAC/LTV).
- **Дофаминовая петля ИИ [S2] III.7:** «красивая» модель с положительным NPV приятнее реалистичной с NPV≈0; человек обязан сопротивляться.

## 🔭 Связь с Тремя Линзами (3PL)

**Business Lens — ядро EXECUTE.** NPV — вершина метрик Business Lens; Рычаги — операциональная единица Business Sprint (Рычаги/Lever → NPV [S1] VI.1). Каждый Рычаг из ранжирования — кандидат в бизнес-Банч (HARVEST). Связь:
- **Product Lens** — через ARPU/mNSM (продуктовая ценность конвертируется в деньги); через AI-фичи (tokens-per-user как продуктовый параметр AI-COGS);
- **Strategy Lens** — через NPV как критерий инвестиционного решения портфеля (Portfolio Manager); через ставку дисконтирования (WACC = альтернативная стоимость капитала портфеля).

## 🧠 Зона человека (критично для EXECUTE) — повтор и углубление

**NPV-моделирование — зона человека, агенты = инструменты, не акторы** [reformation §5.9, §VIII; S2] III.4. Это **самый чувствительный veto-пункт всей трансформации**: именно здесь соблазн делегировать ИИ «посчитать, окупится ли» максимален, а цена ошибки (масштабирование убыточной БМ) — высока. Конкретно человек делает:
1. **Выбор ставки дисконтирования** (WACC / альтернативная стоимость / требуемая доходность) — обоснованно, не «взяли 15%»;
2. **Выбор распределений** параметров (нормальное для симметричных ошибок, логнормальное для цен/COGS, треугольное для бенчмарков);
3. **Интерпретацию P50/P10/P90** — какой сценарий считать «реалистичным»;
4. **Veto при NPV<0 на P50** — осознанный отказ или пересмотр БМ, а не «подкрутить».

Junior без метапознания + ИИ = нулевой прирост качества финмодели [S2] III.4.
