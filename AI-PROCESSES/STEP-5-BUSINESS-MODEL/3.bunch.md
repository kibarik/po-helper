# 3.bunch — BUNCH

> Шаг **Step 5 — Business Model** · фаза цикла Product Sprint · Линза: **Business** · **ТЯЖЁЛАЯ фаза** · Инструкция для Инженера по развитию бизнеса / Bizdev Architect (PAF [S1],[S2]).
> Референс методов: [[TRADITIONAL/RB-STEP-5-BUSINESS-MODEL-VALIDATION/5.1 Формирование гипотезы бизнес-модели]] · [[5.2 Схема монетизации]] · [[5.3 Структура воронки потребителя]]. Референс «почему»: [[AI-TRANSFORMATION/RB-STEP-5-BUSINESS-MODEL-VALIDATION/5.1 instruction]] · [[5.2 instruction]] · [[5.3 instruction]] · [[reformation]] §5.

## 🎯 Цель фазы для Step 5

**BUNCH** [S1] VI.5, VI.6 — сформировать из текущего состояния Нексуса (кандидаты-паттерны из SCOUT) **гипотезу бизнес-модели** и свернуть её в **бизнес-Банч рычагов** (Lever): связку 2–3 измеримых свойств Б-модели, рост NPV при инвестиции [naming_conventions]. BUNCH Step 5 = агрегация подшагов 5.1–5.3:

- **5.1 — Гипотеза БМ:** BMC Остервальдера (9 блоков) + выбор базового паттерна из 6 + Lean Canvas fallback на null base [5.1];
- **5.2 — Монетизация из mNSM:** 5 измерений (тип платежа / предмет / плательщик / ценообразование / момент) [5.2] + ценовой коридор из 3 методов (cost-plus / value-based / competitor-based);
- **5.3 — Воронка AARRR:** Acquisition → Activation → Retention → Referral → Revenue + Channel Mix + Aha-moment + модель оттока [5.3].

**5 измерений монетизации** [5.2] — каждое становится дочерним узлом:
| Измерение | Вопрос | Варианты |
|---|---|---|
| Тип платежа | Разовый или регулярный? | Разовый / Регулярный / Гибрид |
| Предмет оплаты | За что платят? | Транзакция / Доступ / Ресурсы / Результат / Без монетизации |
| Плательщик | Кто платит? | Потребитель / Поставщик / Обе стороны / Третья сторона |
| Ценообразование | Сколько? | Фикс / % / Tiered / Freemium |
| Момент оплаты | Когда? | До ценности / В момент / После / По результату |

**Воронка AARRR** [5.3] — конверсии как **распределения** (не точки):
`Awareness → Acquisition → Sign Up → Activation (Aha-moment) → Conversion/Revenue → Retention → Referral` — на каждом ребре распределение вероятностей из бенчмарков/ранних данных; Channel Mix по правилу трёх горизонтов (сейчас 0–3 мес / скоро 3–12 мес / потом 12+ мес).

**3 метода ценового коридора** [5.2 Phase 4] — пересечение = целевая ценовая точка:
| Метод | Формула | Источник |
|---|---|---|
| **Cost-plus** | Себестоимость + Маржа (вверх от COGS из 5.4) | Прайс-листы |
| **Value-based** | % от дельты ценности для клиента (из 4.x) | Интервью Stage 2–3 |
| **Competitor-based** | Диапазон цен конкурентов | Нексус рынка |

**Channel Mix** [5.3 Phase 1]: платные (контекст, таргет, инфлюенсеры) / органические (SEO, контент, SMM) / вирусные (рефералы, network effects) / партнёрские / прямые продажи. Оценка по 3 критериям: доступность, масштабируемость, стоимость (CAC по каналу).

**Модель оттока** [5.3 Phase 4]: три группы причин — Product Churn (продукт не даёт ценности), Circumstantial Churn (жизненная ситуация клиента), Competitive Churn (ушёл к конкуренту). Ранние сигналы оттока (early warning) → автоматические действия (push, пауза вместо отмены). Бенчмарки: B2C SaaS <5%/мес, B2B SaaS <2%/мес.

**Ключевой сдвиг PAF:** бизнес-Банч — **не матрица конфигураций** [reformation §5.3 разрыв 4], а связка рычагов в моменте под Bunch Size (скорость восприятия рынка) и Bunch Window (период ожидания бизнес-результата) [S1] VI.6. Каждый рычаг — кандидат с известной (позже, в EXECUTE) эластичностью NPV. Монетизация **выводится из mNSM**, а не из абстрактного «как зарабатывать» [5.2 instruction].

## 👤 Роль

- **Инженер по развитию бизнеса / Bizdev Architect** — владелец узлов `bm-hypothesis`, `monetization`; выбор базового паттерна и модификаций (veto, зона человека [S2] III.4); формирование бизнес-Банча из рычагов.
- **Growth Engineer** — владелец узла `funnel`; определение «юнита оплаты» (1 урок / 1 месяц / 1 сделка) и его связи с mNSM; time-to-activation; Channel Mix.
- **Product Ops** — настройка A/B-симулятора цен, симулятора AARRR, churn-предиктора; RAG по WTP/ценам.

**BMC Остервальдера — 9 блоков** [5.1 Phase 3], каждый становится дочерним узлом с CP:
1. Value Proposition (из Step 4) · 2. Customer Segments (из Step 2–3) · 3. Channels · 4. Customer Relationships · 5. Revenue Streams (связь с mNSM) · 6. Key Resources · 7. Key Activities · 8. Key Partners · 9. Cost Structure (связь с 5.4).

**Lean Canvas** (Эш Маурья, fallback на null base [5.1 Phase 4]): Проблема / Решение / UVP / Unfair Advantage / Сегменты / Метрики / Каналы / Затраты / Доходы — фокус на рисках и допущениях стартапа.

## 🤖 Кортекс-агенты

- **BMC-генератор / Pattern Clusterer** (5.1) — заполняет 9 блоков BMC из узлов конкурентов; кластеризация по 6 паттернам; предлагает базовый паттерн + модификации.
- **Проектировщик монетизации + A/B-симулятор цен + NPV-оценщик схем** (5.2) — из mNSM и `bm-hypothesis` выводит 5 измерений схемы; считает ценовой коридор из 3 методов; **A/B-симулятор цен** генерирует синтетических агентов-клиентов (4-я роль «секретарь/адъютант» [S2] III.8) и симулирует конверсию при разных ценах → кривая эластичности спроса; NPV-оценщик ранжирует схемы по NPV.
- **Симулятор вероятностной воронки AARRR + churn-предиктор + Aha-Moment детектор** (5.3) — граф переходов с конверсиями как распределениями (не точечными); Channel-mix модeller по правилу трёх горизонтов; churn-предиктор по поведенческим сигналам; детектор «момента ценности».
- **Sanity-чек** — маркировка `факт/гипотеза/бенчмарк`; симулированная конверсия ≠ валидированная WTP.

## 🧠 Человек (метапознание)

- **NPV-моделирование — зона человека, агенты = инструменты** [S2] III.4, [reformation §5.9]: Кортекс симулирует/считает, человек интерпретирует и выбирает. Выбор ценового коридора и момента оплаты — стратегическое позиционирование (дорого/дёшево = сигнал качества), veto за Bizdev Architect.
- **Отличить симуляцию от данных:** A/B-симулятор цен и симулятор AARRR дают «убедительные, но потенциально неверные» цифры [Harvard × BCG, S2] III.4; симулированная конверсия маркируется как модель, не a posteriori; реальная WTP подтверждается только в Stage 6–8.
- **Маркировка факт/гипотеза/бенчмарк:** цены конкурентов — факт (URL+дата); WTP из интервью — гипотеза с цитатой; симулированная конверсия — бенчмарк/модель.
- **Проверка mNSM-связи:** каждая схема монетизации и каждый рычаг Банча связывается ребром с mNSM; схема, не монетизирующая mNSM, — отбраковывается [5.2 instruction].

## 📦 Артефакт = узлы Нексуса системы роста (+ Context Ripeness)

- `bm-hypothesis` (агрегирующий) + 9 дочерних BMC-узлов + `pattern-cluster` (выбранный базовый паттерн + отклонённые альтернативы). CR ≥ 40%.
- `monetization` + 5 дочерних узлов измерений + `price-corridor` (3 метода) + `price-elasticity-curve` (маркирована как симуляция) + `monetization-npv`. CR ≥ 50%.
- `funnel` + дочерние AARRR (распределения конверсий) + `channel-mix` + `churn-model` + `aha-moment`. CR ≥ 55%.
- **Бизнес-Банч (черновой):** 2–3 рычага-кандидата с эластичностью NPV (уточнится в EXECUTE) + Bunch Size/Window.

## 🚦 CP-гейт

**Три CP-гейта BUNCH (по подшагам):**
- **5.1:** {CP блока Revenue Streams ≥ порога} ∧ {Customer Segments ≥ порога} ∧ {базовый паттерн выбран и обоснован} ∧ {CR `bm-hypothesis` ≥ 40%}.
- **5.2:** {тип/предмет/плательщик определены} ∧ {коридор из ≥2 методов} ∧ {NPV ≥1 схемы > 0 на реалистичном сценарии} ∧ {схема связана с mNSM} ∧ {CR ≥ 50%}.
- **5.3:** {этапы AARRR связаны с монетизацией} ∧ {Channel Mix по ≥2 каналам с CAC} ∧ {распределения конверсий, не точки} ∧ {модель оттока с сигналами} ∧ {CR ≥ 55%}.

## 🛡️ Анти-workslop

- **Workslop в BMC/воронке/цене:** «картинка воронки» без конверсий; «убедительный» ценовой коридор без метода; BMC с блоками без источника. Гейт: каждый узел — через метод с источником.
- **Галлюцинации цифр** (главный workslop Step 5): бенчмарки конверсий (например, «15–25% визит→регистрация») — только из RAG (отраслевые, teardown) с URL+датой; иначе «гипотеза», CP=1.
- **NPV как чёрный ящик:** формула NPV схемы раскрывается (ARPU × Lifetime − CAC − COGS, ставка дисконтирования); входные параметры трассируются до узлов 5.1/4.x.
- **Бизнес-Банч ≠ матрица конфигураций:** 2–3 рычага в моменте, а не «6–9 конфигураций в таблице» [reformation §5.3 разрыв 4]; бизнес-Банч — не Беклог [naming_conventions].

## 🔭 Связь с Тремя Линзами (3PL)

**Business Lens — ядро BUNCH.** Монетизация — прямой **Рычаг (Lever)**: цена (ARPU) и тип платежа — измеримые свойства БМ, рост NPV при инвестиции [5.2 instruction]. Воронка — источник Рычагов: конверсия на каждом этапе AARRR — измеримое свойство, рост NPV при улучшении [5.3 instruction]. Связь:
- **Product Lens** — через mNSM (схема монетизирует именно mNSM) и Aha-Moment (продуктовое «момент ценности»);
- **Strategy Lens** — через позиционирование цены (дорого/дёшево = ставка портфеля) и Channel Mix как ставку на каналы дистрибуции.

## 🧠 Ключевые veto-точки человека в BUNCH

1. **Выбор базового паттерна БМ и модификаций** — стратегическое распределение ресурсов [S2] III.4; Кортекс кластеризует, человек выбирает.
2. **Выбор ценового коридора и момента оплаты** — позиционирование (дорого/дёшево = сигнал качества); veto за Bizdev Architect.
3. **Выбор приоритетного этапа воронки** для инвестиций («латаем Acquisition или Retention») — за Growth Engineer на основе эластичности NPV (из EXECUTE), а не «где конверсия ниже».
4. **Проверка mNSM-связи** — каждая схема и каждый рычаг Банча связываются ребром с mNSM; схема, не монетизирующая mNSM, отбраковывается.
