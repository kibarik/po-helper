# Подшаг 0.2 — Cortex Setup: настройка Кортекса над Нексусами

> Второй подшаг бутстрапа. Кортекс настраивается **поверх** готовых Нексусов из подшага 0.1. Порядок обязателен: сначала Нексус, потом Кортекс [S1] IX.
> Источник: [[operating-model]] столп 2; [S1] III.1, Часть VII; [S2] III.4.

---

## 🎯 Цель

Поднять **Кортекс** — стандартизированную систему автоматизации всех процессов (агенты / RAG / интеграции / промпты) [S1] III.1 — до начального рабочего состояния. На Step 0 Кортекс стартует на **фазе зрелости 1–2** модели [S1] Часть VII: от ассистента (промпт-шаблоны) к проводнику команд (RAG по Нексусам + спец. агенты). Фаза 3 (сквозной event-based Кортекс всей компании) — цель на будущее, не на Step 0.

**Ключевое правило, вводимое здесь:** инженер работает **только через проводника — Кортекс**, не вручную. Ставит задачу → Кортекс манипулирует Нексусами [S1] III.1.

---

## 👤 Роль и ответственность

**Владелец — Product Ops / Product Architect.** Миссия этой роли — **двигать команду по модели зрелости Кортекса** [S1] Часть VII, [S2] III.6. На Step 0 Product Ops:

- собирает Кортекс из 4 компонентов (агенты / RAG / промпты / интеграции);
- выбирает **начальный набор агентов** под шаги 1–8;
- договаривается с Product Engineer о **AI workflow** (что делегировать агентам, что себе) [S2] III.6.

Product Engineer — **главный пользователь** Кортекса; определяет, какие задачи автоматизировать / аугментировать / агентизировать.

## 🧩 Структура: 4 компонента Кортекса + 3 оптимизации

### Четыре компонента Кортекса [S1] III.1

| Компонент | Что это | На Step 0 |
|---|---|---|
| **Промпты** | Шаблоны под типовые задачи (генерация, скоринг, разбор) | Базовый набор: разбор интервью, скоринг CP, draft Узла Нексуса |
| **RAG** | Поиск по Нексусам (подключение из 0.1) | Готов: индекс по 4 Нексусам |
| **Агенты** | Спец. роли с зоной ответственности | Начальный набор (см. ниже) — скелеты, развиваются в шагах 1–8 |
| **Интеграции** | Связи с внешними системами (аналитика, CRM, репозитории) | Минимум: подключение источников для `источник[]` Узлов |

### Три оптимизации роли [S2] III.4

Кортекс настраивается под три режима усиления инженера:

| Оптимизация | Суть | Пример на Step 0 |
|---|---|---|
| **Автоматизация** | Сокращение срока задач (скорость) | Nexus-Builder проставляет `timestamp`, статус увядания |
| **Аугментация** | Рост качества: альтернативная точка зрения, челендж идей/рисков | CP-скорер предлагает альтернативные оценки доверия |
| **Агентизация** | Полное/частичное делегирование ИИ | Scouting-агент ищет возможности/угрозы по 3 Линзам (черновой) |

> ⚠️ Все три оптимизации дают прирост **только** при достаточном метапознании для валидации [S2] III.4. См. секцию «Человек» ниже.

## 🤖 Кортекс: модель зрелости и начальный набор агентов

### Модель зрелости Кортекса (3 фазы) [S1] Часть VII

| Фаза | Объект ИИзации | Статус на Step 0 |
|---|---|---|
| **1** | Отдельные задачи | ✅ Стартуем здесь: промпт-шаблоны, Кортекс = ассистент |
| **2** | Процессы команд | 🟧 Цель Step 0: RAG по Нексусам + спец. агенты, Кортекс = проводник |
| **3** | Процессы всей компании | ⏳ Будущее: сквозной event-based Кортекс, авто-обновление Нексусов, CP пересчитывается сам |

Product Ops может держать команду на фазе 1 Кортекса и фазе 2 Нексуса одновременно (уровни независимы [S1] Часть VII).

### Начальный набор агентов (скелеты — развиваются в шагах 1–8)

| Агент | Функция | Когда оживает |
|---|---|---|
| **Nexus-Builder** | Создаёт/обновляет Узлы, проставляет `источник[]`, `timestamp` | 0.1 (уже), развивается в 1–8 |
| **Wilting-детектор** | Помечает Узлы с потерей актуальности | 0.1 (уже) |
| **Scouting-агент** | Скаутинг возможностей/угроз через 3 Линзы (Understand [S1] VI.5) | Step 1+ |
| **CP-скорер** | Оценка Confidence Point Узлов/гипотез (аугментация, челендж) | Step 1+ |
| **Bunch-формер** | Генерация чернового Банча из текущего состояния Нексуса (Identify) | Step 1+ |
| **Pitching-оппонент** | «Заваливает» аргументы Банча в Pitching of Trust (аугментация) | Step 4+ |

> Эти агенты — **общие роли**, не финальные реализации. Их компетенции наращиваются по мере прохождения шагов. На Step 0 важна их **структура и привязка к Нексусам**, а не интеллект.

## 🧠 Человек (метапознание)

**Критический фильтр [S2] III.4:** рост качества от Кортекса — **только** при достаточных метапознании и компетенциях для валидации.

- Junior без метапознания + ИИ = **нулевой прирост**.
- Компетентный, но забивший на верификацию = **«убедительные, но неверные»** результаты (Harvard × BCG).

На Step 0 под принимает правило: **каждый результат агента верифицируется человеком**, трассируется до `источник[]`, человек имеет veto. Агенты — инструменты, не акторы; ответственность не передать [S2] III.4.

**Паттерны работы с ИИ** [S2] III.4 (под выбирает осознанно): **Кентавры** (делят/делегируют часть ИИ) или **Киборги** (интегрируют поток задач с ИИ).

## 🛡️ Анти-workslop

- **Не «Кортекс = один большой промпт»** → Кортекс = агенты + RAG + интеграции + промпты [S1] IX.
- **Не доверять агентам без верификации** → метапознание как фильтр [S2] III.4.
- **Не агентизировать стратегию** → распределение ресурсов и синхронизация людей — зона человека [S2] III.4.
- **Не прятать COGS токенов** → AI-COGS (cost-per-token × tokens-per-user) учитываем с самого начала [S2] III.6 (Growth Engineer), [[operating-model]] гвардраилы.
