# 5.execute — EXECUTE / Фиксация Нексуса рынка v1.0 (Step 3: Market)

> Фаза лёгкая (на discovery-этапе) · Линза: Strategy · Discovery-EXECUTE = фиксация + передача допущений на верификацию.
> Источник: [S1] VI.5 (Und-Id-Ex) · `3.6` instruction (Commit, профиль v0.2).

---

## 🎯 Цель фазы для Step 3

В шагах 1–5 (discovery) EXECUTE — **не валидация на рынке** (это произойдёт в RB-STEP-4 Customer Validation), а **commit рыночного контекста в Нексус** и **подготовка допущений к верификации**. На выходе Pitching (GO) Нексус рынка фиксируется как **v1.0** (CR ≥ 60%, все 7 типов Узлов), а Bet-узел порождает явный список `assumptions_to_verify` — гипотез, которые проверит Customer Validation (RB-STEP-4).

PAF: «рост контекста = рост скорости процессов, в т.ч. отказа» [S1] VI.3 — поэтому EXECUTE здесь = закрепление контекста, который ускорит следующие шаги (включая осознанный отказ, если Customer Validation опровергнет допущения).

---

## 👤 Роль

**Product Engineer** — фиксирует Нексус рынка v1.0 через Кортекс, переносит `assumptions_to_verify` в Нексус потребителя (для RB-STEP-4), проверяет целостность связей между Узлами (Force→Trend→TAM-SAM-SOM→Competitor→Gap→Bet). **Product Ops** — коммитит версионирование (v0.x → v1.0), настраивает event-based обновление Узлов (после релиза Кортекс обновляет Нексус → меняется Банч → цикл [S1] базовая логика 5). **Portfolio Manager** — регистрирует GO-решение в Ставках портфеля (Goal Map), обновляет баланс маржинальности/каннибализации.

> **Зона человека [S2] III.4:** регистрация стратегического решения GO в портфеле — не делегируется ИИ.

---

## 🤖 Кортекс-агент

**Nexus-Builder Agent** (рынок) [3.0] — фаза Commit:
- Проверяет полноту: все 7 типов Узлов (Force / Trend / Constant / TAM-SAM-SOM / Competitor / Gap / Bet) представлены хотя бы одним экземпляром.
- Пересчитывает итоговый CR; помечает Нексус рынка как **v1.0**.
- Связывает Bet-узел (Strategy Lens) с Узлами других типов (links[]) — целостный подграф.
- Настраивает **event-based поддержку**: событие (новый отчёт, новость, refresh) → пересчёт CR → аларм при падении CR ниже порога (увядание).

### Lifecycle Нексуса рынка (финал Step 3)
1. Phase-0 (PULSE): CR v0.x, intent.
2. Насыщение (SCOUT): 6 типов Узлов → CR ~70%.
3. **Commit (EXECUTE)**: Bet-узлы фиксируются, Нексус рынка v1.0, CR ≥ 60%.
4. Event-based поддержка: после релиза Кортекс обновляет Нексус → цикл [S1].

## 🧠 Человек (метапознание)

- Проверяет, что Нексус рынка v1.0 — **двойник**, а не «документ с галочками»: 7 типов Узлов, у каждого `source[]`, `cp`, `timestamp`, `decay_status`, `links`.
- **Контрольные вопросы готовности** (из `3.0`):
  - Можно ли ответить без заметок: «на каком рынке работаю?», «какая область потребностей в фокусе?»
  - Все 7 типов Узлов представлены с источником?
  - CR вычисляется автоматически и падает при устаревании?
  - Каждая рыночная цифра имеет URL+дату?
- Формулирует `assumptions_to_verify` — гипотезы Bet-узла, требующие проверки в Customer Validation (RB-STEP-4). Например: «допущение SAM по правилу ФРИИ 10% → проверить willingness-to-pay сегмента Y».
- Отличает зафиксированный контекст (факт с источником) от допущения (гипотеза на верификацию).

---

## 📦 Артефакт = Нексус рынка v1.0 + enriched профиль потребителя

- **Нексус рынка v1.0** (CR ≥ 60%): целостный подграф 7 типов Узлов со связями; Bet-узел с `decision: GO` и `assumptions_to_verify`.
- **Профиль потребителя v0.2** (исходный `3.6` Phase 2): enriched рыночным контекстом — область потребностей + TAM/SAM/SOM (привязка к сегменту) + ближайшие конкуренты + конкурентная позиция + рыночные пробелы + решение GO. Проверка внутренней согласованности: «кто клиент» ↔ «для кого считался TAM»; gaps ↔ боли клиента.
- **Goal Map обновлена**: GO-решение зарегистрировано в Ставках портфеля.

---

## 🚦 CP-гейт фазы

EXECUTE — фиксирующая фаза (не gate). Условие завершения: Нексус рынка v1.0 закоммичен с CR ≥ 60%, все 7 типов Узлов представлены, `assumptions_to_verify` переданы в Нексус потребителя для RB-STEP-4. Если хотя бы один тип Узла отсутствует → возврат в SCOUT.

---

## 🛡️ Анти-workslop

- **Документ как финальный артефакт = будущий workslop** [S2] III.7. EXECUTE фиксирует **живой Нексус** с event-based обновлением, не статичный «рыночный отчёт в Obsidian».
- **Иллюзия живого Нексуса** (reformation §3.9): CR заявлен как метрика с фазами созревание/увядание, но обновление вручную → «живость» теряется. Снятие: event-based обновление только через Кортекс-агентов.
- **NPV-потенциал как чёрный ящик**: при фиксации Bet-узла — явная трассировка допущений (агент-оценщик выдал число → список допущений для верификации).
- **Увядание**: Узлы без event-based refresh теряют актуальность; Product Ops настраивает аларм при падении CR.
