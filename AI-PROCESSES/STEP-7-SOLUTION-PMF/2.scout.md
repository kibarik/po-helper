# 2.scout — SCOUT (Step 7: Solution & PMF)

> Фаза средняя · Линза: Product · **Identify-фаза Und-Id-Ex** [S1] VI.5.
> Источник: [[AI-TRANSFORMATION/RB-STEP-7-SOLUTION-VALIDATION/7.1 instruction]] · [S1] Принцип 5.

---

## 🎯 Цель фазы для Step 7

**Скаутинг возможностей/угроз** через Линзу продукта (Принцип 5 [S1] — вместо приоритизации списка): Кортекс генерирует кандидатов фич Банча из Нексуса продукта и Нексуса потребителя, AI UX Designer проектирует ИИ-метафоры взаимодействия, строятся CJM и User Flow. Объект анализа сменился: теперь это не свойства потребителя, а **его поведение внутри конкретного продукта**. Продукт = способ изменения текущего поведения потребителя.

Цель — **поднять Confidence Point гипотез решения** до уровня, достаточного для входа в вайбкодинг-прототипирование (7.2), и **насытить Нексус продукта** узлами решения. Логика PAF [S1]: в более чем половине случаев гипотезу ценности невозможно провалидировать без конкретного решения, с которым может взаимодействовать пользователь.

> Это **Bunch Saturation** (насыщение банча) — старт цикла Product Sprint [S1] VI.4. Контекст из предыдущих стадий (JTBD, персоны, поведение AS-IS) живёт в Нексусе и служит входом; гипотезы генерируются RAG по этим узлам, а не «с нуля в пустой комнате».

---

## 👤 Роль

**Product Engineer** — управляет насыщением Банча через Кортекс; задаёт целевую mNSM гипотезы решения; формулирует Гэп; проектирует цифровой профиль. **AI UX Designer** — проектирует **ИИ-метафоры** по 5 уровням [S2] III.8. **Technical/AI Product Manager** — если продукт содержит ИИ-свойства: оценивает LLM-дизайн, COGS токенов на кандидат-фичу. **Discovery/Launcher PM** — определяет минимум знаний для перехода к прототипу.

> **Зона человека** [S2] III.4: выбор фокусного JTBD и решение, какая гипотеза закрывает Гэп с максимальным CP. Стратегическое планирование не делегируется.

---

## 🤖 Кортекс-агент (из 7.1)

- **Генератор гипотез решения** — RAG по узлам `current-state`/`vision`/`gap` и Нексусу потребителя (JTBD, поведение AS-IS, боли) → кандидаты в формате **Solution Hypothesis Card**:
  - «Если [фича/UX-решение], то [ожидаемое изменение поведения], что подтвердится [метрика+порог], ведь [обоснование из исследований]»;
  - поля: CP, источник уверенности, способ проверки, критерий успеха, срок проверки;
  - трассировка до узла-источника (без источника CP=0);
  - **минимум 3 альтернативы** — конкуренция гипотез, а не монополия первой идеи (первая идея редко лучшая).
- **AI UX Designer-агент** — для каждого кандидата проектирует ИИ-метафору по 5 уровням [S2] III.8: (1) ментальная модель «намерение→черновик→уточнение»; (2) метафора «партнёр/соавтор»; (3) ошибка = альтернатива/уточнение (новый промпт); (4) ввод — естественный язык/референсы (валидация: синтаксис→семантика); (5) время — streaming + «мысли вслух». Возвращает варианты tone-of-voice. ИИ в продукт ≠ чат-бот без изменений UX.
- **CJM-билдер** [NNGroup] — Customer Journey Map из поведения AS-IS и JTBD: этапы (Осознание→Поиск→Сравнение→Запись→Получение ценности), фокус 5–7 этапов, один ключевой сценарий; строки: цели/ожидания, действия, точки контакта, мысли, эмоции 😊/😐/😠, боли, возможности; выявляет 3–5 ключевых инсайтов (самая болезненная точка, основная причина отказа, узкое место воронки, момент получения ценности).
- **User Flow-билдер** — детализирует Motivation → Stages → Actions внутри продукта; связывает с экранами; техническая схема навигации (конкретные шаги и экраны, чтобы дизайнер/разработчик нарисовал без доп. вопросов).
- **CP-скорер (колесо Гилада)** [S1] VI.7 — объективизирует «уверенность 1–10» через ICE/RICE с **оценкой стоимости риска**; расставляет кандидатов на скаутинг-карте возможностей/угроз (замена RAT-приоритизации списка [naming_conventions]).

## 🧠 Человек (метапознание — валидация ИИ-генерации гипотез)

- **Валидация каждой гипотезы** [S2] III.4: (а) проверяемость метрики с порогом; (б) обоснование из реального узла-источника; (в) различимость альтернатив (≥3 кандидата); (г) баланс конкретности («если X / то Y / что подтвердится Z на N% / ведь…»). Хотя бы один пункт отсутствует → гипотеза не готова.
- **Veto** на workslop-формулировки: «пользователь будет доволен», «удобно», «правильно» — непроверяемо. Техника «Шесть шляп»: первая идея / что делают конкуренты / что будет, если убрать самое сложное — три угла.
- **Конвергенция JTBD**: выбирает фокусный JTBD и фокусную боль на CJM как точку приложения Банча.
- **ИИ-метафора**: решение, какая метафора («партнёр» vs «инструмент») соответствует Видению — за Product Engineer + AI UX Designer. Четвёртая роль потребителя — «секретарь/адъютант» + сущность «поручение» [S2] III.8 — учитывается.

## 📦 Артефакт = узлы Нексуса продукта (+CR)

- `solution-hypothesis` — {фича/UX-решение, ожидаемое поведение, метрика+порог, обоснование, CP, источник-узел[], способ проверки, критерий успеха, срок проверки}.
- `cjm` — Customer Journey Map (этапы → цели/действия/точки контакта/мысли/эмоции/боли/возможности) + 3–5 ключевых инсайтов.
- `user-flow` — детальный поток внутри продукта (Motivation → Stages → Actions → экраны).
- `ai-metaphor` — {ментальная модель, метафора, поведение при ошибке, режим ввода, режим времени, tone-of-voice}.
- `gap` — узел Гэпа (текущее → Видение) как якорь Банча.
- Каждый узел: `источник`, `CP`, `timestamp`, `статус увядания`. **CR после 7.1 → ~20–25%**.

## 🚦 CP-гейт фазы

**Gate 7.1 → 7.2 (Understand → Identify)** [7.1]:
- CP ≥ порога по ≥3 кандидатам фич Банча;
- CJM построена для фокусного сценария (5–7 этапов);
- User Flow покрывает путь от входа до получения ценности;
- сработал CP-скорер с оценкой стоимости риска;
- сгенерированы ИИ-метафоры для каждого кандидата;
- есть понимание, с какой гипотезы начинать прототипирование.
- До порога — итерация насыщения (дособрать поведение AS-IS, дополнить JTBD).

> CP растёт по колесу Гилада [S1] VI.7; падает при срабатывании рисков/отклонениях. Чем выше CP, тем выше шанс прохождения гейта **или осознанного отказа** [S1] VI.3.

## 🛡️ Анти-workslop

- **Workslop в формулировках гипотез** (метрика без порога, обоснование «потому что логично») — главный риск 7.1. ~15,4% материалов перекладывают когнитивную нагрузку; MIT: 95% без отдачи от ИИ именно из-за workslop [S2] III.7.
- **Метапознание-фильтр**: каждый кандидат проходит проверку «можно ли это опровергнуть?». Нет → workslop. Junior без метапознания + ИИ = нулевой прирост; компетентный, но забивший на верификацию = «убедительные, но неверные» (Harvard × BCG) [S2] III.4.
- **Трассировка до Нексуса**: гипотеза без связи с узлом JTBD/боли/поведения → CP=0, в Банч не попадает. Запрещены сгенерированные «персоны» без доказательной базы.
- **Premature PMF-риск**: не путать «гипотеза решения подтверждена в голове команды» с ростом CP — CP растёт только от данных.
