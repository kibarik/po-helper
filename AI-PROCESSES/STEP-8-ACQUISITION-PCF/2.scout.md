# 2.scout.md — Скаутинг конфигураций привлечения

> Фаза **SCOUT** цикла Product Sprint на Step 8 · `[[operating-model#Цикл Product Sprint]]` [S1] Принцип 5, VI.5.

---

## 🎯 Цель фазы для Step 8

**Скаутинг вместо приоритизации** [S1] Принцип 5: нет списка задач, есть карта целей. На Step 8 скаутинг = поиск **возможностей и угроз** дистрибуции через 3 Линзы, насыщение Нексуса роста по методике **Und-Id-Ex** (Understand → Identify) [S1] VI.5.

Скаутинг переводит базовый каталог из Pulse (каналы/сегменты/офферы) в **набор гипотез конфигураций** «сегмент–канал–оффер» (метод `8.1`). Каждая конфигурация — кандидата на Банч (фаза 3).

**Understand** (Data base [S1] VI.5 — есть продукт с PMF): анализ Нексуса роста → какие каналы/сегменты из GTM недоисследованы, где угрозы (дорогой CAC, малая ёмкость).
**Identify** (Кортекс генерит идеи, рост CP): формирование 5–15 гипотез конфигураций с целевым CAC, воронкой, обоснованием.

> Data base [S1] VI.5: на Step 8 продукт есть → извлекаем знания о дистрибуции, находим бутылочные горлышки (не discovery с нуля).

---

## 👤 Роль

- **Growth Engineer / Growth PM** [S2] — ведёт скаутинг: извлекает каналы из GTM, сегменты из Customer Validation, офферы из Sales & Marketing Kit (метод `8.1` Phase 1).
- **Product Engineer** — формулирует офферы как **инкременты** (каждая конфигурация = Bale = инкремент + оффер + план дистрибуции [S1]).
- **Product Ops** — насыщает Нексус роста новыми узлами (конфигурации-кандидаты) с метаданными: источник, CP, timestamp.

---

## 🤖 Кортекс-агент

**Acquisition Bunch Agent** (`8.1 instruction`) — генерирует гипотезы конфигураций в моменте из текущего состояния Нексуса роста (Understand→Identify). На скаутинге агент:
- Извлекает **каталог каналов** (performance/органика/direct/referral — нейтрально к типу, метод `8.1`), **каталог сегментов** (демография + JTBD из Customer Validation), **каталог офферов** (из Sales & Marketing Kit).
- Комбинирует в конфигурации с 7 элементами (сегмент, канал, оффер, воронка, целевые метрики, ресурсы, обоснование — метод `8.1` структура).
- Скармливает воронку: точка входа → шаги → конверсия, с целевым CAC из правила LTV/3.

---

## 🧠 Человек (метапознание)

- Валидирует **обоснование каждой гипотезы** (метод `8.1` элемент 7): почему сегмент, почему канал, почему оффер — трассировка до интервью, GTM, аналогов. AI-сгенерированное обоснование без источника = workslop [S2] III.7.
- Проверяет **целевой CAC** (метод `8.1`): рассчитан из LTV/3, а не «из головы» (метод `8.1` Problem 3).
- Правит ошибки скаутинга: слишком broad сегменты («наша аудитория»), слишком много конфигураций (>15 — размазывается бюджет, метод `8.1` Q1).
- Veto: не пускать в Банч конфигурации без критерия успеха (конкретное число, не «будет работать»).

---

## 📦 Артефакт = узлы Нексуса системы роста

- **Каталог каналов** (тип, платформа, формат, бюджет в медиаплане) — метод `8.1` Step 1.1.
- **Каталог сегментов** (описание, размер, боль/JTBD, источник инсайта) — метод `8.1` Step 1.2.
- **Каталог офферов** (ЦП, CTA, формат) — метод `8.1` Step 1.3.
- **Гипотезы конфигураций** (5–15) — карточки с 7 элементами как узлы Нексуса роста (метод `8.1` Phase 2).
- Рост CP: появление структурированных гипотез снижает неопределённость [S1] VI.7.

---

## 🚦 CP-гейт

Скаутинг — фаза насыщения, не гейта. CP **растёт** по мере формализации гипотез (колесо Гилада, ICE/RICE как объективизация [S1] VI.7). Приоритизация порядка тестирования (метод `8.1` Phase 3): быстрые и дешёвые тесты первыми (performance > органика), высокоуверенные первыми.

> SCOUT → BUNCH: гипотезы-кандидаты собираются в Банч активных конфигураций в моменте (фаза 3).

---

## 🛡️ Анти-workslop

- **AI-офферы/креативы** — топ-зона workslop [S2] III.7: каждый AI-сгенерированный оффер маркируется, проходит A/B (фаза 5), трассируется до интервью/Value Validation.
- **Рыночные цифры TAM/размер сегмента** — workslop-зона: трассировка до Market Research (Step 3), SimilarWeb, рекламного кабинета.
- **Сегменты из воздуха**: AI-персоны без интервью = workslop; проверять через Customer Validation (Step 2).

---

## 🧮 Структура гипотезы конфигурации (метод `8.1` — 7 элементов)

Каждая гипотеза-кандидат из скаутинга несёт 7 обязательных элементов (метод `8.1` структура):
1. **Сегмент** — демография + поведение + психография + JTBD + размер.
2. **Канал** — тип (performance/органика/direct/referral) + платформа + формат + инструмент.
3. **Оффер** — ЦП + формат (freemium/tripwire/страховочный/B2B) + CTA + посадочная.
4. **Воронка** — точка входа → путь → целевое действие (что = «привлечённый клиент»).
5. **Целевые метрики** — целевой CAC (из LTV/3), ожидаемая конверсия, целевой LTV, период окупаемости, критерий успеха.
6. **Ресурсы** — бюджет теста, ответственный, материалы, срок запуска, длительность.
7. **Обоснование** — почему сегмент/канал/оффер (инсайты Customer Validation, GTM, аналоги).

> Целевой CAC = LTV/3 (правило большого пальца, метод `8.1` Problem 3); критерий успеха — конкретное число (CAC ≤ X при выборке ≥ 50), не «будет работать».
