## 🎯 Цель фазы для Step 4

**Harvesting** [S1] VI.4 — собрать урожай цикла: зафиксировать подтверждённое ЦП как commit в цифровой профиль продукта, сформулировать **mNSM** (монетизируемая северная метрика) и построить дерево метрик с корнем = mNSM и ребром → NPV. Здесь же — **финальный гейт шага: 🟧 Need/Value Fit** (CP ≥ gate).

HARVEST = три последовательных подшага (методы 4.4 → 4.5 → 4.6):
1. **4.4 Фиксация need/value fit** — VP-node v1.0 + VPC-node + доказательная база.
2. **4.5 mNSM-формулировка** — NSM через «чёрный ящик» + **mNSM-проверка** (монетизируема ли?) + ребро mNSM→NPV.
3. **4.6 Дерево метрик** — корень = mNSM, 3 уровня, формулы, ребро mNSM→NPV, Harvesting-узел.

> **КЛЮЧЕВОЙ акцент шага**: различение **NSM** и **mNSM** (Принцип 6). Классическая NSM ценностна «в вакууме» (может расти без связи с деньгами). **mNSM** — NSM, прошедшая проверку монетизируемости. Корень дерева = **mNSM**, не NSM, не выручка.

## 👤 Роль

- **Product Engineer** — формулирует VP (Остервальдер), заполняет VPC, определяет границы применимости и anti-scope, версионирует (v1.0/v1.1/v2.0); владеет структурой дерева (корень → Ур.1 → Ур.2 → Ур.3).
- **Growth Engineer** — **владелец mNSM**: формулирует NSM через «чёрный ящик», проводит mNSM-проверку, строит ребро mNSM→NPV, владеет драйверами роста mNSM (новые клиенты + LTV).
- **Product Ops** — поддерживает версионирование узлов, следит за фазой «увядания», настраивает Кортекс-валидацию каузальности рёбер дерева.

> **Зона человека**: стратегическое решение «достигнут ли need/value fit» (конвергенция ≥2 источников); стратегический выбор финального VP и северной метрики — целеполагание, не делегируется.

## 🤖 Кортекс-агенты

Три агента (по одному на подшаг) [AI-TRANSFORMATION 4.4–4.6]:

**① Агент-фиксация в Нексус (4.4)**
1. Сводит результаты EXECUTE в таблицу: эксперимент / метод / сегмент / метрика / цель / факт / подтверждено.
2. Расставляет доказательства по **иерархии достоверности**: пилот (использовал и готов продолжать) > оплата/предзаказ > заявка > клик/CTR > цитата.
3. Генерирует VPC (Customer Profile: Jobs/Pains/Gains ↔ Value Map: Products & Services/Pain Relievers/Gain Creators) с трассировкой Pain Reliever→Pain, Gain Creator→Gain.
4. Формирует **commit** VP-node + VPC-node + Evidence-node + Boundary-node в цифровой профиль (Нексус).
5. **Anti-workslop-фильтр VP**: широкий смысл — фокус на профите, не на фичах.
6. Триггерит обновление связанных узлов; флаг «увядания» при устаревании доказательной базы.

**② Агент-mNSM — проверка монетизируемости (4.5)**
1. Берёт зафиксированные VP из 4.4; ведёт вторичный скоринг (если VP несколько): сила подтверждения / SAM / дельта vs альтернативы / стратегическое соответствие / сложность реализации / защищаемость.
2. Применяет алгоритм NSM «чёрный ящик»: ключевая потребность → Состояние 1 (as-is) → Состояние 2 (to-be) → Вход/Выход → NSM = Выход/Вход или скорость перехода 1→2.
3. Экстраполирует на рынок: NSM (рынок) = NSM (потребитель) × количество потребителей («единица потребности» = событие удовлетворения: поездка/урок/сделка/просмотр).
4. **★ mNSM-проверка (ключевой шаг)**: отвечает «монетизируема ли NSM?» — есть ли метрика выручки/маржи, растущая вместе с ней? Если ценностна, но не монетизируема → помечает как **NSM, не mNSM**, требует перевода в монетизируемую форму или поиска платящей связи.
5. Формирует **ребро mNSM → NPV**.
6. **Отбраковка vanity**: DAU / выручка / NPS / «количество загрузок» — явно НЕ NSM.

**③ Агент-дерева метрик (4.6)**
1. Берёт mNSM-node из 4.5 как **корень** (Ур.0). Если корень — выручка/DAU, отбраковывает.
2. Строит декомпозицию: Ур.1 (бизнес-метрики, lagging) → Ур.2 (драйверы/воронка) → Ур.3 (операционные листья, leading, контролируемы).
3. **Записывает формулу** каждого родителя из детей и проверяет бьётся ли подстановкой.
4. **Кортекс-валидация каузальности** каждого ребра: «если принудительно увеличить дочернюю (зафиксировав остальное), вырастет ли родительская механически?» Каузальность = да; корреляция = не ребро.
5. **Гигиена A/B листьев**: primary metric, α=0.1, мощность 1−β=80%, MDE, sample size, длительность ≥2 бизнес-циклов.
6. **Ребро mNSM → NPV**: явное ребро от корня к NPV-узлу (Ур.1).
7. **Harvesting-узел**: фиксация NPV-инкремента после цикла → обновление контекста → новый цикл.

## 🧠 Человек (метапознание — отличать «оффер сработал» от «ценность подтверждена»)

- **★ Различать NSM и mNSM (главный риск шага)**: NSM отражает ценность, но может быть не монетизируема. Главный риск — остановиться на ценностной NSM без mNSM-проверки. NSM «успешные уроки» без связи с платящими действиями — не mNSM; gate не пройден.
- **Отличать «подтверждено» от «нам кажется»**: need/value fit достигнут при конкретных измеримых сигналах — по иерархии доказательств уровень «оплата/предзаказ» или «пилот с готовностью продолжать» + конвергенция ≥2 источников.
- **Тест корня дерева**: если метрика растёт — ценность точно растёт? Можно ли расти в деньгах при падающей метрике? Если да — это не mNSM.
- **Границы продукта**: NSM отражает то, на что продукт влияет напрямую, не «за границами» (оценки в школе — за границами сервиса репетиторов; NSM = скорость подбора).
- **Veto** на корень = выручка/DAU по требованию стейкхолдеров; veto на фиксацию без иерархии доказательств (только CTR → уровень «Низкий»).

## 📦 Артефакт = узлы Нексуса продукта (+CR)

- **VP-node** (версионированный, v1.0) — широкий + узкий смысл + статус «✅ Подтверждено — need/value fit» + сегмент + дата.
- **VPC-node** — Customer Profile ↔ Value Map + Fit Score по каждой потребности.
- **Evidence-node** — сводная таблица валидации + ключевые цитаты с источником/датой.
- **Boundary-node** — для какого сегмента, при каких условиях, что НЕ входит (anti-scope).
- **NSM-node** (потребитель) — формулировка, единица, Состояние 1/2, Вход/Выход, границы.
- **mNSM-node** (рынок) — «единица потребности», проверка монетизируемости ✅/❌, цели 6/12 мес.
- **NPV-edge** — ребро mNSM → NPV.
- **MetricTree-node** — корень = mNSM (Ур.0), 3 уровня, формулы, leading/lagging-разметка.
- **NPV-node** (Ур.1) — вершина возврата монетизируемой ценности.
- **Leaf-node** (Ур.3) — контролируемая метрика с гигиеной A/B.
- **Harvesting-node** — фиксация NPV-инкремента, обновление контекста, триггер нового цикла.
- **Causality-edge** — каждое ребро прошло Кортекс-тест.
- **Context Ripeness** по шагу: ≥70% (HARVEST поднимает до порога gate).

## 🚦 CP-гейт — 🟧 Need/Value Fit (финальный gate Step 4)

**HARVEST = гейт [[fit-points#🟧 Need/Value Fit]]**. Критерии прохождения:
- Confidence Point ценностной гипотезы ≥ порога stage-gate **после валидации**.
- По иерархии доказательств есть сигнал уровня «оплата/предзаказ» или «пилот с готовностью продолжать» (а не только CTR/заявка).
- Конвергенция ≥2 источников (количественный + качественный) в одном направлении.
- Need/value fit зафиксирован в Нексусе (VP-node, VPC-node).
- **★ mNSM-проверка пройдена**: NSM проверена на монетизируемость → mNSM (не vanity). [Принцип 6]
- Дерево метрик: корень = mNSM, формулы бьются, рёбра каузальны, ребро mNSM→NPV построено, Harvesting-узел определён.
- Context Ripeness Нексуса продукта ≥ порога (≥70%).

**Не пройден →** осознанный отказ / возврат к Step 2–3 (проблема в потребителе/рынке, не в ценности). *«Рост контекста = рост скорости процессов, в т.ч. отказа»* [S1] VI.3.

## 🛡️ Анти-workslop

- **★ NSM↔mNSM-путаница** — главный риск этапа. Классическая NSM ценностна, но может быть не монетизируема. Без mNSM-проверки этап считается незавершённым. Корень дерева обязан быть **mNSM** с ребром → NPV.
- **Vanity vs mNSM**: DAU / выручка / NPS / загрузки — отбраковываются (показывают активность/монетизацию/субъективное мнение, не ценность). Запрещённый синоним: НЕ «метрика ради метрики» → **mNSM как монетизируемая ценность** (Принцип 6).
- **Workslop формулировок VP**: широкий смысл не перегружает фичами — фокус на профите; Pain/Gain подтверждены интервью, не придуманы.
- **Документация исчезает**: артефакт = узел цифрового профиля (Нексус), не разрозненный документ. Запрещённый синоним: НЕ «документация по продукту» / «запись в базе знаний» → **commit в цифровой профиль**.
- **Дерево ≠ дашборд**: формула каждого родителя записана и бьётся подстановкой; корреляция ≠ каузальность; 1 mNSM + 5–9 узлов (не >20).
- **Увядание**: данные >3 месяцев → флаг переподтверждения, иначе фиксация не коммитится.
