# 3.bunch — BUNCH

> Шаг **Step 1 — Idea & Context** · фаза цикла Product Sprint · Линза: **над 3PL** (Банч собирает все три Линзы) · Инструкция для Инженера Продукта (PAF [S1],[S2]).
> Референс метода: [[TRADITIONAL/RB-STEP-1-IDEA/3.1 Консолидация]] · [[TRADITIONAL/RB-STEP-1-IDEA/3.2 Confidence Points]] · [[TRADITIONAL/RB-STEP-1-IDEA/3.3 AI Bunch Formation]]. Референс «почему»: [[AI-TRANSFORMATION/RB-STEP-1-IDEA/3.1 instruction]] · [[AI-TRANSFORMATION/RB-STEP-1-IDEA/3.2 instruction]] · [[AI-TRANSFORMATION/RB-STEP-1-IDEA/3.3 instruction]] · [[reformation]] §1.

## 🎯 Цель фазы для этой вехи

Сформировать **Банч (Feature Bunch)** — связку фич **в моменте** под текущее состояние Нексуса, чтобы выиграть конкуренцию (а не «распределить разработку» — **Беклог упразднён** [S1] Принцип 1, VI.6). Это **Identify-фаза** Und-Id-Ex [S1] VI.5: Кортекс генерирует фич-кандидаты из графа opportunities (SCOUT) и отбирает 5–7 фич.

В Step 1 BUNCH — **тяжёлая фаза**: из opportunities всех трёх Линз (3.1 Консолидация) → initial CP (3.2) → **Bunch-v1** из 5–7 features с mNSM-impact и **комплексным NPV Банча** [S1] VI.6.

**Лимиты Банча** [S1] VI.6: **Bunch Size** — ограничен скоростью восприятия рынка (не разработки); **Bunch Window** — период ожидания бизнес-результата.

**Три критерия отбора** [S1] VI.6: ① макс. mNSM; ② мин. рисков (CP); ③ эффект в окне → **комплексный NPV** (сумма NPV фич с учётом синергии).

## 👤 Роль

- **Product Engineer** — ставит задачу Bunch Architect (порог CP, Bunch Size, Bunch Window); валидирует комплексный NPV Банча; veto на фичи без mNSM-ребра. Проверяет NPV-вклад каждой фичи и допущения о синергии.
- **Product Ops** — поддерживает **Nexus RAG** (индекс по 3 Нексусам); следит, что Банч формируется из текущего состояния, а не из старого.
- **Growth Engineer** — валидирует mNSM-потенциал фич Банча; проверяет, что success criteria фич двигают mNSM (не vanity).
- **Зона человека:** состав Банча как инвестиционное решение (какие фичи максимизируют NPV) — за человеком [S2] III.4. Инвестиционная логика распределения ресурсов не делегируется.

## 🤖 Кортекс-агент фазы

**Bunch Architect Agent** ([reformation] §1.5) — ядро стадии:

- RAG по 3 Нексусам + граф opportunities (3.1) с CP (3.2).
- Генерация фич-кандидатов из opportunities (Identify-фаза [S1] VI.5).
- Отбор 5–7 фич по 3 критериям [S1] VI.6: ① макс. mNSM; ② мин. рисков (CP); ③ эффект в Bunch Window.
- **Комплексный NPV Банча** (сумма NPV фич с учётом синергии) [S1] VI.6 — ключевой артефакт (не «low/med/high», а число).
- **Bunch Size** (5–7, по скорости восприятия рынка) и **Bunch Window**.
- **Anti-workslop фильтр:** фичи без mNSM-ребра → отсев; CP < порога → отсев; дубликаты → отсев.

- **Input:** Nexus RAG (3 Нексуса) + `Opportunity-Cluster` (3.1) + CP (3.2).
- **Action:** генерация + отбор + расчёт NPV с синергией.
- **Output:** `Feature`-узлы Банча (5–7) с `mNSM-ребро`, `CP`, `NPV-вклад`; `Bunch-v1`-мета-узел с комплексным NPV, Size, Window.

## 🧠 Человек (метапознание)

- **Критический чекпоинт:** NPV Банча — потенциальный чёрный ящик [reformation] §1.3. Инженер проверяет: каков NPV-вклад каждой фичи? Какие допущения? Что если синергия переоценена?
- **Veto-право:** фича без явного mNSM-ребра (vanity) — отсев, даже если CP высокий.
- **Вопрос:** «Банч формирован из текущего состояния Нексуса или из старого списка? Bunch Size = 6 — это по восприятию рынка или по вместимости разработки?» (должно по рынку [S1] VI.6).

## 📦 Артефакт = узлы Нексуса (+ Context Ripeness)

- **Создаются:** `Feature`-узлы (5–7) с `mNSM-ребро`, `CP`, `NPV-вклад`, `Bunch-Window`; `Bunch-v1`-мета-узел (комплексный NPV Банча = сумма NPV фич с учётом синергии, Size, Window).
- **Метаданные каждого Feature-узла:** `источник` (opportunity[]), `CP`, `mNSM-numeric`, `NPV-вклад`, `timestamp`, `Bunch-Window`, `статус увядания`; рёбра к `JTBD-Opportunity` (Product), `Monetization-Hypothesis` (Business), `Strategic-Opportunity` (Strategy).
- **Bunch-v1 мета-узел:** `комплексный-NPV`, `Bunch-Size` (5–7, по скорости восприятия рынка), `Bunch-Window` (период ожидания бизнес-результата), `синергия` (допущение, проверяемое инженером).
- **Прирост CR:** +5–10% (Банч замыкает связи между Нексусами — каждый Feature-узел имеет рёбра ко всем трём).

## 🚦 Confidence Point (гейт к след. фазе)

- **Критерий перехода к PITCH:** `Bunch-v1` существует; комплексный NPV > 0 (базовый сценарий); каждая фича имеет mNSM-ребро и CP ≥ порога.
- **Стоимость риска:** «Какой NPV теряем, держа в Банче фичу, чья валидация failed?» — drop по NPV-логике.

## 🛡️ Анти-workslop

- **Workslop-риск:** **Банч как старый беклог под новым именем** (накопленный, а не в моменте); mNSM = low/med/high без числа; NPV без разложения.
- **Признак:** фича без mNSM-ребра; CP фичи > CP opportunity-источника; Банч формирован из списка, а не из текущего состояния.
- **Правило верификации:** Банч — только из текущего состояния Нексуса [S1] VI.6; каждая фича — mNSM-ребро + NPV-вклад; комплексный NPV Банча обязателен.

**Банч vs Беклог** [S1] VI.6 (контраст, чтобы не скатиться в запрещённый синоним):

| | Беклог (упразднён) | Банч |
|---|---|---|
| Цель | С чем работать в первую очередь | Как выиграть конкуренцию |
| Объекты мышления | Ресурсы + идеи + метрики | Свойства продукта + конкуренция + гэп |
| Срок жизни | Накопленный | Только в моменте |
| Источник | Стейкхолдеры + старые штурмы | Только анализ текущего состояния |
| Миссия | Утилизировать разработку | Макс. финансовая и продуктовая ценность быстро |

## 🔭 Связь с Тремя Линзами (3PL)

BUNCH работает **над 3PL**: Банч собирает фичи из opportunities всех трёх Линз.
- **Product Lens** — фичи → mNSM;
- **Business Lens** — NPV Банча, Рычаги;
- **Strategy Lens** — Банч обслуживает `Bet`.
Связь: каждый `Feature`-узел имеет рёбра к `JTBD-Opportunity` (Product), `Monetization-Hypothesis` (Business), `Strategic-Opportunity` (Strategy). Фича без хотя бы одного ребра → подозрение на workslop.
