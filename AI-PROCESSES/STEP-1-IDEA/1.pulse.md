# 1.pulse — PULSE

> Шаг **Step 1 — Idea & Context** · фаза цикла Product Sprint · Линза: **над 3PL** (собирает состояние всех трёх) · Инструкция для Инженера Продукта (PAF [S1],[S2]).
> Референс метода: [[TRADITIONAL/RB-STEP-1-IDEA/1.0 Базовые концепции]]. Референс «почему»: [[AI-TRANSFORMATION/RB-STEP-1-IDEA/0.1 instruction]] · [[AI-TRANSFORMATION/RB-STEP-1-IDEA/1.0 instruction]] · [[reformation]] §1.

## 🎯 Цель фазы для этой вехи

**Progress Pulse** — первое событие цикла Product Sprint [S1] VI.4: фиксация состояния трёх Нексусов «как пилоты у приборной панели», измерение **Context Ripeness** (полнота × актуальность) [S1] III.1 и формирование из разрыва между текущим состоянием и Видением **стартового Гэпа (gap)** [S1] базовая логика.

В Step 1 (null base) Гэп = «весь Vision, продукта ещё нет». На data base Гэп = конкретные бутылочные горлышки. Цель Pulse — **запустить event-based цикл** [S1] VI.2, VI.4 и зафиксировать базис (null/data), который определит, будет ли SCOUT генерировать Vision с нуля или заливать Current State из product analytics [S1] VI.5. Pulse — это фиксация, а не насыщение.

## 👤 Роль

- **Product Engineer** — запускает цикл; считывает состояние Нексусов через Кортекс; формулирует стартовый Гэп; определяет базис (null/data). Проектирует **AI workflow** цикла (что делегировать Кортексу, что оставить себе) [S2] III.6; не вайбкодер, а инженер контекста.
- **Product Ops / Product Architect** — проверяет, что Нексусы существуют как графы Узлов (не разрозненные документы), что CR вычисляется как метрика (не самооценка), что у каждого Узла есть метаданные. Выставляет фазу зрелости Кортекса (1/2/3) [S1] Часть VII.
- **Portfolio Manager** (если есть) — сверяет стартовый Гэп с Картой Целей портфеля [S1] III.2, чтобы прелюдия не отклонилась от Ставок стратегической линзы.

**Паттерн работы с ИИ** [S2] III.4 — инженер фиксирует, какой паттерн у этого цикла: **Кентавр** (делит/делегирует часть задач ИИ) или **Киборг** (интегрирует поток задач с ИИ).

## 🤖 Кортекс-агент фазы

**Nexus-Pulse Agent** (фаза зрелости Кортекса 2+ [S1] Часть VII) — событийно считывает три Нексуса и возвращает:
- список Узлов с метаданными по каждому Нексусу;
- значение CR = `полнота(0–1) × актуальность(0–1)` с фазой «созревание»/«увядание» [S1] III.1;
- флаги устаревших Узлов (`timestamp` старше порога свежести по типу);
- предварительный список «дыр» (отсутствующих Узлов), критичных для решений BIG Idea.

- **Input:** RAG-индекс по 3 Нексусам + (на null base) пустые скелеты.
- **Action:** вычисляет CR; помечает Узлы в стадии «увядания»; генерирует стартовый дашборд Pulse.
- **Output:** `Pulse`-узел Кортекса со снимком состояния + триггер `GAP_DEFINED` → передача в SCOUT.

**Пороги свежести** (настраивает Product Ops): TAM-SAM-SOM — квартал; Trend — месяц; Competitor-отзывы — неделя; Unit-Economics — месяц; JTBD — полгода.

## 🧠 Человек (метапознание)

- **Чекпоинт валидации:** Кортекс не «придумал» Узлы — каждый обязан иметь `источник` [S1] III.1. На null base пустые скелеты — это честно, а не заполненные галлюцинациями.
- **Veto-право:** CR ≥ 60% по Нексусу, которого физически не существует, = **иллюзия живого Нексуса** [reformation] Часть VIII → инженер обнуляет метрику.
- **Чекпоинт базиса:** честно фиксирует null vs data — это определяет путь SCOUT.
- **Junior-ограничение:** на низкой зрелости Кортекса не доверять авто-CR, сверять выборку Узлов вручную [S2] III.4.

## 📦 Артефакт = узлы Нексуса (+ Context Ripeness)

- **Создаётся/обновляется:** `Pulse`-узел (мета-узел Кортекса, не Нексуса) — снимок состояния 3 Нексусов в момент старта цикла.
- **Метаданные Pulse-узла:**
  - `cycle_id` — идентификатор цикла Product Sprint;
  - `timestamp` — момент Pulse;
  - `CR_рынка`, `CR_продукта`, `CR_роста` — спелость каждого Нексуса (0–1) с фазой «созревание»/«увядание»;
  - `gap_summary` — стартовый Гэп (что закрываем этим циклом);
  - `базис` — null/data [S1] VI.5;
  - `trigger=GAP_DEFINED`;
  - `бюджет_цикла` — лимит ресурсов на насыщение (стоимость риска);
  - `первая_линза` — какая Линза включается первой для насыщения (Strategy/Product/Business).
- **Прирост CR:** **только фиксация**, не насыщение. Pulse — «приборная панель», а не источник новых Узлов. Прирост приходит в SCOUT (+20–35%) + BUNCH (+5–10%) + EXECUTE (+10–15%) + PITCH (+5%). Цель цикла — CR ≥ 60% по всем трём к HARVEST.

## 🚦 Confidence Point (гейт к след. фазе)

- **Критерий перехода к SCOUT:** существует `Pulse`-узел + стартовый Гэп сформулирован (даже «gap = весь Vision» на null base). Без Гэпа цикл не запускается [S1] базовая логика.
- **Стоимость риска** (Принцип 4 [S1]): «Сколько ресурсов готовы потратить на насыщение, прежде чем Гэп станет измеримым?» — фиксируется как бюджет цикла. Риски — будущее, не потери.

## 🛡️ Анти-workslop

- **Workslop-риск:** «дашборд ради дашборда» — Pulse красиво выглядит, но Узлы без источников [S2] III.7 (~15,4% материалов перекладывают когнитивную нагрузку; MIT — 95% не получают отдачи от ИИ именно из-за workslop).
- **Признак:** CR посчитан из Узлов с `источник = "Кортекс предположил"`; нет drill-down до источника; инженер не может ответить «зачем этот Нексус для решения о Банче».
- **Правило верификации:** каждый Узел — внешний/внутренний источник (URL, дата, интервью, product analytics); без источника → CP = 0; дашборд показывает `источник` в один клик. Каркас PAF считается усвоенным, когда инженер устно проговаривает путь «Гэп → скаутинг через 3 Линзы → Банч → CP → Pitching» без подглядывания [S1] VI.4–VI.5.

## 🔭 Связь с Тремя Линзами (3PL)

Progress Pulse **не привязан к одной Линзе** — он собирает состояние всех трёх и определяет, какая Линза включается первой для насыщения [S1] VI.1:
- новый рынок → первой **Strategy Lens** (SCOUT → Нексус рынка);
- новый продукт → первой **Product Lens** (SCOUT → Нексус продукта);
- новая модель роста → первой **Business Lens** (SCOUT → Нексус роста).

Сверка Гэпа с целью бизнес-юнита — через **Карту Целей (Goal Map)** [S1] III.2. Зона человека: целеполагание — какой Гэп закрываем этим циклом, какие Ставки портфеля обслуживает прелюдия [S2] III.4.

## 🧩 Skill Map / профстандарт

- **Skill Map [S3]:** сектор 1 **Discovery & Researches** (контекст продукта/рынка) + сектор 4 **Growth & Experiments** (цикл Sprint). Уровень — **внутренний круг (junior)**: атомарная задача фиксации состояния. Навыки связаны лучами: Pulse предваряет «формирование Банча» [S3] IV.4 (нельзя формировать Банч без фиксации состояния).
- **Профстандарт 06.012 [S4]:** обобщённая функция **A** (сопровождение, A/01.4 оргсопровождение исследований), уровень квалификации 4 — ассистент/junior. Pulse — входной ритуал для всех вышестоящих функций B/C/D.
