# 5.execute — EXECUTE

> Шаг **Step 1 — Idea & Context** · фаза цикла Product Sprint · Линза: **Product** (валидация ценности фич Банча) · Инструкция для Инженера Продукта (PAF [S1],[S2]).
> Референс метода: [[TRADITIONAL/RB-STEP-1-IDEA/4.1 Генерация гипотез]] · [[TRADITIONAL/RB-STEP-1-IDEA/4.2 Rapid Validation]] · [[TRADITIONAL/RB-STEP-1-IDEA/4.3 Обновление CP]]. Референс «почему»: [[AI-TRANSFORMATION/RB-STEP-1-IDEA/4.1 instruction]] · [[AI-TRANSFORMATION/RB-STEP-1-IDEA/4.2 instruction]] · [[AI-TRANSFORMATION/RB-STEP-1-IDEA/4.3 instruction]] · [[reformation]] §1, §4.

## 🎯 Цель фазы для этой вехи

**Rapid validation гипотез Банча**, чтобы **поднять CP** фич по stage-gate (Принцип 4 [S1]) — не «проверить ради галочки», а снизить неопределённость через evidence. В Step 1 EXECUTE — **лёгкая фаза** (discovery-шаг): это не build (Build+Bale+Germination активны в Step 6–8), а **валидация гипотез**.

В Identify/Execute-фазе Und-Id-Ex [S1] VI.5: гипотезы → тест → рост CP → gate. Цель — **NPV фичи растёт по мере роста CP** (доказанная ценность → больший NPV [S1] Принцип 6).

Три подшага:
1. **Гипотезы (4.1)** — Hypothesis Card для каждой фичи Банча, связанная с mNSM;
2. **Rapid Validation (4.2)** — smoke test / прототип / customer validation по CP фичи (вайбкодинг [S2] III.6);
3. **Обновление CP (4.3)** — анализ результатов, рост/падение CP по колесу Гилада [S1] VI.7.

## 👤 Роль

- **Product Engineer** — формулирует success criteria (измеримые пороги); валидирует связку гипотезы с mNSM; veto на premature scale; выбирает метод валидации.
- **Discovery / Launcher PM** (если есть) — **вайбкодит** smoke test/прототип через Кортекс (full-stack: продуктовые компетенции + вайбкодинг [S2] III.6); rapid-итерации.
- **Product Ops** — настройка вайбкодинг-пайплайна (генерация лендинга/прототипа из Hypothesis Card).
- **Growth Engineer** — валидирует, что success criteria двигают mNSM (не vanity).
- **Зона человека:** решение о масштабировании по результатам валидации — за человеком (ложное ощущение готовности — сквозной риск [reformation] Часть VIII).

## 🤖 Кортекс-агент фазы

Два агента в связке:

**Hypothesis-генератор** (режим Scouting Agent [reformation] §1.5):
- Для каждой `Feature` Банча генерирует Hypothesis Card по шаблону: «We believe that [feature] for [segment] will achieve [mNSM outcome] if [success criteria]».
- Связка с mNSM: success criteria обязан включать метрику с ребром к `mNSM-numeric` (не vanity).
- Трассировка к Узлам: ребро к `Opportunity-Cluster` → `JTBD-Opportunity`/`Strategic-Opportunity`/`Monetization-Hypothesis`. Гипотеза без трассировки → CP = 0.
- Предлагает метод валидации по CP фичи.
- **Output:** `Hypothesis-Card`-узлы.

**Validation Agent** ([reformation] §1.5):
- По CP фичи выбирает метод: smoke (CP 1–3) / prototype (4–6) / customer validation (7–8).
- **Автогенерация smoke test/прототипа через вайбкодинг:** из `Hypothesis-Card` — лендинг (Tilda/web) / Figma-прототип / no-code.
- Сбор сигналов: конверсия, completion rate, willingness-to-pay, цитаты.
- **Авто-обновление CP** по результатам (колесо Гилада [S1] VI.7): сигнал ≥ порога → рост CP; < порога → падение/переформулировка.
- **Различает «оффер сработал» vs «ценность подтверждена»** [reformation] §4.6 — главный анти-workslop-чек.
- **Output:** `Validation-Result`-узел (метод, сигнал, pass/fail, дельта CP), автообновлённый CP, `Validation-Artifact` (URL).

**Пороги** (преемственность из исходника 4.2): smoke — 5–10% конверсия в signup; prototype — 70%+ completion rate задач; customer validation — 80% выражают сильный интерес.

- **Input:** `Feature`-узлы Банча (3.bunch) + `Opportunity-Cluster` (2.scout) + `mNSM-numeric`.
- **Output:** `Hypothesis-Card`, `Validation-Result`, автообновлённый CP, `Validation-Artifact`.

## 🧠 Человек (метапознание)

- **Главный workslop-риск:** «оффер сработал» ≠ «ценность подтверждена» [reformation] §4.6. Конверсия на лендинге может быть от любопытства, не от ценности. Инженер проверяет: сигнал подтверждает JTBD или только интерес?
- **Veto-право:** premature scale (масштабировать по одному smoke-сигналу) — veto; нужна конвергенция ≥ 2 сигналов.
- **Вопрос:** «Конверсия 8% — это curiosity-click или реальная потребность? Какой сигнал доказывает, что JTBD выполнена?» · «Success criteria = "40% retention" — связано с mNSM или это vanity?»

## 📦 Артефакт = узлы Нексуса (+ Context Ripeness)

- **Создаются:** `Hypothesis-Card`-узлы (по числу фич) с рёбрами к `Feature`, `mNSM-numeric`, `Opportunity-Cluster`; `Validation-Result`-узлы; автообновлённый `CP` фич; `Validation-Artifact`.
- **Метаданные:** `источник` (Feature + Opportunity), `CP`, `success-criteria` (с mNSM-ребром), `метод`, `сигнал` (число), `pass/fail`, `delta-CP`, `тип-сигнала` (offer-click vs value-confirmation), `timestamp`, `статус увядания`.
- **Прирост CR:** +10–15% (evidence из реальных сигналов — ядро созревания).

## 🚦 Confidence Point (гейт к след. фазе)

- **Критерий перехода к HARVEST:** каждая фича Банча имеет `Validation-Result` (pass/fail) с дельтой CP; фичи с CP < порога после валидации → отсев или переформулировка; Confirmed-фичи формируют `Bunch-v2`.
- **Стоимость риска:** «Сколько ресурсов на smoke test? Какой NPV-потенциал теряем, если валидация поверхностная?» — приоритет по NPV-потенциалу.

## 🛡️ Анти-workslop

- **Workslop-риск:** **«оффер сработал» как ложное подтверждение ценности** (главный риск Value-стадий [reformation] §4.9); premature PMF/готовности по одному сигналу; правдоподобные, но неверные гипотезы (галюцинации [S2] III.4); success criteria = vanity без mNSM-ребра.
- **Признак:** конверсия на лендинге принята как подтверждение JTBD; CP поднят по одному smoke без повторной проверки; гипотеза без трассировки к Узлу; success criteria = DAU/registrations (не монетизируемые); CP назначен без evidence.
- **Правило верификации:** различать оффер/ценность; конвергенция ≥ 2 сигналов для масштабирования; CP растёт только от снижения неопределённости; каждая Hypothesis Card — ребро к Opportunity-Cluster (трассировка); success criteria — mNSM-ребро; CP = 0 без трассировки.

**Шаблон Hypothesis Card** (преемственность из исходника 4.1):
- **Hypothesis Statement:** «We believe that [feature] for [segment] will achieve [mNSM outcome] if [success criteria]»;
- **We'll Know This Is True When** — метрика + порог + timeframe;
- **Validation Method** — smoke / prototype / customer validation;
- **Current CP + Risk level**;
- **Dependencies** (must exist / blocked by).

**Лестница методов по CP** (преемственность из исходника 4.2): CP 1–3 → smoke (лендинг, concierge, Wizard-of-Oz); CP 4–6 → prototype (Figma, pre-sale); CP 7–8 → customer validation (deep interviews, data analysis).

## 🔭 Связь с Тремя Линзами (3PL)

EXECUTE — **активная Product Lens** (валидация ценности фичи).
- **Связь с Business Lens:** подтверждённая ценность → рост NPV фичи → рост NPV Банча; success criteria → mNSM → NPV;
- **Связь со Strategy Lens:** валидация подтверждает/опровергает `Bet`; фича, не подтвердившая Ставку, → переоценка.
