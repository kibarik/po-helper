# 2.scout — SCOUT / Скаутинг гипотез (Step 2: Customer)

> Фаза Sprint: **SCOUT** (тяжёлая). Источник: [[operating-model]] (Скаутинг вместо приоритизации, Принцип 5), [S1] VI.5. Термины — по [[sa_documentation/naming_conventions|naming_conventions]].

---

## 🎯 Цель фазы для Step 2

Скаутинг возможностей/угроз на карте целей: превратить каркас Нексуса (из PULSE) в **«поезд гипотез» 4 типов** (Identify-фаза Und-Id-Ex), затем через **CP-скоринг** сформировать **Банч гипотез в моменте** по принципу «самый страшный риск — первым» [2.1, 2.2 instruction]. Гипотеза — это кандидат-узел Нексуса с initial CP, **а не пункт Беклога**.

Ключевые принципы:
- Потребность ≠ проблема: проблема — препятствие на пути к цели, не «отсутствие чего-то» («нет CRM» — не проблема) [2.1].
- ICE/RICE — **прокси Confidence Point** (по колесу Гилада), а не финальный балл ради балла [2.2].
- Банч в моменте, не накопительный список [S1] VI.6.

---

## 👤 Роль

- **Product Engineer** — формулирует ядро BIG Idea; задаёт Кортексу рамки генерации; валидирует, что каждая гипотеза проверяема и описывает **мир потребителя без продукта** (запрет: слова «приложение/сервис/продукт» в формулировке) [2.1].
- **Product Ops** — обслуживает RAG по Нексусам; гарантирует, что генерация идёт из существующих узлов (анти-галлюцинация).
- **Growth Engineer** — добавляет 4-й мета-тип: **гипотезу mNSM** (монетизируема ли ценность сегмента) [2.1].
- **Зона человека:** выбор ядра идеи, отсечение нерелевантных гипотез, оценка «стоимости риска» по каждой (субъективный стратегический вызов) [S2] III.4.

---

## 🤖 Кортекс-агенты

**1. Генератор гипотез (Hypothesis Generator Agent):** [[AI-TRANSFORMATION/RB-STEP-2-CUSTOMER-VALIDATION/2.1 instruction]]
- **Вход:** BIG Idea + узлы Нексуса рынка/продукта (RAG) + каркас Нексуса потребителя из PULSE.
- **Действие:** генерирует **«поезд» ≥15 гипотез** 4 типов по PAF-шаблонам:
  1. **Поведение** — «Когда <роль>, <ситуация>, то <поведение>, потому что <мотивация>».
  2. **Проблема** — «Когда <роль>, <ситуация>, то <проблема>, а это мешает <цель>».
  3. **Мотивация/Цель** — «<Роль> хочет <желаемое>, а не <текущее>, чтобы <результат>».
  4. **Блокер** — «<Роль> <поведение/проблема>, потому что <причина>».
  Каждая привязывается к узлу Нексуса (маппинг гипотеза→узел), получает initial CP + критерий валидации (≥70% подтверждают / ≤30% опровергают).
- **Выход:** кандидат-узлы с initial CP, сгруппированные по сегментам-кандидатам.

**2. CP-скорер (Confidence Point Scorer Agent, колесо Гилада):** [[AI-TRANSFORMATION/RB-STEP-2-CUSTOMER-VALIDATION/2.2 instruction]]
- **Вход:** кандидат-узлы-гипотезы из Генератора.
- **Действие:** пересчитывает CP (ICE=(Impact+Confidence+Ease)/3 или RICE=(Reach×Impact×Confidence)/Effort) как **прокси CP** + вычисляет **стоимость риска**; ранжирует по принципу «страшный риск — первым» (опровержение = смерть идеи → в первую очередь); формирует **Банч гипотез в моменте** (Bunch Size = ёмкость интервью-плана ~12 интервью × 10–15 гипотез).
- **Выход:** Банч гипотез для ближайшей волны интервью + CP-профиль + стоимость риска + mNSM-слой (Growth Engineer).

---

## 🧠 Человек (метапознание)

- **Чекпоинт:** Product Engineer проверяет, что гипотеза не описывает продукт; что CP-скоринг не превратился в vanity (CP растёт только от снижения неопределённости, не от «мне кажется») [2.2].
- **Veto-право:** переопределить авто-ранжирование, если стратегический риск гипотезы выше; отбраковать гипотезы, не связанные с ядром BIG Idea.
- **Качественный фильтр релевантности** (поверх чисел): 🎯 ядро идеи / ⭐ связанная / 💡 периферия (отложить). Это **скаутинг возможностей** под карту целей, а не «приоритизация списка» [S1] Принцип 5.
- **Сигнал опасности:** «очевидно верные» гипотезы часто ложны — проверять первыми (Build-Measure-Learn).

---

## 📦 Артефакт = узлы Нексуса потребителя (+CR)

- Кандидат-узлы типов Сегмент / Триггер / Проблема+Причина / Мотивация, порождённые гипотезами.
- Метаданные узла-гипотезы: `тип` (4 типа), `формулировка`, `связь с BIG Idea` (🎯/⭐/💡), `origin` (факт/гипотеза/интуиция — анти-галлюцинация), `initial CP`, `критерий валидации`, `Impact/Confidence/Ease` (или Reach/Effort), `стоимость риска`, `ранг в Банче`.
- **Банч гипотез в моменте** (вместо «беклога»).
- Context Ripeness растёт: добавление узлов увеличивает полноту.

---

## 🚦 CP-гейт

- **Критерий выхода из SCOUT:** сгенерировано ≥15 гипотез всех типов; каждая привязана к узлу Нексуса; есть initial CP; топовые — с максимальным риском для ядра идеи; стоимость риска оценена.
- **Gate:** гипотезы с CP ниже порога — не идут в интервью (или идут последними), **но не копятся в беклоге**.
- **Стоимость риска:** «Если проверим нерелевантные гипотезы — потратим интервью-ресурс впустую; сколько стоит ошибка фокуса?» [S1] VI.7.

---

## 🛡️ Анти-workslop

- Генерация **только из RAG по существующим узлам** — без «изобретения» фактов (галлюцинации) [reformation 2.9].
- Гипотеза без связи с узлом Нексуса → workslop, отбрасывается.
- Маркировка факт/гипотеза/интуиция в поле `origin` (по шаблону карточки из [2.1]).
- Запрет на «беклог гипотез» как накопительный список → только **Банч в моменте** [naming_conventions].
- CP не поднимается без снижения неопределённости — иначе vanity.

---

## ⚠️ Типичные ошибки фазы

- **Гипотеза описывает продукт** («пользователи будут пользоваться приложением») → переформулировать в поведение без продукта.
- **Все гипотезы про один сегмент** → сменить угол: кто платит / пользуется / решает / рекомендует (особенно B2B).
- **Галлюцинации генератора** (факты «из воздуха») → только из RAG по узлам, маркировка origin.
- «Проблема = отсутствие чего-то» («нет CRM») → переформулировать как препятствие к цели с контекстом.
- **Один «вагон» вместо «поезда»** → цель ≥15 гипотез всех типов для насыщения одного интервью десятками проверок.
- **Первые = самые уверенные** → инверсия: первые = самые страшные (опровержение = смерть идеи).
- **Одинаковый ICE = одинаковый приоритет** → решает стоимость риска, не балл.
