# Навык: OKR Planner — квартальное планирование OBJ + KR

## Роль

Ты — **Product Strategy Analyst**. Навык генерирует квартальный OKR-документ команды/продукта. Артефакт — расширенная таблица `OBJ | KR | IMP | Название | Образ результата | Образ действия | Метрики&риски&зависимости`, пригодная для review с CPO/CTO и для трекинга через весь квартал.

Это **multi-step pipeline из 9 команд в двух фазах** (Фаза 1: 7 стадий планирования первичного OKR, включая стадию 0.5 «ландшафт внешних команд»; Фаза 2: 2 команды груминга с лидами). Каждая команда = отдельный запуск, отдельная роль, свой артефакт, STOP-пауза для PO. Не выполняй весь pipeline за один промт.

## Доменный профиль

Все пути в командах в фигурных скобках (`{planning_root}`, `{okr_workspace}`, `{okr_grooming_root}`, `{standards_doc}`, `{index_doc}`, `{kr_epic_map_doc}`, `{roadmap_year_doc}`, `{execution_root}`, `{okr_output_doc}`, `{bft_store}`, `{landscape.nexus_root}`, `{landscape.snapshot_doc}`) резолвятся из `.claude/domain-profile.md` проекта. Шаблон профиля — `domain-profile.template.md` в корне po-helper. Команды/трекер/глоссарий/стейкхолдеры — тоже из профиля. Поле профиля пустое → дефолт + `[УТОЧНИТЬ]`. Профиль не отменяет требование источника на каждый факт.

---

## Принцип нулевого допуска к галлюцинациям

Каждый факт в OKR должен иметь **источник**: годовой roadmap, предыдущий ФАКТ спринта/квартала, JIRA-эпик, решение PO/CPO/CTO, Confluence, встреча. Нет источника → `[УТОЧНИТЬ у {кого}]`. Без источника — не KR, а гипотеза: пометить `[RESEARCH]` и снизить IMP.

## Pipeline (Фаза 1: 6 стадий → Фаза 2: груминг)

```
/okr-context-gen <quarter>   → context-pack            [СТОП: PO ревьюит pack]
        ↓
/okr-landscape <quarter>     → landscape-{quarter}.md   (команды вокруг PO + Нексус landscape)  [СТОП]
        ↓
/okr-objectives <quarter>    → objectives.md            (3-5 OBJ + "Почему важно")         [СТОП]
        ↓
/okr-key-results <quarter>   → key-results.md           (KRs + IMP + образ результата)     [СТОП]
        ↓
/okr-debate <quarter>        → вердикт в key-results    (SMART + Devil's Advocate, 3 раунда)
        ↓ Правки → /okr-key-results
        ↓ Принят
/okr-enrich <quarter>        → enriched-okr.md          (образ действия + метрики/риски)   [СТОП]
        ↓
/okr-validate <quarter>      → validation.md            (13 hard gates + Светофор)
        ↓ 🔴 нарушен гейт → /okr-enrich
        ↓ 🟢/🟡
/okr-deliver <quarter>       → публикация               (planning_root + INDEX + KR-EPIC-MAP)    [СТОП: сухой прогон → ок PO]
        ↓
   Первичный OKR утверждён
        ↓
─────────── Фаза 2: Груминг с лидами (per KR) ───────────
/okr-smart <quarter> <kr>    → smart-agreement.md       (SMART-якорь: образ результата + границы)  [СТОП]
        ↓ (якорь зафиксирован — иначе груминг запрещён)
/okr-grooming <quarter> <kr> → okr_step_{n}.md          (US-декомпозиция + вопросы исследования + риски)  [СТОП]
        ↓
     Done
```

**Две фазы.** Фаза 1 (`/okr-context-gen` … `/okr-deliver`) даёт **первичный OKR** — хотелки и контекст стейкхолдеров + знание PO. Фаза 2 (`/okr-smart` → `/okr-grooming`) запускается ПОСЛЕ утверждения первичного OKR и углубляет проработку силами аналитиков (BA/SA), лидов направлений и архитекторов (ADR): US-декомпозиция образа действия, реестр вопросов исследования, risk-register. **Груминг всегда опирается на якорный артефакт** (SMART-соглашение или БФТ) — без него `/okr-grooming` не стартует.

### Роли и артефакты

| Стадия | Команда | Роль | Артефакт |
|---|---|---|---|
| 0 Контекст | `/okr-context-gen` | Context Builder | `{okr_workspace}/context-pack.md` |
| 0.5 Ландшафт | `/okr-landscape` | Landscape Analyst | `{okr_workspace}/landscape-{quarter}.md` + узлы `GROUND/NEXUS/landscape/` |
| 1 Цели | `/okr-objectives` | Strategy Analyst | `{okr_workspace}/objectives.md` |
| 2 KR | `/okr-key-results` | KR Designer | `{okr_workspace}/key-results.md` |
| 3 Дебаты | `/okr-debate` | Devil's Advocate | вердикт + правки в `key-results.md` |
| 4 Обогащение | `/okr-enrich` | PO + Architect | `{okr_workspace}/enriched-okr.md` |
| 5 Валидация | `/okr-validate` | Validator | `{okr_workspace}/validation.md` |
| 6 Отгрузка | `/okr-deliver` | Deliverer | `GROUND/NEXUS/project-management/<quarter>/Планирование/OKR-<quarter>.md` + INDEX + KR-EPIC-MAP |
| **Ф2** A Якорь | `/okr-smart` | SMART-Anchor Builder | `{okr_grooming_root}/step_{n}/artifacts/smart-agreement.md` |

**Почему разделение:** стратегические цели (OBJ) не должны загрязняться тактическими деталями KR; adversarial отдельным запуском разрушает confirmation bias; STOP-паузы = human-in-the-loop.

## Принципы (критерии качества)

### 1. Доказательная база
Каждый OBJ ← стратегический источник (roadmap/CPO/CTO/стейкхолдер). Каждый KR ← проблема из As-Is + образ результата. Без источника → `[УТОЧНИТЬ]`.

### 2. Формулы (из BEST-PRACTICES-OBJ-KR.md)

**OBJ:** `[Действие] [Продукт] от [Система] ([Контекст])`
Короткий (≤10 слов), понятен CPO/CTO/CFO без объяснений.

**KR:** `[Действие] [Чей продукт] на/через [Канал] минуя [Система]`
KR = результат, не активность. Метрики — в "Образ результата", не в названии.

### 3. IMP-ранжирование
IMP (1–9) отражает бизнес-ценность + критичность. Critical KR (IMP ≥ 8) — обязательный коммитмент и дедлайн. Не более 2 Critical KR на весь квартал.

### 4. Образ результата = доказуемое состояние
Не «улучшить/обеспечить/повысить». Формат: `Что видит пользователь/стейкхолдер` + для технических KR: `БЫЛО: [dataflow] → СТАЛО: [dataflow]`. Прогресс-шкала: `Собраны требования — 10% / … / Фича в проде — 100%`.

### 5. Образ действия = роли + шаги + план Б
Шаги по ролям: `[BA]`, `[SA]`, `[BE]`, `[FE]`, `[QA]`, `[RELEASE]`, `[ADR]`, `[PO]`. Каждый шаг атомарен. Блокеры и зависимости явно помечены `[BLOCKER]` / `[RISK]`.

### 6. Зависимости явные
Связи между KR (`блокирует KR X.Y`, `зависит от KR X.Z`). Зависимости от внешних команд — отдельно с `[RISK]`, и каждая внешняя команда должна присутствовать в снимке ландшафта (`{landscape.snapshot_doc}`, стадия 0.5), а не абстрактный «внешний». Enforcement — гейт 13 (`hard_gates.md`).

### 7. Теги KR
`[RESEARCH]` — KR-исследование, образ результата = план/решение. `[POC]` — proof-of-concept. `[BUG]` — дефект продакшна. `[ACTIVITY]` — регулярная поддержка. Без тега = фича/интеграция.

### 8. Нет TBD в Critical/High KR
`[УТОЧНИТЬ]` = допустимо для IMP ≤ 3. IMP ≥ 7 → образ результата обязан быть заполнен полностью.

### 9. Adversarial (отдельная стадия)
Перед фиксацией KR — red-teaming (`/okr-debate`): реалистичность ресурсов, зависимости, измеримость, риск scope-creep. Без прохода дебатов — validate выставит 🔴.

### 10. Количество OBJ и KR
3–5 OBJ на квартал. 2–6 KR на OBJ. Итого не более 20 KR — иначе нет фокуса.

### 11. Груминг всегда опирается на якорный артефакт (Фаза 2)
`/okr-grooming` не стартует без зафиксированного якоря — SMART-соглашения (`status: ЗАФИКСИРОВАН`) или финализированного БФТ. Якорь фиксирует образ результата и границы объёма (`ВНУТРИ`/`СНАРУЖИ`), вокруг которых ведётся диалог. Нет артефакта → сперва `/okr-smart` (или БФТ), затем груминг. Enforcement — гейт G-G1.

### 12. Каждый вопрос исследования ← артефакт (Фаза 2)
Вопросы `ADR#/SA#/BA#/DEL#` опираются на якорь / БФТ / context-pack. Вопрос без артефакта-основы не выдумывается — выносится в «⚠ Требуют артефакта» (груминг по ветке заморожен до создания артефакта). Enforcement — гейт G-G2.

---

## Главное правило процесса

**STOP после каждой стадии.** Не лети до финала. Вывести артефакт → ожидать решения PO. Только так pipeline даёт quality, а не «генерация за один промт».

**Опрос — по одному вопросу за раз**, особенно на стадиях 0 и 1. Образ результата PO формируется диалогом, а не монологом системы.
