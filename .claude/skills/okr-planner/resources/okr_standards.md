# OKR Standards — форматы, шкалы, структуры

## IMP-шкала (Importance)

| IMP | Уровень | Значение |
|-----|---------|---------|
| 9 | Critical | Коммитмент перед топ-менеджментом/партнёрами. Дедлайн зафиксирован. Провал = эскалация. |
| 7–8 | High | Стратегический приоритет квартала. Должно быть сделано, иначе квартал не закрыт. |
| 5–6 | Medium | Важно, но есть запас по срокам. При ресурсном конфликте — уступает Critical/High. |
| 3–4 | Low | Желательно. Берётся только при наличии capacity после Critical+High. |
| 1–2 | Backlog | Исследование / гипотеза. В квартал только при явном решении PO. |

Не более 2 KR с IMP = 9 на квартал. Если их больше — пересмотреть приоритеты.

## Теги KR

| Тег | Что означает | Образ результата |
|-----|-------------|-----------------|
| (без тега) | Фича / интеграция | Рабочая система в продакшне |
| `[POC]` | Proof of concept | Согласованный план + PoC или детальная декомпозиция |
| `[RESEARCH]` | Исследование | Понятный согласованный план дальнейших действий |
| `[BUG]` | Дефект продакшна | Баг устранён, регресс не введён |
| `[ACTIVITY]` | Регулярная активность | Поддержка работает без инцидентов |

## Теги ролей (образ действия)

Базовые роли-теги шага: `[BA] [SA] [BE] [FE] [QA] [RELEASE] [ADR] [PO]` (резолвятся из `role_tags` профиля).

| Модификатор | Что означает |
|-------------|-------------|
| `[EXT]` | Шаг исполняется **внешней командой** (партнёр / смежная система). Ставится рядом с ролью: `[BE][EXT]`, `[FE][EXT]`, `[ADR][EXT]`. Сигнал кроссфункциональной зависимости и риска по срокам. |
| `[BLOCK]` / `[BLOCKER]` | Жирный блокер / зависимость от другой задачи. |

## Структура артефакта: итоговая таблица OKR

Финальный документ (`OKR-{quarter}.md`) содержит:

### Заголовок
```
Проектные цели:
- **OBJ 1**: [Название]
- **OBJ 2**: [Название]
...

Регулярные активности:
- [Активность 1]
```

### Основная таблица
Колонки (порядок фиксированный):
```
| OBJ | KR | IMP | Название | Образ результата | Образ действия | Метрики&риски&зависимости |
```

### Образ результата — обязательные элементы
- Для фич/интеграций: описание что видит пользователь/стейкхолдер
- Для технических KR: `БЫЛО: [dataflow] → СТАЛО: [dataflow]`
- Прогресс-шкала (если KR многошаговый):
  `Собраны требования — 10% / Согласована документация — 30% / Фича разработана — 50% / Пройдено ревью — 70% / Закончены тесты и отладка — 90% / Фича в проде — 100%`
- Критичность/коммитмент (если есть): `**Критичность:** коммитмент перед {кем} на {месяц}`

### Образ действия — структура
Шаги по ролям, блокеры и зависимости:
```
- [BA] Название шага
- [SA] Название шага
- [ADR] Согласование решения
- [BE] Название шага
- [FE] Название шага
- [QA] Отладка
- [RELEASE] Пошаговая раскатка
- [BLOCK] Название блокера (жирный блокер)
- [BLOCKER] Зависимость от другой задачи

Зависимости:
- [RISK] Описание риска
```

### Метрики&риски&зависимости
- Оценка сроков (диапазон в неделях)
- Технические риски
- Кроссфункциональные зависимости
- Стейкхолдеры согласования

## Фаза 2 — Груминг с лидами (форматы)

> Груминг (`/okr-smart` → `/okr-grooming`) запускается ПОСЛЕ утверждённого первичного OKR и углубляет проработку силами аналитиков, лидов направлений и архитекторов. **Груминг всегда опирается на якорный артефакт** (SMART-соглашение или БФТ) — без него `/okr-grooming` не стартует. Эталон вывода — `examples/ideal_grooming.md`.

### SMART-соглашение (якорь груминга)

Лёгкий артефакт, фиксирующий образ результата одного KR перед грумингом. Создаётся `/okr-smart`, статус `ЗАФИКСИРОВАН`. Поля:
- **Образ результата** — копия из утверждённого OKR (с якорем на `{okr_output_doc}`).
- **SMART-разбор** — S (Specific) / M (Measurable, критерий демо-прод) / A (Achievable) / R (Relevant, какой OBJ) / T (Time-bound, срок + источник).
- **Границы объёма** — `ВНУТРИ` / `СНАРУЖИ` (фиксирует объём диалога).
- **Контекст диалога** — команды/системы в орбите (внешние → кандидаты на `[EXT]`), стейкхолдеры.
- **БЫЛО → СТАЛО** — для технических KR.
- **Якоря** — таблица «утверждение → источник».

Альтернатива: при наличии финализированного БФТ в `{bft_store}` — `anchor_type: БФТ` + ссылка `bft_ref` (БФТ — более сильный якорь).

### Образ действия — US-декомпозиция

Груммированный образ действия = общий префикс + блоки user stories:
```
Общий префикс:
- [BA] Сбор и описание БФТ
- [SA] Первичная аналитика
- [SA] Декомпозиция и разработка

US{k}: [название user story]
- [BE][EXT] [шаг внешней команды]
- [BE] [шаг]
- [FE][EXT] [шаг]
- [QA] Отладка по US{k}
- [RELEASE] Отгрузка US{k} в Production
```
Правила: `[EXT]` — на каждый шаг внешней команды; каждый US-блок закрывается `[QA] Отладка по US{k}` + `[RELEASE] Отгрузка US{k}`. Для `[RESEARCH]`/`[POC]` US-декомпозиция может отсутствовать — центр тяжести смещается в «Вопросы исследования».

### Вопросы исследования (реестр)

Нумерованный реестр открытых вопросов. Префикс = ответственная роль:
| Префикс | Роль | Пример |
|---------|------|--------|
| `ADR#` | Архитектор (архитектурное решение) | `ADR1: TC отличается от первичного представления` |
| `SA#` | Системный аналитик | `SA8: Следствие US7 → как меняем Dataflow?` |
| `BA#` | Бизнес-аналитик | `BA1: Какие бизнес-договорённости нужны?` |
| `DEL#` | Delivery / devops / инфраструктура | `DEL12: Как переезжают авторизации сервисов?` |

Кросс-ссылки `US#` связывают вопрос с user story. **Правило артефакта:** каждый вопрос опирается на якорь / БФТ / context-pack. Вопрос без артефакта-основы → блок «⚠ Требуют артефакта» (груминг по ветке заморожен до создания артефакта), не выдумывать.

## Workspace (папки артефактов)

```
{okr_workspace}/
├── context-pack.md       (стадия 0)
├── objectives.md          (стадия 1)
├── key-results.md         (стадия 2, обновляется после дебатов)
├── enriched-okr.md        (стадия 4)
└── validation.md          (стадия 5)

{okr_grooming_root}/                  (Фаза 2 — груминг, 1 степ = 1 KR)
└── step_{n}/
    ├── okr_step_{n}.md               (/okr-grooming — груммированный образ действия)
    └── artifacts/
        ├── smart-agreement.md        (/okr-smart — якорь, статус ЗАФИКСИРОВАН)
        └── research-questions.md     (/okr-grooming — реестр вопросов как backlog)

{planning_root}/
├── {okr_output_doc}        (финальный артефакт, стадия 6)
├── {index_doc}             (обновляется: KR-таблица)
└── {kr_epic_map_doc}       (обновляется: новые KR → TODO эпики)
```

## Naming OBJ / KR

**OBJ formula:** `[Действие] [Продукт] от [Система] ([Контекст])`
- Длина ≤ 10 слов
- Понятен CPO/CTO/CFO без пояснений
- Контекст в скобках: `(виджеты&контент)`, `(B2B)`, `(Кино)`

**KR formula:** `[Действие] [Чей продукт] на витрине/через [Канал] минуя [Система]`
- KR = результат, не активность
- Метрики и цифры — в "Образ результата", не в названии KR
- Если KR пока не готов → тег `[RESEARCH]` или `[POC]`

---

## Нумерация

- OBJ: `OBJ 1`, `OBJ 2`, … (порядок = убывание стратегической важности)
- KR: `{OBJ}.{номер}`, например `1.1`, `1.2`, `2.1`
- Если KR уточняется → суффикс: `1.6.2`
- Неопределённые KR (конфликт нумерации): `⚠?` в поле KR + пометка

## Confidence для capacity planning

| Уровень | Когда | Коэффициент в план |
|---------|-------|-------------------|
| High | Детализирован, зависимости известны | 100% |
| Medium | Частично проработан, есть вопросы | 70% |
| Low | Гипотеза, объём неизвестен | 50% |
