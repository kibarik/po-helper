# OKR Standards — форматы, шкалы, структуры

## IMP-шкала (Importance)

| IMP | Уровень | Значение |
|-----|---------|---------|
| 9 | Critical | Коммитмент перед топ-менеджментом/партнёрами. Дедлайн зафиксирован. Провал = эскалация. |
| 7–8 | High | Стратегический приоритет квартала. Должно быть сделано, иначе квартал не закрыт. |
| 5–6 | Medium | Важно, но есть запас по срокам. При ресурсном конфликте — уступает Critical/High. |
| 3–4 | Low | Желательно. Берётся только при наличии capacity после Critical+High. |
| 1–2 | Backlog | Исследование / гипотеза. В квартал только при явном решении PO. |

Не более 2 KR с IMP = 9 на квартал. Если их больше — пересмотреть приоритеты.

---

## Теги KR

| Тег | Что означает | Образ результата |
|-----|-------------|-----------------|
| (без тега) | Фича / интеграция | Рабочая система в продакшне |
| `[POC]` | Proof of concept | Согласованный план + PoC или детальная декомпозиция |
| `[RESEARCH]` | Исследование | Понятный согласованный план дальнейших действий |
| `[BUG]` | Дефект продакшна | Баг устранён, регресс не введён |
| `[ACTIVITY]` | Регулярная активность | Поддержка работает без инцидентов |

## Структура артефакта: итоговая таблица OKR

Финальный документ (`OKR-{quarter}.md`) содержит:

### Заголовок
```
Проектные цели:
- **OBJ 1**: [Название]
- **OBJ 2**: [Название]
...

Регулярные активности:
- [Активность 1]
```

### Основная таблица
Колонки (порядок фиксированный):
```
| OBJ | KR | IMP | Название | Образ результата | Образ действия | Метрики&риски&зависимости |
```

### Образ результата — обязательные элементы
- Для фич/интеграций: описание что видит пользователь/стейкхолдер
- Для технических KR: `БЫЛО: [dataflow] → СТАЛО: [dataflow]`
- Прогресс-шкала (если KR многошаговый):
  `Собраны требования — 10% / Согласована документация — 30% / Фича разработана — 50% / Пройдено ревью — 70% / Закончены тесты и отладка — 90% / Фича в проде — 100%`
- Критичность/коммитмент (если есть): `**Критичность:** коммитмент перед {кем} на {месяц}`

### Образ действия — структура
Шаги по ролям, блокеры и зависимости:
```
- [BA] Название шага
- [SA] Название шага
- [ADR] Согласование решения
- [BE] Название шага
- [FE] Название шага
- [QA] Отладка
- [RELEASE] Пошаговая раскатка
- [BLOCK] Название блокера (жирный блокер)
- [BLOCKER] Зависимость от другой задачи

Зависимости:
- [RISK] Описание риска
```

### Метрики&риски&зависимости
- Оценка сроков (диапазон в неделях)
- Технические риски
- Кроссфункциональные зависимости
- Стейкхолдеры согласования

## Naming OBJ / KR

**OBJ formula:** `[Действие] [Продукт] от [Система] ([Контекст])`
- Длина ≤ 10 слов
- Понятен CPO/CTO/CFO без пояснений
- Контекст в скобках: `(виджеты&контент)`, `(B2B)`, `(Кино)`

**KR formula:** `[Действие] [Чей продукт] на витрине/через [Канал] минуя [Система]`
- KR = результат, не активность
- Метрики и цифры — в "Образ результата", не в названии KR
- Если KR пока не готов → тег `[RESEARCH]` или `[POC]`

---

## Нумерация

- OBJ: `OBJ 1`, `OBJ 2`, … (порядок = убывание стратегической важности)
- KR: `{OBJ}.{номер}`, например `1.1`, `1.2`, `2.1`
- Если KR уточняется → суффикс: `1.6.2`
- Неопределённые KR (конфликт нумерации): `⚠?` в поле KR + пометка

## Confidence для capacity planning

| Уровень | Когда | Коэффициент в план |
|---------|-------|-------------------|
| High | Детализирован, зависимости известны | 100% |
| Medium | Частично проработан, есть вопросы | 70% |
| Low | Гипотеза, объём неизвестен | 50% |
