# Эталонный груминг-степ — Фаза 2 (аннотированный)

> Образец артефакта `okr_step_{n}.md`, который генерирует `/okr-grooming`. Аннотации `<!-- -->` — пояснения для LLM, в финал не включаются.
>
> **Предметная область — иллюстративная** (билетная экосистема: TC = внешний партнёр, TL/BI/БАЗИС/DBA — внутренние системы). Доменные имена замените на свои из `.claude/domain-profile.md`. Универсальна **структура**: общий префикс + US-блоки, тег `[EXT]`, нумерованные вопросы `ADR#/SA#/BA#/DEL#`, risk-register.
>
> **Предусловие:** каждый степ опирается на зафиксированный якорь (`artifacts/smart-agreement.md` или БФТ). Без якоря `/okr-grooming` не стартует.

---

## Пример 1 — зрелый KR с US-декомпозицией

<!-- Образ действия: общий префикс (BA/SA) → US-блоки. [EXT] на шагах внешней команды (TC). Каждый US закрывается [QA]+[RELEASE]. -->

```
Общий префикс:
- [BA] Сбор и описание БФТ
- [SA] Первичная аналитика
- [SA] Декомпозиция и разработка

US1: Отображение заказов
- [BE][EXT] Доработки на стороне TC по Kafka
- [BE] Сохранение заказов
- [BE] Доработка BFF для вывода заказов
- [FE][EXT] Доработка логики отображения заказов
- [QA] Отладка по US1
- [RELEASE] Отгрузка US1 в Production

US2: Возвраты
- [SA] Сбор и описание БФТ/СТ/ADR
- [ADR] Согласование дизайна решений
- [ADR][EXT] Согласование плана и требований с TC
- [BE] Разработка механизма возвратов
- [BE] Доработка BFF
- [FE][EXT] Доработка логики отображения в ЛК TL
- [QA] Отладка по US2
- [RELEASE] Отгрузка US2 в Production

US3: Обмен билетов
- [SA] Сбор и описание БФТ/СТ/ADR
- [ADR] Согласование дизайна решений
- [ADR][EXT] Согласование плана и требований с TC
- [BE] Разработка механизма обмена
- [BE] Доработка BFF
- [FE][EXT] Доработка логики отображения в ЛК TL
- [QA] Отладка по US3
- [RELEASE] Отгрузка US3 в Production
```

### Риски
- [RISK] US2, US3 не исследованы на момент старта квартала, объём может изменяться
- [RISK] Задержки на стороне TC по Kafka
- [RISK] Юридические риски пользовательских соглашений — задержка релиза до правок
- [RISK] Сроки слетят из-за Legacy при коннекте TC и TL.DBA
- [RISK] Сетевая доступность между TC, TL, UMC
- [RISK] CI/CD процессы из-за переезда
- [RISK] Для US2, US3 могут потребоваться доп. работы на TC, о которых сейчас не знаем
- [RISK] Может потребоваться замена API для TL.Frontend, объём не понятен
- [RISK] Задержка релиза из-за кроссфункциональных взаимодействий TC, TL, Devops

<!-- ПАТТЕРН: US2/US3 ещё не исследованы → часть рисков прямо говорит «объём может меняться». Это норма: риск зафиксирован явно, а не спрятан. -->

## Пример 2 — исследовательский KR ([RESEARCH]/[POC]): центр тяжести в вопросах

<!-- Зрелой US-декомпозиции нет — KR про передачу BI-модуля во внешнюю систему TC. Образ действия минимален, основное — реестр вопросов. -->

```
Общий префикс:
- [ADR] BI-шлюзы: стратегия интеграции с TC
- [ADR][EXT] Согласование дизайна TC/BI
```

### Вопросы исследования
<!-- Префикс = ответственная роль. Кросс-ссылки US#. Каждый вопрос ← артефакт-основа (здесь: SMART-якорь + context-pack). -->

```
ADR1: TC архитектурно отличается от первичного представления Q1, Q2
ADR2: Следствие ADR1 → нужно согласовать дизайн передачи BI-модуля
ADR3: Как изолировать BI-модуль от БАЗИСа и передать в TC?
SA1: Как сделать BI независимым от БАЗИС?
SA2: Как подключить BI на вход к TC?
SA3: Нужно ли мигрировать историч. данные из BI в TC и как?
SA4: Какие бизнесовые договорённости нужны?
SA5: Что делаем с бронированием заказов в TC и БАЗИС?
SA6: Что становится со шлюз. мероприятиями, если BI больше не в БАЗИС?
SA7: Если шлюзовых (BI) мероприятий нет в БАЗИС — какой эффект на TL, GDS, Live?
SA8: Следствие US7 → как меняем Dataflow для сохранения продаж без BI?
SA9: Следствие US7 → как меняется бизнес-модель экосистемы?
SA10: Как и в какие сроки организаторы переезжают с БАЗИС в TicketsCloud?
SA11: Как происходит миграция админок и бизнес-процессов без шлюзов?
SA12: Есть ли связь BI-модуля с оповещениями (Email, пуши, номер)?
DEL12: Как происходит переезд авторизаций сервисов?
```

### Риски
- [RISK] Научно-исследовательская работа без гарантий результата
- [RISK] Кто и как проводит исследование со стороны TC?
- [RISK] Кто и как проводит регресс BI-модуля?
- [RISK] Кто и как проводит приёмочное тестирование со стороны TC?
- [RISK] Legacy BI-сервиса без автотестов
- [RISK] Дебаты между лидами BI, TC, DBA, TL о дизайне перехода
- [RISK] Сетевая доступность между TC, BI, TL
- [RISK] Не знакомы с культурой и архитектурой TC
- [RISK] Не понятен объём работ на стороне TC
- [RISK] Не понятен объём работ на стороне BI

## Пример 3 — узкий research-KR

```
### Вопросы исследования
SA1: Что делать с модулем авторизации extapi?
SA2: Как мигрировать и адаптировать правила доступности под TC?
SA3: Как переедут мероприятия из БАЗИС в TC? → сильно зависит переезд extapi_go
```

### Риски
- [RISK] Научно-исследовательская работа без гарантий результата
- [RISK] Сейчас не описаны критерии результата → реализация не понятна
- [RISK] extapi — монолитный сервис, нельзя переносить частями
- [RISK] extapi сильносвязан с Live.Платформой, логикой BI-шлюзов и TicketSale
- [RISK] Наполнение и дистрибуция extapi_go сильносвязаны с CDC и структурой БД
- [RISK] Нет доменной экспертизы по TicketsCloud
- [RISK] Сетевая доступность между TC, TL, DBA, БАЗИС из разных корпоративных контуров
