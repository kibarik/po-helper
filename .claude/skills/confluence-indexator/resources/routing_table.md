# Таблица маппинга — Confluence → node_type → нексус

Ядро стадии `/cindex-route`. Роутинг — **только** в существующий реестр
(`GROUND/NEXUS/_registry.yaml`); новые нексусы не создаются. Источник —
дизайн-спека §5, значения ниже дословно.

| Сигнал на странице | node_type | Целевой нексус |
|---|---|---|
| ADR / «Решение», «выбрали X потому что» | `decision` | `precedents` (или `system-landscape` для тех-решений) |
| Бизнес-правило BR-*, «система должна» | `rule` | `requester-domain` |
| Термин глоссария, Ubiquitous Language | `term` | `requester-domain` / `product` |
| Метрика, KPI, целевое значение | `metric` | `strategy` (KR) / `growth` |
| Сервис/API/интеграция/поток данных | `component` | `system-landscape` |
| Требование закон/стандарт/политика/security | `regulation` | `compliance` |
| SLA/НФТ/производительность | `nfr` | `compliance` |
| Риск, угроза, «что если» | `risk` | `problem` / `capacity` |
| Персона, роль, владелец, RACI | `person` | `team` |
| Внешняя команда, зона ответственности | (ext-team) | `landscape` / `ownership` |
| Сегмент, JTBD, боль клиента | (empirical) | `customer` |
| Конкурент, рыночная динамика | (empirical) | `market` |
| Roadmap, OKR, приоритет квартала | (empirical) | `strategy` |
| Ничего из выше / уверенность низкая | — | `_intake/unrouted.md` (очередь PO) |

## Правило множественности и неоднозначности

Одна страница Confluence может дать **несколько узлов разных строк таблицы**
(семантический разбор постранично, не «страница = узел»). Каждый узел
классифицируется независимо и может попасть в разный нексус; связь между
такими узлами сохраняется через `[[wiki-links]]` (см. `linking_rules.md`),
не через совместное размещение.

Строки с двумя нексусами через «/» (`decision`, `term`, `risk`) разрешаются
Router'ом по контексту страницы (тех-решение vs бизнес-решение, платформенный
термин vs домен заказчика, продуктовый риск vs риск мощности). **При
неоднозначности между двумя нексусами** — не гадать: понизить
`route_confidence` (см. `confidence_rules.md`) и увести узел в очередь
`_intake/unrouted.md` на решение PO.

## Правила применения

1. Роутинг — только в slug из `GROUND/NEXUS/_registry.yaml`. Нет
   зарегистрированного нексуса под сигнал → строка «Ничего из выше» →
   `_intake/unrouted.md`.
2. Таблица — единственный источник соответствий для Router'а; расширять
   строками можно только синхронно с обновлением спеки и `_registry.yaml`.
3. `node_type` для строк без явного значения в скобках (`(ext-team)`,
   `(empirical)`) выбирается Digitizer'ом по природе сущности — см.
   `sa_documentation/nexus_schema.md` §3 для полного списка `node_type`.

   Честно: строки с плейсхолдером `(empirical)`/`(ext-team)` пока не имеют
   канонического node_type в `nexus_schema.md` §3 → такие узлы уходят в
   очередь `_intake/` до расширения схемы (открытый вопрос дизайна, спека §10).
