# Разрыв ожиданий — As-Is / Проблема / To-Be

> Стадия `/req-gap` превращает ImpactMap в **конкретное продуктовое изменение** — краткое и понятное. Механика: показать текущее состояние (As-Is), назвать разрыв (Проблема), описать целевое (To-Be). Высота — результат и приёмка, не реализация.

## Три блока

### As-Is (как сейчас)
- Как процесс/продукт работает на текущий момент в зоне запроса.
- Только подтверждённое: цитата запрашивающего (R1) о том, как он делает сейчас, + домен команды (R2: NEXUS/CORTEX/JIRA).
- Не додумывать реализацию → `[УТОЧНИТЬ]`.

### Проблема (разрыв)
- В чём конкретно разрыв между текущим и желаемым. Не «плохо», а измеримая/наблюдаемая боль.
- Каждая проблема ← цитата (R1). Цена проблемы — из интервью (время/деньги/риск).
- Это **диагноз**, не решение. Запрещено «надо добавить X».

### To-Be (как должно стать)
- Целевое наблюдаемое состояние — на языке **результата и пользователя**, не системы.
- Что изменится с точки зрения запрашивающего, когда задача решена.
- Включает **образ приёмки**: по каким признакам стейкхолдер поймёт, что готово (из Блока 5 интервью). Нет образа приёмки → `[УТОЧНИТЬ]`, запрос не созрел.

## Итог стадии — краткое продуктовое изменение

После трёх блоков — **одно-двухфразовая формулировка изменения**, понятная без контекста:
> «Сделать сверку выгрузки с биллингом автоматической: пользователь видит расхождения в одном отчёте, закрытие месяца не требует ручной сверки.»

Это «короткое и понятное продуктовое изменение» из реального процесса PO (шаг 2). Оно — мост к SMART-задаче.

## Правила

1. **As-Is заякорено**, не выдумано. Текущее поведение ← R1 (как делает запрашивающий) / R2 (домен).
2. **Проблема — диагноз, не решение.** Никаких «надо сделать». Решение придёт в SMART.
3. **To-Be — на высоте результата.** Без API/таблиц/User Stories (это `bft-writer`). Принцип altitude.
4. **Образ приёмки обязателен** в To-Be. Без него → `[УТОЧНИТЬ]`.
5. **Краткость.** Итоговое изменение — 1-2 фразы, без жаргона, понятно заказчику.

## Шаблон

```markdown
# Разрыв ожиданий — <request_id>

## As-Is (как сейчас)
<текущее состояние/процесс> — якорь R1/R2

## Проблема (разрыв)
- <боль 1, измеримая> — R1
- <боль 2> — R1
Цена: <время/деньги/риск> — R1

## To-Be (как должно стать)
<целевое наблюдаемое состояние, язык результата> — R1
Образ приёмки: <по каким признакам готово> — R1 / [УТОЧНИТЬ]

## Краткое продуктовое изменение
> <1-2 фразы, понятно без контекста>

## Открытые вопросы
- [УТОЧНИТЬ]: <…>
```
