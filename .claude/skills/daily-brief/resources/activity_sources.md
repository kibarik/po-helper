# Источники активности — порядок сбора и деградация

`/daily-brief` собирает «кто что сделал за окно» из нескольких источников, каждый read-only. Порядок фиксирован: сначала самый богатый (трекер), затем сверочные (PULSE, Backlog), затем опциональные (git). Недоступный источник не валит прогон — помечается `[НЕДОСТУПНО]` и сводка деградирует.

## Лестница источников

### 1. Трекер (JIRA) через MCP — главный источник

- **Только `access: read-only`.** Навык не делает переходов статусов, не пишет коменты, ничего не создаёт. Если профиль в режиме `build` — навык всё равно читает как read-only.
- **Доступ строго через MCP** (`tracker.mcp`). Токены/PAT руками не собирать. Нет MCP/VPN → `[НЕДОСТУПНО]`, деградация на vault (шаг 2–3).
- **Что тянуть:** карточки проектов `tracker.projects`, обновлённые в окне (`updated >= <начало окна>`), по каждому исполнителю: текущий статус, переход за окно (если виден в истории), закрытия, свежие коменты. Ориентир JQL: `project in (...) AND updated >= "<окно>" ORDER BY assignee`.
- **Агрегация по человеку:** сгруппировать карточки по `assignee` → одна строка на человека (несколько карточек через `·`). Исполнитель → `@username` через People Graph.
- **Стейлнесс:** для карточек «в работе», не обновлённых ≥ порога (`reminder_horizon_days`, дефолт 2 дня) — пометка «N дней без движения» (⚠️).

### 2. Прошлый снапшот `/daily-review` (PULSE) — сверка обязательств

- Последняя папка в `paths.meeting_notes` (`daily-*` / `planning-*`). Читать `commitments.md` (кто-что-к-сроку) и `blockers.md` (что мешало).
- Для каждого вчерашнего обязательства: сопоставить с сегодняшней активностью трекера → `выполнено ✅ / поехало ⚠️ / в работе 📌`. Формулировать вопросом к дейлику.
- Нет папки (первый запуск) → пропустить сверку, сводка только из трекера. Не выдумывать вчерашние обязательства.

### 3. Backlog PO (`planner`) — что висит

- Читать открытые задачи с labels `commitment` / `blocker` / `agreement` (`planner.labels`). Срок парсится из описания (в Backlog.md нет поля `due`).
- Просроченные и «скоро-по-сроку» (`reminder_horizon_days`) → блок «На кого смотреть» и «План PO на дейлик».
- `planner.type: none` или нет MCP `backlog` → пропустить, работать без слоя Backlog.

### 4. People Graph (`GROUND/NEXUS/team`) — якорение и навигация

- Имя исполнителя → `@username`/`handle` из person-узла. Не сматчилось → `[КТО?]`, не додумывать.
- «К кому идти» по вопросу/блокатору — для блока «План PO на дейлик» (`/people-map`).

### 5. Sprint Goal + KR — группировка

- Sprint Goal ← план спринта (`planning_root`/`sprint_output_doc`). KR отгрузки ← `okr_output_doc`, привязка карточка→KR ← `kr_epic_map_doc`.
- Каждая строка активности тегируется `на цель / по касательной / мимо / вне плана`; строки группируются под заголовками KR/цели. Не сматченный KR → `[KR?]`.

### 6. Git / GitHub (опционально, по умолчанию ВЫКЛ)

- Включается флагом (`/daily-brief --git`) или профилем. Тянет коммиты/PR/ревью за окно по авторам (`git log --since`, GitHub MCP).
- Даёт инженерный срез, но **не покрывает не-кодовую работу** (дизайн, аналитика, созвоны) — поэтому вспомогательный, не основной. Автор коммита → `@username` через People Graph.

## Лестница деградации

| Что доступно | Как собирается сводка | Пометка в шапке |
|---|---|---|
| Трекер + PULSE + Backlog | Полная: активность из трекера, сверка обязательств, догляд PO | `источники: jira+pulse+backlog` |
| Трекер, нет PULSE | Активность из трекера, без сверки «обещали→сделали» | `источники: jira` |
| Нет трекера, есть PULSE + Backlog | По последнему дейлику и открытым обязательствам; строка `Трекер [НЕДОСТУПНО]` | `источники: vault-only` |
| Ничего | Честно сообщить PO: нет источников, сводку не собрать | — |

## Правила

- **Никаких записей.** Ни в трекер, ни в Backlog. Единственная запись — снапшот `brief.md` в `brief_notes` и вывод в чат.
- **Источник на строку.** Каждая строка активности заякорена (ключ JIRA / строка PULSE / задача Backlog / коммит). Нет якоря → плейсхолдер.
- **Недоступность честная.** Нет MCP/VPN → `[НЕДОСТУПНО]` и деградация, а не выдуманный состав досок.
