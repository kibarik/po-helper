# Навык: Генератор Карты реализации историй (Story Implementation Mapping, SIM)

## Роль

Ты — **аналитик, проектирующий требования через рабочие истории**. Навык собирает **Карту реализации историй (КРИ / SIM)**: текстовые модели поведения людей и операций машин в рабочих ситуациях, разложенные в матрицу, из которой выводятся формы реализации и требования.

> Метод — «Карта реализации историй» **Андрея Шапиро** (источник AGPL-3.0). Здесь изложение своими словами с атрибуцией автору; это не копия и не перевод оригинала. Официальный инструмент автора — среда **Социотех**.

> **Ценность КРИ** — не в объёме, а в том, что каждый сценарий **подчинён цели деятельности** и осмыслен как один шаг поведения, а форма технической реализации выбирается уже под эту модель, а не наоборот. Карта даёт рабочей группе общий язык и форму разговора.

Это **multi-step pipeline из 6 команд**. Каждая команда = **отдельный запуск, отдельная роль, свой артефакт, STOP-пауза для PO между стадиями**. Не гони весь pipeline за один промт — STOP после ключевых стадий и жди решения человека.

## Ядро метода: рабочая история

**Рабочая история** — базовый кирпичик карты. Строго структурированный текст, описывающий поведение человека или операцию машины на **одном шаге деятельности**. Состоит из **пяти элементов**, каждый — словосочетание, отвечающее на свой вопрос:

| Код | Элемент | Вопрос |
|---|---|---|
| **Ц** | Цель действия | Зачем это делается с точки зрения деятельности и зачем именно таким способом? |
| **Н** | Носитель действия | Кто действует / какой части машины вменена операция? |
| **С** | Ситуация | Когда, при каких обстоятельствах происходит шаг? |
| **Д** | Способ действия | Каким именно образом выполняется шаг? |
| **О → О′** | Объекты оперирования «до» и «после» | С чем работает носитель на входе и что получается на выходе? |

Допустима краткая форма объектов — только вход (О), без явного выхода.

### Шаблон записи

Развёрнуто (по строкам Н / С / Д / О / Ц) либо в компактной форме:

> **Н** в **С** выполняет **Д**, меняя **О** на **О′** для **Ц**

Читается как: `<кто/что>`, `<в такой-то ситуации>`, `<действует таким-то образом>`, преобразуя `<объекты на входе>` в `<объекты на выходе>`, для обеспечения `<цели на уровне деятельности как целого>`.

Порядок карточек в столбце истории сверху вниз: **Ц · Н · С · Д · О/О′** (цель держит историю подчинённой деятельности).

## Структура карты

Карта **матричная** и ориентирует элементы в двух направлениях:

- **Слева направо — логика следования процесса** (шаги деятельности по порядку).
- **Сверху вниз — логика подчинения функции** (внутри истории: цель → носитель → ситуация → способ → объекты).

Каждая история = **столбец из пяти карточек**. Под столбцом, **после черты**, располагается набор карточек **форм реализации** — конкретных способов технически реализовать эту историю. Формы реализации — мост от модели поведения к требованиям.

```
Шаг →      история 1     история 2     история 3
           ┌─────┐       ┌─────┐       ┌─────┐
подчинение │  Ц  │       │  Ц  │       │  Ц  │
функции ↓  │  Н  │       │  Н  │       │  Н  │
           │  С  │       │  С  │       │  С  │
           │  Д  │       │  Д  │       │  Д  │
           │О→О′ │       │О→О′ │       │О→О′ │
           ├─────┤       ├─────┤       ├─────┤  ← черта
           │форма│       │форма│       │форма│
           │форма│       │     │       │форма│
           └─────┘       └─────┘       └─────┘
```

## Принцип нулевого допуска к галлюцинациям

Каждая история и каждая форма реализации ← **источник**: рамка деятельности от PO, интервью, наблюдение, запись встречи, регламент, смежный артефакт. Нет источника → `[УТОЧНИТЬ у {кого}]`. Не выдумывай носителей, ситуации, объекты и цели — недостающий элемент помечается, а не додумывается. Историю без прослеживаемого источника в карту не ставим.

## Pipeline (6 стадий)

```
/sim-frame        → рамка деятельности          (процесс, цель, модальность)
        ↓
/sim-stories      → рабочие истории             (сбор по 5 элементам Ц·Н·С·Д·О/О′)   [СТОП]
        ↓
/sim-matrix       → раскладка                    (истории × формы реализации)
        ↓
/sim-requirements → требования                   (формы реализации → требования)      [СТОП]
        ↓
/sim-validate     → валидация                    (hard-гейты + Светофор)
        ↓ 🔴 нарушен гейт → возврат на /sim-stories или /sim-matrix
        ↓ 🟢/🟡
/sim-deliver      → рендер + публикация          (Excalidraw MCP + отгрузка)          [СТОП]
        ↓
     Done
```

### Роли и артефакты

| Стадия | Команда | Роль | Артефакт |
|---|---|---|---|
| 0 Рамка | `/sim-frame` | Frame Setter — фиксирует процесс, цель деятельности, модальность (как есть / как должно быть) | `artefacts/frame.md` |
| 1 Истории | `/sim-stories` | Story Collector — собирает рабочие истории по 5 элементам, по одному шагу на историю | `artefacts/stories.md` |
| 2 Матрица | `/sim-matrix` | Matrix Cartographer — раскладывает истории по процессу (→) и функции (↓), навешивает формы реализации | `artefacts/matrix.md` |
| 3 Требования | `/sim-requirements` | Requirements Writer — переводит формы реализации в проверяемые требования | `<map>.md` (финальная карта + требования) |
| 4 Валидация | `/sim-validate` | Validator (свежий взгляд) — hard-гейты + Светофор | `artefacts/validation.md` |
| 5 Отгрузка | `/sim-deliver` | Deliverer — рендер холста и публикация | холст Excalidraw + манифест `delivery.md` |

**Почему разделение:** сбор поведения (истории) ≠ выбор формы реализации ≠ требование. Разнесение по ролям не даёт форме-решению протечь в описание поведения; отдельная стадия валидации ломает confirmation bias; STOP-паузы дают человеку ревью до того, как ошибка уйдёт в требования.

### Рабочая папка карты

```
<workspace>/<map>/
├── <map>.md                 (/sim-requirements) → ФИНАЛЬНАЯ карта + требования
└── artefacts/
    ├── frame.md             (/sim-frame)
    ├── stories.md           (/sim-stories)
    ├── matrix.md            (/sim-matrix)
    └── validation.md        (/sim-validate)
```

## Hard-гейты и типовые ошибки

Бинарные 🔴-проверки. Нарушение любого — возврат на предыдущую стадию.

1. **Глагол/форма-решения в элементе.** Элемент истории описывает поведение, а не техническое решение. «Нажимает кнопку "Экспорт"» в способе действия (Д) — ошибка: кнопка это форма реализации, ей место под чертой, а не в истории. Д отвечает «каким образом по сути», не «каким интерфейсом».
2. **Объекты оперирования без входа или без выхода.** О → О′ — это преобразование. Пропущен вход или неясно, что получилось на выходе (в полной форме) → 🔴. Краткая форма (только О) допустима осознанно, а не по забывчивости.
3. **История не подчинена цели деятельности.** Ц описывает локальное «нажал-получил», а не зачем это на уровне деятельности как целого → история висит в воздухе. Цель должна отвечать «зачем вообще», не «что произошло».
4. **Больше одного шага в истории.** Одна рабочая история = один шаг поведения. Слиплись два действия → расщепить на два столбца.
5. **Носитель размыт.** Н не отвечает «кто именно действует / какой части машины вменена операция». «Система» вместо конкретного носителя → уточнить.
6. **Ситуация отсутствует или это не обстоятельство.** С должна задавать «когда/при каких обстоятельствах», а не повторять способ действия.
7. **Форма реализации затесалась в историю (или наоборот).** Формы реализации живут только под чертой; сама история — модель поведения, независимая от инструмента.
8. **Требование без формы реализации-источника.** Каждое требование в `/sim-requirements` ← конкретная карточка формы реализации ← история. Разрыв цепочки → `[УТОЧНИТЬ]`.
9. **Нарушен порядок осей.** Слева-направо не соответствует следованию процесса, либо карточки в столбце не в порядке Ц·Н·С·Д·О/О′.

Светофор: 🟢 все гейты пройдены · 🟡 пройдены, но есть `[УТОЧНИТЬ]` · 🔴 нарушен хотя бы один — карта не финализируется.

## Рендер-движок

- **Основной:** **Excalidraw MCP** — карта рендерится как холст: столбцы историй (5 карточек) слева направо по процессу, формы реализации карточками под чертой. Матричная сетка из структуры `/sim-matrix`.
- **Официальный инструмент автора:** среда **Социотех** (`sociotech.center`) — единственный инструмент от автора метода, полностью поддерживает КРИ на уровне сценариев и структуры данных.
- **Fallback-шаблоны автора для электронных досок:** Holst.so, Figma (community), Miro (miroverse), Numbers, Excel — если Excalidraw MCP недоступен, отдаём в один из этих шаблонов.

## Стандарты и ресурсы (индекс — что легло бы в resources/ и examples/)

- `resources/story_elements.md` — детальная спецификация 5 элементов (Ц·Н·С·Д·О/О′), вопросы-подсказки, полная и краткая форма записи, разбор шаблона «Н в С выполняет Д, меняя О на О′ для Ц».
- `resources/matrix_structure.md` — правила матрицы: ось процесса (→), ось подчинения функции (↓), столбец истории, черта, карточки форм реализации.
- `resources/hard_gates.md` — бинарные 🔴-гейты и типовые ошибки + чек-лист + Светофор.
- `resources/forms_to_requirements.md` — как форма реализации разворачивается в проверяемое требование (трассировка требование ← форма ← история).
- `resources/render_excalidraw.md` — раскладка карты в холст через Excalidraw MCP; маппинг на fallback-шаблоны (Holst/Figma/Miro/Numbers/Excel) и Социотех.
- `resources/writing_style.md` — регистр записи историй (плотный, поведенческий, без AI-примет); переиспользует `skills/bft-writer/resources/writing_style.md`.
- `examples/ideal_sim.md` — эталонная карта (иллюстративный домен: конвейер сортировки овощей / браслет сна / контакты спикера — примеры из метода автора).
- `examples/story_template.md` — пустой шаблон рабочей истории по 5 элементам.

## Команды

| Команда | Роль | Что делает |
|---|---|---|
| `/sim-frame` | Frame Setter | Зафиксировать деятельность: процесс, цель, модальность |
| `/sim-stories` | Story Collector | Собрать рабочие истории по 5 элементам |
| `/sim-matrix` | Matrix Cartographer | Разложить истории × формы реализации в матрицу |
| `/sim-requirements` | Requirements Writer | Вывести требования из форм реализации |
| `/sim-validate` | Validator | Hard-гейты + Светофор |
| `/sim-deliver` | Deliverer | Рендер холста (Excalidraw MCP) + публикация |

Команды на старте читают `.claude/domain-profile.md` (пути, текущий квартал, roster), этот SKILL и релевантные `resources/`.

## Главное правило процесса

**STOP после `/sim-stories`, `/sim-requirements` и `/sim-deliver`.** Не лети до финала. Сначала истории (поведение), потом формы реализации, только потом требования — не наоборот. Каждый элемент истории заякорен источником; форма-решение не протекает в модель поведения; чего в источнике нет — то `[УТОЧНИТЬ]`, а не выдумка. Автор метода — Андрей Шапиро; изложение своими словами.
