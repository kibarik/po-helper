# people-map · рецепты навигации

Точные рецепты для режимов `/people-map`. Цель — детерминированная навигация по People Graph без догадок. Все рецепты read-only.

## Ось запроса → поле (таблица решений)

PO-вопрос почти всегда сводится к одной из четырёх осей. Сначала определи ось, потом матч по её полю — не смешивай.

| Намерение PO (примеры формулировок) | Ось | Первичное поле | Fallback | Что НЕ ответ |
|---|---|---|---|---|
| «с кем согласовать», «кто даёт добро», «нужен sign-off», «кто подпишет», «кто решает» | **approve** | `approves` | `influence_zones` (зона ответственности) | эксперт без `approves` — у него нет права решения |
| «у кого уточнить», «кто знает детали», «к кому за контекстом», «кто в курсе истории» | **clarify** | `clarify_with` | `context_holds`, `expertise_topics` | тот, кто только согласует, но контекст не держит |
| «к кому по теме», «кто отвечает за X», «кто поможет с X» | **contact** | `contact_for` | `expertise_topics`, `influence_zones` | — |
| «от кого ждать запрос», «кто сам придёт», «что прилетит» | **inbound** | `inbound_topics` | — | — |

> **Ключевое различие** (повторяется в SKILL.md): `contact_for`/`expertise_topics` = «человек полезен по теме вообще». `clarify_with` = «я-как-PO иду к нему за деталями». `approves` = «без него решение не принять». Право решения и наличие знаний — разные вещи.

## Скоринг кандидата (режим route)

Для каждого узла, попавшего в матч по выбранной оси, посчитай ранг (сортировка по убыванию):

```
score = match_quality * 100      # 3 = тема дословно в поле; 2 = явная синонимия; 1 = смежно
      + proximity_weight         # core=8, close=6, extended=4, peripheral=2, нет=0
      + freshness_bonus          # fresh=+3, ripening=+1, wilting=0 (и пометить ⚠️)
      + confidence * 2           # CP узла 0..1 → 0..2
```

- `match_quality` оценивай семантически: «дата релиза» ⊇ запрос «когда релиз» → 3; «релизы» (influence_zone) под запрос «согласовать дату релиза» → 2.
- При равенстве score выше тот, у кого больше `proximity_weight` (ближе к PO — быстрее достучаться).
- В ответе показывай топ-3..5; остальных сворачивай в «ещё N кандидатов».

## Разрешение «не он → к кому дальше»

Если выбранный человек может не решить вопрос:
1. `escalate_via` узла задан → это следующий адресат (покажи его имя/роль).
2. Не задан, но есть `reports_to` → предложи руководителя как вероятную эскалацию с пометкой `[УТОЧНИТЬ: escalate_via не задан, беру reports_to]`.
3. Ни того, ни другого → `[УТОЧНИТЬ путь эскалации у PO]`.

## Покрытие графа (показывать в каждом ответе)

- `proximity_coverage = узлов с proximity / всего узлов`.
- Перечисли узлы с пустыми ключевыми полями оси, по которой шёл запрос → блок `[УТОЧНИТЬ]`. Это превращает навигацию в чек-лист до-наполнения онбординга.

## Анти-паттерны

- ❌ Ответить «согласует Иванов», если у Иванова `approves: []`, но он эксперт. → Это `[УТОЧНИТЬ]` + кандидат, не ответ.
- ❌ Слить inbound и outbound: «приходит ко мне» (`inbound_topics`) ≠ «я иду уточнять» (`clarify_with`).
- ❌ Назвать человека без `node_id`-якоря.
- ❌ Дорисовать в карту людей/рёбра, которых нет в узлах.
