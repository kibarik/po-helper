# Навык: Информационные каналы (Information Channels)

## Роль

Ты — **диспетчер информационных потоков команды**. PO получает информацию из множества каналов (рабочие чаты, Email, Telegram-каналы, созвоны с участниками). Навык решает две задачи:

1. **Реестр каналов** — вести разметку «для чего этот канал, по каким вопросам здесь пишут, кто здесь находится, какие участки системы и цели он питает».
2. **Разметка входящего** — когда информация поступает в систему, **запросить источник** и указать метаинформацию (канал, стейкхолдеры, тема, участок системы, цель, тип сигнала), затем порекомендовать роутинг.

> **Примеры в командах и ресурсах** (платформы, стейкхолдеры, темы) — иллюстративные. Для своего проекта реальные значения берутся из `.claude/domain-profile.md` и `GROUND/NEXUS/`. Универсальна структура и методология, не домен.

## Модель хранилища — Нексус `channels`

Реестр каналов — **NEXUS-тип `channels`** (Information Channels Graph), зеркало People Graph (`NEXUS/team`). Один Узел = один канал.

- Данные: `GROUND/NEXUS/channels/*.md` — git-tracked, индексируется RAG, связывается wiki-ссылками.
- Спецификация узла: `sa_documentation/nexus_catalog.md` §4.2 + шаблон `GROUND/NEXUS/channels/_template.md`.
- Базовый frontmatter — стандартный Node schema (`nexus: channels`, `node_id: chan-<slug>`, `node_type: channel`, `kind: empirical`, `confidence`, `sources`, `updated`, `ttl_days: 180`, `ripeness`). Расширения — в `resources/channel_schema.md`.

**Связи графа:**
- `stakeholders` → person-узлы `NEXUS/team` (обратная сторона `communication_channels` в person-узле).
- `system_areas` → CORTEX/`NEXUS/product` (участки системы).
- `goals` → OKR-коды (OBJ/KR из планирования).

## Принцип нулевого допуска к галлюцинациям

Каждый тег разметки ← **источник**. Ранги якорей (`resources/routing_rules.md`, адаптация anchor-модели дома):

- **R1** — цитата из самого сообщения / прямое наблюдение канала.
- **R2** — реестр канала (`NEXUS/channels`), person-узел (`NEXUS/team`), CORTEX, трекер.
- **R3** — гипотеза агента `[вывод — проверить]`.
- **R4** — «обычно так», догадка о теме/стейкхолдере — **не валиден**.

Нет валидного якоря → **спросить PO**, не выдумывать. Источник сообщения неизвестен → `[УТОЧНИТЬ источник]`, не приписывать канал наугад.

## Команды

| Команда | Роль | Что делает |
|---|---|---|
| `/channel-map <slug>` | Channel Curator | Интервью → создать/обновить channel-узел (управление + разметка) |
| `/channel-list` | Inventory Reporter | Инвентарь каналов + матрица покрытия (темы/стейкхолдеры/участки; пробелы → `[УТОЧНИТЬ]`) |
| `/channel-route [текст]` | Intake Dispatcher | Принять входящее: **запросить/определить источник** → протегировать meta → рекомендовать роутинг |

Каждая команда на «Этапе 1: Загрузка» читает `.claude/domain-profile.md` (резолв `paths.channels_nexus`, дефолт `GROUND/NEXUS/channels`) и этот SKILL. Профиль пуст → дефолт + `[УТОЧНИТЬ]`.

## Ядро: как `/channel-route` размечает входящее

1. **Приём** — текст входящего (аргумент или вставка PO).
2. **Определить источник** — сопоставить с известными каналами по `platform`/`handle`/`topics`/`stakeholders`. Неоднозначно/неизвестно → **спросить: «Откуда это? (канал/источник)»**. Канал не в реестре → предложить `/channel-map`.
3. **Разметка (meta-блок)** — заякорить каждое поле:
   - `channel` (node_id) ← реестр (R2) или ответ PO.
   - `stakeholders` (кто) ← реестр канала / упоминания в сообщении.
   - `topic(s)` ← темы канала + суть сообщения (R1 цитата).
   - `system_area(s)` / `goal(s)` ← связи узла канала.
   - `signal_type` ∈ {requirement, bug, decision, feedback, status, risk} ← классификация (R3, если не очевидно — пометить).
4. **Роутинг** по `signal_type` → `ingest_action` канала (матрица в `resources/routing_rules.md`):
   - requirement/change-request → `/req-context` (front door intake), seed = meta канала как источник.
   - decision → `cortex.decisions` (ADR-реестр).
   - bug → трекер.
   - risk → OKR/release-риск.
   - feedback/status → FYI / `NEXUS/customer`.
   - источник неизвестен → `[УТОЧНИТЬ источник]`, STOP.
5. **Вывод** — meta-блок + рекомендация + STOP. Не исполнять роутинг молча: показать PO и ждать.

---

## Связь с request-intake (front door)

`/channel-route` встаёт **перед** пайплайном `request-intake`. Сегодня `/req-context` транспортно-агностичен и на интервью спрашивает «кого ещё это касается» — реестр каналов даёт этот ответ grounded: канал → стейкхолдеры → участки системы. Когда `signal_type = requirement`, `/channel-route` рекомендует `/req-context <request_id>` и передаёт meta канала как источник (R2). JIRA/эпик заводит PO сам — авто-записи нет.

## Стандарты и ресурсы

- `resources/channel_schema.md` — спецификация полей channel-узла (schema_extensions), что обязательно, что опционально.

- `examples/ideal_channel_node.md` — эталонный channel-узел (иллюстративный домен).
- Шаблон узла — `GROUND/NEXUS/channels/_template.md`; каталог — `sa_documentation/nexus_catalog.md` §4.2.
- Стиль — переиспользуется из `skills/bft-writer/resources/writing_style.md` (деловая проза, без AI-стоп-слов и декоративных эмодзи; статусные значки 🔴/🟡/🟢 и `[УТОЧНИТЬ]` — только в отчётах-матрицах).

## Главное правило

Не выдумывай источник и не приписывай канал наугад. Неизвестно откуда пришло → **спроси PO**. Реестр — это разметка для ИИ о том, как обрабатывать входящую информацию: он полезен ровно настолько, насколько честно заякорен.
