# Правила разметки и роутинга — /channel-route

Стадия `/channel-route` совмещает две роли: **Source Identifier** (откуда пришло) и **Dispatcher** (куда роутить). Сначала якорим источник, затем тегируем meta, затем роутинг-решение, затем Светофор.

## Часть A — Ранги якоря (против галлюцинаций)

| Ранг | Тип якоря | Когда валиден | Пример |
|---|---|---|---|
| **R1** 🥇 | Цитата из самого сообщения / прямое наблюдение канала | суть сообщения, тема, упомянутые люди | `«списание прошло дважды» (сообщение)` |
| **R2** 🥈 | Реестр канала (`NEXUS/channels`), person-узел (`NEXUS/team`), CORTEX, трекер | канал, стейкхолдеры, участок системы, цель | `NEXUS/channels/chan-billing-tg.md`, `team-ivanov-ivan` |
| **R3** 🥉 | Вывод агента — гипотеза | классификация `signal_type`, реконструкция участка/цели | `[вывод — проверить]` |
| **R4** ❌ | «Обычно так», догадка о теме/канале/человеке | **не валиден** | — |

**Правило источника:** если канал не определяется по `platform`/`handle`/`topics`/`stakeholders` — **спросить PO: «Откуда это?»**. Не приписывать канал наугад. Неизвестно → `[УТОЧНИТЬ источник]`, STOP.

## Часть B — Meta-блок разметки

Каждое поле — со своим якорем:

```
Источник:      chan-billing-tg (Telegram @billing_team)   [R2 реестр | ответ PO]
Стейкхолдеры:  team-ivanov-ivan, team-petrov-dmitry        [R2]
Тема:          двойное списание                            [R1 «прошло дважды»]
Участок:       [[cortex-billing]]                          [R2]
Цель:          KR-3.2                                       [R2] / [УТОЧНИТЬ]
Тип сигнала:   bug                                          [R3 — проверить]
```

Поле не выводится из якоря → `[УТОЧНИТЬ]`, не заполнять догадкой.

## Часть C — Роутинг-матрица (решение по signal_type)

Решение — по `signal_type` и полю `ingest_action` узла канала. Только «→ intake» ведёт в `request-intake`.

| signal_type | Куда | Действие агента |
|---|---|---|
| **requirement** / change-request | `/req-context <request_id>` | Front door intake. Передать meta канала как источник (R2). JIRA заводит PO сам. |
| **decision** | `cortex.decisions` (ADR-реестр) | Зафиксировать как решение с якорем на сообщение. |
| **bug** | трекер (`tracker` из профиля) | Оформить дефект; участок системы — из `system_areas`. |
| **risk** | OKR / release-риск | Привязать к `goals`; при релизе — сигнал в release-guard. |
| **feedback** / status | FYI / `NEXUS/customer` | Не создаёт задачу; фиксируется как сигнал/фидбек. |
| источник неизвестен | — | `[УТОЧНИТЬ источник]`, STOP. Не роутить вслепую. |

Один вход может нести несколько сигналов (напр. баг + запрос) — разметить каждый отдельной строкой meta и роутить раздельно.

## Часть D — Светофор покрытия (для /channel-list)

| Слой | 🔴/🟡/🟢 | Что проверяем |
|---|---|---|
| Назначение (`purpose`) | | у канала явно описано, для чего он |
| Темы (`topics`) | | перечислены вопросы, по которым пишут |
| Стейкхолдеры (`stakeholders`) | | привязаны к реальным `NEXUS/team` |
| Участки системы (`system_areas`) | | привязаны к CORTEX/product |
| Правило обработки (`ingest_action`) | | канал знает, куда роутить входящее |

- 🟢 — обязательные поля заполнены и заякорены.
- 🟡 — заполнены `purpose`/`topics`/`stakeholders`, но пусты `system_areas`/`goals` (`[УТОЧНИТЬ]`).
- 🔴 — нет одного из обязательных (`purpose`/`topics`/`stakeholders`) → канал не размечен, `/channel-map` доработать.

## Правила

1. Источник ← R1/R2 или вопрос PO. Никогда не выдумывать канал.
2. Классификация `signal_type` — R3, помечается `[проверить]`, если не очевидна из R1.
3. Стейкхолдеры/участки/цели ← только существующие узлы (R2). Нет — `[УТОЧНИТЬ]`.
4. Роутинг показывается PO, не исполняется молча. STOP после meta-блока.
5. Несколько сигналов в одном входе — раздельная разметка и раздельный роутинг.
