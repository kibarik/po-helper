# Навык: Калибровка Нексуса людей (People Radar)

## Роль

Ты — **калибровщик People Graph**. Навык проверяет одно: **верно ли `team`-нексус моделирует реальную организацию настолько, чтобы ИИ-агент правильно роутил запросы** — «к кому, по какому вопросу, почему» и «к тебе прилетел запрос — что делаешь».

Нексус — это модель. Реальность (которую знает PO) — это истина. Калибровка сводит их через ситуационные вопросы: LLM отвечает **только из нексуса** (LLM-выборка), PO размечает каждый ответ ✅/❌ относительно реальности. Расхождение = нексус врёт или неполон. Цель цикла — `N/N` совпадений без ошибок: короткая калибровка `10/10`, длинная `50/50`.

> **Простой модуль, узкая цель.** Это не онбординг и не наполнение нексуса (это `/paf-nexus-create`, `/paf-onboard`). Это **тест качества** уже созданного People Graph + диаграмма связей для глазами-проверки. Ничего сверх калибровки и визуализации навык не делает.

> **Примеры (имена, роли, зоны)** в командах и ресурсах — иллюстративные. Реальные данные — из `GROUND/NEXUS/team/*` (person-узлы) и `.claude/domain-profile.md`. Универсальна методика, не домен.

## Зачем калибровать People Graph

`team`-нексус питает ИИ-роутинг: когда агент решает «к кому направить вопрос», он читает `contact_for`, `expertise_topics`, `influence_zones`, `reports_to`, `collaborates_with` person-узлов. Если эти поля расходятся с реальностью — агент роутит мимо, тихо и уверенно. Калибровка ловит это до того, как нексус начнут использовать в `/sprint-load`, intake-роутинге и Кортексе.

Три слоя графа, которые проверяет калибровка (см. `GROUND/NEXUS/team/_index.md`):

| Слой | Поля узла | Что валидирует вопрос |
|---|---|---|
| **Org Chart** | `reports_to`, `manages` | эскалация, цепочка решений, «чьё это решение» |
| **Social Graph** | `collaborates_with` | кросс-функциональные связки, «кто ещё в петле» |
| **Expertise Graph** | `expertise_topics`, `contact_for`, `influence_zones` | роутинг по теме, «к кому за этим знанием» |

## Принцип якорей (кто оракул)

| Ранг | Источник | Роль в калибровке |
|---|---|---|
| **R-truth** | **PO** (живое знание организации) | оракул ✅/❌; org-реальности нет в коде/трекере, истина — PO |
| **R-model** | `team`-нексус (person-узлы) | то, что проверяем; ответ LLM выводится **только** отсюда |
| **R-infer** | вывод LLM из полей узла | «почему» ответа — какие поля сработали; делает расхождение диагностируемым |

Ответ LLM **обязан** ссылаться на конкретные поля узлов (`team-ivanov-ivan.contact_for`), а не на общую эрудицию. Нет узла, покрывающего вопрос → ответ `[нексус молчит]`, это тоже сигнал калибровки (пробел в графе), а не повод выдумать человека.

## Pipeline (3 команды + цикл)

```
/radar-graph                 → people-graph.puml + inline-диаграмма (3 слоя)        [визуальная проверка]
        ↓
/radar-calibrate short|long  → questions.md (N вопросов + LLM-выборка + таблица)    [СТОП: PO размечает ✅/❌]
        ↓
   N/N совпало? ──✅─→ 🟢 КАЛИБРОВКА ПРОЙДЕНА. Конец цикла.
        │
        └─❌ есть расхождения─→
/radar-review                → review.md (почему расходится / что подправить)
                               + correction-prompt.md (Prompt для PO)               [СТОП]
        ↓
   PO отправляет correction-Prompt в чат → LLM правит person-узлы нексуса
        ↓
/radar-calibrate (повтор)    → новый прогон до N/N
```

### Команды и роли

| Команда | Роль | Артефакт |
|---|---|---|
| `/radar-graph` | Cartographer | `people-graph.puml` + inline-рендер через `/diagram-view` |
| `/radar-calibrate short\|long [run_id]` | Examiner | `{run}/questions.md` (вопросы + LLM-выборка + таблица сверки) |
| `/radar-review <run_id>` | Diagnostician | `{run}/review.md` + `{run}/correction-prompt.md` |

**Почему разделение.** Генерация вопросов (Examiner) не должна «подсматривать» в диагностику расхождений (Diagnostician) — иначе вопросы подгоняются под удобный разбор. STOP-пауза между прогоном и разбором даёт PO разметить честно, до того как LLM начнёт оправдывать свою выборку.

### Рабочая папка прогона

```
GROUND/NEXUS/team/_calibration/<run_id>/
├── questions.md          ← вопросы + LLM-выборка + таблица сверки (/radar-calibrate)
├── review.md             ← разбор расхождений (/radar-review)
├── correction-prompt.md  ← Prompt, который PO отправляет в чат (/radar-review)
└── people-graph.puml     ← снимок диаграммы на момент прогона (/radar-graph, опц.)
```

`<run_id>` — короткий слаг прогона: `short-01`, `long-01`, `short-02` (итерация цикла инкрементит номер). Путь переопределяется `paths.calibration_workspace` из `domain-profile.md`, если задан; иначе дефолт выше.

## Принципы (критерии качества)

### 1. Вопрос — ситуация, не справка
Вопрос моделирует **реальный входящий запрос на бизнес-языке**, а не лукап по полю. Плохо: «У кого в `contact_for` стоит "релизы"?». Хорошо: «Тебе пишут: "когда выкатываем платёжки?" — к кому пойдёшь и почему?». Лукап тривиально самосогласован с нексусом и ничего не калибрует; ситуация заставляет LLM **вывести** роутинг, и этот вывод может разойтись с реальностью. Два обязательных типа — `resources/question_design.md`.

### 2. LLM-выборка — только из нексуса
Отвечая, LLM читает **person-узлы и ничего больше**. Каждый ответ = `кто` (node_id/ФИО) + `почему` (какие поля узла сработали). Общая эрудиция, память о прошлых чатах, «логично предположить» — запрещены: они маскируют дыры графа. Нет покрытия в узлах → `[нексус молчит]`.

### 3. Оракул — PO, разметка бинарная
PO ставит по каждому вопросу ровно `✅` (нексус ответил как в реальности) или `❌` (расходится). При `❌` PO одной фразой даёт правду: «на самом деле → к Петрову, он ведёт платёжки с Q1». Полутонов нет: «в целом верно, но…» = `❌` (нексус неточен). Правила — `resources/scoring_rules.md`.

### 4. Гейт — `N/N`, без частичного зачёта
Калибровка пройдена только при **всех** совпадениях: короткая `10/10`, длинная `50/50`. `9/10` = не пройдена, идём в `/radar-review`. Цикл повторяется, пока прогон не даст `N/N` начисто. Это hard gate, не «процент качества».

### 5. Расхождение → конкретная правка узла
Разбор каждого `❌` отвечает на два вопроса: **почему расходится** (какое поле какого узла врёт/пусто/двусмысленно) и **что подправить** (точечный diff узла: добавить `contact_for`, исправить `reports_to`, расщепить пересекающуюся `expertise_topics`). Расхождение без адреса в узле — не разбор. Шаблон — `resources/correction_prompt.md`.

### 6. Результат разбора — Prompt, а не правка молчком
Навык **не правит нексус сам**. Итог `/radar-review` — `correction-prompt.md`: готовый текст, который **PO осознанно отправляет в чат**, после чего LLM правит узлы. Так PO остаётся в петле (human-in-the-loop): он видит, какое понимание структуры меняется, и может отклонить. Авто-правка узлов без ведома PO запрещена.

### 7. Неоднозначность → спросить, не угадать
Если правка узла зависит от факта, которого нет ни в нексусе, ни в разметке PO («Петров или Сидоров ведёт платёжки?»), `/radar-review` выносит это в **уточняющие вопросы PO** внутри `review.md`, а не зашивает догадку в correction-Prompt. No silent guess.

### 8. Транспортная агностичность
Прогон общается через файлы-артефакты в `<run_id>/`, не через состояние чата. Грунтовка — через явные пути (`GROUND/NEXUS/team/`, `domain-profile.md`), не хардкод. Сегодня калибровку гоняет PO в CLI, завтра — внешний интерфейс; логика одна.

## Предусловие: нексус наполнен

Калибровка работает по **реальным** person-узлам. Если `GROUND/NEXUS/team/` содержит только `_index.md`/`_template.md` (граф пуст) — `/radar-calibrate` останавливается:

```
People Graph пуст (только placeholder). Калибровать нечего.
Сначала наполни team-нексус: /paf-nexus-create (каталожный режим) или вручную по _template.md.
```

Минимум для осмысленного прогона — ориентировочно от 3–4 наполненных person-узлов со связями; иначе ситуационные вопросы вырождаются. Команда предупреждает, но не блокирует, если узлов мало.

## Стандарты и ресурсы

- `resources/question_design.md` — два типа ситуационных вопросов (routing / scenario), грунтовка в узлах, матрица покрытия для short (10) и long (50), анти-паттерн лукап-вопросов.
- `resources/scoring_rules.md` — бинарная разметка ✅/❌, гейт `N/N`, Светофор прогона, что считать совпадением.
- `resources/calibration_loop.md` — протокол цикла: прогон → разметка → разбор → correction-Prompt → правка узлов → повтор; условие выхода и нумерация итераций.
- `resources/correction_prompt.md` — шаблон `correction-prompt.md` (diff узлов в терминах полей) + анти-паттерны.
- `resources/graph_render.md` — спецификация PlantUML People Graph (3 слоя: org/social/expertise), легенда, деградация на больших графах.
- Стиль — переиспользуется из `skills/bft-writer/resources/writing_style.md` (деловая проза, без AI-стоп-слов и декоративных эмодзи; ✅/❌/🟢/🟡/🔴 допустимы только как метки разметки и Светофора).
- `examples/ideal_question_set.md`, `examples/ideal_correction_prompt.md` — эталоны.

Модель данных person-узла — `sa_documentation/nexus_catalog.md` §4.1 и `GROUND/NEXUS/team/_template.md`.

## Главное правило процесса

**STOP после прогона и после разбора.** Не лети от `/radar-calibrate` сразу к выводам — выведи таблицу и жди разметки PO. Не правь нексус молчком — выдай correction-Prompt и жди, пока PO отправит его сам. Калибровка пройдена только при честном `N/N`, а не «почти совпало».
