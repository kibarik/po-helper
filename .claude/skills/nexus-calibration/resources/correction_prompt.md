# Шаблон correction-Prompt

Итог `/radar-review` — два артефакта: `review.md` (диагноз для PO) и `correction-prompt.md` (готовый текст, который **PO осознанно отправляет в чат**, чтобы LLM правил person-узлы). Навык узлы сам не трогает (принцип 6 SKILL).

## review.md — диагноз

Для каждого `❌` из прогона:

```markdown
## Расхождение #7 — платёжки

- **Вопрос:** «Заказчик спрашивает дату выката платёжек — к кому?»
- **Ответ LLM (выборка):** team-petrov-ivan · почему: petrov.contact_for "статус релиза" + influence_zones "релизы"
- **Правда PO:** платёжки с Q1 ведёт Сидорова; у Петрова только инфра-релизы
- **Почему расходится:** влияние/контакт по платёжкам не выражены в графе. У `team-petrov-ivan` `influence_zones` слишком широкая ("релизы" покрывает и платёжки), а у `team-sidorova-anna` нет `contact_for`/`expertise_topics` про платёжки → LLM выбрал ближайшего по тексту.
- **Что подправить (адрес в узлах):**
  - `team-petrov-ivan`: сузить `influence_zones` "релизы" → "инфра-релизы, деплой"
  - `team-sidorova-anna`: добавить `contact_for: "дата выката платёжек"`, `expertise_topics: "платёжный модуль"`, `influence_zones: "платёжки"`
- **Слой:** Expertise (+ дизамбигуация)
```

Если правка зависит от факта, которого нет ни в нексусе, ни в разметке PO → **не угадывай**, вынеси в раздел:

```markdown
## Уточнить у PO (блокирует correction-Prompt)
- Платёжки ведёт только Сидорова, или Петров тоже частично (тогда оба в contact_for с разными зонами)?
```

correction-Prompt не выпускается, пока такие вопросы открыты (принцип 7 SKILL).

## correction-prompt.md — Prompt для PO

Готовый к отправке текст. PO копирует его в чат как есть; адресован LLM, который держит нексус. Структура:

```markdown
# Correction-Prompt (калибровка <run_id>, итерация N)

> Скопируй блок ниже и отправь в чат. После применения — запусти /radar-calibrate для повторного прогона.

---

Откалибруй team-нексус (People Graph) по итогам калибровки. Внеси точечные правки в person-узлы `GROUND/NEXUS/team/`. Меняй только перечисленные поля, остальное не трогай. После каждой правки подними `confidence` затронутого узла по факту источника и обнови `updated`/`sources` (`sources: [..., "calibration:<run_id>"]`).

## Узел team-petrov-ivan
- `influence_zones`: было ["релизы", ...] → стало ["инфра-релизы, деплой", ...]
- причина: платёжки ведёт не он; широкая зона уводила роутинг

## Узел team-sidorova-anna
- `contact_for`: + "дата выката платёжек"
- `expertise_topics`: + "платёжный модуль"
- `influence_zones`: + "платёжки"
- причина: реальный владелец платёжек, не был выражен в графе

## После правок
Покажи diff изменённых узлов и подтверди, что поля `contact_for`/`expertise_topics`/`influence_zones`/`reports_to` теперь отражают: «платёжки → Сидорова, инфра-релизы → Петров».
```

## Принципы correction-Prompt

1. **Адресность.** Каждая правка указывает узел + поле + было/стало. «Уточни роли» без адреса — не correction-Prompt.
2. **Минимальность.** Меняем только то, что вызвало `❌`. Не переписываем узел целиком, не «улучшаем» поля без расхождения.
3. **Прослеживаемость.** Prompt велит проставить `sources: ["calibration:<run_id>"]` и поднять `confidence` — правка должна оставлять след в нексусе (узел без источника = workslop, `nexus_schema`).
4. **Проверяемость.** Финальный блок Prompt'а просит LLM показать diff и сформулировать новое понимание словами — чтобы PO видел, какое представление структуры поменялось (это и есть «LLM доработала своё понимание структуры»).
5. **Человек в петле.** Prompt отправляет PO, не навык. Текст самодостаточен: PO не должен ничего дописывать, кроме случаев, вынесенных в «Уточнить у PO».

## Анти-паттерны

- ❌ Навык сам редактирует `GROUND/NEXUS/team/*` в обход PO.
- ❌ Prompt вида «исправь нексус, чтобы калибровка прошла» — без конкретных полей (подгонка под тест, а не под реальность).
- ❌ Правка узла без `sources`/`confidence`/`updated` (теряется прослеживаемость).
- ❌ Догадка вместо «Уточнить у PO», когда факта нет.
