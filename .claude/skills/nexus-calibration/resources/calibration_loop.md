# Протокол цикла калибровки

Калибровка — замкнутый цикл, который крутится, пока прогон не даст `N/N` начисто. Ни одна итерация не «почти проходит» — гейт бинарный.

```
        ┌──────────────────────────────────────────────┐
        ▼                                                │
/radar-calibrate short|long  →  questions.md            │
        │   (N вопросов + LLM-выборка + таблица)         │
        ▼                                                │
   ── СТОП ── PO размечает ✅/❌ + правду к каждому ❌     │
        │                                                │
        ▼                                                │
   N/N совпало? ──✅──→ 🟢 ПРОЙДЕНА. Выход из цикла.      │
        │                                                │
        └──❌──→ /radar-review <run_id>                  │
                  →  review.md (почему / что подправить) │
                  →  correction-prompt.md                │
                     │                                   │
                ── СТОП ── PO отправляет Prompt в чат     │
                     │                                   │
                     ▼                                   │
              LLM правит person-узлы нексуса             │
                     │                                   │
                     └──→ /radar-calibrate (повтор) ─────┘
```

## Нумерация итераций

`<run_id>` инкрементит номер на каждом новом прогоне той же длины:

- `short-01` → разбор → правка → `short-02` → … → `short-NN` (🟢).
- Длинная — отдельная серия: `long-01`, `long-02`, …

Каждый прогон — своя папка `GROUND/NEXUS/team/_calibration/<run_id>/`. Старые прогоны не перезаписываются: видна история сходимости (сколько итераций до 🟢, какие узлы чинились).

## Условие выхода

Цикл закрывается **только** при `N/N` (🟢). Это единственный success-state. После 🟢:

```
🟢 КАЛИБРОВКА ПРОЙДЕНА (<short 10/10 | long 50/50>) за <K> итераций.
People Graph откалиброван: ИИ-роутинг по нему совпадает с реальностью на всех вопросах.
Снимок: GROUND/NEXUS/team/_calibration/<run_id>/questions.md
```

## Связь между итерациями (регрессия)

Новый прогон **не обязан** повторять те же вопросы. Но при `short-02+` хотя бы половина вопросов должна щупать узлы, которые чинились в прошлой итерации, — иначе правка не проверена. Остальные — свежие, чтобы не переобучить нексус под фиксированный тест. Команда `/radar-calibrate` читает `review.md` прошлой итерации (если есть) и нацеливает часть вопросов на исправленные узлы.

## Когда короткая, когда длинная

| | Короткая (10) | Длинная (50) |
|---|---|---|
| Когда | быстрая регрессия после правок; «не сломали ли роутинг» | финальная приёмка нексуса перед боевым использованием |
| Стоимость | минуты PO | заметное время PO на разметку |
| Гейт | 10/10 | 50/50 |

Типичный путь: гонять короткую до 🟢 итерациями, затем один прогон длинной для приёмки. Длинная упала → точечные правки → снова короткая на затронутых узлах → длинная.

## Что цикл НЕ делает

- Не правит нексус сам — только выдаёт correction-Prompt (принцип 6 SKILL).
- Не наполняет пустой граф — это `/paf-nexus-create` / `/paf-onboard`. Калибровка предполагает наполненный нексус.
- Не меняет схему узла — работает в рамках полей `team`-нексуса (`sa_documentation/nexus_catalog.md` §4.1).
