# Пример: email → JIRA Issue через jira-task

> **Примечание:** пример иллюстративный. Коды вида `PROJ-*`, ключи и URL — вымышленные; реальные значения берутся из `.claude/domain-profile.md`.

---

## Входные данные

**Источник:** сообщение PO в Slack

```
Привет, нужно что-то делать с API /users/profile. 
На нагрузку свыше 500 запросов в секунду он отвечает медленнее, 
чем хотелось бы. Пользователи жалуются. 
Это критично для наших SLA.
```

---

## Шаг 1: Parse

Разборка входного текста по полям шаблона:

| Поле | Результат | Источник |
|------|-----------|---------|
| **Образ результата** | API /users/profile отвечает быстро даже при пиках нагрузки | Инферировано из контекста |
| **ASIS** | Текущий эндпойнт замедляется при нагрузке > 500 req/s | Явно из сообщения |
| **ПРОБЛЕМА** | Пользователи жалуются на медленные отклики; это влияет на SLA | Явно из сообщения |
| **TOBE** | *(пусто)* | — |
| **Описание** | Критично для SLA | Явно из сообщения |

**Заключение:** критичное поле `TOBE` не заполнено → переход к шагу 2 (Gate-интервью).

---

## Шаг 2: Gate-интервью

**Вопрос к PO:**
> Спасибо за контекст. Вижу проблему с задержками API /users/profile при пиках нагрузки. Чтобы правильно сформулировать целевое состояние: какое время отклика вы считаете приемлемым при нагрузке 500+ req/s? Есть ли конкретный SLA (например, < 100ms, < 200ms)?

**Ответ PO:**
> Да, нужно чтобы отклик был < 100ms даже при 1000 req/s. Это наш стандартный SLA для критичных эндпойнтов.

**Итог интервью:**

Обновленное поле TOBE:

| Поле | Значение |
|------|----------|
| **TOBE** | API /users/profile отвечает за < 100ms при нагрузке до 1000 req/s |

---

## Шаг 3: Classify

**Issue type:** Task (сигналов дефекта нет, это улучшение производительности)

**Project:** PROJ (иллюстративный, из конфига `tracker.projects[0]`)

**Summary** (на основе Образа результата):  
`Оптимизировать производительность API /users/profile до < 100ms при пиках 1000+ req/s`

---

## Шаг 4: Relevance-research

### Извлечение ключевых слов
Из Summary, Образа результата и TOBE:
- `API`, `производительность`, `/users/profile`, `нагрузка`, `SLA`, `оптимизация`

### JIRA-поиск
```jql
project = PROJ AND (text ~ "производительность" OR text ~ "API" OR text ~ "/users/profile" OR text ~ "SLA") ORDER BY updated DESC
```

**Найдено 2 кандидата (релевантные):**
- `PROJ-101` — «Добавить кэширование для /users/profile»  
  Updated: 2026-06-15
- `PROJ-87` — «Расширить connection pool в auth-сервисе»  
  Updated: 2026-05-20

### Confluence-поиск
```
space = ENG (из `wiki.space`) AND (text ~ "производительность" OR text ~ "SLA")
```

**Найдено (релевантный материал):**
- `https://wiki.internal.com/spaces/ENG/pages/98765/Performance-SLA-patterns`  
  Заголовок: «Performance SLA patterns для критичных API»

### Отбор кандидатов (до ~5 наиболее релевантных)

Ранжирование:
1. **PROJ-101** — явное упоминание `/users/profile` и кэширования (прямое решение)
2. **https://wiki.internal.com/spaces/ENG/pages/98765/Performance-SLA-patterns** — документация по SLA-паттернам (контекст)

*Исключена:* PROJ-87 (про connection pool, менее релевантна, не про /users/profile)

### Представление кандидатов PO

**Найдены релевантные материалы:**
- PROJ-101 — обсуждение кэширования для эндпойнта /users/profile
- https://wiki.internal.com/spaces/ENG/pages/98765/Performance-SLA-patterns — документация по паттернам оптимизации SLA

**Вопрос к PO:**  
> Какие из этих материалов вы хотите прикрепить к Issue?

**Ответ PO:**
> Приложи оба. И PROJ-101 (может быть уже решение), и документацию.

## Шаг 5-6: Preview + Create

### Preview (до создания)

**Проект:** PROJ  
**Тип:** Task  
**Summary:** `Оптимизировать производительность API /users/profile до < 100ms при пиках 1000+ req/s`

**Описание:**
```
Образ результата:
API /users/profile отвечает быстро даже при пиках нагрузки

ASIS: Текущий эндпойнт замедляется при нагрузке > 500 req/s
ПРОБЛЕМА: Пользователи жалуются на медленные отклики; это влияет на SLA
TOBE: API /users/profile отвечает за < 100ms при нагрузке до 1000 req/s

Описание:
Критично для SLA. Требуется анализ узких мест и оптимизация (кэширование, пулинг, профилирование).

Связанные материалы:
- PROJ-101 — обсуждение кэширования для эндпойнта /users/profile
- https://wiki.internal.com/spaces/ENG/pages/98765/Performance-SLA-patterns — документация по паттернам оптимизации SLA
```

**Подтверждение от PO:**
> ОК, создавай.

### Create (выполнение)

✅ Вызов `jira_create_issue`:
- `project_key` = `"PROJ"`
- `summary` = `"Оптимизировать производительность API /users/profile до < 100ms при пиках 1000+ req/s"`
- `issue_type` = `"Task"`
- `description` = *(вышеприведённый блок)*
- `labels` = `[]` (не выдумываем)

✅ Вызов `jira_create_issue_link` (связь JIRA ↔ JIRA):
1. Связь PROJ-124 ↔ PROJ-101 (тип: "Relates")

✅ Вызов `jira_create_remote_issue_link` (связь на внешний URL):
2. Связь PROJ-124 → https://wiki.internal.com/spaces/ENG/pages/98765/Performance-SLA-patterns

### Результат

**✅ Issue создана:**

| Параметр | Значение |
|----------|----------|
| Ключ | `PROJ-124` |
| URL | `https://jira.internal.com/browse/PROJ-124` |

**Финальное сообщение:**
> Готово! Issue `PROJ-124` создана: https://jira.internal.com/browse/PROJ-124
