# Шаблон описания и правила классификации для jira-task

## Канонический блок шаблона тела Issue

```
Образ результата:
<краткое описание образа результата>

ASIS: <описание текущего состояния>
ПРОБЛЕМА: <ключевая проблема, почему текущее решение не устраивает>
TOBE: <целевое состояние>

Описание:
<остальной дополнительный контекст для решения>

Связанные материалы:
- <ключ/URL> — <почему релевантно>
```

**Примечание:** Блок «Связанные материалы» опускается, если в relevance-research ничего не выбрано. Остальные разделы обязательны.

---

## Критичные и некритичные поля

### Критичные поля
Обязательны при создании Issue. Отсутствие → интервью с PO (см. SKILL.md):
- **Образ результата** — краткое описание желаемого состояния/результата
- **ПРОБЛЕМА** — ключевая проблема, которую решает Issue
- **TOBE** — целевое состояние после решения

### Некритичные поля
Допускают заполнение паттерном `[УТОЧНИТЬ у {кого}]`, если данные отсутствуют:
- **ASIS** — текущее состояние (если неясно, допускается пропуск с пояснением)
- **Описание** — дополнительный контекст (может остаться пустым, если главные поля достаточны)

## Summary

**Правило:** краткая императивная формулировка на основе «Образа результата» — кратко, императивно, без деталей.

**Алгоритм:**
1. Извлечь ключевую идею из «Образа результата»
2. Переформулировать в повелительное наклонение (например: «Реализовать…», «Добавить…», «Исправить…», «Оптимизировать…»)
3. Указать объект/компонент, если не очевидно
4. Опустить детали; они идут в тело Issue

**Примеры (иллюстративные):**
- Образ результата: «пользователи могут подписаться на уведомления по email»  
  → Summary: `Добавить подписку на email-уведомления`
- Образ результата: «API возвращает 504 при нагрузке >1000 req/s»  
  → Summary: `Оптимизировать производительность API под пики 1000+ req/s`

## Классификация: Task vs Bug

### Task (по умолчанию)
Используется для:
- Нового функционала
- Улучшений существующей функции
- Технических работ (рефакторинг, документация, инфра)
- Неоднозначных случаев → Task с пометкой предположения в тело

### Bug
Используется только при явных сигналах дефекта:
- Ошибка, сбой, не работает корректно
- Регресс (ранее работало, сейчас нет)
- Падение, краш, зависание
- Сломалось в production

**Правило неоднозначности:** если характер Issue двусмыслен (например, «улучшить обработку ошибок» может быть и Bug, и Task), то:
1. Классифицировать как **Task** по умолчанию
2. Добавить пометку в тело Description: `[ПРЕДПОЛОЖЕНИЕ: может быть Bug, если...]`

## Маппинг полей → `jira_create_issue(...)`

Функция подписи:
```
jira_create_issue(project_key, summary, issue_type, description, labels)
```

**Маппинг:**

| Поле шаблона | Аргумент jira_create_issue | Правило |
|---|---|---|
| *Образ результата* + *ASIS* + *ПРОБЛЕМА* + *TOBE* + *Описание* (+ *Связанные материалы*, если непусто) | `description` | Собрать всё содержимое блока в едином аргументе, сохраняя структуру и разделители |
| Summary | `summary` | Краткая императивная формулировка (см. раздел Summary) |
| Классификация (Task/Bug) | `issue_type` | `"Task"` или `"Bug"` в зависимости от сигналов дефекта |
| Профиль пользователя (project) | `project_key` | Из аргумента функции jira-task или профиля domain-profile (иллюстративно: `"PROJ"`, `"INFRA"`, `"QA"`) |
| Теги категоризации | `labels` | labels остаётся пустым по умолчанию — не выдумывать теги |

## Обработка пустых полей

### Критичное поле пусто
→ **Не создавать Issue, запросить интервью** с PO для уточнения (см. SKILL.md):
- Пусто *Образ результата* → фокус Issue непонятен
- Пусто *ПРОБЛЕМА* → мотивация неясна
- Пусто *TOBE* → целевое состояние не определено

### Некритичное поле пусто
→ **Заполнить паттерном** `[УТОЧНИТЬ у {кого}]`, где `{кого}` — владелец информации:

Примеры (иллюстративные):
- ASIS пусто: `ASIS: [УТОЧНИТЬ у PO текущее состояние системы]`
- Описание пусто: `Описание: [УТОЧНИТЬ у tech-lead детали интеграции с API]`

### Связанные материалы пусто
→ **Опустить блок целиком.** Не добавлять placeholder-строки типа `- нет` или `- пусто`.

## Примечания к реализации

1. **Блок Связанные материалы** включается в `description` только если в нём есть конкретные ссылки/материалы. Пусто → не включать.

2. **Символьное совпадение:** блок шаблона в Description должен совпадать символ-в-символ с версией в этом файле (пробелы, переводы строк, пунктуация).

3. **Пример кода:** любые коды (PROJ-*, INFRA-*, etc.) в примерах помечены как *иллюстративные* и заменяются реальными значениями из контекста.

4. **Язык:** все содержимое — на русском языке, соответствуя корпоративному стилю и устоявшимся терминам (например: «Образ результата», не «Vision»; «ASIS/TOBE», не «As-is/To-be»).
