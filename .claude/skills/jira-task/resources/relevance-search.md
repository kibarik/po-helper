# Процедура Relevance-research для jira-task

## Назначение

На шаге 4 пайплайна jira-task перед показом превью Issue выполняется **поиск возможно релевантного контекста** в JIRA и Confluence. Шаг помогает исполнителю найти связанные задачи, зависимости и вспомогательные материалы, но не обязателен для создания Issue. Нулевой результат поиска — это **норма**, не повод для блокировки.

---

## 1. Извлечение ключевых слов

### Источники ключевых слов

Извлечь существительные, термины и имена систем из следующих полей задачи:
- **Summary** — заголовок Issue (главный источник)
- **Образ результата** — желаемое состояние
- **ПРОБЛЕМА** — описание проблемы
- **TOBE** — целевое состояние после решения

### Алгоритм

1. **Выделить** из каждого поля главные существительные, технические термины, названия компонентов/систем/интеграций.
2. **Отфильтровать** стоп-слова (служебные глаголы вроде «сделать», «улучшить», предлоги, артикли, деепричастия типа «не работает»).
3. **Дедублицировать** — привести синонимы к одному виду (напр. «бэкенд» и «backend» → выбрать один).
4. **Получить** список из 3–6 ключевых слов/фраз для поиска.

**Пример (иллюстративный):**
- Summary: `Оптимизировать производительность API авторизации`
- Образ результата: `API авторизации возвращает результат быстрее, чем за 100ms`
- ПРОБЛЕМА: `Текущий эндпойнт /auth/login блокируется на кэше Redis`
- TOBE: `эндпойнт работает напрямую из памяти`

→ **Ключевые слова:** `API`, `авторизация`, `производительность`, `Redis`, `auth`, `/auth/login`

## 2. JIRA-запрос

### Инструмент

Использовать MCP-инструмент **`jira_search`** с параметром JQL (JIRA Query Language).

### Параметры запроса

- **Проект:** `project = <PROJ_KEY>` (где `<PROJ_KEY>` берётся из `tracker.projects[0]` или `bft.default_project` из `.claude/domain-profile.md`)
- **Текстовый поиск:** `text ~ "<keyword>"` для каждого ключевого слова (с оператором `OR` между ними)
- **Сортировка:** `ORDER BY updated DESC` (самые свежие сначала)
- **Лимит выдачи:** ~10 результатов (для дальнейшего ранжирования до ~5)

### Пример (иллюстративный)

```jql
project = INFRA AND (text ~ "API" OR text ~ "авторизация" OR text ~ "Redis" OR text ~ "performance") ORDER BY updated DESC
```

### Обработка результатов

Для каждого найденного Issue сохранить:
- **Ключ** (напр. `INFRA-1234`)
- **Summary**
- **Description** (первые 200–300 символов для понимания контекста)
- **Updated** (дата последнего обновления, поможет оценить свежесть)

---

## 3. Confluence-запрос

### Инструмент

Использовать MCP-инструмент **`confluence_search`** с теми же ключевыми словами.

### Параметры запроса

- **Пространство:** `space = <SPACE_KEY>`, где `<SPACE_KEY>` берётся из `wiki.space` в `.claude/domain-profile.md`
- **Текстовый поиск:** `text ~ "<keyword>"` для каждого ключевого слова
- **Лимит выдачи:** ~10 результатов

### Fallback: отсутствие wiki-конфига

**Если в `.claude/domain-profile.md` не заданы ключи `wiki.mcp` и `wiki.space`:** Confluence-поиск **пропускается**. Шаг выполняет поиск только по JIRA. Явно сообщить в превью: «Confluence недоступна; поиск выполнен только в JIRA».

### Обработка результатов

Для каждой найденной страницы сохранить:
- **URL** (полная ссылка на страницу)
- **Заголовок** (title)
- **Краткий отрывок** (первые 200–300 символов содержимого)
- **Space** (пространство, из которого страница)

---

## 4. Ранжирование кандидатов

### Принцип

Объединить результаты JIRA и Confluence в единый список. Отранжировать по релевантности исходя из совпадения ключевых слов и сущностей в заголовке и описании.

### Алгоритм

1. **Совпадение терминов:** чем больше ключевых слов из списка пункта 1 присутствует в Summary/заголовке, тем выше релевантность.
2. **Точность совпадения:** полные совпадения термина выше, чем частичные (напр. `Redis` > `redis-cache`).
3. **Свежесть:** для JIRA — Updated DESC (новые задачи выше). Для Confluence — если дата опубликования доступна, приоритизировать свежие.
4. **Тематическое соответствие:** результаты, явно упоминающие проблему или решение из ПРОБЛЕМА/TOBE, выше.

### Отбор

Выбрать **до ~5 наиболее релевантных** кандидатов (JIRA + Confluence вместе). Если результатов меньше 5 — вывести все. Если совпадений нет — вывести пустой список (см. раздел 7 «Правило тишины»).

---

## 5. Формат кандидата

### Строка кандидата

```
- <ключ Issue или полный URL страницы> — <одна строка: почему релевантно>
```

### Правила оформления

- **Для JIRA Issue:** `- <ключ> — <обоснование>` (напр. `- INFRA-1234 — в этой задаче обсуждалось кэширование Redis`)
- **Для Confluence:** `- <полный URL> — <обоснование>` (напр. `- https://wiki.example.com/spaces/ARCH/pages/12345/... — документация по интеграции с Redis`)
- **Обоснование (почему релевантно):** одна лаконичная строка, указывающая какой из ключевых слов совпал, или в чём совпадение тематики. Не писать многостраничное резюме.

### Примеры (иллюстративные)

```
- INFRA-1234 — обсуждение оптимизации производительности API
- INFRA-5678 — связанная задача про Redis-кэш для авторизации
- https://wiki.example.com/ARCH/redis-patterns — гайд по шаблонам Redis
```

## 6. Принцип нулевого допуска к галлюцинациям

### Правило

Включить в список кандидатов **только те записи, которые реально вернул поиск** (JIRA или Confluence). Каждый кандидат должен иметь **якорь**:
- Для JIRA: реальный ключ Issue (напр. `INFRA-1234`), вернённый `jira_search`
- Для Confluence: реальный URL страницы, вернённый `confluence_search`

### Запрещено

- ✗ Додумывать или изобретать Issue-коды, которых нет в результатах поиска
- ✗ Создавать новые ссылки на основе предположений («похоже, должна быть задача про...»)
- ✗ Добавлять ссылки на документацию/страницы, которые не вернула `confluence_search`

### Следствие

Если поиск не вернул релевантных результатов → список кандидатов остаётся **пустым**. Это **не ошибка**.

---

## 7. Правило тишины (silence rule)

### Принцип

Нулевой или пустой результат поиска по JIRA и Confluence — это **нормальное явление**, не повод для алерта или уточнения.

### Поведение

- **Совпадений не найдено** → при показе превью (шаг 5) явно указать: «Релевантные материалы не найдены; это нормально».
- **Блок не добавляется** в финальное описание Issue (шаг 6), если кандидатов нет.
- **Не писать** `[УТОЧНИТЬ]`, `TODO`, `?` или другие маркеры незаполненности.

### Следствие

Шаг Relevance-research **никогда не блокирует создание Issue**. Если поиск ничего не вернул — Issue всё равно создаётся по явному подтверждению PO.

---

## 8. Read-only принцип

### Границы шага

На этапе поиска (шаг 4) выполняются **только операции чтения**:
- `jira_search(...)` — поиск по JQL
- `confluence_search(...)` — поиск по ключевым словам

### Запрещено

- ✗ Создание Issue (`jira_create_issue`)
- ✗ Создание связей (`jira_create_issue_link`, `jira_create_remote_issue_link`)
- ✗ Редактирование существующих задач
- ✗ Добавление комментариев, меток, переходов

### Следствие

Результаты поиска (кандидаты) **показываются PO в превью** (шаг 5) без автоматического прикрепления. PO выбирает, какие материалы оставить. Прикрепление ссылок происходит только на шаге 6 (Create) по явному подтверждению PO.

---

## Конфигурация из domain-profile.md

Все параметры поиска берутся из `.claude/domain-profile.md`:

| Параметр | Источник | Применение |
|----------|----------|-----------|
| Проект JIRA | `tracker.projects[0]` или `bft.default_project` | Значение `project = <PROJ_KEY>` в JQL |
| MCP-сервер JIRA | `tracker.mcp` | Подключение к `jira_search` |
| Space Confluence | `wiki.space` | Ограничение поиска в `confluence_search` |
| MCP-сервер Confluence | `wiki.mcp` | Подключение к `confluence_search` |

**Если `wiki.mcp` или `wiki.space` не установлены:** шаг выполняет поиск только по JIRA (fallback, см. раздел 3).

---

## Резюме процедуры

1. Извлечь 3–6 ключевых слов из Summary, Образ результата, ПРОБЛЕМА, TOBE
2. Выполнить `jira_search` с JQL, включив все ключевые слова
3. Выполнить `confluence_search` по тем же словам (или пропустить, если wiki не настроена)
4. Объединить и отранжировать результаты, выбрав до ~5 наиболее релевантных
5. Оформить каждого кандидата в формате `- <ключ/URL> — <почему релевантно>`
6. Убедиться, что все якоря реальные (нет галлюцинаций)
7. Если совпадений нет → вывести пустой список (это норма)
8. Показать результаты PO в превью; PO выберет, что оставить
9. Прикрепить выбранные ссылки только после явного подтверждения (шаг 6)
