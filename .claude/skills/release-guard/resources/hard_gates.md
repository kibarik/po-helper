# Hard Gates — валидация baseline релиза (10 бинарных гейтов)

Проверяются на `/release-baseline` перед фиксацией v1 и на `/release-gate` перед каждой новой версией. Каждый гейт бинарный: ✅ / 🔴. Один 🔴 в критичных (1–6) → baseline не фиксируется.

## Светофор

| Статус | Условие | Действие |
|--------|---------|----------|
| 🟢 Зелёный | Все 10 гейтов ✅ | Зафиксировать baseline, → handoff в bft-writer / запуск /release-sync |
| 🟡 Жёлтый | Нарушены гейты 7–10 (некритичные) | Зафиксировать с явными `[УТОЧНИТЬ]`, PO принимает риск |
| 🔴 Красный | Нарушен хотя бы один гейт 1–6 | Вернуться, дозаполнить baseline |

---

## Критичные гейты (1–6) — блокирующие

### Гейт 1: Форма обязательства задана
**Правило:** `commitment_mode` ∈ {date_fixed, scope_fixed, both_fixed}. Для `date_fixed` есть `fixed_date`.
**Нарушение:** не выбрана вершина iron triangle.

### Гейт 2: both_fixed не зафиксирован молча
**Правило:** Если `commitment_mode = both_fixed` → в baseline есть раздел встречки (что некоммитбельно при текущем CP) И явное решение PO/стейкхолдера принять риск.
**Нарушение:** обещаны обе вершины без встречки — это и есть корень продавливания.

### Гейт 3: Уровень обязательства соответствует CP
**Правило:** `commitment_level = commitment` требует `cp = high` И заполненную оценку. Нет оценки/БФТ → максимум `target`. Нет ни того ни другого → `intent`.
**Нарушение:** `Commitment` при `cp = low` или без оценки — обещание на дне неопределённости.

### Гейт 4: Definition of Releasable заполнен
**Правило:** ≥ 1 проверяемый критерий отгружаемости. Закладывается сейчас, не в конце.
**Нарушение:** пустой DoR → «не можем выкатить» в финале.

### Гейт 5: Scope IN/OUT с границами
**Правило:** заполнены и `Scope IN`, и `Scope OUT` (границы важнее содержимого).
**Нарушение:** пустой `Scope OUT` — нет границы, любое требование «всасывается».

### Гейт 6: Нет факта без источника
**Правило:** оценка, дата, заказчик, риски имеют якорь (tracker_key, PO-решение, риск-реестр, встреча).
**Нарушение:** «оценка 5 спринтов» без того, кто и на чём её дал.

## Некритичные гейты (7–10) — 🟡 допустимы

### Гейт 7: Буфер обоснован риск-реестром
**Правило:** `contingency_buffer > 0` и `risk_refs` непуст.
**Нарушение:** буфер 0 или с потолка → первый же лебедь сразу в дрейф.

### Гейт 8: Параметры мониторинга заданы
**Правило:** `drift_threshold`, `staleness_n`, `sync_period_days`, `estimate_unit`, `sprint_capacity` заполнены (дефолты из профиля допустимы).
**Нарушение:** не из чего считать дрейф/staleness.

### Гейт 9: У требований есть owner
**Правило:** каждая строка трекинг-таблицы имеет `owner` (кто актуализирует статус).
**Нарушение:** требование без владельца → staleness не на ком считать.

### Гейт 10: Получатели алертов заданы
**Правило:** `alert_recipients` непуст.
**Нарушение:** алерт некому отправить → раннее предупреждение в пустоту.
