# Sprint Standards — форматы, capacity-модель, шкалы

## Модель ёмкости (capacity)

Цель планирования — **максимально загрузить каждого** без перегруза. Ёмкость считается по человеку на спринт:

```
ёмкость(человек) = velocity(человек) × focus_factor − отпуска/недоступность
```

| Параметр | Дефолт | Источник | Комментарий |
|---|---|---|---|
| `velocity` | из истории (3–5 спринтов) | ФАКТ прошлых спринтов | SP, которые человек реально закрывает за спринт |
| `focus_factor` | 0.7 | профиль `capacity.focus_factor` | доля времени на плановую работу (Scrum: 0.6–0.8) |
| отпуска | — | диалог `/sprint-sync` | вычитаются пропорционально дням |
| `sp_per_person_sprint` | — | профиль `capacity` | если истории нет — стартовая оценка ёмкости |

Если velocity неизвестна (новый человек/нет истории) → использовать `sp_per_person_sprint` из профиля + пометить `[УТОЧНИТЬ velocity]`. Не выдумывать цифры.

**Флаги загрузки** (стадия `/sprint-load`):
- 🔴 **Перегруз** — сумма SP > ёмкость. Действие: снять Should/Could или передать другому.
- 🟡 **Недогруз** — сумма SP < 70% ёмкости. Действие: добрать историю из бэклога/N+1/внеплановых.
- 🟢 **Сбалансирован** — 70–100% ёмкости.

## SP-шкала (Story Points, Фибоначчи)

| SP | Объём | Ориентир |
|----|-------|----------|
| 1 | тривиально | < полудня, понятно полностью |
| 2 | малая | ~1 день |
| 3 | средняя | 2–3 дня |
| 5 | крупная | ~неделя, есть неизвестность |
| 8 | очень крупная | весь спринт одним человеком — кандидат на декомпозицию |
| 13+ | эпик | **обязательно декомпозировать** на стадии `/sprint-decompose` |

История > 8 SP в Must → 🔴 на гейте: разбить.

---

## Приоритеты (MoSCoW)

| Приоритет | Значение | Правило |
|-----------|----------|---------|
| **Must** | Без этого Sprint Goal не достигнут | Берётся первым. Не должно содержать TBD-исполнителя. |
| **Should** | Важно, но Sprint Goal выживет | Берётся после Must при наличии ёмкости. |
| **Could** | Желательно | Только если осталась ёмкость. Первый кандидат на снятие при перегрузе. |
| **Won't** | Сознательно не в этом спринте | В блок N+1 или бэклог. |

## Теги историй

| Тег | Что означает |
|-----|--------------|
| (без тега) | Плановая история, наследует KR из roadmap |
| `[CARRYOVER]` | Перенос незакрытой истории прошлого спринта |
| `[TECH-DEBT]` | Технический долг |
| `[BUG]` | Дефект продакшна |
| `[SPIKE]` | Исследование/PoC, образ результата = решение/декомпозиция |
| `[UNPLANNED]` | Влёт вне OKR (служит стратегии, но не отображён в планах) |

`[CARRYOVER]`/`[TECH-DEBT]`/`[BUG]`/`[UNPLANNED]`/`[SPIKE]` → секция «Необходимые внеплановые работы», если не наследуют KR напрямую.

## Колонки таблицы плана (порядок фиксирован)

```
| ЭПИК | История | Образ результата | Образ действия | Исполнитель | Приоритет | SP |
```

- **ЭПИК** — ключ JIRA-эпика или `TBD` (с пометкой «завести эпик»).
- **История** — `[РОЛЬ]: краткое действие` (`BE: …`, `FE: …`, `SA: …`, `QA: …`, `DevOps: …`).
- **Образ результата** — доказуемое состояние к концу спринта; для технических — `БЫЛО: … → СТАЛО: …`.
- **Образ действия** — атомарные шаги (через `<br>` в ячейке), теги ролей `[BA]/[SA]/[BE]/[FE]/[QA]/[RELEASE]/[ADR]/[PO]`, `[BLOCKER]`/`[RISK]`.
- **Исполнитель** — имя из ростера (NEXUS/team). `TBD` запрещён в Must.
- **Приоритет** — Must/Should/Could.
- **SP** — число по шкале выше.

## Структура финального документа (план спринта)

```markdown
# {sprint} — ПЛАН спринта

**Период:** {даты} ({sprint_weeks} недели)
**Ответственный:** {po_name}
**Статус:** 📋 Планирование

## ЦЕЛЬ: {название OBJ} (OBJ N)
KR над которыми работаем:
- KR N.M [[ссылка]]

**🎯 Образ результата команды (Sprint Goal):** {что команда показывает руководителю к концу спринта}

| ЭПИК | История | … таблица … |

**Дополнительный контекст и риски:** …

## Необходимые внеплановые работы
| ЭПИК | История | … | тег |

## 📊 Capacity по командам
| Исполнитель | Ёмкость (SP) | Загрузка (SP) | Статус | Основная нагрузка |

## 👤 Персональный фокус (что лично каждый показывает к концу)
| Исполнитель | Что показываю к концу спринта |

## ⚠️ Общие риски
| Риск | Mitigation |

## 🔭 Спринт N+1 (предварительно)
{образ результата + крупные истории без точного SP}

*📋 Следующие шаги: …*
```

Двойная оптика обеспечивается двумя блоками: **Sprint Goal** (что команда показывает руководителю) + **Персональный фокус** (что лично каждый показывает).

## Блок N+1 (горизонт двух спринтов)

Лёгкий, без жёсткого коммита (rolling-wave): по каждому продолжающемуся OBJ/KR — образ результата следующего спринта и 2–4 крупные истории с **грубой** оценкой (`~SP` или T-shirt S/M/L), без исполнителей. Источник — SPRINT-ROADMAP. Назначение: чтобы команда видела ближайшие 2 спринта и готовила бэклог (backlog refinement).

---

## Матрица SPRINT-ROADMAP (уровень A)

Распределение срезов KR по спринтам квартала:

```markdown
# SPRINT-ROADMAP {quarter}

## Легенда capacity
Спринтов в квартале: N · Ёмкость команды/спринт: ~X SP

| OBJ | KR | Образ результата KR (квартал) | S1 | S2 | S3 | … | Итого SP |
|-----|----|-------------------------------|----|----|----|----|---------|
| 2 | 2.1 | Кино через UMC в проде | срез: BE загрузка (8) | срез: FE витрина (8) | … | | 30 |

## Вехи квартала
- S{N}: {milestone}

## Грубая загрузка по спринтам
| Спринт | Плановые SP | Ёмкость команды | Запас |
```

Ячейка спринта = краткий срез KR, который планируется продвинуть в этом спринте (+ грубый SP). Пустая ячейка = KR в этом спринте не двигается. Каждый KR ссылается только на существующие KR из `{okr_output_doc}`/`{index_doc}` — не выдумывать.
