# JIRA Build Know-How — материализация спринта в трекере (BUILD-проход)

Правила и ограничения записи спринта в трекер. Читаются `/sprint-build` и `/sprint-activate`.
Специфика инстанса (id полей/типов, ключи проектов) — из секции `tracker` domain-profile, **не хардкодить**. Значения-примеры ниже (Jira Server/DC, кейс GDSLV/TLND) — иллюстративны.

---

## Доступ и гигиена

- Трекер — **только через MCP** (`tracker.mcp`). Токены/PAT руками не собирать, в чат не вставлять.
- Нет MCP / нет VPN → **остановиться**, попросить поднять MCP. Не искать обходной маршрут.
- BUILD-запись разрешена только при `tracker.access: build` в профиле. `read-only` → BUILD запрещён.
- Секреты сессии (если всплыли) — ротировать после хендоффа.

---

## Модель связей (Jira Server/DC)

- Связь **история → эпик** = **Epic Link** (`tracker.epic_link_field`, напр. `customfield_10101`), **НЕ** `parent`.
- **Подзадача наследует спринт от родителя** — независимо в спринт не встаёт. Ставить в спринт родительскую историю.
- Типы подзадач — карта `tracker.issuetype.subtask` (роль → id). Story — `tracker.issuetype.story`.
- ⚠️ **Существующую задачу нельзя сделать подзадачей через REST/MCP.** Только Move-визард в UI. → Группировку проектировать сразу через **Epic Link**, а не через конвертацию в подзадачу.

---

## Слоты спринтов

- Перед записью — инвентарь слотов (active/future) **с числом задач в каждом**.
- Писать только в **ПУСТОЙ** слот (или явно выбранный PO). Имя+даты зафиксировать один раз, не дрейфить.
- 🚪 **slot-safety**: не репойнтить непустой слот без явного решения PO (см. `build_gates.md`).

---

## Порядок записи (батчами, с верификацией)

Каждый батч: **dry-run (что и куда) → approve PO → write → верификация чтением**. Не лить всё разом.

1. **Существующие истории → спринт** (подзадачи подтянутся сами от родителя).
2. **Переназначить исполнителей** по образу результата PO (только валидные учётки — см. discovery).
3. **Снять `On Hold`/`В ожидании`** где нужно — доступным transition.
4. **Создать NEW** как подзадачи под родными историями (валидный тип из `issuetype.subtask`). Только то, чего нет в беклоге (см. reuse-матрицу из `/sprint-decompose`).
5. **Кросс-проектный reuse** (напр. TLND↔GDSLV) — допустим, помечать флагом источника.
6. **Orphan-группировка** — привязать сирот через **Epic Link** к эпику (+ link «Relates» на umbrella-US). Цель — ноль орфанов.

🚪 **cascade-preview**: перед добавлением эпика/родителя в спринт — показать, сколько детей подтянется.

---

## Connectivity (fallback, если MCP недоступен и PO разрешил прямой маршрут)

Приоритет всегда — MCP. Если MCP лёг, а доступ нужен:
- Убрать `HTTP(S)_PROXY`/`ALL_PROXY` из окружения → прямой маршрут.
- Системный `curl`, PAT bearer; корпоративный сертификат валиден напрямую.
- Это аварийный путь; штатно — MCP.

---

## Верификация после батча

- Прочитать записанное обратно (spot-check ключей): спринт проставлен, Epic Link не пуст, исполнитель валиден, статус снят с hold.
- Расхождение → откатить/поправить до следующего батча, не накапливать дрейф.
