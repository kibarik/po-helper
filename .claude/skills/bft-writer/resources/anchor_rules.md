# Правила якорения (Anchor Rules) — что считается доказательством факта

> Главный рычаг против галлюцинаций: каждое утверждение в БФТ ← якорь. Нет валидного якоря → `[УТОЧНИТЬ]`, не выдумывать. Адаптация sa-helper (якорь `code:line`) под БФТ.

## Ранги достоверности якоря

| Ранг | Тип якоря | Когда валиден | Пример |
|---|---|---|---|
| **R1** 🥇 | **Код** `file:line` (через sa-helper graph/repomix или прямой код) | **ТОЛЬКО As-Is** (текущее поведение системы) | `extapi/order.go:142` |
| **R2** 🥈 | Задача трекера (`KEY-123`), страница wiki (`pageId`), бизнес-правило (`BR-*`), ADR (`ADR-ID`) | As-Is и To-Be | `[TLND-16189](https://jira.mts.ru/browse/TLND-16189)`, `[Confluence 2272447498](https://confluence.mts.ru/pages/viewpage.action?pageId=2272447498)`, `BR-DATA-002` |
| **R3** 🥉 | Решение PO/заказчика: **кто ответил + дата + источник** (встреча/почта/ТЗ) | To-Be (forward-looking) | `PO (Иванов, 21.06, встреча)` |
| **R4** ❌ | «Обычно так», «лучшая практика», общие знания | **НЕ валиден** | — |

## Корректива для forward-looking БФТ

БФТ описывает To-Be — того, чего часто ещё нет в коде. Поэтому:

- **As-Is** (что есть сейчас) → якорь R1 (код) или R2 (трекер/wiki/BR/ADR).
- **To-Be** (требования, целевое) → якорь R2 (ТЗ/задача) или R3 (решение PO).
- **Требовать R1 (код) для To-Be — ошибка**: нельзя доказать кодом то, чего ещё нет. Для forward-требований валиден R3 (PO-решение).

## Hallucination Gate (🔴)

Утверждение о факте (As-Is или To-Be) без якоря R1/R2/R3 → 🔴. Исключение — явно помеченное `[УТОЧНИТЬ у {кого}]`.

Оценочные/рыночные утверждения без ранга R1–R3 («быстро», «масштабируемо», «обычно так делают») → 🔴.

## Правила
1. Каждое ключевое утверждение в As-Is, БТ, ФТ, НФТ — имеет якорь R1/R2/R3.
2. Код-якорь (R1) — только для As-Is.
3. To-Be требования → R2 (ТЗ/задача) или R3 (PO-решение), не R1.
4. Несколько якорей лучше одного (перекрёстная проверка).
5. Решение PO всегда с «кто + когда + источник», иначе это не якорь.
