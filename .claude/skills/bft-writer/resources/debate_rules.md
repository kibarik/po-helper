# Правила Adversarial Review для БФТ

Адаптация sa-helper `/fnr-debate` (`architectural-debate/resources/debate_rules.md`): красная команда проходит по **требованиям** БФТ, а не по коду. Цель — выловить пропущенные negative flows, скрытые зависимости, неизмеримые НФТ и потерянную трассировку ДО фиксации документа.

## Общие правила

1. Review проводится в **3 раунда**. Количество фиксировано.
2. Каждый раунд чередует: **Аналитик** (защита) → **Адвокат Дьявола** (атака).
3. Модератор (LLM) не занимает сторону — обеспечивает структуру.
4. Каждый аргумент/контраргумент привязан к **источнику** (JIRA / PO / ТЗ) или к конкретному месту в черновике БФТ.
5. Результат дописывается в конец черновика БФТ в блок `## Adversarial Review`.

---

## Раунд 1 — Защита

**Аналитик (обоснование требований):**
- Проходит по БТ/ПТ/ИТ/ФТ/НФТ и формулирует 3–5 аргументов «почему требование корректно и достаточно».
- Каждый аргумент — с источником (JIRA / решение PO / ТЗ).

**Адвокат Дьявола (атака):**
- 3–5 контраргументов. Каждый конкретен и привязан к реальному риску.
- **Обязательные линзы атаки** (минимум по одной):
  - **Нагрузка** — что при пике (топ-мероприятие, старт распродажи)?
  - **Отказ компонента** — что если поставщик (Legacy-платформа/Партнёр-API) недоступен?
  - **Идемпотентность / гонки** — двойной клик, повторный запрос, конкурентное бронирование?
  - **Изменение требований** — «а если через 6–12 месяцев добавится новый поставщик / метод оплаты?»
  - **ПДн / Compliance** — 54-ФЗ, ФЗ-193 (возвраты), хранение ПДн?
  - **Скрытые зависимости** — между командами (API-слой↔Шина↔Процессинг↔BI↔1С↔Финтех), статус согласования?
  - **Трассировка** — есть ли «висячие» требования без связи к БТ/эпику?

**Чек-лист Аналитика:** выбраны конкретные требования · ≥3 аргумента · каждый с источником.
**Чек-лист Адвоката:** ≥3 контраргумента · каждый конкретен (не «это плохая практика») · пройдены обязательные линзы.

---

## Раунд 2 — Пересмотр

**Аналитик (ответ):**
- Отвечает на каждый контраргумент. Усиливает требование (добавляет negative flow / НФТ / уточняет Границы) или пивотит.
- Если усиливает — явно вносит правку в черновик.

**Адвокат Дьявола (поверка):**
- Проверяет усиленный вариант на новые слабости.
- Задаёт: «А что если условие X изменится через 6–12 месяцев?».

**Допустимые исходы:** Аналитик усилил → Раунд 3; пивот → проверка нового; не ответил → вердикт «забраковано»/«пивот».

---

## Раунд 3 — Вердикт

**Аналитик:** итоговая позиция (1–2 предложения).
**Адвокат Дьявола:** итоговая позиция (1–2 предложения).

**Вердикт модератора:**

| Тип | Условие |
|-----|---------|
| Принят | Обе стороны согласны, критика адресована |
| Принят с модификациями | Аналитик усилил требования, Адвокат признал достаточным |
| Пивот | Изменён состав/приоритет требований |
| Забраковано | Выявлены неприемлемые риски/пробелы, не митигированы |
| Не техническая проблема | Требует организационного/процессного решения (эскалация PO) |

---

## Запреты

1. Не формулировать вердикт до Раунда 3.
2. Не игнорировать контраргументы Адвоката.
3. Не фабриковать источники — только JIRA / PO / ТЗ / реальная запись.
4. Не менять субъект review (переходить на другой БФТ).
5. Не «сглаживать» — если Адвокат нашёл реальный пробел, он фиксируется как `[УТОЧНИТЬ]` или правка требования.
