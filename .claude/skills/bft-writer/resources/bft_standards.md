# Стандарты БФТ (BFT Standards)

Единообразие — условие проверяемости. Все БФТ следуют одному формату. Адаптируйте имена полей/сервисов под ваш проект.

## Стиль и голос

БФТ пишется плотным деловым русским, как золотые референсы (`{bft_store}/золотой референс …`, `examples/golden_bft_example.md`). Стиль — enforcement-слой, проверяется гейтом 13. Полный набор правил: `resources/writing_style.md`.

Кратко:
- Без эмодзи в теле (статусные значки только в `validation.md`).
- Без `**bold**` внутри ячеек таблиц требований (кроме story-ключей ПТ: `**Когда**/`**Я хочу**/`**Чтобы**`).
- Без callout-врезок `> **⚠️ …`.
- Em-dash — для определений и перечислений, не как связка.
- Без стоп-слов: «позволяет», «обеспечивает», «осуществляет», «производит», «представляет собой», «важно отметить», «в рамках данного».
- `[УТОЧНИТЬ]` без кросс-рефов; неизвестное предпочтительно естественной прозой («уточняется у X», «предварительно», «TBD») как в золотой референс.
- Frontmatter без diff-narrative; история правок — в разделе «История изменений».

## Порядок разделов (фиксирован — ЗМ-005, lean-форма)

БФТ пишется в **lean-форме**: минимум разделов, максимум сканируемости. Разделы идут строго в этом порядке:

0. **Оглавление** — в самом начале, после H1/frontmatter (обязательно; ЗМ-006)
1. **О задаче** — краткое погружение в контекст, без вдумчивого вычитывания (ЗМ-007)
   - 1.1 As-Is → Gap → To-Be
   - 1.2 Контекст обращения
   - 1.3 Стейкхолдеры
2. **План демо для стейкхолдера** — договорённость команды об образе приёмки (ЗМ-008)
   - 2.1 Ключевой сценарий (таблица)
   - 2.2 Альтернативные ветки — ЧТО ЕСЛИ (таблица)
   - 2.3 Недопустимые события (таблица)
   - 2.4 Диаграмма демо (actor-level sequence + таблица акторов)
3. **Критерии успеха**
   - 3.0 Границы: Не входит / Вынесено в СА (scope-exclusions; ЗМ-001)
   - 3.1 Бизнес-требования БТ
   - 3.2 Пользовательские требования ПТ
   - 3.3 Требования к интерфейсам ИТ
   - 3.4 Функциональные требования ФТ
   - 3.5 Нефункциональные требования НФТ
4. **Риски и зависимости** — единый раздел (зависимость = форма риска; ЗМ-009)
5. **Открытые вопросы** (+ Ключевые решения)
6. _(опц. подвал)_ История изменений · Ревью · Якоря истины · Дополнительные материалы · Словарь

**Что изменилось против прежнего 19-разд. шаблона:**
- Верхние разделы (Общая информация / Бизнес описание / Заинтересованные стороны) свёрнуты в один краткий **«О задаче»** с подпунктами.
- **As-Is → Gap → To-Be** — теперь подпункт 1.1 «О задаче», не отдельный верхнеуровневый раздел.
- **Границы системы** больше не отдельный обязательный раздел: сохраняются только scope-exclusions (**Не входит / Вынесено в СА**) как подпункт 3.0 внутри «Критерии успеха». Блок «Входит» — свернуть в «О задаче».
- **БТ/ПТ/ИТ/ФТ/НФТ** — подпункты 3.1–3.5 под зонтиком «Критерии успеха», не отдельные разделы.
- **Сценарии взаимодействия** переработаны в **«План демо для стейкхолдера»** (3 обязательных блока + actor-level sequence).
- **Зависимости** и **Риски** объединены в один раздел «Риски и зависимости».
- **Якоря истины / Ревью / История / Словарь** — опциональный подвал (не обязательные разделы). Дисциплина якорения при этом сохраняется inline (гейт 1): каждый факт ссылается на якорь в тексте / в «Связанные требования».

Открытые вопросы — ближе к концу, но всегда присутствуют (блокеры на виду через «Контекст обращения» в начале). Эталон структуры — `examples/ideal_bft.md`.

## Идентификаторы требований

```
{TYPE}-{N}
TYPE  ∈ {БТ, ПТ, ИТ, ФТ, НФТ}
N     = порядковый в пределах типа (1, 2, 3 …)
```

**Идентификаторы локальные** (ЗМ-003): действительны **строго внутри документа** — это якоря для быстрой навигации/трассировки между разделами. Не глобальные ключи. **Не добавляй EPIC/проект/сервис в идентификатор** (`БТ-TL-RestNotify-1` ❌ → `БТ-1` ✅). Связь с эпиком/трекером идёт **только** через колонку «Связанные требования» (ключ задачи `PROJECTKEY-XXX`), не через идентификатор.

Примеры: `БТ-1`, `ПТ-2`, `ИТ-3`, `ФТ-1`, `НФТ-5`.

- **БТ** — Бизнес-Требование (ценность).
- **ПТ** — Пользовательское требование (story).
- **ИТ** — Требование к интерфейсу приёмки: что видит и может сделать актор на экране и в результате. Не API-контракт.
- **ФТ** — Функциональное требование (поведение).
- **НФТ** — Нефункциональное требование (измеримое -ility).

Трассировка каскадом: `БТ ← ПТ ← ИТ` и `БТ ← ФТ ← НФТ`. Связь с эпиком в трекере — через «Связанные требования», не через идентификатор.

> Единообразие: один и тот же шаблон `{TYPE}-{N}` и в колонке «Идентификатор», и в «Связанные требования». Смешение (`БТ-TL-RestNotify-1` в заголовке, `БТ-1` в связях) — ошибка (гейт 3).

## Различение типов (частая ошибка LLM — не путать)

| Тип | О чём | НЕ должен содержать |
|---|---|---|
| **БТ** | Ценность, зачем (почему) | системного поведения, -ilities |
| **ПТ** | Что хочет пользователь (story) | как система устроена внутри |
| **ИТ** | Интерфейс приёмки: экраны, доступность, адаптив, что видит и может актор | API-контракта (эндпоинты, протоколы, схемы сообщений), внутренней реализации |
| **ФТ** | Поведение системы на уровне результата (что делает для пользователя/бизнеса) | ценности (это БТ), -ilities (это НФТ), кода/API-деталей |
| **НФТ** | Измеримое -ility — порог приёмки (как качественно) | функционального поведения |

> Гейт: ПТ, описывающий системное поведение → это ФТ. ФТ со словом «быстро/надёжно» → это НФТ.

> **Высота БФТ (гейт 14).** API-контракты (эндпоинты, REST, схемы запросов/ответов, идемпотентность и auth как контракт), sequence-диаграммы с вызовами между сервисами, имена таблиц/полей БД, код — **вне зоны БФТ**. Это зона СА / системных требований (FNR-пайплайн). БФТ описывает результат и приёмку, не внутреннее устройство системы.

## Frontmatter (синхронизация с wiki)

```yaml
---
source: <URL страницы в вашей wiki>
space: <PROJ>            # пространство wiki, опц.
version: <N>
synced: <YYYY-MM-DD>
jira: <PROJECTKEY-XXX>   # эпик в трекере, опц.
status: Черновик 0.X     # опц.
---
```

H1: `# [БФТ] {EPIC}: {Название}` или `# БФТ {EPIC} – {Название}`.

## Ссылки на Jira / Confluence (ЗМ-004)

Любая ссылка на Jira/Confluence — **полноценной markdown-ссылкой**, не голым текстом.

| Источник | Голый текст ❌ | Markdown-ссылка ✅ |
| --- | --- | --- |
| Jira | `JIRA GDSLV-1409` | `[JIRA GDSLV-1409](https://jira.mts.ru/browse/GDSLV-1409)` |
| Confluence (pageId) | `Confluence 2272447498` | `[Confluence 2272447498](https://confluence.mts.ru/pages/viewpage.action?pageId=2272447498)` |

**Базовые URL** (корпоративные, поправь под свой проект):
- Jira: `https://jira.mts.ru/browse/{KEY}`
- Confluence: `https://confluence.mts.ru/pages/viewpage.action?pageId={pageId}`

Касается **всех** мест: «Связанные требования», «Якоря истины», «Дополнительные материалы», frontmatter `source`, текст разделов. Текст ссылки сохраняет ключ/pageId для читаемости (`JIRA GDSLV-1409`), URL — в скобках. Локальный идентификатор требования (`{TYPE}-{N}`) ссылкой НЕ оборачивается — ссылку получает трекер-ключ в «Связанных требованиях».

## Форматы таблиц по типам требований

**БТ:** `Идентификатор | Наименование | Ценность | Краткое описание | Связанные требования`

**ПТ:** `Идентификатор | Наименование | Story | Связанные требования`
Story-формат: `**Когда** я (актор) {действие} **Я хочу** {результат} **Чтобы** {обоснование}`

**ИТ:** `Идентификатор | Наименование | Требование | Связанные требования`
Требование — интерфейс приёмки: что видит и может актор, экраны, доступность, адаптив. Контекст продукта (платформа / браузеры / разрешение / адаптив) — если применимо. Не API-контракт.

**ФТ:** `Идентификатор | Наименование | Приоритет | Функциональные требования | Параметры, ограничения | Связанные требования`
Приоритет: **Высокий / Средний / Низкий** (пусто запрещено).

**НФТ:** `Идентификатор | Наименование | Описание | Связанные требования`

## Стандартный НФТ-набор (включать по умолчанию для бэкенд-API)

| Группа | О чём |
|--------|-------|
| **Идемпотентность** | Повторный вызов не выполняет бизнес-логику повторно |
| **SLA / латентность** | P95 / P99 по операциям (напр. запись 2000 ms P95, чтение 200 ms P95) |
| **Нагрузка** | RPS на чтение / на запись (с пометкой «предварительно», если не подтверждено) |
| **RED-метрики** | Попытки / успех / неуспех, сквозная латентность, ошибки интеграций |
| **Алертинг** | Пороги: рост ошибок, рост таймаутов, аномальный рост конфликтов |
| **Логирование** | Состав событий, **без чувствительных данных**, срок хранения, сквозной trace-id |
| **Доступ / аудит** | Сервисные учётки, TLS, аудит ключевых операций |
| **Надёжность / целостность** | Атомарность операций, идемпотентность на уровне БД, алерты на критические ошибки |

## Корпоративный реестр НФТ (приоритет)

Если в проекте есть **реестр НФТ** (каталог стандартных нефункциональных требований) — **приоритет брать НФТ оттуда**, а не формулировать с нуля. Ссылайтесь на ID из реестра. Не выдумывайте НФТ, если есть готовый корпоративный стандарт (безопасность, хранение, SLA и т.д.).

## Оглавление (обязательно — ЗМ-006)

В начале документа, сразу после H1/frontmatter — markdown-список ссылок-якорей на все разделы и ключевые подпункты (минимум до 2-го уровня: 1.1, 2.1 …). Цель — быстрая навигация без прокрутки. Пример:

```markdown
## Оглавление
- [1. О задаче](#1-о-задаче)
  - [1.1 As-Is → Gap → To-Be](#11-as-is--gap--to-be)
  - [1.2 Контекст обращения](#12-контекст-обращения)
  - [1.3 Стейкхолдеры](#13-стейкхолдеры)
- [2. План демо для стейкхолдера](#2-план-демо-для-стейкхолдера)
  …
```

## О задаче (ЗМ-007)

**Краткий вводный раздел — погрузить в контекст без вдумчивого вычитывания каждой строки.** Максимум таблиц и буллетов, минимум прозы. 1–2 вводные строки + три подпункта:

- **1.1 As-Is → Gap → To-Be** — таблица (формат ниже).
- **1.2 Контекст обращения** — 2–4 буллета: причина/повод обращения, статус эпика в трекере, важные оговорки. Неизвестное — естественной прозой («уточняется у X»), не `[УТОЧНИТЬ]` в каждой строке.
- **1.3 Стейкхолдеры** — таблица `Роль | Кто | Интерес`.

Блок «Входит» (что в скоупе) — 1–2 строки здесь же, если помогает погружению. Детальные scope-exclusions — в 3.0.

**As-Is → Gap → To-Be (подпункт 1.1).** Формат — таблица, НЕ сплошной текст (гейт 13 / `ЗМ-002`). Вертикальная таблица: колонка-срез + содержимое списком, пункты через `<br>`, каждый со ссылкой на US/требование где уместно.

| Срез | Содержание |
| --- | --- |
| **As-Is** | • `<текущее поведение>` <br> • `<текущее поведение>` |
| **Gap (проблема)** | • `<разрыв>` (US1) <br> • `<разрыв>` (US2) |
| **To-Be** | • `<целевое поведение>` <br> • `<целевое поведение>` |

Правила: один пункт = один факт/разрыв; не слеплять; As-Is с якорями R1/R2; To-Be не якорить на код (гейт 12).

## План демо для стейкхолдера (ЗМ-008)

Раздел фиксирует, **как команда презентует БФТ на демо**, чтобы стейкхолдер принял результат и посчитал его достаточным. Цель — договорённость команды о едином образе приёмки. **Три блока показываются всегда**, каждый — таблицей (не прозой, гейт Demo-completeness):

- **2.1 Ключевой сценарий** — таблица `Шаг | Действие ведущего | Ожидаемый результат`. Основной путь приёмки.
- **2.2 Альтернативные ветки — ЧТО ЕСЛИ** — таблица `Ветка | Действие | Ожидаемый результат | Что доказывает`. Развилки (другой вход, повтор, отмена, недоступно).
- **2.3 Недопустимые события** — таблица `# | Недопустимое событие | Почему это провал приёмки`. Отработка ограничений: что НЕ должно происходить.
- **2.4 Диаграмма демо** — actor-level sequence + таблица акторов.

**Диаграмма (2.4):**
- PlantUML sequence — только **actor-level / black-box**: акторы и витрина, без вызовов между внутренними сервисами и без имён сообщений/API. Показывает поток приёмки, а не архитектуру.
- `alt/else`-ветки для «ЧТО ЕСЛИ», `note` для недопустимых событий. Надписи на языке документа, технические имена в скобках. Mermaid не используется.
- Таблица акторов: `Актор | Роль на демо`.
- Атрибутивный состав сообщений (параметры и типы запросов/ответов) — **не входит в БФТ**. Это зона СА / системных требований (гейт 14).

## Критерии успеха

Зонтичный раздел. Требования сгруппированы по уровням: границы → бизнес → пользователь → интерфейсы → функциональные → нефункциональные. Каждый уровень фиксирует **результат и признак приёмки**, без внутреннего устройства системы (высота БФТ, гейт 14).

**3.0 Границы: Не входит / Вынесено в СА (scope-exclusions — ЗМ-001).** Структурой, НЕ абзацем. Один пункт = одна зона:

**Не входит:**
- `<пункт>` — `<где реализуется / почему вне>`

**Вынесено в СА / системные требования** _(«высота документа»)_:
- `<что вынесено: контракты, схемы БД, последовательность вызовов, имена полей, эндпоинты>`

**3.1–3.5 БТ/ПТ/ИТ/ФТ/НФТ** — подпункты этого раздела (форматы таблиц — см. «Форматы таблиц по типам требований»).

## Риски и зависимости (объединённый раздел — ЗМ-009)

Зависимость — форма риска: влияет на сроки и ресурсы. Ведутся **единой таблицей** с колонкой типа:

`# | Тип | Описание | Влияние | Митигация / владелец`

- **Тип** ∈ {Зависимость, Риск, Зависимость / риск}.
- Каждая зависимость — со статусом согласования в «Митигация / владелец».
- Каждый риск — с митигацией (пустая митигация запрещена).

## Открытые вопросы

`Вопрос | Ответ | Кто ответил | Источник (встреча / почта / ТЗ)`.
Закрытые → блок «Ключевые решения из открытых вопросов»: `# | Вопрос | Решение | Кто ответил`.

`[УТОЧНИТЬ]`-дисциплина (`writing_style.md` §6): маркер без кросс-рефов (`[УТОЧНИТЬ]`, не `[УТОЧНИТЬ O2]`); каждый вопрос живёт один раз в этой таблице; рамка известна, число не подтверждено → «(предварительно)» / «уточняется у X» / «TBD», а не `[УТОЧНИТЬ]` в каждой ячейке. Запрещён дубль «верхняя таблица + нижняя таблица» одних и тех же вопросов.

## Ревью

Таблица согласования: `Роль | Исполнитель | Статус`.
Роли: Продакт, Архитектор, Аналитик (кросс-ревью), Разработка, Тестирование.

## Якоря истины (Truth Anchors)

Финальный раздел: `Факт | Якорь | Ранг | Тип`. Правила ранжирования якорей — `resources/anchor_rules.md`.

## Словарь (Ubiquitous Language) — пример

Зафиксируйте каноничные термины проекта (понятие → технический термин). Пример структуры:

| Понятие | Термин |
|---------|--------|
| <Бизнес-сущность> | <имя в коде/API> |
| <Роль/актор> | <имя> |

> Если в БФТ появился новый термин — зафиксируйте его в словаре проекта. Единый язык исключает двусмысленность.

## Регуляторика (если применимо)

Если домен регулируемый (финансы / ПДн / гос) — вынести ограничения в отдельный чек-лист: какие законы/стандарты затрагивает БФТ (напр. защита ПДн, PCI DSS для карт, фискализация), и кто согласовывает. Адаптируйте под юрисдикцию вашего проекта.
