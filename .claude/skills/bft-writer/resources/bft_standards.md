# Стандарты БФТ (BFT Standards)

Единообразие — условие проверяемости. Все БФТ следуют одному формату. Адаптируйте имена полей/сервисов под ваш проект.

## Стиль и голос

БФТ пишется плотным деловым русским, как золотые референсы (`{bft_store}/золотой референс …`, `examples/golden_bft_example.md`). Стиль — enforcement-слой, проверяется гейтом 13. Полный набор правил: `resources/writing_style.md`.

Кратко:
- Без эмодзи в теле (статусные значки только в `validation.md`).
- Без `**bold**` внутри ячеек таблиц требований (кроме story-ключей ПТ: `**Когда**/`**Я хочу**/`**Чтобы**`).
- Без callout-врезок `> **⚠️ …`.
- Em-dash — для определений и перечислений, не как связка.
- Без стоп-слов: «позволяет», «обеспечивает», «осуществляет», «производит», «представляет собой», «важно отметить», «в рамках данного».
- `[УТОЧНИТЬ]` без кросс-рефов; неизвестное предпочтительно естественной прозой («уточняется у X», «предварительно», «TBD») как в золотой референс.
- Frontmatter без diff-narrative; история правок — в разделе «История изменений».

## Порядок разделов (фиксирован — ЗМ-005)

Порядок ЧИСТОВОГО БФТ строго по **корпоративному шаблону MTS v10** (начинается с «Бизнес описание»):

1. Бизнес описание (скоуп/границы — абзацем здесь)
2. Общая информация
3. Заинтересованные стороны
4. История изменений
5. Дополнительные материалы и артефакты
6. Проблема которую решаем (ДЛЯ КОГО / ЧТО ДЕЛАЕМ / ASIS / ПРОБЛЕМА / TOBE)
7. Изменение в UJM (таблица Было (As-Is) / Стало (To-Be))
8. План демонстрации (таблица Акторов + PlantUML-сценарий)
9. Бизнес-Требования
   - Вводные для разрабатываемого функционала\* (сюда входят открытые вопросы)
   - Ценность разрабатываемого функционала для бизнес-заказчиков\*
10. Пользовательские требования\*
11. Требования к интерфейсам\*
    - Макеты интерфейса\*
12. Функциональные требования\*
13. Нефункциональные требования\*
14. Зависимости
15. Риски
16. Ревью требований (список ролей)
17. Якоря истины (служебный раздел трассировки, в конце)

**Нет в чистовике:** отдельных разделов «Границы» и «Критерии успеха» (метрики). Объём читается из «Образа результата» (Бизнес описание) + строк ДЛЯ КОГО/ЧТО ДЕЛАЕМ в «Проблеме»; отдельной строки «Скоуп документа —» НЕ вставлять (даёт самопротиворечие, ЗМ-014). Границы/критерии/As-Is-Gap — рабочие аиды стадии решения (`problem.md`, `concept.md`), не разделы чистовика. Раздела «Ключевые решения» нет — открытые вопросы живут в подразделе **«Вводные для разрабатываемого функционала»** внутри Бизнес-Требований. Маркер `*` на заголовке = обязательный раздел корп-шаблона. Эталон порядка — `examples/ideal_bft.md`.

## Бизнес описание (образ результата + зачем + спойлер)

Кратко и точно, без многословия — работает и для человека, и для LLM (ЗМ-013). Структура:
- **Образ результата:** одна-две строки, конкретно и операционно — как выглядит целевое состояние, какие сущности/страницы затронуты, что меняется при рендере. Не абстрактное «документ описывает требования к…».
- **Зачем делать:** ценность одной строкой + метрика (метрику при отсутствии — «уточнить у <кто>»).
- **Спойлер «Подробный контекст»:** `<details><summary>…</summary>…</details>` с развёрнутым описанием домена/механики/сценариев — свёрнуто для человека, доступно для LLM. При публикации → макрос Expand.

**Не вставлять строку «Скоуп документа —»** (ЗМ-014): она регулярно расходится с деталями требований. Объём — из «Образа результата» и «Проблемы».

## Идентификаторы требований

```
{TYPE}-{N}
TYPE  ∈ {БТ, ПТ, ИТ, ФТ, НФТ}
N     = порядковый в пределах типа (1, 2, 3 …)
```

**Идентификаторы локальные** (ЗМ-003): действительны **строго внутри документа** — это якоря для быстрой навигации/трассировки между разделами. Не глобальные ключи. **Не добавляй EPIC/проект/сервис в идентификатор** (`БТ-TL-RestNotify-1` ❌ → `БТ-1` ✅). Связь с эпиком/трекером идёт **только** через колонку «Связанные требования» (ключ задачи `PROJECTKEY-XXX`), не через идентификатор.

Примеры: `БТ-1`, `ПТ-2`, `ИТ-3`, `ФТ-1`, `НФТ-5`.

- **БТ** — Бизнес-Требование (ценность).
- **ПТ** — Пользовательское требование (story).
- **ИТ** — Требование к интерфейсу приёмки: что видит и может сделать актор на экране и в результате. Не API-контракт.
- **ФТ** — Функциональное требование (поведение).
- **НФТ** — Нефункциональное требование (измеримое -ility).

Трассировка каскадом: `БТ ← ПТ ← ИТ` и `БТ ← ФТ ← НФТ`. Связь с эпиком в трекере — через «Связанные требования», не через идентификатор.

> Единообразие: один и тот же шаблон `{TYPE}-{N}` и в колонке «Идентификатор», и в «Связанные требования». Смешение (`БТ-TL-RestNotify-1` в заголовке, `БТ-1` в связях) — ошибка (гейт 3).

## Различение типов (частая ошибка LLM — не путать)

| Тип | О чём | НЕ должен содержать |
|---|---|---|
| **БТ** | Ценность, зачем (почему) | системного поведения, -ilities |
| **ПТ** | Что хочет пользователь (story) | как система устроена внутри |
| **ИТ** | Интерфейс приёмки: экраны, доступность, адаптив, что видит и может актор | API-контракта (эндпоинты, протоколы, схемы сообщений), внутренней реализации |
| **ФТ** | Поведение системы на уровне результата (что делает для пользователя/бизнеса) | ценности (это БТ), -ilities (это НФТ), кода/API-деталей |
| **НФТ** | Измеримое -ility — порог приёмки (как качественно) | функционального поведения |

> Гейт: ПТ, описывающий системное поведение → это ФТ. ФТ со словом «быстро/надёжно» → это НФТ.

> **Высота БФТ (гейт 14).** API-контракты (эндпоинты, REST, схемы запросов/ответов, идемпотентность и auth как контракт), sequence-диаграммы с вызовами между сервисами, имена таблиц/полей БД, код — **вне зоны БФТ**. Это зона СА / системных требований (FNR-пайплайн). БФТ описывает результат и приёмку, не внутреннее устройство системы.

## Frontmatter (синхронизация с wiki)

```yaml
---
source: <URL страницы в вашей wiki>
space: <PROJ>            # пространство wiki, опц.
version: <N>
synced: <YYYY-MM-DD>
jira: <PROJECTKEY-XXX>   # эпик в трекере, опц.
status: Черновик 0.X     # опц.
---
```

H1: `# [БФТ] {EPIC}: {Название}` или `# БФТ {EPIC} – {Название}`.

## Ссылки на Jira / Confluence (ЗМ-004, ЗМ-009)

Любое упоминание Jira/Confluence — **кликабельная ссылка на существующую страницу**, не голый текст и не выдуманный ключ.

**Правило 1 — только существующие страницы (ЗМ-009).** Ссылку получает лишь реально существующая задача/страница. Если объекта ещё нет (эпик не создан, страница не заведена) — это **не ссылка**, а `[УТОЧНИТЬ]` или обычная пометка (`[СОЗДАТЬ эпик]` без URL). Битые/выдуманные ссылки запрещены — PO должен из документа сразу перейти и открыть превью (ЗМ-008).

**Правило 2 — в рабочем `.md`: markdown-ссылка.**

| Источник | Голый текст ❌ | Markdown-ссылка ✅ |
| --- | --- | --- |
| Jira | `JIRA GDSLV-1409` | `[JIRA GDSLV-1409](https://jira.mts.ru/browse/GDSLV-1409)` |
| Confluence (pageId) | `Confluence 2272447498` | `[Confluence 2272447498](https://confluence.mts.ru/pages/viewpage.action?pageId=2272447498)` |

**Правило 3 — при публикации в Confluence: макросы для превью (ЗМ-009).** На отгружаемой странице (`/bft-deliver`) предпочтительны **макросы Confluence** — они дают быстрый доступ и live-превью со статусом, а не просто гиперссылку:
- Jira-задача → **Jira-макрос** (ключ + актуальный статус + hover-превью).
  - storage: `<ac:structured-macro ac:name="jira"><ac:parameter ac:name="key">GDSLV-1610</ac:parameter></ac:structured-macro>`
  - wiki: `{jira:key=GDSLV-1610}`
- Confluence-страница → **smart-link/inline-card** или макрос `include`/`excerpt-include` для превью содержимого.
  - storage-ссылка на страницу по заголовку: `<ac:link><ri:page ri:content-title="<Заголовок>"/></ac:link>`
  - wiki: `[<Заголовок страницы>]` или `{include:<Space>:<Заголовок>}`

**Правило 4 — линкуются и проекты/пространства, и КАЖДОЕ вхождение (ЗМ-012).** Ссылка ставится не только на задачу-ключ, но и на **JIRA-проект** и **Confluence-пространство**, если они упомянуты. Линкуется **каждое** вхождение в каждой колонке и абзаце (не только первое) — цель прямая: читатель кликает и переходит, не тратит время на поиск по внешним ресурсам. Ни одного голого ключа/pageId/имени проекта в готовом документе.

| Объект | Голый текст ❌ | Ссылка ✅ |
| --- | --- | --- |
| JIRA-задача | `GDSLV-1409` | `[GDSLV-1409](https://jira.mts.ru/browse/GDSLV-1409)` |
| JIRA-проект | `проект GDSLV` | `[проект GDSLV](https://jira.mts.ru/projects/GDSLV)` |
| Confluence-страница | `Confluence 2272447498` | `[Confluence 2272447498](https://confluence.mts.ru/pages/viewpage.action?pageId=2272447498)` |
| Confluence-пространство | `space MTSLIVE` | `[space MTSLIVE](https://confluence.mts.ru/display/MTSLIVE)` |

**Базовые URL** (корпоративные, поправь под свой проект):
- Jira задача: `https://jira.mts.ru/browse/{KEY}`
- Jira проект: `https://jira.mts.ru/projects/{PROJECT}`
- Confluence страница: `https://confluence.mts.ru/pages/viewpage.action?pageId={pageId}`
- Confluence пространство: `https://confluence.mts.ru/display/{SPACE}`

Касается **всех** мест: «Связанные требования», «Якоря истины», «Дополнительные материалы», «Общая информация» (проект, эпик), frontmatter `source`, текст разделов. Текст ссылки сохраняет ключ/pageId/имя для читаемости, URL — в скобках. Локальный идентификатор требования (`{TYPE}-{N}`) ссылкой НЕ оборачивается — ссылку получает трекер-ключ в «Связанных требованиях». Бизнес-правила (`BR-*`) и `US#N` — не внешние артефакты, ссылкой не оборачиваются.

## Форматы таблиц по типам требований

Заголовки колонок и разделов — по корпоративному шаблону MTS (см. `examples/ideal_bft.md`).

**БТ** (раздел «Ценность разрабатываемого функционала для бизнес-заказчиков\*»): `Идентификатор | Наименование требования | Ценность разрабатываемого функционала для бизнес-заказчиков | Краткое описание доработки из данного БФТ | Связанные требования`

**ПТ** (раздел «Пользовательские требования\*»): `Идентификатор | Наименование требования | Story | Комментарий | Связанные требования`
Story-формат: `**Когда** я (актор) {действие} **Я хочу** {результат} **Чтобы** {обоснование}`

**ИТ** (раздел «Требования к интерфейсам\*»): `Идентификатор | Наименование требования | Требование к пользовательскому интерфейсу | Предложения по элементам UI, механике и композиции | Связанные требования`
Требование — интерфейс приёмки: что видит и может актор, экраны, доступность, адаптив. Контекст продукта (платформа / браузеры / разрешение / адаптив) — строкой перед таблицей. Макеты — в подразделе «Макеты интерфейса\*». Не API-контракт.

**ФТ:** `Идентификатор | Наименование | Приоритет | Функциональные требования | Параметры, ограничения | Связанные требования`
Приоритет: **Высокий / Средний / Низкий** (пусто запрещено).

**НФТ:** `Идентификатор | Наименование | Описание | Связанные требования`

## Стандартный НФТ-набор (включать по умолчанию для бэкенд-API)

| Группа | О чём |
|--------|-------|
| **Идемпотентность** | Повторный вызов не выполняет бизнес-логику повторно |
| **SLA / латентность** | P95 / P99 по операциям (напр. запись 2000 ms P95, чтение 200 ms P95) |
| **Нагрузка** | RPS на чтение / на запись (с пометкой «предварительно», если не подтверждено) |
| **RED-метрики** | Попытки / успех / неуспех, сквозная латентность, ошибки интеграций |
| **Алертинг** | Пороги: рост ошибок, рост таймаутов, аномальный рост конфликтов |
| **Логирование** | Состав событий, **без чувствительных данных**, срок хранения, сквозной trace-id |
| **Доступ / аудит** | Сервисные учётки, TLS, аудит ключевых операций |
| **Надёжность / целостность** | Атомарность операций, идемпотентность на уровне БД, алерты на критические ошибки |

## Корпоративный реестр НФТ (приоритет)

Если в проекте есть **реестр НФТ** (каталог стандартных нефункциональных требований) — **приоритет брать НФТ оттуда**, а не формулировать с нуля. Ссылайтесь на ID из реестра. Не выдумывайте НФТ, если есть готовый корпоративный стандарт (безопасность, хранение, SLA и т.д.).

## План демонстрации (акторы + сценарий)

Раздел перед «Бизнес-Требованиями» (в корп-шаблоне называется «План демонстрации», не «Сценарии взаимодействия»). Содержит таблицу акторов + PlantUML-сценарий приёмки.

- Таблица акторов (актор → описание).
- Сценарий на уровне **актор → действие → результат**, с `alt/else`-ветками (ошибка, недоступно, отмена, таймаут).
- PlantUML sequence — только **actor-level / black-box**: акторы и витрина, без вызовов между внутренними сервисами и без имён сообщений/API. Диаграмма показывает поток приёмки, а не архитектуру.
- **Рендер (ЗМ-015):** в рабочем `.md` — блок ` ```plantuml `; при публикации оборачивается в **макрос «PlantUML»** (одноимённый плагин Confluence/JIRA), чтобы диаграмма рендерилась картинкой, а не кодом. Плагина нет → `[УТОЧНИТЬ: плагин PlantUML]`, оставить кодом.
- Атрибутивный состав сообщений (параметры и типы запросов/ответов) — **не входит в БФТ**. Это зона СА / системных требований.

## Границы и критерии успеха — только в черновиках (не в чистовике)

Разделов «Границы» (Входит/Не входит) и «Критерии успеха» (метрики) в **чистовике нет** (соответствие корп-шаблону v10). Это рабочие аиды стадии принятия решения: границы и As-Is/Gap живут в `problem.md`, критерии/образ результата — в `concept.md`. В чистовик из них попадают: объём → в «Образ результата» (Бизнес описание), без отдельной строки «Скоуп документа —» (ЗМ-014); ДЛЯ КОГО/ЧТО ДЕЛАЕМ → строками в «Проблему»; было/стало → в «Изменение в UJM».

**Высота документа (гейт 14)** сохраняется: БФТ фиксирует результат и приёмку. Системные требования, API-контракты, схемы сообщений, sequence-вызовы между сервисами, БД-схемы и код — вне документа (зона СА / системных требований).

## Проблема которую решаем

Первый смысловой раздел после «Доп. материалов» (бывш. «As-Is → Gap → To-Be»). **Формат — таблица, НЕ сплошной текст** (гейт 13 / `ЗМ-002`). Пять строк: `ДЛЯ КОГО` / `ЧТО ДЕЛАЕМ` / `ASIS` (с якорями R1/R2) / `ПРОБЛЕМА` (бизнес-разрыв) / `TOBE` (не якорить на код).

**Форматирование ячеек (ЗМ-011):** одна мысль → простой строкой (ДЛЯ КОГО / ЧТО ДЕЛАЕМ / ASIS обычно однострочны); несколько пунктов → **настоящий список `<ul><li>…</li></ul>`**, а НЕ `• … <br> • …`. Глиф `•` с `<br>` в Confluence даёт плоский текст без висячего отступа — визуально не список. `<ul><li>` рендерится корректным маркированным списком и в `.md`, и после публикации.

| Срез | Содержание |
| --- | --- |
| **ДЛЯ КОГО** | `<актор-бенефициар>` |
| **ЧТО ДЕЛАЕМ** | `<одно предложение сути доработки>` |
| **ASIS** | `<текущее поведение одной строкой>` ([JIRA KEY](https://jira.mts.ru/browse/KEY)) |
| **ПРОБЛЕМА** | `<ul><li>`в чём боль для бизнеса/пользователя`</li><li>`разрыв`</li></ul>` |
| **TOBE** | `<ul><li>`целевое поведение`</li><li>`целевое поведение`</li></ul>` |

Правила: один пункт = один факт/разрыв; не слеплять; ASIS с якорями R1/R2; ПРОБЛЕМА — бизнес-разрыв (не технический Gap-жаргон); TOBE не якорить на код (гейт 12). То же правило списков-в-ячейке — для любой таблицы БФТ с несколькими пунктами в клетке.

## Изменение в UJM

Отдельный раздел сразу после «Проблемы». Таблица «было/стало» на уровне приёмки: `Точка демо | Было (As-Is) | Стало (To-Be)`. Числовых метрик успеха (метрика→базовое→цель→срок) в чистовике нет.

## Зависимости и риски

**Зависимости:** `Команда / система | Тип зависимости | Статус согласования`.
**Риски:** `Риск | Вероятность | Влияние | Митигация`.

## Вводные для разрабатываемого функционала (открытые вопросы)

Подраздел Бизнес-Требований. Единая таблица корп-шаблона: `Информация или вопрос | Конспект | Ответ | В рамках чего был получен ответ | Кто ответил | Дата`. Незакрытое → строка с пустым «Ответ» и `[УТОЧНИТЬ]`; полученный ответ вписывается в те же колонки (отдельного блока «Ключевые решения» нет — исключён).

`[УТОЧНИТЬ]`-дисциплина (`writing_style.md` §6): маркер без кросс-рефов (`[УТОЧНИТЬ]`, не `[УТОЧНИТЬ O2]`); каждый вопрос живёт один раз в этой таблице; рамка известна, число не подтверждено → «(предварительно)» / «уточняется у X» / «TBD», а не `[УТОЧНИТЬ]` в каждой ячейке.

## Ревью требований

Предпоследний раздел (после «Рисков», перед «Якорями истины»). Простой **список ролей** (не таблица со статусом): Продакт, Архитектор, Аналитик (кросс-ревью), Разработка, Тестирование. При наличии — с ФИО: `* Продакт — <ФИО>`. Раздела «Ключевые решения» нет.

## Якоря истины (Truth Anchors)

Финальный раздел: `Факт | Якорь | Ранг | Тип`. Правила ранжирования якорей — `resources/anchor_rules.md`.

## Словарь (Ubiquitous Language) — пример

Зафиксируйте каноничные термины проекта (понятие → технический термин). Пример структуры:

| Понятие | Термин |
|---------|--------|
| <Бизнес-сущность> | <имя в коде/API> |
| <Роль/актор> | <имя> |

> Если в БФТ появился новый термин — зафиксируйте его в словаре проекта. Единый язык исключает двусмысленность.

## Регуляторика (если применимо)

Если домен регулируемый (финансы / ПДн / гос) — вынести ограничения в отдельный чек-лист: какие законы/стандарты затрагивает БФТ (напр. защита ПДн, PCI DSS для карт, фискализация), и кто согласовывает. Адаптируйте под юрисдикцию вашего проекта.
