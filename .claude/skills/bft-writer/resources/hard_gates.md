## Чек-лист самовалидации

### 1. Структура и идентификаторы
- [ ] Frontmatter заполнен (source / space / version / synced / jira)
- [ ] H1 содержит EPIC-код и название
- [ ] «Оглавление» в начале документа (ЗМ-006, гейт 16)
- [ ] Все обязательные разделы присутствуют
- [ ] Разделы в каноническом lean-порядке (ЗМ-005): Оглавление → Продуктовое обоснование → О задаче (2.1 As-Is→Gap→To-Be / 2.2 Контекст / 2.3 Стейкхолдеры) → План демо → Критерии успеха (4.0 scope-exclusions / 4.1–4.5 БТ-НФТ) → Риски и зависимости → Открытые вопросы (см. `bft_standards.md`)
- [ ] Раздел «Продуктовое обоснование» есть (ЗМ-010): единственная метрика верификации (база → цель), A/B-дизайн, МДО со ссылками на требования, порог засчёта KR — гейт 18
- [ ] Раздел заполнен из `artefacts/value.md` (стадия `/bft-value` пройдена), а не оставлен пустым. Пустой раздел → 🟡 «дурной тон» (не блокер, но продуктовая ценность не обоснована)
- [ ] Идентификаторы локальные и единообразны `{TYPE}-{N}` (без EPIC; один шаблон в «Идентификатор» и «Связанные требования») — ЗМ-003
- [ ] В «Критерии успеха» есть блок 4.0 «Не входит / Вынесено в СА» (scope-exclusions) И соблюдена высота БФТ (приёмка, не системный/API/код) — гейт 4

### 2. Трассировка
- [ ] Нет пустых «Связанные требования»
- [ ] Каждая строка ссылается вверх (к БТ) + на эпик в трекере
- [ ] Каскад `БТ ← ПТ ← ИТ` и `БТ ← ФТ ← НФТ` соблюдён

### 3. Качество требований
- [ ] БТ и каждый ФТ имеют приоритет
- [ ] НФТ только измеримые (число / % / RPS / срок)
- [ ] ПТ в формате Когда / Я хочу / Чтобы
- [ ] Каждый ФТ ≈ атомарная задача (1–2 дня), со ссылкой на трекер
- [ ] Типы не смешены (ПТ≠ФТ≠НФТ) — гейт 11

### 4. Якорение (Truth Anchors — `anchor_rules.md`)
- [ ] Каждое ключевое утверждение имеет якорь R1 (код, As-Is) / R2 (трекер/wiki/BR/ADR) / R3 (PO-решение) — inline, в тексте / «Связанные требования» (гейт 1)
- [ ] To-Be не заякорен на код (R1) — гейт 12
- [ ] Решения PO — с «кто + когда + источник»
- [ ] _(опц.)_ Раздел «Якоря истины» в подвале, если ведётся (Факт | Якорь | Ранг | Тип) — в lean-форме необязателен, но якорение inline обязательно
- [ ] Все ссылки Jira/Confluence — markdown-ссылкой, не голым ключом (ЗМ-004, гейт 10)

### 5. План демо для стейкхолдера и визуализация
- [ ] Все три блока присутствуют и оформлены таблицами: 3.1 Ключевой сценарий / 3.2 ЧТО ЕСЛИ / 3.3 Недопустимые события (ЗМ-008, гейт 17)
- [ ] Альтернативные ветки покрывают развилки (другой вход / повтор / отмена / недоступно)
- [ ] PlantUML — actor-level / black-box, без вызовов между внутренними сервисами и без имён сообщений
- [ ] Есть таблица акторов
- [ ] Нет атрибутивного состава сообщений (это зона СА) — гейт 14
- [ ] Mermaid не используется
- [ ] Надписи на языке документа, технические имена в скобках

### 6. Полнота и неизвестное
- [ ] Раздел «Открытые вопросы» присутствует
- [ ] Закрытые вопросы → «Ключевые решения» (с «кто ответил»)
- [ ] Все пробелы знаний помечены `[УТОЧНИТЬ]` и собраны в начале документа
- [ ] Ни одного выдуманного ключа эпика / статуса / оценки — только из трекера / PO

### 7. Контекст и риски
- [ ] Зависимости между командами вашего проекта со статусом согласования
- [ ] Риски имеют митигацию
- [ ] Стейкхолдеры указаны с контактами
- [ ] Измеримые критерии успеха (метрика → цель → срок)
- [ ] Раздел «Ограничения релиза» есть (если есть `constraints.md`); каждое ограничение с классом Ф/Г, измеримо, атрибутировано; гипотезы (Г) не внесены жёстким порогом в НФТ — гейт 19

### 8. Стиль и голос (`writing_style.md`)
- [ ] В теле документа нет эмодзи (статусные значки только в `validation.md`)
- [ ] В ячейках таблиц БТ/ИТ/ФТ/НФТ нет `**bold**` (кроме story-ключей в ПТ)
- [ ] Нет callout-блоков `> **⚠️ …`
- [ ] Em-dash — для определений и перечислений, не как связка; не больше одного на предложение
- [ ] Нет стоп-слов («позволяет», «обеспечивает», «осуществляет», «представляет собой», «важно отметить»)
- [ ] `[УТОЧНИТЬ]` без кросс-рефов `[УТОЧНИТЬ Ox]`; каждый вопрос один раз в «Открытые вопросы»; нет дубля таблиц
- [ ] Frontmatter без diff-narrative; история правок — в «История изменений»

### 9. Замечания ревью (Known-Mistake — `review_feedback.md`)
- [ ] Все `активно`-правила реестра соблюдены (гейт 15)
- [ ] Ни одно ранее зафиксированное замечание PO не повторено
