# Реестр замечаний ревью (Review Feedback / Lessons Learned)

**Назначение.** Превращать замечания PO/ревьюеров к сгенерированным БФТ в
**durable-правила**. Каждое замечание → правило, которое `/bft-draft` применяет
**при генерации**, а `/bft-validate` **проверяет**. Цель — не повторять одну и ту
же ошибку дважды. Реестр — машина памяти навыка между эпиками.

> Это НЕ «Замечания» внутри конкретного БФТ (те живут в «Открытых вопросах» /
> «Вводные для разрабатываемого функционала» документа). Здесь — **сквозные уроки**, применимые ко всем
> будущим БФТ.

---

## Как пополнять (intake)

PO присылает замечание к сгенерированному БФТ → добавь **одну строку** в реестр.
Сформулируй из замечания **анти-паттерн** (что нельзя) и **правило** (как надо).

Поля строки:

| Поле | Что писать |
|------|-----------|
| **ID** | `ЗМ-NNN` — сквозной, не переиспользовать |
| **Замечание (анти-паттерн)** | Что было не так в сгенерированном БФТ. Конкретно, в прошедшем времени: «ФТ заякорен на код To-Be» |
| **Правило (как правильно)** | Императив, проверяемый: «To-Be → только R2/R3» |
| **Область** | Этап (`context`/`problem`/`concept`/`debate`/`draft`/`validate`/`общее`) + тип требования (`БТ`/`ПТ`/`ИТ`/`ФТ`/`НФТ`/`общее`) |
| **Связь** | Гейт из `hard_gates.md` (№) / принцип `SKILL.md` (№) / `новый` |
| **Статус** | `активно` / `устарело` (устаревшее не удалять — помечать, для аудит-следа) |

**Правила пополнения:**
1. Одно замечание = одна атомарная строка. Несколько проблем в одном замечании → несколько строк.
2. Если замечание совпадает с уже существующим правилом — **не дублируй**, усиль формулировку существующего.
3. Если замечание требует нового бинарного запрета — заведи **гейт 13+** в `hard_gates.md` и поставь его № в «Связь».
4. Дата/автор — в комментарии под таблицей (история), чтобы таблица оставалась читаемой.

---

## Как применять (enforcement)

- **`/bft-draft`** (Этап 1): загрузи этот реестр. Все строки `активно` — **обязательные ограничения генерации**. Нарушение активного правила недопустимо (см. запреты команды).
- **`/bft-validate`**: отдельная итерация рефлексии — прогон по активным правилам реестра. Нарушение = 🔴 (гейт «Known-Mistake»).
- При конфликте правила реестра с базовым стандартом — **реестр приоритетнее** (это явная воля PO), но зафиксируй конфликт в «Открытых вопросах» БФТ.
