# Ограничения релиза — модель и правила (Release Constraints)

> Ресурс стадии `/bft-constraints`. Определяет, что считать ограничением релиза, как отделять факт от гипотезы, как измерять, атрибутировать и заякоривать. Совместное исследование LLM + PO: техника (repowise) + People Graph (Нексус `team`) + подтверждение человеком.

## Что такое ограничение релиза

**Ограничение релиза** — точное, конкретное, **измеримое** условие окружения, которое релиз обязан соблюсти, иначе он не состоится успешно или нанесёт вред смежной команде/системе/бизнесу.

Это **не** требование к продукту (что мы делаем), а **рамка среды**: чего нельзя нарушить, с чьим ожиданием обязаны совпасть, в какое окно можно выкатываться. Проблема, которую стадия закрывает: побочные требования внешних стейкхолдеров всплывают в момент выката, потому что они не знали о задаче. Ограничения собираются **до черновика**, пока изменения ещё можно заложить в требования.

Отличие от смежных сущностей:

| Сущность | Отвечает на вопрос | Раздел БФТ |
|---|---|---|
| БТ/ФТ | Что система делает | Требования |
| НФТ | Как качественно (порог) | НФТ |
| **Ограничение релиза** | Что среда/смежник требует, чтобы релиз состоялся | Ограничения релиза (+ проекция в Границы/НФТ/Зависимости/Риски) |
| Риск | Что может пойти не так и с какой вероятностью | Риски |

Нарушенное ограничение — это не «риск с вероятностью», это условие, которое либо соблюдено, либо нет.

## Два класса (разделение обязательно)

Каждое ограничение относят к одному из двух классов. Смешивать запрещено — это ядро стадии.

| Класс | Признак | Негативный эффект | Режим |
|---|---|---|---|
| **Факт (Ф)** — фактологически-ограничивающее | Подтверждено якорем: документированная зависимость, регламент, договорённость, инвариант в коде | **Реальный, доказанный** негативный бизнес-эффект при нарушении (сорванный SLA смежника, инцидент, штраф, простой, откат) | Жёсткое. Blocking. Проецируется в НФТ/Границы/Зависимости, согласуется с ответственным |
| **Гипотеза (Г)** — гипотетически-ограничивающее | Гипотеза blast-radius: «если не учесть X, где-то может пойти так». Основана на технической связи, не на подтверждённой договорённости | **Предполагаемый**, не доказанный | Проверить с ответственным владельцем → подтвердить (переходит в Ф с якорем) или снять с обоснованием. До проверки живёт в «Открытых вопросах» |

**Правило честности:** гипотеза не выдаётся за факт. Класс Г держится отдельно, пока владелец компонента/эффекта не подтвердит его (тогда Г → Ф, якорь меняется на подтверждение) либо не снимет (фиксируем «снято, кто, почему»). Перевод Г → Ф без нового якоря-подтверждения запрещён.

## Измеримость

Ограничение проверяемо: число / порог / дата / булев инвариант. Не «нельзя ломать смежников», а конкретно, что именно и как проверить.

| Плохо (не измеримо) | Хорошо (измеримо, проверяемо) |
|---|---|
| «Не сломать интеграцию с биллингом» | «Событие `payment.created` в топике X сохраняет поля {id, amount, currency}; контракт vN принимается биллингом до 31.12» |
| «Не перегрузить смежный сервис» | «Пик исходящих вызовов к сервису Z в окне релиза ≤ 100 RPS» |
| «Выкатывать в удобное время» | «Окно релиза вне закрытия месяца в 1С (1–5 число включительно)» |
| «Согласовать с безопасностью» | «ПДн в логах отсутствуют; согласование ИБ по чек-листу PCI DSS §3 получено до выката» |

Если рамка известна, а число ещё нет — «(предварительно)» / «уточняется у X», не выдуманное значение.

## Атрибуция (кто отвечает)

У каждого ограничения — **ответственный**: владелец затронутого компонента или стейкхолдер бизнес-эффекта. Без ответственного ограничение неполно → `[УТОЧНИТЬ у {кого}]`.

Поиск ответственного — через **People Graph** (Нексус `GROUND/NEXUS/team`): сопоставь затронутый компонент/эффект с `influence_zones` / `contact_for` / `expertise_topics` person-узлов. Возвращай `full_name` + `role_title` + канал связи. Если в People Graph совпадения нет → `[УТОЧНИТЬ у {кого}]`, не выдумывай владельца.

## Якорь

- **Ф** ← якорь R1/R2/R3 (`anchor_rules.md`): инвариант в коде — R1 (только для As-Is-факта, напр. текущий контракт); регламент/договорённость/ADR/тикет — R2; решение владельца — R3.
- **Г** ← якорь-гипотеза: blast-radius из repowise (`symbol_id` / модуль) + явная пометка «гипотеза». Гипотеза без технической опоры (просто предположение) — слабая, помечается как требующая проверки в первую очередь.

## Источники исследования (совместное LLM + PO)

Стадия — **не автогенерация**, а совместное исследование. Три источника:

1. **repowise (техническая аналитика, blast-radius).** По затронутым компонентам выбранного концепта:
   - `get_context` — зависимости файла/модуля (кто с ним связан).
   - `search_codebase` / `get_answer` — кто вызывает изменяемый компонент, где ещё используется контракт.
   - `get_risk` — hotspot и риск изменения компонента.
   - `get_why` — почему компонент устроен так (ADR/история), какие инварианты нельзя ломать.
   - Результат: список затронутых компонентов и зависящих от них команд/систем → кандидаты в ограничения.
2. **People Graph (Нексус `team`, PO-helper).** Для каждого затронутого компонента и каждого бизнес-эффекта — ответственный PO/команда по зонам влияния. Роутинг к тому, с кем ограничение надо согласовать.
3. **PO (человек).** Подтверждает негативный бизнес-эффект, переводит гипотезы в факты или снимает их, добавляет ограничения, которых нет в коде: регламенты, календарь релизов, договорённости с внешними стейкхолдерами, юридические/комплаенс-рамки.

repowise / People Graph недоступны (VPN, пустой Нексус) → честно сказать, собрать доступное, остальное `[УТОЧНИТЬ]`. Не выдумывать компоненты, владельцев, эффекты.

## Матрица ограничений (формат артефакта)

`constraints.md` содержит таблицу. Один ряд = одно ограничение.

| Поле | Содержание |
|---|---|
| **ОГР-{N}** | Локальный идентификатор (в пределах документа) |
| **Класс** | `Ф` (факт) / `Г` (гипотеза) |
| **Формулировка** | Измеримое условие (число / порог / дата / инвариант) |
| **Затронутый компонент** | Из repowise (`symbol_id` / модуль / сервис) |
| **Ответственный / стейкхолдер** | Из People Graph (`full_name`, роль) или `[УТОЧНИТЬ у {кого}]` |
| **Негативный эффект при нарушении** | Ф — доказанный (с якорем); Г — предполагаемый (пометка «гипотеза») |
| **Как проверить соблюдение** | Конкретная проверка перед/на выкате |
| **Статус согласования** | Согласовано / Требует согласования / Гипотеза (не проверена) |
| **Якорь** | R1/R2/R3 для Ф; blast-radius `symbol_id` + «гипотеза» для Г |

## Проекция в БФТ (как ограничения входят в черновик)

`/bft-draft` разносит ограничения из `constraints.md` так, чтобы ни одно не потерялось:

- **Раздел «Ограничения релиза»** (специальный) — полная атрибутированная таблица обоих классов. Это первичное место, где ограничение видно целиком.
- **Ф, измеримое про среду/качество** → дополнительно в **НФТ** (CATWOE `E`): порог как требование приёмки.
- **Ф про «за это не отвечаем / этот инвариант держим»** → **Границы** (Не входит / защищённый инвариант).
- **Ф про смежную команду/систему** → **Зависимости** (со статусом согласования и ответственным).
- **Г (непроверенная)** → **Открытые вопросы** («проверить с {владелец}») + при необходимости строка в **Рисках**.

Правило: класс сохраняется при проекции. Гипотеза, попавшая в НФТ как жёсткий порог, — ошибка; сначала перевод Г → Ф через подтверждение владельцем.
