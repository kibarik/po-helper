# CATWOE — декомпозиция трансформации (Soft Systems Methodology)

> Для каждого крупного БТ (значимой трансформации) прогнать CATWOE. Метод структурирует шапку требования: `W` → ценность (формулировка БТ), `O` → согласующий (ревью), `E` → заготовки НФТ.

## Расшифровка

| Буква | Значение | Что даёт в БФТ |
|---|---|---|
| **C** Customer | Кто выигрывает / терпит от трансформации | → акторы ПТ, стейкхолдеры |
| **A** Actor | Кто реализует / участвует в работе | → исполнители, команды |
| **T** Transformation | Что во что превращается (input → output) | **суть БТ** |
| **W** Worldview | Зачем это нужно (ценность, взгляд на мир) | → **«Ценность» в БТ** |
| **O** Owner | Кто может остановить / согласовать | → **согласующий в Ревью** |
| **E** Environment | Ограничения среды (внешние) | → **заготовки НФТ** (регуляторика, контракты, legacy, SLA) |

## Шаблон (per крупный БТ)

```
БТ-N: <Название>
  C: <кто получает выгоду/ущерб>
  A: <кто делает>
  T: <вход> → <выход>
  W: <зачем / ценность>
  O: <согласующий / владелец>
  E: <ограничения: закон, контракт, legacy, нагрузка>
```

## Пример

```
БТ-2: Персонализация FAQ
  C: Покупатель (релевантные ответы), SEO-команда (трафик), поддержка (меньше обращений)
  A: Контент-менеджер (Direct), разработка витрины
  T: пустая/базовая страница FAQ → персонализированный FAQ под сущность
  W: релевантные ответы снижают обращения в поддержку и растят конверсию
  O: PO витрины (Кульков), SEO (Любимов)
  E: SSR обязателен (SEO); store_db источник истины; таймаут SSR (E → НФТ-3)
```

## Где встроено

- **`/bft-concept`** — каждый концепт трансформации прогоняется через CATWOE (особенно `T` и `W`), чтобы сравнивать концепты по сути трансформации.
- **`/bft-draft`** — `W` становится формулировкой ценности в БТ; `O` попадает в раздел Ревью; `E` даёт заготовки НФТ.

## Когда применять
- На каждый **крупный** БТ (значимая трансформация).
- Мелкие/производные БТ — без полного CATWOE, но с явными `W` (ценность) и `O` (owner).
