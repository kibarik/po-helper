# Golden эталон БФТ: BOOK-2 «Booking сервис (Бронирование)»

> Few-shot эталон для генератора. Показывает **гибридный паттерн** (каркас + строгие таблицы + диаграммы + акторы) — целевой стандарт качества. Полный оригинал: `{bft_store}/BOOK-2 …`.

## Почему BOOK-2 — эталон

BOOK-2 сочетает **нарративный каркас** (проблема/границы/решения) со **строгими таблицами требований** (БТ/ПТ/ИТ/ФТ/НФТ), PlantUML-диаграммами и таблицей акторов. Это и есть гибрид, к которому надо стремиться.

---

## Эталонные элементы (учи эти паттерны)

### ✓ 1. Границы системы — явные, в самом начале

> «**Границы системы**: в рамках данной задачи API-слой выступает абстракцией (прокси-слоем) между витринами продаж и поставщиками/мастер-системами инвентаря. API-слой **не** принимает платежи и **не** отвечает за эквайринг/платёжные сценарии витрин.»

**Чему учит:** формулируй, что система ДЕЛАЕТ и чего НЕ делает. Закрывает споры о зоне ответственности.

### ✓ 2. «Ключевые решения из открытых вопросов» — закрытые вопросы как артефакт

| # | Вопрос | Решение | Кто ответил |
| --- | --- | --- | --- |
| 1 | Нужно ли разрешить покупку, если Мастер система недоступна? | Если система поставщика недоступна — не выполняем финальное бронирование и продажу | PO |
| 4 | TTL бронирования при разнородной корзине? | TTL = минимальный среди поставщиков | PO |

**Чему учит:** каждый вопрос → решение → **кто ответил** (это и есть якорь). Нет анонимных решений.

### ✓ 3. Трассировка — сплошная, каскадом

```
ПТ-BOOK-2-1 (Мгновенное бронирование) → БТ-BOOK-2-1, ФТ-BOOK-2-1, ФТ-BOOK-2-2
ФТ-BOOK-2-1 (Резервирование)          → БТ-BOOK-2-1, ПТ-BOOK-2-1
```

**Чему учит:** каждое требование связано вверх (к БТ) + к зависимому ФТ. Нет «висячих» строк.

### ✓ 4. Story-формат ПТ — Когерентный

> «**Когда** я (витрина/партнер) вызываю метод бронирования **Я хочу** чтобы другие витрины не смогли забронировать этот же экземпляр **Чтобы** обеспечить гарантированную продажу»

**Чему учит:** актор → желание → обоснование ценности. Обоснование отвечает на «зачем».

### ✓ 5. Negative flows в PlantUML (не только happy path)

В диаграмме BOOK-2 есть ветки `alt`: «Бронирование не найдено» → откат; «шлюз недоступен» → отмена + снятие брони + ошибка пользователю.

**Чему учит:** каждый сценарий содержит обработку ошибок/недоступности. Pure happy path = брак.

### ✓ 6. Измеримые НФТ — конкретные числа

| НФТ | Описание |
| --- | --- |
| Скорость ответа API | Создание бронирования: **2000 ms P95**. Получение статуса: **200 ms P95** |
| Нагрузка | **100 RPS** на чтение, **10 RPS** на создание (предварительно) |
| Логирование | … Хранение: **7 дней**. Сквозной идентификатор сессии |

**Чему учит:** НФТ = число/процент/RPS/срок. Не «быстро/надёжно».

### ✓ 7. Таблица акторов — кто есть кто в сценарии

| Актор | Описание |
| --- | --- |
| BOOK.BOOKING.WORKER | Воркер для авто-отмены просроченных броней |
| BOOK.INVENTORY | Компоненты контекста инвентаря в API-слой |

**Чему учит:** перечисли всех участников взаимодействия до диаграммы — генератор и читатель видят полный состав.

### ✓ 8. Приоритеты проставлены + «Параметры, ограничения»

ФТ-BOOK-2-1 «Резервирование» — Приоритет: **Высокий**, параметр: «TTL = TTL поставщика. Если хотя бы один элемент недоступен — **ни один** не бронируется».

**Чему учит:** приоритет не пустой; ограничения зафиксированы рядом с требованием.

---

## Чему равняться при генерации

1. Начинай с **Границ системы** (явное ВНУТРИ/СНАРУЖИ).
2. Все вопросы → **Ключевые решения** (с «кто ответил»).
3. **Трассировка** сплошная, каскадом.
4. **Story** в формате Когда/Я хочу/Чтобы.
5. **Negative flows** в каждой PlantUML.
6. **НФТ** измеримые.
7. **Таблица акторов** перед диаграммой.
8. **Приоритеты** + параметры на каждом ФТ.

Если твой БФТ по этим 8 пунктам совпадает с BOOK-2 — качество на уровне sa-helper.
