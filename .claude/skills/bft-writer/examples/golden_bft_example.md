# Golden эталон БФТ: «Каталог — Полнотекстовый поиск» (вымышленный пример)

> Few-shot эталон для генератора. Показывает **гибридный паттерн** (каркас + строгие таблицы + диаграммы + акторы) — целевой стандарт качества. Полностью вымышленный сценарий; замените на реальный golden своего проекта.

## Почему это эталон

Документ сочетает **нарративный каркас** (проблема/границы/решения) со **строгими таблицами требований** (БТ/ПТ/ИТ/ФТ/НФТ), PlantUML-диаграммой и таблицей акторов. Это гибрид, к которому надо стремиться.

---

## Эталонные элементы (учи эти паттерны)

### ✓ 1. Границы системы — явные, в начале

> «**Границы:** сервис поиска отвечает за индексацию каталога и полнотекстовый запрос/ранжирование. Сервис НЕ хранит мастер-данные каталога (источник истины — основная БД) и НЕ отвечает за фильтрацию по доступности/цене (это слой выдачи).»

**Чему учит:** формулируй, что сервис ДЕЛАЕТ и чего НЕ делает.

### ✓ 2. «Ключевые решения из открытых вопросов» — закрытые вопросы как артефакт

| # | Вопрос | Решение | Кто ответил |
| --- | --- | --- | --- |
| 1 | Синхронный или асинхронный поиск? | Синхронный (P95 < 200 ms достижим на индексе); асинхронность — позже при >1000 RPS | PO |
| 2 | Реиндексация при изменении каталога — как быстро? | Задержка ≤ N минут через очередь событий | Архитектор |

**Чему учит:** каждый вопрос → решение → **кто ответил** (якорь).

### ✓ 3. Трассировка — сплошная, каскадом

```
ПТ-SEARCH-1 (поиск) → БТ-SEARCH-1, ФТ-SEARCH-1, ФТ-SEARCH-2
ФТ-SEARCH-1 (запрос) → БТ-SEARCH-1, ПТ-SEARCH-1
```

**Чему учит:** каждое требование связано вверх (к БТ) + к зависимому ФТ. Нет «висячих» строк.

### ✓ 4. Story-формат ПТ — когерентный

> «**Когда** я (покупатель) ввожу запрос **Я хочу** видеть релевантные результаты за <200 ms **Чтобы** быстро найти товар»

**Чему учит:** актор → желание → обоснование ценности.

### ✓ 5. Negative flows в PlantUML

В диаграмме есть ветки `alt`: «индекс недоступен» → fallback на прямую БД (медленнее) или ошибку; «пустой запрос» → валидация.

**Чему учит:** каждый сценарий содержит обработку ошибок/недоступности.

### ✓ 6. Измеримые НФТ — конкретные числа

| НФТ | Описание |
| --- | --- |
| Скорость ответа | **P95 ≤ 200 ms** |
| Нагрузка | **1000 RPS** на чтение (предварительно) |
| Актуальность индекса | задержка реиндексации **≤ N минут** |

**Чему учит:** НФТ = число/процент/RPS/срок. Не «быстро/надёжно».

### ✓ 7. Таблица акторов — кто есть кто

| Актор | Описание |
| --- | --- |
| Покупатель | Конечный пользователь витрины |
| Сервис каталога | Источник истины, публикует события об изменениях |
| Индекс (поисковый движок) | Хранит поисковый индекс |

### ✓ 8. Приоритеты + параметры на каждом ФТ

ФТ-SEARCH-1 «Запрос» — Приоритет **Высокий**, параметр: «если индекс недоступен → fallback на БД с таймаутом».

---

## Чему равняться при генерации

1. Начинай с **Границ системы** (явное ВНУТРИ/СНАРУЖИ).
2. Все вопросы → **Ключевые решения** (с «кто ответил»).
3. **Трассировка** сплошная, каскадом.
4. **Story** в формате Когда/Я хочу/Чтобы.
5. **Negative flows** в каждой PlantUML.
6. **НФТ** измеримые.
7. **Таблица акторов** перед диаграммой.
8. **Приоритеты** + параметры на каждом ФТ.

Если твой БФТ по этим 8 пунктам совпадает с эталоном — качество на уровне sa-helper.
