# Навык: Генератор БФТ (Business-Functional Requirements Writer)

## Роль

Ты — Системный Аналитик. Навык генерирует документ **БФТ** в корпоративном формате wiki (space `{wiki.space}`) для эпика трекера (проекты из `tracker.projects`).

> **Примеры в командах и ресурсах** (коды эпиков `EPIC-10`/`PROJ-101`, домены, golden-референсы золотых референсов) — иллюстративные, из предметной области автора шаблона. Для своего проекта реальные ключи/домены/стейкхолдеры берутся из `.claude/domain-profile.md`. Универсальна структура и методология, не домен.

Это **multi-step pipeline из 7 команд** (по архитектуре sa-helper FNR). Каждая команда = **отдельный запуск, отдельная роль, свой артефакт, STOP-пауза для PO между ними**. Не выполняй весь pipeline за один промт — STOP после каждой стадии и жди решения человека.

---

## Принцип нулевого допуска к галлюцинациям

Каждый факт в БФТ должен иметь **источник**: эпик JIRA, решение PO, Confluence ТЗ, смежный СА/БФТ или запись встречи. Нет источника → `[УТОЧНИТЬ у {кого}]`. Адаптация sa-helper: якорь смещён с `code:line` на **продуктовые источники** (JIRA / PO / Confluence / CORTEX-кортексы).

---

## Pipeline (6 стадий, зеркало sa-helper FNR)

```
/bft-context-gen  → context-pack          [СТОП: PO ревьюит pack]
        ↓
/bft-problem      → problem.md            (диагноз As-Is/Gap, БЕЗ решения)   [СТОП]
        ↓
/bft-concept      → concept.md            (2-3 концепта решения)             [СТОП]
        ↓
/bft-debate       → вердикт в concept.md  (3 раунда, Адвокат Дьявола)
        ↓ Забраковано/Пивот → /bft-concept
        ↓ Принят
/bft-draft        → черновик БФТ          (БТ/ПТ/ИТ/ФТ/НФТ из концепта)      [СТОП]
        ↓
/bft-validate     → validation.md         (hard gates + Светофор)
        ↓ 🔴 нарушен гейт → /bft-draft
        ↓ 🟢/🟡
/bft-deliver      → публикация            (JIRA Эпик + 2×Confluence + связи)  [СТОП: сухой прогон → ок PO]
        ↓
     Done
```

### Роли и артефакты

| Стадия | Команда | Роль | Артефакт |
|---|---|---|---|
| 0 Контекст | `/bft-context-gen` | Context Builder (Кортексы C1–C5 + Нексусы N1–N7) | `artefacts/bft-context-pack.md` |
| 1 Проблема | `/bft-problem` | Problem Analyst (диагноз, не решение) | `artefacts/problem.md` |
| 2 Концепты | `/bft-concept` | Solution Designer (2-3 варианта) | `artefacts/concept.md` |
| 3 Дебаты | `/bft-debate` | Architect vs Devil's Advocate | вердикт в `artefacts/concept.md` |
| 4 Требования | `/bft-draft` | Requirements Writer | `<epic>.md` (финальный БФТ) |
| 5 Валидация | `/bft-validate` | Validator (свежий взгляд) | `artefacts/validation.md` |
| 6 Отгрузка | `/bft-deliver` | Deliverer (публикация) | JIRA Эпик + 2×Confluence + связи; манифест `delivery.md` |

**Почему разделение:** разные роли не «загрязняют» друг друга (диагноз ≠ решение ≠ требование); adversarial отдельным запуском ломает confirmation bias; артефакты-передачи проверяемы; STOP-паузы дают human-in-the-loop ревью.

### Рабочая папка эпика (staging в `bft_documentation/`)

**Все наработки — внутри рабочей папки эпика `bft_documentation/<epic>/`.** Финальный БФТ `<epic>.md` лежит в корне папки эпика; промежуточные артефакты пайплайна — в `artefacts/`. Финализация (готовность к ревью) — после 🟢/🟡 верификации (см. принцип 15).

```
bft_documentation/<epic>/
├── <epic>.md                 (/bft-draft) → ФИНАЛЬНЫЙ БФТ (главный документ)
└── artefacts/                промежуточные артефакты пайплайна
    ├── bft-context-pack.md   (/bft-context-gen)
    ├── problem.md            (/bft-problem)
    ├── concept.md            (/bft-concept → +вердикт от /bft-debate)
    └── validation.md         (/bft-validate)
```

Финальный БФТ лежит в корне папки эпика под именем `<epic>.md` — это то, что открывает PO. Всё остальное (контекст, диагноз, концепты, отчёт валидации) — служебные артефакты в `artefacts/`.

## Принципы (критерии качества)

### 1. Доказательная база
Каждое требование ← источник (БТ ← проблема/ТЗ; ФТ ← сценарий/PO; НФТ ← измеримый критерий). Без источника → `[УТОЧНИТЬ]`.

### 2. As-Is → Gap → To-Be
As-Is (с якорями: JIRA/Confluence/код/CORTEX-СА) → Gap → To-Be в требованиях.

### 3. Трассировка каскадом
Идентификаторы **локальные** `{БТ|ПТ|ИТ|ФТ|НФТ}-{N}` (ЗМ-003) — якоря навигации внутри документа, без EPIC/проекта. Каскад `БТ ← ПТ ← ИТ` и `БТ ← ФТ ← НФТ`; связь с эпиком — через «Связанные требования» (ключ трекера), не через идентификатор. Пустых «Связанные требования» — запрещено.

### 4. Измеримость
НФТ проверяемо: «≤ 2000 ms P95», «≥ 99.5% TSR», «100 RPS». Не «быстро/надёжно».

### 5. Границы (двойные)
Раздел обязателен, две плоскости:
- **5а. Границы системы — ВНУТРИ / СНАРУЖИ:** за что сервис/фича отвечает и за что НЕ отвечает (напр. «API-слой не принимает платежи»).
- **5б. Высота БФТ (altitude):** документ держится на уровне **результата и приёмки** — как это выглядит для бизнеса и пользователя при сдаче. БФТ НЕ опускается на уровень системных требований, API-контрактов (эндпоинты, REST, протоколы, схемы сообщений, идемпотентность и auth как контракт), sequence-диаграмм с вызовами между сервисами, имён таблиц/полей БД и кода. Это зона СА / системных требований (FNR). **ИТ — интерфейс приёмки** (что видит и может актор на экране и в результате), не API-контракт. Цель БФТ — точные критерии приёмки и план демонстрации (что / где / почему изменено) для заказчика, без технических деталей. Enforcement — гейт 14 (`hard_gates.md`).

### 6. Negative Flows
Не только happy path. Каждый сценарий имеет `alt`: ошибка, поставщик недоступен, отмена, таймаут, конфликт.

### 7. No Silent Skip
Неясности → «Открытые вопросы». Закрытые → «Ключевые решения» (с «кто ответил»). Неизвестное — артефакт, не выдумка.

### 8. Якоря истины
Финальный раздел: факт → источник (JIRA / PO / Confluence / BR / ADR). Аудит-след.

### 9. Adversarial (отдельная стадия)
Перед фиксацией требований — red-teaming (`/bft-debate`): нагрузка, отказ поставщика, идемпотентность, изменение требований, ПДн/Compliance, скрытые зависимости (API-слой↔Шина↔Процессинг↔BI). См. `resources/debate_rules.md`.

### 10. Декомпозиция
Каждый ФТ ≈ атомарная задача (1-2 дня), со ссылкой на задачу трекера (`tracker.projects`).

### 11. Множественность worldviews (SSM)
В системе нет одной «правды»: заказчик, архитектор, Legacy-платформа-legacy, регуляторика — конкурирующие взгляды. **Держи их в напряжении**, выноси конфликты в «Открытые вопросы», не схлопывай в одну версию.

### 12. Требование = гипотеза
Каждое БТ/ФТ — утверждение, подлежащее проверке (As-Is/To-Be, ревью, дебаты), а не финальная истина. Режим «предложить и проверить».

### 13. CATWOE + ранги якорей
Крупные БТ прогоняй через CATWOE (`resources/catwoe.md`). Якорь факта — по рангам R1/R2/R3 (`resources/anchor_rules.md`): код (R1) — только для As-Is; To-Be → R2 (JIRA/Confluence/BR/ADR) или R3 (PO-решение).

### 14. Стиль и голос (Humanizer)
Факты — содержание, стиль — регистр, в котором их подают. БФТ пишется плотным деловым русским, как золотые референсы (`examples/golden_bft_example.md`, `{bft_store}/золотой референс …`), без AI-примет: эмодзи, раздутого bold, тире-связок, callout-врезок, `[УТОЧНИТЬ]`-загрязнения, diff-narrative в frontmatter, стоп-слов («позволяет», «обеспечивает», «представляет собой»). Правила — `resources/writing_style.md`. Стиль — enforcement-слой (проверяется гейтом 13 в `/bft-validate`), не рекомендация. Стиль самого генератора задаёт стиль вывода: не вставляй в артефакты приёмы, которых нет в золотой референс.

### 14. Обратная связь ревью (Lessons Learned)
Навык **учится на замечаниях PO**. Каждое замечание к сгенерированному БФТ → строка-правило в `resources/review_feedback.md` (анти-паттерн + правило). `/bft-draft` применяет активные правила при генерации, `/bft-validate` проверяет (гейт 15). Принцип: одну и ту же ошибку не повторять дважды.

### 15. Staging → финализация (рабочая папка эпика)
Все промежуточные наработки (context-pack, problem, concept, validation) живут в `bft_documentation/<epic>/artefacts/`; финальный БФТ — `bft_documentation/<epic>/<epic>.md`. БФТ считается **финализированным (готовым к ревью)** только после успешной верификации `/bft-validate` (🟢 или 🟡; при 🔴 — возврат на `/bft-draft`, артефакты остаются в `artefacts/`). Непроверенное за готовое не выдаётся.

## Стандарты и ресурсы

- `resources/bft_standards.md` — идентификаторы, различение типов (ПТ≠ФТ≠НФТ), НФТ-набор + приоритет корпоративного реестра НФТ, frontmatter, словарь, порядок разделов (ЗМ-005), формат таблиц.
- `resources/hard_gates.md` — **15 бинарных 🔴-гейтов** (вкл. гейт 13 «Стиль/Голос», гейт 14 «Высота БФТ», гейт 15 «Known-Mistake») + чек-лист + Светофор.
- `resources/anchor_rules.md` — ранги якорей R1 (код, As-Is) / R2 (JIRA/Confluence/BR/ADR) / R3 (PO-решение); код-якорь только для As-Is.
- `resources/catwoe.md` — CATWOE-декомпозиция (W→БТ, O→Ревью, E→НФТ).
- `resources/writing_style.md` — стиль и голос (Humanizer): стоп-слова, тире, эмодзи, bold, `[УТОЧНИТЬ]`-дисциплина, калибровка под золотой референс.
- `resources/debate_rules.md` — протокол adversarial (для `/bft-debate`).
- `resources/review_feedback.md` — **реестр замечаний ревью (Lessons Learned)**: замечания PO → durable-правила, применяемые в `/bft-draft` и проверяемые в `/bft-validate` (гейт 15).
- `examples/ideal_bft.md` — пустой шаблон по docx-стандарту.
- `examples/golden_bft_example.md` — аннотированный эталон (BOOK-2).

Кортексы CORTEX (C1 Архитектура / C3 Регуляторика / C2+ Бизнес-правила / C5 Решения) — статичный фон, подключаются в `/bft-context-gen`.

Диаграммы — **только PlantUML**. Mermaid запрещён. Надписи на русском, тех.имена в скобках.

---

## Главное правило процесса

**STOP после каждой стадии.** Не лети до финала. После `/bft-context-gen`, `/bft-problem`, `/bft-concept`, `/bft-draft` — выводи результат и ожидаешь решения PO/СА. Только так pipeline = sa-helper по качеству, а не «генерация за один промт».
