# /paf-install — развернуть экосистему po-helper

Оркестратор установки. `install.sh` делает всё детерминированное (копия репозитория, установка CLI, адаптация конфигов, self-check). Этот скилл его запускает, разбирает машиночитаемый отчёт, добивает шаги, требующие рассуждения/выбора, честно рапортует состояние и передаёт финальную доменную настройку в `/paf-init`.

> Это пошаговый план для LLM. Выполняй по порядку. Не объявляй экосистему готовой, пока не свёл отчёт Phase 5 и не проверил факты. Ничего не выдумывай про статус тулов — бери из отчёта.

## Принцип разделения

- **install.sh** = «сделал одинаково и проверил факты» (копия, CLI, пути, init).
- **Этот скилл** = «разрулил выбор, объяснил человеку, довёл до готового».

Не дублируй логику скрипта. Не переписывай конфиги руками, если это уже сделала Phase 4 — сверяйся с отчётом.

---

## 1. Определить режим и запустить движок

1. Реши режим:
   - target уже содержит `.claude/skills/` **и** `.claude/domain-profile.md` → это обновление → `--update`.
   - иначе → первая установка → без флага.
2. Запусти установщик из корня проекта:
   - если пользователь в клоне po-helper и ставит в другой проект: `bash <клон>/install.sh [--update] <путь-к-target>`;
   - если ставит «сюда» из клона: `bash install.sh [--update]`;
   - через curl (нет клона): `curl -fsSL https://raw.githubusercontent.com/kibarik/po-helper/main/install.sh | bash -s -- [--update]`.
3. **Захвати весь stdout.** Ненулевой exit — это фатал Phase 0–2 (нет источника/rsync/сети): покажи ошибку пользователю дословно и остановись, не притворяйся, что установил.

---

## 2. Разобрать отчёт Phase 5

Найди в выводе блок между `=== PAF-INSTALL-REPORT ===` и `=== END-REPORT ===`. Разбери строки `key: value`:

| Строка | Что значит | Твоё действие |
|---|---|---|
| `repo_copied: OK` | дерево скопировано | ок |
| `MISSING_PATH: <p>` | ключевой путь не появился | эскалируй как проблему установки |
| `ruflo: OK/DEGRADED/MISSING` | MCP-память | при DEGRADED/MISSING → §3 |
| `entire: OK/DEGRADED` | session-tracking | при DEGRADED → §3 |
| `backlog: OK/DEGRADED` | операционный штаб | при DEGRADED → §3 |
| `serena: OK/DEGRADED` | код-MCP рантайм | при DEGRADED → §3 |
| `mcp_json: ADAPTED/SKIPPED` | путь ruflo в .mcp.json | информационно |
| `serena_name: SET/SKIPPED` | project_name серены | сверь в §4 |
| `domain_profile: TEMPLATE/PRESENT` | доменный профиль | TEMPLATE → §5 (обязательно) |
| `backlog_init: OK/SKIPPED/FAILED` | доска | FAILED → §3 |

Составь для пользователя короткую сводку «что встало / что деградировало», по фактам из отчёта.

---

## 3. Ремедиация деградаций (best-effort → честность)

Для каждого `DEGRADED`/`MISSING`/`FAILED` покажи пользователю причину и **точную** команду из отчёта. Затем:

- Если причина — отсутствие пакетного менеджера/рантайма (npm, bun, uvx, git, rsync): предложи поставить его и повторить (например `pip install uv` для serena, `brew install rsync`). Спроси разрешение перед установкой чего-либо глобального.
- Если пользователь не хочет чинить сейчас — зафиксируй как «опционально, экосистема работает частично» и **не** помечай эти пункты как готовые в финальном чеклисте.
- ruflo/serena/entire/backlog — все опциональны для базовой работы пайплайнов; не блокируй установку из-за них, но и не ври об их наличии.

Никогда не объявляй «всё готово», если в отчёте остались непогашенные DEGRADED/MISSING — перечисли их явно.

---

## 4. Рассуждающая адаптация (то, что скрипт намеренно не делает)

1. **Имя проекта (serena/backlog).** Скрипт проставил `project_name` = имя папки. Если оно неудачное (случайное/техническое) — предложи осмысленное и поправь `.serena/project.yml` (и переиспользуй в backlog при необходимости).
2. **Язык(и) `.serena/project.yml`.** Скрипт НЕ трогает `languages` (там дефолт `python`). Определи по target:
   - репозиторий доменных `.md`/заметок без кода → `markdown`;
   - есть код → соответствующий язык (ts/python/go/…).
   Проставь корректно, чтобы serena-LSP не стартовал зря.
3. **Конфликты при `--update`.** Если пользователь ждал обновления защищённого файла (например новый MCP-сервер в `.mcp.json`), а он защищён от перезаписи — покажи diff сорса и target и спроси, применять ли вручную. Не затирай молча.

---

## 5. Хендофф в /paf-init (доменная настройка)

Если отчёт вернул `domain_profile: TEMPLATE` — экосистема установлена, но не адаптирована под домен. Финальную настройку делает существующий скилл, не дублируй его:

1. Предложи заполнить `.claude/domain-profile.md` (пути, доменные термины) — из `.claude/domain-profile.template.md`.
2. Запусти `/paf-init` — интервью → `GROUND/config.yaml` + засев каталога Нексусов.

Если `domain_profile: PRESENT` — профиль уже есть, `/paf-init` не нужен (только если пользователь хочет пересобрать GROUND).

---

## 6. Финальная верификация и отчёт пользователю

Прежде чем сказать «готово», проверь факты (а не отчёт):

1. Ключевые деревья на месте: `.claude/skills` (ожидается 14), `.claude/commands`, `GROUND/NEXUS`, `docs/`, `sa_documentation/`.
2. MCP-серверы поднимутся после Reload Window: `.mcp.json` присутствует и путь ruflo валиден (или bare `ruflo` на PATH).
3. `.git` источника **не** утёк в target (у проекта свой репозиторий).

Выдай итоговый чеклист в таком духе (по фактам):

```
Экосистема po-helper: <проект>
  ✓ Репозиторий скопирован (N скиллов, M команд)
  ✓/⚠ ruflo · entire · backlog · serena  — <статус каждого из отчёта>
  ✓ .mcp.json адаптирован · serena project_name = <имя>
  ⚠ domain-profile — нужен /paf-init   (или ✓ если PRESENT)
Дальше: Reload Window → /paf-init → первый пайплайн (/okr-* · /bft-* · /sprint-*)
```

Честно помечай `⚠` всё, что деградировало. «Готовая экосистема» = либо всё зелёное, либо зелёное ядро + явно перечисленные опциональные пробелы с командами их закрыть.

## Guardrails

- Не запускай глобальные установки (npm/bun/pip/curl) без согласия пользователя — предлагай команду.
- Не объявляй успех при непогашенных DEGRADED/MISSING.
- Не перезаписывай domain/project-слой руками в обход защиты скрипта.
- Ненулевой exit install.sh — это стоп: разбери причину, не продолжай «как будто установилось».
