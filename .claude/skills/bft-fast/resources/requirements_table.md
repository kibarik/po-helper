# Вложение — таблица требований на согласование границ (requirements_table.md)

Второй артефакт `/bft-fast` (после письма): **блок персон** + **сводная таблица ключевых требований** в структуре БФТ (БТ/ПТ/ИТ/ФТ/НФТ). Назначение — согласование границ задач на этапе первичного брифинга.

**Главное отличие от полного БФТ:** таблица содержит **только то, что сказали люди на встрече**. Каждая строка формируется из **цитаты** участника. Никакого обогащения, порогов, приоритетов, выдуманных параметров, домысленного ASIS — это скелет на цитатах; детальная проработка и обогащение происходят на следующем этапе (deep `bft-*`).

## Блок персон (домен знаний)

Перед таблицей требований — краткое описание участников: **не каждая цитата является достоверным требованием**, вес реплики зависит от роли и зоны ответственности говорящего (реплика владельца требования весомее реплики вне его зоны). Блок даёт читателю основу для оценки достоверности.

| Колонка | Правило |
|---|---|
| ФИО | из `участники[]` (frontmatter/упоминания) |
| Роль | из People Graph (`capacity.roster_source` = `GROUND/NEXUS/team`) или явного упоминания; только по контексту встречи → пометить (напр. «PO инициативы»); неизвестно → `[роль не подтверждена]` |
| Департамент | из People Graph / явного упоминания; не озвучено → `[не указано]` |

Правила: ФИО не выдумывать; роль/департамент — из ростера или сказанного, домысел из поведения помечать, полностью неизвестное → `[не указано]`/`[роль не подтверждена]`. Персоны — кандидаты на фиксацию/сверку в People Graph (`GROUND/NEXUS/team`) на этапе проработки; это тот самый «домен знаний», по которому взвешивается достоверность цитат.

**Частая ошибка (не делать):** департамент не звучал на встрече, но проставлен «Product»/«Engineering»/«QA» по догадке. Это галлюцинация. Не сказано → `[не указано]`. То же для роли: если человек только подал реплику, но его роль не названа и не в ростере → `[роль не подтверждена]`, а не выдуманная «Backend Lead».

## Колонки

| Колонка | Что | Правило |
|---|---|---|
| ID | `{ТИП}-{N}` (локальный) | `БТ-1`, `ПТ-1`, … — тип из префикса; N без ведущих нулей (`БТ-1`, не `БТ-001`) |
| ASIS (сейчас) | как работает сейчас | **только из цитаты**; не озвучено на встрече → `[ASIS не озвучен]`, не додумывать |
| TOBE (после) | как будет после изменения | из цитаты; целевое состояние, которое проговорили |
| Связанные | **только ID** других строк (`БТ-1`, `ПТ-1`), не свободный текст; нет связи → пустая ячейка (не `[не указано]`) | каскад `БТ ← ПТ ← ИТ`, `БТ ← ФТ ← НФТ`; только где связь видна из встречи |
| Источник (цитата) | транскрипт → `ФИО: «…»`; `#summary`-заметка → `Summary: «…»` | обязателен; без цитаты/строки-источника строки нет |

## Фиксированный порядок иерархии

Строки **всегда** в этом порядке (тип — из префикса ID):

1. **Бизнес-требование (БТ)**
2. **Пользовательские требования (ПТ)**
3. **Требование к интерфейсам (ИТ)**
4. **Функциональные требования (ФТ)**
5. **Нефункциональные требования (НФТ)**

Различение типов:
- **БТ** — ценность/зачем (бизнес-требование).
- **ПТ** — что хочет пользователь.
- **ИТ** — что видит/делает актор на экране (интерфейс приёмки), не API.
- **ФТ** — поведение системы на уровне результата.
- **НФТ** — измеримое -ility, **но только если озвучено** (не выдумывать пороги вроде «≤2000ms»).

## Правила грунтования (zero hallucination — критично на брифинге)

1. **Строка ← цитата.** Нет реплики участника, из которой прямо следует требование — строки нет. «Источник» обязателен и непуст.
2. **ASIS только из сказанного.** Текущее состояние — лишь если его проговорили. Не проговорили → `[ASIS не озвучен]`. Не реконструировать «как оно наверное сейчас».
3. **Без обогащения.** Не добавлять пороги, приоритеты, параметры, негативные сценарии, если их не проговорили. Это работа следующего этапа.
4. **Тип по факту.** Включать тип, только если под него есть цитаты. ИТ/НФТ на discovery часто пусты — тогда их в таблице нет, не добивать.
5. **Цитата верна.** Дословно или максимально близко к источнику; ASR-искажения чинить осторожно, смысл не менять. Атрибуция — по говорящему. **Таймстемпы не цитировать** (формат `ФИО: «текст»`, без `(18:37:97)`).
6. **out-of-scope не в таблицу.** Что отнесли к «не делаем»/следующему кварталу — не требование этой итерации (уходит в «Границы» письма, не в таблицу).

## Форма вывода

Отдельным блоком после письма. Сначала блок персон, затем таблица требований (порядок строк — БТ → ПТ → ИТ → ФТ → НФТ):

```
Вложение — таблица требований на согласование границ:

Персоны (домен знаний):

| ФИО | Роль | Департамент |
|---|---|---|
| … | … | [не указано] |

Роли/департаменты — из People Graph или сказанного; неизвестное помечено. Вес цитаты зависит от роли.

Требования:

| ID | ASIS (сейчас) | TOBE (после) | Связанные | Источник (цитата) |
|---|---|---|---|---|
| БТ-1 | … | … | ПТ-1 | ФИО: «…» |
| ПТ-1 | [ASIS не озвучен] | … | БТ-1 | ФИО: «…» |
…

Только зафиксированное на встрече. ASIS без реплики — [ASIS не озвучен]. Обогащение — на этапе глубокой проработки.
```

Таблица — часть вложения, отдельно от тела письма (письмо остаётся самодостаточным для пересылки).
