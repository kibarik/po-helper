# Правила промоута — pulse-promote

Справочник фильтра устойчивости, шаблонов и дедуп-логики для скилла `/pulse-promote`.

---

## Фильтр устойчивости

Таблица: тип сигнала радара → действие.

| Тип сигнала | Маркер радара | Действие |
|---|---|---|
| `incident` | 🔴 | Закрытый/разовый → **пропуск**. Повторяющийся / структурный (упоминался в прошлых радарах или явно «снова») → **new-problem**. |
| `blocker` | 🚧 | Тормозит конкретный KR → **kr-risk-update**. Структурная зависимость / отсутствие ответственного → **system-fact** или **ownership-fact**. |
| `decision/change` | ✅ | Длящееся поведение системы (архитектурный костыль, постоянное ограничение) → **system-fact**. Разовое решение → **пропуск**. |
| `risk` | 🚧 🟠 | Привязан к KR → **kr-risk-update**. Не привязан, но структурная боль → **new-problem**. |
| `action-on-PO` | ❗ | **Пропуск** действия. НО: если строка несёт **встроенный длящийся факт** (структурная проблема/риск за формулировкой «нужна встреча / сделай X») — извлечь этот факт как new-problem/kr-risk, действие пропустить. |
| `📈 влияние` | 📈 | **Пропуск** (перекрёстные ссылки, не первичный факт). |
| `🔕 инфошум` | 🔕 | **Пропуск**. |

**Колонка «Маркер радара» = секция радара** (`##`-заголовок), а не per-line severity.
Внутри строк секций стоят маркеры важности `🔴`/`🟠`/`⚪` — они **не** влияют на
промоутируемость (сигнал `🟠` в `🚧` промоутится так же, как `🔴`). Фильтр — по типу/секции
и длящейся ценности, не по severity.

**Секция — подсказка, не абсолют:** первичен длящийся факт. Строка из `❗`/`✅` со встроенной
структурной проблемой промоутится; чисто эфемерное (статус, разовое действие) — нет.

**Итоговые типы кандидатов после фильтра:**
- `new-problem` → `GROUND/NEXUS/problem/problem-<slug>.md`
- `kr-risk-update` → `GROUND/NEXUS/okr/kr-*.md` (additive)
- `system-fact` → `GROUND/NEXUS/system/system-<slug>.md` (апдейт или new)
- `ownership-fact` → `GROUND/NEXUS/ownership/ownership-<slug>.md` (апдейт или new)

## Шаблон нового узла

Использовать дословно; заполнять все поля; `<угловые скобки>` = плейсхолдер.

```yaml
---
nexus: <problem|system|ownership>
node_id: <problem|system|ownership>-<slug>
node_type: <problem → step-overview | новый system/ownership из радара → fact>
kind: empirical
owner: <po_name из .claude/domain-profile.md>
confidence: 0.3
sources: [radar-ГГГГ-ММ-ДД, "<Чат> · <Автор> · <время>"]
updated: ГГГГ-ММ-ДД
ttl_days: 90
ripeness: fresh
title: <краткий заголовок — факт, не вопрос>
source_note: GROUND/PULSE/radar/ГГГГ-ММ-ДД-pulse-radar.md
owns_node: <team-node-id PO из domain-profile (node_id PO-узла в NEXUS/team)>
mentions: [<node_id смежных system-*/kr-*/ownership-* из текста сигнала>]
involves: [<team-* участники из якоря>]
---
# <Заголовок — тот же, что title>

<Факт одним абзацем: что происходит, почему проблема, с якорем чата.>
```

**Правила заполнения:**
- `confidence: 0.3` — всегда для узлов из радара (чат = одиночный неверифицированный источник; ниже 0.4 curated-проблем).
- `sources` — два элемента: `radar-ГГГГ-ММ-ДД` (идентификатор радар-прогона) + строка якоря `"<Чат> · <Автор> · <время>"`.
- `source_note` — абсолютный путь к радар-заметке относительно корня vault.
- `mentions` — `node_id` смежных узлов, упомянутых в сигнале (KR, system, ownership). Не выдумывать; если нет — `[]`.
- `involves` — `node_id` team-узлов участников. Если нет уверенного матча — `[]`.
- `owns_node` — team-узел PO (node_id из `NEXUS/team`, PO из domain-profile); всегда для новых узлов из этого скилла.

## Формат additive-апдейта существующего узла

Применять строго в указанном порядке; никакого другого изменения файла не делать.

### 1. Тело: раздел «Сигналы из чатов (радар)»

Найти в конце тела раздел `## Сигналы из чатов (радар)`.
Если раздела нет — создать в самом конце файла:

```markdown

## Сигналы из чатов (радар)
```

Дописать под раздел новый булет:

```
- [ГГГГ-ММ-ДД] <факт одной фразой> ← <Чат> · <Автор> · <время> (radar)
```

**Никогда** не удалять и не редактировать уже существующие булеты этого раздела.

### 2. Frontmatter: `updated` и `sources`

- Поднять `updated` до даты текущего прогона.
- Добавить `radar-ГГГГ-ММ-ДД` в список `sources`, если его там ещё нет.
- **Больше ничего** во frontmatter не менять: `confidence`, `title`, `node_type`, `kind` — не трогать. Curated `system-*` могут иметь богатый `node_type` (напр. `system-component`) — при additive-апдейте его НЕ менять.

## Дедуп и slug

### Slug

- 2–4 слова латиницей, кебаб-кейс, по теме узла.
- Примеры: `faq-redis-scale`, `vtb-fintech-block`, `hallview-iframe-php`.

### Перед предложением new-node

1. Определить целевой домен (`problem` / `system` / `ownership`).
2. Поискать в `NEXUS/<domain>/` файлы, содержащие slug-термины темы.
3. Проверить близость `title` существующих узлов к предлагаемому.
4. Если найден узел с совпадающей темой/системой → предлагать **апдейт** найденного узла, не создавать новый.

### Идемпотентность

При повторном прогоне по той же радар-заметке:
- Новый узел с тем же `node_id` уже существует → кандидат помечается `already-promoted`, пропускается.
- Апдейт-булет с тем же якорем (дата + чат + автор) уже есть в разделе `## Сигналы из чатов (радар)` → `already-promoted`, пропускается.
- Повторный прогон **не плодит дубли**.
