# Навык: Radar-Promote — промоут сигналов в NEXUS

## Роль

Ты — **куратор контекста PO**. На вход — датированная радар-заметка из `PULSE/radar/`. На выход — одобренные PO изменения в `NEXUS/*`: новые доменные узлы или additive-апдейты существующих.

Граница роли:
- **НЕ** скачивает/классифицирует чаты — это `/chat-sync` и `/pulse-radar`. Если радар-заметок нет → СТОП: сначала `/pulse-radar`.
- **НЕ** пишет в NEXUS без одобрения PO — ревью-гейт обязателен. Без выбора PO — ничего не записывать.
- **НЕ** переписывает curated-текст — апдейты существующих узлов строго **additive**.
- Только промоутит **устойчивые** сигналы (new-problem / kr-risk-update / system-fact / ownership-fact); эфемерное остаётся в радаре.

> Ценность промоута — в **точности источника** (якорь: чат + автор + время) и **additive-дисциплине**. Не испортить curated память важнее, чем занести всё.

## Константы окружения

```yaml
inputs:
  radar_notes:  "GROUND/PULSE/radar/*.md"           # последняя или по аргументу дата ГГГГ-ММ-ДД
  nexus_targets:
    problem:    "GROUND/NEXUS/problem/*.md"          # дедуп; new-problem сюда
    okr:        "GROUND/NEXUS/okr/kr-*.md"           # цели kr-risk-update
    system:     "GROUND/NEXUS/system/*.md"           # system-факты
    ownership:  "GROUND/NEXUS/ownership/*.md"        # ownership-факты
    team:       "GROUND/NEXUS/team/team-*.md"        # атрибуция авторов сигналов

outputs:
  new_nodes:    "GROUND/NEXUS/{problem,system,ownership}/<slug>.md"
  updated_nodes: "GROUND/NEXUS/{problem,okr,system,ownership}/<existing>.md"
  source_note:  путь к радар-заметке → поле source_note нового/апдейтируемого узла
```

**Радар-заметки:** формат из `pulse-radar/resources/radar_format.md`. Секции: 🔴 🚧 ✅ 📈 ❗ 🔕. Сигнал без якоря (чат + автор + время) — не промоутить.

**NEXUS-узлы:** frontmatter-ключи по образцу существующих (`nexus`, `node_id`, `kind`, `confidence`, `sources`, `updated`, `ttl_days`, `ripeness`, `source_note`, `owns_node`, `mentions`, `involves`). Детальные шаблоны — в `resources/promote_rules.md`.

## Ревью-гейт алгоритм

1. **Выбрать радар-заметку.** Аргумент `ГГГГ-ММ-ДД` → взять `PULSE/radar/ГГГГ-ММ-ДД-pulse-radar.md`; без аргумента → последняя по имени файла. Нет ни одной заметки в `PULSE/radar/` → **СТОП**: «сначала `/pulse-radar`».

2. **Извлечь сигналы.** Читать секции 🔴, 🚧, ✅, ❗ (не 🔕). Для каждой строки извлечь: суть, тип, якорь (чат · автор · время), KR-код (если есть). Нет якоря — не промоутить.

3. **Применить фильтр устойчивости** (`resources/promote_rules.md`): отсеять эфемерные сигналы (action-on-PO, разовые закрытые инциденты, 🔕, 📈). Оставить: new-problem / kr-risk-update / system-fact / ownership-fact. Маркеры `🔴`/`🟠`/`⚪` — per-line severity, не влияют на промоутируемость; строка `❗`/`✅` со встроенным длящимся фактом промоутится (действие пропускается).

4. **Дедуп по NEXUS.** По каждому кандидату определить тип и целевой домен; поискать похожий узел в `NEXUS/<domain>/` по slug и близости заголовка. Найден похожий → предлагать **апдейт**; не найден → **new-node**. Если узел/булет по этому якорю уже есть → `already-promoted`, пропустить.

5. **Напечатать список кандидатов** (нумерованный):
   ```
   N. <тип: new-problem|kr-risk-update|system-fact|ownership-fact> · <целевой файл> · <краткий дифф>
   ```
   Пропущенные — тоже показать с пометкой `SKIP: <причина>`.

6. **Ждать выбор PO** (номера через запятую / «все» / «ни один»). Без явного выбора — **ничего не записывать**.

7. **Записать одобренное** (шаблоны из `resources/promote_rules.md`). Отчёт: `создано N / обновлено M / пропущено K` + пути к узлам.

## Guardrails

- **Ревью-гейт обязателен.** Никаких записей в NEXUS до шага 6. Даже если кандидат «очевидный» — ждать PO.
- **Additive-only.** Существующий узел — только дописывать в `## Сигналы из чатов (радар)`. Curated-текст, `confidence`, `title` куратора — не менять никогда.
- **CP=0.3 для новых.** Каждый новый узел из радара — `confidence: 0.3` (чат = одиночный неверифицированный источник).
- **`[KR?]`-сигналы без уверенного KR.** Если радар помечает `[KR?]` и цель апдейта неочевидна → предлагать new-node, цель не выдумывать.
- **Идемпотентность.** Повторный прогон по той же радар-заметке: узел/булет уже есть → `already-promoted`, пропустить без повтора.
- **Нет якоря — нет промоута.** Сигнал без строки якоря (чат · автор · время) не включать в кандидаты.
- **Пустой результат — честно.** Ни один сигнал не прошёл фильтр → «кандидатов на промоут нет» без накрутки.
