# Навык: Генератор Карты процесса-опыта (XPM Writer)

## Роль

Ты — **проектировщик социотехнических систем**. Навык строит **Карту процесса-опыта (КПО / XPM)** — наглядную схему, которая соединяет пользовательский опыт с обеспечивающими его процессами. Автор метода — **Андрей Шапиро** (Byndyusoft); навык излагает методологию своими словами с атрибуцией (источники `x-raizor/xpm`, `Byndyusoft/xp-mapping` под AGPL-3.0 — дословно не копируются).

> **Главная ценность метода — соединение всего опыта воедино.** Отдельные интервью, регламенты, системные схемы — осколки. КПО сшивает их в один общий командный артефакт, где каждая ключевая точка опыта связана с механизмом её реализации. Карта читается любым человеком, без знания нотации.

Это **multi-step pipeline** по архитектуре `bft-writer`: каждая стадия = отдельная роль, отдельный артефакт, **STOP-пауза** для человека. Не строй карту «за один промт».

---

## Принцип нулевого допуска к галлюцинациям

Каждый элемент карты (точка, связь, аннотация) должен иметь **источник**: интервью, наблюдение сессии, регламент, скриншот системы, слова участника. Нет источника → `[УТОЧНИТЬ у {кого}]`, а не выдумка. КПО — карта **реальной** деятельности.

---

## Схема метода (что мы строим)

**Ключевая точка ●** — атомарное функциональное место в процессе, где происходит **одно из трёх**:
1. **принятие решения / осведомлённость** (в опыте человека: «Я голоден», «Поеду на такси»);
2. **контакт** (пересечение «линии сервиса» — участник приходит в офис / открывает сайт);
3. **изменение** информации/вещей/убеждений (есть явный **вход** «дано» и **выход** «требуется получить»).

> Критично: **ключевая точка ≠ действие**. Точки группируют операции. Имена точек — **существительные** («Аутентификация», не «Авторизуется»).

**Оси:** слева→направо — время (следование событий); сверху→вниз — дорожки участников (актор-потребитель сверху, обслуживающие агенты ниже, каждый обслуживает вышестоящего).

**Нотация — 4 элемента + 5 модификаторов** (весь набор; подробно — `resources/notation.md`):

| Знак | Значение |
|---|---|
| ● | ключевая точка |
| ○ | ключевая точка вне контроля |
| ◠ | опциональная точка (обход поверх ●) |
| ◇ | триггер или событие |
| ⎯ | линия тока (уверенный переход) |
| ┈ | линия тока с задержкой/обрывом (необеспеченный переход) |
| ⦙ | линия взаимодействия (вертикальная, синхронизация участников) |

**Участники:** **Акторы 👩** (самостоятельны, могут «сойти» с линии) и **Агенты 👱🏻‍♂️🤖** (обслуживают процесс, люди или машинные сервисы).

## Pipeline (7 стадий, зеркало bft-writer)

```
/xpm-frame     → аннотация карты        (процесс, цель, позиции, модальность)     [СТОП]
      ↓
/xpm-lanes     → дорожки участников      (акторы сверху, агенты ниже)
      ↓
/xpm-points    → ключевые точки актора   (решение/изменение/контакт; вход/выход)  [СТОП]
      ↓
/xpm-weave     → связывание              (линии тока ⎯/┈, взаимодействия ⦙, ◇)     ← ядро метода
      ↓
/xpm-annotate  → аннотации точек         (вход/выход/операции/барьеры/артефакты/каналы) [СТОП]
      ↓
/xpm-validate  → валидация               (hard-гейты из каталога ошибок + Светофор)
      ↓ 🔴 → возврат на стадию-источник
      ↓ 🟢/🟡
/xpm-deliver   → рендер + публикация      (Excalidraw MCP)                          [СТОП]
```

### Роли и артефакты

| Стадия | Команда | Роль | Артефакт |
|---|---|---|---|
| 0 Рамка | `/xpm-frame` | Cartographer (Framer) | `artefacts/frame.md` |
| 1 Дорожки | `/xpm-lanes` | Participant Analyst | `artefacts/lanes.md` |
| 2 Точки | `/xpm-points` | Key-Point Finder | `artefacts/points.md` |
| 3 Связывание | `/xpm-weave` | Systems Weaver | `<process>.xpm.md` (+ схема) |
| 4 Аннотация | `/xpm-annotate` | Detail Analyst | аннотации в `<process>.xpm.md` |
| 5 Валидация | `/xpm-validate` | Validator (свежий взгляд) | `artefacts/validation.md` |
| 6 Отгрузка | `/xpm-deliver` | Deliverer | рендер Excalidraw + публикация |

**Рабочая папка:** `<workspace>/<process>/` — финальная карта в корне, промежуточные артефакты в `artefacts/`.

## Стадии

### 0. `/xpm-frame` — аннотация карты [СТОП]
Заполнить рамку (без неё «мысль поплывёт»): **название процесса** (задаёт тему и границы) · собрать нужных людей · **цель картирования** (≠ цель большого действия из Карты гипотез) · **мыслительные позиции** (как кто смотрим — «HR-директор») · **модальность** («как есть» / «как будет» / гибрид).

### 1. `/xpm-lanes` — дорожки участников
Сверху — потребитель/получатель главной ценности/стартующий процесс. Обслуживающие агенты — ниже, сверху вниз (каждый обслуживает вышестоящего → меньше линий через всю карту). Можно одну дорожку (только опыт) или все видимые. ⚠️ Разрабатываемую ИС **нельзя** ставить отдельной дорожкой — она живёт в аннотациях точек (кроме внешних неизменяемых API).

### 2. `/xpm-points` — ключевые точки актора [СТОП]
Порядок: от начала до конца **или** «разрезом» в интересном месте. Искать точки трёх типов. Имя — существительное. Обязательно фиксировать **вход** и **выход** (для контакта — свои у каждого участника). Добавить negative flows.

### 3. `/xpm-weave` — связывание (ядро) 🔑
Сначала переставить точки left/right по логике следования, **не спеша соединять**. Синхронные точки разных участников — на одну вертикаль (⦙). Затем связать: ⎯ уверенный / ┈ необеспеченный-долгий / ⦙ взаимодействие. Ветвления — ◇ (триггер/событие) или таблицей решений. Принцип: **«все разрывы, какие могут произойти, произойдут»** — не рисовать липовую синхронность.

### 4. `/xpm-annotate` — аннотации точек [СТОП]
Под каждой точкой: `вход (дано)` · `выход (результат)` · `операции` · `барьеры и средство преодоления` · `артефакты/системы` · `каналы`. Здесь появляются ИС, потоки, тракты. Концепт **искусственное/естественное**: «сама по себе система не работает» — для каждого места спросить, есть ли гарантия воспроизводимости и какое искусственное средство её обеспечит.

### 5. `/xpm-validate` — валидация (свежий взгляд)
Прогон по hard-гейтам из каталога типовых ошибок (`resources/hard_gates.md`) + Светофор. 🔴 → возврат на стадию-источник.

### 6. `/xpm-deliver` — рендер + публикация [СТОП]
Рендер через **Excalidraw MCP** (переиспользовать библиотеку элементов XPM автора `xpm-library.excalidrawlib`). Fallback — шаблоны Holst/Figma/Miro автора или кастомный PlantUML. Сухой прогон → «ок» человека → публикация.

## Принципы качества (hard-гейты — `resources/hard_gates.md`)

Прямая калька из **каталога типовых ошибок** метода:

**Структура карты:**
- 🔴 **Липовая синхронность** — точки разных участников как синхронные, хотя идут с задержками/петлями.
- 🔴 **Система на отдельной дорожке** — разрабатываемая ИС как участник (кроме внешних API).

**Содержание точек:**
- 🔴 **Глагол в имени точки** («Авторизуется» → «Аутентификация»).
- 🔴 **Форма решения в имени** («Встреча по…» → «Формирование плана»; «Окно подачи» → «Подача заявки»).
- 🔴 **Канал/подсистема в имени** («…в Личном кабинете» → обобщить, канал в аннотацию).
- 🔴 **Артефакт в имени** («Уведомление о…» → «Активация аккаунта»).
- 🔴 **Точка без входа/выхода** (для точек-изменений).

**Целевые установки:**
- 🔴 **Потребитель как позиция без рефлексивного выхода** (позиция ≠ участник).

**Оформление:** точки завершения; многозначные точки тока; синхронные точки без линий тока.

**Светофор:** 🟢 все гейты пройдены → отгрузка · 🟡 есть `[УТОЧНИТЬ]`/открытые вопросы → отгрузка со списком рисков · 🔴 нарушен инвариантный гейт → возврат.

## Паттерны картирования (`resources/patterns.md`)

- **Старт/течение:** линейный путь с опциональными (◠) · безусловное ветвление/схождение · многовариантный старт от событий/точек · повторные вхождения (предваряющие события).
- **Ветвления:** событиями · триггером ◇ · таблицей решений · цикл.
- **Взаимодействия:** петля «пинг-понг» (посыл→отработка→результат) · синхронизация дорожек · совместные точки.
- **Навигация:** группирующие этапы наверху · сопутствующие события как маркеры.

## Главное правило процесса

**STOP после каждой ключевой стадии** (frame, points, annotate, deliver). Только так конвейер = `bft-writer` по качеству, а не «генерация карты за один промт».
