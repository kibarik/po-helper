# Формат `#summary` — спецификация и стиль

Единый источник правил оформления заметок встреч. Калиброван по правкам PO (эталон — `examples/ideal_summary.md`).

---

## Шаблон файла

```markdown
---
дата: 2026-07-03
источник: <созвон / встреча / транскрибация + id, если есть>
участники: <полные имена, свести с NEXUS/team>
теги: [summary, <домен>, <домен>]
---

#summary
**Тема**: <название темы 1>
**Проблема**: <в чём боль — опционально, убрать строку если темы-проблемы нет>
**Ключевое:**
- <факт / тезис / контекст>
- <мнение с инлайн-атрибуцией — «(Ишманов)» или «Ишманов есть гипотеза …»>
**Договоренности:**
- <кто именно> <что делает> <к какому сроку>
- <принятое решение>

#summary
**Тема**: <название темы 2>
**Ключевое:**
- <...>
**Договоренности:**
- <...>
```

Между темами в одной встрече — новый блок `#summary` (либо `---`-разделитель, если так ведёт PO).

---

## Поля блока

| Поле | Обязательное | Заметки |
|---|---|---|
| `**Тема**:` | да | Формулируется как **решение / KR / действие**, не как описание. «снятие KR 2.1 …», не «Переквалификация задачи поиска». Короткая. |
| `**Проблема**:` | нет | Только если у темы есть явная проблема/боль. Одной фразой. |
| `**Ключевое:**` | да | Телеграфные буллеты — факты, ограничения, риски, гипотезы, мнения. Мнение → инлайн-атрибуция. |
| `**Договоренности:**` | да | Буллеты — действия с ответственным и сроком, либо принятые решения. Конкретно: кто именно что делает. |

---

## Что отличает хорошее саммари (правки PO)

1. **Тема-как-решение.** Не «Переквалификация задачи поиска», а «снятие KR 2.1 "РЕСТОРАНЫ — добавление в полнотекстовый поиск"».
2. **Привязка к KR и к людям/датам решений.** `KR2.1`, `KR2.8`; «согласовано Даня, Настя, Вася на встрече 2 июля».
3. **Атрибуция мнений инлайн-префиксом** прямо в буллете: `Ишманов есть гипотеза, что …`, `мнение (Осташева): …`. Не выносить «мнение (…)» в отдельную структуру.
4. **Доменная точность вместо общих слов:** A/B-эксперимент, content-transfer-api, SEO-оптимизации, слой данных, dataflow, МРС, web_db — а не «фишка», «система», «данные».
5. **Причинно-следственные ветки:** «если хорошо → никаких работ GDS; если отрабатывают опасения → GDS примет БФТ на конкретные участки».
6. **Договорённости конкретны:** кто именно что делает — «команда Кати выведет поиск на витрину», не «доработать поиск».
7. **Краткость.** Телеграфный буллет, а не абзац с тремя придаточными. «Краткость — сестра таланта».

## Обработка фактуры (zero-hallucination)

- **Искажения распознавания** — одну сущность под разными написаниями (Рексис/Рексист/Алексис; ЮМС; web_db) свести к единому написанию. В конце отчёта пометить: «термины сведены — проверь глазами».
- **Дата** — нет в транскрипте (только время) → `currentDate` как заглушка + пометка «поправь по календарю».
- **Участники / KR-коды / сроки** — только из транскрипта или из ответа PO. Нет → не выдумывать, `[УТОЧНИТЬ]`.
- **Автор мнения** — атрибутировать только если в транскрипте видно, кто сказал. Неясно → факт без автора.

## Стиль прозы

Наследуется из `skills/bft-writer/resources/writing_style.md`: прямые сказуемые, конкретика (имена сервисов, KR-коды, даты, %), без AI-стоп-слов («позволяет», «обеспечивает», «представляет собой») и декоративных эмодзи. Статусные пометки `[УТОЧНИТЬ]` — по делу.
