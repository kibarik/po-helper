## Использование

```
/sprint-build <sprint>
```

Вход: `{sprint_output_doc}` (vault-ПЛАН из `/sprint-deliver`) + матрица покрытия из `/sprint-decompose`. Выход: спринт наполнен в JIRA (существующие истории в слоте, NEW созданы, исполнители/статусы выставлены, орфанов 0) + обновлённый `{sprint_output_doc}` с реальными ключами.

## Важно

**Роль: Delivery Builder. Это BUILD-проход — запись в JIRA НЕОБРАТИМА.** PLAN-проход (`/sprint-sync`…`/sprint-deliver`) обратим в vault; здесь начинается материализация. Человек — на каждой развилке, автономия — внутри батча. Каждый батч: **dry-run → approve PO → write → верификация**. Слот **не активировать** — это `/sprint-activate`.

> Разрешено только при `tracker.access: build`. `read-only` → СТОП, остаться в PLAN. Нет MCP → СТОП, поднять MCP/VPN (не искать обход, токены руками не собирать).

## Инструкция для LLM

### Этап 1: Загрузка и предусловия
1. Прочитай `skills/sprint-planner/SKILL.md` (модель двух проходов, принципы reuse-first / ноль орфанов / гигиена).
2. Прочитай `skills/sprint-planner/resources/jira_build.md` (модель связей, порядок записи, верификация) и `resources/build_gates.md` (гейты + антипаттерны).
3. Прочитай `.claude/domain-profile.md` секцию `tracker`: `access`, `mcp`, `projects`, `epic_link_field`, `sprint_field`/`board_id`, `issuetype`, источник назначаемых учёток.
4. Прочитай `{sprint_output_doc}` (vault-ПЛАН) и матрицу покрытия из `{sprint_workspace}/sprint-stories.md` (REUSE/NEW/activity).
5. **Проверь предусловия P1–P4** (`build_gates.md`): `access: build` · MCP поднят · PLAN завершён · метаданные проекта резолвлены. Любое нарушение → СТОП с указанием, куда вернуться.

### Этап 2: Проверка DoR (усиленная)
Прогони по каждой истории Definition of Ready (`hard_gates.md`): образ результата · измеримый критерий демо · валидный исполнитель · заведена под эпиком. Не готова → в PLAN (`/sprint-load`/`/sprint-decompose`), не тащить в BUILD.

### Этап 3: Слот-безопасность (🚪 G2 slot-safety)
1. Через MCP собери слоты спринтов (active/future) **с числом задач в каждом**.
2. Выбери ПУСТОЙ слот (или явно указанный PO). Имя+даты зафиксируй один раз.
3. Слот непустой и PO не подтвердил явно «в этот» → **СТОП**, не репойнтить.

### Этап 4: Сухой прогон плана записи (batched)
Собери обозримый dry-run — что и куда, батчами (не лить разом):
```
СУХОЙ ПРОГОН BUILD — {sprint} · слот: {имя} (пуст) · проект(ы): {...}
Батч 1 — существующие истории → спринт: [ключи]  (каскад подзадач: N — 🚪G3)
Батч 2 — переназначить исполнителей: [ключ → учётка (валидна)]
Батч 3 — снять hold: [ключи → transition]
Батч 4 — создать NEW (reuse-check пройден): [под какой историей, тип подзадачи]
Батч 5 — orphan-группировка через Epic Link: [ключ → эпик (+Relates на umbrella-US)]
── СТОП ── PO: подтвердите батч 1 (или скорректируйте).
```
🚪 **G1 reuse-before-create**: в батче 4 — только то, чего нет активного в беклоге (все `tracker.projects`). 🚪 **G3 cascade-preview**: перед добавлением эпика/родителя — сколько детей подтянется.

### Этап 5: Запись батчами (после «ок» на каждый батч)
Для каждого батча по порядку из `jira_build.md`:
1. Записать через MCP трекера.
2. **Верифицировать чтением** (spot-check): спринт проставлен · Epic Link не пуст · исполнитель валиден · hold снят.
3. Расхождение → поправить до следующего батча, не копить дрейф.
4. Следующий батч — снова dry-run → approve → write → verify.

Помни (`jira_build.md`): подзадача наследует спринт от родителя; существующую задачу **нельзя** сделать подзадачей через REST — группировать через **Epic Link**. Кросс-проектный reuse (напр. TLND↔GDSLV) — помечать флагом источника.

### Этап 6: Orphan-группировка (🚪 G5 zero-orphans)
1. Найди орфанов: не подзадача, `Epic Link` пуст, не эпик.
2. Сведи в измеримые User Story (образ результата + критерий демо).
3. Привяжи: `Epic Link` к эпику (+ link «Relates» на umbrella-US).
4. Проверь: орфанов **0**. Остались → СТОП, показать список PO.

### Этап 7: Обновление ПЛАНа + STOP
1. Впиши в `{sprint_output_doc}` реальные ключи JIRA + ссылки на истории/подзадачи (не «на словах»).
2. Сохрани через `obsidian.vault_write`.
3. **Выведи и ОСТАНОВИСЬ:**
   ```
   BUILD {sprint} записан: слот {имя}
   Истории в спринте: N · NEW создано: K · переназначено: M · hold снято: L
   Орфанов: 0 · Epic Link проставлен: да · верификация батчей: пройдена
   ⚠️ Спринт НЕ активирован. Дальше → /sprint-activate {sprint}
   ── СТОП ── PO: проверьте состав спринта в JIRA.
   ```

## Запреты
1. `access: read-only` или нет MCP → НЕ писать, СТОП.
2. НЕ репойнтить непустой слот без явного «да» PO.
3. НЕ создавать NEW до reuse-check (G1). НЕ гадать issuetype-id — брать из профиля/discovery.
4. НЕ записывать батч без dry-run→approve и без верификации после.
5. НЕ активировать спринт (Start sprint) — это `/sprint-activate`.
6. НЕ конвертировать существующую историю в подзадачу через REST — только Epic Link.
7. Токены руками не собирать, в чат не вставлять — только MCP.
