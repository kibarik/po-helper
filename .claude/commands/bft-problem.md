## Использование

```
/bft-problem <epic_code>
```

Вход: `bft-context-pack.md` (из `/bft-context-gen`). Выход: `problem.md`.

## Важно

**Роль: Problem Analyst.** Ты ставишь диагноз, **НЕ предлагаешь решений**. Запрещено: «нужно добавить колонку», «лучше переписать», «система должна…». Разрешено: описывать текущее поведение (As-Is с якорями), фиксировать разрыв (Gap), идентифицировать затронутые компоненты.

> Аналог sa-helper: `/fnr-new-task` → `task.md`. Решения — задача `/bft-concept`.

**Принцип:** каждый диагноз-факт ← якорь (трекер / wiki / PO / код). Нет якоря → `[УТОЧНИТЬ]`.

## Инструкция для LLM

### Этап 1: Загрузка роли и входа
1. Прочитай `skills/bft-writer/SKILL.md` — раздел принципов (особенно As-Is→Gap→To-Be, доказательная база).
2. Прочитай контекст-пак: `<workspace>/<epic>/bft-context-pack.md`.

### Этап 2: As-Is (текущее поведение)
- Опиши как система работает СЕЙЧАС. Только подтверждённое якорями.
- Ключевые компоненты: `Компонент | Роль | Якорь (задача трекера / wiki / file)`.
- Не додумывай реализацию — если не подтверждено, `[УТОЧНИТЬ]`.

### Этап 3: Gap (разрыв)
- Что не так в As-Is относительно желаемого (из постановки PO / context-pack).
- Конкретные проблемы (не абстрактное «плохо»), каждая с ≥1 доказательством.

### Этап 4: Затронутые компоненты (blast radius)
- Прямое воздействие (что менять), косвенное (что зависит), защищённые (backward compat).

### Этап 5: Открытые вопросы (No Silent Skip)
- Все неясности/блокеры/противоречия — список с «у кого уточнить».

### Этап 6: Сохранение + STOP
- Сохрани `<workspace>/<epic>/problem.md` (разделы: As-Is, Gap, Затронутые компоненты, Открытые вопросы, Якоря).
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Диагноз готов: <workspace>/<epic>/problem.md
  As-Is: <кратко>. Gap: <кратко>. [УТОЧНИТЬ]: N шт.
  ── СТОП ── PO/СА: проверьте диагноз. Дальше — /bft-concept <epic> (варианты решения).
  ```

## Запреты
1. НЕ предлагать решения / архитектуру / «система должна».
2. НЕ оценивать подходы.
3. Только As-Is + Gap + компоненты + вопросы. Каждый факт ← якорь.
