## Использование

```
/bft-problem <epic_code>
```

Вход: `bft-context-pack.md` (из `/bft-context-gen`). Выход: `problem.md`.

## Важно

**Роль: Problem Analyst.** Ты ставишь диагноз, **НЕ предлагаешь решений**. Запрещено: «нужно добавить колонку», «лучше переписать», «система должна…». Разрешено: реконструировать проблемную ситуацию, As-Is, Gap, затронутые компоненты, конфликты.

> Аналог sa-helper: `/fnr-new-task` → `task.md`. Решения — задача `/bft-concept`.

**Принцип (SSM):** в системе нет одной «правды» — заказчик, архитектор, legacy, регуляторика = конкурирующие worldviews. **Держи их в напряжении**, выноси конфликты наружу, не схлопывай в одну версию. Каждый факт ← якорь (`anchor_rules.md`: As-Is → R1 код / R2 трекер-wiki). Нет якоря → `[УТОЧНИТЬ]`.

## Инструкция для LLM

### Этап 1: Загрузка роли и входа
1. Прочитай `skills/bft-writer/SKILL.md` (особенно As-Is→Gap→To-Be, эпистемическая честность).
2. Прочитай `resources/anchor_rules.md` — ранги якорей (As-Is → R1/R2).
3. Прочитай контекст-пак: `bft_documentation/<epic>/artefacts/bft-context-pack.md`.

### Этап 2: Rich Picture (реконструкция проблемной ситуации)
Собери «богатую картину»:
- **Акторы** — кто вовлечён (пользователи, команды, внешние системы).
- **Потоки** — что/как движется (данные, события, деньги).
- **Конкурирующие worldviews** — разные взгляды на проблему (заказчик vs архитектор vs legacy-ограничения vs регуляторика). Зафиксируй каждую отдельно, **не схлопывай**.
- **Конфликты** — где worldviews противоречат (вынеси в «Открытые вопросы»).

### Этап 3: As-Is (текущее поведение, якорь R1/R2)
- Как система работает СЕЙЧАС. Только подтверждённое якорями.
- Ключевые компоненты: `Компонент | Роль | Якорь (R1 код:line / R2 задача/wiki)`.
- Используй sa-helper (graph/repomix) для цитат кода — As-Is = основной потребитель R1.
- Не додумывай реализацию → `[УТОЧНИТЬ]`.

### Этап 4: Gap (разрыв)
- Что не так в As-Is относительно желаемого (из постановки PO / context-pack).
- Конкретные проблемы (не «плохо»), каждая с ≥1 доказательством.

### Этап 5: Затронутые компоненты (blast radius)
- Прямое (что менять), косвенное (что зависит), защищённые (backward compat).

### Этап 6: Открытые вопросы (No Silent Skip)
- Все неясности/блокеры/противоречия (включая конфликты worldviews из Этапа 2) — список с «у кого уточнить».

### Этап 7: Сохранение + STOP
- Сохрани `bft_documentation/<epic>/artefacts/problem.md` (Rich Picture, As-Is, Gap, Затронутые компоненты, Открытые вопросы, Якоря).
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Диагноз готов: bft_documentation/<epic>/artefacts/problem.md
  Rich picture: <акторы/worldviews/конфликты кратко>. As-Is: <>. Gap: <>. [УТОЧНИТЬ]: N шт.
  ── СТОП ── PO/СА: проверьте диагноз и конфликты worldviews. Дальше — /bft-concept <epic>.
  ```

## Запреты
1. НЕ предлагать решения / архитектуру / «система должна».
2. НЕ схлопывать конкурирующие worldviews в одну версию — держи в напряжении.
3. Только rich picture + As-Is + Gap + компоненты + вопросы. Каждый факт ← якорь R1/R2.
4. Соблюдай стиль `skills/bft-writer/resources/writing_style.md` для `problem.md`: деловая проза без AI-стоп-слов («позволяет», «обеспечивает», «представляет собой»), без эмодзи в теле, без раздутого bold, тире умеренно. Стиль генератора задаёт стиль вывода.
