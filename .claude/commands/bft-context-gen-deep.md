---
description: Углублённый контекст-пак для БФТ — параллельные субагенты по всем источникам (JIRA дочерние, Confluence, код, ADR, стейкхолдеры). Как Perplexity, но для корпоративных ресурсов. Полная доказательная база перед /bft-problem.
---

## Использование

```
/bft-context-gen-deep <epic_code> <jira_key>
```

**Параметры:**
- `<epic_code>` — короткий код БФТ (напр. `EPIC-10`).
- `<jira_key>` — эпик JIRA (напр. `PROJ-101`).

## Примеры

```
/bft-context-gen-deep EPIC-10 PROJ-101
/bft-context-gen-deep EPIC-PROC PROJ-103
```

## Важно

**Режим:** углублённый. Запускает параллельные субагенты по каждому источнику — аналог Perplexity для корпоративных ресурсов. Занимает больше времени, но на выходе — максимально полный pack с доказательной базой, минимумом `[УТОЧНИТЬ]` и готовым основанием для `/bft-problem`.

**Ключевое отличие от /bft-context-gen:**
- Дочерние задачи JIRA (полная декомпозиция, статусы, SP, assignee)
- Глубокий поиск Confluence (многократные запросы, смежные страницы, ТЗ, решения)
- Анализ кода/СА через repowise (blast radius, текущие зависимости)
- Параллельное выполнение через субагентов → результат быстрее, чем последовательно
- Кросс-валидация: факты из разных агентов сверяются между собой
- Конфликты фиксируются явно, не схлопываются

## На выходе

```
{bft_workspace}/
├── bft-context-pack.md          ← финальный синтез (читает /bft-problem)
└── raw/
    ├── agent-jira.md            ← отчёт субагента JIRA
    ├── agent-confluence.md      ← отчёт субагента Confluence
    ├── agent-code.md            ← отчёт субагента кода/СА
    ├── agent-decisions.md       ← отчёт субагента ADR/решений
    └── agent-stakeholders.md   ← отчёт субагента стейкхолдеров
```

## Инструкция для LLM

### Этап 0: Подготовка

1. Создай папки `{bft_workspace}/` и `{bft_workspace}/raw/`.
2. Прочитай CORTEX-фон (C1/C3/BR/C5) — это общий контекст для всех агентов.
3. Составь **план исследования**: какие вопросы должен ответить каждый агент.
4. Объяви пользователю: «Запускаю 5 параллельных субагентов...»

### Этап 1: CORTEX — читаем локальный фон

Идентично /bft-context-gen, Этап 1. Результат — выдержки по C1/C3/BR/C5 для передачи агентам.

### Этап 1.6: GROUND NEXUS — слой фактов онбординга (локально)

Идентично /bft-context-gen, Этап 1.5: прочитай `GROUND/NEXUS/_registry.yaml` и релевантные Узлы шести Нексусов БФТ (`problem`, `requester-domain`, `system-landscape`, `ownership`, `precedents`, `compliance` — матрица `sa_documentation/nexus_process_map.md`). Выдержки с якорями `(NEXUS:<slug>)`, CP и `ripeness` передаются агентам как **опорные факты для кросс-валидации**: агент E сверяет стейкхолдеров с `ownership`, агенты C/B — As-Is/границы с `system-landscape`, агент D — с `precedents`. Расхождение факта агента с Нексусом (или наоборот) → в «Разногласия источников» (Этап 3), не схлопывать. Нексус пуст → агенты работают по своим источникам, раздел помечается `[УТОЧНИТЬ: нет в <slug>]`.

### Этап 1.5: Resolve capability (роли → коннекторы)

Перед спавном субагентов A–E определи доступность ролей:

1. Прочитай `role_bindings` из `.claude/domain-profile.md` (роль `id` → MCP-сервер / `builtin`).
   Маппинг агентов на роли: A→`jira`, B→`conf`, C→`code`, D→`conf`+`code`, E→синтез (без своих tools).
2. Роль не привязана / сервер не отвечает → **соответствующего субагента НЕ спавним**; его раздел в pack → `[НЕДОСТУПНО: роль <id>]`. Не выдумывать факты за недоступный источник.
3. Прочитай `source_policy`, класс = `bft-critical`. Посчитай `required ∩ available`:
   - `on_missing_required: block` → **СТОП**: «<роль> недоступна — `bft-critical` требует роли из source_policy.classes.bft-critical (дефолт [jira, conf]). Проверь `.mcp.json`/`role_bindings` или используй `/bft-context-gen` (быстрый).»
   - `on_missing_required: warn` → продолжай, флаг в шапке pack и в матрице покрытия.
4. В матрице покрытия (Этап 4) добавь колонки `required?/available?/used?`.

### Этап 2: Запуск параллельных субагентов

Запускай субагентов **одновременно** (в одном сообщении, не последовательно). Каждый агент получает контекст из CORTEX + свою задачу.

---

#### 🔵 Субагент A — JIRA Deep

**Цель:** полная картина эпика и всего беклога.

**Задачи:**
1. `jira_get_issue(<jira_key>)` — эпик: описание, статус, приоритет, labels, components, sprint.
2. `jira_search("Epic Link" = <jira_key>)` или `"parent" = <jira_key>` — все дочерние задачи.
   - Для каждой: ключ, статус, assignee, SP, summary, блокеры.
3. `jira_search("issueType = Epic AND project = <tracker.projects> AND summary ~ '<ключевые слова из эпика>'")` — смежные эпики того же домена.
4. Проверь linked issues (blocks/is blocked by/relates to).
5. Зафиксируй: незакрытые задачи из прошлого (статус ≠ Done) — кандидаты на перенос.

**Артефакт:** `raw/agent-jira.md`
- Скоуп и декомпозиция
- Таблица дочерних задач (key | status | assignee | SP | summary)
- Смежные эпики того же домена
- Блокеры и зависимые задачи (с ключами)
- Незакрытые задачи

---

#### 🟢 Субагент B — Confluence Deep

**Цель:** найти всю документацию по теме эпика.

**Задачи (несколько поисковых запросов):**
1. `confluence_search("\"<jira_key>\"")` — прямые упоминания эпика.
2. `confluence_search("<ключевые слова из JIRA description>")` — по теме.
3. `confluence_search("space = {wiki.space} AND <домен> AND (БФТ OR ТЗ OR постановка)")` — документация домена.
4. Для каждой найденной страницы: `confluence_get_page(pageId)` → извлечь образ результата, As-Is, решения.
5. Проверь {bft_store}/ и {cortex.sa_store} на смежные документы.

**Что искать в документах:**
- Образ результата (что должно получиться)
- As-Is (как работает сейчас)
- Ключевые решения и открытые вопросы
- Стейкхолдеры из документа
- Упомянутые зависимости

**Артефакт:** `raw/agent-confluence.md`
- Список найденных страниц (pageId | title | релевантность)
- Выдержки по каждому блоку: образ результата, As-Is, ключевые решения
- Конфликты между документами (если нашёл)
- Что не найдено → `[УТОЧНИТЬ у {кого}]`

---

#### 🟡 Субагент C — Код и СА (As-Is Blast Radius)

**Цель:** понять, что в текущем коде/СА затрагивает эпик.

**Задачи:**
1. Если доступен repowise/sa-helper: запроси граф зависимостей по компонентам из C1.
   - `graph_impact(<компонент>)` — что затронется при изменении.
   - `graph_call_chain(<ключевой метод/эндпоинт>)` — цепочка вызовов.
2. Из {cortex.sa_store} — найди системно-аналитические документы по домену.
3. Если кода нет (greenfield) → опирайся только на C1 и {cortex.sa_store}, явно пометь.
4. Зафиксируй: какие сервисы/БД читают/пишут в зоне изменений.

**Артефакт:** `raw/agent-code.md`
- Blast radius (какие компоненты затрагивает эпик)
- Текущие интеграции (сервис → источник/приёмник данных)
- Потенциальные точки риска (shared state, legacy-зависимости)
- Если greenfield: явная пометка, что As-Is из кода не применимо

---

#### 🟠 Субагент D — ADR и PO-решения

**Цель:** собрать все ранее принятые решения по теме.

**Задачи:**
1. `{cortex.decisions}` — все ADR по домену эпика.
2. Из смежных БФТ ({bft_store}/) — блоки «Ключевые решения из открытых вопросов».
3. `confluence_search("ADR AND <домен>")` или `confluence_search("Архитектурное решение AND <тема>")`.
4. Для каждого решения: что решено, кто решил, когда, источник.

**Артефакт:** `raw/agent-decisions.md`
- Реестр ADR по теме (ADR-ID | суть | кто принял | дата | источник)
- PO-решения из смежных БФТ
- Конфликты решений (если ADR противоречат друг другу)
- Устаревшие решения (помечены как superseded)
- Что не решено → `[УТОЧНИТЬ у PO/Архитектора]`

---

#### 🔴 Субагент E — Стейкхолдеры и зависимости

**Цель:** карта людей и межкомандных зависимостей.

**Задачи:**
1. Из `NEXUS:ownership` (владельцы доменов, согласующие, скрытые стейкхолдеры, эскалации) + C1 — владельцы доменов (API-слой/Шина/Процессинг/BI) и их контакты. Нексус `ownership` — приоритетный источник владения; C1 дополняет.
2. Из JIRA assignee/reporter + упоминаний в описании — кто вовлечён.
3. Из Confluence-документов (агент B) — кто фигурирует в ревью и согласованиях.
4. Карта зависимостей: API-слой↔Шина↔Процессинг↔BI↔1С↔Финтех — по теме эпика, статус согласования.
5. Регуляторные владельцы (из C3): юристы/бухгалтерия/СБ по применимым законам.

**Артефакт:** `raw/agent-stakeholders.md`
- Таблица стейкхолдеров (роль | имя | команда | что согласует)
- Карта зависимостей (команда A → команда B | тип зависимости | статус)
- Конкурирующие интересы / потенциальные конфликты
- `[УТОЧНИТЬ]` — кого не удалось найти

---

### Этап 3: Кросс-валидация

После получения результатов всех агентов — **сверь факты между ними**:

| Проверка | Источник 1 | Источник 2 | Статус |
|---|---|---|---|
| As-Is совпадает? | agent-code (C) | agent-confluence (B) | ✅/⚠️ Конфликт |
| Стейкхолдеры из JIRA ↔ Confluence | agent-jira (A) | agent-stakeholders (E) | ✅/⚠️ |
| ADR не противоречат JIRA? | agent-decisions (D) | agent-jira (A) | ✅/⚠️ |
| Blast radius ↔ зависимости | agent-code (C) | agent-stakeholders (E) | ✅/⚠️ |
| Границы кода ↔ ландшафт онбординга | agent-code (C) | NEXUS:system-landscape | ✅/⚠️ |
| Стейкхолдеры ↔ владение доменами | agent-stakeholders (E) | NEXUS:ownership | ✅/⚠️ |

**Конфликты** — выноси явно в раздел «Разногласия источников» финального pack. Не схлопывай в одну версию.

### Этап 4: Матрица покрытия (полная)

| Раздел БФТ | Питающий источник | Агент | Статус |
|---|---|---|---|
| Домен и контекст | JIRA + C1 + NEXUS:requester-domain | A + CORTEX | ✅/[УТОЧНИТЬ] |
| Проблема (не симптом) | NEXUS:problem + JIRA | CORTEX + A | ✅/[УТОЧНИТЬ] |
| As-Is (код) | repowise + {cortex.sa_store} + NEXUS:system-landscape | C | ✅/[УТОЧНИТЬ] |
| As-Is (документация) | Confluence | B | ✅/[УТОЧНИТЬ] |
| Стейкхолдеры | JIRA + Confluence + C1 + NEXUS:ownership | A + B + E | ✅/[УТОЧНИТЬ] |
| БТ (ценность) | JIRA + Confluence ТЗ | A + B | ✅/[УТОЧНИТЬ] |
| ПТ (story) | Confluence + ADR | B + D | ✅/[УТОЧНИТЬ] |
| ФТ (поведение) | ADR + Confluence | D + B | ✅/[УТОЧНИТЬ] |

### Этап 5: Сборка финального pack

`{bft_workspace}/bft-context-pack.md`:

1. **Шапка:** epic, jira_key, дата, `Режим: DEEP (bft-context-gen-deep)`, статус всех 5 агентов.
2. **CORTEX-выдержки** (C1/C3/BR/C5) + **NEXUS-факты** (problem/system-landscape/ownership/requester-domain/precedents/compliance) с якорями `(NEXUS:<slug>)`.
3. **Синтез по блокам** (проблема / домен / As-Is / стейкхолдеры / зависимости / регуляторика) — факты с якорями.
4. **Разногласия источников** — конфликты, требующие решения PO.
5. **Матрица покрытия** (из Этапа 4).
6. **Требует уточнения** — сводный список всех `[УТОЧНИТЬ]` с указанием: что, у кого, как проверить.
7. **Ссылки на raw-артефакты** (для детального изучения).

## Жёсткие правила

1. **Параллельность обязательна.** Агенты A–E запускаются одновременно, не последовательно.
2. **Raw-артефакты сохраняются.** `/raw/*.md` — для аудита и повторного использования.
3. **Конфликты не схлопываются.** Разногласие между источниками → явный раздел в pack.
4. **Нулевой допуск к галлюцинациям.** Каждый факт ← JIRA-key / Confluence-pageId / ADR-ID / PO-решение+кто.
5. **Честно про недоступность.** VPN/MCP — явно помечать, что агент не мог достать источник.
6. **Стиль pack и raw-отчётов** — по `skills/bft-writer/resources/writing_style.md`. Pack/raw — отчёты пайплайна (не БФТ): деловая проза без AI-стоп-слов; эмодзи только семантические (цветовые маркеры субагентов в легенде, статус `✅`/`⚠️` в матрице и кросс-валидации); декоративных эмодзи в шапке режима не ставить — режим текстом.
