## Использование

```
/okr-validate <quarter>
```

Вход: `enriched-okr.md` (из `/okr-enrich`). Выход: `validation.md` (Светофор + детальный отчёт).

## Важно

**Роль: Validator — свежий взгляд.** Не «соавтор» OKR, а аудитор. Проверяй строго по гейтам из `resources/hard_gates.md`. Цель — не найти повод для 🔴, а честно зафиксировать состояние.

> Аналог bft-pipeline: `/bft-validate`. Без 🟢/🟡 вердикта `/okr-deliver` не запускается.

## Инструкция для LLM

### Этап 1: Загрузка контекста
1. Прочитай `skills/okr-planner/resources/hard_gates.md` — 13 гейтов, Светофор.
2. Прочитай `{okr_workspace}/enriched-okr.md` — финальная таблица.
3. Прочитай `{okr_workspace}/context-pack.md` — для проверки гейта 6 (якоря).
4. Прочитай `{landscape.snapshot_doc}` (`landscape-{quarter}.md`) — для гейта 13 (внешние зависимости заземлены на ландшафт). Файла нет → гейт 13 = 🟡.

### Этап 2: Проверка каждого гейта

Проходи гейты последовательно (1→13). Для каждого:

```
ГЕЙТ N: [Название]
Статус: ✅ / 🔴 / 🟡
Обнаружено: [конкретное нарушение или OK]
Затронуто KR: [список если нарушение]
```

Гейты 1–6 → 🔴 при нарушении (блокируют deliver).
Гейты 7–12 → 🟡 при нарушении (допустимо с явным [УТОЧНИТЬ] и подтверждением PO).
Гейт 13 → 🔴 если снимок ландшафта есть, но команда не заземлена; 🟡 если снимка нет вовсе; ✅ если внешних зависимостей нет.

### Этап 3: Итоговый Светофор

```markdown
## Итого: Светофор OKR {quarter}

| Гейт | Название | Статус |
|------|---------|--------|
| 1 | OBJ count 3–5 | ✅ / 🔴 |
| 2 | Каждый OBJ: Проблема + Эффект | ✅ / 🔴 |
| 3 | Каждый KR имеет IMP | ✅ / 🔴 |
| 4 | Образ результата IMP≥7 без TBD | ✅ / 🔴 |
| 5 | Технические KR: БЫЛО→СТАЛО | ✅ / 🔴 |
| 6 | Факты с якорями | ✅ / 🔴 |
| 7 | KR per OBJ: 2–6 | ✅ / 🟡 |
| 8 | Образ действия IMP≥7: ≥3 шагов | ✅ / 🟡 |
| 9 | Зависимости явные | ✅ / 🟡 |
| 10 | Critical KR → дедлайн | ✅ / 🟡 |
| 11 | [RESEARCH] → конкретный образ | ✅ / 🟡 |
| 12 | Итого KR ≤ 20 | ✅ / 🟡 |
| 13 | Внешние зависимости ← ландшафт | ✅ / 🔴 / 🟡 |

### Финальный статус: 🟢 / 🟡 / 🔴

**Действие:**
- 🟢 → `/okr-deliver {quarter}`
- 🟡 → `/okr-deliver {quarter}` (PO принимает риск по 🟡-гейтам)
- 🔴 → вернуться на `/okr-enrich {quarter}` или `/okr-key-results {quarter}`
```

### Этап 4: Детальный отчёт нарушений

Для каждого нарушенного гейта — детализировать:
```markdown
### 🔴 Гейт N: [Название]
**Нарушение:** [конкретно]
**Затронутые KR:** KR X.N, KR Y.M
**Что исправить:** [конкретная инструкция]
**Куда вернуться:** `/okr-enrich` / `/okr-key-results`
```

### Этап 5: Сохранение + STOP
- Сохрани через `obsidian.vault_write` в `{okr_workspace}/validation.md`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Валидация завершена: {okr_workspace}/validation.md
  Статус: 🟢/🟡/🔴
  Гейты 🔴: [список]. Гейты 🟡: [список].
  ── СТОП ──
  🟢/🟡 → /okr-deliver {quarter}
  🔴 → исправить [список KR], затем /okr-validate {quarter} повторно
  ```

## Запреты
1. НЕ смягчай нарушения — называй 🔴 красным.
2. НЕ пропускай гейт «потому что очевидно всё OK» — каждый гейт проверить явно.
3. Статус «🟡» — не для критичных гейтов (1-6). Для них только ✅ или 🔴.
