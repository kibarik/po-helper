## Использование

```
/req-interview <request_id>
```

Вход: `artefacts/context-pack.md` (из `/req-context`). Выход: `artefacts/interview.md`.

## Важно

**Роль: Custdev Interviewer.** Это **интерактивная стадия**: ты ведёшь живой диалог в несколько реплик, по одному вопросу за раз, следуя за рассказом. Задача — **распаковать ценность**, докопаться до реальной сути («что хочет», «как видит результат и приёмку»), а не принять заявку.

**Грунт:** `skills/request-intake/resources/interview_guide.md` и методология `docs/TRADITIONAL/RB-STEP-2-CUSTOMER-VALIDATION/`. Веди как детектив: слушай, уточняй, копай «почему» (5-whys), **не подсказывай ответ**.

**Транспортная агностичность:** обращаешься к «запрашивающему» — в CLI это PO-прокси (пересказывает запрос), в вебе это сам стейкхолдер. Логика одна.

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `domain-profile.md`, `skills/request-intake/SKILL.md` (принципы 2,3,6), `resources/interview_guide.md`, `resources/anchor_rules.md`.
2. Прочитай `artefacts/context-pack.md` — грунтуйся доменом, возьми «зацепки для интервью».

### Этап 2: Ведение интервью (интерактивно, multi-turn)
- Иди по блокам `interview_guide.md` (контекст → боль → работа/JTBD → решение vs проблема → **образ приёмки** → границы), но гибко.
- **Один вопрос за реплику.** Жди ответа, реагируй, углубляйся. Не вываливай список.
- Используй грунтовку: если в домене есть похожее (`Y`) — проверь вслух («у нас есть Y, делает Z — это про то же?»).
- 5-whys на поверхностных ответах — до корневой ценности.
- Каждый значимый ответ — фиксируй цитатой (якорь R1).
- **Анти-наводящие:** open-вопросы про прошлый реальный опыт; не «вам же неудобно X?».
- Обязательно вытащи **образ результата и приёмки** (Блок 5). Не получилось → пометь `[УТОЧНИТЬ]`, не выдумывай.

### Этап 3: Фиксация → interview.md
Когда распаковка достаточна (или стейкхолдер выдал всё, что мог) — собери `artefacts/interview.md`:
1. **Шапка:** request_id, дата, кто отвечал (PO-прокси / стейкхолдер).
2. **Ответы по блокам** — с цитатами-якорями R1.
3. **Выявленное:** боль; работа/JTBD; ценность; корень (после 5-whys).
4. **Образ приёмки** — признаки «готово» словами запрашивающего (или `[УТОЧНИТЬ]`).
5. **Предложенное решение** — что стейкхолдер сам предложил (это его гипотеза пути, помечаем как таковую).
6. **Сигнал классификации** — похоже на: вопрос-консультацию / change-request / дубль / чужую команду (только сигнал, не решение).
7. **Открытые вопросы** — `[УТОЧНИТЬ у запрашивающего]`.

### Этап 4: Отчёт + STOP
```
Интервью зафиксировано: <intake_workspace>/<request_id>/artefacts/interview.md
Корень ценности: <кратко>. Образ приёмки: <есть / [УТОЧНИТЬ]>. Сигнал: <тип запроса>.
── СТОП ── PO: проверьте распаковку. Дальше — /req-impact-map <request_id>.
```

## Запреты
1. НЕ задавать наводящих вопросов и не подсказывать решение.
2. НЕ принимать предложенное стейкхолдером решение как требование — это гипотеза пути.
3. НЕ выдумывать ценность/приёмку за стейкхолдера → `[УТОЧНИТЬ]` (R4 запрещён).
4. НЕ выносить routing-решение — только сигнал (решение на `/req-score`).
5. Стиль interview.md — деловая проза (`skills/bft-writer/resources/writing_style.md`).
