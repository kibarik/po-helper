## Использование

```
/bft-validate <epic_code>
```

Вход: финальный БФТ `bft_documentation/<epic>/<epic>.md` (из `/bft-draft`). Выход: отчёт `artefacts/validation.md` + правки БФТ.

## Важно

**Роль: Validator (свежий взгляд).** Это **отдельный запуск** — критический проход по чужому черновику, не «самопроверка» автора. Прогон 18 бинарных 🔴-гейтов + чек-лист + отчёт «Светофор».

> Аналог sa-helper: `/validate-doc`. При 🔴 — вернуть на `/bft-draft`.

**Принцип:** проверяй против источников (якорей), не «верь на слово» черновику.

## Инструкция для LLM

### Этап 1: Загрузка ролей и чек-листов
1. Прочитай `skills/bft-writer/SKILL.md` (принципы — это критерии).
2. Гейты: `resources/hard_gates.md`. Стандарты: `resources/bft_standards.md`. Стиль: `resources/writing_style.md`. Шаблон: `examples/ideal_bft.md`.
3. **`resources/review_feedback.md` — реестр замечаний ревью.** Все строки `активно` = проверяемые правила (уроки прошлых БФТ).

### Этап 2: 4 итерации рефлексии (разные линзы)

**Итерация 1 — Структура и трассировка (Gateway):**
- **Оглавление** в начале (ЗМ-006, гейт 16). Все разделы **в каноническом lean-порядке** (ЗМ-005: Оглавление → Продуктовое обоснование → О задаче [2.1 As-Is→Gap→To-Be / 2.2 Контекст / 2.3 Стейкхолдеры] → План демо → Критерии успеха [4.0 scope-exclusions / 4.1–4.5 БТ-НФТ] → Риски и зависимости → Открытые вопросы); идентификаторы локальные `{TYPE}-{N}` (без EPIC, ЗМ-003) — единый шаблон в «Идентификатор» и «Связанные требования»; frontmatter; scope-exclusions 4.0 (Не входит/Вынесено в СА + высота БФТ, гейт 4).
- Нет пустых «Связанные требования»; каскад БТ←ПТ←ИТ и БТ←ФТ←НФТ; эпик-якорь.
- **Продуктовое обоснование (ЗМ-010, гейт 18):** если раздел заполнен — метрика верификации единственная (база → цель, источник данных); A/B-дизайн заполнен; МДО определён и ссылается на требования; порог засчёта KR сформулирован; гипотеза/метрика/МДО не заякорены на код (это To-Be — R2/R3) → нарушение = 🔴. Если раздел пуст (стадия `/bft-value` пропущена) — это **не 🔴, а 🟡 «дурной тон»**: зафиксируй, что продуктовая ценность не обоснована, порекомендуй пройти `/bft-value`, но документ не блокируй по этому пункту.

**Итерация 2 — Качество требований (Storage):**
- БТ и каждый ФТ имеют приоритет. НФТ только измеримые (число/%/RPS/срок) — оценочные = 🔴.
- ПТ в формате Когда/Я хочу/Чтобы. ФТ ≈ атомарные.
- Критерии успеха оцифрованы отдельным разделом.

**Итерация 3 — План демо и высота БФТ (Human-Readable + Altitude):**
- «План демо для стейкхолдера»: все 3 блока таблицами (3.1 Ключевой / 3.2 ЧТО ЕСЛИ / 3.3 Недопустимые) — отсутствие блока или проза = 🔴 (гейт 17, ЗМ-008).
- PlantUML — actor-level / black-box, с ≥1 веткой `alt` (negative flow) + `note` для недопустимых. Вызовы между сервисами, имена сообщений/API = 🔴 (гейт 14). Mermaid = 🔴.
- Таблица акторов. Надписи на языке документа, тех.имена в скобках.
- Атрибутивный состав сообщений отсутствует (он — зона СА). Эндпоинты/REST/схемы/БД-поля/код в теле = 🔴 (гейт 14).

**Итерация 4 — Источники и неизвестное (Hallucination):**
- Раздел «Открытые вопросы» есть; закрытые → «Ключевые решения».
- Якорение inline: каждый факт → источник в тексте / «Связанные требования» / As-Is. Отдельный раздел «Якоря истины» опционален (подвал). Неподтверждённое → `[УТОЧНИТЬ]`.
- Ни одного факта без якоря и без `[УТОЧНИТЬ]` (Hallucination Gate).

**Итерация 5 — Стиль и голос (`writing_style.md`):**
- В теле нет эмодзи (`⚠️ ✅ ❌ ⚡ …`). Статусные значки только в самом `validation.md`.
- Нет callout-врезок `> **⚠️ …`. Скоуп-предположения живут в прозе или «Открытых вопросах».
- В ячейках таблиц БТ/ИТ/ФТ/НФТ нет `**bold**` (кроме story-ключей ПТ).
- Em-dash — для определений и перечислений, не как связка; не больше одного на предложение.
- Нет стоп-слов («позволяет», «обеспечивает», «осуществляет», «представляет собой», «важно отметить»).
- `[УТОЧНИТЬ]` без кросс-рефов `[УТОЧНИТЬ Ox]`; нет дубля «верхняя + нижняя» таблицы вопросов.
- Frontmatter без diff-narrative (`supersedes …исправлено`, многословный `status`).

**Итерация 6 — Замечания ревью (Known-Mistake):**
- Прогон по всем `активно`-строкам `resources/review_feedback.md`.
- Любое нарушенное правило реестра = 🔴 (гейт 15). В отчёте укажи ID замечания (`ЗМ-NNN`).

### Этап 3: Hard Gates (18 бинарных 🔴)
Прогон по таблице из `hard_gates.md` (гейты 1–18, включая гейт 13 «Стиль/Голос», гейт 14 «Высота БФТ», гейт 15 «Known-Mistake», гейт 16 «TOC», гейт 17 «Demo-completeness», гейт 18 «Продуктовое обоснование»). Любой 🔴 = документ не готов.

### Этап 4: Отчёт «Светофор»
Таблица по слоям (Оглавление / Продуктовое обоснование / Структура / Границы (scope-exclusions) / Высота БФТ / Трассировка / НФТ / План демо (3 блока) / Критерии успеха / Открытые вопросы / Риски и зависимости / Стиль и голос / Замечания ревью / Шаблон) → 🟢/🟡/🔴 с доказательством и «что исправить».

### Этап 5: Маршрут
- Сохрани `bft_documentation/<epic>/artefacts/validation.md`.
- **Если есть 🔴:** внеси правки в БФТ ИЛИ выведи:
  ```
  🔴 Найдены нарушения Hard Gates: <список>.
  → /bft-draft <epic> (исправить и пересдать). БФТ не финализирован.
  ```
- **Если 🟢/🟡 только — выведи РОВНО это (финал пайплайна), ничего сверх:**
  ```
  Файл БФТ: bft_documentation/<epic>/<epic>.md

  Артефакты — bft_documentation/<epic>/artefacts/:
  | Файл | Стадия |
  |---|---|
  | value.md | продуктовая ценность (зачем инвестировать) |
  | bft-context-pack.md | контекст |
  | problem.md | диагноз As-Is/Gap |
  | concept.md | концепты + дебаты + вердикт |
  | validation.md | hard gates + Светофор |

  Валидация: <светофор> (<N> блокирующих 🔴, <гейты>/18).
  Перед разработкой закрыть (если есть 🟡): <одной строкой>.
  ```
- **Только эти блоки.** НЕ добавляй: пересказ покрытия end-to-end, список вшитых решений (PD-*), «следующий логичный шаг» с предложением другого скила, похвалу. Главное — путь к файлу БФТ и таблица артефактов. Если папка эпика содержит файлы сверх перечисленных, добавь их строками в таблицу; выдуманные строки не добавляй.

## Запреты
1. Не «сглаживай» 🔴 — если гейт нарушен, фиксируй честно.
2. Не переписывай черновик целиком — точечные правки или возврат на /bft-draft.
3. Проверяй против якорей, не против самого черновика.
4. Не финализируй БФТ при 🔴 — готовым к ревью считается только верифицированное (🟢/🟡). Непроверенное за готовое не выдаётся (принцип 15).
