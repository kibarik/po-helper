## Использование

```
/bft-validate <epic_code>
```

Вход: черновик БФТ (из `/bft-draft`). Выход: отчёт `validation.md` + правки черновика.

## Важно

**Роль: Validator (свежий взгляд).** Это **отдельный запуск** — критический проход по чужому черновику, не «самопроверка» автора. Прогон 13 бинарных 🔴-гейтов + чек-лист + отчёт «Светофор».

> Аналог sa-helper: `/validate-doc`. При 🔴 — вернуть на `/bft-draft`.

**Принцип:** проверяй против источников (якорей), не «верь на слово» черновику.

## Инструкция для LLM

### Этап 1: Загрузка ролей и чек-листов
1. Прочитай `skills/bft-writer/SKILL.md` (принципы — это критерии).
2. Гейты: `resources/hard_gates.md`. Стандарты: `resources/bft_standards.md`. Шаблон: `examples/ideal_bft.md`.
3. **`resources/review_feedback.md` — реестр замечаний ревью.** Все строки `активно` = проверяемые правила (уроки прошлых БФТ).

### Этап 2: 4 итерации рефлексии (разные линзы)

**Итерация 1 — Структура и трассировка (Gateway):**
- Все разделы по `ideal_bft.md`; идентификаторы локальные `{TYPE}-{N}` (без EPIC, ЗМ-003) — единый шаблон в «Идентификатор» и «Связанные требования»; frontmatter; Границы системы (Входит/Не входит).
- Нет пустых «Связанные требования»; каскад БТ←ПТ←ИТ и БТ←ФТ←НФТ; эпик-якорь.

**Итерация 2 — Качество требований (Storage):**
- БТ и каждый ФТ имеют приоритет. НФТ только измеримые (число/%/RPS/срок) — оценочные = 🔴.
- ПТ в формате Когда/Я хочу/Чтобы. ФТ ≈ атомарные.
- Критерии успеха оцифрованы отдельным разделом.

**Итерация 3 — Сценарии и визуализация (Human-Readable):**
- PlantUML с ≥1 веткой `alt` (negative flow). Mermaid = 🔴.
- Таблица акторов. Атрибутивный состав. Надписи на языке документа, тех.имена в скобках.

**Итерация 4 — Источники и неизвестное (Hallucination):**
- Раздел «Открытые вопросы» есть; закрытые → «Ключевые решения».
- Якоря истины заполнены (факт → источник). Неподтверждённое → `[УТОЧНИТЬ]`.
- Ни одного факта без якоря и без `[УТОЧНИТЬ]` (Hallucination Gate).

**Итерация 5 — Замечания ревью (Known-Mistake):**
- Прогон по всем `активно`-строкам `resources/review_feedback.md`.
- Любое нарушенное правило реестра = 🔴 (гейт 13). В отчёте укажи ID замечания (`ЗМ-NNN`).

### Этап 3: Hard Gates (13 бинарных 🔴)
Прогон по таблице из `hard_gates.md`. Любой 🔴 = документ не готов.

### Этап 4: Отчёт «Светофор»
Таблица по слоям (Структура / Границы / Трассировка / НФТ / Negative flows / Критерии успеха / Открытые вопросы/Якоря / Риски-зависимости / Шаблон) → 🟢/🟡/🔴 с доказательством и «что исправить».

### Этап 5: Маршрут + Promotion
- Сохрани `po-helper/БФТ/<EPIC>/validation/validation.md`.
- **Если есть 🔴:** артефакт **остаётся в песочнице** `po-helper` (в основное хранилище НЕ переносить). Внеси правки в черновик ИЛИ выведи:
  ```
  🔴 Найдены нарушения Hard Gates: <список>.
  → /bft-draft <epic> (исправить и пересдать). Остаётся в po-helper.
  ```
- **Если 🟢/🟡 только → Promotion (принцип 15):** перенеси финальный БФТ из песочницы в основное хранилище проекта:
  ```
  Promotion: po-helper/БФТ/<EPIC>/draft/draft.md → <основное-хранилище>/БФТ/<EPIC>.md
  ```
  Промежуточные артефакты (context/problem/concept/validation) НЕ переносить. Затем выведи:
  ```
  ✅ БФТ <epic> прошёл валидацию. Светофор: <итог>.
  Перенесён в основное хранилище: <путь>. Готов к ревью: Продакт / Архитектор / Аналитик / Разработка / Тестирование.
  ```

## Запреты
1. Не «сглаживай» 🔴 — если гейт нарушен, фиксируй честно.
2. Не переписывай черновик целиком — точечные правки или возврат на /bft-draft.
3. Проверяй против якорей, не против самого черновика.
4. Не переноси в основное хранилище при 🔴 — только верифицированное (🟢/🟡). Непроверенное остаётся в `po-helper` (принцип 15).
