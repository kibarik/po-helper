## Использование

```
/sprint-roadmap <quarter>
```

Пример: `/sprint-roadmap 2026Q3`

Вход: опубликованный OKR квартала. Выход: `{sprint_roadmap_doc}` — матрица распределения срезов KR по спринтам.

## Важно

**Роль: Release Planner.** Превратить квартальные OKR в верхнеуровневый план по спринтам: какой KR в каком спринте двигается. Это rolling-артефакт (перезапуск/патч при смене видения) и вход для детального планирования `/sprint-sync`.

> Строится один раз после `/okr-deliver`. Не путать с детальным планом спринта — здесь только грубое распределение, без историй/исполнителей.

## Инструкция для LLM

### Этап 1: Загрузка роли и стандартов
1. Прочитай `skills/sprint-planner/SKILL.md`.
2. Прочитай `skills/sprint-planner/resources/sprint_standards.md` (раздел «Матрица SPRINT-ROADMAP»).
3. Прочитай `.claude/domain-profile.md` — резолвинг путей, `cadence` (число спринтов, формат), `capacity`.

### Этап 2: Чтение входов (параллельно)
4. Прочитай `{okr_output_doc}` — OBJ/KR квартала, IMP, образы результата.
5. Прочитай `{index_doc}` и `{kr_epic_map_doc}` — приоритизация KR и привязки эпиков.
6. Определи ёмкость команды/спринт: ростер из `GROUND/NEXUS/team/*` + `capacity` профиля. Нет данных → `[УТОЧНИТЬ]`.
7. Если доступен ФАКТ прошлого квартала (`{execution_root}` предыдущего квартала) — учти незакрытые KR как кандидатов на ранние спринты.

### Этап 3: Опрос PO (по одному вопросу за раз)
**Q1 (порядок):** «В каком порядке двигаем OBJ/KR по кварталу? Что критично сделать раньше (коммитменты, зависимости)?»
**Q2 (зависимости):** «Есть ли жёсткие последовательности — KR X должен закрыться до старта KR Y?»
**Q3 (вехи):** «Какие вехи/демо/релизы зафиксированы на конкретные спринты?»
**Q4 (ёмкость):** «Подтверди ёмкость команды на спринт ({оценка}) и крупные отпуска по кварталу.»

PO может ответить «достаточно» — работаем с собранным.

### Этап 4: Сборка матрицы
Для каждого KR определи срезы и разложи по спринтам S1..Sn (формат — `sprint_standards.md` → «Матрица SPRINT-ROADMAP»):
- Срез в ячейке = краткое действие + грубый SP.
- Не заполнять спринты KR-историями под 100% ёмкости — оставить ~25% резерва на внеплановые (техдолг/баги/влёты).
- Учесть IMP: Critical/High KR — в ранние спринты.
- Каждый KR ссылается только на существующие KR из OKR. Не выдумывать новые.

### Этап 5: Сохранение + STOP
- Сохрани через `obsidian.vault_write` в `{sprint_roadmap_doc}`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  SPRINT-ROADMAP {quarter} готов: {sprint_roadmap_doc}
  KR распределены: N · Спринтов: M · Средняя загрузка: X% ёмкости.
  Запас на внеплановые: ~Y SP/спринт.
  [УТОЧНИТЬ]: {перечень}.
  ── СТОП ── PO: проверьте распределение и вехи.
  Дальше → /sprint-sync {first-sprint} (детальный план первого спринта)
  ```

## Запреты
1. НЕ детализируй истории/исполнителей/точные SP — это уровень B (`/sprint-decompose`, `/sprint-load`).
2. НЕ выдумывай KR — только существующие из `{okr_output_doc}`.
3. НЕ заполняй спринты под 100% — оставляй резерв на внеплановые.
4. Ёмкость/отпуска неизвестны → `[УТОЧНИТЬ]`, не выдумывать цифры.
