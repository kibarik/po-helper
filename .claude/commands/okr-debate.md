## Использование

```
/okr-debate <quarter>
```

Вход: `key-results.md` (из `/okr-key-results`). Выход: вердикт + обновлённый `key-results.md`.

## Важно

**Роль: Devil's Advocate + Architect.** Не соглашайся. Ищи слабые места: нереалистичные IMP, неизмеримые образы результата, скрытые зависимости, риск scope-creep. Каждое замечание — конкретное, с аргументом.

> Аналог bft-pipeline: `/bft-debate`. Без прохода дебатов `/okr-validate` выставит 🔴 гейт 9.

## Инструкция для LLM

### Этап 1: Загрузка контекста
1. Прочитай `skills/okr-planner/SKILL.md`, `resources/hard_gates.md`.
2. Прочитай `{okr_workspace}/key-results.md` — все KR с IMP.
3. Прочитай `{okr_workspace}/context-pack.md` — риски и зависимости.
4. **BFT-ярус (если инстанцирован — см. `GROUND/NEXUS/_registry.yaml`):** прочитай `risk` (узлы `risk` — реестр рисков домена с вероятностью/влиянием/митигацией) и `decisions` (узлы `decision` — принятые ограничения). Нет Нексуса → пропусти, риски бери только из context-pack. Adversarial сильнее, когда сверяет KR с зафиксированным реестром, а не выдумывает риски.

### Этап 2: Раунд 1 — SMART-проверка

Для каждого KR:
- **Specific:** Название по формуле? Конкретный продукт/канал/система?
- **Measurable:** Образ результата доказуемый? Можно ли на демо показать «выполнено»?
- **Achievable:** Ресурсы реалистичны за квартал? Не оставить в IMP=9 то, что займёт год?
- **Relevant:** KR действительно закрывает OBJ, или это «заодно» задача?
- **Time-bound:** Для Critical/High — есть дедлайн?

**Формат замечания:**
```
KR [X.N] «[название]»
Проблема: [SMART-нарушение]
Аргумент: [почему это проблема]
Предложение: [конкретное исправление или вопрос PO]
```

### Этап 3: Раунд 2 — зависимости и скрытые риски

Построй граф зависимостей KR. Ищи:

- **Циклические зависимости:** KR A блокирует KR B, KR B блокирует KR A.
- **Неявные блокеры:** KR X требует готовности системы Z, которую делает KR Y с более низким IMP.
- **Внешние зависимости:** требуется согласование от команды/стейкхолдера — не проставлен ли нереалистичный IMP?
- **Scope-creep риски:** KR с TBD в образе действия + High IMP = будущий перенос.
- **Совпадение с реестром рисков (`risk`-Нексус, если есть):** KR затрагивает зону известного риска (`affects` узла) с высоким `impact`, но без митигации в образе действия → флаг. Якорь замечания — `node_id` риска.

Проверь: не перегружен ли один OBJ (>6 KR)? Нет ли дублирования между KR?

**Формат:**
```
⚠️ Зависимость: KR [X.N] неявно блокируется [KR Y.M / внешней командой Z]
Риск: [что произойдёт если блокер не решён]
Рекомендация: [понизить IMP / добавить [BLOCKER] / уточнить у PO]
```

### Этап 4: Раунд 3 — IMP-калибровка

Проверь соответствие IMP реальности:
- IMP = 9, но нет коммитмента → предложи снизить до 7-8.
- IMP = 7-8, но объём неизвестен → предложи тег `[RESEARCH]` и снизить до 3-5.
- IMP = 9 у более 2 KR → выбери какие реально Critical, остальное High.
- IMP-порядок OBJ соответствует стратегическому приоритету?

### Этап 5: Формирование вердикта

```markdown
## Вердикт дебатов: OKR {quarter}

### Статус: Принят / Требует правок / Заблокирован

### Критичные замечания (требуют изменений)
[Замечания из Раунда 1 + 2, которые нарушают гейты 1-6]

### Некритичные замечания (рекомендации)
[Замечания из Раунда 2 + 3, допустимые но улучшаемые]

### Изменения IMP
| KR | Было | Стало | Причина |

### Что нужно уточнить у PO
[Список вопросов]
```

### Этап 6: Применение правок + STOP
- Если PO согласен с правками — обнови `key-results.md` через `obsidian.vault_patch`.
- Сохрани вердикт в конец файла `key-results.md` (секция `## Вердикт дебатов`).
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Дебаты завершены. Статус: [Принят / Требует правок]
  Критичных замечаний: N. Изменений IMP: M.
  ── СТОП ── PO: подтвердите правки или уточните.
  Если правки одобрены → /okr-enrich {quarter}
  Если KR пересматриваются → /okr-key-results {quarter}
  ```

## Запреты
1. НЕ отменяй KR без подтверждения PO — только помечай как проблемный.
2. Замечания без аргумента — не допустимы. Каждое замечание = конкретная проблема + конкретное предложение.
3. Не пиши «всё хорошо» если нашёл нарушения гейтов — это стадия честной критики.
