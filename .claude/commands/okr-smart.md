## Использование

```
/okr-smart <quarter> <kr>
```

Пример: `/okr-smart 2026-Q4 1.2`

Вход: утверждённый OKR (`enriched-okr.md` / `{okr_output_doc}`) + конкретный KR. Выход: `{okr_grooming_root}/step_{n}/artifacts/smart-agreement.md` со статусом `ЗАФИКСИРОВАН`.

## Важно

**Роль: SMART-Anchor Builder.** Цель — быстро зафиксировать **образ результата** одного KR, вокруг которого будет вестись груминг: что именно делаем, какие границы объёма, в каком контексте шёл диалог. Это якорь, без которого `/okr-grooming` не стартует.

> **ГРУМИНГ ВСЕГДА ОПИРАЕТСЯ НА АРТЕФАКТ.** SMART-соглашение — лёгкий якорь, когда полного БФТ ещё нет. Если для KR уже есть финализированный БФТ в `{bft_store}` — он сильнее: достаточно сослаться на него (см. Этап 4).

> Аналог bft-pipeline: «договорённость об образе результата» перед декомпозицией. Это шаг A Фазы 2 — без него `/okr-grooming` запрещён.

## Инструкция для LLM

### Этап 1: Загрузка контекста
1. Прочитай `.claude/domain-profile.md` — резолв `{okr_workspace}`, `{okr_grooming_root}`, `{okr_output_doc}`, `{bft_store}`, `stakeholders`, `po_name`.
2. Прочитай `skills/okr-planner/SKILL.md`, `skills/okr-planner/resources/okr_standards.md` (секция «SMART-соглашение (якорь груминга)»).
3. Прочитай утверждённый OKR: `{okr_workspace}/enriched-okr.md` (или `{okr_output_doc}`). Найди строку KR `{kr}` — образ результата, IMP, теги, БЫЛО→СТАЛО.
4. Если KR `{kr}` не найден → СТОП: «KR {kr} нет в утверждённом OKR {quarter}. Сначала проведи Фазу 1 (/okr-deliver)».
5. **BFT-ярус (если инстанцирован — см. `GROUND/NEXUS/_registry.yaml`):** прочитай `metrics` (узлы `metric`) — для якоря **M (Measurable)** реальным baseline→target с источником. Нет Нексуса → измеримость из OKR/PO.

### Этап 2: Определение номера шага
1. Просмотри `{okr_grooming_root}/` — найди существующие `step_*`.
2. Если для KR `{kr}` уже есть `step_{n}` → переиспользуй его (обновляешь якорь). Иначе `n` = следующий свободный номер.
3. Запомни путь степа: `{okr_grooming_root}/step_{n}/`.

### Этап 3: Опрос PO (один вопрос за раз)
Образ результата из OKR уже есть — здесь уточняем **границы и контекст**, чтобы груминг не расползался. После каждого ответа — следующий вопрос. PO может ответить «достаточно».

**Q1 (границы объёма):** «KR {kr}: что ТОЧНО входит в этот квартал, а что осознанно НЕ входит (выносим за рамки)? Это зафиксирует объём диалога.»

**Q2 (контекст диалога):** «С кем и о чём уже шли разговоры по этому KR — какие команды/системы/стейкхолдеры в орбите? (для разметки [EXT] и вопросов на груминге)»

**Q3 (критерий результата):** «Как мы поймём, что KR закрыт — что конкретно можно показать на демо/в проде в конце квартала?»

**Q4 (дедлайн/коммитмент, если IMP ≥ 7):** «Есть зафиксированный срок или коммитмент перед {кем}? Назови источник.»

### Этап 4: Альтернатива — привязка существующего БФТ
1. Проверь `{bft_store}` на наличие финализированного БФТ, релевантного KR `{kr}` (по теме/эпику).
2. Если найден и PO подтверждает релевантность → `smart-agreement.md` собирается в режиме **ссылки**: образ результата и границы берутся из БФТ, в шапке указывается `Якорь: БФТ {путь}` со статусом `ЗАФИКСИРОВАН (БФТ)`. SMART-разбор можно не дублировать — сослаться на разделы БФТ.
3. Если БФТ нет → собираем полноценное SMART-соглашение (Этап 5).

### Этап 5: Сборка SMART-соглашения
Структура (см. `okr_standards.md` → «SMART-соглашение»):

```markdown
---
kr: "{kr}"
quarter: "{quarter}"
step: {n}
status: ЗАФИКСИРОВАН
anchor_type: SMART | БФТ
bft_ref: "{путь к БФТ, если anchor_type=БФТ}"
---

# SMART-соглашение: KR {kr} — [название KR]

## Образ результата (из OKR)
[Скопировать из утверждённого OKR. Источник: {okr_output_doc}]

## SMART-разбор
- **S (Specific):** [что конкретно делаем]
- **M (Measurable):** [как измеряем «готово» — критерий демо/прод; при наличии `metrics`-Нексуса — `node_id` метрики + baseline→target]
- **A (Achievable):** [реалистичность за квартал, известные ограничения]
- **R (Relevant):** [какой OBJ закрывает, чей коммитмент]
- **T (Time-bound):** [срок/коммитмент + источник, или [УТОЧНИТЬ]]

## Границы объёма
- **ВНУТРИ:** [что входит в квартал]
- **СНАРУЖИ:** [что осознанно вынесено за рамки]

## Контекст диалога
- Команды/системы в орбите: [список, отметить внешние → кандидаты на [EXT]]
- Стейкхолдеры: [кто согласует / кому коммитмент]

## БЫЛО → СТАЛО (для технических KR)
БЫЛО: [dataflow/состояние]
СТАЛО: [целевой dataflow/состояние]

## Якоря
| Утверждение | Источник |
|-------------|----------|
| [образ результата] | {okr_output_doc} |
| [срок/коммитмент] | PO / встреча / JIRA |
```

Незаполнимое без источника → `[УТОЧНИТЬ у {кого}]`, не выдумывать.

### Этап 6: Сохранение + STOP
- Создай папку `{okr_grooming_root}/step_{n}/artifacts/`.
- Сохрани через `obsidian.vault_write` → `{okr_grooming_root}/step_{n}/artifacts/smart-agreement.md`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  SMART-якорь зафиксирован: step_{n}/artifacts/smart-agreement.md
  KR {kr} — [название]. Якорь: SMART | БФТ. Границы зафиксированы.
  [УТОЧНИТЬ]: N пунктов.
  ── СТОП ── PO: проверьте границы и критерий результата.
  Дальше → /okr-grooming {quarter} {kr}
  ```

## Запреты
1. НЕ проводи груминг (US-декомпозицию, вопросы) здесь — это задача `/okr-grooming`.
2. НЕ выдумывай образ результата или границы без источника → `[УТОЧНИТЬ]`.
3. НЕ ставь `status: ЗАФИКСИРОВАН`, если ключевые поля (образ результата, границы, критерий) пусты — тогда статус `ЧЕРНОВИК` и явный список недостающего для PO.
4. БФТ-режим — только при подтверждении PO релевантности БФТ данному KR.
