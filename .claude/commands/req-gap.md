## Использование

```
/req-gap <request_id>
```

Вход: `artefacts/impact-map.md`. Выход: `artefacts/gap.md`.

## Важно

**Роль: Problem→Change Analyst.** Превращаешь ImpactMap в **конкретное продуктовое изменение** через разрыв As-Is / Проблема / To-Be. Краткое и понятное. Высота — результат и приёмка, не реализация (altitude).

As-Is и Проблема — **диагноз** (как сейчас, в чём боль). To-Be — целевое наблюдаемое состояние + **образ приёмки**. Запрещено опускаться в «как реализуем».

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `domain-profile.md`, `skills/request-intake/resources/gap_rules.md`, `resources/anchor_rules.md`.
2. Прочитай `artefacts/impact-map.md`, `artefacts/interview.md`, `artefacts/context-pack.md`.

### Этап 2: Три блока (по gap_rules.md)
- **As-Is** — как процесс/продукт работает сейчас в зоне запроса. Якорь R1 (слова запрашивающего) / R2 (домен). Не додумывать → `[УТОЧНИТЬ]`.
- **Проблема** — конкретный разрыв (наблюдаемая/измеримая боль), цена. Диагноз, не решение. Каждая ← R1.
- **To-Be** — целевое наблюдаемое состояние на языке результата + **образ приёмки** (признаки «готово»). Нет приёмки → `[УТОЧНИТЬ]`, запрос не созрел.

### Этап 3: Краткое продуктовое изменение
Сформулируй **одной-двумя фразами** изменение, понятное без контекста (мост к SMART). Образец — в `gap_rules.md`.

### Этап 4: Сохранение + STOP
- Сохрани `artefacts/gap.md` (As-Is / Проблема / To-Be / Краткое изменение / Открытые вопросы).
```
Разрыв готов: <intake_workspace>/<request_id>/artefacts/gap.md
Изменение: «<1-2 фразы>». Образ приёмки: <есть / [УТОЧНИТЬ]>.
── СТОП ── PO: проверьте разрыв и формулировку изменения. Дальше — /req-smart <request_id>.
```

## Запреты
1. As-Is/Проблема — НЕ предлагать решение («надо добавить X»).
2. To-Be — НЕ опускаться на системный уровень (API/БД/User Stories) — это bft-writer.
3. Образ приёмки в To-Be обязателен или `[УТОЧНИТЬ]`.
4. Краткость итогового изменения. Каждый факт ← якорь. Стиль — `skills/bft-writer/resources/writing_style.md`.
