## Использование

```
/sprint-deliver <sprint>
```

Вход: `sprint-load.md` (+ goal/stories/context). Выход: `{sprint_output_doc}` — финальный план спринта для обсуждения с командой.

## Важно

**Роль: Validator + Deliverer.** Прогнать hard-gates (Светофор), собрать финальный документ в формате эталона + блок N+1, показать сухой прогон, и **только после «ок PO»** записать в vault. Это **конец PLAN-прохода** (обратимо, vault). JIRA/Confluence — не трогать (только чтение); материализация в JIRA — отдельный **BUILD-проход** (`/sprint-build`).

> Валидация встроена в эту стадию (отдельной нет — каденция лёгкая). 🔴 → вернуться на `/sprint-load` или `/sprint-decompose`.
> После vault-ПЛАНа PO решает: остаться в плане или материализовать спринт в JIRA через `/sprint-build` (необратимо, требует `tracker.access: build`).

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `skills/sprint-planner/SKILL.md`, `resources/hard_gates.md`, `resources/sprint_standards.md` (структура финального документа).
2. Прочитай артефакты стадий: `sprint-load.md`, `sprint-goal.md`, `sprint-stories.md`, `sprint-context.md`.
3. Прочитай `{sprint_roadmap_doc}` — для блока N+1 (срез следующего спринта).
4. Прочитай эталон `skills/sprint-planner/examples/ideal_sprint_plan.md` — целевая структура.

### Этап 2: Прогон hard-gates
Прогони чеклист из `hard_gates.md` (8 гейтов). Выведи Светофор:
- 🔴 (нарушен гейт 1–5) → **СТОП**, сообщи PO что исправить и куда вернуться (`/sprint-load` или `/sprint-decompose`). Не продолжать.
- 🟡 (нарушены 6–8) → продолжить с явными `[УТОЧНИТЬ]`, PO примет риск.
- 🟢 → продолжить.

### Этап 3: Сборка финального документа (сухой прогон)
Собери `ПЛАН-{sprint}.md` по структуре `sprint_standards.md` → «Структура финального документа»:
- Шапка (период из `cadence`, ответственный = `po_name`, статус).
- По каждой OBJ: KR-ссылки + **🎯 Образ результата команды (Sprint Goal)** + таблица историй (`ЭПИК|История|Образ результата|Образ действия|Исполнитель|Приоритет|SP`) + контекст/риски.
- Секция «Необходимые внеплановые работы».
- «📊 Capacity по командам» (ёмкость/загрузка/статус).
- «👤 Персональный фокус» (что лично каждый показывает).
- «⚠️ Общие риски».
- «🔭 Спринт N+1 (предварительно)» — образ результата + крупные истории без точного SP (из roadmap).
- «📋 Следующие шаги».

### Этап 4: Сухой прогон → СТОП
```
СУХОЙ ПРОГОН — план спринта {sprint}
Светофор валидации: 🟢/🟡 [если 🟡 — какие гейты]

📄 СОЗДАТЬ: {sprint_output_doc}
  Целей (Sprint Goals): N
  Историй: M (план) + K (внеплановые)
  Загрузка: 🟢 X / 🟡 Y / 🔴 Z
  Блок N+1: есть

⚠️ Не изменяется: JIRA, Confluence (только чтение).

── СТОП ── PO: подтвердите запись или скорректируйте.
Команда: «да, публикуй» / «ок»
```

### Этап 5: Запись (только после «ок» PO)
1. `obsidian.vault_write` → `{sprint_output_doc}`.
2. (опционально, если PO просит) пометить в `{kr_epic_map_doc}` истории без эпика как `TODO`.

### Этап 6: Манифест + финал
```markdown
## Delivery Manifest: ПЛАН {sprint}
**Дата:** {дата} · **Светофор:** 🟢/🟡 · **Ответственный:** {po_name}
### Артефакты
- {sprint_output_doc} — ✅
- {sprint_workspace}/{context,goal,stories,load}.md — ✅
### Следующие шаги
1. Обсудить план с командой (Sprint Planning встреча), уточнить SP `[УТОЧНИТЬ]`.
2. Завести эпики для TBD-историй.
3. Материализовать в JIRA (если нужно) → `/sprint-build {sprint}` (BUILD-проход, необратимо).
4. По завершении спринта — зафиксировать ФАКТ (вход для `/sprint-sync` следующего).
```
```
ПЛАН {sprint} готов: {sprint_output_doc}
── PLAN-проход завершён ──
Материализовать в JIRA → /sprint-build {sprint}   |   Следующий спринт → /sprint-sync {next-sprint}
```

## Запреты
1. НЕ записывать файл до «ок PO» — только сухой прогон.
2. 🔴 на гейтах 1–5 → СТОП, не публиковать.
3. НЕ создавать задачи в JIRA, НЕ публиковать в Confluence — это PLAN-проход (только vault). Запись в JIRA — отдельный BUILD-проход `/sprint-build`.
4. НЕ выдумывать данные для заполнения пустот — `[УТОЧНИТЬ]`.
