## Использование

```
/release-gate <release-id>
```

Вход: сработавший алерт / решение PO пересмотреть обязательство. Выход: новая версия `baseline.md` (vN+1) + decision-запись.

## Важно

**Роль: Re-baseliner.** Когда дрейф пересёк порог или наступил плановый стейдж-гейт — обязательство **пере-фиксируется явно**: режем объём, двигаем дату или осознанно отказываемся. Старый baseline не удаляется (`superseded`) — история обязательств сохраняется как доказательная база.

> Это перенос «осознанного отказа» из fit-points на релиз: рост контекста → новое честное обязательство, а не подкрутка параметров.

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `skills/release-guard/resources/release_standards.md` + `hard_gates.md`.
2. Прочитай активный `baseline.md`, `change-log.md`, `ledger.md`, последний алерт из `{release_alerts_root}/`.

### Этап 2: Диагноз (из ledger + алерта)
Выведи PO: текущий `total_drift`, разбивку demand/supply с атрибуцией, что под угрозой (DoR/дата), `commitment_mode`.

### Этап 3: Решение по commitment_mode (один вопрос за раз)
Предложи действие, соответствующее форме обязательства:
- `date_fixed` → **cut**: какие REQ режем/в defer, чтобы удержать дату? (объём — клапан)
- `scope_fixed` → **move-date**: на сколько двигаем дату? (дата — клапан)
- `both_fixed` / исчерпаны клапаны → **refuse/эскалация**: что эскалируем стейкхолдеру.

Зафиксируй, **кто** (названный стейкхолдер) утвердил решение.

### Этап 4: Новая версия baseline
- Скопируй активный baseline → `version: N+1`, обнови `date`, `status: active`.
- Старую версию пометь `status: superseded` (не удаляй).
- Примени решение: убери/перемести REQ (cut/defer), обнови `fixed_date` (move-date), пересчитай `estimate_baseline`.
- Влитые change requests: их Δ теперь часть нового нуля → в change-log они «закрыты» текущей версией (новые CR пойдут с `baseline_ver = N+1`, дрейф обнуляется от новой версии).
- Обнови CP, при необходимости `commitment_level`.

### Этап 5: Прогон hard gates
Прогони 10 гейтов на новой версии. 🔴 → не фиксируй, верни PO.

### Этап 6: Decision-запись + сброс ledger + STOP
- Сохрани новую версию `baseline.md` (можно вести `baseline-vN.md` в истории + `baseline.md` = активная).
- Допиши в `baseline.md` раздел `## Re-baseline log`: `vN→vN+1 | дата | причина (дрейф/гейт) | решение | утвердил <кто>`.
- Сбрось `ledger.md`: дрейф от новой версии = 0.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Baseline vN+1 зафиксирован: причина <дрейф T сп. / стейдж-гейт>
  Решение: <cut REQ-x / move-date к <дата> / refuse>, утвердил <стейкхолдер>
  Светофор гейтов: 🟢/🟡. Дрейф обнулён от vN+1.
  ── СТОП ── PO: новое обязательство зафиксировано. Уведомить заказчика <автор>.
  ```

## Запреты
1. НЕ пере-baseline без названного утвердившего — решение должно иметь владельца.
2. НЕ удаляй старую версию — `superseded`, история обязательств = твоя защита.
3. НЕ «подкручивай» молча: ре-baseline всегда фиксирует причину и решение явно.
