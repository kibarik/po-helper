## Использование

```
/radar-calibrate <short|long> [run_id] [--lens routable|context|both]
```

- `<short|long>` — длина: `short` = 10 вопросов, `long` = 50. Обязательно.
- `[run_id]` — слаг прогона; по умолчанию `short-01` / `long-01`, при итерации цикла инкременть номер (`short-02`…).
- `[--lens]` — линза калибровки (см. `question_design.md`): `routable` (только структурные поля, строго), `context` (+ `context_holds`, как RAG), `both` (по умолчанию: A/B под routable + Тип C под обеими, ищет routable-gap).

Выход: `<calibration_workspace>/<run_id>/questions.md` (вопросы + LLM-выборка + таблица сверки) + STOP для разметки PO.

## Важно

**Роль: Examiner.** Генерируешь **ситуационные** вопросы из реальных person-узлов и отвечаешь на каждый **только из нексуса** (LLM-выборка): кто + почему (какие поля сработали). Не размечаешь сам — разметку ✅/❌ ставит PO (оракул-реальность). Твоя честность здесь = не подсматривать в реальность и не подгонять вопросы под удобный ответ.

**Два типа вопросов обязательны:** routing («к кому/почему») и scenario («к тебе обратились — что делаешь»). Вопрос — ситуация на бизнес-языке, не лукап по полю.

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `skills/nexus-calibration/SKILL.md`, `resources/question_design.md`, `resources/scoring_rules.md`.
2. Прочитай `.claude/domain-profile.md` (если есть) — резолвь `paths.calibration_workspace` (дефолт `GROUND/NEXUS/team/_calibration/{run_id}`) и название команды.
3. Прочитай все `GROUND/NEXUS/team/*.md`, кроме `_index`/`_template`.
4. Итерация цикла (`run_id` ≠ `*-01`) → прочитай `review.md` прошлой итерации, чтобы нацелить часть вопросов на исправленные узлы (регрессия, см. `calibration_loop.md`).

### Этап 2: Предусловие
- Граф пуст (только placeholder) → СТОП: «People Graph пуст, калибровать нечего. Наполни нексус: /paf-nexus-create или вручную». Не генерируй вопросы по воздуху.
- Узлов мало (<3–4) → предупреди, что прогон вырожденный, но продолжай.

### Этап 3: Генерация вопросов (N = 10 short / 50 long)
По матрице покрытия из `question_design.md` с учётом линзы:
- типы A/B примерно поровну (long routable: ~20 routing + ~30 scenario); при линзе `both` часть слотов — Тип C (short: 1; long: ~7), по одному на узел с непустым `context_holds`;
- покрыть все слои (org/social/expertise) и (для long) каждый наполненный узел минимум 1 вопросом;
- включить сложные кейсы: дизамбигуация (пересечение экспертизы), эскалация по цепочке, кросс-команда, **негативные** (ожидается `[нексус молчит]`), конфликт слоёв;
- формулировка — реальный входящий запрос, без подсказки адресата в тексте.

### Этап 4: LLM-выборка (ответ из нексуса)
Для каждого вопроса зафиксируй:
- **Кто:** `node_id` (+ ФИО); для scenario — упорядоченный список действий/адресатов.
- **Почему:** конкретные поля узлов (`team-x.contact_for "…"`, `team-y.reports_to`). Только нексус — никакой эрудиции/памяти.
- **Линза:** A/B отвечаешь под `routable` (структурные поля). Тип C — под `context` (можно опираться на `context_holds`), затем перепроверь тот же вопрос под `routable`: достаётся ли факт из структурных полей? Нет → пометь **routable-gap** в колонке «routable?».
- Нет покрытия → `[нексус молчит]` (валидный ответ, не выдумывай человека).

### Этап 5: Сборка questions.md + таблица сверки
Создай `<calibration_workspace>/<run_id>/questions.md`:
1. **Шапка:** run_id, длина (N), дата, список узлов прогона, ссылка на review прошлой итерации (если есть).
2. **Таблица:** `# | Вопрос | Тип/слой | Линза | Ответ LLM (кто · почему) | routable? | PO: ✅/❌ + правда` — колонки `routable?` (для Типа C: да/нет = routable-gap) и `PO` пустые под разметку. При линзе `routable` колонки `Линза`/`routable?` можно опустить.
3. Формат — как в `examples/ideal_question_set.md`.

### Этап 6: Отчёт + STOP
```
Прогон <run_id>: <N> вопросов готовы. Файл: <calibration_workspace>/<run_id>/questions.md
Покрытие: routing <a> / scenario <b>; слои org<>/social<>/expertise<>; негативных <n>.
── СТОП ── PO: размечай по каждому вопросу ✅ (нексус ответил как в реальности) / ❌ (расходится).
К каждому ❌ допиши ОДНУ фразу-правду («на самом деле → Сидорова, она ведёт платёжки»).
Гейт: <10/10 | 50/50>. Совпало всё → 🟢 пройдена. Есть ❌ → /radar-review <run_id>.
```

## Запреты
1. НЕ размечать ✅/❌ за PO — оракул реальности только PO (принцип 3 SKILL).
2. НЕ отвечать из эрудиции/памяти — только из полей узлов; нет покрытия → `[нексус молчит]` (принцип 2).
3. НЕ генерировать лукап-вопросы («у кого contact_for = X») — только ситуации (анти-паттерн в `question_design.md`).
4. НЕ подсказывать адресата в тексте вопроса (наводящий вопрос → ложное ✅).
5. НЕ объявлять «калибровка пройдена» без полной разметки PO и гейта `N/N`.
6. Стиль — деловая проза (`skills/bft-writer/resources/writing_style.md`); ✅/❌ только как метки таблицы.
