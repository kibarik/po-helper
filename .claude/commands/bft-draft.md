## Использование

```
/bft-draft <epic_code>
```

Вход: `concept.md` (с вердиктом от `/bft-debate`) + `problem.md` + context-pack. Выход: черновик БФТ.

## Важно

**Роль: Requirements Writer.** Сформируй БФТ по **утверждённому дебатами концепту**. Формулируешь **что** и **зачем**, не **как**. Строго по шаблону `ideal_bft.md`.

**Высота БФТ (главное ограничение):** документ на уровне результата и приёмки — как это выглядит для бизнеса и пользователя при сдаче. Не опускайся на системные требования, API-контракты, схемы сообщений, sequence-вызовы между сервисами, БД-схемы, код. Это зона СА / системных требований (FNR). Цель — точные критерии приёмки и план демонстрации (что / где / почему изменено). Гейт 14.

> Аналог sa-helper: `/fnr-system-requirements`. Adversarial уже в `/bft-debate`; валидация — `/bft-validate`.

**Принципы:**
- Каждое требование ← якорь (`anchor_rules.md`). **As-Is → R1 (код) / R2; To-Be → R2 (ТЗ/задача) / R3 (PO-решение).** Не заякоривай To-Be на код (гейт 12).
- CATWOE из концепта → `W` в формулировку БТ, `O` в Ревью, `E` в НФТ (`resources/catwoe.md`).
- Типы не смешивать: ПТ≠ФТ≠НФТ (гейт 11, таблица в `bft_standards.md`).

## Инструкция для LLM

### Этап 1: Загрузка роли, стандартов, входа
1. `skills/bft-writer/SKILL.md` (принципы).
2. Стандарты: `resources/bft_standards.md` (форматы + различение типов + NFR-реестр). Шаблон: `examples/ideal_bft.md`. Эталон: `examples/golden_bft_example.md`.
3. `resources/anchor_rules.md` (ранги якорей). `resources/catwoe.md`. `resources/writing_style.md` (стиль и голос, калибровка под золотой референс).
4. **`resources/review_feedback.md` — реестр замечаний ревью.** Все строки `активно` = обязательные ограничения генерации (уроки прошлых БФТ). Применяй на всех этапах ниже; при конфликте с базовым стандартом реестр приоритетнее — конфликт вынеси в «Открытые вопросы».
5. Входы (все из `bft_documentation/<epic>/artefacts/`): `concept.md` (концепт + CATWOE + вердикт), `problem.md` (As-Is/Gap), `bft-context-pack.md`.

### Этап 2: Каркас (lean-форма)
- **Порядок разделов фиксирован (ЗМ-005, см. `bft_standards.md`):** Оглавление → О задаче (1.1 As-Is→Gap→To-Be / 1.2 Контекст обращения / 1.3 Стейкхолдеры) → План демо для стейкхолдера → Критерии успеха (3.0 scope-exclusions / 3.1–3.5 БТ-ПТ-ИТ-ФТ-НФТ) → Риски и зависимости → Открытые вопросы → опц. подвал (История/Ревью/Якоря/Словарь).
- **Оглавление** (ЗМ-006, гейт 16): в начале, markdown-список ссылок-якорей на разделы и подпункты.
- **О задаче** (ЗМ-007): краткий вводный раздел, макс. таблиц/буллетов. As-Is → Gap → To-Be — **таблицей, НЕ текстом** (ЗМ-002): срезы `As-Is` / `Gap (проблема)` / `To-Be`, пункты через `<br>` (из problem.md; As-Is с якорями R1/R2). Контекст обращения — 2–4 буллета. Стейкхолдеры — таблица `Роль | Кто | Интерес`.
- **Критерии успеха:** блок 3.0 scope-exclusions (**Не входит / Вынесено в СА**, структурой не абзацем — ЗМ-001, гейт 4) + подпункты 3.1–3.5 БТ/ПТ/ИТ/ФТ/НФТ. Метрика → базовое → цель → срок для успеха. Высота документа: БФТ = приёмка, не системный/API/код (гейт 14).

### Этап 3: Каскад требований (с трассировкой + CATWOE + якоря)
- **БТ** (ценность из CATWOE `W`, приоритет) → **ПТ** (story Когда/Я хочу/Чтобы) → **ИТ** (интерфейс приёмки: что видит и может актор; не API-контракт) → **ФТ** (поведение на уровне результата, приоритет) → **НФТ** (только измеримые; из CATWOE `E` и **корпоративного реестра НФТ**, если есть).
- Идентификаторы **локальные** `{БТ|ПТ|ИТ|ФТ|НФТ}-{N}` (ЗМ-003): без EPIC/проекта/сервиса, только для навигации внутри документа. Один шаблон и в «Идентификатор», и в «Связанные требования».
- КАЖДАЯ строка: «Связанные требования» заполнены вверх + эпик.
- Каждый ФТ ≈ атомарная задача (1-2 дня).
- **Различение типов:** ПТ = что хочет пользователь; ФТ = поведение системы; НФТ = измеримое -ility. Не смешивай.
- **Якорение:** As-Is-факты → R1/R2; To-Be-требования → R2/R3 (не R1).
- **Ссылки:** каждую ссылку на Jira/Confluence — markdown-ссылкой (ЗМ-004): `[JIRA KEY](https://jira.mts.ru/browse/KEY)`, `[Confluence pageId](https://confluence.mts.ru/pages/viewpage.action?pageId=pageId)`. Не голым ключом.

### Этап 4: План демо для стейкхолдера (ЗМ-008, гейт 17)
- Три обязательных блока таблицами: **2.1 Ключевой сценарий** (`Шаг | Действие | Ожидаемый результат`), **2.2 ЧТО ЕСЛИ** (`Ветка | Действие | Ожидаемый результат | Что доказывает`), **2.3 Недопустимые события** (`# | Недопустимое событие | Почему это провал приёмки`).
- **2.4 Диаграмма демо:** PlantUML sequence — только actor-level / black-box: акторы и витрина, без вызовов между сервисами и без имён сообщений/API. `alt`-ветки для «ЧТО ЕСЛИ», `note` для недопустимых. Mermaid запрещён. Таблица акторов (из CATWOE `C`/`A`).
- Атрибутивный состав сообщений НЕ описывать — это зона СА / системных требований (гейт 14).

### Этап 5: Риски и зависимости, открытые вопросы
- **Риски и зависимости** — единый раздел (ЗМ-009): `# | Тип | Описание | Влияние | Митигация / владелец`, Тип ∈ {Зависимость, Риск, Зависимость / риск}. Зависимости — со статусом согласования; риски — с митигацией (из минусов концепта + контраргументов дебатов).
- Открытые вопросы: всё незакрытое → `[УТОЧНИТЬ]`. Закрытые → «Ключевые решения» (с CATWOE `O` = согласующий).

### Этап 6: Inline-якорение + опц. подвал
- **Якорение inline** (гейт 1): каждый факт → источник в тексте / «Связанные требования» / As-Is (R1/R2/R3). Отдельный раздел «Якоря истины» (`Факт | Якорь | Ранг | Тип` по `anchor_rules.md`) — опционален, в подвале.
- Ревью (опц. подвал): CATWOE `O` (owner) → согласующий по каждой роли.
- Все `[УТОЧНИТЬ]` → раздел **Открытые вопросы** в НАЧАЛЕ документа (после Общей информации, ЗМ-005).

### Этап 7: Стиль-проходка (Humanizer)
Перед сборкой прогони чек-лист `resources/writing_style.md` по черновику и почисти:
- **Эмодзи:** убери из тела `⚠️ ✅ ❌ ⚡` и пр. (статусные значки только в `validation.md`). Скоуп-предположения и предупреждения → обычный абзац или строка в «Открытые вопросы», не callout-врезка `> **⚠️ …`.
- **Bold:** убери `**bold**` внутри ячеек таблиц БТ/ИТ/ФТ/НФТ. Оставь только story-ключи в ПТ (`**Когда**/`**Я хочу**/`**Чтобы**`) и редкие смысловые акценты в прозе.
- **Тире:** em-dash оставь для определений и перечислений; связки-заменители замени на запятую/точку/скобки. Не больше одного тире на предложение.
- **Стоп-слова:** «позволяет», «обеспечивает», «осуществляет», «производит», «представляет собой», «важно отметить», «в рамках данного» — замени на прямые формулировки в стиле золотой референс.
- **`[УТОЧНИТЬ]`:** убери кросс-рефы `[УТОЧНИТЬ Ox]` → `[УТОЧНИТЬ]`; если рамка известна, а число нет — «(предварительно)» / «уточняется у X» / «TBD». Убери дубль «верхняя таблица + нижняя таблица» одних и тех же вопросов (оставь «Открытые вопросы» + «Ключевые решения»).
- **Frontmatter:** убери diff-narrative (`supersedes: v0.1 (…исправлено)`, многословный `status`). История правок — в разделе «История изменений» одной строкой на версию.

Стиль генератора задаёт стиль вывода: если приёма нет в золотой референс, его не должно быть в БФТ. Запомни количество правок для STOP-вывода.

### Этап 8: Сборка + STOP
- Frontmatter + H1 `# [БФТ] {EPIC}: {Название}` + все разделы по `ideal_bft.md`.
- **Сохрани финальный БФТ в КОРЕНЬ папки эпика: `bft_documentation/<epic>/<epic>.md`** (не `draft.md`, не в `artefacts/`). Это главный документ, который открывает PO.
- **Выведи РОВНО три блока и ОСТАНОВИСЬ** (формат фиксирован, ничего сверх):
  ```
  БФТ: bft_documentation/<epic>/<epic>.md

  Открытые вопросы (<N>):
  - [У<n>] <вопрос одной строкой>
  - …
  (если N=0 → строка «Открытых вопросов нет».)

  Дальше: /bft-validate <epic>
  ```
- **Только эти три блока.** НЕ добавляй: пересказ «что внутри», счётчики БТ/ПТ/ИТ/ФТ/НФТ, как заякорено, список зафиксированных решений (PD-*), описание sequence-веток, отчёт стиль-проходки, эмодзи, преамбулы, похвалу. Документ для этого и есть — PO откроет файл. Сводка = указатель (ссылка + что требует ответа + один шаг), не дайджест.
- Открытые вопросы — это все `[УТОЧНИТЬ]` из черновика, одной строкой каждый, без обоснований и кросс-ссылок.

## Запреты
1. Не проводи adversarial заново (был в /bft-debate) — включи его выводы.
2. Не делай финальную валидацию (это /bft-validate).
3. Не выдумывай факты — только из концепта/problem/context-pack или `[УТОЧНИТЬ]`.
4. Не заякоривай To-Be на код (R1) — гейт 12.
5. Не смешивай типы требований — гейт 11.
6. Не лезь в код / API / системные требования — БФТ на уровне приёмки (гейт 14). ИТ = интерфейс приёмки, не API-контракт.
7. Не нарушай ни одного `активно`-правила из `resources/review_feedback.md` — гейт 15.
