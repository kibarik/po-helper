## Использование

```
/bft-draft <epic_code>
```

Вход: `concept.md` (с вердиктом от `/bft-debate`) + `problem.md` + context-pack. Выход: черновик БФТ.

## Важно

**Роль: Requirements Writer.** Сформируй БФТ по **утверждённому дебатами концепту**. Формулируешь **что** и **зачем**, не **как** (реализация — разработке). Строго по шаблону `ideal_bft.md`.

> Аналог sa-helper: `/fnr-system-requirements` → BR/FR/NFR. Adversarial уже прошёл в `/bft-debate`; финальная валидация — `/bft-validate`.

**Принцип:** каждое требование ← якорь (трекер / PO / ТЗ / концепт). Нет якоря → `[УТОЧНИТЬ]`.

## Инструкция для LLM

### Этап 1: Загрузка роли, стандартов, входа
1. Прочитай `skills/bft-writer/SKILL.md` (принципы: трассировка, измеримость, границы, negative flows).
2. Стандарты: `resources/bft_standards.md`. Шаблон: `examples/ideal_bft.md`. Эталон: `examples/golden_bft_example.md`.
3. Входы: `concept.md` (выбранный концепт + модификации из вердикта), `problem.md` (As-Is/Gap), context-pack.

### Этап 2: Каркас
- Бизнес описание; Общая информация (проект/PO/отв./эпик); Стейкхолдеры; История изменений; Доп. материалы.
- **Границы системы:** явное ВНУТРИ / СНАРУЖИ.
- **Критерии успеха:** метрика → базовое → цель → срок (отдельным разделом; обязательно для фичей/миграций).
- As-Is → Gap → To-Be (из problem.md).

### Этап 3: Каскад требований (с трассировкой)
- БТ (ценность, приоритет) → ПТ (story Когда/Я хочу/Чтобы) → ИТ (контракты, идемпотентность, auth) → ФТ (поведение, приоритет) → НФТ (только измеримые).
- Идентификаторы `{БТ|ПТ|ИТ|ФТ|НФТ}-{EPIC}-{N}`.
- КАЖДАЯ строка: «Связанные требования» заполнены вверх + эпик.
- Каждый ФТ ≈ атомарная задача (1-2 дня).
- **Опирайся на концепт:** если вердикт внёс модификации (extra negative flow / НФТ) — включи их.

### Этап 4: Диаграммы и атрибутивный состав
- PlantUML sequence: happy path + ВСЕ `alt`-ветки (ошибка/недоступно/отмена/таймаут). Mermaid запрещён.
- Таблица акторов. Атрибутивный состав ключевых сообщений (Параметр | Обяз | Тип | Описание).

### Этап 5: Зависимости, риски, открытые вопросы
- Зависимости между командами/системами со статусом согласования.
- Риски: `Риск | Вероятность | Влияние | Митигация` (из минусов концепта + контраргументов дебатов).
- Открытые вопросы: всё незакрытое → `[УТОЧНИТЬ]`. Закрытые → «Ключевые решения».

### Этап 6: Якоря истины + блок «Требует уточнения»
- Финальный раздел «Якоря истины» (факт → источник).
- Все `[УТОЧНИТЬ]` собери в блок **Требует уточнения у PO** в НАЧАЛЕ документа.

### Этап 7: Сборка + STOP
- Frontmatter + H1 `# [БФТ] {EPIC}: {Название}` + все разделы по `ideal_bft.md`.
- Сохрани черновик.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Черновик БФТ готов: <путь>
  Концепт: <выбранный>. [УТОЧНИТЬ]: N шт.
  ── СТОП ── PO/СА: проверьте черновик. Дальше — /bft-validate <epic> (hard gates + Светофор).
  ```

## Запреты
1. Не проводи adversarial заново (он был в /bft-debate) — только включи его выводы.
2. Не делай финальную валидацию (это /bft-validate).
3. Не выдумывай факты — только из концепта/problem/context-pack или `[УТОЧНИТЬ]`.
