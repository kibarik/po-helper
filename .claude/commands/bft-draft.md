## Использование

```
/bft-draft <epic_code>
```

Вход: `concept.md` (с вердиктом от `/bft-debate`) + `problem.md` + context-pack. Выход: черновик БФТ.

## Важно

**Роль: Requirements Writer.** Сформируй БФТ по **утверждённому дебатами концепту**. Формулируешь **что** и **зачем**, не **как**. Строго по шаблону `ideal_bft.md`.

**Высота БФТ (главное ограничение):** документ на уровне результата и приёмки — как это выглядит для бизнеса и пользователя при сдаче. Не опускайся на системные требования, API-контракты, схемы сообщений, sequence-вызовы между сервисами, БД-схемы, код. Это зона СА / системных требований (FNR). Цель — точные критерии приёмки и план демонстрации (что / где / почему изменено). Гейт 14.

> Аналог sa-helper: `/fnr-system-requirements`. Adversarial уже в `/bft-debate`; валидация — `/bft-validate`.

**Принципы:**
- Каждое требование ← якорь (`anchor_rules.md`). **As-Is → R1 (код) / R2; To-Be → R2 (ТЗ/задача) / R3 (PO-решение).** Не заякоривай To-Be на код (гейт 12).
- CATWOE из концепта → `W` в формулировку БТ, `O` в Ревью, `E` в НФТ (`resources/catwoe.md`).
- Типы не смешивать: ПТ≠ФТ≠НФТ (гейт 11, таблица в `bft_standards.md`).

## Инструкция для LLM

### Этап 1: Загрузка роли, стандартов, входа
1. `skills/bft-writer/SKILL.md` (принципы).
2. Стандарты: `resources/bft_standards.md` (форматы + различение типов + NFR-реестр). Шаблон: `examples/ideal_bft.md`. Эталон: `examples/golden_bft_example.md`.
3. `resources/anchor_rules.md` (ранги якорей). `resources/catwoe.md`. `resources/writing_style.md` (стиль и голос, калибровка под золотой референс).
4. **`resources/review_feedback.md` — реестр замечаний ревью.** Все строки `активно` = обязательные ограничения генерации (уроки прошлых БФТ). Применяй на всех этапах ниже; при конфликте с базовым стандартом реестр приоритетнее — конфликт вынеси в «Открытые вопросы».
5. Входы (все из `<workspace>/<epic>/artefacts/`): `concept.md` (концепт + CATWOE + вердикт), `problem.md` (As-Is/Gap), `bft-context-pack.md`.

### Этап 2: Каркас
- **Порядок разделов чистовика фиксирован по корп-шаблону MTS v10 (ЗМ-005, см. `bft_standards.md`):** Бизнес описание → Общая информация → Заинтересованные стороны → История изменений → Доп. материалы → Проблема которую решаем → Изменение в UJM → План демонстрации → Бизнес-Требования (Вводные\* + Ценность\*) → Пользовательские\* → Интерфейсы\* (+ Макеты\*) → Функциональные\* → Нефункциональные\* → Зависимости → Риски → Ревью требований → Якоря. **Начинай с «Бизнес описание».** Раздела «Ключевые решения» НЕ генерировать.
- **Бизнес описание (ЗМ-013)** — кратко и точно: строка **«Образ результата:»** (конкретное операционное целевое состояние + затронутые сущности) + строка **«Зачем делать:»** (ценность + метрика). Полный контекст — под спойлер `<details><summary>Подробный контекст…</summary>…</details>` (для LLM). Не многословить. **Не вставлять отдельную строку «Скоуп документа —»** — она даёт самопротиворечие с деталями (ЗМ-014); объём читается из «Образа результата» и ДЛЯ КОГО/ЧТО ДЕЛАЕМ в «Проблеме».
- **В чистовике НЕТ разделов «Границы» и «Критерии успеха»** (метрики). Границы и As-Is/Gap — рабочие аиды стадии решения (`problem.md`), в чистовик не выносятся (гейт 14 высоты сохраняется).
- **Открытые вопросы** живут в подразделе **«Вводные для разрабатываемого функционала\*»** внутри Бизнес-Требований (таблица `Информация или вопрос | Конспект | Ответ | В рамках чего получен | Кто ответил | Дата`), а не отдельным разделом.
- **Проблема которую решаем** — **таблицей, НЕ текстом** (ЗМ-002): пять строк `ДЛЯ КОГО` / `ЧТО ДЕЛАЕМ` / `ASIS` / `ПРОБЛЕМА` / `TOBE` (из problem.md; ASIS с якорями R1/R2; ПРОБЛЕМА — бизнес-разрыв). Однострочные ячейки — простым текстом; многопунктовые (ПРОБЛЕМА/TOBE) — списком `<ul><li>…</li></ul>`, НЕ `•`+`<br>` (ЗМ-011).
- **Изменение в UJM** — отдельный раздел после «Проблемы»: таблица `Точка демо | Было (As-Is) | Стало (To-Be)` на уровне приёмки.
- **План демонстрации** — раздел перед «Бизнес-Требованиями»: таблица акторов + PlantUML-сценарий (см. Этап 4).

### Этап 3: Каскад требований (с трассировкой + CATWOE + якоря)
- **БТ** (ценность из CATWOE `W`, приоритет) → **ПТ** (story Когда/Я хочу/Чтобы) → **ИТ** (интерфейс приёмки: что видит и может актор; не API-контракт) → **ФТ** (поведение на уровне результата, приоритет) → **НФТ** (только измеримые; из CATWOE `E` и **корпоративного реестра НФТ**, если есть).
- Идентификаторы **локальные** `{БТ|ПТ|ИТ|ФТ|НФТ}-{N}` (ЗМ-003): без EPIC/проекта/сервиса, только для навигации внутри документа. Один шаблон и в «Идентификатор», и в «Связанные требования».
- КАЖДАЯ строка: «Связанные требования» заполнены вверх + эпик.
- Каждый ФТ ≈ атомарная задача (1-2 дня).
- **Различение типов:** ПТ = что хочет пользователь; ФТ = поведение системы; НФТ = измеримое -ility. Не смешивай.
- **Якорение:** As-Is-факты → R1/R2; To-Be-требования → R2/R3 (не R1).
- **Ссылки (ЗМ-004, ЗМ-009, ЗМ-012):** линкуй **каждое** вхождение Jira/Confluence в каждой колонке и абзаце (не только первое) — markdown-ссылкой **на существующую** страницу. Линкуй не только ключи задач, но и **JIRA-проекты** (`https://jira.mts.ru/projects/{PROJECT}`) и **Confluence-пространства** (`https://confluence.mts.ru/display/{SPACE}`): `[GDSLV-1409](https://jira.mts.ru/browse/GDSLV-1409)`, `[проект GDSLV](https://jira.mts.ru/projects/GDSLV)`, `[Confluence pageId](https://confluence.mts.ru/pages/viewpage.action?pageId=pageId)`. Цель — клик из документа, без поиска снаружи. Ни одного голого ключа. Объекта ещё нет (эпик не создан) → не ссылка, а `[УТОЧНИТЬ]`/пометка без URL. `BR-*`/`US#N` не линкуются. Макросы Confluence для превью — на публикации (`/bft-deliver`).

### Этап 4: План демонстрации (акторы + сценарий)
- Раздел **«План демонстрации»** (перед Бизнес-Требованиями): таблица акторов (из CATWOE `C`/`A`) + PlantUML-сценарий приёмки.
- PlantUML sequence — только actor-level / black-box: акторы и витрина, без вызовов между сервисами и без имён сообщений/API. happy path + ВСЕ `alt`-ветки (ошибка/недоступно/отмена/таймаут). Mermaid запрещён. В `.md` — блок ` ```plantuml `; при публикации оборачивается в макрос «PlantUML» для визуального рендера (ЗМ-015, `/bft-deliver`).
- Атрибутивный состав сообщений НЕ описывать — это зона СА / системных требований (гейт 14).

### Этап 5: Зависимости, риски, вводные (открытые вопросы)
- Зависимости между командами/системами со статусом согласования.
- Риски: `Риск | Вероятность | Влияние | Митигация` (из минусов концепта + контраргументов дебатов).
- Вводные: всё незакрытое → строка в таблице «Вводные для разрабатываемого функционала\*» с пустым «Ответ» и `[УТОЧНИТЬ]`; полученный ответ (CATWOE `O` = согласующий) вписывается в те же колонки. Отдельного блока «Ключевые решения» нет.

### Этап 6: Ревью требований + Якоря истины
- Раздел **«Ревью требований»** (после «Рисков») — список ролей: Продакт, Архитектор, Аналитик (кросс-ревью), Разработка, Тестирование (при наличии с ФИО через CATWOE `O`). Не таблица со статусом. Раздела «Ключевые решения» нет.
- Финальный служебный раздел **«Якоря истины»**: `Факт | Источник | Ранг | Тип` (по `anchor_rules.md`).
- Все `[УТОЧНИТЬ]` продублированы строкой в «Вводных» (единое место открытых вопросов, ЗМ-005).

### Этап 7: Стиль-проходка (Humanizer)
Перед сборкой прогони чек-лист `resources/writing_style.md` по черновику и почисти:
- **Эмодзи:** убери из тела `⚠️ ✅ ❌ ⚡` и пр. (статусные значки только в `validation.md`). Скоуп-предположения и предупреждения → обычный абзац или строка в «Открытые вопросы», не callout-врезка `> **⚠️ …`.
- **Bold:** убери `**bold**` внутри ячеек таблиц БТ/ИТ/ФТ/НФТ. Оставь только story-ключи в ПТ (`**Когда**/`**Я хочу**/`**Чтобы**`) и редкие смысловые акценты в прозе.
- **Тире:** em-dash оставь для определений и перечислений; связки-заменители замени на запятую/точку/скобки. Не больше одного тире на предложение.
- **Стоп-слова:** «позволяет», «обеспечивает», «осуществляет», «производит», «представляет собой», «важно отметить», «в рамках данного» — замени на прямые формулировки в стиле золотой референс.
- **`[УТОЧНИТЬ]`:** убери кросс-рефы `[УТОЧНИТЬ Ox]` → `[УТОЧНИТЬ]`; если рамка известна, а число нет — «(предварительно)» / «уточняется у X» / «TBD». Каждый вопрос — один раз в таблице «Вводные для разрабатываемого функционала\*».
- **Служебные леса:** не выноси в тело внутренние пути инструмента (CORTEX/_context-packs, concept.md, context-pack) — ни в frontmatter, ни в «Доп. материалах», ни сносками. Автор в «Истории изменений» — `bft-draft` (имя инструмента), не «Claude (bft-draft)». Не выдумывай стейкхолдеров/метрики/сроки «для полноты» — неизвестное → `[УТОЧНИТЬ]` или не вносить.
- **Frontmatter:** убери diff-narrative (`supersedes: v0.1 (…исправлено)`, многословный `status`). История правок — в разделе «История изменений» одной строкой на версию.

Стиль генератора задаёт стиль вывода: если приёма нет в золотой референс, его не должно быть в БФТ. Запомни количество правок для STOP-вывода.

### Этап 8: Сборка + STOP
- **Самопроверка на противоречия (ЗМ-014, гейт 16):** перед сохранением сверь «Образ результата» / «Проблема» (ДЛЯ КОГО/ЧТО ДЕЛАЕМ/TOBE) / БТ / ФТ / перечень сущностей / «Изменение в UJM» — они не должны противоречить друг другу (частый случай: «пилот на одном типе» в одном месте и «все типы» в другом). Нашёл расхождение — устрани до сборки.
- Frontmatter + H1 `# [БФТ] {EPIC}: {Название}` + все разделы по `ideal_bft.md`.
- **Сохрани финальный БФТ в КОРЕНЬ папки эпика: `<workspace>/<epic>/<epic>.md`** (не `draft.md`, не в `artefacts/`). Это главный документ, который открывает PO.
- **Выведи РОВНО три блока и ОСТАНОВИСЬ** (формат фиксирован, ничего сверх):
  ```
  БФТ: <workspace>/<epic>/<epic>.md

  Открытые вопросы (<N>):
  - [У<n>] <вопрос одной строкой>
  - …
  (если N=0 → строка «Открытых вопросов нет».)

  Дальше: /bft-validate <epic>
  ```
- **Только эти три блока.** НЕ добавляй: пересказ «что внутри», счётчики БТ/ПТ/ИТ/ФТ/НФТ, как заякорено, список зафиксированных решений (PD-*), описание sequence-веток, отчёт стиль-проходки, эмодзи, преамбулы, похвалу. Документ для этого и есть — PO откроет файл. Сводка = указатель (ссылка + что требует ответа + один шаг), не дайджест.
- Открытые вопросы — это все `[УТОЧНИТЬ]` из черновика, одной строкой каждый, без обоснований и кросс-ссылок.

## Запреты
1. Не проводи adversarial заново (был в /bft-debate) — включи его выводы.
2. Не делай финальную валидацию (это /bft-validate).
3. Не выдумывай факты — только из концепта/problem/context-pack или `[УТОЧНИТЬ]`.
4. Не заякоривай To-Be на код (R1) — гейт 12.
5. Не смешивай типы требований — гейт 11.
6. Не лезь в код / API / системные требования — БФТ на уровне приёмки (гейт 14). ИТ = интерфейс приёмки, не API-контракт.
7. Не нарушай ни одного `активно`-правила из `resources/review_feedback.md` — гейт 15.
