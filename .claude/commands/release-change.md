## Использование

```
/release-change <release-id>
```

Вход: новое требование/изменение от заказчика или сработавший риск. Выход: строка в `change-log.md` + пересчёт дрейфа.

## Важно

**Роль: Change Controller.** PO защищает каждое изменение до конца — поэтому гейт **не режет, а атрибутирует**: фиксирует автора, считает цену в спринтах, возвращает решение названному владельцу последствия. Это превращает «техкоманда тормозит» в «объём вырос на X спринтов из N запросов продакта Y — вот реестр».

> Реестр change-log — политический инструмент: каждое изменение получает имя автора и владельца последствия.

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `skills/release-guard/resources/release_standards.md` (§4 change-log, §1 конвертация).
2. Прочитай `baseline.md` (активная версия), `change-log.md`.
3. **BFT-ярус (если инстанцирован — см. `GROUND/NEXUS/_registry.yaml`):** держи под рукой `decisions` (узлы `decision` — чтобы `demand`-изменение привязать к решению-первопричине) и `risk` (узлы `risk` — чтобы `supply`-изменение привязать к сработавшему риску). Нет Нексуса → атрибутируй как раньше, без node-якоря.

### Этап 2: Опрос (один вопрос за раз)
**Q1 (автор):** «Кто инициатор изменения? (имя/роль — атрибуция обязательна)»
**Q2 (что):** «Что именно меняется/добавляется?»
**Q3 (lane):** определи сам: изменение требований → `demand`; сработавший риск/баг/блокер → `supply`. Уточни у PO при сомнении.
**Q4 (цена):** «Оценка изменения в `{estimate_unit}`? Кто оценил?» → сконвертируй в спринты.
**Q5 (решение):** «Как поступаем: absorb (влезает) / cut-<REQ> (режем другое) / move-date / defer (следующий банч) / refuse (отказ)?»

### Этап 3: Определение владельца последствия
- `demand` → владелец последствия = автор запроса (Q1). Он принимает трейд-офф.
- `supply` → владелец = PO (+ стейкхолдер при move-date).
- `buffer` (риск в пределах буфера) → владелец «—».

### Этап 4: Запись строки
Допиши строку в `change-log.md` по схеме §4: `id | дата | автор | описание | lane | Δ | решение | владелец последствия | baseline_ver`. `baseline_ver` = текущая активная версия. Если изменение отражает решение или сработавший риск из BFT-яруса — добавь node-якорь в «описание» (`(decisions:node_id)` / `(risk:node_id)`); это делает атрибуцию дрейфа трассируемой до первопричины.

### Этап 5: Пересчёт дрейфа
Прогони расчёт §5 (как `/release-sync`, без чтения трекера — только по change-log + предыдущий ledger). Обнови `ledger.md`. Если `total_drift` пересёк порог или это `black_swan` → выведи рекомендацию `/release-gate` (сам алерт-файл пишет ближайший `/release-sync`, либо при решении `move-date`/`cut`/`refuse` сразу зови `/release-gate`).

### Этап 6: Вывод
```
CR-<n> записан: <автор> | <lane> | <Δ> сп. | решение: <...> | владелец: <...>
дрейф теперь: total <T> сп. → band <...>
[если порог] → /release-gate <release-id> (нужен ре-baseline)
```

## Запреты
1. НЕ записывай change без автора — атрибуция обязательна, это суть реестра.
2. НЕ «всасывай» молча: даже absorb фиксируется строкой с ценой.
3. Оценку без источника → `[УТОЧНИТЬ кто оценил]`, не выдумывай.
