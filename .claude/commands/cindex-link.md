## Использование

```
/cindex-link <space_key>
```

Вход: `artefacts/confluence/routing.jsonl` (+ `inventory.json`,
`nodes-draft.jsonl` для контекста связей). Выход: тот же
`artefacts/confluence/routing.jsonl`, обновлённый на месте полем `links`.

## Важно

**Роль: Linker.** Ты достраиваешь граф `[[wiki-links]]` между узлами,
которые уже разложил по нексусам `/cindex-route`. Ты **не** меняешь
`target_nexus`, `route_confidence`, `action` или `node_id` — это поля
Router'а. Ты **не** проверяешь, существуют ли узлы физически в
`GROUND/NEXUS/<slug>/` — это делает `/cindex-deliver` вместе с
`sa_documentation/lint_graph.py` после записи.

Пятая стадия пайплайна (`.claude/skills/confluence-indexator/SKILL.md`).
Авто-batch, без STOP — следующая обязательная пауза (сухой прогон → ок PO)
уже внутри `/cindex-deliver`.

**Ноль галлюцинаций (жёсткое правило):** ребро ставится только если есть
основание — структурное (`ancestors`/`links` страницы Confluence) или
семантическое (общая сущность, подтверждённая текстом `title`/`body`). Связь
«по ощущению», без опоры на конкретную страницу/сущность, не создаётся.

## Инструкция для LLM

### Этап 1: Загрузка входа и правил

1. Прочитай `artefacts/confluence/routing.jsonl`. Нет файла →
   `[УТОЧНИТЬ: запусти /cindex-route <space_key>]`.
2. Прочитай `artefacts/confluence/nodes-draft.jsonl` — источник `source_page`,
   `title`, `body` для каждого `tmp_id` (в `routing.jsonl` их уже нет, только
   маршрутные поля — join делай по `tmp_id`).
3. Прочитай `artefacts/confluence/inventory.json` — `ancestors`/`links`
   каждой страницы Confluence (карта пространства из `/cindex-crawl`).
4. Прочитай
   `.claude/skills/confluence-indexator/resources/linking_rules.md` — три
   правила рёбер (структурные, семантические, висячие) дословно; ниже —
   только порядок применения, сами правила не пересказываются повторно.

### Этап 2: Структурные рёбра (Правило 1 `linking_rules.md`)

Для каждой записи `routing.jsonl` с резолвленным `node_id`:

1. Найди её `source_page` (join через `tmp_id` → `nodes-draft.jsonl`).
2. В `inventory.json` найди эту страницу, возьми её `ancestors` (страницы-
   предки) и `links` (явные исходящие ссылки на другие страницы).
3. Для каждой страницы-предка/ссылки: если у неё тоже есть узел в
   `routing.jsonl` (через `tmp_id` с тем же `source_page`) и у него
   резолвлен `node_id` — добавь `[[<этот node_id>]]` в `links` текущей
   записи. Ставь ребро в обе стороны, если направление это оправдывает
   (родитель-потомок, взаимная ссылка страниц) — как описано в
   `linking_rules.md`.

### Этап 3: Семантические кросс-нексусные рёбра (Правило 2 `linking_rules.md`)

Для пар записей с резолвленным `node_id` **в разных `target_nexus`**: если их
`title`/`body` явно указывают на одну и ту же сущность (тот же сервис, тот же
термин, та же метрика) — добавь взаимные `[[node_id]]` рёбра между ними, даже
если структурной связи в Confluence не было. Совпадение сущности должно быть
текстуально подтверждено (то же имя/явный синоним в `title` или `body`), не
предположено по тематической близости.

### Этап 4: Узлы без резолвленного `node_id`

Запись с `target_nexus: null` (в очереди `_intake/unrouted.md`, без `node_id`)
не может быть целью ребра — ссылаться не на что. Пропусти такую запись как
цель связи; это не ошибка, узел получит `node_id` после ревью PO и повторного
роутинга. Сама такая запись всё равно получает поле `links` (пустой список,
если ничего не нашлось) для единообразия схемы.

### Этап 5: Висячие ссылки (Правило 3 `linking_rules.md`)

На этой стадии **все** сформированные `[[node_id]]` технически «висячие» —
`GROUND/NEXUS/<slug>/` ещё не записан (`/cindex-deliver` ещё не выполнялся).
Это ожидаемо и не требует дополнительной проверки существования узла здесь.
Фактическую проверку битых ссылок делает `sa_documentation/lint_graph.py` как
verify-гейт после `/cindex-deliver` — не дублируй эту проверку на этой
стадии.

### Этап 6: Сохранение

Обнови `artefacts/confluence/routing.jsonl` **на месте** — та же одна строка
JSON на запись, с добавленным полем `links` (список строк вида
`"[[<node_id>]]"`, `[]` если рёбер не нашлось). Остальные поля записи
(`tmp_id`, `node_type`, `target_nexus`, `node_id`, `route_confidence`,
`action`, `reason`) не трогай.

```json
{"tmp_id":"n-131074-1","node_type":"decision","target_nexus":"precedents","node_id":"precedents-vybor-sinhronnogo-platezhnogo-potoka","route_confidence":0.82,"action":"auto","links":["[[system-landscape-payment-gateway]]"]}
{"tmp_id":"n-131090-2","node_type":"risk","target_nexus":null,"route_confidence":0.34,"action":"queue","reason":"нет уверенного нексуса","links":[]}
```

### Этап 7: Отчёт (без STOP)

```
Линковка завершена: artefacts/confluence/routing.jsonl обновлён (<N> записей).
Рёбра: <M> всего (<S> структурных, <K> семантических кросс-нексусных).
Записей без исходящих связей (links: []): <L>.

Дальше: /cindex-deliver <space_key> — там STOP на сухом прогоне перед записью.
```

STOP не требуется на этой стадии — следующая обязательная пауза внутри
`/cindex-deliver` (сухой прогон → ок PO).

## Запреты

1. НЕ создавать ребро без основания — только `ancestors`/`links` страницы
   (структурное) или текстуально подтверждённая общая сущность (семантическое).
2. НЕ придумывать более строгое правило направления ребра, чем сказано в
   `linking_rules.md` — следуй ему дословно, не усиливай и не ослабляй.
3. НЕ проверять существование целевого узла в `GROUND/NEXUS/<slug>/` —
   висячие ссылки допустимы, проверка — задача `lint_graph.py` после
   `/cindex-deliver`.
4. НЕ менять `target_nexus`, `route_confidence`, `action`, `reason` или
   `node_id` — это поля `/cindex-route`, Linker только добавляет `links`.
5. НЕ ставить ребро на запись без резолвленного `node_id` (в очереди) — не на
   что ссылаться, пока PO не проревьюит и не перероутит.
