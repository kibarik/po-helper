## Использование

```
/sprint-load <sprint>
```

Вход: `sprint-stories.md` + `sprint-context.md`. Выход: `{sprint_workspace}/sprint-load.md`.

## Важно

**Роль: Capacity Balancer. Это вопрос HOW из Scrum-планирования.** Назначить исполнителей и SP, **максимально загрузив каждого** без перегруза, и дать персональную оптику — что лично каждый показывает к концу спринта. Это главная ценность процесса: каждый участник однозначно понимает «что хотят от меня».

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `skills/sprint-planner/SKILL.md` (принципы 5, 6).
2. Прочитай `skills/sprint-planner/resources/sprint_standards.md` — модель ёмкости, SP-шкала, флаги загрузки.
3. Прочитай `{sprint_workspace}/sprint-stories.md` и `sprint-context.md` (раздел 2 — ёмкости/отпуска).

### Этап 2: Оценка SP
Проставь SP каждой истории по шкале Фибоначчи (`sprint_standards.md`). Если объём неясен — диапазон + `[УТОЧНИТЬ на оценке с командой]`, не выдумывать. История > 8 SP в Must → вернуть на декомпозицию (пометить).

### Этап 3: Назначение и балансировка
1. Посчитай ёмкость каждого: `velocity × focus_factor − отпуска` (из context/профиля). Нет velocity → `sp_per_person_sprint` + `[УТОЧНИТЬ velocity]`.
2. Назначь исполнителей по экспертизе (роль из истории ↔ `role_title`/`expertise_topics` из NEXUS/team). Carryover — тому, кто вёл.
3. Сложи загрузку по каждому и сравни с ёмкостью:
   - 🔴 **Перегруз** (>100%): сними Should/Could или передай другому. Предложи PO варианты.
   - 🟡 **Недогруз** (<70%): добери историю из бэклога/N+1/внеплановых. Предложи PO конкретные кандидаты.
   - 🟢 **70–100%**: ок.
4. Цель — **ни одного недогруженного и ни одного перегруженного**. Итерируй, пока не сбалансировано (или PO принял остаток).

### Этап 4: Диалог при дисбалансе (по одному вопросу)
Если есть 🔴/🟡 — спроси PO точечно: «{Имя} перегружен на X SP. Снять {история} (Should) или передать {кому}?» / «{Имя} недогружен на Y SP. Добрать {кандидат}?»

### Этап 5: Сборка sprint-load.md
```markdown
# Sprint Load: {sprint}

## Истории с назначением (полные таблицы по OBJ + внеплановые)
| ЭПИК | История | Образ результата | Образ действия | Исполнитель | Приоритет | SP |

## 📊 Capacity по командам
| Исполнитель | Ёмкость (SP) | Загрузка (SP) | Статус | Основная нагрузка |

## 👤 Персональный фокус (что лично каждый показывает к концу)
| Исполнитель | Что показываю к концу спринта |

## Действия по балансу
[что снято/добрано/передано и почему]
```

### Этап 6: Сохранение + STOP
- Сохрани в `{sprint_workspace}/sprint-load.md`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Sprint Load готов: {sprint_workspace}/sprint-load.md
  Загрузка: 🟢 X · 🟡 Y недогруз · 🔴 Z перегруз. Σ SP: {сумма} / ёмкость {总}.
  TBD-исполнители в Must: [нет / список].
  ── СТОП ── PO: подтвердите распределение и баланс.
  Дальше → /sprint-deliver {sprint}
  ```

## Запреты
1. Перегруз (🔴) нельзя оставлять молча — предложить PO решение.
2. Нет TBD-исполнителя в Must-историях.
3. SP неизвестен → диапазон + `[УТОЧНИТЬ]`, не выдумывать.
4. Назначение по экспертизе из ростера, не случайно.
