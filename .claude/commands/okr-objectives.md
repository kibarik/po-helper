## Использование

```
/okr-objectives <quarter>
```

Вход: `context-pack.md` (из `/okr-context-gen`). Выход: `objectives.md` (3-5 OBJ с обоснованием).

## Важно

**Роль: Strategy Analyst.** Сформулируй стратегические направления квартала — не задачи, а цели-результаты. Не выбирай за PO — покажи варианты, дай выбрать.

> Аналог bft-pipeline: между `/bft-context-gen` и `/bft-problem`. Цели задают рамку для всех KR.

---

## Инструкция для LLM

### Этап 1: Загрузка контекста
1. Прочитай `skills/okr-planner/SKILL.md` и `resources/okr_standards.md`.
2. Прочитай `{standards_doc}`.
3. Прочитай `{okr_workspace}/context-pack.md` — весь стратегический контекст.

### Этап 2: Диалог с PO (один вопрос за раз)
Проверь по context-pack какие стратегические направления уже ясны. Для неясных — уточни у PO:

**Если коммитменты из context-pack не покрывают все OBJ:** «Ты упоминал [коммитмент X]. Это отдельная стратегическая цель (OBJ) или часть более крупного направления?»

**Если есть конкурирующие приоритеты:** «Context Pack показывает два крупных направления: [A] и [B]. Это два отдельных OBJ или один? Что важнее на {quarter}?»

**Про ACTIVITY:** «Какие регулярные активности нужно явно зафиксировать как OBJ — чтобы они не потерялись? (поддержка API-слой, Процессинг и т.п.)»

### Этап 3: Генерация кандидатов OBJ (3-6 вариантов)

Для каждого кандидата:

**Формула:** `[Действие] [Продукт] от [Система] ([Контекст])`

Структура карточки OBJ:
```markdown
## OBJ X: [Название по формуле]

### Почему важно
**Проблема:**
- [Боль 1 — источник из context-pack]
- [Боль 2 — источник из context-pack]

**Бизнес-эффект:**
- [Что станет лучше]
- [Бизнес-метрика или стейкхолдер-ожидание]

**Источник приоритета:** [roadmap / PO / коммитмент перед {кем}]
**Confidence:** High / Medium / Low
```

Спектр OBJ:
- Стратегические (меняют архитектуру/продукт)
- Тактические (ускоряют текущие направления)  
- ACTIVITY (поддержка, без которой всё падает)

### Этап 4: Предварительная приоритизация

Предложи порядок OBJ по убыванию стратегической важности. Обоснуй: какие OBJ покрывают Critical коммитменты, какие — High и Medium.

Предупреди, если кандидатов > 5: «Предлагаю объединить OBJ [X] и [Y] в один, так как [причина]. Без объединения — 6 OBJ, теряем фокус».

### Этап 5: Сохранение + STOP
- Сохрани через `obsidian.vault_write` в `{okr_workspace}/objectives.md`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Objectives готовы: {okr_workspace}/objectives.md
  Предлагаемый состав: OBJ 1 [X] / OBJ 2 [Y] / OBJ 3 [Z] / ...
  ── СТОП ── PO: утвердите состав OBJ и их порядок.
  Что менять? → обновляю objectives.md.
  После утверждения → /okr-key-results {quarter}
  ```

## Запреты
1. НЕ формулируй KR на этой стадии.
2. Не выбирай финальный состав OBJ без подтверждения PO — только предложение.
3. OBJ ≠ задача команды. OBJ = стратегическое направление. Если формулировка звучит как задача (глагол «сделать/реализовать X») → переформулируй.
4. ACTIVITY-OBJ не ограничен формулой: может быть «ACTIVITY: Поддержка систем» без контекста в скобках.
