## Использование

```
/bft-context-gen <epic_code> <jira_key>
```

**Параметры:**
- `<epic_code>` — короткий код БФТ (напр. `EPIC-10`).
- `<jira_key>` — эпик JIRA (напр. `PROJ-101`).

## Примеры

```
/bft-context-gen EPIC-10 PROJ-101
/bft-context-gen EPIC-PROC PROJ-103
```

## Важно

**Режим:** быстрый первичный контекст. Читает CORTEX (локальные файлы vault) + один запрос к JIRA, далее LLM синтезирует вероятную картину домена. Неизвестное — честно помечает `[УТОЧНИТЬ]`. Для полного глубокого разбора по всем источникам используй `/bft-context-gen-deep`.

**Когда использовать /bft-context-gen (быстрый):**
- Домен хорошо знаком, CORTEX актуален
- Нужно быстро войти в контекст перед встречей с PO
- Первый черновик для ориентира перед глубоким исследованием
- JIRA/Confluence временно недоступны (VPN)

**Когда лучше /bft-context-gen-deep:**
- Новый/незнакомый эпик
- Много неизвестных зависимостей между командами
- Требуется полная доказательная база для БФТ без дыр

## На выходе

```
bft_documentation/<epic>/artefacts/
└── bft-context-pack.md   ← pack с пометкой [БЫСТРЫЙ]: CORTEX + базовый JIRA + LLM-синтез + матрица покрытия
```

**Иные файлы не создавать.** Pack — единственный артефакт, который читает `/bft-gen`. Кладётся в `artefacts/` (служебная папка пайплайна); финальный БФТ `<epic>.md` появится в корне папки эпика на стадии `/bft-draft`.

## Инструкция для LLM

### Этап 0: Подготовка

1. Создай папку `bft_documentation/<epic>/artefacts/`.
2. Зафиксируй `epic_code` и `jira_key`.
3. В шапке pack сразу пометь: `Режим: БЫСТРЫЙ (bft-context-gen). Для полного разбора — /bft-context-gen-deep.`

### Этап 1: CORTEX — читаем локальный фон (без сетевых запросов)

Прочитай из vault только релевантное по теме эпика:

- **C1 Архитектура** (`{cortex.architecture}`) — границы, потоки, сервисы.
- **C3 Регуляторика** (`{cortex.regulatory}`) — применимые законы.
- **C2+ Бизнес-правила** (`{cortex.business_rules}`) — релевантные BR-*.
- **C5 Решения** (`{cortex.decisions}`) — известные ADR и PO-решения.
- **Смежные БФТ/СА** (`{bft_store}/`, `{cortex.sa_store}`) — есть ли документ по этому домену?

Фиксируй: что нашёл / чего нет (→ `[УТОЧНИТЬ: нет в CORTEX]`).

### Этап 1.5: GROUND NEXUS — слой фактов онбординга (без сети)

Прочитай реестр `GROUND/NEXUS/_registry.yaml`, затем по теме эпика — релевантные Узлы **ключевых Нексусов БФТ** (матрица — `sa_documentation/nexus_process_map.md`, колонка `/bft-context-gen`). Они прямо ложатся на блоки синтеза A–F:

| Нексус | блок | что берём |
|---|---|---|
| `problem` | B (боль/As-Is) | метрика под угрозой, выгодоприобретатель, цена бездействия — проблема, не симптом |
| `requester-domain` | A (домен) | бизнес-логика и KPI команды-заказчика |
| `system-landscape` | A, D (границы/зависимости) | bounded contexts, API-контракты, что есть vs строить заново |
| `ownership` | C (стейкхолдеры) | владельцы доменов, согласующие, скрытые стейкхолдеры |
| `precedents` | B, D | похожие БФТ/отказы, связанный техдолг |
| `compliance` | E (регуляторика/НФТ) | security/legal, NFR-бейзлайн, обязательный шаблон БФТ |

Каждый факт из Нексуса ← якорь `(NEXUS:<slug>)`, учитывай CP и `ripeness` (`wilting` → `⚠️ верифицировать`). Нексус пуст → `[УТОЧНИТЬ: нет в <slug>]`. Нексусы — **факты онбординга** (наполнены `/paf-onboard`), CORTEX — слой знаний; дополняют друг друга.

### Этап 2: N1 — JIRA базовый (один запрос на эпик)

1. `mcp__atlassian__jira_get_issue(<jira_key>)` — описание, статус, приоритет, labels, assignee, components.
2. Из описания эпика выдели: цель, scope, связанные компоненты, упомянутые зависимости.
3. **Дочерние задачи не запрашиваем** (это для deep). Только эпик.
4. Если JIRA недоступен → `[УТОЧНИТЬ: JIRA недоступна (VPN?)]`, продолжай с CORTEX.

### Этап 3: LLM-синтез на основе CORTEX + JIRA

На основе прочитанного — рассуждай как опытный аналитик домена. **Явно разделяй факт и вывод:**

- **Факт** (из базы знаний / трекера / Нексусов) — помечай источником: `(C1)`, `(JIRA)`, `(BR-*)`, `(ADR-*)`, `(NEXUS:<slug>)`
- **Вывод LLM** — помечай: `[вывод LLM, требует проверки]`
- **Неизвестное** — помечай: `[УТОЧНИТЬ у {кого}]`

> Факт из Нексуса (CP высокий, свежий) **сильнее** вывода LLM: если `ownership`/`system-landscape`/`problem` дают прямой ответ — блок C/D/B перестаёт быть «вывод LLM» и становится фактом `(NEXUS:<slug>)`.

Синтезируй по блокам:

**A. Домен и контекст**
- Что делает эпик (своими словами, 2-3 предложения)
- К какому домену относится (домены/команды из `teams` и `glossary` профиля; KPI заказчика — `NEXUS:requester-domain`)
- Какие сервисы и БД затрагивает (из C1 + JIRA + `NEXUS:system-landscape`)

**B. As-Is (вероятная текущая ситуация)**
- Как эта область работает сейчас (из C1, смежных БФТ/СА, `NEXUS:system-landscape`)
- Где источник истины (системы из `glossary` профиля)
- Что считается болью/проблемой (из JIRA description + CORTEX + `NEXUS:problem` — метрика под угрозой, цена бездействия; отличай проблему от предложенного решения)

**C. Стейкхолдеры**
- Кто владелец домена (из `NEXUS:ownership` / C1 / известных контактов)
- Кого затрагивают изменения, включая **скрытых стейкхолдеров** (`NEXUS:ownership`)
- Кто согласует (PO / архитектор / юристы по C3; эскалационные пути — `NEXUS:ownership`)
- Есть факт из `ownership` → `(NEXUS:ownership)`; иначе `[вывод LLM — проверить с PO]`

**D. Зависимости**
- Какие смежные системы/команды вовлечены (из C1 + `NEXUS:system-landscape`)
- Какие интеграции затрагивает (существующие API-контракты — `NEXUS:system-landscape`: что есть vs строить заново)
- Известные блокеры, связанный техдолг, зависимые эпики (из JIRA description + `NEXUS:precedents`)
- Есть факт из Нексуса → `(NEXUS:<slug>)`; иначе `[вывод LLM — статус согласования неизвестен, уточнить]`

**E. Регуляторика и риски (из C3 + `NEXUS:compliance`)**
- Применимые законы/требования из C3, NFR-бейзлайн и обязательный шаблон БФТ из `NEXUS:compliance`
- Если ни C3, ни `compliance` не покрывают тему → `[УТОЧНИТЬ: регуляторика для данного домена]`

**F. Открытые вопросы**
- Всё неизвестное из блоков выше — сводный список `[УТОЧНИТЬ у {кого}]`

### Этап 4: Матрица покрытия (быстрая)

| Раздел БФТ | Источник | Режим | Статус |
|---|---|---|---|
| Домен и контекст | JIRA + C1 + NEXUS:requester-domain/system-landscape | факт | ✅/[УТОЧНИТЬ] |
| As-Is | C1 + смежные БФТ/СА + NEXUS:system-landscape | факт | ✅/[УТОЧНИТЬ] |
| Проблема (не симптом) | NEXUS:problem + JIRA | факт+вывод | ✅/[УТОЧНИТЬ] |
| Стейкхолдеры | NEXUS:ownership / LLM-вывод | факт/вывод | ✅/⚠️ проверить |
| БТ (ценность) | JIRA description + NEXUS:problem | факт+вывод | ✅/[УТОЧНИТЬ] |
| ПТ/ФТ | LLM-вывод | вывод | ⚠️ проверить |
| НФТ (нагрузка/SLA) | C1 + NEXUS:compliance + LLM | вывод | ⚠️ /bft-context-gen-deep |
| Negative flows | LLM-вывод | вывод | ⚠️ /bft-context-gen-deep |
| Риски/Compliance | C3 + NEXUS:compliance + LLM | факт+вывод | ✅/[УТОЧНИТЬ] |
| Зависимости | C1 + NEXUS:system-landscape/precedents | факт+вывод | ✅/⚠️ проверить |
| Дочерние задачи | — | не собрано | ➡️ /bft-context-gen-deep |

### Этап 5: Сборка pack

Собери `bft_documentation/<epic>/artefacts/bft-context-pack.md`:

1. **Шапка:** epic, jira_key, дата, `Режим: БЫСТРЫЙ`, статус источников.
2. **CORTEX-выдержки** (C1/C3/BR/C5) — только релевантное.
3. **NEXUS-факты** (problem/system-landscape/ownership/requester-domain/precedents/compliance) — с якорями `(NEXUS:<slug>)`, CP и freshness.
4. **JIRA-факты** — что взято из эпика.
5. **LLM-синтез** (блоки A–F) с явными пометками факт / вывод LLM.
6. **Матрица покрытия**.
7. **Требует уточнения** — сводный список всех `[УТОЧНИТЬ]`.
8. **Рекомендация:** если >3 блоков помечены `⚠️ вывод LLM` в критичных разделах (НФТ, Negative flows, Зависимости) → предложи запустить `/bft-context-gen-deep` перед `/bft-problem`.

## Жёсткие правила

1. **Явная метка режима.** Pack помечается `БЫСТРЫЙ` — читатель знает, что часть — вывод LLM.
2. **Факт vs вывод** — всегда разделять. Не смешивать.
3. **Нет выдумок.** Вывод LLM ≠ факт. Вывод помечается явно.
4. **JIRA — только эпик.** Без дочерних задач и поиска по Confluence (→ deep).
5. **Честно про недоступность** — VPN/MCP проблемы помечать явно.
6. **Стиль pack** — по `skills/bft-writer/resources/writing_style.md`. Pack — это отчёт пайплайна (не БФТ): деловая проза без AI-стоп-слов («позволяет», «обеспечивает», «представляет собой»); эмодзи только семантические (статус источника в матрице покрытия: `✅`/`⚠️`/`➡️`); декоративных эмодзи в шапке режима не ставить — режим текстом.
