## Использование

```
/bft-context-gen <epic_code> <epic_key>
```

**Параметры:**
- `<epic_code>` — короткий код БФТ (напр. `SEARCH-1`).
- `<epic_key>` — эпик в трекере (напр. `PROJECTKEY-123`).

## Примеры

```
/bft-context-gen SEARCH-1 PROJECTKEY-123
```

## Важно

**Назначение:** зеркало sa-helper `/context-gen`. sa-helper собирает ВЕСЬ код (repomix) + граф → детерминированный контекст. БФТ — forward-looking, кода часто нет. Поэтому контекст-пак собирается из **Кортексов** (статичный фон, каталог знаний проекта) + **Нексусов** (живые факты, трекер/wiki/PO). Принцип тот же: дать генератору ПОЛНЫЙ контекст до `/bft-gen`, чтобы не галлюцинировать.

**Принцип Кортекс/Нексус:**
- **Кортекс** — статичный фон, переиспользуется, подключается по ссылке + релевантные выдержки (архитектура, регуляторика, бизнес-правила, история решений).
- **Нексус** — живые факты для ЭТОГО эпика, тянутся из источников per-запрос (трекер, wiki, PO, код/СА).

**Нулевой допуск к галлюцинациям:** каждый факт ← источник (трекер / wiki / PO / ADR). Источника нет → `[УТОЧНИТЬ у {кого}]`. Если трекер/wiki недоступны — честно сказать, собрать доступное, остальное пометить `[УТОЧНИТЬ]`, не выдумывать.

## На выходе ожидается

```
<context-packs>/<epic>/artefacts/
└── bft-context-pack.md   ← единый pack: Кортекс-выдержки + Нексус-факты + матрица покрытия + [УТОЧНИТЬ]
```

**Иные файлы не создавать.** Pack — единственный артефакт, который читает `/bft-gen`. Кладётся в `artefacts/` (служебная папка пайплайна); финальный БФТ `<epic>.md` появится в корне папки эпика на стадии `/bft-draft`.

## Инструкция для LLM

### Этап 0: Подготовка

1. Создай папку `<context-packs>/<epic>/artefacts/`.
2. Зафиксируй `epic_code` и `epic_key`.

### Этап 1: Кортексы (статичный фон — подключить)

Подгрузи из каталога знаний проекта релевантное (по ссылке + короткие выдержки, НЕ копировать целиком):
- **Архитектура** — границы, потоки, сервисы, затрагиваемые данным эпиком.
- **Регуляторика** — применимые законы/стандарты по теме эпика (если домен регулируемый).
- **Бизнес-правила** — релевантные политики продукта по доменам эпика.
- **Решения/ADR** — релевантные архитектурные и продуктовые решения.

> Фильтр релевантности (аналог `graph_select_files` sa-helper): брать только то, что относится к теме эпика.

### Этап 2: Нексус — трекер (скоуп, декомпозиция, статус)

1. Через интеграцию трекера — эпик: описание, статус, приоритет.
2. Дочерние задачи: ключ, статус, assignee, оценка.
3. Если трекер недоступен → `[УТОЧНИТЬ]`, продолжай.

### Этап 3: Нексус — wiki (образ результата, As-Is)

1. По эпику/теме найди ТЗ/постановку.
2. Смежные БФТ/СА из каталога знаний на релевантные аспекты.
3. Зафиксируй: образ результата, As-Is (текущее поведение), границы из постановки.

### Этап 4: Нексус — PO-решения (якорь №1 для forward-looking)

1. Из реестра решений/ADR — релевантные решения.
2. Из смежных БФТ — блоки «Ключевые решения из открытых вопросов».
3. Недостающее → `[УТОЧНИТЬ у PO]`.

### Этап 5: Нексус — Код/СА As-Is (blast radius)

- Если код есть и подключён sa-helper: `repomix`/граф (`graph_impact`, `graph_call_chain`) по затрагиваемым компонентам.
- Если кода нет (greenfield): опирайся на СА и архитектуру.

### Этап 6: Нексус — стейкхолдеры, зависимости, регуляторный

- **Стейкхолдеры:** PO, архитектор, владельцы смежных команд; позиции сторон где расходятся.
- **Зависимости:** между командами/системами проекта — по теме эпика, со статусом согласования.
- **Регуляторный:** из Кортекса регуляторики — конкретные ограничения + владельцы (юристы/безопасность).

### Этап 7: Матрица покрытия (ядро — откуда чем питается каждый раздел)

Собери таблицу: каждый раздел будущего БФТ ← питающий источник. Источника нет → `[УТОЧНИТЬ]`.

| Раздел БФТ | Питающий Кортекс/Нексус | Статус |
|---|---|---|
| Границы системы | Кортекс архитектуры + wiki | ✅/[УТОЧНИТЬ] |
| БТ (ценность) | wiki + PO | |
| ПТ (story) | wiki + PO | |
| ФТ (поведение) | PO + код/СА | |
| НФТ (нагрузка/SLA) | PO + метрики прод + архитектура | |
| Negative flows | PO + код/СА + регуляторика | |
| Риски/Compliance | регуляторный + бизнес-правила | |
| Зависимости | архитектура + зависимости | |

### Этап 8: Сборка pack

Собери `bft-context-pack.md`:
1. Заголовок + мета (epic, ключ, дата сборки, статус источников).
2. **Кортекс** — релевантные выдержки (с указанием, что это фон).
3. **Нексус** — конкретные факты + якоря (ключ трекера / pageId wiki / PO-решение / ADR).
4. **Матрица покрытия** (из Этапа 7).
5. **Требует уточнения** — список всех `[УТОЧНИТЬ]` (что, у кого, как проверить).

### Этап 9: Отчёт

```
Контекст-пак собран: <context-packs>/<epic>/artefacts/bft-context-pack.md
Кортексы подключены (релевантные выдержки).
Нексусы собраны (источники: трекер/wiki/PO/СА).
Матрица покрытия: <X>/<Y> разделов имеют источник. <Z> → [УТОЧНИТЬ].
⚠️ Недоступно: <список>.
── СТОП ── PO/СА: проверьте pack и дополните факты/[УТОЧНИТЬ].
Дальше — /bft-problem <epic> (диагноз As-Is/Gap, без решения).
```

## Жёсткие правила

1. Не выдумывать. Нет источника → `[УТОЧНИТЬ]`. Трекер/wiki/MCP — единственный источник фактов (плюс PO/СА из каталога знаний).
2. Не тянуть лишнее. Кортексы — по релевантности (тема эпика), не целиком.
3. Pack — один файл. `/bft-gen` читает только его.
4. Честно про недоступность — что не собрано, помечать явно.
5. Каждый факт в Нексусе — с якорем (ключ/pageId/ADR-ID/PO-решение+кто).
