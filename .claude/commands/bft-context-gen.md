---

## Использование

```
/bft-context-gen <epic_code> <jira_key>
```

**Параметры:**
- `<epic_code>` — короткий код БФТ (напр. `EPIC-10`).
- `<jira_key>` — эпик JIRA (напр. `PROJ-101`).

## Примеры

```
/bft-context-gen EPIC-10 PROJ-101
/bft-context-gen EPIC-PROC PROJ-103
```

## Важно

**Режим:** быстрый первичный контекст. Читает CORTEX (локальные файлы vault) + один запрос к JIRA, далее LLM синтезирует вероятную картину домена. Неизвестное — честно помечает `[УТОЧНИТЬ]`. Для полного глубокого разбора по всем источникам используй `/bft-context-gen-deep`.

**Когда использовать /bft-context-gen (быстрый):**
- Домен хорошо знаком, CORTEX актуален
- Нужно быстро войти в контекст перед встречей с PO
- Первый черновик для ориентира перед глубоким исследованием
- JIRA/Confluence временно недоступны (VPN)

**Когда лучше /bft-context-gen-deep:**
- Новый/незнакомый эпик
- Много неизвестных зависимостей между командами
- Требуется полная доказательная база для БФТ без дыр

## На выходе

```
bft_documentation/<epic>/artefacts/
└── bft-context-pack.md   ← pack с пометкой [БЫСТРЫЙ]: CORTEX + базовый JIRA + LLM-синтез + матрица покрытия
```

**Иные файлы не создавать.** Pack — единственный артефакт, который читает `/bft-gen`. Кладётся в `artefacts/` (служебная папка пайплайна); финальный БФТ `<epic>.md` появится в корне папки эпика на стадии `/bft-draft`.

## Инструкция для LLM

### Этап 0: Подготовка

1. Создай папку `bft_documentation/<epic>/artefacts/`.
2. Зафиксируй `epic_code` и `jira_key`.
3. В шапке pack сразу пометь: `Режим: БЫСТРЫЙ (bft-context-gen). Для полного разбора — /bft-context-gen-deep.`

### Этап 1: Локальный фон — граф Нексусов (приоритет) → CORTEX (fallback)

**Порядок чтения (без сетевых запросов):** сначала живой граф Нексусов BFT-яруса, затем плоский CORTEX как fallback. Узел Нексуса приоритетнее плоского реестра — он несёт `sources`/`confidence`/`ripeness` (свежесть). Wilting-узел (`ripeness: wilting`) → помечай `[УСТАРЕЛО: узел <id> протух, проверить]`, не доверяй молча.

1. Прочитай `GROUND/NEXUS/_registry.yaml`. Для каждого Нексуса BFT-яруса, если он инстанцирован — читай релевантные узлы по теме эпика; иначе — fallback на CORTEX-предка:

   | Знание для БФТ | Нексус яруса (приоритет) | CORTEX-fallback |
   |---|---|---|
   | Архитектура: границы, потоки, сервисы, БД | `system` (`component`) | `{cortex.architecture}` C1 |
   | Решения: ADR + PO-решения | `decisions` (`decision`) | `{cortex.decisions}` C5 |
   | Бизнес-правила BR-* | `rules` (`rule`) | `{cortex.business_rules}` C2 |
   | Регуляторика | `compliance` (`regulation`) | `{cortex.regulatory}` C3 |
   | НФТ-стандарты, SLA-классы | `quality` (`nfr`) | — (нет CORTEX-предка) |
   | Риски | `risk` (`risk`) | — |
   | Единый язык (термины) | `lexicon` (`term`) | `glossary` из профиля |
   | Метрики/измеримость | `metrics` (`metric`) | — |

2. **Смежные БФТ/СА** (`{bft_store}/`, `{cortex.sa_store}`) — есть ли документ по этому домену?

Фиксируй: что нашёл (Нексус-узел с `node_id` или CORTEX-реестр) / чего нет (→ `[УТОЧНИТЬ: нет ни в графе, ни в CORTEX]`). При использовании узла — источником в pack ставь `node_id` (напр. `(system:comp-billing)`), при fallback — CORTEX-метку (`(C1)`).

### Этап 2: N1 — JIRA базовый (один запрос на эпик)

1. `mcp__atlassian__jira_get_issue(<jira_key>)` — описание, статус, приоритет, labels, assignee, components.
2. Из описания эпика выдели: цель, scope, связанные компоненты, упомянутые зависимости.
3. **Дочерние задачи не запрашиваем** (это для deep). Только эпик.
4. Если JIRA недоступен → `[УТОЧНИТЬ: JIRA недоступна (VPN?)]`, продолжай с CORTEX.

### Этап 3: LLM-синтез на основе CORTEX + JIRA

На основе прочитанного — рассуждай как опытный аналитик домена. **Явно разделяй факт и вывод:**

- **Факт** (из базы знаний / трекера) — помечай источником: `(C1)`, `(JIRA)`, `(BR-*)`, `(ADR-*)`
- **Вывод LLM** — помечай: `[вывод LLM, требует проверки]`
- **Неизвестное** — помечай: `[УТОЧНИТЬ у {кого}]`

Синтезируй по блокам:

**A. Домен и контекст**
- Что делает эпик (своими словами, 2-3 предложения)
- К какому домену относится (домены/команды из `teams` и `glossary` профиля)
- Какие сервисы и БД затрагивает (из C1 + JIRA)

**B. As-Is (вероятная текущая ситуация)**
- Как эта область работает сейчас (из C1, смежных БФТ/СА)
- Где источник истины (системы из `glossary` профиля)
- Что считается болью/проблемой (из JIRA description + CORTEX)

**C. Стейкхолдеры (вывод LLM)**
- Кто владелец домена (из C1 / известных контактов)
- Кого затрагивают изменения (команды API-слой/Шина/Процессинг/BI)
- Кто согласует (PO / архитектор / юристы по C3)
- Пометь весь блок: `[вывод LLM — проверить с PO]`

**D. Зависимости (вывод LLM)**
- Какие смежные системы/команды вовлечены (из C1)
- Какие интеграции затрагивает (API-слой↔Шина↔Процессинг↔BI↔1С↔Финтех)
- Известные блокеры или зависимые эпики (из JIRA description)
- Пометь: `[вывод LLM — статус согласования неизвестен, уточнить]`

**E. Регуляторика и риски (из C3)**
- Применимые законы/требования из C3
- Если C3 не покрывает тему → `[УТОЧНИТЬ: регуляторика для данного домена]`

**F. Открытые вопросы**
- Всё неизвестное из блоков выше — сводный список `[УТОЧНИТЬ у {кого}]`

### Этап 4: Матрица покрытия (быстрая)

| Раздел БФТ | Источник | Режим | Статус |
|---|---|---|---|
| Домен и контекст | JIRA + C1 | факт | ✅/[УТОЧНИТЬ] |
| As-Is | C1 + смежные БФТ/СА | факт | ✅/[УТОЧНИТЬ] |
| Стейкхолдеры | LLM-вывод | вывод | ⚠️ проверить |
| БТ (ценность) | JIRA description | факт+вывод | ✅/[УТОЧНИТЬ] |
| ПТ/ФТ | LLM-вывод | вывод | ⚠️ проверить |
| НФТ (нагрузка/SLA) | C1 + LLM | вывод | ⚠️ /bft-context-gen-deep |
| Negative flows | LLM-вывод | вывод | ⚠️ /bft-context-gen-deep |
| Риски/Compliance | C3 + LLM | факт+вывод | ✅/[УТОЧНИТЬ] |
| Зависимости | C1 + LLM | вывод | ⚠️ проверить |
| Дочерние задачи | — | не собрано | ➡️ /bft-context-gen-deep |

### Этап 5: Сборка pack

Собери `bft_documentation/<epic>/artefacts/bft-context-pack.md`:

1. **Шапка:** epic, jira_key, дата, `Режим: БЫСТРЫЙ`, статус источников.
2. **CORTEX-выдержки** (C1/C3/BR/C5) — только релевантное.
3. **JIRA-факты** — что взято из эпика.
4. **LLM-синтез** (блоки A–F) с явными пометками факт / вывод LLM.
5. **Матрица покрытия**.
6. **Требует уточнения** — сводный список всех `[УТОЧНИТЬ]`.
7. **Рекомендация:** если >3 блоков помечены `⚠️ вывод LLM` в критичных разделах (НФТ, Negative flows, Зависимости) → предложи запустить `/bft-context-gen-deep` перед `/bft-problem`.

### Этап 6: Отчёт

```

Источники: CORTEX (C1/C3/BR/C5) + JIRA basic.
Факты: <X> из JIRA/CORTEX.
Выводы LLM (требуют проверки): <Y> блоков.
Открытые вопросы: <Z> [УТОЧНИТЬ].

<если много выводов LLM>
⚠️ Рекомендую: запусти /bft-context-gen-deep для полной доказательной базы перед /bft-problem.
</если>

── СТОП ── PO/СА: проверьте pack, особенно блоки ⚠️ (вывод LLM).
Дальше:
• /bft-context-gen-deep <epic_code> <jira_key>  ← полный разбор источников
• /bft-problem <epic_code>                       ← диагноз As-Is/Gap (если контекста достаточно)
```

## Жёсткие правила

1. **Явная метка режима.** Pack помечается `БЫСТРЫЙ` — читатель знает, что часть — вывод LLM.
2. **Факт vs вывод** — всегда разделять. Не смешивать.
3. **Нет выдумок.** Вывод LLM ≠ факт. Вывод помечается явно.
4. **JIRA — только эпик.** Без дочерних задач и поиска по Confluence (→ deep).
5. **Честно про недоступность** — VPN/MCP проблемы помечать явно.
6. **Стиль pack** — по `skills/bft-writer/resources/writing_style.md`. Pack — это отчёт пайплайна (не БФТ): деловая проза без AI-стоп-слов («позволяет», «обеспечивает», «представляет собой»); эмодзи только семантические (статус источника в матрице покрытия: `✅`/`⚠️`/`➡️`); декоративных эмодзи в шапке режима не ставить — режим текстом.
