## Использование

```
/bft-concept <epic_code>
```

Вход: `problem.md` (из `/bft-problem`). Выход: `concept.md` (2-3 концепта).

## Важно

**Роль: Solution Designer.** Сгенерируй **спектр решений** проблемы из `problem.md` — от «чистой архитектуры» до «компромисса/быстрого». Не выбирай за PO — дай варианты с плюсами/минусами.

> Аналог sa-helper: `/fnr-concept`. Выбор — задача `/bft-debate`.

**Метод:** каждый концепт трансформации прогони через **CATWOE** (`resources/catwoe.md`) — это выявит суть трансформации (T) и ценность (W) до фиксации требований.

**Принцип:** концепты опираются на As-Is/Gap из problem.md; плюсы/минусы — конкретны (не «лучше/хуже»).

## Инструкция для LLM

### Этап 1: Загрузка роли и входа
1. Прочитай `skills/bft-writer/SKILL.md`.
2. Прочитай `resources/catwoe.md` — метод CATWOE.
3. Прочитай диагноз: `po-helper/БФТ/<EPIC>/problem/problem.md` (+ context-pack).

### Этап 2: Генерация концептов (2-3)
Спектр:
- **Концепт 1 — «чистый»** (правильная долгосрочная модель).
- **Концепт 2 — «компромисс»** (баланс усилий/качества).
- **Концепт 3 (опц.) — «быстрый/костыль»** (минимум усилий, техдолг).

### Этап 3: CATWOE для каждого концепта
Для каждого концепта (см. `resources/catwoe.md`):
- **C** Customer — кто выигрывает/теряет.
- **A** Actor — кто реализует.
- **T** Transformation — вход → выход (суть).
- **W** Worldview — зачем / ценность.
- **O** Owner — согласующий.
- **E** Environment — ограничения среды (→ заготовки НФТ).

### Этап 4: Плюсы / минусы / влияние
Для каждого концепта:
- **Плюсы** (2-3, конкретно, со ссылкой на закрытый Gap).
- **Минусы / риски** (2-3, конкретно).
- **Влияние** (затронутые компоненты, миграция, backward compat).
- **Оценка усилий** (S/M/L) — если возможно.

### Этап 5: Учёт открытых вопросов и worldviews
- Какие открытые вопросы / конфликты worldviews из problem.md каждый концепт закрывает / оставляет.

### Этап 6: Сохранение + STOP
- Сохрани `po-helper/БФТ/<EPIC>/concept/concept.md` (концепты + CATWOE + сравнительная таблица).
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Концепты готовы: po-helper/БФТ/<EPIC>/concept/concept.md (N вариантов, каждый с CATWOE)
  1. <чистый> · 2. <компромисс> · 3. <быстрый>
  ── СТОП ── Дальше — /bft-debate <epic> (красная команда выберет/усилит).
  ```

## Запреты
1. НЕ выбирай «победителя» сам — это задача `/bft-debate`.
2. НЕ формулируй БТ/ФТ (это `/bft-draft`).
3. Концепты должны РАЗЛИЧАТЬСЯ по существу (разные T/W), не косметически.
4. Соблюдай стиль `skills/bft-writer/resources/writing_style.md` для `concept.md`: деловая проза без AI-стоп-слов, без эмодзи в теле, без раздутого bold, тире умеренно. Плюсы/минусы — конкретны, без «лучше/хуже». Стиль генератора задаёт стиль вывода.
