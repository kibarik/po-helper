## Использование

```
/cindex-digitize <space_key>
```

Вход: `artefacts/confluence/comprehension/<id>.md`. Выход:
`artefacts/confluence/nodes-draft.jsonl`.

## Важно

**Роль: Digitizer.** Ты выделяешь атомарные узлы-кандидаты (node_type +
черновик тела) из осмысления — **не** классифицируешь узел в конкретный
нексус и **не** подгоняешь его под нексус: маршрутизация в
`GROUND/NEXUS/<slug>/` — работа `/cindex-route`. Твоя ответственность —
только «что это за тип сущности» и «какой у неё черновой текст», по
`resources/routing_table.md` (колонка `node_type`, без колонки «нексус»).

Третья стадия пайплайна (`.claude/skills/confluence-indexator/SKILL.md`).
Авто-batch, без STOP.

**Ноль галлюцинаций (жёсткое правило):** каждый узел-кандидат обязан нести
`sources: ["confluence:<url>"]`. **Узел-кандидат без источника не
создаётся** — ни один, ни в каком виде, даже как черновик. Это прямое
следствие SKILL.md («Принцип нулевого допуска к галлюцинациям») и канона
`sa_documentation/nexus_schema.md` §2 («узел без sources = workslop»).

## Инструкция для LLM

### Этап 1: Загрузка входа и правил

1. Прочитай все `artefacts/confluence/comprehension/<id>.md`. Нет файлов →
   `[УТОЧНИТЬ: запусти /cindex-comprehend <space_key>]`.
2. Прочитай `.claude/skills/confluence-indexator/resources/routing_table.md`
   — таблица «сигнал на странице → `node_type`»: используй только колонки
   «Сигнал на странице» и `node_type`, колонку «Целевой нексус» на этой
   стадии не трогай (это вход для `/cindex-route`, не для тебя).
3. Прочитай `resources/node_schema.md` — фиксированные поля Confluence-узла
   (`kind: empirical`, `confidence: 0.3`, `tags` включает
   `confluence-indexed`); канон полей — `sa_documentation/nexus_schema.md`.

### Этап 2: Выделение узлов-кандидатов (батч, без STOP)

Для каждого файла `comprehension/<id>.md`:

1. Пройди по разделу «Сущности» — сопоставь каждую с строкой
   `routing_table.md` по сигналу (ADR/решение → `decision`, BR-* → `rule`,
   термин → `term`, метрика/KPI → `metric`, сервис/API → `component`,
   закон/стандарт/security → `regulation`, SLA/НФТ → `nfr`, риск → `risk`,
   персона/роль/RACI → `person`, внешняя команда → ext-team, сегмент/JTBD →
   customer-empirical, конкурент → market-empirical, roadmap/OKR →
   strategy-empirical).
2. **Одна страница может дать несколько узлов** разных строк — это
   ожидаемо (семантический разбор, не «страница = узел»). Не схлопывай
   разные сущности в один узел ради простоты.
3. Для каждого найденного узла-кандидата собери:
   - `tmp_id` — временный id вида `n-<id>-<порядковый номер на странице>`.
   - `node_type` — по таблице.
   - `source_page` — `id` страницы.
   - `title` — короткое название сущности (не заголовок страницы целиком).
   - `body` — черновой текст узла (описание сущности своими словами, по
     материалу осмысления и исходной странице; без выдумывания деталей,
     которых нет на странице).
   - `sources` — **обязательно** `["confluence:<url>"]`, где `<url>` — URL
     страницы из `inventory.json`.
4. **Сигнал не сопоставляется ни с одной строкой таблицы, или на странице
   вообще нет оцифровываемой сущности** — не создавай узел; это уже должно
   было попасть в `skipped.md` на стадии comprehend, здесь дополнительных
   действий не требуется.

### Этап 3: Гейт «нет источника — нет узла»

Перед записью в `nodes-draft.jsonl` проверь **каждый** кандидат: если по
какой-то причине `url` страницы неизвестен или пуст — **не пиши этот
кандидат в файл вообще**, независимо от того, насколько уверенно
определился `node_type`. Отсутствие `sources` не компенсируется
уверенностью в содержании — это и есть определение workslop в этом
пайплайне.

### Этап 4: Сохранение

Сохрани `artefacts/confluence/nodes-draft.jsonl` — один JSON-объект на
строку (дизайн-спека §6):

```json
{"tmp_id":"n-131074-1","node_type":"decision","source_page":"131074","title":"Выбор синхронного платёжного потока","body":"...","sources":["confluence:https://.../131074"]}
```

Финальные поля узла (`nexus`, `node_id`, `route_confidence`, `confidence`,
`ttl_days`, `ripeness`, полный `sources`-формат канона) достраивают
`/cindex-route` и `/cindex-deliver` — на этой стадии фиксируем только то,
что перечислено выше.

### Этап 5: Отчёт (без STOP)

```
Оцифровка завершена: artefacts/confluence/nodes-draft.jsonl (<N> узлов-кандидатов)
Разбивка по node_type: <decision: X, rule: Y, term: Z, ...>
Пропущено кандидатов без источника: <K> (не записаны — нет sources).

Дальше: /cindex-route <space_key>.
```

STOP не требуется — следующая обязательная пауза после `/cindex-link`,
перед `/cindex-deliver`.

## Запреты

1. НЕ создавать узел-кандидат без `sources`.
2. НЕ выбирать целевой нексус — это задача `/cindex-route`, не Digitizer'а.
3. НЕ схлопывать несколько сущностей страницы в один узел ради удобства.
4. НЕ выдумывать содержание `body` сверх того, что подтверждено осмыслением
  и исходной страницей.
