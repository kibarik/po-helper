## Использование

```
/release-status <release-id>
```

Пример всего портфеля: `/release-status --all`

Вход: `baseline.md` (все версии) + `change-log.md` + `ledger.md` + трекер. Выход: `{release_workspace}/<release-id>/status.md`.

## Важно

**Роль: Reporter.** Переопределяет «полноту закрытия квартала» на три честных числа вместо одного перверсного. Цель — мерить полноту против **актуального** baseline, а не против фантазии, придуманной при нулевом CP, и показать, **кто** нагнал дрейф.

> Эту таблицу PO защищает перед CPO/бизнесом: scope drift — диагностика качества входа бизнеса, а не вина команды.

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `skills/release-guard/resources/release_standards.md` (§8 метрики).
2. Прочитай `baseline.md` (активная + v1 + все `superseded`), `change-log.md`, `ledger.md`.
3. Через `{tracker.mcp}` — текущие статусы требований (Done/Released). Недоступно → `[НЕДОСТУПНО]`.

### Этап 2: Расчёт трёх метрик (§8)
- **Baseline-полнота** = `delivered / current_baseline_scope` (против активной версии).
- **Scope drift** = `(current_estimate − v1_estimate) / v1_estimate` (и в спринтах, и в %).
- **Commitment hit-rate** = для уровня `commitment`: попал ли в срок/объём (для одного релиза — да/нет; для `--all` — доля).

### Этап 3: Атрибуция дрейфа
Из change-log сгруппируй Δ по авторам (`demand`) и по причинам (`supply`): «продакт A: +1.2 сп. (CR-1,4); лебеди: +0.9 сп. (блокер БД)». Это ядро отчёта.

### Этап 4: Сборка status.md
```markdown
# Release Status: <release-id> @ <дата>

## Метрики
| Метрика | Значение | Комментарий |
|---|---|---|
| Baseline-полнота | <X>% | <delivered>/<scope> против baseline v<N> |
| Scope drift | +<Y> сп. (+<Z>%) | рост от v1 = качество входа |
| Commitment hit-rate | <hit/miss> | только для уровня commitment |

## Атрибуция дрейфа
- demand: <автор → Δ, CR-id>
- supply: <причина → Δ> (буфер исчерпан на <...>)

## История baseline
v1 → … → v<N> | причины ре-baseline

## Текущий band: <🟢|🟡|🔴> (из ledger)
```

### Этап 5: Сохранение + вывод
- Сохрани через `obsidian.vault_write` в `{release_workspace}/<release-id>/status.md`.
- Для `--all` — собери сводную таблицу по всем релизам в workspace.
- Без STOP. Вывод — краткое саммари трёх метрик + кто главный источник дрейфа.

## Запреты
1. НЕ считай Baseline-полноту против v1, если был ре-baseline — только против активной версии (иначе метрика снова врёт).
2. НЕ замалчивай атрибуцию — кто нагнал дрейф, видно поимённо.
3. Нет данных из трекера → `[НЕДОСТУПНО]`, не оценивай delivered «на глаз».
