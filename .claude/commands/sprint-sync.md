## Использование

```
/sprint-sync <sprint>
```

Пример: `/sprint-sync 2026Q3-S7`

Вход: SPRINT-ROADMAP квартала + ФАКТ прошлого спринта. Выход: `{sprint_workspace}/sprint-context.md`.

## Важно

**Роль: Context Builder + ресинк.** Стадия 0 пайплайна спринта. Главная задача — **пересинхронизировать контекст** (не поменялось ли видение/цели/команда/ресурсы) и собрать актуальную базу для планирования. Без неё `/sprint-goal` запрещён.

> Это точка синхронизации всех контекстов — самая ценная часть процесса. Опрос строго по одному вопросу за раз.

---

## Инструкция для LLM

### Этап 1: Загрузка роли и стандартов
1. Прочитай `skills/sprint-planner/SKILL.md`.
2. Прочитай `skills/sprint-planner/resources/sync_questions.md` — сценарий диалога.
3. Прочитай `.claude/domain-profile.md` — пути, `cadence`, `capacity`, трекер.

### Этап 2: Чтение входов (параллельно, ДО опроса)
4. Прочитай `{sprint_roadmap_doc}` — какие KR двигаются в этом спринте (срез).
5. Найди и прочитай ФАКТ прошлого спринта в `{execution_root}` — незакрытые истории = carryover-кандидаты. Нет файла → отметь, спроси PO.
6. Собери ростер и ёмкости: `GROUND/NEXUS/team/*` (person-узлы) + `capacity` профиля.
7. Прочитай `{okr_output_doc}` (секции активных OBJ/KR) — образы результата KR.

### Этап 3: JIRA (если трекер доступен)
Через MCP трекера (`tracker.mcp` из профиля), только чтение:
- Активные эпики/истории по KR этого спринта.
- Открытые баги/техдолг (кандидаты во внеплановые).
- Недоступен (нет VPN) → `[НЕДОСТУПНО: JIRA]`, не выдумывать беклог.

### Этап 4: Ресинк-диалог (по одному вопросу за раз)
Задавай вопросы Q1–Q7 из `sync_questions.md` последовательно, про **дельту** к подтянутому состоянию. После каждого ответа — следующий. Фиксируй каждое изменение с источником. PO может сказать «без изменений» / «достаточно».

### Этап 5: Сборка sprint-context.md
```markdown
# Sprint Context: {sprint}

## 1. Актуальные KR спринта (источник: SPRINT-ROADMAP + дельта PO)
[KR + срез на спринт + изменения относительно roadmap]

## 2. Команда и ёмкость (источник: NEXUS/team + профиль + PO)
| Исполнитель | Роль | Ёмкость (SP) | Отпуска/недоступность |

## 3. Carryover из прошлого спринта (источник: ФАКТ)
[Незакрытые истории → перенос/снятие]

## 4. Внеплановые работы (источник: PO + JIRA)
[Техдолг/баги/влёты/онбординг + теги]

## 5. Коммитменты и риски спринта (источник: PO)
[Дедлайны/демо/релизы + риски]

## 6. Состояние эпиков JIRA (если доступно)
[Список или [НЕДОСТУПНО]]

## 7. Открытые вопросы [УТОЧНИТЬ]
```

### Этап 6: Сохранение + STOP
- Создай папку `{sprint_workspace}/`, сохрани через `obsidian.vault_write` в `sprint-context.md`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Sprint Context готов: {sprint_workspace}/sprint-context.md
  KR спринта: [список]. Изменений к roadmap: N.
  Команда: M чел, ёмкость ~X SP. Отпуска: [кто].
  Carryover: K историй. Внеплановые: L.
  ── СТОП ── PO: проверьте контекст, добавьте пропущенное.
  Дальше → /sprint-goal {sprint}
  ```

## Запреты
1. НЕ формулируй Sprint Goal/истории — это `/sprint-goal` и `/sprint-decompose`.
2. Задавай по одному вопросу. Не вываливай всё сразу.
3. Факт неизвестен → `[УТОЧНИТЬ у {кого}]`. JIRA недоступна → `[НЕДОСТУПНО]`.
4. НЕ выдумывай ёмкости/отпуска/состав беклога.
