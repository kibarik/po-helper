## Использование

```
/sprint-decompose <sprint>
```

Вход: `sprint-goal.md` + `sprint-context.md`. Выход: `{sprint_workspace}/sprint-stories.md`.

## Важно

**Роль: Story Designer. Это вопрос WHAT из Scrum-планирования.** Разложить каждый Sprint Goal на конкретные истории с образом результата, и отдельно собрать внеплановые работы. **Без исполнителей и SP** — они появятся на стадии `/sprint-load`.

---

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `skills/sprint-planner/SKILL.md` (принципы 2, 3, 4).
2. Прочитай `skills/sprint-planner/resources/sprint_standards.md` — теги историй, колонки таблицы, конвенция образа результата/действия.
3. Прочитай `{sprint_workspace}/sprint-goal.md` и `sprint-context.md`.
4. Если есть привязки — прочитай `{kr_epic_map_doc}` (эпики KR) и доступные JIRA-истории (только чтение).
5. **BFT-ярус (если инстанцирован — см. `GROUND/NEXUS/_registry.yaml`):** прочитай `system` (узлы `component` — сервисы/БД/интеграции домена + `team_owner`) для точной декомпозиции по компонентам и `risk` (узлы `risk`) для тегов `[RISK]`. Нет Нексуса → пропусти, декомпозируй по context-pack/JIRA.

### Этап 1a: Reuse-first сверка с беклогом (🚪 gate — до любого NEW)
**Reuse-first: ничего не помечать NEW, пока не проверен беклог.** По каждому требованию Sprint Goal / «образа результата PO» (доски) — поиск в беклоге через MCP трекера (`text ~`, все `tracker.projects`, оба проекта при кросс-проекте):
- **REUSE** — есть активная задача под требование → взять её ключ.
- **NEW** — только если активного дубля нет.
- **activity-без-тикета** — работа без задачи в трекере, помечается отдельно.
- Закрытые аналоги **не** переиспользовать, но показывать как контекст.

Трекер недоступен → `[НЕДОСТУПНО: reuse-check]`, все требования как `NEW?` с пометкой «проверить на BUILD». Артефакт — **матрица покрытия** (ниже), вход для `/sprint-build`.

### Этап 2: Декомпозиция плановых историй
Для каждого Sprint Goal разложи KR на истории:
- **История** = `[РОЛЬ]: краткое действие` (BE/FE/SA/QA/DevOps).
- **Образ результата** истории = доказуемое состояние к концу спринта; для технических — `БЫЛО→СТАЛО`.
- **Образ действия** = атомарные шаги с тегами ролей (`[BA]/[SA]/[BE]/[FE]/[QA]/[RELEASE]/[ADR]`), `[BLOCKER]`/`[RISK]`. `[RISK]` заякоривай на `node_id` из `risk`-Нексуса, если история пересекает известный риск (`affects`); шаг по компоненту — на `system`-узел (владелец = `team_owner`).
- **Приоритет** = MoSCoW (Must — без чего Sprint Goal не достигнут).
- **ЭПИК** = ключ из KR-EPIC-MAP или `TBD` (пометить «завести эпик»).
- История > 8 SP по объёму → разбить на несколько.
- Каждая история наследует KR (поле для трассировки).

### Этап 3: Внеплановые работы
Из `sprint-context.md` (раздел 4) собери истории вне OKR в отдельную секцию с тегами `[CARRYOVER]`/`[TECH-DEBT]`/`[BUG]`/`[UNPLANNED]`/`[SPIKE]`. Carryover — первыми.

### Этап 4: Сборка sprint-stories.md
```markdown
# Sprint Stories: {sprint}

## OBJ {N}: {название} — Sprint Goal: {из sprint-goal.md}
| ЭПИК | История | Образ результата | Образ действия | KR | Приоритет |
[без Исполнитель/SP — добавит /sprint-load]

[повтор для каждого OBJ]

## Необходимые внеплановые работы
| ЭПИК | История | Образ результата | Образ действия | Приоритет | Тег |

## Трассировка
| История | KR / внеплановое |
[каждая история ← KR или тег внеплановой; «сирот» быть не должно]

## Матрица покрытия (reuse-first) — вход для /sprint-build
| Требование (доска/KR) | Статус | Ключ JIRA / примечание |
|-----------------------|--------|------------------------|
| {требование} | REUSE / NEW / activity-без-тикета | {ключ или «создать под {история}»} |
[закрытые аналоги — строкой «контекст: {ключ} (закрыт)», не REUSE]
```

### Этап 5: Сохранение + STOP
- Сохрани в `{sprint_workspace}/sprint-stories.md`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Sprint Stories готовы: {sprint_workspace}/sprint-stories.md
  Плановых историй: N (Must M / Should S / Could C). Внеплановых: K.
  Все истории трассированы на KR/внеплановое: да/нет [список сирот].
  Reuse-матрица: REUSE {r} · NEW {n} · activity {a} [или НЕДОСТУПНО].
  ── СТОП ── PO: проверьте состав историй, приоритеты и reuse/NEW.
  Дальше → /sprint-load {sprint}
  ```

## Запреты
1. НЕ назначай исполнителей и SP — это `/sprint-load`.
2. Каждая история ← KR или тег внеплановой. «Сирот» (без KR и без тега) быть не должно.
3. Образ результата ≠ TBD для Must-историй.
4. НЕ выдумывай JIRA-ключи и состав беклога — `TBD`/`[НЕДОСТУПНО]`.
5. Reuse-first: НЕ помечай требование `NEW`, пока не проверен беклог (`text ~`, все `tracker.projects`). Закрытые аналоги — контекст, не REUSE.
