## Использование

```
/summary [транскрипт | путь к .md/.txt файлу]
```

- `[транскрипт]` — сырая расшифровка созвона/встречи (аргумент, путь к файлу или вставка PO). Не передан → попроси PO вставить.

Выход: `.md`-заметка с краткой выжимкой, сохранённая в `{summary_notes}` (дефолт `GROUND/PULSE/summaries`), + превью и заметки по фактуре в чат. STOP — ждать подтверждения PO.

## Важно

**Роль: Стенографист-аналитик.** Разложи транскрипт по формату `#summary`: одна встреча → несколько блоков «Тема». Ценность — в точности терминов, привязке к KR/людям/датам и авторстве позиций, а не в развёрнутых абзацах.

**Главное:** каждый факт ← транскрипт. Чего в транскрипте нет — того нет в саммари. Неизвестное → `[УТОЧНИТЬ]`, не выдумывать.

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `.claude/domain-profile.md` — резолвь `paths.summary_notes` (дефолт `GROUND/PULSE/summaries`), `capacity.roster_source` (дефолт `GROUND/NEXUS/team`), `meta.current_quarter`. Профиль пуст → дефолты + `[УТОЧНИТЬ]`.
2. Прочитай `skills/summary/SKILL.md` и `resources/summary_format.md` (спецификация формата + стиль). Эталон — `examples/ideal_summary.md`.
3. Загрузи ростер: имена/роли из `roster_source` — для сверки участников.

### Этап 2: Приём транскрипта
Возьми `[транскрипт]` (аргумент / путь к файлу / вставка PO). Пусто → попроси вставить. Путь к файлу → прочитай его.

### Этап 3: Шапка
Определи для frontmatter:
- `дата` — из транскрипта. Только время без даты → `currentDate` как заглушка + пометь «дата — заглушка».
- `источник` — созвон / встреча / транскрибация (+ id, если есть).
- `участники` — из транскрипта, сведи с ростером (полные имена/роли). Не сматченное имя не додумывай.
- `теги` — `[summary, <домен по темам>]`.

### Этап 4: Сегментация
Разбей встречу на смысловые темы (обычно 2-4). Одна тема = один блок `#summary`.

### Этап 5: Разложение по формату
На каждую тему (`resources/summary_format.md`):
- `**Тема**` — как **решение / KR / действие**, не как описание.
- `**Проблема**` — опционально, одной фразой, только если есть явная боль.
- `**Ключевое:**` — телеграфные буллеты; мнения — инлайн-атрибуцией («(Осташева)», «Ишманов: есть гипотеза …»); доменная точность вместо общих слов; причинно-следственные ветки «если → …».
- `**Договоренности:**` — конкретно: кто именно что делает к какому сроку / принятое решение.
Привязывай к KR-кодам и датам решений, где они есть в транскрипте.

### Этап 6: Обработка фактуры (zero-hallucination)
- Искажения распознавания (одна сущность под разными написаниями) → свести к единому написанию, запомнить для отчёта.
- KR-коды / сроки / авторы мнений — только из транскрипта. Нет → `[УТОЧНИТЬ]`.

### Этап 7: Сохранение
Собери файл: frontmatter + блоки `#summary`. Запиши в `{summary_notes}/{ГГГГ-ММ-ДД}-{slug}.md` (`slug` — 2-4 слова по главной теме). Файл существует → предложи суффикс или перезапись (спроси PO).

### Этап 8: Отчёт + STOP
```
── Summary сохранён ──
Файл:      {summary_notes}/{дата}-{slug}.md
Темы:      <N> (<короткий перечень>)
Заметки:   <заглушка даты? / сведённые термины — проверь глазами / [УТОЧНИТЬ]-поля>
── СТОП ── PO: проверьте фактуру и термины.
```
Покажи превью заметки в чат.

## Запреты
1. НЕ выдумывать участников, KR-коды, сроки, решения — только из транскрипта или ответа PO.
2. НЕ атрибутировать мнение автору, если в транскрипте не видно, кто сказал.
3. НЕ раздувать: Тема — решение, а не пересказ; буллет — телеграфный, а не абзац. Стиль — `skills/bft-writer/resources/writing_style.md`.
