## Использование

```
/sprint-fact <sprint>
```

Пример: `/sprint-fact 2026Q3-S1`

Вход: `{sprint_output_doc}` (ПЛАН) + финальное состояние спринта в JIRA (read-only) + `{sprint_workspace}/sprint-goal.md`. Выход: `{sprint_fact_doc}` — ФАКТ спринта, вход для `/sprint-sync` следующего.

## Важно

**Роль: Fact Recorder.** Замыкает rolling-цикл: по завершении спринта фиксирует, **что реально произошло** — достигнут ли Sprint Goal, что закрыто/перенесено/снято, фактическая velocity по каждому. Это не ретро-фасилитация, а сухая фиксация факта с якорями (JIRA/ПЛАН). JIRA — **только чтение**, ничего не писать в трекер.

> Каждый факт ← источник (JIRA-статус / ПЛАН / слова PO). Нет источника → `[УТОЧНИТЬ]`. JIRA недоступна → `[НЕДОСТУПНО: JIRA]`, velocity/статусы не выдумывать.

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `skills/sprint-planner/SKILL.md` (принципы 5, 7 — velocity и carryover) и `resources/sprint_standards.md` (модель ёмкости: velocity = SP, реально закрытые за спринт).
2. Прочитай `.claude/domain-profile.md` — `tracker` (MCP, слот/поля), пути `sprint_output_doc`/`sprint_fact_doc`, `capacity`.
3. Прочитай `{sprint_output_doc}` (план: истории, исполнители, плановые SP, Sprint Goal) и `{sprint_workspace}/sprint-goal.md`.

### Этап 2: Снятие факта из JIRA (только чтение)
Через MCP трекера по слоту спринта:
- Статус каждой истории/подзадачи: закрыто / открыто (перенос) / снято.
- Фактически закрытые SP по каждому исполнителю.
- Внеплановое, попавшее в спринт (влёты/баги — чего не было в ПЛАНе).
Недоступно → `[НЕДОСТУПНО: JIRA]`, снять факт со слов PO, пометить `[УТОЧНИТЬ]`.

### Этап 3: Достижение Sprint Goal
По каждому Sprint Goal из плана — вердикт: ✅ достигнут / 🟡 частично / 🔴 не достигнут + доказательство (что демонстрировали, ключи закрытых Must-историй). Без «в целом норм» — только доказуемое.

### Этап 4: Velocity по человеку (для следующего capacity)
Для каждого исполнителя: плановые SP → фактически закрытые SP. Это velocity-точка для `/sprint-sync` следующего спринта (модель ёмкости `sprint_standards.md`). Систематический разрыв план↔факт — отметить (переоценка/недооценка).

### Этап 5: Carryover
Незакрытые истории → список кандидатов на перенос с тегом `[CARRYOVER]` (кто вёл — для назначения в следующем спринте).

### Этап 6: Сборка ФАКТ + сохранение + STOP
```markdown
# {sprint} — ФАКТ спринта

**Период:** {даты} · **Ответственный:** {po_name} · **Статус:** ✅ Завершён

## 🎯 Достижение Sprint Goal
| Sprint Goal (OBJ N) | Вердикт | Доказательство (демо / ключи) |
| {…} | ✅/🟡/🔴 | {…} |

## 📊 Velocity по исполнителю (план → факт)
| Исполнитель | План SP | Факт закрыто SP | Δ | Комментарий |

## ✅ Закрыто
| История | Ключ | SP | KR / тег |

## ↩️ Carryover (кандидаты на перенос)
| История | Ключ | Кто вёл | Остаток SP |

## ✂️ Снято / не начато
| История | Ключ | Причина |

## ⚡ Внеплановое (влетело в спринт)
| История | Ключ | Тег | SP |

## 📝 Отклонения и уроки (кратко)
- {что систематически расходилось план↔факт, что учесть в следующем планировании}
```
- Сохрани через `obsidian.vault_write` → `{sprint_fact_doc}`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  ФАКТ {sprint} готов: {sprint_fact_doc}
  Sprint Goal: ✅ {a} / 🟡 {b} / 🔴 {c}. Velocity: закрыто {X} SP из {Y} плановых.
  Carryover: {k} историй. Снято: {m}. Внеплановое: {n}.
  ── Цикл замкнут. Вход для → /sprint-sync {next-sprint}
  ```

## Запреты
1. JIRA — только чтение. НЕ менять статусы/спринт в трекере (это `/sprint-build`/`/sprint-activate`).
2. Velocity/статусы не выдумывать — из JIRA/ПЛАНа/слов PO, иначе `[УТОЧНИТЬ]`/`[НЕДОСТУПНО]`.
3. Вердикт Sprint Goal — доказуемый (ключи/демо), не «в целом норм».
4. Не раздувать в ретро-протокол — сухой факт + краткие уроки.
