## Использование

```
/okr-deliver <quarter>
```

Вход: `enriched-okr.md` + `validation.md` (🟢/🟡). Выход: публикация в vault + обновление индексов.

## Важно

**Роль: Deliverer.** Сухой прогон → PO подтверждает → записываем. Без явного «ок PO» — никакой записи. JIRA и Confluence — **только чтение** (не создавать задачи без отдельного запроса).

> Аналог bft-pipeline: `/bft-deliver`. STOP перед записью обязателен.

## Инструкция для LLM

### Предусловие: Проверка статуса валидации
1. Прочитай `{okr_workspace}/validation.md`.
2. Если статус 🔴 → **СТОП**, сообщить PO: «Валидация не пройдена. Исправить [список], затем /okr-validate повторно».
3. Если 🟢 или 🟡 → продолжить.

### Этап 1: Загрузка артефактов
1. Прочитай `{okr_workspace}/enriched-okr.md` — финальная таблица.
2. Прочитай `{index_doc}` — текущая структура.
3. Прочитай `{kr_epic_map_doc}` — текущий маппинг.
4. Прочитай `{planning_root}/BEST-PRACTICES-OBJ-KR.md` — структура финального документа.

### Этап 2: Подготовка артефактов (сухой прогон)

**Артефакт 1: OKR-{quarter}.md**
Итоговый документ в формате эталона (`examples/ideal_okr.md`):
```
[Заголовок с проектными целями]
[Регулярные активности]
[Основная таблица OBJ|KR|IMP|Название|Образ результата|Образ действия|Метрики&риски&зависимости]
```

**Артефакт 2: Изменения в INDEX.md**
- Обновить таблицу KR в каждом разделе OBJ (добавить новые KR, обновить образы результата).
- Добавить блок «Приоритизация» если отсутствует: Critical/High/Medium KR.

**Артефакт 3: Изменения в KR-EPIC-MAP.md**
- Добавить строки для новых KR со значением `TODO` в колонке «Эпики JIRA».
- Не трогать существующие привязки.

### Этап 3: Сухой прогон → СТОП для PO

Выведи план публикации:
```
СУХОЙ ПРОГОН — план публикации OKR {quarter}

📄 СОЗДАТЬ:
  {okr_output_doc}
  [Превью: первые 10 строк таблицы]

📝 ОБНОВИТЬ INDEX.md:
  - Добавить KR: [список новых KR]
  - Обновить образы результата: [список KR]

📝 ОБНОВИТЬ KR-EPIC-MAP.md:
  - Добавить строки: [список KR → TODO]

⚠️ Не изменяется:
  - JIRA (задачи, статусы)
  - Confluence (не публикуется автоматически)

── СТОП ── PO: подтвердите публикацию или скорректируйте.
Команда для старта: «да, публикуй» или «ок»
```

### Этап 4: Запись (только после «ок» PO)

В порядке:
1. `obsidian.vault_write` → `{okr_output_doc}`
2. `obsidian.vault_patch` → `{index_doc}` (добавить/обновить KR-секции)
3. `obsidian.vault_patch` → `{kr_epic_map_doc}` (добавить TODO-строки)

### Этап 5: Манифест доставки

```markdown
## Delivery Manifest: OKR {quarter}
**Дата:** {дата}
**Статус валидации:** 🟢/🟡
**Исполнитель pipeline:** PO (`po_name` из профиля)

### Созданные файлы
- {okr_output_doc} — ✅
- {okr_workspace}/context-pack.md — ✅ (стадия 0)
- {okr_workspace}/enriched-okr.md — ✅ (стадия 4)
- {okr_workspace}/validation.md — ✅ (стадия 5)

### Обновлённые файлы
- INDEX.md — добавлено N KR
- KR-EPIC-MAP.md — добавлено M строк TODO

### Следующие шаги
1. Уточнить эпики JIRA для новых KR → обновить KR-EPIC-MAP (TODO → реальные ключи)
2. Показать OKR команде на kick-off спринта
3. Построить roadmap по спринтам: `/sprint-roadmap {quarter}` → затем `/sprint-sync {first-sprint}`
```

### Этап 6: STOP — финал
```
OKR {quarter} опубликован.
Файл: {okr_output_doc}
KR заведены в KR-EPIC-MAP (TODO → нужно уточнить эпики JIRA).
── Цикл планирования завершён ──
Следующий шаг: /sprint-roadmap {quarter} (roadmap по спринтам) → /sprint-sync {first-sprint-id}
```

## Запреты
1. НЕ записывать файлы до «ок PO» — только сухой прогон.
2. НЕ создавать задачи в JIRA автоматически — только читать.
3. НЕ публиковать в Confluence без отдельного запроса PO.
4. НЕ менять существующие привязки эпиков в KR-EPIC-MAP.
