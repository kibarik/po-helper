## Использование

```
/okr-grooming <quarter> <kr>
```

Пример: `/okr-grooming 2026-Q4 1.2`

Вход: зафиксированный якорь (`smart-agreement.md` или БФТ) + `context-pack.md`. Выход: `{okr_grooming_root}/step_{n}/okr_step_{n}.md` (груммированный образ действия) + `research-questions.md`.

## Важно

**Роль: Lead Grooming Facilitator.** Углубляешь проработку утверждённого KR силами **аналитиков (BA/SA), лидов направлений и архитекторов (ADR)**. Инженеры (BE/FE) напрямую не сидят — их объём заходит через лидов. Результат: US-декомпозиция образа действия, реестр вопросов исследования, полный risk-register.

> **🚫 ГРУМИНГ НЕВОЗМОЖЕН БЕЗ ЯКОРНОГО АРТЕФАКТА.** Нет зафиксированного SMART-соглашения или привязанного БФТ → жёсткий STOP (Предусловие). Каждый вопрос исследования опирается на артефакт; вопрос без артефакта-основы = сигнал «нужен новый артефакт», а не повод выдумывать.

> Аналог bft-pipeline: декомпозиция и red-teaming после фиксации образа результата. Это шаг B Фазы 2.

## Инструкция для LLM

### Предусловие: проверка якоря (ЖЁСТКИЙ STOP)
1. Прочитай `.claude/domain-profile.md` — резолв `{okr_grooming_root}`, `{okr_workspace}`, `{bft_store}`, `teams`, `stakeholders`.
2. Найди степ KR `{kr}`: `{okr_grooming_root}/step_{n}/` (по `kr` во frontmatter `smart-agreement.md`).
3. Проверь `step_{n}/artifacts/smart-agreement.md`:
   - файл существует **И** `status: ЗАФИКСИРОВАН` → ОК;
   - `anchor_type: БФТ` + `bft_ref` указывает на финализированный БФТ в `{bft_store}` → ОК.
4. Если якоря нет или `status` ≠ `ЗАФИКСИРОВАН` → **СТОП**, без груминга:
   ```
   🚫 Груминг невозможен для KR {kr} — нет зафиксированного якорного артефакта.
   Запусти: /okr-smart {quarter} {kr}   (или привяжи существующий БФТ)
   Груминг всегда опирается на артефакт. Старт без якоря запрещён.
   ```

### Этап 1: Загрузка якоря и контекста
1. Прочитай якорь: `smart-agreement.md` (образ результата, границы ВНУТРИ/СНАРУЖИ, контекст диалога, БЫЛО→СТАЛО) или БФТ по `bft_ref`.
2. Прочитай `{okr_workspace}/context-pack.md` — риски, зависимости, эпики, внешние команды.
3. Прочитай `skills/okr-planner/resources/okr_standards.md` (секции «Образ действия — US-декомпозиция», «Вопросы исследования», тег `[EXT]`) и `resources/hard_gates.md` (секция «Гейты груминга»).
4. Прочитай эталон `skills/okr-planner/examples/ideal_grooming.md` — формат вывода.

**Граница груминга = границы из якоря.** Не выходи за `ВНУТРИ`; всё из `СНАРУЖИ` — не груммим.

### Этап 2: US-декомпозиция образа действия
Собери образ действия в формате (см. `okr_standards.md`):

```
Общий префикс:
- [BA] Сбор и описание БФТ
- [SA] Первичная аналитика
- [SA] Декомпозиция и разработка

US1: [название user story]
- [BE][EXT] Доработки на стороне [внешняя команда]
- [BE] ...
- [FE][EXT] ...
- [QA] Отладка по US1
- [RELEASE] Отгрузка US1 в Production

US2: [название] ...
```

Правила:
- `[EXT]` — на каждый шаг, исполняемый внешней командой (из «контекст диалога» якоря / `context-pack`).
- Каждый US-блок завершается шагом `[QA] Отладка по US{k}` и `[RELEASE] Отгрузка US{k}`.
- Для незрелых KR (`[RESEARCH]`/`[POC]`) US-декомпозиция может отсутствовать — тогда центр тяжести смещается в «Вопросы исследования» (Этап 3).

### Этап 3: Вопросы исследования (реестр)
Сгенерь кандидаты-вопросы, **каждый с привязкой к артефакту-основе**. Префикс = ответственная роль:
- `ADR#` — архитектурное решение (архитектор);
- `SA#` — системная аналитика (системный аналитик);
- `BA#` — бизнес-аналитика (бизнес-аналитик);
- `DEL#` — delivery/devops/инфраструктура;
- кросс-ссылки `US#` — связь вопроса с user story.

Формат:
```
ADR1: [вопрос]              ← основа: [якорь/БФТ-раздел/context-pack]
SA8: Следствие US7 → [...]  ← основа: [...]
```

**Правило артефакта (G-G2):** если вопрос не опирается ни на якорь, ни на БФТ, ни на context-pack → НЕ добавляй его как обычный вопрос. Вынеси в блок «⚠ Требуют артефакта»:
```
⚠ Требуют артефакта (груминг по ветке заморожен):
- [тема]: нет артефакта-основы → сначала /okr-smart или БФТ по этому подвопросу.
```

Для каждого вопроса пометь, **чей он** (аналитик BA/SA / лид направления / архитектор) — чтобы PO собрал ответы у команды.

### Этап 4: Risk-register
`[RISK]` списком, без потолка. Источники: `context-pack`, контекст диалога якоря, внешние зависимости, Legacy, сетевые/CI-CD/юридические риски. Каждый риск конкретен (что именно может пойти не так), без воды.

### Этап 5: Сборка okr_step_{n}.md
```markdown
---
kr: "{kr}"
quarter: "{quarter}"
step: {n}
anchor: smart-agreement.md | БФТ {bft_ref}
status: НА РЕВЬЮ
---

# Груминг KR {kr} — [название]

## Образ действия
[US-декомпозиция из Этапа 2]

## Вопросы исследования
[реестр из Этапа 3 + блок ⚠ Требуют артефакта, если есть]

## Риски
[risk-register из Этапа 4]
```

Параллельно собери `research-questions.md` — тот же реестр вопросов как backlog (кандидаты в research-таски / БФТ через `/bft-*`), с колонкой «ответственный».

### Этап 6: Сбор ответов у команды (опрос PO)
Для вопросов с известным ответом — PO отвечает (или передаёт ответ команды). Каждый ответ ← источник; вписывается рядом с вопросом. Неотвеченные остаются открытыми в backlog.

### Этап 7: Сохранение + STOP
- Сохрани `okr_step_{n}.md` и `artifacts/research-questions.md` через `obsidian.vault_write`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Груминг KR {kr} готов: step_{n}/okr_step_{n}.md
  US: K. Вопросов исследования: M (открытых: X). Рисков: R.
  ⚠ Требуют артефакта: [список или нет].
  ── СТОП ── PO: проверьте US-декомпозицию, вопросы, риски.
  Следующий KR → /okr-smart {quarter} <kr> затем /okr-grooming {quarter} <kr>
  ```

## Запреты
1. НЕ стартовать груминг без зафиксированного якоря — только жёсткий STOP (Предусловие).
2. НЕ добавлять вопрос исследования без артефакта-основы — выносить в «⚠ Требуют артефакта».
3. НЕ выходить за границы `ВНУТРИ` из якоря; `СНАРУЖИ` не груммить.
4. НЕ выдумывать шаги, внешние команды или риски — только из якоря, context-pack или ответов PO.
5. НЕ записывать финал без явного «ок PO» по содержанию степа.
