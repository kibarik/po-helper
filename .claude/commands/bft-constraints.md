## Использование

```
/bft-constraints <epic_code>
```

Вход: `concept.md` (с вердиктом от `/bft-debate`) + `problem.md` + context-pack. Выход: `constraints.md` (матрица ограничений релиза).

## Важно

**Роль: Constraint Researcher.** Собери набор точных, конкретных, **измеримых** ограничений релиза от стейкхолдеров — рамок среды, которые релиз обязан соблюсти, иначе он не состоится или навредит смежникам. Проблема, которую стадия закрывает: побочные требования внешних стейкхолдеров всплывают в момент выката, потому что они не знали о задаче. Собираем **до черновика**, пока изменения ещё можно заложить в требования.

**Это совместное исследование LLM + PO, не автогенерация.** История напрямую зависит от технической реализации выбранного концепта: нужно точно понимать, как изменение технического компонента влияет на остальные команды. Инструменты:
- **repowise** (техническая аналитика) — blast-radius: какие компоненты меняются, кто от них зависит.
- **People Graph** (Нексус `GROUND/NEXUS/team`, PO-helper) — найти ответственного PO/команду каждого затронутого компонента и бизнес-эффекта.
- **PO (человек)** — подтверждает негативный эффект, переводит гипотезы в факты, добавляет ограничения вне кода (регламенты, календарь релизов, договорённости).

**Ключевое разделение (ядро стадии):** ограничения делятся на два класса, смешивать запрещено —
- **Ф (фактологически-ограничивающее)** — доказанный негативный бизнес-эффект при нарушении, есть якорь.
- **Г (гипотетически-ограничивающее)** — гипотеза «если не учесть X, где-то может пойти так», не доказана.

Гипотеза не выдаётся за факт. Модель, измеримость, атрибуция и якоря — `resources/constraint_rules.md`.

> Стадия между `/bft-debate` и `/bft-draft`: концепт уже выбран (техническая форма известна → blast-radius считается), требования ещё не написаны (ограничения успеют войти в НФТ/Границы/Зависимости).

## Инструкция для LLM

### Этап 1: Загрузка роли и входа
1. Прочитай `skills/bft-writer/SKILL.md`.
2. Прочитай `resources/constraint_rules.md` — модель ограничений (Ф/Г, измеримость, атрибуция, якоря, проекция в БФТ).
3. Прочитай `resources/anchor_rules.md` — ранги якорей.
4. Входы (все из `bft_documentation/<epic>/artefacts/`): `concept.md` (выбранный концепт + вердикт дебатов), `problem.md` (затронутые компоненты / blast radius), `bft-context-pack.md`.

### Этап 2: Техническая карта изменения (repowise)
По затронутым компонентам выбранного концепта (из `concept.md` «Влияние» + `problem.md` «Затронутые компоненты»):
- `get_context` — зависимости изменяемого модуля (кто с ним связан).
- `search_codebase` / `get_answer` — кто вызывает компонент, где ещё используется контракт/событие/поле.
- `get_risk` — hotspot и риск изменения.
- `get_why` — инварианты, которые нельзя ломать (ADR/история).

Итог: список затронутых компонентов + зависящих от них команд/систем. Каждый — кандидат в ограничение. repowise недоступен → собери из `problem.md`/context-pack, недостающее → `[УТОЧНИТЬ]`, не выдумывай.

### Этап 3: Атрибуция (People Graph)
Для каждого затронутого компонента и каждого потенциального бизнес-эффекта найди ответственного через Нексус `GROUND/NEXUS/team`: сопоставь с `influence_zones` / `contact_for` / `expertise_topics` person-узлов. Верни `full_name` + `role_title` + канал связи. Нет совпадения в People Graph → `[УТОЧНИТЬ у {кого}]`, владельца не выдумывай.

### Этап 4: Формулировка ограничений (измеримо)
Для каждого кандидата сформулируй ограничение: число / порог / дата / булев инвариант (примеры — `constraint_rules.md`). Не «не сломать биллинг», а «событие X сохраняет поля {a,b,c}; контракт vN принимается до 31.12». Рамка известна, число нет → «(предварительно)» / «уточняется у X».

### Этап 5: Классификация Ф / Г
Для каждого ограничения определи класс:
- **Ф** — есть якорь (документированная зависимость / регламент / инвариант в коде) И доказанный негативный бизнес-эффект. Якорь R1/R2/R3.
- **Г** — гипотеза blast-radius, эффект предполагаемый, не подтверждён. Якорь = blast-radius `symbol_id` из repowise + пометка «гипотеза».

Не смешивай. Сомнение между Ф и Г → это Г (пока владелец не подтвердил).

### Этап 6: Совместная сверка с PO (STOP-точка исследования)
Стадия совместная — вынеси PO на подтверждение:
- Гипотезы (Г): какие проверить с владельцем в первую очередь (перевод Г → Ф требует нового якоря-подтверждения; снятие — фиксируем «кто, почему»).
- Факты (Ф): подтверждает ли PO негативный эффект и его величину.
- Пробелы: ограничения вне кода, которых LLM не видит (календарь релизов, договорённости с внешними стейкхолдерами, комплаенс).

### Этап 7: Сохранение + STOP
- Сохрани `bft_documentation/<epic>/artefacts/constraints.md`: матрица ограничений (формат — `constraint_rules.md`, поля `ОГР-N | Класс | Формулировка | Затронутый компонент | Ответственный | Негативный эффект | Как проверить | Статус согласования | Якорь`), сгруппированная по классам Ф и Г.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Ограничения релиза готовы: bft_documentation/<epic>/artefacts/constraints.md
  Ф (факт): N · Г (гипотеза): M · [УТОЧНИТЬ]: K
  Проверить с владельцами (гипотезы → факты): <кратко, кого спросить>
  ── СТОП ── PO: подтвердите факты, проверьте гипотезы. Дальше — /bft-draft <epic>.
  ```

## Запреты
1. НЕ выдавай гипотезу (Г) за факт (Ф) — без нового якоря-подтверждения перевод Г → Ф запрещён.
2. НЕ выдумывай затронутые компоненты (только repowise / problem.md), владельцев (только People Graph) и негативные эффекты (только подтверждённые PO/якорем). Нет данных → `[УТОЧНИТЬ]`.
3. НЕ формулируй БТ/ФТ/НФТ — это `/bft-draft`. Здесь только ограничения среды.
4. Ограничения — измеримые (число/порог/дата/инвариант), не «быстро/надёжно/не сломать».
5. Соблюдай стиль `skills/bft-writer/resources/writing_style.md` для `constraints.md`: деловая проза без AI-стоп-слов, без эмодзи в теле, без раздутого bold, тире умеренно. Стиль генератора задаёт стиль вывода.
