## Использование

```
/req-context <request_id> [краткая суть запроса]
```

- `<request_id>` — короткий слаг запроса (напр. `REQ-2026-014` или `intake-billing-export`).
- `[краткая суть]` — одна фраза о запросе, если уже известна (опц.).

Выход: `<intake_workspace>/<request_id>/artefacts/context-pack.md`.

## Важно

**Роль: Context Builder.** Твоя задача — **грунтануть агента доменом команды**, чтобы следующее интервью шло *информированно* (принцип 1 SKILL). Это не приём заявки — это подготовка к умной распаковке: что команда уже умеет, нет ли похожего/готового решения, какие смежные системы.

**Транспортная агностичность:** контекст собирается про домен команды, независимо от того, кто запустит интервью дальше.

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `.claude/domain-profile.md` — резолвь `paths.intake_workspace`, `teams`, `tracker`, `cortex`, глоссарий. Профиль не заполнен → дефолт + `[УТОЧНИТЬ]`.
2. Прочитай `skills/request-intake/SKILL.md` (принципы 1, 9) и `resources/anchor_rules.md`.
3. Создай папку `<intake_workspace>/<request_id>/artefacts/`.

### Этап 2: Грунтовка доменом (GROUND + CORTEX)
Прочитай релевантное по теме запроса:
- **GROUND NEXUS** (`GROUND/NEXUS/`) — узлы знаний команды (product/customer/team/…): что команда делает, какие решения уже есть.
- **CORTEX** (из `cortex.*` профиля): C1 архитектура (границы/сервисы), BR бизнес-правила, C5 решения/ADR.
- Фиксируй: что нашёл / чего нет (→ `[УТОЧНИТЬ: нет в домене]`).
- Грунтовка тонкая (нет онбординга PAF) → graceful degradation: помечай пробелы `[УТОЧНИТЬ]`, не блокируй.

### Этап 3: Похожие прошлые запросы (анти-дубль)
- Поиск в трекере (если `tracker` доступен) и в `<intake_workspace>` по теме: нет ли уже закрытого/похожего запроса или существующего решения.
- Каждый найденный — с якорем R2 (ключ трекера/путь). Это топливо для роутинга (early-сигнал «дубль/уже есть»).
- Трекер недоступен → `[НЕДОСТУПНО: трекер]`, продолжай с GROUND/CORTEX.

### Этап 4: Сборка context-pack
Собери `artefacts/context-pack.md`:
1. **Шапка:** request_id, суть (если есть), дата, статус источников.
2. **Домен команды** — что команда делает в зоне запроса (NEXUS/CORTEX, R2).
3. **Существующие решения** — есть ли уже `Y`, закрывающий часть запроса (для информированного интервью).
4. **Похожие запросы** — найденные дубли/смежники (R2).
5. **Смежные системы/команды** — что затрагивается (C1).
6. **Зацепки для интервью** — 3-5 пунктов, что проверить вслух у запрашивающего («у нас есть Y — это про то же?»).
7. **Открытые вопросы** — `[УТОЧНИТЬ]`.

### Этап 5: Отчёт + STOP
```
Контекст собран: <intake_workspace>/<request_id>/artefacts/context-pack.md
Домен: <кратко>. Существующее решение: <есть Y / нет>. Похожие запросы: <N>.
Зацепки для интервью: <2-3 пункта>.
── СТОП ── PO: проверьте контекст. Дальше — /req-interview <request_id>.
```

## Запреты
1. НЕ начинать интервью здесь — только грунтовка домена.
2. НЕ выдумывать содержимое домена → `[УТОЧНИТЬ]`.
3. Каждый факт о домене ← якорь R2. Стиль — `skills/bft-writer/resources/writing_style.md` (деловая проза, без декоративных эмодзи).
