## Использование

```
/req-context <request_id> [краткая суть запроса]
```

- `<request_id>` — короткий слаг запроса (напр. `REQ-2026-014` или `intake-billing-export`).
- `[краткая суть]` — одна фраза о запросе, если уже известна (опц.).

Выход: `<intake_workspace>/<request_id>/artefacts/context-pack.md`.

## Важно

**Роль: Context Builder.** Твоя задача — **грунтануть агента доменом команды**, чтобы следующее интервью шло *информированно* (принцип 1 SKILL). Это не приём заявки — это подготовка к умной распаковке: что команда уже умеет, нет ли похожего/готового решения, какие смежные системы.

**Транспортная агностичность:** контекст собирается про домен команды, независимо от того, кто запустит интервью дальше.

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `.claude/domain-profile.md` — резолвь `paths.intake_workspace`, `teams`, `tracker`, `cortex`, глоссарий. Профиль не заполнен → дефолт + `[УТОЧНИТЬ]`.
2. Прочитай `skills/request-intake/SKILL.md` (принципы 1, 9) и `resources/anchor_rules.md`.
3. Создай папку `<intake_workspace>/<request_id>/artefacts/`.

### Этап 2: Грунтовка доменом (GROUND NEXUS + CORTEX)
Прочитай реестр `GROUND/NEXUS/_registry.yaml`, затем по теме запроса — релевантные Узлы **ключевых Нексусов intake→БФТ** (матрица — `sa_documentation/nexus_process_map.md`, колонка `/req-context`). Эти шесть Нексусов и есть каркас грунтовки запроса:

| Нексус | что берём для контекста запроса |
|---|---|
| `problem` | бизнес-метрика под угрозой, выгодоприобретатель, цена бездействия — чтобы отличить проблему от предложенного решения |
| `requester-domain` | бизнес-логика и KPI команды-заказчика — их специфическая боль vs платформенная потребность |
| `system-landscape` | bounded contexts, существующие API-контракты, что уже есть vs строить заново |
| `ownership` | владельцы затронутых доменов, кого согласовывать, скрытые стейкхолдеры |
| `precedents` | обсуждался ли похожий запрос, почему отклоняли, связанный техдолг (топливо анти-дубля, Этап 3) |
| `compliance` | security/legal-ограничения, NFR-бейзлайн, обязательный шаблон БФТ |

- Каждый факт из Нексуса ← якорь `R2` (путь `GROUND/NEXUS/<slug>/<node>.md`), учитывай CP и `ripeness` Узла (`wilting` → `⚠️ верифицировать`).
- **CORTEX** (из `cortex.*` профиля): C1 архитектура (границы/сервисы), BR бизнес-правила, C5 решения/ADR — дополняет Нексусы слоем знаний.
- Фиксируй: что нашёл / чего нет (→ `[УТОЧНИТЬ: нет в <slug>]`).
- Нексус пуст (онбординг не наполнил) → graceful degradation: помечай пробелы `[УТОЧНИТЬ]`, не блокируй.

### Этап 3: Похожие прошлые запросы (анти-дубль)
- Поиск в трекере (если `tracker` доступен) и в `<intake_workspace>` по теме: нет ли уже закрытого/похожего запроса или существующего решения.
- Каждый найденный — с якорем R2 (ключ трекера/путь). Это топливо для роутинга (early-сигнал «дубль/уже есть»).
- Трекер недоступен → `[НЕДОСТУПНО: трекер]`, продолжай с GROUND/CORTEX.

### Этап 4: Сборка context-pack
Собери `artefacts/context-pack.md` (блоки грунтуются Нексусами из Этапа 2):
1. **Шапка:** request_id, суть (если есть), дата, статус источников.
2. **Домен заказчика** — бизнес-логика/KPI команды-заказчика (`requester-domain`, R2): их боль vs платформенная потребность.
3. **Существующие решения** — есть ли уже `Y`, закрывающий часть запроса (`system-landscape` + CORTEX C1, для информированного интервью).
4. **Похожие запросы / прецеденты** — найденные дубли/смежники и прошлые отказы (`precedents`, R2 трекер/путь).
5. **Смежные системы/команды и владельцы** — что затрагивается и кто владелец/согласующий (`system-landscape` + `ownership`, C1).
6. **Ограничения** — security/legal/NFR, обязательный шаблон БФТ (`compliance`) — если применимо.
7. **Зацепки для интервью** — 3-5 пунктов, что проверить вслух у запрашивающего («у нас есть Y — это про то же?», «какая метрика страдает?» — `problem`).
8. **Открытые вопросы** — `[УТОЧНИТЬ]`.

### Этап 5: Отчёт + STOP
```
Контекст собран: <intake_workspace>/<request_id>/artefacts/context-pack.md
Домен: <кратко>. Существующее решение: <есть Y / нет>. Похожие запросы: <N>.
Зацепки для интервью: <2-3 пункта>.
── СТОП ── PO: проверьте контекст. Дальше — /req-interview <request_id>.
```

## Запреты
1. НЕ начинать интервью здесь — только грунтовка домена.
2. НЕ выдумывать содержимое домена → `[УТОЧНИТЬ]`.
3. Каждый факт о домене ← якорь R2. Стиль — `skills/bft-writer/resources/writing_style.md` (деловая проза, без декоративных эмодзи).
