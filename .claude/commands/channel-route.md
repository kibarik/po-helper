## Использование

```
/channel-route [текст входящего]
```

- `[текст входящего]` — сообщение/выдержка из чата/письма/расшифровки созвона. Не передан → попроси PO вставить.

Выход: meta-блок разметки + рекомендация роутинга в чат (STOP, роутинг не исполняется молча).

## Важно

**Роль: Intake Dispatcher.** Когда информация поступает в систему — **запроси источник** и укажи метаинформацию (канал, стейкхолдеры, тема, участок системы, цель, тип сигнала), затем порекомендуй, куда это направить. Это front door перед `request-intake`.

**Главное:** не выдумывай источник. Канал не определяется → спроси PO «Откуда это?». Неизвестно → `[УТОЧНИТЬ источник]`, STOP.

## Инструкция для LLM

### Этап 1: Загрузка
1. Прочитай `.claude/domain-profile.md` — резолвь `paths.channels_nexus` (дефолт `GROUND/NEXUS/channels`), `paths.intake_workspace`, `tracker`, `cortex.decisions`, `stakeholders`.
2. Прочитай `skills/info-channels/SKILL.md` и `resources/routing_rules.md` (ранги якорей + роутинг-матрица).
3. Загрузи реестр каналов: frontmatter всех `chan-*.md` (`platform`/`handle`/`topics`/`stakeholders`/`signal_types`/`ingest_action`).

### Этап 2: Приём входящего
Возьми `[текст входящего]` (аргумент или вставка PO). Пусто → попроси вставить сообщение.

### Этап 3: Определить источник (R1/R2 или вопрос)
Сопоставь входящее с реестром по `platform`/`handle`/`topics`/`stakeholders`/упоминаниям.
- Одно совпадение → канал определён (R2).
- Неоднозначно/нет совпадения → **спроси: «Откуда это? (канал/источник)»**. Ответ PO = якорь.
- Названный канал не в реестре → предложи `/channel-map <slug>`, продолжай с меткой `[новый канал — разметить]`.
- Отказ/неизвестно → `[УТОЧНИТЬ источник]`, STOP (не роутить вслепую).

### Этап 4: Разметка (meta-блок)
Заякори каждое поле (`routing_rules.md` §B):
```
Источник:      <chan-id> (<platform> <handle>)   [R2 / ответ PO]
Стейкхолдеры:  <node_ids из канала/упоминаний>    [R2]
Тема:          <тема>                             [R1 цитата]
Участок:       <system_areas>                     [R2] / [УТОЧНИТЬ]
Цель:          <goals>                            [R2] / [УТОЧНИТЬ]
Тип сигнала:   <requirement|bug|decision|feedback|status|risk>  [R3 — проверить, если не очевидно]
```
Несколько сигналов в одном входе → отдельная строка meta на каждый.

### Этап 5: Роутинг (матрица §C)
По `signal_type` и `ingest_action` канала:
- **requirement/change-request** → `/req-context <request_id>` (front door intake); meta канала = источник (R2). Предложи `request_id`-слаг.
- **decision** → `cortex.decisions` (ADR-реестр).
- **bug** → трекер (`tracker` из профиля); участок из `system_areas`.
- **risk** → OKR/release-риск (привязка к `goals`).
- **feedback/status** → FYI / `NEXUS/customer`.
- источник неизвестен → `[УТОЧНИТЬ источник]`.

### Этап 6: Отчёт + STOP
```
── Разметка входящего ──
<meta-блок>
Рекомендация: <куда роутить + следующая команда, напр. «/req-context billing-tariff-export»>
── СТОП ── PO: подтвердите источник и роутинг. Канал не размечен → /channel-map <slug>.
```

## Запреты
1. НЕ выдумывать источник/канал — спросить PO или `[УТОЧНИТЬ источник]`.
2. НЕ приписывать стейкхолдеру то, чего нет в сообщении/реестре (R4 запрещён).
3. НЕ исполнять роутинг молча — показать PO, ждать. Стиль — `skills/bft-writer/resources/writing_style.md`.
