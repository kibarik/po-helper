## Использование

```
/okr-key-results <quarter>
```

Вход: `objectives.md` (из `/okr-objectives`). Выход: `key-results.md` (KRs + IMP + образ результата).

## Важно

**Роль: KR Designer.** Для каждого утверждённого OBJ — сгенерируй 2-6 KR: конкретные результаты-задачи с IMP, тегами, образом результата и командами. Не выбирай за PO — давай варианты с обоснованием IMP.

> Аналог: `/bft-concept` в bft-pipeline. KR = конкретизация OBJ через проверяемые результаты.

## Инструкция для LLM

### Этап 1: Загрузка контекста
1. Прочитай `skills/okr-planner/SKILL.md`, `resources/okr_standards.md`.
2. Прочитай `{okr_workspace}/context-pack.md` (коммитменты, риски, эпики JIRA).
3. Прочитай `{okr_workspace}/objectives.md` (утверждённые OBJ).
4. Прочитай `{kr_epic_map_doc} (предыдущий квартал)` — какие эпики уже привязаны.

### Этап 2: Генерация KR-кандидатов (для каждого OBJ)

Для каждого OBJ (по очереди):

**Шаг 2а: Анализ что нужно сделать**
Из context-pack + objectives.md — выпиши конкретные изменения/результаты, которые нужны для достижения OBJ.

**Шаг 2б: Формулировка KR**
Формула: `[Действие] [Чей продукт] на/через [Канал] минуя [Система]`

Спектр KR на OBJ:
- Основные (интеграция, фича, миграция) — без тега
- Исследования (`[RESEARCH]`) — нет ясности в решении
- PoC (`[POC]`) — нужна проверка гипотезы
- Дефекты (`[BUG]`) — критичные баги продакшна

**Шаг 2в: Карточка KR**
```markdown
### KR {OBJ}.{N}: [тег] [Название по формуле]

**IMP:** [число 1-9]
**Теги:** [RESEARCH] / [POC] / [BUG] / (пусто)
**Команды:** [команды из профиля (`teams`) + роли SA/BA/QA]

**Образ результата:**
[Что видит пользователь/стейкхолдер после выполнения]

БЫЛО: [dataflow или состояние системы]
СТАЛО: [dataflow или целевое состояние]

**Зависимости:**
- Блокируется: [KR X.N если есть]
- Блокирует: [KR Y.M если есть]
- Внешние: [команды, системы]

**Источник:** [откуда этот KR: roadmap / PO / JIRA / коммитмент]
**Confidence:** High / Medium / Low
```

### Этап 3: IMP-обоснование

После карточек — свести в таблицу и обосновать IMP:

```markdown
## Предварительное IMP-ранжирование

| KR | Название | IMP | Обоснование IMP |
|----|---------|-----|----------------|
| 1.1 | ... | 9 | Коммитмент перед стейкхолдером (`stakeholders`) на дату |
| 1.2 | ... | 9 | Стратегический коммитмент август 2026 |
| 1.3 | ... | 6 | Важно, но объём неизвестен → Medium |
```

Предупреди если IMP=9 более чем у 2 KR: «Больше 2 Critical KR — нет фокуса. Предлагаю пересмотреть IMP для [список]».

### Этап 4: Вопросы PO по неясным KR

Для каждого KR с `confidence = Low` или пустым образом результата — задай уточняющий вопрос PO:

«KR [X.N] "[название]": образ результата пока TBD. Что должно быть видно в конце квартала — план/декомпозиция или рабочая интеграция в проде?»

После ответа — заполнить образ результата.

### Этап 5: Сохранение + STOP
- Сохрани через `obsidian.vault_write` в `{okr_workspace}/key-results.md`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Key Results готовы: {okr_workspace}/key-results.md
  OBJ 1: N KR (Critical: X, High: Y, Medium: Z)
  OBJ 2: N KR (...)
  Итого: N KR. [УТОЧНИТЬ]: M пунктов.
  ── СТОП ── PO: проверьте KR, IMP, образы результата.
  Что менять? → обновляю key-results.md.
  После утверждения → /okr-debate {quarter}
  ```

## Запреты
1. НЕ заполняй "Образ действия" — это задача `/okr-enrich`.
2. НЕ выбирай финальный состав KR — PO утверждает.
3. Образ результата без источника → `[УТОЧНИТЬ]`, не придумывай.
4. Нумерация KR: если конфликт → поставь `⚠?` и укажи «нужно уточнить у PO».
