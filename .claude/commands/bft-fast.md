## Использование

```
/bft-fast <источник> [epic_slug]
```

**Параметры:**
- `<источник>` — путь к `#summary`-заметке, путь к файлу транскрипта или инлайн-текст. Пусто → команда просит указать.
- `[epic_slug]` — опц., имя для файла-копии; не задан → slug из главной темы Summary (`{ГГГГ-ММ-ДД}-{slug}`).
- `--no-deep` — не форкать фоновую deep-проработку (по умолчанию форкается: RICH-контекст → параллельно письму, THIN → следом, seed=письмо).

## Примеры

```
/bft-fast GROUND/PULSE/summaries/2026-07-03-direct-faq.md
/bft-fast .claude/skills/bft-fast/examples/golden_summary.md
/bft-fast "сырой транскрипт встречи ..." direct-faq
```

## Важно

**Роль: PO-аналитик быстрого прохода.** Результат — **одно письмо в чат**, пересылаемое коллегам как есть. Не полный БФТ-документ. Глубина (полный сбор контекста, deep-проработка) по умолчанию форкается автоматически в фоне (этап 7, `bft-deep-swarm`) сразу после письма; `--no-deep` отключает форк — тогда это отдельный ручной шаг PO после ответов на вопросы письма, вне этой команды.

**Анти-правило:** ничего не выдумывать. Факт без источника в Summary → «Открытый вопрос», не утверждение.

## Инструкция для LLM

### ФОРМАТ ВЫВОДА — копируй структуру точно

Письмо — ровно 4 блока, строго в порядке. Затем вложение.

```
Цель: <1-2 строки: кто хочет что, зачем, ключевой enabler>

How to demo:
1. <шаг от лица роли-приёмщика; дизамбигуация «X, не Y» в скобках, если сказана>
2. …

Открытые вопросы:
* <ФИО> — <вопрос кратко>
* <ФИО> — <вопрос кратко>
(плоский список; БЕЗ заголовков-групп `ФИО:`, БЕЗ чекбоксов `- [ ]`; владелец неясен → `* [кому?] — вопрос`)

Границы: <in-scope> (<out-of-scope> — не входят в зону БФТ).
Дополнительно, если возможно: <nice-to-have, если было>.
<операционные заметки: сроки/эффекты>

Вложение — таблица требований на согласование границ:

Персоны (домен знаний):

| ФИО | Роль | Департамент |
|---|---|---|
| … | … | … |

Требования:

| ID | ASIS (сейчас) | TOBE (после) | Связанные | Источник (цитата) |
|---|---|---|---|---|
| БТ-1 | … | … | … | ФИО: «…» |
(порядок строк: БТ → ПТ → ИТ → ФТ → НФТ)
```

### СТОП-ГАЛЛЮЦИНАЦИИ — на брифинге критично, не додумывать

- **Департамент/роль не сказаны на встрече → `[не указано]` / `[роль не подтверждена]`.** НЕ писать «Product», «Engineering», «QA» из головы.
- **Технологии/алгоритмы (BM25, TF-IDF, типы кухни, фильтры), которых НЕ было в репликах → НЕ добавлять.** Вопрос заводится, только если сам участник его поднял.
- **Таймстемпы НЕ цитировать.** Колонка «Источник» = текст реплики: `ФИО: «дословно»`.
- **ASIS не проговорён → `[ASIS не озвучен]`.** НЕ реконструировать «как оно наверное сейчас».
- **out-of-scope (следующий квартал / «не делаем») → только в «Границы».** НЕ в How to demo, НЕ в таблицу требований.
- **Блок «Цель» обязателен.** Вопросы — только `* ФИО — вопрос` (не группировать, не чекбоксить).

### САМОПРОВЕРКА ПЕРЕД ВЫВОДОМ (пройди каждый пункт, иначе переделай)

1. Первая строка письма — `Цель: …`? Если нет — добавь Цель, не начинай с «How to demo».
2. Блок «Открытые вопросы» — каждая строка вида `* ФИО — вопрос`? Нет заголовков-групп `ФИО:` и нет `- [ ]`? Если есть — переписать в плоский список.
3. В «Персоны» департамент = `[не указано]` там, где не звучал? Роль не в ростере и не названа → `[роль не подтверждена]`? Нет выдуманных «Product/Engineering»?
4. Каждая строка таблицы требований имеет **реальную цитату** в «Источник»? Строка с `[не указано]`/пустым источником — **удалить** (нет цитаты — нет требования).
5. «Связанные» — только ID (`БТ-1`, `ПТ-1`), не свободный текст, не `[не указано]` (нет связи → пустая ячейка)? ID вида `БТ-1` (без нулей `БТ-001`)?
6. ASIS без реплики = `[ASIS не озвучен]`? Нет реконструкции «как наверное сейчас»?
7. В «How to demo» нет шагов про следующий квартал/будущие итерации («в Q4 будет…») и нет out-of-scope? Всё это — только в «Границы», не в демо и не в таблице?
8. В вопросах нет выдуманных технологий (BM25, кухни, фильтры), которых не поднимали участники?

### Этап 1: Загрузка
1. `skills/bft-fast/SKILL.md` (роль, принципы, этапы, резолв конфига).
2. `skills/bft-fast/resources/`: `extraction_schema.md`, `demo_synthesis.md`, `letter_format.md`, `requirements_table.md`.
3. Голос: `skills/bft-writer/resources/writing_style.md`.
4. Резолв конфига из `.claude/domain-profile.md` (нет → `domain-profile.template.md` + `[УТОЧНИТЬ]`).

### Этап 2: resolve-source
Определить форму `<источник>` (СТ-2). Транскрипт → внутренняя выжимка до `#summary`. Пусто → спросить путь/текст, не выдумывать. Путь указан, но файл не читается/не найден → сообщить ошибку пути и попросить корректный источник; не эмулировать содержимое.

### Этап 3: extract
Извлечь сущности по `extraction_schema.md`; каждая с source; участники сверить с ростером.

### Этап 4: synth
Построить `Цель` (кто/что/зачем + enabler, 1-2 строки), `How to demo` (demo_synthesis.md; дизамбигуации «X, не Y» в скобках у шагов), `Открытые вопросы` (атрибуция по стейкхолдерам; неясно → `[кому?]`; No Silent Skip), `Границы` (in/out + тир «Дополнительно, если возможно» + заметки).

### Этап 5: assemble + gate
Собрать письмо по `letter_format.md` (порядок блоков, без обвязки) + вложение по `requirements_table.md`: блок персон (ФИО/Роль/Департамент) + таблица требований ASIS/TOBE (порядок БТ→ПТ→ИТ→ФТ→НФТ, каждая строка ← цитата, без обогащения). Гейт: утверждение письма ← source или вопрос; строка таблицы ← непустая цитата; роль/департамент ← ростер/сказанное или `[не указано]`.

### Этап 6: emit
Вывести письмо в чат (готовое к пересылке), затем вложение — таблицу требований отдельным блоком. Опц. копия в `{bft_store}/{epic_slug}/letter.md` (письмо) и `{bft_store}/{epic_slug}/requirements.md` (таблица). Финал: по умолчанию (без `--no-deep`) — строка о том, что фоновый deep-форк (этап 7) уже запущен, уведомление придёт по завершении; при `--no-deep` — строка о развилке: «Ответы получены — запусти глубину отдельно».
