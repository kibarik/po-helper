## Использование

```
/okr-enrich <quarter>
```

Вход: `key-results.md` (после `/okr-debate`). Выход: `enriched-okr.md` — полная таблица с образом действия + метриками.

## Важно

**Роль: PO + Architect.** Заполнить колонки «Образ действия» и «Метрики&риски&зависимости» для каждого KR. Работать итеративно с PO: сначала образ действия, потом метрики. Опираться только на факты из context-pack и ответы PO.

> Это самая трудоёмкая стадия. KR с IMP ≥ 7 требуют детального образа действия. Для TBD-KR — минимальный образ или честный `[УТОЧНИТЬ]`.

## Инструкция для LLM

### Этап 1: Загрузка контекста
1. Прочитай `skills/okr-planner/SKILL.md`, `resources/okr_standards.md`.
2. Прочитай `{okr_workspace}/key-results.md` (KR после дебатов).
3. Прочитай `{okr_workspace}/context-pack.md` (риски, зависимости, эпики).
4. Прочитай `{landscape.snapshot_doc}` (`landscape-{quarter}.md`, стадия 0.5) — команды вокруг: имена, PO, тип связи. Нужно, чтобы шаги `[EXT]` ссылались на конкретную команду. Файла нет → пометь `[УТОЧНИТЬ: запусти /okr-landscape]`.
4. Если есть прошлые похожие KR — прочитай соответствующие БФТ из `{bft_store}/` для шаблона образа действия.

### Этап 2: Образ действия (для каждого KR с IMP ≥ 5)

Структура «Образ действия»:
```
- [BA] Шаг 1 (сбор требований, БФТ, исследование)
- [SA] Шаг 2 (архитектура, ADR, декомпозиция)
- [ADR] Согласование решения (если нужно)
- [BE] Шаг backend разработки
- [FE] Шаг frontend (если есть)
- [QA] Отладка и тестирование
- [RELEASE] Раскатка (пошаговая / feature-flag / миграция)

[BLOCK] Название блокера (жирный блокер — без него KR не стартует)

Зависимости:
- [RISK] Описание риска
- [BE][EXT] Доработки на стороне внешней команды {название команды из landscape-снимка} (если есть)
```

**Правило [EXT]:** каждый шаг внешней команды помечается `[EXT]` и называет **конкретную** команду из снимка ландшафта (`{landscape.snapshot_doc}`), не абстрактный «внешний». Команды нет в снимке → сначала добавь её через `/okr-landscape`. Это проверяет гейт 13 в `/okr-validate`.

**Для [RESEARCH]-KR образ действия минимальный:**
```
- [BA/SA/PO] Исследование: [что конкретно изучается]
- [PO] Согласование плана с [стейкхолдеры]
- [SA] Декомпозиция (если [RESEARCH] → [POC])
```

**Для [ACTIVITY]-KR:**
```
Регулярная активность. Исправление инцидентов и багов в [система].
```

**Для [BUG]-KR:**
```
- [SA/BE] Диагностика причины: [описание проблемы]
- [BE] Реализация фикса
- [QA] Регрессионный тест
```

### Этап 3: Уточнения у PO (для неясных образов действия)

Для KR с неизвестным образом действия — задать PO по одному вопросу:

«KR [X.N]: как мы планируем двигаться? Знаешь ли уже что нужно сделать первым шагом, или нужно начать с исследования?»

«Кто исполнитель этого KR — наша команда целиком или нужна внешняя команда [Партнёр / FinTech / ...] ?»

### Этап 4: Метрики&риски&зависимости

Для каждого KR — заполнить поле «Метрики&риски&зависимости»:

```markdown
**Оценка:** [диапазон: X–Y недель] / [S/M/L]

**Риски:**
- [Технический риск]
- [Организационный риск]

**Зависимости согласования:**
- [RISK] согласование от [кого]

**Метрики успеха:** (для фич/интеграций)
- [Метрика 1: что измеряем, порог]
```

Для `[RESEARCH]`/`[POC]` — метрики могут быть `TBD`, но риски обязательны.

### Этап 5: Сборка enriched-okr.md

Собери итоговую таблицу в формате финального артефакта (эталон — `examples/ideal_okr.md`):

```markdown
Проектные цели:
- **OBJ N**: [Название]

Регулярные активности:
- [ACTIVITY]

| OBJ | KR | IMP | Название | Образ результата | Образ действия | Метрики&риски&зависимости |
|-----|-----|-----|---------|-----------------|----------------|--------------------------|
[все KR]
```

### Этап 6: Сохранение + STOP
- Сохрани через `obsidian.vault_write` в `{okr_workspace}/enriched-okr.md`.
- **Выведи и ОСТАНОВИСЬ:**
  ```
  Enriched OKR готов: {okr_workspace}/enriched-okr.md
  KR с образом действия: N из M.
  [УТОЧНИТЬ] в образах действия: K пунктов.
  ── СТОП ── PO: проверьте образы действия и метрики.
  После подтверждения → /okr-validate {quarter}
  ```

## Запреты
1. НЕ выдумывай шаги образа действия — только из context-pack, аналогичных KR или ответов PO.
2. НЕ заполняй метрики-цифры без источника (IMP=9 + «100 RPS» без обоснования → `[УТОЧНИТЬ]`).
3. [ACTIVITY]-KR не требуют детального образа действия.
