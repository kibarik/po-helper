## Использование

```
/cindex-crawl <space_key>
```

Вход: ключ пространства Confluence (напр. `PROD` — иллюстративный пример,
реальное пространство — из `.claude/domain-profile.md`). Выход:
`artefacts/confluence/inventory.json`.

## Важно

**Роль: Crawler.** Ты только обходишь дерево страниц пространства и собираешь
метаданные + сырой контент — **не осмысляешь, не оцифровываешь, не
классифицируешь**. Это первая стадия пайплайна (`.claude/skills/confluence-indexator/SKILL.md`);
после неё — обязательный **STOP**: PO подтверждает scope/фильтры пространства,
прежде чем контент пойдёт в `/cindex-comprehend`.

Полные принципы (живой доступ, честные отказы, идемпотентность) — в
SKILL.md, разделы «Предусловия» и «Принципы» (пункт 6). Этот файл их не
дублирует.

## Инструкция для LLM

### Этап 1: Проверка Confluence MCP

1. Прочитай `.claude/skills/confluence-indexator/SKILL.md`, раздел
   «Предусловия».
2. Через `ToolSearch` проверь, доступны ли в текущей сессии инструменты
   обхода пространства (листинг дерева страниц) и чтения содержимого
   страницы через Confluence MCP. **Не угадывай и не хардкодь** конкретное
   имя MCP-инструмента — какой сервер настроен, определяется на месте.
3. Если подходящего MCP нет — **СТОП**:
   ```
   ── СТОП ── Confluence MCP не обнаружен в .mcp.json.
   Добавь сервер в mcpServers.<name>, перезапусти сессию Claude Code,
   убедись, что инструменты обхода пространства и чтения страницы видны
   через ToolSearch, и повтори /cindex-crawl <space_key>.
   ```
   Не имитируй и не подделывай контент Confluence при отсутствии MCP.

### Этап 2: Проверка прошлого inventory

Если `artefacts/confluence/inventory.json` уже существует — прочитай его
(`crawled_at`, число страниц). Реши по STOP-логике из Этапа 5, будешь ли
перезаписывать: любая ошибка в процессе обхода (Этап 4) **не должна**
затронуть уже существующий файл — пиши в новый файл (или в память) и
переименуй/перезапиши только по успешному завершению обхода целиком.

### Этап 3: Обход пространства

Для `<space_key>`:
1. Получи дерево страниц пространства (иерархия ancestors/parent-child).
2. Для каждой страницы собери поля (дизайн-спека §6):
   - `id` — идентификатор страницы Confluence.
   - `title` — заголовок.
   - `ancestors` — цепочка родителей (заголовки или id, по данным MCP).
   - `labels` — метки страницы.
   - `links` — id страниц, на которые страница явно ссылается.
   - `url` — прямая ссылка на страницу.
3. Прочитай сырое содержимое каждой страницы (нужно `/cindex-comprehend` на
   следующей стадии) — храни его вместе с записью страницы либо в
   отдельном кэше, если MCP отдаёт контент большим объёмом; главное, чтобы
   `/cindex-comprehend` мог получить текст по `id`.

### Этап 4: Обработка ошибок и краевых случаев

- **401 / rate-limit / MCP недоступен посреди обхода** — **СТОП** сразу,
  не пиши частичный `inventory.json` поверх старого:
  ```
  ── СТОП ── Обход прерван: <причина: 401 / rate-limit / MCP недоступен>.
  Прошлый inventory.json (если был) не тронут. Устрани причину и повтори
  /cindex-crawl <space_key>.
  ```
- **Пустая страница** (нет содержимого) — включи в `inventory.json` с
  пометкой `"note": "empty"`, не исключай молча.
- **Архивная страница** — включи с пометкой `"note": "archived"`.
- Ни одна страница не отбрасывается без записи — полнота обхода важнее
  скорости (SKILL.md, принцип 6 «Живой доступ, честные отказы»).

### Этап 5: Сохранение + STOP

Собери и сохрани `artefacts/confluence/inventory.json`:

```json
{"space":"<space_key>","crawled_at":"<YYYY-MM-DD>",
 "pages":[{"id":"...","title":"...","ancestors":["..."],
           "labels":["..."],"links":["..."],"url":"https://..."}]}
```

Выведи и **ОСТАНОВИСЬ**:

```
Обход завершён: artefacts/confluence/inventory.json
Пространство: <space_key>. Страниц: <N> (пустых: <a>, архивных: <b>).

── СТОП ── PO: подтверди scope — это те страницы, что нужно индексировать?
Если нужно сузить/расширить (фильтр по ancestors/labels) — уточни, обход
можно перезапустить. Дальше: /cindex-comprehend <space_key>.
```
